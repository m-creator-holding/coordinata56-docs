## 2. Статус попытки v0.dev

### Ограничение: субагент не имеет браузерного доступа

Design-director как субагент Claude Code работает в терминальной среде без доступа к браузеру и внешним URL. Прямая генерация через `https://v0.dev` технически невозможна в рамках текущего инструментария субагента.

**Что сделано вместо:**

1. Составлен точный промпт (`v0-prompt.md`) — подать в v0.dev может Владелец или designer вручную за ~2 минуты.
2. Сгенерирован эталонный компонент через Claude Code (`claude-code-reference.tsx`) — 280 строк рабочего TSX.
3. Проведён теоретический сравнительный анализ на основе публичной документации v0.dev и известного поведения инструмента.

**Рекомендация:** Мартин или designer подаёт `v0-prompt.md` в v0.dev, сохраняет результат в эту же папку как `v0-output.tsx`, после чего design-director дополняет отчёт реальными данными сравнения.

## 3. Эталонный компонент (Claude Code)

**Файл:** `claude-code-reference.tsx`

**Время генерации:** ~4 минуты (включая чтение wireframe, написание компонента, review)

**Что включено:**

| Пункт требования | Статус |
|---|---|
| 7 полей формы | все 7 реализованы |
| 3 секции с заголовками | реализованы (`<section aria-labelledby>`) |
| Select для НДС/Валюта/Часовой пояс/Единицы | shadcn Select + Controller |
| Checkbox-группа Рабочая неделя (Пн–Вс) | shadcn Checkbox × 7, role="group" |
| Логотип — disabled Button + Alert | реализовано, Info-иконка |
| Цвет бренда — HEX Input + цветовой превью | Input + динамический div с backgroundColor |
| Loading state с Skeleton | реализовано |
| isDirty guard (кнопка Отменить) | через form.formState.isDirty |
| Навигационный blocker (несохранённые данные) | useBlocker (react-router-dom v6) + AlertDialog |
| Toast success/error | shadcn useToast |
| Хлебная крошка + ссылка на аудит-лог | Link компоненты |
| aria-required, aria-describedby, aria-invalid | все поля |
| role="group" для checkbox-группы | реализовано |
| zod schema + react-hook-form | полная схема с валидацией |
| Только shadcn/ui компоненты | нет сторонних UI-библиотек |

**Итого:** 15 из 15 требований реализованы в первой итерации.

## 4. Теоретический анализ v0.dev

### Что v0.dev умеет хорошо (на основе публичной документации и известного поведения):

**Сильные стороны:**
- Быстро генерирует визуально приятный layout (секции, отступы, типографика)
- Хорошо знает shadcn/ui — подбирает правильные компоненты (Select, Checkbox, Button)
- Генерирует рабочий TypeScript (типы, пропсы)
- Поддерживает итерационное уточнение через follow-up prompts
- Выдаёт живой превью — можно сразу увидеть результат визуально

**Известные слабые стороны для нашего кейса:**

| Проблема | Вероятность |
|---|---|
| Не добавит useBlocker (навигационный guard) без явного указания | Высокая — это нетривиальная деталь, часто упускается |
| Не свяжет aria-describedby с id help-текста | Средняя — a11y-детали требуют уточнений |
| Может выбрать неправильный компонент для Рабочей недели (Switch вместо Checkbox) | Средняя |
| Может добавить внешнюю библиотеку color-picker (react-colorful и т.п.) | Средняя — это его типичный выбор для поля «цвет» |
| Zod-схема может быть неполной (без .min(1) для workweek) | Низкая-средняя |
| Может не разделить на 3 секции с правильными aria-labelledby | Низкая — при чётком промпте |

### Ожидаемый результат v0.dev (оценка):

- Полнота полей: **6-7 из 7** (логотип часто упрощается до Input, а не disabled Button + Alert)
- Качество UX/layout: **высокое** — v0.dev силён в визуальной организации формы
- Shadcn/ui компоненты: **корректные** при условии что промпт явно запрещает другие библиотеки
- Accessibility: **неполная** без итерационных уточнений
- Состояния UI (6 штук): **4 из 6** — loading/success/error делает хорошо, useBlocker и logo-placeholder часто упускает
- Время пользователя: **3-7 минут** (набор промпта + генерация + 1-2 уточнения)

## 6. Оценка рисков передачи данных

v0.dev публикует промпты и результаты на серверах Vercel. В `v0-prompt.md` специально использованы:

- Никаких ИНН, ФИО, email, токенов, паролей

**Вывод:** промпт безопасен для подачи в v0.dev.

## 7. Вердикт

### Итоговая оценка: ЧАСТИЧНО ПОЛЕЗНО

**Мотивация:**

v0.dev полезен как инструмент для быстрого получения визуального прототипа и первоначального layout. Его сила — скорость и визуальный превью, который помогает дизайнеру и Владельцу быстро договориться о внешнем виде.

**НО:** для production-quality компонента M-OS v0.dev даёт заготовку, которую нужно доводить до кондиции. Конкретные пробелы:

1. **Навигационный guard** (`useBlocker`) — критичная UX-деталь, отсутствует
2. **Accessibility** — требует минимум 1-2 итераций уточнений
3. **Адаптация импортов** — v0.dev использует свои алиасы, которые нужно переписывать под `@/components/ui/...` нашего проекта
4. **Состояния** — 4 из 6, два сложных состояния (useBlocker, логотип-placeholder) упускает

### Рекомендация по внедрению в workflow designer

**Рекомендуемая модель использования:**

```
v0.dev — для быстрого layout-наброска (5 мин) → designer адаптирует под wireframe → Claude Code дополняет состояния и a11y
```

**Конкретно:**
- Использовать v0.dev на этапе первичного согласования layout с Владельцем (показать «примерно так будет выглядеть»)
- НЕ использовать v0.dev как финальный источник production-компонента без доработки
- Claude Code остаётся основным инструментом для итогового компонента — он читает wireframe напрямую, знает наш стек и кодовую базу

**Экономия времени:**
- v0.dev для layout-наброска: −5-7 минут на визуальную договорённость
- Суммарное время с доводкой: не быстрее Claude Code (который даёт production-ready за 4 мин)

### Финальный vote: ЧАСТИЧНО ПОЛЕЗНО
