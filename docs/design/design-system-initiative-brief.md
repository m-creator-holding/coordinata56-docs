## Контекст

Владелец поставил задачу: создать передовую design-систему M-OS — единую точку истины для всех визуальных и UX-решений системы. Финальный документ `docs/design/design-system-initiative.md` должен быть готов к 25 апреля 2026.

Design-система — это не просто набор цветов и кнопок. Это:
- Соглашения о компонентах, состояниях UI и поведении интерфейса
- Технические ограничения и возможности стека (React, shadcn/ui, Tailwind)
- Стандарты производительности, доступности, тестируемости
- Правила документирования дизайн-решений как ADR

Чтобы design-система была полезной для всей команды — нам нужен input от каждого направления. Вы лучше знаете свои ограничения, возможности и боли, чем дизайн-отдел. Один час вашего времени сейчас сэкономит месяц переделок потом.

## Что нужно от каждого директора

Ответ в свободной форме на вопросы ниже. Не нужно описывать всё — выбирайте то, что считаете критичным для вашего направления. Одна страница текста максимум.

---

### Backend-director

**Тема: UI-паттерны для backend-состояний**

Вопросы:
1. Какие состояния данных backend должен явно транслировать в UI? (loading, error, success — понятно; что ещё?)
2. Как должен выглядеть optimistic update в UI с точки зрения backend: когда «откатывать» визуальное состояние, если запрос провалился?
3. Какие типы ошибок бэкенда требуют разных UI-реакций? (например: 422 validation — подсветить поля; 403 — показать 403-экран; 500 — toast с кнопкой «Сообщить»)
4. Есть ли специфические требования к формату ответов API, которые влияют на то, как frontend должен отображать данные? (пагинация, вложенность, enum-значения)

---

### Frontend-director

**Тема: Возможности и ограничения React + shadcn/ui + Tailwind 2026**

Вопросы:
1. Какие компоненты shadcn/ui на текущем стеке хорошо работают для сложных UX-сценариев M-OS? (drag-and-drop, большие таблицы, матрицы прав)
2. Какие компоненты shadcn/ui проблемные — и как планируете с этим работать?
3. Есть ли в Tailwind 2026 возможности, которые стоит закрепить в design-системе как стандарт? (анимации, кастомные токены, темизация)
4. Какие технические ограничения frontend должен design учитывать при проектировании экранов? (максимальное число колонок в таблице, вложенность компонентов)
5. Какой механизм темизации планируете (dark mode, brand colors per-company)?

---

### Infra-director

**Тема: Производительность и observability UI**

Вопросы:
1. Какой performance budget для UI считаете разумным для M-OS MVP? (LCP, TTI, bundle size)
2. Как планируете мониторить производительность фронтенда в production? (RUM, Sentry, кастомные метрики)
3. Есть ли инфраструктурные ограничения, влияющие на UI? (CDN, кэширование статики, размер ответов API)
4. Как должны выглядеть UI-индикаторы состояния системы? (downtime-баннер, деградация функций, maintenance mode)

---

### Quality-director

**Тема: Тестируемость UI и стандарты доступности**

Вопросы:
1. Какой уровень WCAG планируете как минимум для M-OS? (AA предполагается — есть специфические требования?)
2. Какой инструмент для visual regression testing планируете? (Chromatic, Percy, playwright screenshots)
3. Что должен design обязательно включать в wireframes, чтобы QA могли тестировать UI без дополнительных уточнений? (aria-labels, data-testid-конвенции)
4. Есть ли паттерны, которые при текущем стеке сложно тестировать — и которых design должен избегать?

---

### Governance-director

**Тема: Дизайн-решения как ADR и brand guidelines**

Вопросы:
1. Нужен ли формальный тип ADR для дизайн-решений? (отдельная серия ADR-D01+, или достаточно design-system-initiative.md как живого документа)
2. Как должен выглядеть процесс пересмотра дизайн-токенов? (кто предлагает, кто утверждает, как документируется)
3. Есть ли требования к brand guidelines как нормативному документу? (логотип, цвета, шрифты как ADR или как отдельный артефакт)
4. Какой уровень review нужен для изменения design-системы? (только design-director, или требуется governance-approve)

---

### Tech-writer

**Тема: Документирование дизайн-системы и docs-as-code**

Вопросы:
1. В каком формате лучше документировать design-систему для команды? (markdown в репозитории, Storybook, комбинация)
2. Как связывать wireframes с User Stories и ADR? (ссылки, теги, отдельный индекс)
3. Какие требования к UX-копирайтингу (микрокопи) планируете стандартизировать? (формулировки ошибок, кнопок, подтверждений)
4. Нужен ли глоссарий UX-терминов для команды (единый язык: что такое «Sheet» vs «Dialog» vs «Drawer» в контексте M-OS)?
