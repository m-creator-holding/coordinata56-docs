## Критичные ограничения

**1. Storybook — слишком дорог для MVP-фазы.**
Storybook требует постоянной поддержки: каждый новый компонент нужно вручную документировать историями, следить за актуальностью снапшотов, обновлять при изменениях пропсов. В нашей команде субагентов нет выделенной роли «frontend-documenter». Это значит: Storybook быстро устаревает и становится источником дезинформации, а не помощи. Рекомендация: исключить из MVP design-системы. Вернуться к вопросу в M-OS-2 или M-OS-3, когда будет стабильный компонентный слой.

**2. Два языка в команде — технический и бизнесовый.**
Владелец читает документацию. Любой UX-термин (Sheet, Drawer, Dialog) должен быть объяснён простыми словами, иначе это барьер при ревью дизайна. Глоссарий — обязателен, не опционален.

**3. Документация должна жить рядом с кодом.**
По нашей практике (см. туториалы в `docs/knowledge/tutor/` и строительный глоссарий в `docs/knowledge/construction/04-glossary.md`): документ, который лежит отдельно от репозитория, неизбежно устаревает. Design-система должна быть в `docs/design-system/`, под version control, рядом с ADR.

## Рекомендации

### Вопрос 1. Формат документирования design-системы

Трёхуровневая структура, всё в Markdown в репозитории:

```
docs/design-system/
  README.md              — навигатор: что здесь и как пользоваться
  tokens/
    colors.md            — цветовые токены (имя → значение → когда использовать)
    typography.md        — шрифты, размеры, отступы
    spacing.md           — сетка отступов
  components/
    button.md            — описание компонента: варианты, состояния, микрокопи
    dialog.md
    sheet.md
    ...
  patterns/
    empty-states.md      — паттерн «пустой экран»
    error-states.md      — паттерн «ошибка»
    confirmations.md     — паттерн «подтверждение действия»
  glossary.md            — UX-термины на русском
  microcopy.md           — справочник формулировок
```

Структура следует Diátaxis: `tokens/` и `components/` — Reference; `patterns/` — How-to.

Mermaid-диаграммы — для иллюстрации состояний компонента.

### Вопрос 2. Связь wireframes с User Stories и ADR

Система перекрёстных ссылок без отдельного индекса:

- В каждом файле компонента — раздел `## Контекст использования` со ссылками на User Stories
- В каждой User Story — раздел `## Дизайн-артефакты` с именем wireframe и компонента
- В ADR — раздел `## Затронутые компоненты UI` при визуальном решении
- Wireframes в `docs/design-system/wireframes/` с именованием по User Story: `US-042-house-list.md`

### Вопрос 3. Стандарты UX-копирайтинга

Обязательные разделы в `docs/design-system/microcopy.md`:

**Ошибки.** «Что пошло не так» + «Что сделать». Без технических кодов. Пример: вместо «Error 422» — «Заполните обязательные поля: Название, Сумма».

**Кнопки действий.** Глагол + объект. «Сохранить черновик», «Удалить договор», «Согласовать платёж».

**Подтверждения необратимых действий.** Обязательно называть, что именно удаляется. «Удалить договор №КП-2026-014? Это действие необратимо. Платежи по договору сохранятся.»

**Пустые состояния.** Три элемента: объяснение + причина + действие. «Договоров пока нет. Добавьте первый договор. [Добавить договор]».

Дополнительно для M-OS: все формулировки в мужском роде: «Договор сохранён», «Платёж отклонён».

### Вопрос 4. Глоссарий UX-терминов

Обязателен. Минимальный набор:

| Термин (EN) | Русское название | Когда использовать |
|---|---|---|
| Dialog | Диалог | Блокирующий запрос подтверждения |
| Modal | Модальное окно | Форма или контент поверх страницы с тёмным фоном |
| Sheet | Шторка | Панель, выезжающая снизу/сбоку. Не блокирует контент |
| Drawer | Боковая панель | Постоянная или временная панель сбоку. Шире Sheet |
| Toast | Уведомление | Временное сообщение в углу |
| Tooltip | Подсказка | Появляется при наведении |
| Banner | Баннер | Постоянная плашка (например, «Система на техобслуживании») |
| Empty state | Пустой экран | Состояние страницы когда данных нет |
| Loading state | Состояние загрузки | Компонент ожидает ответа сервера |
| Skeleton | Скелетон | Анимированные заглушки при загрузке |

## Вопросы к design-director

1. **Figma vs code-first.** Wireframes живут в Figma или сразу в коде? Влияет на формат хранения.
2. **Кто владеет глоссарием?** Предлагаю: tech-writer создаёт и ведёт, design-director утверждает термины.
3. **Язык компонентов в коде.** В React — английские имена (Button, Sheet). В задачах команды — русские («шторка») или английские? Нужно однозначное соответствие.
4. **Приоритет компонентов для документирования.** Начать с используемых в M-OS-1 «Скелет». Список от design-director поможет не документировать компоненты, которых ещё нет.
