**Версия**: 1.0  
**Дата**: 2026-04-18  
**Автор**: data-analyst (субагент coordinata56)  
**Источник схемы**: ADR 0001 v1.1  
**SQL-файл**: `docs/analytics/queries/owner-top10-reports.sql`

---

## Общие соглашения

- Все денежные поля в БД хранятся в **копейках** (bigint). В отчётах делим на 100 — результат в рублях.
- Мягкое удаление: везде применяется фильтр `deleted_at IS NULL` для таблиц, поддерживающих soft-delete.
- Запросы предназначены только для чтения (SELECT). Выполнять на реплике или dev-копии БД.
- Временная зона БД — UTC; при отображении переводить в MSK (UTC+3) на слое отчётности.

## R-01: Cash Burn (расход наличности)

**Бизнес-вопрос**: сколько денег фактически ушло из кассы и в каком темпе?

**Таблицы**: `payments`, `contracts`

**Метрика**: сумма платежей в статусе `approved` (финальный положительный статус) и `pending` (ожидают одобрения), сгруппированная по месяцу и статусу.

**Колонки результата**:
| Колонка | Значение |
|---|---|
| `month` | Первое число месяца (усечение по дате платежа) |
| `status` | Статус платежа (approved / pending) |
| `payments_count` | Число платежей за месяц |
| `total_rub` | Сумма к выплате, руб. |
| `avg_payment_rub` | Средний платёж, руб. |
| `max_payment_rub` | Максимальный разовый платёж, руб. |

**Допущения**: paid = `approved`. В схеме нет отдельного статуса PAID — approved является окончательным.  
**Ограничения**: не учитывает возвраты (отдельная модель не предусмотрена в v1).

## R-02: Дома по стадиям

**Бизнес-вопрос**: где находится каждый из 85 домов прямо сейчас?

**Таблицы**: `houses`, `stages`, `house_types`

**Метрика**: количество домов на каждой стадии строительства с разбивкой по типу дома.

**Колонки результата**:
| Колонка | Значение |
|---|---|
| `stage_name` | Название стадии (или "Не назначена") |
| `stage_order` | Порядковый номер стадии |
| `houses_count` | Количество домов на стадии |
| `pct_of_total` | Доля от всех активных домов, % |
| `house_type_name` | Тип дома |

**Допущения**: `current_stage_id = NULL` выводится как "Не назначена" (order_index = -1, сортируется последним).  
**Ограничения**: один дом = одна текущая стадия; история прошлых стадий — в R-09.

## R-03: Просроченные платежи

**Бизнес-вопрос**: какие договоры просрочены и сколько денег зависло?

**Таблицы**: `contracts`, `contractors`, `payments`, `houses`

**Метрика**: активные контракты с `end_date < CURRENT_DATE`, у которых есть платежи в статусе `draft` или `pending`.

**Колонки результата**:
| Колонка | Значение |
|---|---|
| `contract_number` | Номер договора |
| `contractor_name` | Краткое наименование подрядчика |
| `end_date` | Дата окончания договора |
| `overdue_days` | Дней просрочки |
| `pending_payments_count` | Число незакрытых платежей |
| `pending_amount_rub` | Сумма незакрытых платежей, руб. |
| `house_plot` | Номер участка (если договор на конкретный дом) |

**Допущения**: "просрочен" = contract.status='active' И end_date < сегодня. Cancelled/completed — исключены.  
**Ограничения**: в схеме нет таблицы плановых дат выплат (payment schedule); только факт по статусу платежа.

## R-04: Отставание от плана

**Бизнес-вопрос**: какие дома строятся медленнее остальных?

**Таблицы**: `house_stage_history`, `houses`, `stages`, `house_types`

**Метрика**: время (в днях) нахождения на текущей стадии vs медиана по всем домам на той же стадии. Порог отставания: > 1.5 × медиана.

**Колонки результата**:
| Колонка | Значение |
|---|---|
| `plot_number` | Номер участка |
| `house_type` | Тип дома |
| `current_stage` | Текущая стадия |
| `days_on_stage` | Дней на текущей стадии |
| `median_days_on_stage` | Медиана по стадии среди всех домов |
| `days_behind_median` | Отставание от медианы (дни) |

**Допущения**: берём последнюю запись в `house_stage_history` по каждому дому как "текущую". Порог 1.5× — стандартный в строительной аналитике.  
**Ограничения**: в схеме нет поля `planned_end_date` на стадию; медиана — суррогатный план. При добавлении нормативных сроков в схему — заменить на прямое сравнение.

## R-05: Подрядчики в спорах

**Бизнес-вопрос**: с кем из подрядчиков проблемные отношения?

**Таблицы**: `contractors`, `contracts`, `payments`

**Метрика**: подрядчики с cancelled-договорами или с active-договорами без единого платежа.

**Колонки результата**:
| Колонка | Значение |
|---|---|
| `contractor_name` | Краткое наименование |
| `inn` | ИНН |
| `contractor_category` | Категория подрядчика |
| `contracts_total` | Всего договоров |
| `cancelled_count` | Число аннулированных |
| `active_no_payment_count` | Active-договоров без оплат |
| `total_contract_value_rub` | Общая сумма договоров, руб. |

**Допущения**: "спор" = признак конфликта; прямой таблицы disputes в схеме v1 нет. Два суррогата: (1) cancelled — явный разрыв; (2) active без оплат — замороженные отношения.  
**Ограничения**: не отражает частичные выплаты; для полной картины нужен процент закрытия суммы договора.

## R-06: Топ-10 расходов по категориям

**Бизнес-вопрос**: какие статьи расходов поглощают больше всего денег?

**Таблицы**: `payments`, `contracts`, `budget_plan`, `budget_categories`

**Метрика**: сумма approved-платежей, сгруппированная по статье бюджета, с долей план/факт.

**Колонки результата**:
| Колонка | Значение |
|---|---|
| `category_name` | Статья расходов |
| `payments_count` | Число платежей |
| `fact_spent_rub` | Фактически потрачено, руб. |
| `plan_amount_rub` | Плановый бюджет по категории, руб. |
| `fact_vs_plan_pct` | Исполнение бюджета, % |

**Допущения**: связь платёж → категория строится через `budget_plan` по (project_id, house_id, stage_id) контракта. Платежи без записи в budget_plan попадают в NULL-категорию.  
**Ограничения**: агрегация на уровне всего проекта; для drill-down по дому/стадии — параметризовать запрос.

## R-07: План/Факт бюджета

**Бизнес-вопрос**: где мы перерасходуем, а где укладываемся в план?

**Таблицы**: `budget_plan`, `projects`, `stages`, `budget_categories`, `payments`, `contracts`

**Метрика**: план из `budget_plan` vs фактические расходы (approved-платежи) с отклонением в рублях и процентах на разрезе проект × стадия × статья.

**Колонки результата**:
| Колонка | Значение |
|---|---|
| `project_name` | Проект |
| `stage_name` | Стадия |
| `category_name` | Статья расходов |
| `houses_planned` | Число домов с планом |
| `plan_rub` | Плановый бюджет, руб. |
| `fact_rub` | Фактические расходы, руб. |
| `deviation_rub` | Отклонение (+ перерасход, − экономия), руб. |
| `deviation_pct` | Отклонение, % от плана |

**Допущения**: для строк без фактических расходов `fact_rub = 0`. NULL в `deviation_pct` означает отсутствие плана по строке.  
**Ограничения**: агрегация не учитывает внутрипроектные переносы бюджета между статьями — модель v1 их не поддерживает.

## R-08: Загрузка прораба

**Бизнес-вопрос**: равномерно ли распределена нагрузка между прорабами?

**Таблицы**: `houses`, `users`, `stages`

**Метрика**: количество активных домов под каждым `responsible_user_id` и средний возраст домов в работе.

**Колонки результата**:
| Колонка | Значение |
|---|---|
| `foreman_email` | Email прораба |
| `active_houses_count` | Домов в работе |
| `distinct_stages_count` | Разных стадий одновременно |
| `stages_list` | Перечень стадий через запятую |
| `avg_house_age_days` | Средний возраст домов (дни с создания) |

**Допущения**: "загрузка" = число домов; нет данных о трудозатратах часов. Колонка `responsible_user_id` в `houses` идентифицирует ответственного.  
**Ограничения**: не учитывает прорабов без назначенных домов (LEFT JOIN не добавлен намеренно — они не имеют загрузки).

## R-09: Средний срок этапа vs план

**Бизнес-вопрос**: какие стадии строительства систематически затягиваются?

**Таблицы**: `house_stage_history`, `houses`, `stages`

**Метрика**: для каждой стадии — медиана и среднее фактических длительностей (завершённых и текущих).

**Колонки результата**:
| Колонка | Значение |
|---|---|
| `stage_name` | Название стадии |
| `total_entries` | Всего записей (дом-стадия) |
| `completed_count` | Завершённых |
| `avg_duration_days` | Среднее время на стадии, дни |
| `median_duration_days` | Медиана (все), дни |
| `median_completed_days` | Медиана только завершённых, дни |
| `min_days` / `max_days` | Мин/макс, дни |

**Допущения**: для незавершённых стадий длительность = дни с `started_at` по сегодня (нижняя оценка). "Плановый срок" как норматив отсутствует в схеме v1 — медиана завершённых служит суррогатом.  
**Ограничения**: при малом числе завершённых домов медиана нерепрезентативна. На старте проекта доверять только avg по незавершённым.

## R-10: Контракты без оплат

**Бизнес-вопрос**: какие активные договоры подписаны, но ни разу не оплачивались?

**Таблицы**: `contracts`, `contractors`, `houses`, `stages`

**Метрика**: active-договоры без единой записи в `payments` (независимо от статуса платежа).

**Колонки результата**:
| Колонка | Значение |
|---|---|
| `contract_number` | Номер договора |
| `contract_subject` | Предмет договора |
| `signed_at` | Дата подписания |
| `contract_amount_rub` | Сумма договора, руб. |
| `contractor_name` | Подрядчик |
| `house_plot` | Участок (или "—") |
| `contract_stage` | Стадия договора |
| `days_since_signed` | Дней с момента подписания |

**Допущения**: "без оплат" = ноль записей в payments, любой статус. Чем больше `days_since_signed` при нулевых платежах — тем выше приоритет проверки.  
**Ограничения**: не учитывает договоры с корректно заведёнными будущими сроками аванса — такая логика потребует таблицы payment schedule.

## Рекомендации по запуску

1. Выполнять на read-replica или dev-копии базы.
2. Для R-04 и R-09: при числе домов < 10 на стадии медиана нерепрезентативна — добавить фильтр `COUNT(*) >= 5`.
3. R-07 возвращает все строки budget_plan — при большом числе домов добавить WHERE по project_id.
4. Нетривиальные запросы (R-04, R-07) используют оконные функции и CTE — для PostgreSQL 16 без ограничений.
