## Структурная особенность Фазы 3: три батча

Фаза закрывается **поочередным закрытием трёх батчей** (A → B → C). Каждый батч проходит мини-DoD. Основной DoD ниже проверяется только после закрытия всех трёх.

### Мини-DoD батча (выполняется для каждого A, B, C)

- [ ] Все роутеры батча реализованы согласно ADR 0004
- [ ] Swagger: все эндпоинты имеют `summary`, `description`, `response_model`, примеры
- [ ] RBAC: каждый write-эндпоинт имеет зависимость `require_role(...)` и тест на 403 от чужой роли
- [ ] Аудит-лог: каждая write-операция производит запись в `audit_log` (проверено интеграционным тестом)
- [ ] Формат ошибок соответствует ADR 0005
- [ ] Пагинация работает согласно ADR 0006
- [ ] Soft-delete семантика: `DELETE` выставляет `deleted_at`, `GET list` не возвращает удалённые без флага `?include_deleted=true`
- [ ] Интеграционные тесты: ≥1 happy + ≥1 403 + ≥1 422 на каждый эндпоинт
- [ ] Reviewer `approve` на staged diff (до коммита — §1 регламент v1.3)
- [ ] Ретро-заметка по батчу в `docs/knowledge/retros/phase_3_batch_X_notes.md` (короткая, не полная ретро)

## Зелёный билет Фазы 3 (обязательно)

### 1. Покрытие сущностей
- [ ] 15 бизнес-роутеров работают (см. список в scope §1)
- [ ] `AuditLog` read-only эндпоинт (list + get) существует, write недоступен
- [ ] `User` admin-CRUD реализован (create user, deactivate, список)

### 2. Кросс-срезовые компоненты
- [ ] Базовый CRUD-миксин / generic-слой согласно ADR 0004 — один на весь проект
- [ ] Формат ошибок унифицирован (ADR 0005) — проверка: 5 ручных проверок в Swagger, формат идентичен
- [ ] Пагинация унифицирована (ADR 0006) — все list-эндпоинты отвечают одним envelope
- [ ] Аудит-лог унифицирован (ADR 0007) — ни одна write-операция не проходит мимо

### 3. Безопасность (сверх общего DoD)
- [ ] Нет эндпоинта без `Depends(get_current_user)` (кроме health-check и swagger)
- [ ] Ни один эндпоинт не возвращает `password_hash`, `jwt_secret` или другие секретные поля (проверка через Pydantic response_model)
- [ ] SQL-инъекции невозможны: все запросы через ORM, нет `text()` с f-string
- [ ] Массовые операции (bulk budget upload) имеют лимит размера (предлагается 1000 строк за запрос)
- [ ] Rate-limiting на уровне роутера — **отложено** до Фазы 9 (зафиксировать в Open Questions)

### 4. Иммутабельность и бизнес-правила
- [ ] `Payment` со статусом `approved` нельзя редактировать / удалять (тест обязателен)
- [ ] `HouseStageHistory` — append-only (нет DELETE / UPDATE эндпоинта)
- [ ] `AuditLog` — append-only (нет DELETE / UPDATE эндпоинта)
- [ ] Soft-deleted сущности нельзя «случайно» создать заново с тем же unique-ключом (решение зафиксировать в ADR 0004)

### 5. Тесты (сверх общего)
- [ ] Матрица RBAC: для каждого эндпоинта ×4 роли = явный тест-кейс (параметризованный)
- [ ] Тесты на soft-delete семантику на каждой сущности с `SoftDeleteMixin`
- [ ] Тесты на пагинацию (offset=0, offset=last, limit=1, limit=max)
- [ ] Тесты на формат ошибок: 404, 403, 422, 409 — каждый возвращает единую схему

### 6. Документация
- [ ] `docs/api/phase-3-endpoints.md` — сводная таблица всех эндпоинтов с ролями
- [ ] Swagger экспортирован в `docs/api/openapi-phase-3.json` (для ретроспективной сверки)
- [ ] README обновлён: как тестировать эндпоинты через Swagger

### 7. Ревью (сверх общего)
- [ ] Каждый из трёх батчей прошёл независимое ревью
- [ ] Финальное ревью на всю фазу — проверка единообразия (одинаковый ли CRUD-паттерн в 15 роутерах, одинаковы ли ошибки)

### 8. Согласование с Владельцем
- [ ] Мартин прошёл вручную через Swagger ≥3 сценария (создание дома, ввод платежа, ввод закупки)
- [ ] Мартин подтвердил формат ошибок и пагинации на живых эндпоинтах
- [ ] Явное согласование перехода к Фазе 4

## Жёлтый билет (рекомендуется)

- Покрытие ветками ≥85% (цель ≥90% для service-слоя)
- Postman/Bruno-коллекция для ручных проверок
- Нагрузочный замер `bulk budget upload` (чтобы понимать, выдерживает ли 1000 строк)
- Логирование длительности каждого запроса (p50/p95)

## Риск-митигации из scope

| Риск | Митигация в DoD |
|---|---|
| R1: рутина на 16 моделей | ADR 0004 задаёт единый паттерн; ревью на единообразие |
| R2: кросс-модельные зависимости | Строгий порядок батчей A→B→C |
| R3: аудит-лог как bottleneck | Замер после Батча A; решение до Батча B (пункт в мини-DoD А) |
| R4: Swagger перегружен | Tag по батчам, обязательные `summary` и `description` |

---

## Правила перехода между батчами

1. Батч A не закрыт — Батч B не начинается.
2. Кросс-срезовые компоненты (CRUD-миксин, формат ошибок, аудит) **замораживаются** после закрытия Батча A. Изменения в них в Батчах B/C — только через отдельный ADR amendment.
3. Если в Батче B/C обнаруживается проблема в кросс-срезовом компоненте — Координатор останавливает батч, делает фикс в рамках Батча A (отдельный коммит), возобновляет батч.
