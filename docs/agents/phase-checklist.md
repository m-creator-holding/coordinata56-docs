# Definition of Done — критерии закрытия фазы

**Версия:** 1.0
**Дата:** 2026-04-15
**Статус:** действующий
**Автор:** Координатор

Фаза дорожной карты считается **закрытой**, только если выполнены ВСЕ пункты ниже. Без зелёного билета по каждому пункту фаза остаётся открытой, переход к следующей запрещён.

## Зелёный билет (обязательно)

### 1. Код и реализация
- [ ] Реализованы все функциональные требования критерия перехода (из `ROADMAP.md`)
- [ ] Все тесты зелёные: `pytest -q && pytest --cov-branch` показывает ≥80% веток
- [ ] Линтеры чистые: `ruff check`, `mypy` без ошибок (warnings допустимы при пометке)
- [ ] Локально приложение запускается без ошибок: `docker compose up`
- [ ] Все ветки `try/except`, `if not authorized`, `if expired` покрыты тестами на 100%

### 2. Архитектура и ADR
- [ ] ADR фазы существует в `docs/adr/` и согласован
- [ ] Все MUST из ADR реализованы (сверка по скилу `adr-compliance-checker`)
- [ ] Заявленные отклонения от ADR зафиксированы как amendment в самом ADR (не только в commit message)
- [ ] Незаявленных отклонений нет

### 3. Безопасность
- [ ] Прогон чек-листа OWASP Top 10 (скил `owasp-top10-checklist`) — без P0/P1
- [ ] Threat-model в `docs/security/` обновлена под изменения фазы
- [ ] Секреты не закоммичены: `git ls-files | xargs grep -l -i "secret\|password\|token\|api[-_]key"` — нет реальных значений
- [ ] Зависимости проверены: `pip-audit` / `npm audit` — без CVE высокого уровня

### 4. Ревью
- [ ] `reviewer` дал вердикт **approve** на финальный commit фазы
- [ ] Если был `request-changes` — все P0/P1 закрыты, повторное ревью прошло

### 5. Тесты (по скилу `test-design-equivalence-classes`)
- [ ] Все классы эквивалентности по входным параметрам перечислены и покрыты
- [ ] Граничные значения покрыты (min−1, min, min+1, max−1, max, max+1)
- [ ] Негативные тесты безопасности (без токена, чужая роль, истёкший токен)
- [ ] Нет «зелёных тестов без assert» или с моками там, где должна быть test-БД

### 6. Документация
- [ ] `docs/knowledge/glossary.md` пополнен новыми терминами фазы
- [ ] `docs/ONBOARDING.md` обновлён, если изменился процесс запуска
- [ ] Swagger/ReDoc актуальны (для фаз с API)
- [ ] `README.md` упоминает новую функциональность

### 7. Юридика (если применимо)
- [ ] `legal` подтвердил отсутствие нарушений 152-ФЗ / 214-ФЗ / 63-ФЗ / ТК РФ для функционала фазы
- [ ] Чек-лист 152-ФЗ обновлён, если добавлены новые поля с PII

### 8. Память Координатора
- [ ] Статус фазы зафиксирован в `~/.claude/projects/-root/memory/project_cottage_mvp_status.md`
- [ ] Извлечённые уроки записаны в `docs/knowledge/lessons.md` (если есть)
- [ ] Ретроспектива фазы сохранена в `docs/knowledge/retros/phase_N_retro.md`

### 9. Git
- [ ] Все коммиты фазы атомарны и имеют осмысленные сообщения
- [ ] `ROADMAP.md` помечен «Фаза N — done» с датой
- [ ] Нет «битых» коммитов (тесты не падают на каждом коммите фазы)

### 10. Согласование с Владельцем
- [ ] Владелец ознакомлен с результатом фазы (отчёт через Telegram)
- [ ] Владелец явно согласовал переход к следующей фазе

---

## Жёлтый билет (рекомендуется, не блокер)

- Покрытие тестами ≥90% строк
- E2E-тесты через Playwright (для фаз с UI)
- Performance-замеры (для фаз с массовыми операциями)
- Security audit от внешнего инструмента (semgrep, bandit)

---

## Процесс закрытия фазы (порядок)

1. `backend-dev` / `frontend-dev` / `db-engineer` — реализация
2. `qa` — тесты по методике
3. `reviewer` — ревью **до** коммита
4. После approve — Координатор делает коммит
5. `security` — финальный security-чек
6. `legal` — финальный legal-чек (если применимо)
7. `tech-writer` — обновление документации
8. Координатор — обновление памяти, ретроспектива, отчёт Владельцу
9. Владелец — формальное согласование закрытия
