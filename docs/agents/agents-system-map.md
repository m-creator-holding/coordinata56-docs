## 1. Общее назначение системы

coordinata56 строится одним Владельцем (Мартин) с нулевым опытом программирования при помощи команды ИИ-субагентов Claude Code. Система субагентов выполняет роль компании-разработчика ПО: у неё есть CEO (Координатор), директора направлений, начальники отделов, исполнители и штабные советники.

**Принципы системы:**
1. Владелец — единственный заказчик. Стратегию и бизнес-решения принимает он.
2. Все задачи от Владельца идут через Координатора — других точек входа нет.
3. Агент активируется только тогда, когда для него есть работа (принцип «нет работы — нет роли»).
4. Умная модель (Opus) не делает рутину; простая модель (Sonnet/Haiku) не принимает серьёзных решений.
5. Каждый повторяющийся дефект порождает правило в регламенте. Система обучается через накопление правил, а не через дообучение моделей.

## 5. Кто что делает (краткие карточки)

### Координатор (L1)

**Роль:** я как CEO проекта.
**Вход:** задачи от Владельца.
**Выход:** декомпозированные задачи Директорам, отчёты Владельцу, коммиты, обновления памяти.
**Делегирует:** всем Директорам и Советникам.
**Кто проверяет:** Владелец.
**Память:** `/root/.claude/projects/-root/memory/`, `docs/knowledge/retros/`, `docs/agents/CODE_OF_LAWS.md`.
**Не делает:** не пишет код задач, не принимает бизнес-решения за Владельца, не ревьюит файлы.

### Директора (L2)

Все Директора на Opus, один паттерн работы: принимают от Координатора задачу уровня M, декомпозируют внутри своего направления, владеют департаментским регламентом, не пишут код и не ревьюят файлы. Общаются между собой по серьёзным вопросам напрямую (ADR, security, legal, блокеры). С Владельцем — только через Координатора.

### Начальники отделов (L3) — три варианта

1. **Классические `*-head`** (`backend-head`, `qa-head`, `review-head`) — тактическое управление 1–3 сотрудниками, первичное ревью их работы. Модель — Sonnet, кроме `review-head` (Opus для глубины финальных ревью).

2. **Governance-auditor** — аудит регламента и поведения. Не имеет подчинённых. Пишет еженедельные отчёты.

3. **R&I scout/analyst** — сенсинг и анализ. Scout — Sonnet (быстрый проход по источникам), Analyst — Opus (глубокий RFC). По регламенту 2026-04-15 обязаны расшифровывать каждую находку простым русским для Владельца.

### Сотрудники (L4)

Исполнители конкретной работы в пределах FILES_ALLOWED. Самопроверка перед сдачей, структурированный отчёт Начальнику (или Координатору, если Начальник/Директор dormant). Не коммитят в git, не общаются с Владельцем, не литералят секретов.

### Советники (штабные, не в иерархии)

Консультируют любой уровень, но не управляют задачами. Вызываются Координатором или Директорами. Главные точки — `architect` (ADR), `analyst` (User Stories), `legal` (юр-риски), `memory-keeper` (память).

## 6. Вход / выход / делегирование / ревью (стандартные правила)

### Вход задачи

1. Владелец в Telegram → Координатор.
2. Координатор определяет tier задачи:
   - **XS** — 1 файл, <1ч → Координатор → Директор → Worker (Директор курирует напрямую при одиночном Worker'е)
   - **S** — 2–10 файлов, один отдел → Координатор → Директор → Head → Worker
   - **M** — модуль, одно направление → Координатор → Директор → Head → Workers
   - **L** — фаза или многонаправленная → Координатор → несколько Директоров → Heads → Workers

**Строгое правило (Владелец 2026-04-16 msg 665):** Координатор передаёт задачи **только Директорам**. Прямой маршрут Координатор → Worker или Координатор → Head запрещён. Исключение — Советники (architect, analyst, legal, construction-expert, tech-writer, memory-keeper, tutor, data-analyst): они вне иерархии, Координатор обращается к ним напрямую как к консультантам.

### Выход задачи

Каждый уровень сдаёт пакет вверх по цепи: Worker → Head → Директор → Координатор → Владелец. Ревью на каждом уровне обязательно.

### Делегирование

Делегирование строго по вертикали: L1 → L2 → L3 → L4. Кросс-вертикальная коммуникация:
- Начальники разных направлений — разрешена для рутинных вопросов
- Директора разных направлений — разрешена для серьёзных
- Сотрудники разных направлений — запрещена всегда

### Ревью

- **Reviewer до `git commit`**, не после (v1.3 §1). Работает на `git diff --staged`.
- **Security** — при подозрительном изменении прав, секретов, внешних вызовов.
- **Governance-auditor** — периодический аудит регламента (еженедельно + после фазы).

## 8. Риски и ограничения по ролям

Ключевые ограничения, которые нельзя нарушать (проверяются governance-auditor):

1. **Никто не общается с Владельцем напрямую, кроме Координатора.** При нарушении — поведенческий аудит, правка в `CLAUDE.md`.
2. **Никто не делает `git commit`, кроме Координатора.** Workers предлагают staged-diff, ревьюер даёт вердикт, Координатор коммитит.
3. **Любая правка регламента — через комиссию Governance.** Быстрый путь — только для срочного `CLAUDE.md`, с post-factum заявкой в 24ч.
4. **Любая схема БД — через Alembic-миграцию.** Только `db-engineer` владеет DDL.
5. **`memory-keeper`** не трогает файлы проекта (только `memory/`). Координатор правит память напрямую только при необходимости консистентности.
6. **`integrator`** не делает первый живой вызов внешнего API без одобрения Владельца — потенциальные расходы.
7. **`devops`** не трогает production без явного одобрения Владельца.
8. **`data-analyst`** пишет только SELECT/репликой; никогда не INSERT/UPDATE в production.
9. **Параллельные субагенты** работают только с явными FILES_ALLOWED/FILES_FORBIDDEN в промпте (v1.3 §5).
10. **Секреты не литералятся нигде** — ни в `src/`, ни в `tests/`, ни в фикстурах. Всегда `secrets.token_urlsafe(...)` или `os.environ.get(...)`.

## 10. Как поддерживать эту карту

1. **При создании нового агента** — сначала карточка по `agent-card-template.md`, потом строка в `agents-map.yaml`, потом обновление `agents-system-map.md` и `agents-diagrams.md`.
2. **При активации dormant-направления** — снять dormant-статус в этом документе, создать директора в `~/.claude/agents/`, обновить регламент направления.
3. **При противоречиях с регламентом** — заявка через комиссию Governance, а не правка «на месте».
4. **Ревизия** — governance-auditor проходит эту карту в рамках еженедельного аудита.
