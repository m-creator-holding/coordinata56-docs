## 1. Проблема, которую закрывает v1.6

### 1.1 Инцидент 2026-04-16

Координатор получил от Владельца задачу «проверь свежие находки». Координатор вызвал `ri-director` с промптом «выполни цикл sensing» — и Директор сам прошёлся по источникам, записал 3 находки, написал отчёт. **Директор сделал работу scout'а, нарушив `regulations/director.md` статьи 9 (Директор не вправе писать код задач) и 15 (не должен подменять Сотрудников).**

Корневая причина — **не Директор ошибся, а Координатор неправильно сформулировал задачу**. Вместо «напиши бриф для ri-scout, мы прогоним цикл» — было «выполни цикл sensing».

### 1.2 Техническое ограничение платформы

При проверке документации Anthropic обнаружено жёсткое ограничение одно-уровневой делегации:

- **Claude Code sub-agents** (https://code.claude.com/docs/en/sub-agents): «This prevents infinite nesting (subagents cannot spawn other subagents)». Субагент не может вызвать другого субагента через инструмент `Agent`.
- **Anthropic Managed Agents API** (https://platform.claude.com/docs/en/managed-agents/multi-agent): «Only one level of delegation is supported: the coordinator can call other agents, but those agents cannot call agents of their own.»

То есть нативной иерархии «Директор сам запустит Начальника, Начальник сам запустит Сотрудника» — в Claude Code нет. Единственный, кто может технически вызвать субагента — Координатор (основной Claude в сессии).

### 1.3 Что это значит для нашего регламента v1.4

Правило v1.4 §5 и CLAUDE.md «Президент → Директор → Начальник → Сотрудник» **остаётся в силе по логике ответственности**: Директор принимает решение делегировать, формулирует бриф, проверяет результат, выносит вердикт. Но **физическое нажатие кнопки «запустить следующий уровень» остаётся за Координатором**. Это и есть паттерн «Координатор-транспорт».

Официальный путь, рекомендованный Anthropic (cite: code.claude.com/docs/en/sub-agents): «If your workflow requires nested delegation, chain subagents from the main conversation».

## 2. Паттерн «Координатор-транспорт»

### 2.1 Кто за что отвечает

| Уровень | За что отвечает (логика) | Что физически делает в Claude Code |
|---|---|---|
| Координатор | Принимает задачу Владельца, определяет tier, поддерживает цепочку | Вызывает Агентов поочерёдно, передаёт между ними брифы и результаты, коммитит, отчитывается Владельцу |
| Директор | Стратегия направления, декомпозиция задачи M, вердикт по работе подчинённых | Получает от Координатора запрос на бриф; пишет бриф; получает от Координатора результат Worker'а; пишет вердикт |
| Начальник отдела | Тактическое управление 1-3 сотрудниками, первичное ревью | Получает от Координатора бриф от Директора; распределяет между Сотрудниками (описание); получает от Координатора результаты; ревьюит и агрегирует |
| Сотрудник | Исполнение в пределах FILES_ALLOWED | Получает от Координатора персонализированный бриф + FILES_ALLOWED; выполняет; отчитывается структурированно |

**Ключевое уточнение:** Директор/Начальник «получают» бриф и «отдают» результат **через Координатора как транспорт**. Они НЕ инициируют вызовы субагентов напрямую.

### 2.2 Стадии одной задачи M-уровня

```
Стадия 0. Принятие
  Владелец → Координатор: задача
  Координатор: определяет направление и tier

Стадия 1. Брифинг (Director → Head)
  Координатор → Директор (Agent call): «напиши бриф для <Head>, задача: <X>»
  Директор возвращает бриф (текст) → Координатор
  Координатор передаёт в след. вызов

Стадия 2. Распределение (Head → Workers)
  Координатор → Head (Agent call): передаёт бриф Директора;
    «распредели по Workers, задай FILES_ALLOWED каждому»
  Head возвращает план распределения (какой Worker что делает) → Координатор

Стадия 3. Исполнение (Workers)
  Координатор → Worker-1 (Agent call): персонализированный промпт + FILES_ALLOWED
  Координатор → Worker-2 (Agent call): аналогично (можно параллельно)
  …
  Каждый Worker возвращает результат → Координатор

Стадия 4. Первичное ревью (Head)
  Координатор → Head (Agent call): «вот результаты Workers, сделай первичное ревью»
  Head возвращает вердикт (approve / request-changes по каждому) → Координатор
  Если request-changes → возврат к Стадии 3

Стадия 5. Финальное ревью (Director)
  Координатор → Директор (Agent call): «вот пакет с ревью Head, финальная приёмка»
  Директор возвращает финальный вердикт → Координатор

Стадия 6. Pre-commit ревью (reviewer субагент отдельно)
  Координатор → reviewer: проверка staged diff
  reviewer возвращает approve / request-changes

Стадия 7. Коммит
  Координатор коммитит, пушит

Стадия 8. Отчёт
  Координатор → Владелец (Telegram): короткий отчёт по результату
  Координатор → память (project_tasks_log.md, статус-файл фазы)
```

Для S / XS задач цепочка короче (могут пропускаться стадии), но стадии Брифинг и Финальное ревью Директора **обязательны** для любой задачи — иначе нарушение §1.1.

### 2.3 Формулировка промпта от Координатора Директору

**Плохо (нарушает §1.1):**
> «Директор R&I, проверь свежие находки по GitHub и Anthropic»

**Хорошо:**
> «Директор R&I, сформулируй бриф для ri-scout: какие источники проверить, что искать, критерии отбора. Бриф должен помещаться в 3-5 предложений. Я передам его scout'у, потом верну тебе его результат на ревью.»

**Плохо:**
> «Backend-director, реализуй CRUD для Contractor»

**Хорошо:**
> «Backend-director, Координатор получил задачу M-уровня: Contractor CRUD. Проведи декомпозицию по регламенту и сформулируй бриф для backend-head: какие шаги, какие файлы, какие ADR применимы, критерии DoD. Я передам бриф backend-head, затем подниму тебе результат для финального ревью.»

### 2.4 Формулировка промпта от Координатора Начальнику отдела

**Плохо:**
> «Backend-head, сделай CRUD для Contractor»

**Хорошо:**
> «Backend-head, вот бриф от backend-director: <текст брифа>. Распредели работу между backend-dev-1/2 (если нужно параллелить) с явными FILES_ALLOWED по каждой подзадаче. Верни мне план распределения + персонализированные промпты для каждого Worker'а.»

### 2.5 Формулировка промпта от Координатора Сотруднику

**Плохо:**
> «Backend-dev, вот задача <описание>»

**Хорошо:**
> «Backend-dev, распределил тебе backend-head. Задача: <персонализированный промпт>.
> FILES_ALLOWED: <список>
> FILES_FORBIDDEN: всё остальное.
> Критерии готовности: <DoD>
> Время: <оценка>
> Отчёт мне структурированный: что сделал, где лежит, какие допущения, что требует внимания.»

## 3. Правила для Координатора

### 3.1 Никогда не формулировать Директору «сделай»

Формулировка, предполагающая исполнение Директором самолично (ключевые слова: «сделай», «выполни», «проверь», «напиши код») — запрещена. Разрешены формулировки, предполагающие декомпозицию и делегирование: «сформулируй бриф», «проведи декомпозицию», «укажи план распределения».

Исключение: Координатор может просить у Директора **документ** (обновление regulations/*, departments/*.md, ADR), так как это и есть работа Директора на 25% времени «руководство» и 75% «улучшение системы» (v1.5). Но не «напиши код класса Contractor» — код пишут Сотрудники.

### 3.2 Никогда не обращаться к Сотруднику напрямую от Координатора

После amendment v1.4 §5 (msg 665) прямой маршрут Координатор → Worker отменён. Даже для XS-задач путь: Координатор → Директор (брифинг) → Координатор → Head (опционально) → Координатор → Worker.

Исключение: Советники (architect, analyst, legal, construction-expert, tech-writer, memory-keeper, tutor, data-analyst) — штаб вне иерархии, Координатор обращается к ним напрямую для консультации.

### 3.3 Транзит бриф/результат строго дословный

Когда Координатор передаёт бриф от Директора в Worker — **дословно цитирует текст бри́фа**, не перефразирует. Поскольку Координатор физически является транспортом, он обязан сохранять точность формулировок.

Добавлять можно только FILES_ALLOWED / FILES_FORBIDDEN, критерии времени, формат отчёта — процедурные метаданные, не суть задачи.

### 3.4 Каждая делегация логируется

Все стадии пишутся в `/root/.claude/projects/-root/memory/project_tasks_log.md` секция соответствующего направления. Минимум: «YYYY-MM-DD | стадия | исполнитель | артефакт». Для наблюдаемости на дашборде это обязательно.

### 3.5 При получении задачи от Владельца — сначала план цепочки

Перед первым Agent-вызовом Координатор формулирует **план цепочки на задачу** (как task-routing-template.md). План на 3-7 строк: какие агенты, в каком порядке, что ожидается на входе/выходе. Этот план — артефакт самой задачи.

## 4. Правила для Директора

### 4.1 Бриф — первоочередная реакция на запрос Координатора

Если Координатор просит «напиши бриф» — Директор отдаёт бриф в следующем формате:

```
АДРЕСАТ: <роль, например ri-scout>
ЦЕЛЬ: <что должен получить на выходе>
ОБЪЁМ: <где границы, сколько времени, сколько элементов>
ИСТОЧНИКИ / СРЕДСТВА: <что использовать>
КРИТЕРИИ ГОТОВНОСТИ: <чек-лист>
ФОРМАТ ОТЧЁТА: <структура>
СРОК: <оценка времени>
```

Бриф — текст, не код. Директор не пишет «execute this shell command», он описывает задачу на уровне намерения.

### 4.2 Отказ от задания «сделай сам»

Если Координатор сформулировал задачу в обход §3.1 (например, «выполни sensing»), Директор **обязан отказать** и вернуть:

> «По статье 15 регламента Директора я не вправе делать работу Сотрудников. Вот мой бриф для <роль>: … Передай ему, верни результат мне на ревью.»

Это защитный механизм против процессных ошибок Координатора.

### 4.3 Исключение «критический блокер»

Статья 15 допускает исключение — «если подчинённые недоступны и блокер критический». В Claude Code этот случай практически невозможен (Worker всегда доступен для вызова), поэтому исключение применяется крайне редко. Каждый раз, когда Директор делает работу Сотрудника, он обязан зафиксировать инцидент в `docs/governance/incidents/`.

## 5. Правила для Начальника отдела

### 5.1 При получении брифа от Директора — план распределения

План распределения от Head включает:

```
ПОДЗАДАЧИ (по каждой):
  • Исполнитель: <worker id>
  • Суть: <персонализированный промпт на 3-5 предложений>
  • FILES_ALLOWED: <список абсолютных путей или паттернов>
  • FILES_FORBIDDEN: <что нельзя трогать, минимум «всё остальное»>
  • Оценка времени: <число>
  • Зависимости: <последовательные / параллельные>
ПОРЯДОК: <последовательный / параллельный>
ТОЧКИ СИНХРОНИЗАЦИИ: <когда Head просит у Координатора свести результаты>
```

### 5.2 Первичное ревью — обязанность Head'а

Когда Координатор возвращает результат Worker'а, Head обязан сделать первичное ревью (совместимость с брифом Директора, соответствие регламенту направления, отсутствие очевидных дефектов) и сформулировать вердикт. Head не делает глубокий код-ревью (это reviewer).

## 6. Правила для Сотрудника

### 6.1 FILES_ALLOWED — жёсткая граница

Worker трогает только файлы из FILES_ALLOWED. Если в ходе работы обнаруживается необходимость править файл вне списка — немедленная эскалация Head'у / Координатору, запрос расширения списка. Без молчаливого расширения.

### 6.2 Структурированный отчёт

После исполнения Worker отдаёт отчёт в формате:

```
СДЕЛАНО: <пункты>
ГДЕ ЛЕЖИТ: <файлы, строки>
ДОПУЩЕНИЯ: <что не спросил, но решил>
РИСКИ / ВНИМАНИЕ: <что может сломаться, что перепроверить>
ВОПРОСЫ: <если есть; не блокируют отчёт, но выносятся на ревью>
```

## 7. Работа через инструмент Agent (техника)

### 7.1 Один вызов = одна стадия цепочки

Каждый `Agent(subagent_type=X, prompt=...)` соответствует ровно одной стадии из §2.2. Не упаковывать несколько стадий в один промпт (например, «сделай бриф и потом сделай работу») — это нарушает принцип одноуровневой делегации и возвращает нас к инциденту §1.1.

### 7.2 Параллельные Worker'ы — один messsage с несколькими tool_calls

Если Head распределил работу по 3 backend-dev параллельно, Координатор в одном сообщении вызывает 3 Agent'а с разными subagent_type (если они одного типа — используется индексация -1/-2/-3 через разные промпты с явными FILES_ALLOWED). Результаты приходят одновременно.

### 7.3 Сохранение состояния цепочки

Координатор держит в памяти сессии полный трейс цепочки (брифы, промпты, ответы). Для долгих цепочек — фиксировать в `project_tasks_log.md` или отдельный файл цепочки в `docs/knowledge/chains/`.

## 8. Исключения

### 8.1 Советники (штаб)

Консультация Координатора с `architect`, `analyst`, `legal`, `construction-expert`, `tech-writer`, `memory-keeper`, `tutor`, `data-analyst` — прямой Agent-вызов, без цепочки. Советники — вне вертикали делегации, они отвечают на вопрос, не принимают задачу на исполнение.

### 8.2 XS-задачи строго внутри одного отдела

Для XS (1 файл, <1 час, один Worker, одно направление) допускается сокращение: Координатор → Директор (бриф) → Координатор → Worker (с брифом) → Координатор → Директор (вердикт). Минимум 3 Agent-вызова вместо 5 (без стадии Head). Но Директор всё равно пишет бриф и ревьюит.

### 8.3 Срочные правки CLAUDE.md

Правило «быстрый путь» из `departments/governance.md` — Координатор может править `CLAUDE.md` напрямую при выявлении критического антипаттерна, с обязательной post-factum заявкой в 24 часа. Это не нарушает цепочку, потому что `CLAUDE.md` — зона ответственности Координатора.

## 11. Логическая иерархия vs. технический оркестратор (уточнение 2026-04-16 msg 753-754)

По рекомендации Владельца (Telegram msg 753-754), принцип «Координатор-транспорт» формализован ещё точнее. Суть — **разделить логические связи между агентами и физический запуск**.

### 11.1 Важный нюанс про main-session

Субагент, запущенный как **main session** (`claude --agent <имя>`), имеет доступ к инструменту `Agent` и может технически вызывать других субагентов. Директор в main-session режиме — полноценный main-orchestrator своего направления.

Субагент, запущенный **внутри сессии другого агента** (например Координатор вызывает ri-director через Agent tool), — обычный subagent без доступа к Agent-инструменту. Он физически не может запускать других субагентов.

В coordinata56 мы **выбираем второй режим** — один main-orchestrator (Координатор), все остальные — subagents. Плюсы: одна точка управления, единый event log, проще наблюдаемость. Минус: Координатор становится bottleneck'ом при большой нагрузке — лечится при росте переходом части директоров в main-session режим.

### 11.2 Поля связей между агентами

Каждый агент описывается следующими связями (в `docs/agents/delegation-rules.yaml` — источник истины):

- `reports_to` — логическое подчинение (backend-dev → backend-head → backend-director → coordinator). Используется для ревью и эскалации.
- `can_request` — список агентов, которых этот агент может **попросить привлечь** (но не запускает сам). Например `backend-director.can_request = [backend-head, integrator-head, architect]`.
- `reviewed_by` — кто проверяет результат (массив: первичное + финальное ревью).
- `handoff_to` — куда передаётся результат после приёмки (обычно reports_to + иногда memory-keeper / reviewer).
- `spawned_by` — кто технически запускает. В текущей архитектуре для всех субагентов это `coordinator` (main-orchestrator).
- `main_session_capable` — булев флаг: может ли быть запущен как main-session (Директор — да, Начальник отдела / Сотрудник — нет).

### 11.3 Схема событий — 8 типов

Формальная схема — `docs/agents/task-event-log.schema.yaml`. Восемь типов:

1. **task_created** — Координатор принял задачу от Владельца / внутреннего триггера
2. **delegation_requested** — агент N возвращает запрос привлечь агента M (логическая делегация)
3. **agent_spawned** — Координатор физически вызвал субагента через `Agent` tool
4. **result_returned** — субагент вернул результат
5. **review_requested** — результат передан на ревью
6. **review_completed** — ревьюер вынес вердикт
7. **memory_written** — memory-keeper (или Координатор по v1.2 §A4.7) записал значимое в память
8. **user_reported** — Координатор отчитался Владельцу в Telegram

Каждое событие несёт `task_id` (корневая задача), `chain_depth` (глубина в иерархии), `correlation_id` (для связи request↔response), а также ключевую пару **`requested_by`** (кто логически делегировал) и **`spawned_by`** (кто технически запустил).

### 11.4 Дашборд: логическая vs техническая подсветка

На дашборде паутина делегирования имеет две проекции, переключаемые кнопкой:
- **Логическая** — рёбра идут по `requested_by → target_agent` (Директор → Начальник отдела → Сотрудник). Так Вы видите иерархию.
- **Техническая** — все рёбра идут от `coordinator → target_agent`. Так видна фактическая топология звезды.

По умолчанию показывается логическая, как и в паспорте системы (§Ж диаграмма).

### 11.5 Классификация задач — первая работа Координатора

При получении задачи от Владельца (событие `task_created`) Координатор **определяет направление и tier** до любых Agent-вызовов. Это не делегируется — это прямая обязанность Координатора уровня L1 (v1.4 §«Обязанности Координатора, пункт 2: декомпозиция задач и определение их уровня»). При сложной или неоднозначной задаче — консультация с `analyst` (штаб, вне цепочки).

У нас нет отдельного `product-director` (как в примере Владельца msg 753). Если со временем появится регулярная потребность в product-management работе — добавим через комиссию Governance; пока Координатор справляется с классификацией сам.

### 11.6 Технический кейс: Директор в main-session режиме (как будет в будущем)

Если/когда какой-либо Директор (например backend-director) переводится в main-session режим:
- Запускается как `claude --agent backend-director` в отдельной сессии
- Имеет свой Agent-tool, может вызывать backend-head / integrator-head напрямую
- `spawned_by` для backend-head становится `backend-director`, не `coordinator`
- Координатор всё равно отвечает за классификацию входящей задачи и решение «запустить ли main-session Директора или обойтись субагент-цепочкой»

Это отдельная фаза масштабирования, которая формально запускается через комиссию Governance при достижении критерия (например, >5 одновременных M-задач в одном направлении в неделю).
