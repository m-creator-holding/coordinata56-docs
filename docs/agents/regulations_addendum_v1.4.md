# Регламент субагентов v1.4 — Иерархическая структура

**Дата:** 2026-04-15
**Статус:** ✅ утверждён Владельцем (Мартин, Telegram 2026-04-15)
**Отношение к v1.0/v1.1/v1.2/v1.3:** v1.4 — **крупный переход**, переосмысление организационной структуры. Не заменяет v1.0-v1.3 полностью: роли, скилы, процесс ревью, пред-коммит-чек — остаются. Меняется только **иерархия делегирования**: вместо плоской «Координатор → 17 субагентов» появляется 4-уровневая структура с заместителями и начальниками отделов.

**Решения Владельца, принятые 2026-04-15:**
- Юридическое направление — **отдельная 6-я вертикаль (Директор по юр.вопросам)**, но не активируется до появления задач (первая активация — после подключения к боевым данным / при подготовке договоров с подрядчиками).
- Исследовательская вертикаль (research) — **создаётся только когда дойдём до маркетинга/продаж** (не-MVP модули M4+).
- Роли создаются **по мере надобности**, а не заранее. Принцип: «нет работы — нет роли». Это предотвращает раздувание `~/.claude/agents/` до неуправляемых объёмов.
- Модели — **остаёмся на линейке Claude до закрытия MVP**. Сторонние модели (YandexGPT, DeepSeek, GigaChat, локальные Qwen/DeepSeek-Coder) — внедряем **после MVP**, когда появится объём, оправдывающий настройку мульти-модельного оркестратора. Подтверждено Telegram 2026-04-15.

## 1. Структура (5 уровней)

```
L0  Владелец (Мартин) ─────────────────────┐ (ставит задачи, формально согласует фазы)
      │
      ▼
L1  CEO / Координатор (я, Claude Opus 4.6) ┐ (единая точка контакта Владельца)
      │
      ▼
L2  5 Директорестителей по направлениям ────────┐ (декомпозиция задач внутри направления)
      │
      ▼
L3  Начальники отделов + Проверяющие ──────┐ (назначение исполнителей, ревью их работы)
      │
      ▼
L4  Сотрудники (Workers) ──────────────────┐ (исполнение: код, тесты, документы)

Параллельно: Advisors (советники) ──────── (консультируют любой уровень, но не в цепи управления)
```

## 2. L2 — 6 директоров (5 активных + 1 dormant)

| Должность | Файл-слаг (для ~/.claude/agents/) | Область ответственности | Статус |
|---|---|---|---|
| Директор по бэкенду | `backend-director` | FastAPI, SQLAlchemy, Python, бизнес-логика, REST API | 🟢 активен |
| Директор по фронтенду | `frontend-director` | React, TypeScript, Vite, shadcn/ui, UX-реализация | 💤 активируется на Фазе 4 |
| Директор по дизайну | `design-director` | User flows, wireframes, UI-спецификации, accessibility, копирайтинг (в т.ч. UX-writing) | 💤 активируется на Фазе 4 |
| Директор по качеству | `quality-director` | QA-стратегия, тестирование, security, ревью | 🟡 пилот на Батче B Фазы 3 |
| Директор по инфраструктуре | `infra-director` | Docker, CI/CD, PostgreSQL-схема, интеграции, миграции | 💤 активируется на Фазе 9 |
| Директор по юридическим вопросам | `legal-director` | 152-ФЗ, 214-ФЗ, ТК РФ, 63-ФЗ, договоры с подрядчиками | 💤 активируется при боевых данных или первом договоре |

> Технические слаги — английский snake-case (как существующие `db-engineer`, `tech-writer`, `devops`), потому что это идентификаторы в файловой системе. Должности и упоминания в общении — **только русские**.

**Dormant (💤)** = регламент и роль описаны, но агент в `~/.claude/agents/` **не создан** до появления работы. Координатор не делегирует им задачи.

Каждый зам — **один агент** (не отдел). Внутри его направления — отделы (L3).

- **Отдел инфраструктуры**: начальник = `devops-head` (Начальник отдела инфраструктуры), сотрудник = `devops` (DevOps-инженер)
- **Отдел баз данных**: начальник = `db-head` (Начальник отдела БД), сотрудник = `db-engineer` (Инженер баз данных)

### 3.6 Под Директором по юридическим вопросам — 💤 dormant до боевых данных
- **Отдел юридических исследований**: начальник = `legal-head` (Начальник юридического отдела), сотрудник = `legal-researcher` (Юрист-исследователь — поиск норм, судебной практики)

Советник `legal` из v1.0 (текущий универсальный юрист) **временно заменяется** на трёх специализированных юристов при активации юридической вертикали. До активации — работает как прежде.

## 4. Штабные советники (не в иерархии)

Отвечают на вопросы любого уровня, но не управляют задачами. Вызываются Координатором или Директорами при необходимости.

- `architect` — **Главный архитектор**: архитектурные решения (ADR), подписывает Координатор после Владельца
- `analyst` — **Бизнес-аналитик**: User Stories, бизнес-требования (мост между Владельцем и Директорами)
- `legal` — **Универсальный юрист**: юридические проверки (152-ФЗ, 214-ФЗ, ТК РФ) — пока не активирована полноценная юр-вертикаль
- `construction-expert` — **Эксперт по строительству**: SNiP/ГОСТ, стадии стройки, типовые контракты
- `tech-writer` — **Технический писатель**: документация (README, onboarding, руководства пользователя)
- `memory-keeper` — **Хранитель памяти**: единственный кто пишет в `/root/.claude/projects/-root/memory/`
- `tutor` — **Наставник Владельца**: обучение Мартина программированию параллельно с проектом
- `data-analyst` — **Аналитик данных**: запросы к БД, отчёты
- `claude-code-guide` — **Советник по Claude Code**: вопросы по API/SDK Anthropic

## 5. Маршрутизация задач по размеру

| Tier | Размер | Путь | Ревью-точки |
|---|---|---|---|
| **XS** | 1 файл, <1 час | Координатор → Worker напрямую | Self-check + Координатор |
| **S** | 2-10 файлов, один отдел | Координатор → Head → Worker | Self + Head + Координатор |
| **M** | модуль, одно направление | Координатор → Директор → Head → Workers (parallel) | Self + Head + Директор + Координатор |
| **L** | фаза / многонаправленная | Координатор → несколько Директоров (parallel) → Heads → Workers | Все уровни + Владелец |

Координатор определяет tier при приёме задачи от Владельца.

**Пример из текущего Батча A:** tier M (модуль API, направление — бэкенд).
- Координатор → Директор по бэкенду: «реализуй CRUD для 8 сущностей»
- Директор по бэкенду → backend-head: «разбей на подзадачи, назначь исполнителей»
- backend-head → 3 параллельных backend-dev: по 2-3 сущности каждый
- backend-dev отчитывается head → head ревьюит → поднимает заму → зам ревьюит → Координатор ревьюит → коммит → отчёт Владельцу

## 6. Модели — привязка к уровням и задачам

Модель выбирается **по роли**, но может быть **временно поднята** Координатором/Директором для особо сложной задачи.

### Базовые привязки

| Уровень | Роль | Модель по умолчанию | Почему |
|---|---|---|---|
| L1 | Координатор | **Opus 4.6 (1M)** | Длинный контекст, синтез, финальные решения |
| L2 | Директор по бэкенду, Директор по фронтенду, Директор по инфраструктуре | **Opus 4.6** | Декомпозиция, кросс-отделы, архитектурные компромиссы |
| L2 | Директор по дизайну | **Sonnet 4.6** | UX-спецификации — объёмные, но не глубоко «мыслительные» |
| L2 | Директор по качеству | **Opus 4.6** | Стратегия качества, security-аудит — критично для бизнеса |
| L3 | Все heads | **Sonnet 4.6** | Тактика, ревью кода от workers, паттерн-контроль |
| L4 | backend-dev, frontend-dev, integrator, db-engineer | **Sonnet 4.6** | Кодинг по паттерну — sonnet хватает, в 5× дешевле opus |
| L4 | qa, designer, reviewer-1/2 | **Sonnet 4.6** | Средняя глубина |
| L4 | devops, security (workers) | **Sonnet 4.6** | |
| Advisor | architect, analyst, legal | **Opus 4.6** | Глубина мышления критична |
| Advisor | construction-expert | **Sonnet 4.6** | Домен + факты |
| Advisor | tech-writer, memory-keeper, tutor | **Haiku 4.5** | Механическая работа по шаблону, в 15× дешевле opus |
| Advisor | data-analyst, claude-code-guide | **Sonnet 4.6** | |

### Правила переключения модели на задачу

1. **Up-shift** (повысить модель): Директор или Координатор могут поднять Worker с sonnet → opus, если:
   - задача включает проектирование нового паттерна (не тиражирование)
   - задача на стыке двух подсистем
   - QA-kейс с нестандартной бизнес-логикой
2. **Down-shift** (понизить): Head может опустить Worker с sonnet → haiku, если:
   - задача чисто механическая (scaffolding, форматирование, перевод)
   - объём >70% шаблонный код
3. Решение о переключении — в промпте задачи, с пометкой `MODEL_OVERRIDE: <model> — reason: <...>`. Фиксируется для ретро.

### Ориентировочный расчёт стоимости (per задачу)

Грубый порядок (для сравнения):
- Opus 1M input/output ≈ 15$/M in + 75$/M out (реальные цифры — по Вашему биллингу)
- Sonnet ≈ 3$/M in + 15$/M out (~5× дешевле)
- Haiku ≈ 0.8$/M in + 4$/M out (~18× дешевле)

На tier M задаче ~1M токенов сейчас всё было opus = ~45$. С v1.4:
- Координатор (opus): ~300k = ~15$
- Директор (opus): ~100k = ~5$
- Head (sonnet): ~150k = ~2$
- 3 workers (sonnet): ~300k = ~4$
- Reviewer (sonnet): ~100k = ~1.5$
- **Итого: ~27$ (минус 40%)**, плюс время в ~1.7× быстрее за счёт parallel workers.

## 7. Регламент делегирования (правила)

### 7.1 Прямая коммуникация между субагентами

**Внутри одного отдела (вертикальная цепь)**:
- ✅ Координатор → любой Директор направления
- ✅ Директор → подчинённые ему Начальники отделов
- ✅ Начальник отдела → подчинённые ему Сотрудники

**Внутри одного направления (между отделами одного Директора)**:
- ✅ Начальник одного отдела ↔ Начальник другого отдела того же Директора (горизонтально, с уведомлением Директора в копию)
- ❌ Сотрудники разных отделов одного Директора **не общаются напрямую** — только через своих начальников

**Между разными направлениями (кросс-вертикальная связь)** — это новый раздел, ужесточённый по решению Владельца 2026-04-15:
- ✅ **Только Начальники отделов** разных направлений могут общаться напрямую — для рутинных согласований (например, backend-head ↔ qa-head по приёмке тестов)
- ✅ **Серьёзные вопросы** (изменение архитектуры, спорные приоритеты, конфликт ресурсов) — поднимаются на уровень **Директоров** соответствующих направлений
- ❌ **Сотрудники разных направлений** общаться напрямую **не должны** — никогда. Любая нужда в кросс-направленческой коммуникации эскалируется через начальника отдела
- ❌ **Кто угодно → Владелец** — всегда через Координатора

**Зачем это правило**: предотвращает «теневые договорённости» между исполнителями в обход начальства, сохраняет прозрачность принятия решений и защищает от расползания контекста (если двое Сотрудников сговорились о паттерне, об этом не узнают ни их начальники, ни Координатор — паттерн просочится в код без ревью).

**Что считать «серьёзным вопросом»** (триггеры эскалации Начальник → Директор):
- Конфликт по ADR / архитектуре
- Расхождение приоритетов между отделами
- Запрос блокирует работу более чем одного Сотрудника
- Затрагивает security или legal
- Требует решения Владельца

**Кодификация в промптах**: при делегировании задачи Координатор / Директор / Начальник отдела явно указывает в `COMMUNICATION_RULES`:
- кому подчиняется получатель задачи
- к кому он может обратиться напрямую
- что эскалировать наверх
- какие отделы запрещено беспокоить

### 7.2 Эскалация вверх

- Worker → Head: если встретил отклонение от паттерна/ADR (регламент v1.3 §2)
- Head → Директор: если возникло архитектурное решение или конфликт ревью
- Директор → Координатор: если задача требует решения Владельца
- Координатор → Владелец: только блокирующие вопросы или согласование фаз

### 7.3 Обратная связь с ревью

Каждое ревью записывается в `docs/reviews/` с указанием уровня:
- `tier<size>-<task>-<date>-selfcheck.md` — Worker самоконтроль
- `tier<size>-<task>-<date>-head.md` — ревью от Head
- `tier<size>-<task>-<date>-deputy.md` — ревью от Директора
- `tier<size>-<task>-<date>-coord.md` — финальное от Координатора

### 7.4 Параллелизация

Внутри одного Head-а Workers работают параллельно при условии (регламент v1.3 §5):
- Explicit FILES_ALLOWED / FILES_FORBIDDEN в каждом промпте
- Head согласовывает границы ДО запуска

### 7.5 SLA (ориентиры, не жёсткие)

- Worker (tier XS/S): ответ в течение сессии
- Head ревью: <15 мин от получения
- Директор ревью: <30 мин
- Координатор: real-time (пока сессия активна)

## 8. Что меняется по сравнению с v1.0-v1.3

### Остаётся без изменений
- DoD чек-листы (общий + per-phase)
- ADR-процесс (MUST/SHOULD, amendments)
- Правила v1.3 (reviewer до коммита, отклонения от ADR, секреты)
- Скилы (28 штук из v1.1)
- Память Координатора (MEMORY.md)
- Телеграм-канал как канал связи с Владельцем

### Меняется
- Плоская структура из 17 субагентов → 4-уровневая иерархия с 5 Директорами
- Делегирование: Координатор больше не раздаёт задачи всем — только Директорам
- Появляются новые роли: 5 Директоров + 9 Heads
- Модели — распределены по уровням (см. §6)
- Дублирующие роли workers (backend-dev-1/2/3 и т.д.) — для параллелизации

### Остаётся в v1.3, усиливается
- Параллелизация с файловыми границами (§5 v1.3) — теперь **стандартный** режим для tier M
- Пред-коммит чек Worker (§6 v1.3) — теперь **обязательный** перед передачей Head-у

## 9. Фазировка перехода на v1.4

Резкий переход рискован (мы в середине Фазы 3). Предлагаю:

### Этап 1 — сейчас (до конца Батча A)
- Регламент v1.4 согласован и записан, но **не применяется**
- Батч A закрываем в текущем режиме (v1.3)

### Этап 2 — Батч B Фазы 3 (пилот)
- Применяем v1.4 только внутри Директор по бэкенду (бэкенд-направление)
- Вводим backend-head и 2-3 backend-dev workers
- Смотрим: скорость, бюджет, качество
- Ретроспектива по итогам Батча B

### Этап 3 — Батч C Фазы 3 (расширение)
- При успехе Этапа 2 — добавляем Директор по качеству-вертикаль (qa-head, review-head)
- Остальные вертикали появляются по мере надобности

### Этап 4 — Фаза 4 (фронтенд)
- Активируем Директор по фронтенду + Директор по дизайну вертикали

### Этап 5 — Фаза 9 (деплой)
- Активируем Директор по инфраструктуре вертикаль

Директоры и Heads, которые ещё не активированы, — **не создаём**. Принцип: роль появляется тогда, когда есть работа для неё.
