## Преамбула v1.3

Фаза 2 прошла за 1 день вместо плановых 5–7 — благодаря субагентам, скилам и трём раундам ревью. Ретро выявило четыре процессных дефекта, которые в v1.0–v1.2 не описаны явно. v1.3 их закрывает. Объём изменений минимальный — точечные правки в регламентах backend-dev, qa, reviewer и Координатора.

## §1. Reviewer работает на staged diff ДО коммита

**Правило**: `reviewer` вызывается на результат `git diff --staged` (или на worktree-копию изменений) **до** `git commit`. Коммит в main/feature-ветку допустим только после verdict = `approve`.

**Почему**: в Фазе 2 первый вызов reviewer прошёл после коммита `15a7479`, нашёл 2 P0 — пришлось делать fix-up коммит `312dd50`. Это прошло только потому, что ветка feature; в main недопустимо.

**Как применять** (Координатор):
1. `backend-dev` / `frontend-dev` готовят изменения и делают `git add`, но **не** `git commit`.
2. Координатор вызывает `reviewer` с указанием: «смотри staged diff + описание задачи».
3. `reviewer` пишет отчёт в `docs/reviews/<phase>-<topic>-<date>[-roundN].md`.
4. При `request-changes` — возврат исполнителю, новый раунд. При `approve` — Координатор санкционирует `git commit`.
5. Исключение: правки самой документации ревью (`docs/reviews/*`) — коммитятся без отдельного ревью.

**Кого затрагивает**: Координатор (процесс), backend-dev, frontend-dev, qa, reviewer.

---

## §2. Отклонения backend-dev/frontend-dev от ADR требуют согласования

**Правило**: если исполнитель в ходе работы обнаружил, что буквальное следование ADR невозможно или нецелесообразно, он **обязан** остановиться и запросить согласование у Координатора (или у `architect` через Координатора). Самовольные отклонения запрещены, даже если они кажутся очевидно верными.

**Почему**: в Фазе 2 backend-dev самостоятельно сделал три отклонения от ADR 0003 (`EmailStr→str`, `JWT_SECRET→JWT_SECRET_KEY`, паттерн `B008`). Отклонения по сути приемлемые, но пошли в код без визирования. Это создаёт риск: субагент принимает архитектурные решения, которые не его зона ответственности.

**Как применять** (исполнитель):
1. Наткнулся на расхождение с ADR — **стоп**, не пишешь код.
2. Формируешь короткую записку: какое место ADR, какое отклонение, почему.
3. Возвращаешь Координатору с меткой `ADR-DEVIATION`.
4. Координатор либо решает сам (если тривиально), либо делегирует `architect`, либо эскалирует Владельцу.
5. Результат — запись в ADR как amendment, после чего можно писать код.

**Кого затрагивает**: backend-dev, frontend-dev, db-engineer.

## §3. Секреты — никогда литералом, даже в тестах

**Правило**: любые значения, которые в прод-среде приходят из `.env` (секреты, ключи, токены, пароли), **не** вписываются литералом в код — ни в `src/`, ни в `tests/`, ни в фикстуры, ни в `conftest.py`. Для тестов — генерируется на лету через `secrets.token_urlsafe(32)` или читается из `os.environ.setdefault(...)`.

**Почему**: в Round 2 reviewer поймал `R2-BLOCKER-1` — литерал JWT-секрета в `conftest.py`. Это не только плохая практика, но и реальный риск утечки через git-историю.

**Как применять** (backend-dev, qa):
- В `conftest.py` / фикстурах: `os.environ.setdefault("JWT_SECRET_KEY", secrets.token_urlsafe(32))`.
- В unit-тестах, где нужен предсказуемый ключ — генерируется один раз на сессию pytest, не коммитится.
- Pre-commit hook на детект секретов — задача devops в Фазе 9 (кандидат: `gitleaks` или `detect-secrets`).

Паттерн вынесен в скил `fastapi-auth-hardening` §9.

**Кого затрагивает**: backend-dev, qa, devops.

## §4. QA: для anti-enumeration тестов assert на `detail` обязателен

**Правило**: когда тест проверяет защиту от user enumeration (логин, восстановление пароля, 2FA), недостаточно `assert response.status_code == 401`. Требуется также `assert response.json()["detail"] == "<общая формулировка>"`. Иначе тест пропустит ситуацию, когда код статуса одинаковый, а сообщение разное — и enumeration остаётся возможным.

**Почему**: Round 3 nit — qa проверял только статус, reviewer указал. Исправлено в финальном коммите Фазы 2.

**Как применять** (qa): в каждом тесте «неверный логин / неверный пароль / заблокированный пользователь» проверять и статус, и `detail`.

**Кого затрагивает**: qa.

## §5. Параллелизация субагентов — граница файлов обязательна

**Правило**: когда Координатор запускает двух и более субагентов параллельно (Task tool в одном сообщении), в промпте каждого **явно** прописывается список файлов, которые субагент может трогать, и список, которые трогать **запрещено**. Без этого параллельные правки конфликтуют.

**Почему**: в Фазе 2 параллельный запуск `backend-dev` + `qa` работал только потому, что Координатор явно разграничил: «backend не трогает test_auth.py, qa не трогает auth.py». При забытом разграничении — merge-conflict на worktree.

**Как применять** (Координатор): в каждом prompt-шаблоне для параллельного запуска — секция `FILES_ALLOWED` / `FILES_FORBIDDEN`.

**Кого затрагивает**: Координатор.

## §6. Пред-коммит самопроверка backend-dev (5 минут)

**Правило** (конвенция, не автоматизация): до `git add` backend-dev проходит короткий чек:

1. Есть ли отклонения от ADR? Если да — `STOP`, см. §2.
2. Есть ли новые переменные окружения? Если да — добавлены в `.env.example`?
3. Есть ли литералы секретов? Если да — убрать (см. §3).
4. Тестовые фикстуры с секретами используют `secrets.token_urlsafe` / `os.environ.setdefault`?
5. Все публичные функции и эндпоинты имеют type hints и Pydantic-схемы?

Если любой пункт красный — исполнитель правит до вызова reviewer.

**Кого затрагивает**: backend-dev.

---

## §7. Обновление скилов после фазы

**Правило**: после закрытия каждой фазы Координатор проверяет, какие уроки ретро применимы к SKILL.md релевантных скилов, и предлагает Владельцу обновить. Скилы — носитель процедурных знаний команды; без обновления они устаревают.

**Как применять**: в рамках ретроспективы добавляется раздел «Обновления скилов» с конкретным списком. Владелец визирует — обновление применяется.

**Пример после Фазы 2**: `fastapi-auth-hardening` §9 — «секреты в тестах через `secrets.token_urlsafe`».

**Кого затрагивает**: Координатор.
