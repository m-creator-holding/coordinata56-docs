## Преамбула

Настоящий Свод объединяет все внутренние регламенты проекта M-OS (ранее `coordinata56`) в единый структурированный документ. Свод не вводит новых правил — он систематизирует уже принятые и утверждённые. Каждая статья отсылает к подробному регламенту в репозитории.

Приоритет коллизий между документами регламента определён в `docs/agents/departments/governance.md` §«Приоритет коллизий» (6-уровневая иерархия от `CLAUDE.md` до должностных инструкций). Правило «первоисточник имеет приоритет» действует только в пределах одного уровня иерархии. Если расхождение системное — обновляется Свод через комиссию Governance.

Принцип развития: правила пишем заранее, чтобы избежать ошибок; с накоплением практики корректируем и дополняем (решение Владельца, 2026-04-15).

# КНИГА I. ОБЩАЯ ЧАСТЬ

## Раздел I. Цели и базовые принципы

### Глава 1. Цель проекта

**Статья 1.** Проект развивается как **M-OS (M Operating System)** — единое внутреннее AI-native программное обеспечение холдинга Владельца, построенное на pod-архитектуре. Pod `cottage-platform` (исторически `coordinata56`) — первый pod M-OS, обслуживающий коттеджный посёлок «Координата 56» (85 домов бизнес-класса). Полное определение M-OS — ADR 0008. Принципы pod-архитектуры — ADR 0009.

**Статья 2.** Главная цель — собственное ПО для цифровизации, прозрачности и автоматизации холдинга, написанное самим Владельцем при поддержке команды ИИ-субагентов под управлением Координатора. M-OS строится поэтапно (фазы M-OS-0 — M-OS-5); текущая фаза — M-OS-0 Reframing, пересекается с завершением Фазы 3 `cottage-platform` pod'а.

### Глава 2. Базовые принципы

**Статья 3.** Принцип «Владелец — единственный заказчик». Все стратегические и бизнес-решения принимает Владелец. ИИ-агенты — исполнители и советники, не лица принимающие решения.

**Статья 4.** Принцип фазового развития. Работа ведётся по утверждённой дорожной карте (10 фаз). Следующая фаза не начинается, пока не закрыта текущая по DoD.

**Статья 5.** Принцип «руки работают, головы улучшают». Чем выше уровень в иерархии, тем больше времени на улучшение системы; чем ниже — тем больше на конкретное исполнение. Подробно — Книга II, Глава 1, статьи о распределении времени.

**Статья 6.** Принцип «умные модели не делают рутину, простые модели не принимают серьёзных решений». Подробно — Книга IV, Раздел I.

**Статья 7.** Принцип «роль появляется тогда, когда есть работа». Не активируем спящие роли заранее.

**Статья 8.** Принцип «правило живёт после ошибки». Каждый повторяющийся дефект порождает правило в `CLAUDE.md` или департаментском регламенте.

## Раздел II. Структура иерархии

### Глава 3. Уровни иерархии (v1.4)

**Статья 9.** В компании пять уровней:
- **L0** — Владелец (Мартин)
- **L1** — Координатор (CEO проекта)
- **L2** — Директора направлений (6 ролей: бэкенд, фронтенд, дизайн, качество, инфраструктура, юр.вопросы)
- **L3** — Начальники отделов (11 ролей)
- **L4** — Сотрудники (22+ роли)

Дополнительно: **штабные советники** — 9 ролей вне иерархии, консультируют любой уровень.

> **Примечание (v1.1).** Субагенты классифицируются по таксономии ADR 0010 на 5 типов: **executive** (L1, единственный — Координатор), **core_department** (общие для всей M-OS: backend, quality, infra, R&I и др.), **domain_pod** (специфичные для конкретного pod'а, напр. construction-expert), **governance** (комиссия Свода), **advisory** (штабные советники вне иерархии). Таксономия — в `agents-map.yaml`.

**Статья 10.** Полный регламент структуры — `docs/agents/regulations_addendum_v1.4.md`.

### Глава 4. Канал управления

**Статья 11.** Все задачи Владельца проходят через Координатора. Никто, кроме Координатора, не общается с Владельцем напрямую.

**Статья 12.** Внутрикомпанейское делегирование — строго по вертикали: Координатор → Директора → Начальники → Сотрудники.

**Статья 13.** Кросс-вертикальная коммуникация:
- Между Начальниками разных направлений — разрешена для рутинных вопросов
- Между Директорами разных направлений — разрешена для серьёзных вопросов
- Между Сотрудниками разных направлений — запрещена всегда

Подробно — `regulations_addendum_v1.4.md` §7.1.

# КНИГА VII. POD-АРХИТЕКТУРА И ТАКСОНОМИЯ

**Статья 57.** Pod (модуль) — независимая единица M-OS, обслуживающая одно бизнес-направление холдинга или общую функцию. Определение и принципы — ADR 0009.

**Статья 58.** Граница «общее ядро vs pod»: общее ядро содержит инфраструктуру, одинаково нужную всем направлениям (пользователи, роли, аудит, справочники, уведомления, BPM, интеграции, чат). Предметная логика (модели данных, бизнес-процессы, экраны) принадлежит своему pod'у. Расширение ядра — только через ADR. Подробно — ADR 0009 §2.

**Статья 59.** Таксономия субагентов: каждый субагент классифицирован по одному из 5 типов — `executive`, `core_department`, `domain_pod`, `governance`, `advisory`. Определения типов, мэппинг всех 48 файлов агентов и правила активации при добавлении нового pod'а — ADR 0010.

**Статья 60.** Активация субагентов при добавлении нового pod'а: `core_department`-агенты начинают работать с новым pod'ом автоматически; создаётся минимум один `domain_pod`-агент (носитель предметных знаний направления); `governance` и `advisory` охватывают новый pod без изменений. Подробно — ADR 0010 §3.

**Статья 61.** Миграция документации: pod-специфичные документы (фазы, stories, wireframes, specs) размещаются в `docs/pods/<pod-name>/`. M-OS-wide документы (ADR, agents, governance, legal, security, onboarding) остаются в `docs/`.

**Статья 56.** Порядок изменения:
1. Инициатива (от любого Директора, Координатора или Владельца)
2. Координатор готовит редакцию
3. Согласование с Владельцем
4. Принятие новой минорной/мажорной версии
5. Сохранение в репозиторий

## Приложение Б. Глоссарий ключевых терминов Свода

- **Свод законов** — настоящий документ, единая навигационная точка по всем регламентам
- **Регламент** — отдельный документ с правилами работы (общефирменный, по уровню или по направлению)
- **ADR** — Architecture Decision Record, документ с архитектурным решением и его обоснованием
- **DoD** — Definition of Done, чек-лист критериев готовности фазы или батча
- **CLAUDE.md** — антипаттерник проекта, автоматически загружается в каждой сессии
- **Скил** — методика или сборник рецептов в `~/.claude/skills/`, используется агентами
- **Up-shift / Down-shift** — временное повышение или понижение модели агента под конкретную задачу
- **Dormant** — спящая роль/направление/департамент, описано в регламенте, но не активировано
- **Tier** — уровень размера задачи (XS / S / M / L), определяет маршрут делегирования
- **FILES_ALLOWED / FILES_FORBIDDEN** — границы файлов, в пределах которых работает агент

## Приложение В. Связь со Скилами Anthropic

Скилы — это методики, общие для многих проектов на Claude Code. Некоторые из них применяются в нашем проекте. Полный список используемых — `regulations_addendum_v1.1.md`.
