## Цель отдела

Единая точка владения корпоративным регламентом coordinata56 (`docs/agents/CODE_OF_LAWS.md`, `regulations_*.md`, `departments/*.md`, `ADR/*.md`, `CLAUDE.md`). Отдел отвечает за:
1. **Консистентность** — нет противоречий между документами.
2. **Полнота** — нет пробелов в процессах, которые реально возникают.
3. **Без дублирования** — одно правило живёт в одном месте; остальные ссылаются.
4. **Исполнение** — фиксирует нарушения регламента, эскалирует систематические.
5. **Контроль изменений** — все правки в регламент идут через комиссию этого отдела.

## Структура

| Уровень | Роль | Модель | Основная задача |
|---|---|---|---|
| L2 | `governance-director` | Opus | Председатель комиссии. Рассматривает заявки на изменение регламента, выносит вердикт. Владеет глобальной картой регламента. |
| L3 | `governance-auditor` | Opus | Периодический аудит регламента на противоречия, дубли, пробелы. Пишет отчёты аудита. Проверяет соответствие фактического поведения агентов написанному. |

## Процесс «изменение регламента»

Любая правка любого регламентного документа (исключения: `MEMORY.md` в `/root/.claude/`, `docs/reviews/*`, `docs/knowledge/retros/*`, `docs/research/*` — они не регламент) идёт через комиссию:

1. **Заявка** — инициатор (Координатор, Директор, Владелец, аудитор) создаёт файл `docs/governance/requests/YYYY-MM-DD-<slug>.md` с шаблоном:
   - Что меняется (документ, раздел, формулировка до/после).
   - Почему (инцидент, новое решение, дубль).
   - На что влияет (список документов, которые могут конфликтовать).
2. **Анализ** — governance-auditor проверяет: нет ли противоречия с другими документами, не дублирует ли существующее правило, не противоречит ли ADR.
3. **Вердикт Директора** — approve / reject / request-changes. Approve: правка применяется; Директор обновляет документ и фиксирует изменение в `CHANGELOG.md` отдела governance.
4. **Исключение «быстрый путь»** — срочные правки CLAUDE.md после инцидента может делать Координатор напрямую, но обязан в течение 24 часов подать заявку post-factum, которую аудитор ревьюит.

## Периодический аудит

- **Еженедельный (понедельник):** auditor проходит по `CLAUDE.md` и `departments/*.md` — ищет противоречия, дубли, устаревшие упоминания.
- **После каждой фазы:** auditor проходит по всем ADR + регламентам, написанным за фазу, и пишет отчёт `docs/governance/audits/phase-N-audit.md`.
- **По заявке:** Директор или Координатор может запросить прицельный аудит.

## Поведенческий аудит

**Что это.** Выборочная проверка того, что субагенты реально соблюдают регламент на практике (по логам, коммитам, отчётам), а не только декларативно.

**Триггеры:**
1. `systematic` флаг от Координатора — при обнаружении признаков системного отклонения.
2. ≥3 повторяющихся замечания одного типа в ревью одного субагента/направления.
3. Запрос Директора направления о проверке работы своего отдела.

**Процедура:**
1. `governance-auditor` запрашивает у Координатора выборку из 5–10 последних коммитов / отчётов по указанному субагенту или направлению.
2. Читает выборку, сверяет с регламентом (свой уровень, свой `departments/*.md`, ADR, `CLAUDE.md`).
3. Пишет отчёт в `docs/governance/audits/behavioral/YYYY-MM-DD-<subject>.md` — с цитатами из выборки, ссылками на нарушенные статьи, рекомендациями.
4. Отчёт передаётся Координатору и Директору направления; при необходимости — заявка на правку регламента через стандартный процесс комиссии.

## SLA комиссии

- **Вердикт по заявке:** Директор Governance обязан вынести вердикт (approve / reject / request-changes) в течение **≤ 2 рабочих дней** с момента подачи заявки.
- **Реакция на еженедельный отчёт аудита:** после получения еженедельного отчёта `governance-director` обязан **в одной сессии** либо завести заявки на все W-находки, либо явно сделать `reject` по каждой с письменной мотивировкой в `docs/governance/CHANGELOG.md`. Отсутствие реакции ≥ 1 недели считается процессным дефектом и эскалируется Координатору.

## Что отдел НЕ делает

- Не пишет код.
- Не коммитит.
- Не принимает бизнес-решения.
- Не эскалирует напрямую Владельцу — через Координатора.
- Не блокирует работу Координатора по оперативным задачам; блокирует только изменения регламента.

## Приоритет коллизий

Если аудитор находит противоречие между двумя документами, разрешение по приоритету (от высшего):
1. `CLAUDE.md` (live антипаттерник).
2. `CODE_OF_LAWS.md`.
3. `regulations_addendum_v*.md` (свежая версия бьёт старую).
4. `departments/<name>.md`.
5. `ADR/NNNN-*.md` (архитектурные решения — высокая сила для технических вопросов).
6. Должностные инструкции `~/.claude/agents/<name>.md`.

При конфликте уровней 1-2 эскалируется Координатору.
