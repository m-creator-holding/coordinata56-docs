## Правила работы (выросшие из ошибок)

Общефирменные — в `CLAUDE.md`. Специфичные для качества:

1. **Anti-enumeration: assert на `error.code` + `error.message`**, не только `status_code`. Иначе тест пропустит ситуацию, когда статус одинаковый, а сообщение разное. *(Round 3 Phase 2.)*
2. **Литералы паролей в фикстурах = блокер ревью**. Только `secrets.token_urlsafe(16)`. *(Повторялось 3 раза.)*
3. **IDOR-тесты для вложенных ресурсов — обязательны**: создать ресурс под одним родителем, попытаться править/удалять с правильным cfg_id но чужим parent_id → 404. *(Найдено step 4 round 1.)*
4. **Bulk-операции тестируются на atomic rollback**: подмешать один битый элемент в батч, проверить что весь батч откатился, ни одна запись не осталась. *(Покрыто в test_houses bulk.)*
5. **Soft-delete семантика — отдельная серия тестов**: DELETE → deleted_at выставлен, GET list исключает, GET {id} → 404, повторный DELETE → 404. Для каждой сущности с SoftDeleteMixin.
6. **Pagination edges**: `limit=1`, `limit=200` (max), `limit=201` → 422, `offset=-1` → 422, `offset=last_page`, `offset=99999` (пустой результат + корректный total).
7. **Формат ошибок ADR 0005** — каждый класс ошибок (404/403/422/409) хотя бы одним тестом проверить полный envelope `{error: {code, message, details}}`.
8. **RBAC-матрица**: для каждого write-эндпоинта явно проверить отказ для всех не-допустимых ролей (4 роли — owner, accountant, construction_manager, read_only). Параметризованные тесты приветствуются.
9. **QA не чинит код**. Найден баг → в `bug_log.md` с BUG-id, тест помечен `xfail`/`skip` с комментарием. Чинит код — программист после возврата задачи.
10. **Reviewer работает на staged diff ДО коммита**, не после. Принципиально.

## Стандарты качества тестов

| Параметр | Цель |
|---|---|
| Покрытие строк | ≥85% |
| Покрытие веток | ≥80% |
| На каждый эндпоинт | ≥1 happy + ≥1 403 + ≥1 404 + ≥1 422 + ≥1 audit |
| RBAC | 4 роли × все write-эндпоинты |
| Pagination | минимум 5 граничных тестов на сущность |
| Формат ошибок | минимум 4 класса × 1 тест каждый |
| Тесты flapping | 0 (известный flaky timing-тест из Phase 2 — на marker) |

## Стандарты ревью

| Параметр | Цель |
|---|---|
| Время от запроса до отчёта (один файл) | ≤15 мин |
| Время на финальное ревью батча | ≤30 мин |
| Классификация замечаний | P0 (BLOCKER) / P1 (MAJOR) / P2 (MINOR) / P3 (NIT) |
| Формат отчёта | docs/reviews/<phase>-<step>-<date>[-roundN].md |
| Approve без request-changes | ≥50% (цель к концу Фазы 3) |

## Стандарты security-аудита

- Полный прогон по OWASP Top 10 — раз в фазу
- Грубо: каждая фаза с web-эндпоинтами проходит чек-лист `owasp-top10-checklist`
- Для Batch B специфично: проверка на CSRF (формы), SSRF (если есть внешние интеграции), массовое назначение (mass assignment в Pydantic)

### Шаблон промпта для qa (тесты сущности)

```
Напиши интеграционные тесты для <Name> CRUD по образцу test_projects.py.

ОБЯЗАТЕЛЬНО прочти:
1. /root/coordinata56/CLAUDE.md
2. /root/coordinata56/docs/agents/departments/quality.md
3. backend/tests/test_projects.py — образец стиля
4. ADR 0005 (формат ошибок), ADR 0006 (пагинация)

Покрой:
- happy path: create, read, list, update, delete
- 403: чужая роль на write × 4 роли (параметризованно)
- 404: get/update/delete несуществующего id
- 422: невалидные данные (Pydantic)
- audit: после write — запись в audit_log с правильным entity, action, user_id
- soft-delete: deleted_at + list исключение + повторный 404 (если применимо)
- pagination edges: limit=1, =200, =201 (422), offset=-1 (422)
- формат ошибок: error.code на 404/403/422/409

Пароли в фикстурах — secrets.token_urlsafe(16). Без литералов.

FILES_ALLOWED: backend/tests/test_<name>s.py, backend/conftest.py (если нужны новые фикстуры)
FILES_FORBIDDEN: backend/app/* (qa код приложения не чинит — баги в bug_log.md)

git add. Не коммить.
Отчёт ≤200 слов: число тестов, покрытые категории, найденные баги (BUG-id если есть).
```
