**Версия**: 1.0

**Автор**: UX/UI-дизайнер (субагент `ux-designer`)  
**Персона**: Мартин, владелец холдинга, роль `owner`

## 1. Сценарий использования

### Первичный сценарий (десктоп)
> Мартин садится за ноутбук утром. Хочет за 60 секунд понять: всё ли идёт по плану, нет ли тревожных сигналов, где отставание по деньгам или срокам.

**Первые 3 действия после логина:**

1. **Взгляд на строку KPI-карточек** — видит 4 числа: потрачено / план / отставание / домов в стройке. Сразу понимает общее состояние без кликов.
2. **Проверяет блок алертов** — если красных алертов нет, успокаивается. Если есть — кликает на алерт, переходит к детали (дом, подрядчик, платёж).
3. **Смотрит прогресс по стадиям** — виджет «Прогресс по домам» показывает, сколько домов на каждой из 12 стадий. Если на «сдаче» или «кровле» неожиданно мало — открывает список домов по фильтру этой стадии.

### Вторичный сценарий (мобильный)
> Мартин в машине, 2 минуты на светофоре. Проверяет: были ли новые платежи, нет ли алертов.

**Действия:** открывает приложение → видит сжатый дашборд → 2 карточки KPI (потрачено / план) → алерты → кнопка «Все дома».

## 2. Информационная архитектура

### Разделы бокового меню

```
COORDINATA 56
─────────────────────────
[иконка] Дашборд          ← текущий раздел, подсвечен
[иконка] Дома             ← список 85 домов, фильтр по стадии/типу
[иконка] Финансы          ← план/факт по статьям, отклонения
[иконка] График           ← таймлайн стадий по домам
[иконка] Подрядчики       ← реестр, договоры, оплачено/остаток
[иконка] Отчёты           ← генерация выгрузок (Excel/PDF)
─────────────────────────
[иконка] Настройки
[иконка] Мартин (аватар)  ← выход, смена пароля
```

**Логика порядка:** от оперативного (дашборд → дома → финансы → график) к административному (подрядчики → отчёты → настройки). Самое важное — сверху, техническое — снизу. Соответствует принципу «Recognition over recall» (эвристика Нильсена #6).

## 4. Блоки — детальное описание

---

### 4.1 Топбар (верхняя навигация)

**Что показывает:**
- Логотип / название проекта «Координата 56»
- Заголовок текущего раздела (хлебная крошка: «Дашборд»)
- Иконка колокольчика с бейджем числа непрочитанных алертов (красный, если > 0)
- Аватар пользователя с выпадающим меню (профиль / смена пароля / выход)

**Компоненты shadcn/ui:**
- `NavigationMenu` — верхняя оболочка
- `Badge` (variant: `destructive`) — счётчик алертов, красный при наличии
- `DropdownMenu` — меню пользователя
- `Avatar` — аватар (инициалы «МВ» если фото нет)

**Интерактивность:**
- Клик на колокольчик → панель алертов (Drawer справа или Sheet)
- Клик на аватар → DropdownMenu с пунктами «Профиль», «Сменить пароль», «Выйти»

**Адаптив (мобильный):**
- Логотип схлопывается до иконки
- Заголовок раздела скрывается
- Остаются: иконка меню (Hamburger) + бейдж алертов + аватар
- Боковое меню превращается в Sheet (выезжает снизу вверх или сбоку)

### 4.2 Боковое меню (Sidebar)

**Что показывает:**
- Список разделов с иконками и текстовыми метками
- Активный пункт выделен цветом фона (accent)
- Счётчики рядом с пунктами: «Дома» → число домов с просроченными задачами; «Подрядчики» → число договоров с задержкой оплаты

**Компоненты shadcn/ui:**
- `Sidebar` (из shadcn/ui sidebar primitive) или ручная сборка на `nav` + `Button` (variant: `ghost`)
- `Badge` (variant: `secondary`) — счётчики рядом с пунктами меню
- `Separator` — разделитель между рабочими пунктами и служебными (Настройки, Профиль)
- `Tooltip` — при схлопнутом меню (иконки без текста) показывает название раздела

**Интерактивность:**
- Меню можно свернуть до иконок (кнопка «‹» в верхнем углу сайдбара)
- В свёрнутом состоянии: иконки + Tooltip с названием раздела (эвристика #6 — узнаваемость)
- Активный пункт не является кнопкой — не кликабелен повторно

**Адаптив (мобильный):**
- Сайдбар полностью скрыт
- Вызывается через кнопку «Hamburger» в топбаре → `Sheet` (компонент shadcn/ui) с полным меню поверх контента

### 4.3 Строка KPI-карточек

**Что показывает** (4 карточки в ряд на десктопе, 2×2 на планшете, 1 колонка на телефоне):

```
┌───────────────────┐  ┌───────────────────┐  ┌───────────────────┐  ┌───────────────────┐
│ ПОТРАЧЕНО         │  │ ПЛАН НА СЕГОДНЯ   │  │ ОТКЛОНЕНИЕ        │  │ ДОМОВ В СТРОЙКЕ   │
│                   │  │                   │  │                   │  │                   │
│  ₽ 124 850 000   │  │  ₽ 130 000 000   │  │  -₽ 5 150 000    │  │        47 / 85    │
│                   │  │                   │  │  ▼ -3.96%        │  │                   │
│ с начала проекта  │  │ накоп. план       │  │ (экономия)       │  │ активных домов    │
└───────────────────┘  └───────────────────┘  └───────────────────┘  └───────────────────┘
```

**Семантика цвета отклонения:**
- Зелёный — расходуем меньше плана (экономия или опережение)
- Красный (`destructive`) — перерасход более 5%
- Жёлтый (`warning`) — отклонение 2–5%
- Серый — данных недостаточно

**Компоненты shadcn/ui:**
- `Card` + `CardHeader` + `CardContent` — основа каждой плитки
- `Badge` (variant: `destructive` / `success` / `warning`) — дельта в процентах
- Иконка тренда (стрелка вверх/вниз) рядом с суммой — через lucide-react (`TrendingUp`, `TrendingDown`)

**Интерактивность:**
- Клик на карточку «Потрачено» → переход в раздел «Финансы» с предустановленным фильтром «все расходы»
- Клик на «Домов в стройке» → переход в раздел «Дома» с фильтром «статус: в работе»
- Клик на «Отклонение» → переход в «Финансы» → вкладка «Отклонения»
- Hover на карточке — лёгкое поднятие тени (CSS transition, не компонент)

**Адаптив:**
- Десктоп (≥1024px): 4 карточки в ряд, равная ширина
- Планшет (768–1023px): 2 карточки × 2 строки
- Мобильный (<768px): горизонтальный скролл (scroll-snap) — 2 карточки видно сразу, остальные за экраном, точки-индикаторы снизу

- Красный — критично, требует действия сегодня
- Жёлтый — предупреждение, требует внимания
- Зелёный — позитивное событие (опережение графика, закрытие платежа)

- `Card` — обёртка блока
- `Alert` (variant: `destructive` / `default`) — каждый алерт как элемент

**Интерактивность:**
- Клик на алерт → Sheet (боковая панель) с деталями: что случилось, какой дом/подрядчик, история
- Кнопка «Закрыть алерт» внутри Sheet — убирает из ленты, фиксирует в audit_log
- Клик «Все алерты» → отдельная страница с фильтрами по типу / дому / подрядчику / дате

**Адаптив:**
- На мобильном блок занимает полную ширину
- Детали алерта (вторая строка с суммой/прорабом) скрываются, остаётся только заголовок
- Кнопка «Все алерты» становится `Button` full-width

## 7. Соответствие WCAG 2.2 AA

- **Контрастность текста**: все цвета текста на фоне обеспечивают минимум 4.5:1 (проверить при выборе цветовой палитры)
- **Красный/зелёный без формы**: цветовые маркеры алертов дублируются иконкой (кружок, треугольник, галочка) — не только цвет
- **Фокус клавиатуры**: все интерактивные элементы shadcn/ui имеют видимый outline при focus-visible
- **Размер целевой зоны**: кнопки и строки таблицы — минимум 44×44px на мобильном (WCAG 2.5.8 Target Size)
- **Семантическая разметка**: `nav`, `main`, `aside` для меню/контента/сайдбара; заголовки `h1–h3` в иерархии
- **ARIA**: счётчик алертов — `aria-label="3 непрочитанных алерта"`; Progress — `aria-valuenow`, `aria-valuemax`

## 9. Открытые вопросы (требуют уточнения у Мартина)

1. **KPI — какой период «плана»?** Накоплено с начала проекта или план текущего месяца? Это меняет смысл карточки «Отклонение». Вариант А — «план нарастающим итогом» (показывает общее состояние проекта), Вариант Б — «план текущего месяца» (оперативный контроль периода). Рекомендую Вариант А как основной с переключателем.

2. **Алерты — кто их генерирует?** Нужна логика триггеров (перерасход >X%, нет обновлений >Y дней). Пороговые значения — настраиваемые Мартином или зашитые в код? Рекомендую: зашитые умолчания с возможностью изменить в Настройках.

3. **Домов в стройке — что считается «в стройке»?** Все дома кроме «Не начат» и «Сдача»? Или есть другое определение?

4. **Мобильный — отдельное приложение или адаптивный сайт?** Wireframe построен как адаптивный веб. Если нужно нативное мобильное приложение — архитектура фронтенда меняется.

5. **KPI-каталог от data-analyst** — после его завершения карточки KPI могут измениться. Текущий набор («потрачено / план / отклонение / домов в стройке») — стартовая гипотеза. Финальный набор согласовывается с Мартином после получения каталога KPI.
