## Что такое этот файл

Это главный указатель по всей документации проекта. Он отвечает на три вопроса:

1. Что здесь есть?
2. Кому это читать?
3. С чего начать?

Документация строится по системе **Diátaxis** — это международный стандарт технической документации, который делит все документы на четыре типа в зависимости от цели чтения.

## Четыре типа документов по Diátaxis

### Tutorials (обучение)

Цель: провести читателя за руку через что-то новое. Читатель учится, делая.
Вопрос, на который отвечает: «Как мне начать?»

### How-to (инструкции)

Цель: помочь решить конкретную задачу. Читатель уже знает систему, ему нужно сделать одно конкретное дело.
Вопрос, на который отвечает: «Как мне сделать X?»

### Reference (справочник)

Цель: дать точную информацию без пояснений. Читатель знает, что ищет.
Вопрос, на который отвечает: «Что значит это поле / какой у него формат?»

### Explanation (объяснение)

Цель: дать понимание, как устроено и почему именно так. Читатель хочет разобраться в логике.
Вопрос, на который отвечает: «Почему сделано именно так?»

## Соглашения

**Один документ — одна задача.** Документ охватывает ровно одну тему. Нельзя смешивать tutorial и reference в одном файле.

**Версионирование через git.** Изменения в документации фиксируются в git-коммитах с понятными сообщениями, например: `docs: добавлен how-to по выгрузке отчёта`. Ручной нумерации версий в документах нет — git хранит всю историю.

**Актуальность.** Документация обновляется одновременно с кодом. Устаревшая документация хуже отсутствующей.

**Диаграммы.** Схемы и диаграммы оформляются в формате Mermaid прямо в Markdown-файлах. Это позволяет видеть диаграмму в браузере и хранить её как текст в git.

**Без мусора.** В `docs/` нет черновиков, «временных» файлов, копий с суффиксом `_old`. Если документ не готов — его ещё нет в репозитории.
