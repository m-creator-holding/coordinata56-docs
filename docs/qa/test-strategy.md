# Test Strategy — coordinata56 MVP1

- **Статус**: принято
- **Дата**: 2026-04-14
- **Автор**: QA-инженер (субагент `qa`)
- **Проект**: coordinata56 — ERP коттеджного посёлка «Координата 56»
- **Стек**: Python 3.12 / FastAPI / SQLAlchemy 2.0 / PostgreSQL 16 / React 18 / TypeScript / Vite / shadcn/ui
- **Фазы под покрытием**: Фаза 2–9 (Фазы 0–1 — инфраструктура, тесты ретроспективно)

## 1. Пирамида тестирования

```
        /‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾\
       /       E2E  (10%)       \
      /    Playwright             \
     /‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾\
    /   Integration  (30%)         \
   /   pytest-asyncio + httpx        \
  /     vitest + testing-library      \
 /‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾\
/          Unit  (60%)                 \
/  pytest / vitest, без внешних систем  \
‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾
```

### Пропорции и обоснование

| Уровень | Доля | Время выполнения | Количество тестов |
|---------|------|-----------------|-------------------|
| Unit | 60% | < 10 сек (весь сьют) | 200–400 |
| Integration | 30% | < 60 сек | 80–150 |
| E2E | 10% | < 5 мин | 20–40 |

Пирамида смещена в сторону unit-тестов, потому что:
- Бизнес-логика расчётов (план/факт, остатки по договорам, совместимость опций) концентрируется в сервисном слое backend — именно она критична и сложна.
- E2E-тесты дороги в поддержке: при MVP-скорости разработки их слой остаётся тонким и покрывает только happy path ключевых пользовательских сценариев.

## 2. Что тестируется на каждом уровне

### 2.1 Unit-тесты (backend)

**Где живут**: `tests/unit/`

**Что тестируем:**

| Модуль | Примеры тест-кейсов |
|--------|---------------------|
| `app/models/` — SQLAlchemy-модели | Корректность миксинов (timestamps, soft delete), значения по умолчанию, enum-значения |
| `app/schemas/` — Pydantic-схемы | Валидация обязательных полей, граничные значения (0 копеек, max bigint), отклонение недопустимых типов |
| `app/services/` — бизнес-логика | Расчёт отклонения план/факт, итоги по бюджету, остаток по договору (оплачено / осталось), совместимость опций с типом дома |
| `app/core/security` — пароли и JWT | Хеширование bcrypt, верификация, генерация и парсинг JWT, истечение токена |
| `app/core/permissions` — RBAC | Матрица прав: owner видит всё, read_only не может писать, accountant не имеет доступа к CRUD домов |

**Изолируем**: БД через `unittest.mock` / монки-патч сессии. Нет сетевых вызовов.

---

### 2.2 Unit-тесты (frontend)

**Где живут**: `frontend/src/__tests__/unit/`

**Что тестируем:**

| Что | Примеры |
|-----|---------|
| Утилиты и хелперы | Форматирование копеек в рубли (`100 → "1,00 ₽"`), форматирование дат, обрезка длинных строк |
| Компоненты-примитивы | `<StatusBadge>` рендерит нужный цвет для каждого статуса дома, `<MoneyInput>` принимает только числа |
| Логика форм (zod-схемы) | Обязательность полей, форматы, кросс-полевые правила |
| Хранилище состояния (если Zustand) | Мутации токена, очистка при logout |
| React-хуки (custom hooks) | `useAuth` возвращает корректное состояние при разных вариантах хранилища |

**Изолируем**: API через `vi.mock`, роутер через `MemoryRouter`.

---

### 2.3 Integration-тесты (backend)

**Где живут**: `tests/integration/`

**Что тестируем:**

| Сценарий | Описание |
|----------|----------|
| API-эндпоинты (FastAPI TestClient / httpx) | POST создаёт запись в реальной тестовой БД, GET возвращает её, DELETE (soft) помечает deleted_at |
| JWT flow | Регистрация → логин → получение защищённого ресурса с токеном → запрос без токена → 401 |
| RBAC на эндпоинтах | Пользователь с ролью `read_only` получает 403 при попытке POST /houses |
| Audit log | После каждого мутирующего запроса запись появляется в `audit_log` |
| Алембик-миграции | Применение `upgrade head`, откат `downgrade -1`, повторное `upgrade head` — без ошибок |
| Cascade и FK | Удаление проекта блокируется при наличии дочерних записей (RESTRICT) |
| Seeds | Скрипт seeds наполняет ровно 85 домов, 4 типа, 8 опций |
| Пагинация | GET /houses?page=1&size=10 возвращает 10 записей, `total` = 85 |

**Инфраструктура**: отдельная PostgreSQL в Docker (`coordinata56_test`). Перед каждым тест-запуском — накатываем миграции, после — откатываем (или пересоздаём схему).

---

### 2.4 Integration-тесты (frontend)

**Где живут**: `frontend/src/__tests__/integration/`

**Что тестируем:**

| Сценарий | Описание |
|----------|----------|
| Страница логина | Заполнение формы → нажатие submit → проверка redirect на `/dashboard` (API замокан) |
| Таблица домов | Компонент загружает мок-данные, рендерит строки, поиск фильтрует, сортировка работает |
| Карточка дома | Загружает данные, показывает вкладки, переключение не ломает layout |
| Форма бюджета | Ввод значений → submit → вызов API с правильными данными |
| Обработка 401 | При ответе API 401 пользователь редиректится на `/login`, токен очищается |
| Обработка пустых данных | Таблица без данных показывает empty state, а не крэш |

**Инфраструктура**: `msw` (Mock Service Worker) для перехвата HTTP на уровне браузера.

---

### 2.5 E2E-тесты (Playwright)

**Где живут**: `tests/e2e/`

**Что тестируем — только критические user journey:**

| Сценарий | Шаги |
|----------|------|
| Полный логин-цикл | Открыть `/login` → ввести credentials → попасть на дашборд → выйти → вернуться на `/login` |
| Просмотр реестра домов | Залогиниться → перейти в «Дома» → убедиться, что таблица загрузилась (85 строк) |
| Бюджет: ввод и просмотр | Залогиниться → открыть дом → ввести плановый бюджет → сохранить → проверить в дашборде |
| Запрет доступа по роли | Залогиниться под `read_only` → попытаться создать запись → получить запрет |
| Выгрузка отчёта | Залогиниться → перейти в отчёты → нажать «Скачать Excel» → файл скачан |
| Аудит-лог | Залогиниться под owner → изменить бюджет → зайти в аудит → найти запись |

**Инфраструктура**: Playwright запускается против staging-стенда (или локальный `docker compose up` с seed-данными). Каждый тест начинается с чистого состояния через API-вызов сброса тестовых данных.

## 4. Тестовые данные

### 4.1 Принципы

- Каждый тест создаёт только те данные, которые нужны ему. Не полагается на глобальное состояние БД.
- Unit-тесты: данные создаются через Python-объекты или `factory_boy` без сохранения в БД.
- Integration-тесты: фабрики сохраняют записи в тестовую БД; транзакция откатывается в teardown.
- E2E-тесты: отдельная seed-процедура через API (endpoint `/api/v1/test/reset` доступен только при `TESTING=true`).

### 4.2 Factory Boy — структура фабрик

Файл: `tests/factories.py`

```python
# Пример структуры — реализация задача разработчика
class ProjectFactory(factory.alchemy.SQLAlchemyModelFactory):
    class Meta:
        model = Project
        sqlalchemy_session_persistence = "commit"

    name = factory.Faker("company", locale="ru_RU")
    description = factory.Faker("text", max_nb_chars=200, locale="ru_RU")
    is_active = True

class HouseTypeFactory(factory.alchemy.SQLAlchemyModelFactory):
    class Meta:
        model = HouseType
    code = factory.Sequence(lambda n: f"TYPE_{n}")
    area_sqm = factory.Faker("pydecimal", left_digits=3, right_digits=2, positive=True)

class HouseFactory(factory.alchemy.SQLAlchemyModelFactory):
    class Meta:
        model = House
    number = factory.Sequence(lambda n: n + 1)
    house_type = factory.SubFactory(HouseTypeFactory)
    project = factory.SubFactory(ProjectFactory)
    status = HouseStatus.foundation

class UserFactory(factory.alchemy.SQLAlchemyModelFactory):
    class Meta:
        model = User
    email = factory.Faker("email")
    hashed_password = factory.LazyFunction(lambda: hash_password("Test1234!"))
    role = UserRole.read_only
    is_active = True
```

### 4.3 Фикстуры pytest

Файл: `tests/conftest.py`

Обязательные фикстуры:

| Фикстура | Scope | Назначение |
|----------|-------|-----------|
| `engine` | session | Один движок для всей тест-сессии (PostgreSQL test DB) |
| `db_session` | function | Транзакция + rollback после каждого теста |
| `async_client` | function | `httpx.AsyncClient` с `app` FastAPI, с тестовым `db_session` |
| `owner_token` | function | JWT-токен пользователя с ролью owner |
| `accountant_token` | function | JWT-токен accountant |
| `readonly_token` | function | JWT-токен read_only |
| `seed_houses` | function | 85 домов через HouseFactory в тестовой БД |

### 4.4 Граничные случаи — обязательный минимум

Каждый сервисный метод должен иметь тесты на:

- Пустые коллекции (нет домов, нет строк бюджета)
- Нулевые суммы (план = 0, факт = 0)
- Максимальные суммы (bigint-граница: 9 223 372 036 854 775 807 копеек)
- Soft-deleted записи (не должны попадать в выборки)
- Конкурентные записи (два одновременных POST — нет дублей благодаря уникальным индексам)
- Некорректные foreign key (запись с несуществующим `house_id`)

## 5. Критерии покрытия кода

### 5.1 Backend

| Слой | Минимум coverage | Обоснование |
|------|-----------------|-------------|
| `app/services/` (бизнес-логика) | **80%** | Расчёты денег, план/факт — критические пути |
| `app/api/` (роутеры / эндпоинты) | **75%** | Покрывается integration-тестами |
| `app/models/` (ORM-модели) | **70%** | Миксины, валидаторы, методы |
| `app/core/` (security, config, permissions) | **80%** | JWT, bcrypt, RBAC — безопасность |
| `app/schemas/` (Pydantic) | **65%** | Валидация схем через unit-тесты |
| **Итого по проекту** | **>= 70%** | Без фанатизма: миграции и seed не считаются |

Что НЕ покрывается принудительно:
- `alembic/` — миграции тестируются отдельным интеграционным сценарием, не через coverage
- `app/db/seeds.py` — проверяется вручную
- `app/main.py` — точка входа, только smoke-тест через `/health`

### 5.2 Frontend

| Слой | Минимум coverage | Обоснование |
|------|-----------------|-------------|
| Утилиты и хелперы (`lib/`) | **80%** | Форматирование денег, дат — легко тестируются |
| Хуки (`hooks/`) | **70%** | Логика, которую видит пользователь |
| Компоненты-примитивы | **60%** | Поведение, не вёрстка |
| Страницы (page-компоненты) | **50%** | Покрываются E2E |
| **Итого по frontend** | **>= 60%** | Вёрстка и pixel-perfect — не в coverage |

Что НЕ покрывается: CSS-классы Tailwind, иконки, SVG, Storybook-конфиги.

## 6. Критерии выхода из фазы (Definition of Done)

Фаза считается закрытой, когда **все** пункты ниже выполнены.

- [ ] Все автотесты в `tests/` и `frontend/src/__tests__/` зелёные (0 failed, 0 errors)
- [ ] Покрытие кода соответствует минимумам из раздела 5
- [ ] Нет открытых багов со severity **Blocker** или **Critical** (определения в разделе 8)
- [ ] Код прошёл линтинг: `ruff check` и `mypy --strict` без ошибок (backend), `tsc --noEmit` без ошибок (frontend)

- [ ] Ручная проверка: Мартин входит под своей учёткой, JWT видится в devtools

### Фаза 3 — CRUD API

- [ ] Integration-тест для каждого CRUD-эндпоинта (create, read, read list, update, soft delete)

### Фаза 4 — Frontend-скелет

- [ ] Unit-тесты утилит форматирования
- [ ] Integration: страница логина — форма валидирует, submit вызывает API, redirect работает

### Фаза 5 — Бюджет

- [ ] Unit-тесты сервиса расчёта план/факт/отклонение (в т.ч. нули и граничные суммы)

- [ ] E2E: полный сценарий «ввод бюджета → дашборд»

- [ ] SSL-сертификат валидный (не expired, не self-signed)

## 7. CI Pipeline

### 7.1 На каждый Pull Request

```
PR opened / pushed
│
├── [fast, parallel]
│   ├── ruff check backend/
│   ├── mypy backend/
│   ├── tsc --noEmit frontend/
│   └── vitest run --coverage (frontend unit + integration)
│
└── [after lint pass]
    ├── pytest tests/unit/ -v
    └── pytest tests/integration/ -v --cov=app --cov-report=xml
        └── coverage gate: fail if < 70% backend / < 60% frontend
```

**Время выполнения цели**: < 3 минут. PR не мержится при любом падении.

### 7.2 На merge в main (post-merge)

```
merge → main
│
├── Всё из PR-pipeline (повторно, чистая среда)
│
└── [дополнительно]
    └── playwright test (e2e, headless Chromium)
        └── отчёт HTML сохраняется как artifact
```

**Время**: < 10 минут. Падение E2E блокирует следующий деплой (не откатывает merge).

### 7.3 Переменные окружения для CI

| Переменная | Откуда | Назначение |
|-----------|--------|-----------|
| `DATABASE_URL` | CI secret | Тестовая PostgreSQL (создаётся в Docker service) |
| `SECRET_KEY` | CI secret | JWT signing key |
| `TESTING` | hardcoded `true` | Включает `/api/v1/test/reset` endpoint |
| `BASE_URL` | CI var | URL для Playwright (локальный или staging) |

### 8.2 Severity — определения

| Severity | Определение | Пример для проекта |
|----------|------------|-------------------|
| **Blocker** | Система не запускается или ключевой flow полностью сломан. Разработка стоит. | Backend не стартует, login не работает, БД не подключается |
| **Critical** | Критическая функция работает неверно, данные повреждаются или теряются. Обходной путь отсутствует. | Неверный расчёт суммы бюджета, платёж не сохраняется, аудит-лог не пишется |
| **Major** | Важная функция работает, но с существенными отклонениями. Есть неудобный обходной путь. | Пагинация возвращает лишние записи, фильтр не работает по одному полю, ошибка в PDF-отчёте |
| **Minor** | Функция работает корректно, но есть косметические или UX-проблемы. Не влияет на данные. | Неверный текст ошибки валидации, неправильная сортировка в таблице, лишний пробел |
| **Trivial** | Мелочи: опечатки, стилевые несоответствия, неудобства без последствий. | Опечатка в лейбле, неверный цвет тени кнопки |

### 8.3 SLA на исправление

| Severity | SLA | Действие при нарушении |
|----------|-----|------------------------|
| Blocker | Исправить до конца рабочего дня обнаружения | QA блокирует все другие задачи фазы, эскалация к Владельцу |
| Critical | 2 рабочих дня | QA блокирует переход к следующей фазе |
| Major | До конца текущей фазы | QA фиксирует, включает в DoD фазы |
| Minor | До MVP1 (не блокирует фазу) | Ставится в бэклог |
| Trivial | По усмотрению (после MVP1) | Ставится в бэклог |

## 9. Регрессионный ручной чеклист (перед релизом MVP1)

Выполняется QA и Мартином совместно перед Gate 9 → MVP1. Каждый пункт подписывается.

### 9.1 Аутентификация и безопасность

- [ ] Вход с корректными credentials → попадаем на дашборд
- [ ] Вход с неверным паролем → сообщение об ошибке, не 500
- [ ] Попытка открыть `/dashboard` без токена → редирект на `/login`
- [ ] Токен истекает (проверить с коротким TTL) → редирект на `/login`
- [ ] Пользователь `read_only` не видит кнопки создания/редактирования
- [ ] Пользователь `accountant` не имеет доступа к разделу «Дома» (если ограничено по роли)
- [ ] Выход → токен удалён → повторный доступ требует логина

### 9.2 Реестр домов

- [ ] Таблица показывает 85 домов (4 типа: A/B/C/D)
- [ ] Фильтрация по типу дома работает
- [ ] Поиск по номеру дома работает
- [ ] Карточка дома открывается, показывает тип, статус, стадию
- [ ] Смена статуса дома сохраняется, история стадий обновляется

### 9.3 Бюджет

- [ ] Ввод планового бюджета по статьям — сохраняется
- [ ] Ввод фактического расхода — сохраняется
- [ ] Дашборд: итоги план/факт/отклонение корректны (проверить вручную на 3–5 домах)
- [ ] Разбивка по статьям, по стадиям — суммы совпадают с итогом
- [ ] Нулевой бюджет (нет записей) — пустое состояние, не ошибка

### 9.4 Закупки и материалы

- [ ] Добавление закупки — сохраняется в реестре
- [ ] Поиск по материалу — находит
- [ ] Сводка «материалы на дом» — корректно агрегирует

### 9.5 Подрядчики, договоры, платежи

- [ ] Создать подрядчика → появляется в реестре
- [ ] Создать договор с подрядчиком → привязан, сумма отображается
- [ ] Добавить платёж → «оплачено» увеличивается, «осталось» уменьшается
- [ ] Попытка ввести платёж, превышающий сумму договора → бизнес-ошибка, не 500

### 9.6 Аудит-лог

- [ ] Создание дома → запись в аудит-логе
- [ ] Изменение бюджета → запись с diff
- [ ] Экран аудита: записи видны, фильтрация по дате работает

### 9.7 Отчёты

- [ ] Скачать «Бюджет проекта» → Excel открывается, данные корректны
- [ ] Скачать «Карточка дома» → PDF открывается, не пустой

### 9.8 Производительность (визуальная)

- [ ] Таблица 85 домов загружается < 2 секунды (network throttle: Fast 3G)
- [ ] Дашборд загружается < 3 секунды
- [ ] Нет видимых reflow при переключении вкладок

### 9.9 Браузерная совместимость

- [ ] Chrome (последняя версия) — всё работает
- [ ] Firefox (последняя версия) — всё работает
- [ ] Safari (последняя версия, macOS) — всё работает

### 9.10 Деплой (staging)

- [ ] HTTPS работает, сертификат не просрочен
- [ ] `/health` возвращает 200
- [ ] Вход работает на реальном домене
- [ ] Бэкап БД создан и проверен (файл не пустой)

## 10. Тестирование производительности (нефункциональное)

В рамках MVP1 — только базовая проверка, без нагрузочного тестирования.

| Метрика | Цель | Как проверяем |
|---------|------|---------------|
| GET /houses (85 записей) | < 200 мс (p95) | Ручная проверка через DevTools Network |
| GET /dashboard (агрегаты) | < 500 мс (p95) | Ручная проверка |
| POST /budget-entries | < 300 мс (p95) | Ручная проверка |
| Время загрузки SPA (первый визит) | < 3 сек (Fast 3G) | Lighthouse |

Полноценное нагрузочное тестирование (locust / k6) — откладывается до M2.
