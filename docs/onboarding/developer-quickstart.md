# Быстрый старт разработчика

> **Для кого:** новый разработчик, только что получил доступ к репозиторию.
> **Цель:** поднять рабочую локальную среду с нуля за 15–20 минут.
> **Тип документа (Diátaxis):** Tutorial — пошаговое руководство.

---

## Что вы получите в итоге

После этого руководства у вас будет работающая среда разработки:
- Backend на `http://127.0.0.1:8000` с автоматической перезагрузкой при изменении кода.
- Frontend с Admin UI на `http://127.0.0.1:5173` с мгновенным обновлением в браузере.
- База данных PostgreSQL с тестовыми данными (seed).
- Pre-commit хуки, которые не дадут закоммитить секреты или сломанный код.

## Шаг 1. Убедитесь, что всё необходимое установлено

Вам нужны:

| Инструмент | Зачем | Как проверить |
|------------|-------|---------------|
| Docker Desktop (или Docker Engine) | Запускает все сервисы в контейнерах | `docker --version` |
| Docker Compose v2 | Управляет несколькими контейнерами одной командой | `docker compose version` |
| GNU Make | Удобные команды (`make up`, `make logs`, ...) | `make --version` |
| Git | Клонирование и работа с репозиторием | `git --version` |

**macOS:** Docker Desktop + Xcode CLI Tools (`xcode-select --install`) — этого достаточно.

**Windows:** установите WSL2 (подсистема Linux) и Docker Desktop с интеграцией WSL2.

**Linux:** Docker Engine + Docker Compose plugin + `sudo apt install make`.

## Шаг 3. Настройте переменные окружения

В корне проекта лежит файл-шаблон `.env.dev.example`. Скопируйте его:

```bash
cp .env.dev.example .env.dev
```

Откройте `.env.dev` в любом текстовом редакторе и обязательно смените два значения:

```
POSTGRES_PASSWORD=<придумайте пароль ≥16 символов>
JWT_SECRET_KEY=<случайная строка ≥32 символа, напр. python -c "import secrets; print(secrets.token_urlsafe(32))">
```

Файл `.env.dev` никогда не попадает в git (он в `.gitignore`). Если случайно попадёт — pre-commit хук H-1 заблокирует коммит.

---

## Шаг 4. Установите pre-commit хуки

Один раз после клонирования:

```bash
bash scripts/install-hooks.sh
```

Хуки — это автоматические проверки, которые срабатывают при каждом `git commit`. Они защищают от типичных ошибок:

| Хук | Что проверяет |
|-----|---------------|
| H-1 | Нет ли в коммите паролей, токенов, `.env`-файлов |
| H-2 | Предупреждает, если вы случайно добавляете чужие файлы |
| H-5 | Запускает линтер и тесты для изменённых файлов |

Подробнее: `scripts/hooks/README.md`.

## Шаг 5. Запустите систему

```bash
make up
```

Эта команда:
1. Скачивает Docker-образы (первый раз — 2–5 минут, зависит от интернета).
2. Создаёт контейнеры: PostgreSQL, backend, frontend, Adminer.
3. Применяет миграции базы данных автоматически.
4. Запускает все сервисы.

Дождитесь строки в логах:
```
backend  | Application startup complete.
frontend | ready in XXX ms
```

Откройте в браузере:

| Адрес | Что должны увидеть |
|-------|-------------------|
| http://127.0.0.1:8000/docs | Swagger — список всех API-эндпоинтов |

## Повседневная работа

### Основные команды

```bash
make up           # поднять все сервисы
make down         # остановить (данные сохраняются)
make logs         # смотреть логи в реальном времени
make logs-back    # только логи backend
make psql         # открыть psql-консоль PostgreSQL
make migrate      # применить новые миграции (alembic upgrade head)
make shell-back   # войти в bash внутри backend-контейнера
make build        # пересобрать образы (нужно после изменений Dockerfile)
make reset-db     # ОСТОРОЖНО: удалить все данные и начать заново
```

### Создание новой миграции БД

```bash
make migration MSG="добавить поле description в contracts"
```

Файл миграции появится в `migrations/versions/`. Отредактируйте его, затем:

```bash
make migrate
```

### Запуск тестов

```bash
# Внутри backend-контейнера:
make shell-back
pytest
```

Или для конкретного файла:
```bash
pytest tests/test_companies.py -v
```

## Как устроена hot-reload перезагрузка

- **Backend:** при изменении любого `.py`-файла в `backend/` uvicorn автоматически перезапускается. Ждать `make build` не нужно — изменения применяются через 1–2 секунды.
- **Frontend:** при изменении файла в `frontend/src/` Vite автоматически обновляет нужный модуль в браузере без перезагрузки страницы.

Это работает потому, что исходники монтируются напрямую из вашей файловой системы в контейнер (Docker volume).

## Частые проблемы

**`make up` зависает или контейнеры не стартуют**

Проверьте, запущен ли Docker: `docker ps`. Если нет — запустите Docker Desktop.

**Backend не запускается, ошибка в логах про БД**

Обычно это значит, что PostgreSQL ещё не готов принимать подключения. Подождите 10–15 секунд и повторите `make logs-back`. Если ошибка про пароль — проверьте `.env.dev`.

**Порт уже занят (address already in use)**

Какой-то другой процесс занял порт 8000, 5173 или 5433. Найдите и остановите его, или измените порты в `docker-compose.yml`.

**Pre-commit хук H-1 блокирует коммит**

Хук нашёл в staged-файлах что-то похожее на секрет. Проверьте, не попал ли `.env.dev` или токен в коммит (`git status --short`). Если это ложное срабатывание — добавьте в строку комментарий `# hook-exception: H-1 <причина>`.

**Не знаю, к кому обратиться**

```
Координатор (Президент)
  └── backend-director (Директор)
        └── backend-head (Начальник отдела)
              └── backend-dev (вы)
```

Вопросы по задаче — к **backend-head**. Вопросы про архитектуру — backend-head эскалирует к architect. Нарушение безопасности — сразу к backend-director, без цепочки.
