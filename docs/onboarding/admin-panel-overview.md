# Admin UI — обзор административной панели

> **Для кого:** оператор системы, директор, бухгалтер — тот, кто работает с системой, а не пишет код.
> **Цель:** понять, что делает каждая страница Admin UI и кто что может видеть.
> **Тип документа (Diátaxis):** Explanation — объяснение устройства системы.

---

## Что такое Admin UI

Admin UI — это веб-интерфейс для управления системой: компаниями, пользователями, правами доступа и бизнес-правилами. Он доступен по адресу `http://<адрес-сервера>/admin`.

Admin UI предназначен для сотрудников с правами администратора. Обычные пользователи (бухгалтеры, прорабы, наблюдатели) работают в основном интерфейсе, а не здесь.

---

## Кто может войти в Admin UI

Войти может любой пользователь с учётной записью в системе. Но то, что он увидит и сможет сделать, зависит от его роли:

| Роль | Что доступно |
|------|-------------|
| Суперадмин (holding owner) | Все страницы, полный доступ |
| Администратор компании | Страницы своей компании |
| Бухгалтер | Настройки компании — только чтение (редактировать нельзя) |
| Прораб, Наблюдатель | Административная панель недоступна |

## Страница 1: Матрица прав (`/admin/permissions`)

### Что это

Таблица, которая показывает: **кто** (какая роль) **что может делать** (создавать, читать, редактировать, удалять) **с какими данными** (договоры, платежи, пользователи и т. д.).

### Как выглядит

По горизонтали — роли (Владелец, Бухгалтер, Прораб и другие). По вертикали — ресурсы и действия. На пересечении — галочка (право есть) или пустая ячейка (права нет).

Ресурсы сгруппированы по вкладкам: Финансы, Объекты, Пользователи и т. д.

### Что можно делать

- **Просматривать** матрицу — видеть, у кого какие права.
- **Редактировать** — включать и выключать права. Кнопка «Редактировать» переводит страницу в режим правки. Изменения применяются только после нажатия «Сохранить».
- **Фильтровать** — показать только одну роль через URL-параметр `?role=accountant`.
- **Скачать CSV** — выгрузить матрицу в файл для отчёта или сверки.
- **История** — посмотреть, кто и когда менял права конкретной роли.

### Важно знать

Изменение матрицы прав немедленно влияет на всех пользователей с этой ролью. Если убрать право «читать договоры» для бухгалтеров — они потеряют доступ при следующем запросе, без перезагрузки системы.

Права хранятся в базе данных, а не в коде. Поэтому изменение не требует участия разработчика и деплоя.

## Страница 2: Компании (`/admin/companies`)

### Что это

Реестр всех юридических лиц холдинга. Каждая компания — это отдельное юрлицо с собственными данными, пользователями и настройками.

### Как выглядит

Список компаний с названием, статусом (активна / деактивирована) и кнопкой перехода. По клику на компанию открывается страница с тремя вкладками.

### Вкладка «Детали»

Основная информация: название, ИНН, юридический адрес, контактное лицо.

### Вкладка «Пользователи»

Список сотрудников, привязанных к этой компании. Отсюда можно перейти на карточку конкретного пользователя.

### Вкладка «Настройки»

Семь параметров работы компании:

| Параметр | Что означает |
|----------|-------------|
| Режим НДС | Налоговый режим: ОСН, УСН 6%, УСН 15%, Патент |
| Валюта | Валюта для отображения сумм (только чтение, задаётся при создании) |
| Часовой пояс | Для корректного отображения времени операций |
| Рабочая неделя | Пятидневная или шестидневная (влияет на расчёт сроков) |
| Система единиц | Метрическая или имперская (для объёмов и площадей) |
| Бизнес-сегмент | Профиль компании (строительство, девелопмент и т. д.) |
| Ответственный | Контактный сотрудник по умолчанию |

Бухгалтер видит эту вкладку, но не может редактировать — все поля отображаются в режиме только чтение с информационным баннером.

### Что можно делать

- **Создать** новую компанию.
- **Редактировать** детали и настройки.
- **Деактивировать** компанию (мягкое удаление — данные не уничтожаются, компания скрывается из основного интерфейса).

## Страница 3: Пользователи (`/admin/users`)

### Что это

Реестр всех сотрудников, у которых есть учётные записи в системе. Здесь управляют доступом: создают пользователей, назначают роли, блокируют.

### Как выглядит

Таблица с именем, email, статусом (активен / заблокирован) и текущими ролями пользователя. В каждой строке есть выпадающий список для быстрой смены роли — не нужно заходить на отдельную страницу.

### Фильтрация

Над таблицей — строка поиска (по имени или email) и фильтр по роли. Это удобно, если нужно найти всех бухгалтеров или проверить, у кого есть роль прораба.

### Что можно делать

- **Создать** пользователя: имя, email, пароль (система предложит скопировать сгенерированный пароль при создании), привязка к компании.
- **Назначить роль** прямо в строке таблицы через выпадающий список.
- **Перейти** на страницу пользователя для детальной настройки (несколько компаний, история действий).
- **Заблокировать** пользователя — он не сможет войти в систему, но все его данные сохранятся.

### Важно знать

Email пользователя изменить нельзя после создания — он является уникальным идентификатором. Если сотрудник поменял корпоративный email, нужно создать новую учётную запись.

## Страница 4: Бизнес-правила (`/admin/rules`)

### Что это

Список параметров, которые управляют бизнес-логикой системы. Например: порог суммы, требующей двойного согласования; процент допустимого перерасхода бюджета; сроки уведомлений.

### Как выглядит

Слева — дерево категорий для навигации. Справа — таблица правил с названием, текущим значением, единицей измерения и статусом.

Правила разбиты на три категории:

| Категория | Примеры правил |
|-----------|----------------|
| **Финансы** | Лимит платежа без согласования, процент перерасхода бюджета |
| **Кадры** | Сроки уведомлений, параметры расчёта |
| **Процессы** | Таймауты, пороговые значения для автоматических действий |

### Что можно делать

- **Просматривать** правила и их текущие значения.
- **Редактировать** значение правила (только суперадмин).
- **Посмотреть снимок** — историческое значение правила на конкретный момент времени (кнопка «История» в строке).

### Важно знать

Изменить категорию правила или добавить новое правило может только суперадмин (holding owner). Это сделано намеренно: список правил — это часть конфигурации системы, а не операционных данных.

На текущем этапе (Wave 11) страница работает через тестовые данные (MSW mock). Подключение к реальному backend API запланировано в следующем спринте.

## Итоговая карта: кто что видит

```
Страница             Суперадмин  Администратор  Бухгалтер
─────────────────────────────────────────────────────────
/admin/permissions      RW           RW*            —
/admin/companies        RW           RW*           R (settings)
/admin/users            RW           RW*            —
/admin/rules            RW            —             —

RW = читать и редактировать
R  = только чтение
*  = только в пределах своей компании
—  = нет доступа
```

---

## Как попасть в конкретный раздел

Все разделы доступны через боковое меню Admin UI. Прямые ссылки:

- Матрица прав: `/admin/permissions`
- Матрица для конкретной роли: `/admin/permissions?role=accountant`
- Список компаний: `/admin/companies`
- Настройки компании: `/admin/companies/<id>/settings`
- Список пользователей: `/admin/users`
- Бизнес-правила: `/admin/rules`
