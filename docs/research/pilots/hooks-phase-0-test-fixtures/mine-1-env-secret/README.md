# Mine 1 — .env с литералом секрета

**Тестируемый хук:** H-1 (pre-commit secret block)

**Что симулируем:** разработчик случайно застейджил `.env` с реальным JWT-секретом и пытается закоммитить.

**Ожидаемая реакция:**
- `git commit` падает с exit code 1
- stderr содержит substring `HOOK H-1`
- stderr содержит путь к файлу (`.env.production` или `.env`)
- stderr содержит инструкцию «git reset HEAD» или «переименуйте в .env.example»

**Артефакты:**
- `.env.dirty` — содержит JWT_SECRET_KEY с литералом длиной >8 символов
- `reproduce.sh` — детерминированный сценарий воспроизведения

**Критические требования к фикстуре:**
1. Литерал помечен префиксом `FAKE_KEY_` для безопасности при коммите фикстуры.
2. При этом сам паттерн H-1 **обязан ловить** его: regex из плана `password\s*=\s*["'][a-zA-Z0-9]{8,}["']` матчит `FAKE_KEY_super_secret_12345` (длина 25 > 8, только [a-zA-Z0-9_]). Проверка в подзадаче Г по факту.
3. Если H-1 не сработал — P0 reject.

**Вторичный кейс (опционально в прогоне):**
- Файл имени `api.pem` (паттерн имени `*.pem`) — ожидание block даже если содержимое пустое.
- Файл `database_credentials.json` (паттерн `*credentials*`) — ожидание block.

**Успех = block (exit != 0).** Warning здесь неприемлем — это утечка секрета.
