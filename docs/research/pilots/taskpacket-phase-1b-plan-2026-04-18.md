## 1. Цель пилота

Проверить гипотезу: **структурированный TaskPacket v0.1 сокращает число уточняющих циклов (clarification loops) между Координатором и Директором / Директором и Head'ом и уменьшает брак на выходе (ручные правки после сдачи)** — на одной M-задаче, без масштабирования на все US 1.1A до получения измеримого сигнала.

Пилот — тактический, не платный, решение adopt/reject — моё (ri-director), эскалация Координатору только если пилот выявляет риск regressions в существующем workflow. Срок — ≤ 2 рабочих дня (лимит «Pilot» по регламенту R&I).

## 2. Рекомендация: какую M-задачу брать

### Кандидаты (даны в запросе)

| US | Размер | Исполнитель | Acceptance | Плюсы для пилота | Минусы |
|---|---|---|---|---|---|
| US-02 | M (2–3 дня) | backend-dev-2 | Clear (JWT claims + header + 3 теста) | Чёткий контракт на уровне API; один исполнитель; acceptance легко проверяется тестами | Зависит от US-01 (миграция) — если US-01 задержится, US-02 блокируется |
| US-04 | M (1–2 дня) | db-engineer + backend-dev-3 | Clear (2 таблицы + Pydantic discriminator) | Самый короткий цикл; изолированный скоуп | **Два исполнителя** — нарушает критерий «один исполнитель»; отвлекает на координацию db↔backend |
| US-06 | L (5 дней по декомпозиции) | backend-dev-1 | Чёткое (5 шагов runtime-guard + тесты) | Богатый контент для TaskPacket (много constraints, acceptance); backend-dev-1 | **L-размер** (не M); 5 дней — превышает лимит пилота R&I (2 дня); слишком большой скоуп для первой проверки формата |

### Вердикт: **US-02 (JWT + X-Company-ID middleware)**

Обоснование:
1. **Один исполнитель** (backend-dev-2) — чистая цепочка Координатор → backend-director → backend-head → backend-dev-2, без разрыва на двух Worker'ов.
2. **Размер M (2–3 дня)** — укладывается в лимит пилота R&I (2 дня на пилот, при сдвиге — резать скоуп или отклонять).
3. **Clear acceptance** — 3 конкретных теста (позитив с одной компанией, негатив с двумя, holding-owner bypass), результат измерим без субъективных оценок. Для TaskPacket это идеально: поле `acceptance` заполняется дословно из US, не требует интерпретации Директора.
4. **Contract-level задача** (JWT payload + HTTP header) — TaskPacket покрывает типичный сценарий «делаем API с явным контрактом», который потом повторится на 70% задач M-OS-1.
5. **Блокер — US-01 (миграция)** — реалистичен: в реальной работе задачи часто зависят от предшественников, TaskPacket поле `depends_on` проверяется на практике.

Альтернатива — **US-04**, если к моменту старта пилота US-01 ещё не закрыт. В этом случае пилот проводится только с **одним Worker'ом из пары** (db-engineer пишет миграцию отдельно по обычному процессу, TaskPacket идёт backend-dev-3 на Pydantic discriminator).

## 3. Формат TaskPacket v0.1

TaskPacket — JSON/YAML-структурированный документ, который Координатор формирует однократно и передаёт Директору **дословно** (principle: Координатор-транспорт, v1.6). Директор разворачивает его в бриф для Head'а, Head — в бриф для Worker'а, сохраняя инварианты.

### Обязательные поля

| Поле | Тип | Назначение |
|---|---|---|
| `task_id` | string | Уникальный идентификатор; формат `US-<NN>-<spring-<N>>` (пример: `US-02-sprint-1`). Нужен для связывания метрик и логов. |
| `title` | string | Одной строкой «что делаем». Максимум 100 символов. |
| `context` | string | 3–7 строк: где в M-OS эта задача живёт, какие предшествующие решения (ADR/US/fase) её мотивируют, что уже сделано до неё. Читатель — Director+Head+Worker, не Владелец. |
| `acceptance` | array of strings | Bullet-list измеримых критериев приёмки; каждый критерий — проверяемый (тест, lint-job, grep-check, ручной прогон). Копируется дословно из US. Минимум 3 пункта. |
| `constraints` | array of strings | Жёсткие ограничения: ADR/статьи, которым нельзя противоречить; антипаттерны из CLAUDE.md применимые к задаче; запреты (например, «не трогать US-01 модели — она в работе у backend-dev-1»). |
| `budget` | object | `{max_days: int, max_loc: int | null, max_new_files: int | null}`. Для пилота US-02: `{max_days: 3, max_loc: 400, max_new_files: 5}`. |
| `escalation` | object | `{when: string, to: string}`. Для US-02: `{when: «contract с US-01 меняется или acceptance не покрывает реальный сценарий клиента», to: «backend-director через Координатора-транспорт»}`. |
| `output_path` | array of strings | Явные пути файлов, где ожидается результат. Для US-02: `backend/app/core/security.py`, `backend/app/middleware/company_context.py`, `backend/tests/test_company_context.py`. Worker не создаёт файлы вне списка без явного апдейта TaskPacket. |

### Опциональные поля

| Поле | Тип | Когда применимо |
|---|---|---|
| `depends_on` | array of task_id | Если задача блокируется другими US. Для US-02: `["US-01-sprint-1"]`. |
| `references` | array of paths/URLs | ADR, регламенты, docs, которые Worker обязан прочитать (не «может», а «обязан»). Для US-02: `docs/adr/0011-foundation.md`, `docs/agents/departments/backend.md`. |
| `acceptance_test_command` | string | Точная команда CI/pytest для проверки acceptance. Для US-02: `pytest backend/tests/test_company_context.py -v`. |
| `rollback_plan` | string | Как откатить, если пилот выявит regressions. Для US-02: `git revert <hash>` + откат миграции до US-01 не требуется, т.к. US-02 без миграций. |
| `forbidden_paths` | array of strings | Файлы/директории, в которые Worker не должен вносить изменения. Для US-02: `backend/app/models/*` (модели — зона US-01), `backend/app/core/events/*` (Event Bus — зона US-04/US-05). |

### Формат хранения

- TaskPacket живёт как `docs/research/pilots/taskpackets/taskpacket-US-02.yml` (только на период пилота).
- После adopt → формат переезжает в `docs/agents/task-packet-template.md` + инструкция в `governance.md`.
- После reject → TaskPacket удаляется, инцидент фиксируется в `docs/research/findings.md`.

## 4. План эксперимента: контрольная группа vs пилотная

### Идеальный A/B — невозможен в рамках одной M-задачи

A/B требует двух идентичных M-задач, выполняемых одновременно одним и тем же Worker'ом, одна через TaskPacket, вторая без. У нас:
- US-02 — единственная задача с такими характеристиками в Sprint 1.
- Параллельная запуск US-02 и US-03 даст сравнение, но скоуп разный (3 дня vs 5 дней, разные acceptance) — не научный эксперимент.

### Выбранная схема: **delta против исторического baseline**

Пилот идёт **только через TaskPacket** на US-02. Сравнение — с baseline-метриками предыдущих задач сравнимого размера и характера (M-размер, один исполнитель, API-контракт):
- **Baseline**: US из Phase 3 Batch A Round 1 (например, `POST /api/v1/projects` с аналогичной структурой — валидация, audit, company-scope). Метрики извлекаются из логов Координатора и commit-истории.
- **Пилот**: US-02 с TaskPacket.

### Метрики baseline — как собрать (≤ 30 минут работы, одна сессия)

1. Координатор выгружает из своей истории 3 M-задачи Phase 3 Batch A за последние 2 недели.
2. Считает по каждой:
   - **Время цикла** (from «задача поставлена backend-director» до «PR замержен»): в часах.
   - **Clarification loops** (число сообщений «уточни X» от Директора/Head'а обратно Координатору): count.
   - **Брак на выходе** (дефекты, найденные review-head или reviewer'ом ПОСЛЕ первого commit PR): count (P0/P1/P2/P3).
   - **Ручные правки после сдачи** (fix-commits в течение 3 дней после merge): count.
3. Среднее по 3 задачам — baseline. Заносится в `docs/research/pilots/taskpacket-phase-1b-baseline.md`.

### Если A/B станет возможным — Phase II

В Phase II (после adopt Phase I-b), когда параллельно идут US-03 и US-04 (разные Worker'ы, сравнимый размер), можно сделать полноценный A/B: US-03 — без TaskPacket, US-04 — с TaskPacket. Это уже вне скоупа Phase I-b.

## 5. Метрики пилота

| Метрика | Baseline (ожидаемое) | Пилот (цель) | Измерение |
|---|---|---|---|
| **Время цикла** (часов, от постановки backend-director до merge) | ~20–30 ч (2–3 дня при 8-часовом фокусе) | ≤ baseline × 0.8 (экономия 20%) | Timestamp первого Agent-вызова Координатора → timestamp merge commit |
| **Clarification loops** (раундов уточнений между уровнями) | 2–4 | ≤ 1 | Счётчик сообщений с вопросами «уточни/что имеется в виду/какой формат» в любую сторону |
| **Брак на выходе** (P0+P1 дефектов от review-head на первом PR) | 1–2 | 0 | Читается из review-отчёта |
| **Ручные правки после сдачи** (fix-commits в 3-дневном окне после merge) | 1–2 fix-коммита | 0 | `git log --grep="fix\|hotfix" --since=...` |
| **Время на подготовку TaskPacket Координатором** | N/A | ≤ 30 минут (одноразовые инвестиции) | Timestamp начала составления → timestamp передачи Директору |

### Критерии adopt / reject / iterate

- **Adopt**: все 4 основные метрики улучшились или не ухудшились, при этом хотя бы 2 из 4 улучшились на ≥ 20%. Время подготовки TaskPacket ≤ 30 минут.
- **Iterate**: одна метрика регрессировала (например, clarification loops выросли из-за того, что поле `constraints` оказалось избыточным) → версия v0.2 с правкой и повтор на следующей M-задаче.
- **Reject**: две и более метрик регрессировали ИЛИ подготовка TaskPacket заняла > 1 часа ИЛИ Worker/Head сообщили, что формат избыточен и мешает работе. Инцидент в `findings.md`, формат откатывается.

6. [ ] backend-director разворачивает в бриф для backend-head, сохраняя поля TaskPacket.
7. [ ] backend-head → backend-dev-2 → PR → review-head → merge.
8. [ ] Координатор фиксирует метрики по факту, сравнивает с baseline.
9. [ ] ri-director составляет отчёт пилота: adopt / iterate / reject + причина.
10. [ ] Если adopt — формат переезжает в `docs/agents/task-packet-template.md`, governance-director оформляет ADR-0022 «TaskPacket v1.0» (опционально, может быть article в CODE_OF_LAWS).

## 9. Amendment 2026-04-19 — перенос пилота с US-02 на US-04/US-05

### 9.1 Факт
US-02 закрыто 2026-04-18 **без применения TaskPacket v0.1 при спавне** Координатора-транспорта. Пилот как он описан в §4-6 не произошёл: TaskPacket не писался (§6 п. 4), передача Директору шла в свободной форме (§6 п. 5), метрики «время подготовки TaskPacket», «clarification loops под TaskPacket» не фиксировались.

### 9.2 Рассмотренные варианты

**Вариант A — Post-hoc анализ завершённой US-02.**
Пройтись по истории Agent-вызовов US-02, ретроспективно сконструировать TaskPacket из того, что было сказано в брифах, и сравнить с baseline.

*Отклонено.* Причины:
1. **Ключевые метрики нельзя реконструировать.** «Время на подготовку TaskPacket» требует prospective-замера (таймер при написании); «clarification loops именно из-за TaskPacket» — counter-factual (а было бы их больше/меньше без формата?) — не измеряется постфактум.
2. **Hawthorne effect невозможно учесть.** Пилот с TaskPacket vs. обычный процесс на одной и той же задаче — несравнимо, потому что исполнитель уже прошёл через обычный процесс, знает ответы, clarification loops были бы искусственно снижены.
3. **Ценность post-hoc минимальна** — получаем лишь «ретроспективно TaskPacket выглядит так», а не ответ на вопрос пилота «сокращает ли формат циклы уточнений».
4. **Риск подмены вывода.** Post-hoc reconstruction легко «подогнать» под благоприятную историю, что обесценит решение adopt/reject на следующей итерации.

**Вариант B — Перенос пилота на US-04 или US-05 Sprint 2.**
Дождаться следующей M-задачи с похожими характеристиками (M-размер, один исполнитель, clear acceptance), применить TaskPacket v0.1 уже при спавне.

*Принято.* Причины:
1. Сохраняется **prospective-замер** всех метрик (включая время подготовки и clarification loops).
2. US-04 (2 таблицы + Pydantic discriminator) и US-05 (Event Bus wiring) уже в декомпозиции Sprint 2 — ждать недолго.
3. Единственный минус US-04 — два исполнителя (db-engineer + backend-dev-3). Контрмера из §7: TaskPacket идёт только backend-dev-3 на Pydantic-часть, db-engineer идёт обычным путём.
4. US-05 — одиночный исполнитель (backend-dev-1 или backend-dev-2, TBD), но скоуп L (5 дней по декомпозиции). Подходит, **если** резать скоуп до одного под-шага (runtime-guard или event-publisher, не оба).

### 9.3 Решение ri-director (фиксируется в этом RFC, не требует эскалации)

- **Посадочная площадка пилота TaskPacket v0.1:** первая M-задача Sprint 2 с одним исполнителем и clear acceptance. Приоритет: **US-04 (ограниченный — только backend-dev-3 + Pydantic discriminator)**. Запасной вариант: **US-05, если US-04 стартует раньше, чем backend-dev-3 доступен**, с обязательным резанием L → M (один под-шаг из 5).
- **Ограничение:** пилот не запускается на P0-задачах (высокий риск блокировки критического пути). US-04 и US-05 пока не в критическом пути Sprint 2 — это приемлемо.
- **Baseline из §4** (3 M-задачи Phase 3 Batch A) **остаётся валиден** — его сбор Координатором не привязан к US-02 и может быть выполнен до старта пилота.
- **US-02 baseline приращение.** US-02, закрытая без TaskPacket, **добавляется в baseline как 4-я точка данных** — это усиливает сравнение (сравнимый размер, один исполнитель, JWT/API контракт). Координатор извлекает по US-02 те же 4 метрики, что и по остальным baseline-задачам.
- **Срок переноса:** пилот стартует в первую неделю Sprint 2. Если Sprint 2 не запускается в течение 10 рабочих дней — ri-director эскалирует Координатору, возможна постановка искусственной контролируемой задачи.

### 9.4 Что обновить в плане

- §2 «Рекомендация какую M-задачу брать» — US-02 исключён из кандидатов (закрыт без пилота), US-04/US-05 Sprint 2 — новый top-2.
- §6 «Процесс» п. 4 — `taskpacket-US-02.yml` заменён на `taskpacket-US-04.yml` или `taskpacket-US-05.yml`.
- §4 «Baseline» — US-02 добавлена в baseline-выборку (4 задачи вместо 3), baseline-файл создаётся после Sprint 2 kickoff.

### 9.5 Риски переноса

| Риск | Вероятность | Контрмера |
|---|---|---|
| Sprint 2 стартует не в первую неделю (ждём закрытия Sprint 1) | Средняя | TaskPacket v0.1 может устареть к моменту старта, но пересмотра формата пока не требуется — документ стабилен |
| US-04 и US-05 тоже пройдут без TaskPacket (повторение инцидента US-02) | Низкая-средняя | ri-director **явно** включает TaskPacket в pre-Sprint-2 check-list, эскалирует Координатору напоминание при kickoff |
| Sprint 2 даст только XL/L задачи, M-кандидатов нет | Низкая | Возврат к US-06 с резанием скоупа до M (один runtime-guard шаг) — уже рассмотрено в §2 |
