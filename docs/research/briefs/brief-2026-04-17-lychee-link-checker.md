# Brief: lychee — автоматическая проверка ссылок в документации

- **Ссылка:** https://github.com/lycheeverse/lychee
- **GitHub Action:** https://github.com/lycheeverse/lychee-action
- **Лицензия:** Apache-2.0 / MIT
- **Дата обнаружения:** 2026-04-17
- **Источник:** WebSearch по R-04 (Governance-отчёт 2026-04-17)
- **Теги:** docs-validation, tooling, R-04
- **Запрос, на который отвечает:** R-04 «инструменты валидации регламентных документов» — часть (а) «битые ссылки»

## Что это делает (простым языком)

Маленькая программа, которая обходит все markdown-файлы в `docs/` и проверяет каждую ссылку: на внешний сайт — что страница отвечает (HTTP 200), на соседний файл — что файл реально существует. На выходе — список «сломанных» ссылок. Запускается одной командой: `lychee docs/`.

Аналогия: как проверяющий секретарь, который раз в неделю проходит по всему делу и отмечает красным: «сноска на приказ №145 — такого приказа в папке нет», «ссылка на сайт партнёра — сайт не отвечает». Быстрый (576 ссылок за минуту), кеширует результаты, умеет игнорировать намеренные пустые ссылки через файл `.lycheeignore`.

## Чем это полезно coordinata56

У нас **больше 120 ссылок только между регламентными документами** (CONSTITUTION ↔ PROCEDURAL_CODE ↔ CODE_OF_LAWS ↔ регламенты департаментов ↔ ADR). При каждой реформе (SUPERSEDED-файлы, переименования pod) ссылки разъезжаются. Проблемы, зафиксированные Governance-отчётом:
- Д-05 «head.md и worker.md ссылаются на устаревшие addendum-версии» — нашли вручную, а lychee находит за секунды.
- У-06 «director.md не обновлялся — возможно висят мёртвые ссылки».
- Аудит W5 потратил 40% времени на ручной обход (П-05).

Если сделать это автоматически — governance-auditor тратит время на смысловые противоречия, а не на механику.

## Что предлагаем внедрить

**Хирургическая правка** (принцип Владельца msg 817):
1. Analyst (1 час) устанавливает lychee локально (`cargo install lychee` или готовый бинарник), прогоняет по `docs/`, выводит список битых ссылок на сегодня.
2. Если список короткий (<20 нарушений) — governance-auditor в рамках очередного еженедельного аудита закрывает их.
3. Предложение governance-director'у: добавить в регламент правило «перед закрытием аудита W-N запустить `lychee docs/`, список битых ссылок приложить к отчёту».
4. Позже, когда появится CI (Фаза infra): подключить как GitHub Action на каждый PR.

**Без Docker, без инфраструктуры — одна бинарная команда.** Идеально ложится на ограничение «решение должно работать без CI/CD на старте» из R-04.

**Кто участвует:** ri-analyst (прогон) → governance-director (правка регламента).
**Трудозатраты:** первый прогон 1 час, регулярный прогон <1 минуты.

## Риски и ограничения

- Внешние ссылки (на GitHub, Anthropic docs) иногда дают rate-limit — решается параметром `--github-token`.
- Локальный запуск — нужен Rust toolchain или бинарник, без Claude-специфики.
- Никаких внешних SaaS, никакой отправки данных наружу (кроме HEAD-запросов на публичные URL).

## Вердикт

**Adopt (light)** — пилотный прогон сразу, по результатам — включение в регламент аудита. Высокая уверенность в пользе при минимальной трудозатрате.
