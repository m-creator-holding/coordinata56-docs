## Почему этот скил сейчас

CLAUDE.md строки 67-70 — 3 правила git-гигиены:
- `git add -A` запрещён без предварительного `git status --short`.
- При активных фоновых Agent-вызовах — **ни в коем случае** `git add -A` (чужие файлы попадут в коммит без ревью).
- Commit-message: «почему», не «что».

Правило №2 ловилось **один раз** (2026-04-15, qa Батча A попал в коммит регламента v1.4 без ревью — цитата в CLAUDE.md). Инцидент критичный: reviewer не пропустил работу, а Координатор её закоммитил параллельно, пока qa ещё редактировал. Риск только растёт: Владелец просил 5-15 параллельных Agent-вызовов (msg 1433, `proactive_agent_launches`) + auto-push на GitHub после каждого коммита (msg 1325) — окно ошибки сузилось до нуля.

Keyword-триггеры чёткие (`git add`, `git commit`), применимость ≈10-15% сессий (только когда коммитим) — классический профиль lazy-load skill.

## Что скил должен делать (scope)

1. **Триггер.** user-invocable=false, auto-invoke при любом Bash-вызове содержащем `git add`, `git commit`, `git stash push -u`.
2. **Вход.** Команда, которую Claude Code собирается выполнить.
3. **Шаги SKILL.md.**
   - **Шаг 1: проверка `git add -A` / `git add .` / `git commit -a`.** Если команда содержит любой из этих флагов — **обязательно** сначала выполнить `git status --short` и явно показать список файлов Координатору. FAIL (не запускать команду), предложить конкретный список через `git add path1 path2`.
   - **Шаг 2: детектор параллельных Agent-сессий.** Если в текущей сессии есть активные `run_in_background: true` Agent-вызовы (не завершённые task-notification'ом) — запрет на любой `git add` кроме явно перечисленных файлов. Цитата: CLAUDE.md msg от 2026-04-15.
   - **Шаг 3: проверка commit-message.** Если `-m "..."` содержит паттерн `^(Add|Update|Fix|Adds|Updates|Fixes|Removes?)\s+\w+$` (2-3 слова без «because/так как/чтобы») → WARN «commit-message описывает ЧТО, а не ПОЧЕМУ», предложить шаблон. Не блокировать, но предупредить.
   - **Шаг 4: проверка на секреты в staged diff.** Перед `git commit` — `git diff --staged` grep на типовые паттерны: `.env`, `credentials`, `-----BEGIN`, `eyJhbGci`. Совпадение → P0 FAIL.
   - **Шаг 5: проверка auto-push target.** Если следующая команда после `git commit` — `git push --force` на `main`/`master` → P0 FAIL, требовать подтверждения Владельца (регламент `/root/.claude/CLAUDE.md` §«Безопасность действий»).
   - **Шаг 6: вывод.** ALLOW (команду можно запускать) / WARN[список] / BLOCK[причина + предлагаемая замена].
4. **Выход.** Короткий текст с решением ALLOW/WARN/BLOCK и, при необходимости, готовой заменой команды.

## Что скил НЕ делает

- Не управляет git-процессом целиком (это делает Координатор).
- Не дублирует pre-commit hooks (они работают уже после `git add`).
- Не проверяет сами изменения на качество кода (это review-head).
- Не принимает решений за Координатора — только сигналит и предлагает.
