## Что это делает (простым языком)

Две утилиты-«корректора», которые проверяют структуру markdown-файлов:

- **markdownlint** — следит за правилами оформления: заголовки по порядку (h1 → h2 → h3), нет дублирующихся разделов, единый стиль списков, длина строки. Пример из Governance-отчёта: Д-01 («в research.md раздел Weekly digest встречается дважды») — такое правило ловится одним чеком MD024 «no duplicate headings».
- **remark-lint-frontmatter-schema** — каждый регламент начинается с блока YAML-метаданных (`---` ... `---`), где прописаны «версия», «дата», «статус», «утверждено кем». Утилита проверяет, что блок есть, все обязательные поля заполнены, версия в правильном формате (1.0, 1.1, а не «1»), статус — один из списка (Действующий/SUPERSEDED/Черновик).

Аналогия: как канцелярский регламент — у каждого документа должна быть шапка с номером, датой, грифом. Человек забудет, утилита напомнит.

## Чем это полезно coordinata56

Governance-отчёт показал систематические проблемы:
- Д-03 «приоритет коллизий описан в трёх местах с разными формулировками» — ловится правилом «уникальные якоря + cross-reference check».
- П-04 «нет единого шаблона ADR, 14 ADR разной структуры» — решается одной JSON-схемой для ADR-frontmatter (номер, статус, дата, связи).
- У-02 «дубли в docs/agents/subagents/ и ~/.claude/agents/» — ловится через схему, указывающую canonical location.

Реальный эффект: на следующем аудите governance-auditor сразу видит, какие 5 документов не прошли lint, вместо ручного обхода.

## Что предлагаем внедрить

**Минимальная интеграция** (принцип «хирургические правки»):

1. **Analyst (2 часа):** предлагает минимальный ruleset markdownlint (10–15 правил из ~60), подходящий под наш стиль — без перфекционизма. Пример: MD001 (heading increment), MD024 (no duplicate headings), MD025 (single H1), MD040 (fenced code language).
2. **Analyst (ещё 1 час):** черновик JSON-схемы frontmatter для трёх типов документов: ADR, регламент департамента, заявка в Governance. Обязательные поля: `version`, `date`, `status`, `approved_by`.
3. **governance-director** получает заявку на включение двух новых пунктов в регламент Governance: «перед закрытием заявки — прогнать markdownlint и frontmatter-schema на затронутых файлах».
4. Позже (при CI) — подключить как GitHub Action.

**Кто участвует:** ri-analyst → governance-director.
**Трудозатраты:** 3 часа Analyst'а + обычная заявка в Governance.

## Риски и ограничения

- Требуется Node.js на машине (для markdownlint-cli2) — у нас уже есть.
- 14 действующих ADR могут «посыпаться» по схеме — это ожидаемо и полезно: повод привести их к единой форме (П-04).
- Внедрять **постепенно**, по частям (сначала лёгкие правила оформления, потом frontmatter-схема), чтобы не заблокировать текущую работу.

## Вердикт

**Trial** — Analyst пишет мини-ruleset + схему, прогоняет на текущем состоянии, список нарушений передаёт governance-director. Adopt после успешного прогона и решения Governance о включении в регламент.
