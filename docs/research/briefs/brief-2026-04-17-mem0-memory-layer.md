# Brief: Mem0 — универсальный слой памяти для агентов

- **Ссылки:** https://github.com/mem0ai/mem0 ; научная статья https://arxiv.org/abs/2504.19413
- **Лицензия:** Apache-2.0 (open-source версия), есть платный managed Mem0 Cloud
- **Дата обнаружения:** 2026-04-17
- **Источник:** WebSearch по R-01 (Governance-отчёт 2026-04-17)
- **Теги:** ai-agents, memory, R-01
- **Запрос, на который отвечает:** R-01 «memory management в multi-agent системах»

## Что это делает (простым языком)

Mem0 — это отдельный небольшой сервис-«кладовка», куда агенты складывают важные факты из диалогов: «Владельца зовут Мартин», «Phase 3 закрыта 2026-04-16», «правило git add -A под запретом». При следующем обращении агент не перечитывает всю историю переписки (дорого и долго), а спрашивает Mem0: «что ты помнишь про X?» — и получает только релевантные факты. Внутри Mem0 три хранилища: поиск по смыслу (векторный), связи между фактами (граф) и быстрые ключ-значения. Аналогия: вместо «перечитать всю папку дела перед каждым звонком клиенту» — картотека, где находишь одну карточку за секунду.

Две версии: open-source (ставится локально, в том числе с встроенной графовой БД Kuzu без отдельного сервера) и облачный сервис Mem0 Cloud (отдельная подписка).

## Чем это полезно coordinata56

У нас сейчас память живёт в одном файле `MEMORY.md` + папке `memory/` под Координатора. Работает, но:
- Прямое отображение: чем больше проект, тем больше файл, тем больше контекста тратится на «прочитать всё».
- Нет разграничения «свежее/архив»: последние правила Владельца и уроки Phase 0 лежат рядом с равным весом.
- Нет разграничения памяти между Координатором и Директорами — если Директору нужно «вспомнить решение по миграциям», он перечитывает общие файлы.

Mem0 решает это: каждый агент получает свой namespace, поиск по смыслу находит «что знаем по теме», общие факты проекта хранятся отдельно. Для нашей ситуации (39 агентов, растущий проект) — естественное следующее состояние.

## Что предлагаем внедрить

**Масштаб — минимальный**, по принципу «сначала простота» (Владелец msg 817):
1. Analyst готовит RFC (бюджет 4 часа): читает документацию Mem0 open-source, сравнивает с нашей текущей схемой `MEMORY.md` + `memory/`, пишет раздел «дельта и риски». Ключевой вопрос RFC — можем ли мы поставить **локальный Mem0 с встроенным Kuzu** на тот же сервер, без отдельного графового сервиса.
2. Если RFC положительный — пилот на одной роли (например, memory-keeper получает Mem0 как дополнительный инструмент, старый `MEMORY.md` остаётся). Срок пилота — 2 дня. Метрика: сколько «вспоминаний» проходит через Mem0 vs через файл, насколько меньше контекста тратится.
3. Adopt только если пилот реально экономит контекст и не ломает правило «Координатор-транспорт».

**Кто участвует:** ri-analyst (RFC) → Координатор распределяет пилот (вероятно backend-dev).
**Трудозатраты:** RFC 4 часа, пилот 2 дня.

## Риски и ограничения

- Mem0 **Cloud** — платный сторонний SaaS, требует отправки контекста во внешний сервис. Согласно CODE_OF_LAWS v2.1 ст. 45а («no live external integrations») и ст. 45б — предложение подключить Mem0 Cloud потребует отдельного решения Владельца. **В пилоте — только self-hosted вариант.**
- Mem0 open-source активно эволюционирует, API может меняться (версия 0.x).
- Миграция данных из `MEMORY.md` в Mem0 — одноразовая, не автоматическая.

## Вердикт

**Trial** — отправить Analyst'у на RFC. Полный adopt — только после успешного пилота и явного отказа от облачной версии до отдельного решения Владельца.
