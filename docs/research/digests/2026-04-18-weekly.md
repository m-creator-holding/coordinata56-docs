## §1. Три находки недели (top-3 sensing, отдельные от RFC-003/004/005)

Эти три находки **не покрыты** уже принятыми RFC (003 Mem0, 005 cross-audit) и не являются кирпичами RFC-004. Отобраны по критерию «применимость сейчас × риск пропустить».

### Находка 1 — Claude Code Hooks (официальная документация + paттерн mastery)

- **Источники:** `code.claude.com/docs/en/hooks-guide` + `github.com/disler/claude-code-hooks-mastery` (findings.md строки 34–35)
- **Теги:** claude-code, hooks, workflow-automation, R&I-transport
- **Текущий статус:** Analysis (разбирается в рамках RFC-004 v1.0 как Вариант B')

**Что это делает.** В Claude Code есть встроенный механизм «хуков» — 21 событие жизненного цикла (PreToolUse, PostToolUse, SubagentStop, UserPromptSubmit и т.д.), к каждому можно прицепить обработчик: shell-команду, промпт или вызов другого субагента. Ближайшая аналогия — сигнализация на стройке: сработал датчик открытия ворот → автоматически включилась камера + отправилось сообщение охране. В нашем случае: Worker закончил работу (событие `SubagentStop`) → автоматически запускается Reviewer на `git diff --staged`, без того чтобы Координатор помнил эту стадию маршрута.

**Чем это полезно coordinata56.** Прямой путь закрыть три из четырёх механизмов процессных ошибок, описанных в RFC-004 §1.2 (Transport drift, Skip-step, Cross-route contamination), **без строительства Orchestration Layer по ADR 0012**. То есть вместо 4–6 недель разработки task_router получаем 80 % эффекта за 2–3 дня конфигурации хуков. Кейс «git add -A с чужой работой 15 апреля» и «ri-director подменил ri-scout 16 апреля» — оба механически не возникли бы, если бы PreToolUse-хук блокировал `git commit` без предшествующего запуска reviewer.

**Что предлагаем.** Пилот на 2 дня: настроить 3 ключевых хука (SubagentStop → auto-reviewer; PreToolUse на `git commit` → проверка маркера «reviewer passed»; UserPromptSubmit → логирование в дашборд). Прогнать на одной реальной M-задаче M-OS-1. Если работает — фиксируем как Вариант B' в финальной версии RFC-004 и делаем акселератор к полному ADR 0012. Участники: ri-analyst (конфигурация хуков), один backend-dev (тестовый прогон). **Риск:** хуки — это локальная конфигурация, не self-hosted сервис; миграция на облако (Managed Agents) в будущем может потребовать переписать, но это далёкий горизонт.

### Находка 2 — Microsoft Agent Governance Toolkit (апрель 2026, MIT)

- **Источник:** `github.com/microsoft/agent-governance-toolkit` (findings.md строка 20, брошена в Analysis через brief 2026-04-17)
- **Теги:** governance, agents, policy, R-02
- **Текущий статус:** Discovered → рекомендую двинуть в Analysis (RFC-2026-009?)

**Что это делает.** Microsoft в апреле 2026 выпустил открытый «конституционный движок» для ИИ-агентов: YAML-правила + OPA/Rego/Cedar как выражения политик, интеграция с LangChain/CrewAI/AutoGen. Агент перед выполнением действия прогоняет намерение через политику: «можно ли этому агенту трогать файлы из такой-то папки?», «разрешено ли ему писать в БД?», «нужно ли одобрение L2?». Ответ — allow/deny/escalate. По сути это то, что мы пишем текстом в CODE_OF_LAWS v2.0, только runtime-проверяемое.

**Чем это полезно coordinata56.** У нас уже есть 47-статейный CODE_OF_LAWS и 96-статейная Конституция M-OS — но они существуют как регламент, который читает LLM. Нет технической защиты: Директор *может* нарушить статью 15 «не принимать задачу на исполнение», и это ловится только на ревью post-factum. MS Toolkit даёт путь перевести ключевые запреты (no live external integrations ст. 45а, scope-ограничения полномочий L1/L2/L3, инцидент с git add -A) в машинно-проверяемый слой. Для M-OS-2+ при 30+ субагентах это станет необходимым, не «приятным».

**Что предлагаем.** Analyst-анализ на 4 часа: разобрать, что из Toolkit мы можем взять как идею (YAML-схема политик), не утаскивая сам движок — у нас на сервере Python, не .NET. Результат — RFC-2026-009 с предложением частично перенести CODE_OF_LAWS в YAML-конфиг и подключить валидатор при `create_task()` в Orchestration Layer. Внедрение — синхронно с ADR 0012 Phase 1. Риск: оверинжиниринг — не тащить весь Toolkit, взять паттерн.

### Находка 3 — gitagent (git-native стандарт описания агентов)

- **Источник:** `github.com/open-gitagent/gitagent` (findings.md строка 21)
- **Теги:** governance, agents, portability, R-02
- **Текущий статус:** Discovered

**Что это делает.** Новый open-source стандарт (апрель 2026) для описания ИИ-агентов через версионируемые YAML/Markdown-файлы: system prompt, роль, набор инструментов, жёсткие ограничения (hard constraints), модель. Вся «личность» агента живёт в `git`, меняется через pull request, откатывается как обычный код. По сути — то, что мы делаем в `~/.claude/agents/*.md`, только формализованное как стандарт с валидацией схемы.

**Чем это полезно coordinata56.** Сейчас у нас 39 активных субагентов в `~/.claude/agents/`, каждый — свободной формы markdown с YAML-frontmatter. Расхождения в структуре уже есть (фиксируется в Governance-отчёте 2026-04-17: П-04 «нет единого шаблона»). gitagent даёт готовую схему валидации, совместимую с Claude Agent SDK. Плюс — переносимость: если через год захотим часть агентов запускать на другой платформе, уже сэкономили на описании.

**Что предлагаем.** Scout-сессия на 1 час: проверить, совместим ли формат gitagent с текущими файлами `~/.claude/agents/`, и YAML-схему которую можно сразу подключить к remark-lint-frontmatter-schema (уже в очереди на пилот). Если совместимо — Analyst-проход на 4 часа с миграцией 2–3 агентов в эталон и ADR-поправкой к §10. Риск: стандарт молодой (апрель 2026), может не устояться — держим как референс, не как обязательство.

### Открытый вопрос Координатору

**Что именно подразумевается под «Department Automation & Acceleration»?** В сохранённых сообщениях и findings.md скоуп задачи не описан. Три возможных интерпретации:

- (а) **Автоматизация повторяющихся рутин внутри отделов** — например, weekly digest от каждого Директора, автопроверки регламентов, авто-SPR (scope-permission-review) на каждый бриф. Родственно RFC-001 Routines.
- (б) **Акселераторы для Директоров** — готовые скилы/шаблоны (как superpowers, awesome-claude-code-subagents), ускоряющие типовой цикл Директора на 30–50 %.
- (в) **Параллелизация работы отделов** — минимизация blocking-зависимостей между backend/frontend/infra/quality/design, чтобы отделы могли работать одновременно.

**Нужен бриф от Координатора с уточнением, чтобы Scout не разведывал «всё и сразу» и не потратил недельный бюджет.** Без этого ri-scout пойдёт по умолчанию (а), а Координатор, возможно, имел в виду (в).
