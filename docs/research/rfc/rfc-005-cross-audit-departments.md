# RFC-2026-005: Сквозной аудит 8 департаментов coordinata56

> **Честная оговорка с первой строки.** RFC — это **предложение**, не решение. 26 находок Scout'а отобраны из индустриальных источников за апрель 2026; не все одинаково применимы к нашему масштабу (85 домов, 1 пилотный объект, команда — Координатор + субагенты). Для каждой находки ниже честно указано: применимо сейчас / отложено до M-OS-2 / отклонено. Право принять или отклонить — за Владельцем.

## 0. Расшифровка простым языком для Владельца (обязательный раздел)

### Что делаем

Представьте холдинг, где восемь отделов: бухгалтерия, юристы, служба безопасности, прорабы, дизайнеры, техотдел, архив, и отдел разведки. Раз в квартал Вы проводите **внутренний аудит** — смотрите, где у каждого отдела регламент отстаёт от того, что делают конкуренты в индустрии. Не для того, чтобы всё поменять, а чтобы увидеть: *«вот у нас ещё нет видеокамер на стройке, а у соседей уже три года стоят — надо бы тоже, стоит 2 недели работы одного специалиста»*.

Этот RFC — такой аудит, только для восьми **программных** отделов coordinata56. Scout-разведчик за неделю прочитал 26 внешних статей и выделил улучшения, которые индустрия считает стандартом 2026 года. Я, Analyst, проверил каждую: есть ли у нас уже похожее, насколько это нам нужно сейчас, сколько стоит внедрение. Итог — три рекомендации на каждый отдел по трём критериям (меньше рисков / лучше работа / выше эффективность), плюс сводный Top-10 приоритетов и план первых пяти шагов.

### Зачем это нам

Владельческий бизнес в строительстве 85 домов держится на прозрачности и контроле. coordinata56 — система, через которую Вы будете видеть бюджеты, подрядчиков, статусы. Если фундамент этой системы (инфраструктура, наблюдаемость, безопасность) не соответствует отраслевому минимуму — **риски не видимы Вам до того, как произойдёт инцидент**. Типичный сценарий без этого аудита: через 3 месяца работы production подрядчик получит лишний доступ из-за несработавшего RBAC, мы узнаем об этом через жалобу, разбор займёт неделю без логов.

Аудит закрывает именно эту слепую зону. **Самый критичный разрыв** — отдел инфраструктуры: у нас нет автоматического развёртывания (CI/CD) и мониторинга боевых сбоев; регламент этого отдела — скелет версии 0.1 из одной страницы. Следующий — отдел качества: есть тесты функциональности, нет тестов безопасности и отдельного раздела про тестирование ИИ-субагентов, хотя M-OS — это в первую очередь система субагентов.

### Что именно предлагаем внедрить

Пять первых шагов (детали — §5 ниже), каждый оценён в часах/днях и привязан к M-OS-1:

1. **Наблюдаемость бэкенда** (OpenTelemetry + correlation ID + structured logging) — 2 дня backend-dev, сразу после Волны 1 Foundation.
2. **Контракт миграций Expand-Migrate-Contract** — уже идёт как ADR 0013 (proposed), только дожать до approved + авто-линтер в CI.
3. **Автоматическая проверка безопасности в CI** (Bandit SAST + pip-audit для зависимостей) — 1 день qa.
4. **Регламент отдела инфраструктуры v1.0** вместо скелета — 2 дня infra-director (как только он будет активирован).
5. **OWASP AI Testing Guide** адаптация к субагентам M-OS — 3 дня quality-director + ri-analyst.

Общая оценка первых пяти: ~10 рабочих дней в течение 3–4 недель, не блокирует M-OS-1. Десять следующих пунктов — в §4 (Top-10).

Чего **не предлагаем сейчас**: GitOps (ArgoCD) — overkill для одного production-объекта; Storybook — полезен, но отложен до Фазы 4 «Админка M-OS» вместе с активацией design-director; Diátaxis 4-типа-документов — полезен, но требует не менее недели tech-writer и не закрывает прямой риск.

### Шаблон анализа (единый для всех разделов)

Для каждого департамента:
1. **Текущее состояние (as-is)** — что реально есть: регламент, код, инструменты.
2. **Риски и разрывы (gap)** — чего не хватает относительно отраслевого минимума 2026.
3. **Три рекомендации** по критериям Владельца: 1× минимизация рисков, 1× улучшение работы, 1× эффективность.
4. **Impact / Effort / Приоритет (P0–P2)** для каждой рекомендации.

### Фильтрация

Не все 26 находок одинаково применимы. Часть отклонена/отложена с обоснованием. Критерии фильтрации:
- **overkill для масштаба 85 домов / 1 пилота** → отложить до M-OS-2 или отклонить.
- **конфликтует с уже утверждённым ADR** → отклонить, если ADR не планируется amendments.
- **дублирует уже имеющийся скил/агент** → отклонить как дубль.

## §3. По департаментам

### §3.1 Backend

**Директор:** backend-director. **Регламент:** `departments/backend.md` v1.0 (утверждён). **Зрелость:** высокая — 14 сервисов, 7 API-модулей, 14 моделей SQLAlchemy, тесты, ADR 0004–0007 + 0013.

#### Текущее состояние (as-is)
- Стек: Python 3.12, FastAPI, SQLAlchemy 2.0, Alembic, Pydantic v2.
- Миграции: правило round-trip в CLAUDE.md, ADR 0013 (Expand-Migrate-Contract) — proposed, ожидает governance.
- Логирование: нет structured logging, нет correlation ID, нет OpenTelemetry.
- Feature flags: отсутствуют — каждое изменение требует релиза.
- ADR/RFC: смешаны в `docs/adr/` (0013 — это скорее RFC, а не решение; `_review_2026-04-14.md` показывает, что путаница уже есть).

#### Находки Scout
- **B-1** OpenTelemetry + structured logging + correlation ID (fastlaunchapi.dev, 2026).
- **B-2** Expand-Migrate-Contract миграции (dev.to/young_gao, 2026) — дублирует ADR 0013, нужен авто-линтер.
- **B-3** Feature flags с 30-дневным lifecycle (designrevision.com).
- **B-4** RFC + ADR разделение (itnext.io).

#### Три рекомендации

**1. [Риск] B-1 — Observability FastAPI (OpenTelemetry + correlation ID + structlog).**

Проблема: при production-инциденте в M-OS-1 без correlation ID невозможно связать запрос клиента, вызов БД, запись аудита. Отладка по `docker logs` — часы.
Решение: установить `opentelemetry-instrumentation-fastapi`, `structlog`, middleware с UUID correlation-id, OTLP-экспорт в локальный Jaeger (в `docker-compose.dev`).
Что у нас уже есть: ничего — `backend/app/main.py` не имеет ни одного из этих инструментов.
Impact: **high** — закрывает «чёрный ящик» production. Effort: **~2 дня** (backend-dev + 0.5 дня devops для compose). Приоритет: **P0**.

**2. [Работа] B-2 — ADR 0013 approved + CI-линтер миграций.**

Проблема: ADR 0013 (proposed) описывает правила, но нет инструмента, который запретит `DROP COLUMN`/`RENAME COLUMN` в PR. Риск — именно тот, от которого ADR и защищает.
Решение: дожать ADR 0013 через governance (1-дневная заявка), добавить в CI job проверку Alembic-миграции регулярным выражением или готовым `squawk`/`atlas`-анализатором.
Что у нас уже есть: ADR 0013 в proposed, правило round-trip в CLAUDE.md.
Impact: **high** — один ошибочный PR может заблокировать production. Effort: **~1 день** governance + 0.5 дня devops. Приоритет: **P0**.

**3. [Эффективность] B-3 — Минимальные feature flags.**

Проблема: каждое спорное изменение требует полного релиза + rollback через git revert; на MVP терпимо, в M-OS-1 с 6 параллельными подами — уже нет.
Решение: простейшие boolean-флаги через таблицу `feature_flags` в БД + декоратор на роуте; без сторонних SaaS (LaunchDarkly overkill).
Что у нас уже есть: ничего. Паттерн `app_settings` из ADR 0011 близок, но для бизнес-настроек, не для feature toggles.
Impact: **med** — ускоряет выкатку рискованных фич. Effort: **~1.5 дня** (модель + сервис + декоратор). Приоритет: **P2** — отложить до первого случая, когда реально потребуется.

#### Отклонено / отложено
- **B-4** (RFC vs ADR разделение) — покрывается G-3 в разделе Governance, см. §3.6. Не дублировать.

### §3.2 Frontend

**Директор:** frontend-director (dormant). **Регламент:** `departments/frontend.md` v0.1 — скелет, активируется при старте Фазы 4. **Зрелость:** нулевая — кода фронтенда ещё нет.

#### Текущее состояние (as-is)
- Стек зафиксирован в ADR 0002: React 18 + TypeScript + Vite + shadcn/ui + Tailwind + TanStack Query + react-hook-form + Zod.
- Директория `frontend/` в репозитории отсутствует.
- Wireframes для M-OS-1 Admin UI — в `docs/pods/cottage-platform/tasks/wireframes-m-os-1-1-admin.md` (скелет fe-skeleton-m-os-1-1).
- Регламент отдела v0.1 — страница, без правил.

#### Находки Scout
- **F-1** React Error Boundaries (oneuptime.com) — резервные экраны при падении компонента.
- **F-2** Playwright performance budgets (browserstack.com).
- **F-3** React.Suspense + code splitting (esparkinfo.com) — для 7 блоков M-OS.

#### Три рекомендации

**1. [Риск] F-1 — Error Boundaries в базовом каркасе.**

Проблема: без Error Boundary любая ошибка рендеринга в одном блоке M-OS кладёт весь Admin UI. Для внутреннего ПО холдинга это блокер.
Решение: root-level `<ErrorBoundary fallback={<FallbackScreen/>}/>` + per-route boundaries, логирование в backend через API.
Что у нас уже есть: ничего. Регламент v0.1 не упоминает обработку ошибок UI.
Impact: **high** (для Фазы 4). Effort: **~0.5 дня** как часть fe-skeleton. Приоритет: **P0** — встроить в скелет с нуля.

**2. [Работа] F-3 — Code splitting с первого коммита.**

Проблема: M-OS Admin UI запланирован как 7 функциональных блоков (Vision §3). Single bundle = медленный первый экран, плохой UX руководителя.
Решение: `React.lazy()` + `Suspense` на уровне роутов, Vite split-vendor-chunks.
Что у нас уже есть: ADR 0002 фиксирует стек, но правила bundling не прописаны.
Impact: **med**. Effort: **~0.5 дня** как часть fe-skeleton. Приоритет: **P1**.

**3. [Эффективность] F-2 — Performance budget в Playwright.**

Проблема: без автоматической проверки производительности регрессии (бандл вырос, страница загружается 8 секунд) обнаруживаются пользователем, не разработчиком.
Решение: Playwright-тест с ассертами `expect(loadTime).toBeLessThan(3000)` в CI.
Что у нас уже есть: Playwright упомянут в регламенте v0.1, но не установлен.
Impact: **med**. Effort: **~1 день** когда появится фронтенд + 0.5 дня настройки в CI. Приоритет: **P2** — только после того, как будет 3+ страниц.

#### Действие по регламенту
Заявка в governance: обновить `departments/frontend.md` с v0.1 до v0.5 — добавить раздел «Правила устойчивости UI» с требованиями F-1, F-3. Делать при активации frontend-director в Фазе 4.

### §3.3 Infrastructure (наибольший gap — особое внимание)

**Директор:** infra-director (dormant частично). **Регламент:** `departments/infrastructure.md` v0.1 — 34 строки, 4 правила. **Зрелость:** низкая — Docker Compose для dev есть, CI/CD минимальный (ruff + pytest), мониторинга нет, инцидент-процедуры нет.

#### Текущее состояние (as-is)
- `docker-compose.yml` dev-режим с Postgres 16.
- `.github/workflows/ci.yml` — два job'а (lint ruff + test pytest). Нет security-сканов, нет coverage, нет Playwright, нет build-артефактов.
- Production-деплоя нет, staging нет.
- Backup процедура: отсутствует.
- Мониторинг: отсутствует (ни uptime, ни метрики, ни логи в центр).
- Incident management: не описан.
- GitOps: не применяется.

#### Находки Scout
- **I-1** GitOps — изменения инфраструктуры через Git PR + автоприменение (zenocloud.io).
- **I-2** Observability 3 столпа: metrics + logs + traces → −40% MTTR, +37% reliability (vettedoutsource.com).
- **I-3** Incident management: выделенный канал, blameless post-mortem, SLO (gomboc.ai).

#### Три рекомендации

**1. [Риск] I-2 — Минимальная observability для production (не полные 3 столпа).**

Проблема: при выходе M-OS-1 в production мы не узнаем о сбоях до жалобы Владельца. Без logs/metrics/traces невозможно провести blameless post-mortem.
Решение (MVP, не full-SRE-stack): Jaeger + Loki + Grafana в docker-compose prod (готовые контейнеры, ~2 часа настройки), экспорт из backend через OpenTelemetry (связано с B-1). Uptime monitoring — простой cron-health-check на отдельной виртуалке или UptimeRobot free tier.
Что у нас уже есть: ничего. B-1 — необходимое условие.
Impact: **high** — без этого production небезопасен. Effort: **~3 дня** devops после B-1. Приоритет: **P0**.

**2. [Работа] Регламент отдела v1.0 вместо v0.1-скелета.**

Проблема: текущий регламент v0.1 (34 строки, 4 правила) не покрывает: CI/CD стандарты, deploy-процедуру, backup-стратегию, incident runbook, SLO/SLA. Infra-director, когда активируется, начнёт с нуля — теряя опыт Фазы 0–3.
Решение: заявка governance — infra-director пишет v1.0 по образцу backend.md (120+ строк), разделы: «Сферы ответственности», «CI/CD стандарты», «Deploy-процедура», «Backup», «Incident runbook» (связано с I-3), «Метрики отдела».
Что у нас уже есть: v0.1 скелет + накопленный опыт Фазы 0 (Docker scaffold) и Фазы 1 (миграции).
Impact: **high**. Effort: **~2 дня** infra-director (сразу после активации). Приоритет: **P0**.

**3. [Эффективность] I-3 — Incident runbook (лайт-версия) + ONE-PAGE SLO.**

Проблема: если сейчас БД упадёт — нет документа «что делать первым». Разбор будет бессистемным.
Решение: 1-страничный файл `docs/ops/incident-runbook.md` с 5 сценариями (БД down / backend 500-и / disk full / auth broken / data corruption), каждый — 3–5 шагов; SLO — **одна цифра на сервис** (uptime 99% для M-OS-1 MVP — не 99.9%, реалистично).
Что у нас уже есть: ничего.
Impact: **med**. Effort: **~0.5 дня** infra-director. Приоритет: **P1**.

#### Отклонено / отложено
- **I-1 GitOps с ArgoCD/Flux** — overkill для одной production-площадки. ArgoCD оправдан при 10+ окружениях / кластере Kubernetes. У нас один хост. Отложено до M-OS-2 (когда появится multi-tenancy с физически разными инстансами).
- **Полный 3-pillar observability стек (Prometheus + VictoriaMetrics + Loki + Tempo)** — отложено. На MVP достаточно Jaeger + Loki + Grafana.

### §3.4 Design

**Директор:** design-director (dormant). **Регламент:** `departments/design.md` v0.1 — 29 строк, без правил. **Зрелость:** нулевая — нет design-системы, нет токенов, нет компонентов.

#### Текущее состояние (as-is)
- Регламент v0.1 упоминает WCAG AA, design-токены, shadcn-композиции — декларативно, без инструментов проверки.
- Stack (shadcn/ui + Tailwind) зафиксирован ADR 0002.
- Memory `feedback_ux_over_visual_design.md` — на MVP UX > визуал.
- Нет Storybook, нет axe-core, нет принятой палитры/токенов вне дефолта shadcn.

#### Находки Scout
- **D-1** Accessibility-policy: WCAG 2.2 AA в токенах + axe-core в CI (testparty.ai).
- **D-2** ROI дизайн-системы: +38% дизайнеров, +31% разработчиков, $2.70 на $1 за 5 лет (wpdean.com).
- **D-3** Storybook — каталог компонентов + a11y-тесты (uxpin.com).

#### Три рекомендации

**1. [Риск] D-1 — Axe-core в CI с первого фронтенд-PR.**

Проблема: WCAG AA декларирован, но не проверяется. Для внутреннего ПО холдинга (ADR 0008: «M-OS — внутреннее ПО»), возможно, не критично юридически, но критично для пользы: если цветовой контраст недостаточен — прораб на солнце не прочитает экран.
Решение: `@axe-core/playwright` в E2E-тестах, failover CI при violations уровня `serious`/`critical`.
Что у нас уже есть: ничего.
Impact: **med** для внутреннего использования, **high** если в будущем будет внешний доступ. Effort: **~0.5 дня** qa + frontend-dev. Приоритет: **P1**. Делать на старте Фазы 4.

**2. [Работа] D-2 — Первый пакет дизайн-токенов до фронтенд-скелета.**

Проблема: без токенов (цвета, типографика, отступы) каждый компонент получит свои магические числа, refactor будет болезненным.
Решение: минимальный `tokens.css` с 8 переменными (primary/secondary/accent/bg/fg/border/radius/shadow), привязанными к Tailwind config. Не Figma tokens, не Style Dictionary — просто CSS custom properties.
Что у нас уже есть: ничего за пределами дефолта shadcn.
Impact: **med**. Effort: **~0.5 дня** design-director. Приоритет: **P1**.

**3. [Эффективность] D-3 — Storybook — отложено до первой переиспользуемой библиотеки компонентов.**

Проблема: Storybook даёт ROI когда ≥10 переиспользуемых компонентов. На MVP-скелете с 5 экранами overkill.

### §3.5 Quality

**Директор:** quality-director. **Регламент:** `departments/quality.md` v1.0 — развёрнутый (144 строки), много правил. **Зрелость:** средняя — есть тесты (211 за Батч A), OWASP Top 10 упомянут «раз в фазу», но без инструментов. Нет SAST/DAST, нет тестов для субагентов.

#### Текущее состояние (as-is)

#### Находки Scout

**1. [Риск] Q-4 — Автоматизированный security-скан в CI: Bandit + pip-audit.**

Проблема: сейчас security-аудит — ручная операция «раз в фазу». Классические уязвимости (hardcoded secrets, SQL injection, устаревшие зависимости с CVE) не ловятся автоматически. `ruff` такое не ищет.
Решение: добавить в CI два job'а:
- `bandit -r backend/app/` (SAST для Python, бесплатный, поддерживается OWASP).
- `pip-audit` (CVE-сканер для Python-зависимостей).
DAST (ZAP/nuclei) — отложить до наличия staging, иначе не на чём запускать.
Что у нас уже есть: `ruff` для стиля.
Impact: **high** — ловит CVE в зависимостях и распространённые уязвимости. Effort: **~1 день** qa + devops. Приоритет: **P0**.

**2. [Работа] Q-2 — OWASP AI Testing Guide как раздел регламента quality.md.**

Проблема: **M-OS — система субагентов ИИ**. У нас нет ни одного теста на prompt injection, output validation, model poisoning. Это уникальная слепая зона, отсутствующая у classic-SaaS (у них нет ИИ). Найденный Scout'ом руководящий документ OWASP 2026 — прямо под наш профиль.

Что у нас уже есть: CLAUDE.md фиксирует правила для субагентов, но тестов, проверяющих эти правила, нет.
Impact: **high** — закрывает слепую зону. Effort: **~3 дня** (анализ гайда, адаптация, 5–7 тестов-образцов). Приоритет: **P0**.

Решение: расширить раздел «Чек-лист reviewer» в `departments/quality.md` до 10 пунктов OWASP API Top 10, добавить ссылку на отдельный `docs/qa/owasp-api-2026-checklist.md` с подробностями.

- Критерий готовности: локально в dev — трейсы видны в Jaeger, логи backend попадают в Loki, Grafana открывается на `localhost:3001`.
- Prod-деплой — отдельная задача при первом релизе M-OS-1.
- Откат: remove 3 сервиса из compose, убрать middleware.
- Effort: 2 + 1 = 3 дня.

### Шаг 4 (неделя 2–3): Infrastructure v1.0 регламент

- Ответственный: **активация infra-director** (сейчас dormant — запрос Координатора Владельцу на активацию). Если активация откладывается, по согласованию с Координатором регламент пишет Координатор + devops совместно как bootstrap.
- Действия: регламент `departments/infrastructure.md` v1.0 120+ строк, структура по образцу `backend.md`: Сферы / Правила / CI/CD стандарты / Deploy-процедура / Backup / Incident runbook (включает I-3 лайт-версию) / Метрики.
- Критерий готовности: v1.0 прошла governance-комиссию, Директор infra утверждён.
- Откат: вернуться к v0.1 (тривиально).
- Effort: 2 дня.

### Шаг 5 (неделя 3–4): Q-2 OWASP AI Testing Guide адаптация

- Ответственный: quality-director (бриф) + ri-analyst (адаптация гайда).
- Действия:
  1. ri-analyst читает OWASP AI Testing Guide 2026 (источник Scout'а), пишет 1-страничный чек-лист для субагентов: 7–10 проверок.
  2. qa пишет 5–7 тестов-образцов, покрывающих самые вероятные сценарии (prompt injection через пользовательское поле, выход за пределы FILES_ALLOWED, генерация секрета в output).
  3. quality-director добавляет раздел «Testing AI subagents» в `departments/quality.md` v1.2.
- Критерий готовности: 5+ тестов в `backend/tests/test_subagent_safety.py` зелёные + раздел в регламенте.
- Откат: удалить тесты + revert регламента.
- Effort: 3 дня (1 день analyst + 2 дня qa).

**Итого шаги 1–5:** 1 + 0.5 + 3 + 2 + 3 = 9.5 рабочих дней за 3–4 календарные недели (учитывая параллельность M-OS-1 и зависимости).

**Распараллеливание:**
- Неделя 1: шаги 1 + 2 (разные департаменты, не блокируют).
- Неделя 1–2: шаг 3 (зависит от backend + devops, независим от шагов 1–2 по людям).
- Неделя 2–3: шаг 4 (требует активации infra-director — отдельный шаг Координатора к Владельцу).
- Неделя 3–4: шаг 5 (после шагов 1–3, чтобы уже был CI для запуска новых тестов).

## §6. Риски внедрения

### 6.1 Перегруз команды

**Риск.** M-OS-1 «Скелет» идёт параллельно, 14–15 недель по плану (Memory: `project_m_os_1_decisions.md`). Если воткнуть 10 дней аудит-задач без распараллеливания — M-OS-1 сдвинется на 2 недели.

**Митигация.** План §5 растянут на 3–4 недели и затрагивает **разные субагенты** (quality, governance, backend, infra, ri). Главная нагрузка — backend-dev (2 дня B-1) и infra-director/devops (3 дня I-2 + 2 дня регламент). Их работы в эти дни в M-OS-1 не критические — Волна 1 Foundation закрывается раньше.

### 6.2 Конфликт с текущим регламентом

**Риск.** Шаги 2 и 4 меняют регламенты (governance.md, infrastructure.md). В этот момент могут быть «в полёте» другие правки регламента.

**Митигация.** Заявки идут через governance-комиссию по стандартному процессу — конфликты разрешает сам governance-director. Нет «обходного пути».

### 6.3 Активация infra-director блокирует шаг 4

**Риск.** Сейчас infra-director dormant. Без его активации шаг 4 (регламент v1.0) невозможен либо пишется Координатором как bootstrap.

**Митигация.** Координатор эскалирует Владельцу отдельный запрос: «нужна активация infra-director для M-OS-1 production-подготовки, т.к. в текущем виде регламент v0.1 — не сшитая ткань». Владелец решает: активация до шага 4 или bootstrap v1.0 Координатором.

### 6.4 Q-2 AI Testing — риск false positive

**Риск.** OWASP AI Testing Guide — это гайд для LLM-продуктов вообще. Тесты, адаптированные под Claude Code, могут ловить особенности самого Claude, не наших субагентов. Риск — много xfail'ов.

**Митигация.** Начать с 5 тестов, не 15. Первый раунд — baseline. Если 4 из 5 — xfail, значит гайд неверно адаптирован, правим чек-лист.

### 6.5 Риск «не внедрили — забыли»

**Риск.** RFC-005 принят → через месяц про него забыли → регламенты остались старые, пункты P1/P2 повисли.

**Митигация.** После приёма RFC Координатор открывает 10 задач в `project_tasks_log.md` с привязкой к пунктам Top-10. ri-director включает «% пунктов RFC-005 внедрено» в weekly digest.

### 6.6 Оставшиеся off-table риски

- **Budget.** Ни одна рекомендация не требует денежных затрат (все инструменты — open-source MIT/Apache).
- **Безопасность.** Добавление Jaeger/Loki увеличивает attack surface. Митигация: bind на `127.0.0.1` в dev, в prod — через reverse-proxy с basic-auth на старте.
- **Зависимость от внешних образов.** Jaeger, Grafana — Linux-foundation CNCF, риск abandonment низкий.

## §8. Честные ограничения этого RFC

1. **4 часа на 8 департаментов — поверхностный обзор.** Каждая из 26 находок, взятая отдельно, заслуживает собственный RFC-200-слов. Я сжал до 3 рекомендаций на департамент. Если по какому-то разделу требуется углубление — отдельная задача на ri-analyst в следующую неделю.
2. **Самоаудит R&I (§3.7) — слабое место.** Нужна независимая проверка governance-auditor.
3. **Scout-находки — из статей, не из практики.** Индустрия 2026 — это мейнстрим. Наш масштаб (1 пилотный объект, 85 домов) отличается от масштабов Shopify/IBM/Atlassian, на которых эти практики тестированы. Я фильтровал по размеру (см. раздел «Отклонено» в каждом §3.X), но мог ошибиться.
4. **Я не запускал пилоты.** Все оценки Effort — экспертные. Реальный Effort может отличаться в 1.5–2 раза. Митигация — первые 2 пилота (шаги 1 и 2) короткие (0.5–1 день), ошибка ограничена.
