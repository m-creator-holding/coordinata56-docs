# Phase 0 Hooks — бриф ri-analyst

## 0. Зачем этот документ отдельно

Родительский план (`rfc-004-hooks-phase-0-plan.md`) описывает пилот **целиком**: что делает backend-dev, что делает ri-analyst, что утверждает governance-director. Этот бриф вычленяет **только задачу ri-analyst** и снабжает её:
- чётким разделением на 4 подзадачи (чтобы параллелить с backend-dev, не ждать его);
- форматом отчёта (жёсткий шаблон, чтобы governance-director не возвращал на правки);
- DoD именно для части ri-analyst (а не для пилота в целом);
- **типовыми рисками тестирования хуков**, не описанными в плане — их знает только R&I по предыдущим аналогам (тесты CI-gate'ов, тесты pre-commit в других проектах).

Советую прочитать этот бриф целиком до старта, потом §4 и §5 родительского плана, потом садиться за работу.

---

## 1. Контекст в одном абзаце

Backend-dev пишет 5 хуков (H-1…H-5): три pre-commit (секреты, `git add -A`, ruff+pytest) и два Claude Code hook (SendMessage к dormant, `ultrathink` для Opus). Бюджет пилота — 2 дня, из них 1.5 дня у backend-dev, 0.5 дня у Вас. Ваша часть — **приёмочное тестирование**: убедиться, что хуки ловят то, что должны, не ловят лишнего, не тормозят коммит и откатываются за минуту. Вердикт (adopt / refine / reject) выносите Вы в отчёте; governance-director заверяет; итог идёт Координатору.

---

## 2. Ваши 4 подзадачи (в порядке приоритета)

**Важно:** первые 3 подзадачи не требуют готовых хуков. Стартуйте параллельно с backend-dev, а не после него. Цель — к моменту, когда backend-dev отдаёт deliverables, у Вас уже готов стенд и остаётся только прогон + отчёт (≈1.5–2 часа), а не полные 4 часа с нуля.

### Подзадача А — Минированный стенд (5 сценариев), ~60 мин

Работа в отдельной директории `/root/coordinata56/docs/research/pilots/hooks-phase-0-test-fixtures/` (создать). Для каждой «мины» — подкаталог `mine-<N>-<slug>/` с:
- `README.md` (1 абзац: что тестирует, какой хук должен поймать, ожидаемый результат — block/warn);
- сценарием воспроизведения (bash-скрипт `reproduce.sh` или `.md` с пошаговым списком команд);
- заготовкой «грязных» файлов, если применимо (`.env.sample-dirty` и т.п., **без реальных секретов** — используйте `FAKE_KEY_` префикс на всех литералах, чтобы не словить собственный хук H-1 при коммите фикстур).

5 мин — дословно по §4.1 родительского плана:
1. `.env` со строкой вида `JWT_SECRET_KEY=FAKE_KEY_super_secret_12345` в staged → ждём block от H-1.
2. Симулированный `git add -A` на 15 файлах, из которых 3 созданы сторонним процессом менее 30 мин назад → ждём warning/block от H-2.
3. `SendMessage(to="design-director", ...)` при dormant design-director → ждём warning от H-3.
4. `Agent(subagent_type="backend-director", prompt="...")` без `ultrathink`/`think harder`/`think hard` → ждём warning от H-4.
5. `unused import` в staged `.py` файле внутри `backend/app/` → ждём block от H-5 (ruff).

**Критично:** на шаге 1 литерал секрета должен быть заведомо поддельным. Но именно по этой причине возможна ошибка «H-1 пропустил, потому что у нас в паттерне было требование длины 8+ символов, а `FAKE_KEY_12345` матчится по длине» — то есть FAKE-префикс *не должен* отключать срабатывание. Подтвердите по факту получения паттернов из H-1, что фикстура попадает под regex.

### Подзадача Б — 20 нормальных сценариев (false-positive guard), ~45 мин

По §4.2 плана: не более 1 ложно-положительного срабатывания на 20 легитимных операций. Сценарии записать в `docs/research/pilots/hooks-phase-0-test-fixtures/clean-scenarios.md` — таблица на 20 строк: `№ | сценарий | ожидание | затронутый хук`.

Минимальный состав (закрывает все 5 хуков):
- 4 обычных коммита без секретов: правка кода сервиса, правка теста без пароля, правка markdown в `docs/`, правка `.gitignore`.
- 2 коммита с вполне ожидаемыми вхождениями слова «password» в тексте (например, markdown-документ про политику паролей; `password_hash` колонка модели) — H-1 обязан их **пропустить**.
- 3 нормальных `git add` на 1–3 файла: правка одного сервиса, правка теста + сервиса парой, правка `CHANGELOG.md` — H-2 обязан не срабатывать.
- 2 легитимных `git add -A` на ≤10 файлов в пустом staging → H-2 пропускает (сам список малый).
- 2 `SendMessage` к **активному** background-агенту (имитация — запустить один Agent с `run_in_background: true`, пока работает — отправить; H-3 молчит).
- 2 `Agent`-вызова к Opus-агенту **с** `ultrathink` в промпте → H-4 молчит.
- 2 `Agent`-вызова к Sonnet-агенту (например, scout, quality-worker) без `ultrathink` → H-4 молчит (проверка по справочнику Opus).
- 3 коммита Python-файлов, прошедших `ruff check` заранее → H-5 пропускает.

Сценарии можно не прогонять все сейчас — готовите только чек-лист и скрипты. Прогон — в подзадаче Г.

### Подзадача В — Скрипт измерения оверхеда коммита, ~30 мин

По §4.3 плана: разница между `time git commit` до и после установки хуков ≤2 секунды на среднем коммите (5–10 файлов).

Скрипт `measure-overhead.sh` в той же директории фикстур:
1. Подготовка: 10 staged Python-файлов (можно массивом `touch backend/tmp/benchmark_N.py` с 20 строками каждый, правда затем откатить).
2. **Baseline:** хуки отключены (`git config core.hooksPath /dev/null` или просто `.git/hooks/pre-commit.disabled`), `time git commit -m "benchmark"` × 5 итераций, записать wall-clock из `real`.
3. **Treatment:** хуки включены, те же 5 итераций.
4. Средние значения и разница. При разнице >2 секунд — зафиксировать **какой хук доминирует** (можно быстро проверить, отключая по одному: H-5 с pytest — самый подозрительный).

Формат лога: `overhead-measurements.csv` с колонками `run_id,mode,real_sec,user_sec,sys_sec`. Плюс сводка в отчёте.

**Важная деталь:** если backend-dev реализует H-5 с `pytest` — замеряйте на реальном модуле с реальными тестами, не на пустых `benchmark_N.py`. Пустой модуль тесты не запустит, измерение будет некорректным.

### Подзадача Г — Прогон + отчёт, ~90 мин (после готовности backend-dev)

1. Backend-dev сигнализирует через Координатора о готовности deliverables (см. §5 родительского плана — `scripts/install-hooks.sh`, хуки, доки).
2. Ставите хуки через `scripts/install-hooks.sh` в чистый worktree `/root/worktrees/coordinata56-hooks-pilot/` (если worktree не создан — попросите Координатора; **не работайте в основной ветке**).
3. Прогоняете 5 мин → записываете результат (block / warn / miss / error).
4. Прогоняете 20 чистых сценариев → записываете false-positive count.
5. Прогоняете `measure-overhead.sh` → получаете дельту.
6. Проверяете откат (§4.5): выполняете последовательность из 4 команд, убеждаетесь что `git commit` чистого файла проходит за исходное время.
7. Заполняете отчёт по шаблону из §3 этого брифа.

## 3. Формат отчёта (жёсткий шаблон)

Файл: `docs/research/pilots/2026-04-XX-hooks-phase-0-report.md` (подставить фактическую дату приёмки).

Frontmatter:
```yaml
---
title: Phase 0 Hooks — отчёт по пилоту
parent_plan: docs/research/rfc/rfc-004-hooks-phase-0-plan.md
parent_brief: docs/research/rfc/rfc-004-hooks-pilot-analyst-brief.md
status: ready-for-verdict   # либо: needs-refine / reject-proposed
date: 2026-04-XX
author: ri-analyst
test_environment: /root/worktrees/coordinata56-hooks-pilot/ (worktree pilot branch)
---
```

Обязательные разделы:

### 3.1. Резюме (≤150 слов)
Один абзац: сколько мин поймано, сколько false-positives, оверхед, итоговая рекомендация (adopt / refine / reject). Без воды, без списков — чистая проза для governance-director.

### 3.2. Таблица «мина → хук → результат»
```
| Мина | Хук | Ожидание | Факт         | Exit code | Комментарий |
|------|-----|----------|--------------|-----------|-------------|
| 1    | H-1 | block    | block        | 1         | паттерн сработал |
| 2    | H-2 | warn    | warn+confirm | 0         | требует ENV       |
| ...  | ... | ...      | ...          | ...       | ...              |
```

Включая **цитаты stderr** (3–5 строк вывода хука) для каждой мины — это нужно governance-director'у для оценки текстов сообщений.

### 3.3. False-positive result
Таблица или список 20 сценариев + сколько сработало ложно. Если >1 — объяснить какое правило перезатянуто и какой предлагается патч (regex, условие).

### 3.4. Оверхед
Таблица baseline vs treatment × 5 итераций. Среднее, медиана, максимум. Если H-5 доминирует — вывод: оставить только `ruff check`, `pytest` переносить в CI (как предусмотрено в §4.3 плана).

### 3.5. Откат
Список команд + время отката (секунды). Подтверждение, что посторонних артефактов не осталось.

### 3.6. Обнаруженные недочёты (по приоритетам)
- **P0** — без этого adopt невозможен (например: мина не ловится вообще, оверхед >5 сек).
- **P1** — adopt-with-changes (например: формулировка сообщения хука непонятна, нужна правка текста).
- **P2** — nice-to-have (например: реестр active-workers.json можно завести симлинком, а не файлом).

### 3.7. Вердикт ri-analyst
Один из трёх:
1. **adopt** — 4+ мин пойманы, ≤1 false-positive, оверхед ≤2 с, откат работает.
2. **refine** — 3 критерия из 4 выполнены, 1 требует точечной правки (список P1-замечаний).
3. **reject** — <3 мин пойманы, или >3 false-positive, или оверхед >5 с, или откат ломает репо.

Вердикт — **Ваш** как ri-analyst. Governance-director может его подтвердить, попросить уточнений или оспорить — но не меняет цифры.

### 3.8. Что рекомендовано включить в Phase 1
Не обязательный раздел, но полезный: если в ходе тестирования Вы увидели паттерн, который сам просится в хук, но отсутствует в H-1…H-5 — короткая заметка для следующей итерации. Не больше 200 слов.

## 4. Acceptance criteria Вашей части (DoD ri-analyst)

Ваша работа закончена, когда:
1. Фикстуры `docs/research/pilots/hooks-phase-0-test-fixtures/` созданы и коммитабельны (не содержат реальных секретов, не ломают собственный H-1 при коммите — т.е. все литералы помечены префиксами-маркерами и исключены из паттерна либо отключены через `# hook-exception: H-1 test fixture`).
2. 20 чистых сценариев описаны и проверены.
3. Скрипт оверхеда работает и даёт воспроизводимый результат (3+ запуска с расхождением ≤10 %).
4. Прогон всех 5 мин + 20 чистых + оверхеда выполнен на готовых хуках.
5. Отчёт опубликован по шаблону §3.
6. Вердикт выставлен.
7. Время — ≤4 часа (при превышении — эскалация ri-director с кратким статусом «где застрял»).

## 5. Типовые риски тестирования хуков (критично прочесть)

Это то, что не описано в родительском плане, но регулярно ломает пилоты pre-commit в других проектах. Список из опыта R&I — **не повторяйте чужих ошибок**.

### 5.1. Самосаботаж фикстур
Ваши собственные фикстуры (с «минами») при коммите в репо попадут под хук H-1. Решение: все литералы секретов в фикстурах — с префиксом `FAKE_KEY_`, **но** проверьте что regex H-1 их всё равно ловит (для мины!), а при коммите самих фикстур — используйте маркер `# hook-exception: H-1 test fixture` (предусмотрен §3 H-1 плана). Если маркер не работает — это P0-замечание backend-dev'у, не Ваша проблема.

### 5.2. Путаница «warn vs block»
H-2, H-3, H-4 по плану — **warning**, не block. Новички часто пишут в отчёте «мина не поймана», хотя хук на самом деле вывел warning (exit 0). Проверяйте **stderr**, а не exit code. Для warn-хуков критерий «поймал» = наличие ожидаемой подстроки в stderr.

### 5.3. Загрязнение глобального состояния
Claude Code hooks (`~/.claude/hooks/*`) — глобальные, не per-repo. При тестировании H-3, H-4 Вы можете случайно зацепить текущую реальную сессию Координатора (если хуки установлены на боевом инстансе). Два защитных шага:
- Тестируйте в отдельной shell-сессии с переменной `CLAUDE_HOOKS_DIR=/tmp/claude-hooks-pilot` (если backend-dev поддержал эту опцию; если нет — это P1-замечание).
- После теста обязательно откатите глобальные хуки даже если отчёт ещё не закрыт.

### 5.4. Ложная стабильность оверхеда
Первые коммиты после установки хуков всегда медленнее из-за cold cache Python (import ruff, import pytest). Первый запуск в серии — **отбрасывайте**, он не репрезентативен. Считайте среднее со 2-го по 5-й.

### 5.5. Недетерминированный H-2
Эвристика «чужие файлы» в H-2 по плану — через mtime + hint-файл `active-workers.json`. Это **зависит от состояния файловой системы** в момент теста. Ваша фикстура для мины 2 должна явно подготовить `active-workers.json` с нужными путями перед запуском. Иначе Вы не воспроизведёте кейс.

### 5.6. Pytest в H-5 может не найти тестов
Если backend-dev использует module→test mapping из §5 плана, а Ваш «грязный» файл не попадает под mapping — H-5 запустит `pytest -q` по всему `backend/tests/` с таймаутом 60 с, это скажется на оверхеде. В минной фикстуре 5 используйте файл, для которого **есть** маппинг (например, `backend/app/services/project.py` — тестовый файл заведомо существует).

### 5.7. «Все 5 прошли — значит adopt» — неверно
Критерий adopt (§4.1 плана) — **4 из 5**, с допустимой одной осечкой. Но если осечка — это H-1 (секреты), это **P0** независимо от формального прохождения 4/5: мы не можем adopt'нуть систему, которая не ловит утечку секрета. Явно зафиксируйте в отчёте: «осечка на H-<N> — приемлемо, потому что… / неприемлемо, потому что это критичный класс». H-1 — всегда неприемлемо.

### 5.8. Один прогон ≠ статистика
Для false-positive (§4.2) — 20 сценариев разово. Для оверхеда (§4.3) — 5 итераций минимум. Для мин — по 1 прогону достаточно (бинарный результат). Не пытайтесь считать proxy на малых выборках.

### 5.9. Worktree — обязателен
Работайте в `/root/worktrees/coordinata56-hooks-pilot/`, не в основной ветке. Если Координатор не создал worktree — запросите перед стартом подзадачи Г (к подзадачам А–В — не нужен, работаете с фикстурами локально).

## 6. Эскалации

- **Хуки сломаны (P0):** Не пишите отчёт adopt/refine/reject — пишите вердикт `reject` с указанием конкретного хука и перебрасывайте ri-director. ri-director вернёт backend-dev через Координатора.
- **Время пошло за 4 часа:** остановитесь на текущем шаге, задокументируйте где застряли одним абзацем, эскалируйте ri-director. ri-director решает: дорезать скоуп / дать доп. час / переназначить.
- **Claude Code hooks не работают на Вашей машине вообще:** проверьте `~/.claude/settings.json` — возможно `hooks:` блок не подключён. Если и настройка корректна, а хуки не вызываются — это P0 backend-dev'у, не продолжайте тест H-3 и H-4 (сразу в отчёт).

---

## 7. Когда старт

Старт — **сразу после получения Agent-вызова от Координатора** с ссылкой на этот бриф. Подзадачи А, Б, В — параллельно с работой backend-dev (Вам не нужны готовые хуки для них). Подзадача Г — по сигналу Координатора, что deliverables backend-dev готовы.

После завершения — отчёт сдаёте Координатору; Координатор запрашивает вердикт governance-director; при accept — коммит через стандартный маршрут reviewer → commit.
