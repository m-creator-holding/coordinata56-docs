# Phase 0 Hooks — план пилота

Это **не самостоятельный RFC**, а приложение к RFC-2026-004 «Оптимизация маршрутизации Координатора». В RFC-004 (v0.1, Вариант B') Hooks описаны как «быстрая альтернатива полному Orchestration Layer: 80 % эффекта за 3 % трудозатрат». Weekly digest 2026-04-18 вынес рекомендацию «не ждать v1.0 RFC-004, запустить Phase 0 параллельно». Владелец одобрил 2 дня работы backend-dev + ri-analyst.

Документ описывает **только** план пилота. Сами хуки пишет backend-dev. Тестирует ri-analyst. Вердикт по DoD выносит governance-director. Коммит — Координатор.

---

## 1. Расшифровка простым языком для Владельца

### Что делаем

Представьте цех: есть рабочий, который собирает деталь, и есть ОТК — контролёр, который ловит брак. Сейчас у нас контролёр (это субагент-reviewer) работает **в конце смены**: рабочий всё собрал, положил на стол, контролёр открывает пакет, находит десять мелких ошибок — «гайка не того диаметра», «шов с непроваром», — рабочий переделывает, контролёр смотрит снова, снова правки. В среднем каждый узел проходит 2–3 такие петли.

Хуки — это простой **щит у станка**. Рабочий пытается положить в сборку не ту гайку — станок просто не принимает её, подаёт звуковой сигнал, рабочий берёт правильную. Щит ловит не все ошибки, только типовые и повторяющиеся (те, что встречаются раз за разом и записаны в регламенте). Сложные логические дефекты всё равно пойдут к ОТК. Но 60–80 % механических ошибок отсекаются за секунду, а не за час ручной проверки.

Технически это маленькие скрипты, которые запускаются при git-операциях (commit, push) и при вызовах субагентов. Если скрипт видит запрещённый паттерн — он блокирует действие и печатает одну строку: «это нарушает правило X из CLAUDE.md». Ничего не меняет в коде, ничего не записывает. Просто останавливает руку перед ошибкой.

### Зачем это нам

Три конкретных повторяющихся случая последнего месяца:
1. **Литеральный пароль в тестах** — три раза подряд за две фазы (Phase 2 Round 2 BLOCKER-1, Phase 3 Batch A P0-2, PR#1 Wave 1 P0-2). Каждый — ручное ловлением reviewer'а, каждый — round fix, каждый — задержка на 1–2 часа.
2. **Случайный `git add -A` подхватил чужую работу** — 15 апреля 2026 коммит регламента подобрал файлы QA из другого батча, пришлось откатывать и переревьюивать.
3. **SQL-запросы утекли в сервисный слой** — нарушение ADR 0004 MUST #1a. Поймано дважды (Batch A и PR#2 Wave 1), каждый раз — полный раунд ревью.

По интерим-аудиту quality-director от 2026-04-18 (см. раздел «Рекомендация системная») — оба рецидивирующих класса (пароли и SQL вне репозиториев) должны автоматизироваться через pre-commit + ruff. Хуки убирают механическую работу с ревьюера — он начинает ловить только **смысловые** дефекты, а не «забыл заменить литерал».

Экономия: оценочно 3–5 часов ревьюера в неделю при текущем потоке PR. При запуске M-OS-1 с шестью параллельными Директорами эта экономия масштабируется линейно.

### Что предлагаем внедрить

5 хуков первого приоритета, две категории:
1. **Git-хуки (pre-commit)** — ловят ошибки при попытке закоммитить код. Три штуки: защита секретов, защита от хаотичного `git add -A`, запуск лёгкого линта на затронутых файлах.
2. **Claude Code-хуки (PreToolUse, PostToolUse)** — ловят ошибки в агентных рабочих циклах. Два штуки: предупреждение о SendMessage к dormant-агенту (решает проблему 17 апреля с потерянными сообщениями), напоминание о `ultrathink` в промптах к Opus-агентам.

Срок пилота — 2 рабочих дня: 1.5 дня backend-dev пишет хуки, 0.5 дня ri-analyst тестирует на искусственных «плохих» входах. Стоимость ноль (скрипты на bash + Python). Откат — удаление файлов `.git/hooks/*` и `~/.claude/hooks/*.json`, 1 минута.

## 2. Какую задачу решаем (формально)

Цель пилота: **механически заблокировать ≥4 из 5 ранее словленных рецидивирующих дефектов** на искусственном «заминированном» коммите перед тем, как они дойдут до reviewer'а.

Не-цели:
- Не заменяем reviewer'а. Хуки ловят только то, что описано регулярным выражением или формальной проверкой. Смысловые дефекты (IDOR, fail-open в RBAC, dropped requirements из брифа) — всё ещё задача reviewer/quality-director.
- Не вводим CI/CD — pre-commit работает локально, CI уже существует (`ci.yml`, `docs-validation.yml`) и расширяется отдельной задачей.
- Не автоматизируем запуск субагентов по триггерам — это сфера Orchestration Layer (ADR 0012), не Phase 0.

## 3. Список хуков первого приоритета (5 штук)

Приоритизация по двум осям: (1) частота рецидива проблемы за последние 4 недели, (2) простота реализации.

### H-1. Блокировка `.env*` / `*.key` при коммите — pre-commit

**Что проверяет:** при `git commit` сканирует список staged-файлов и содержимое. Блокирует коммит, если:
- в staged есть файл, чьё имя матчится паттерну `.env*` (кроме `.env.example`), `*.key`, `*.pem`, `*credentials*`, `secrets.json`;
- в diff новых/изменённых файлов встречается подстрока `JWT_SECRET_KEY=`, `DATABASE_URL=postgresql://.*:.*@`, `password\s*=\s*["'][a-zA-Z0-9]{8,}["']` с литералом длиннее 8 символов (не `password=""` или `password=<env>`).

**Что блокирует:** `git commit` падает с exit code 1 и сообщением формата:
```
HOOK H-1: staged файл .env.production содержит секрет. Удалите из индекса (git reset HEAD .env.production) либо переименуйте в .env.example с заменой значений на <placeholder>.
```

**Технология:** bash-скрипт в `.git/hooks/pre-commit` + одна Python-функция для regex-сканирования diff (через `git diff --cached --no-color`). Bash для простоты; без внешних зависимостей.

**Почему этот хук — первый по очереди:** закрывает Б-01 (три рецидива литеральных паролей) и часть П-09 deploy-secrets-checklist. Единственный из пяти, без которого реальный риск — утечка в публичное зеркало `coordinata56-docs`.

**Обходное правило:** маркер `# hook-exception: H-1 <причина>` в комментарии к строке (например, для тестов, где заведомо dummy-secret). Маркер ищется reviewer'ом отдельно и документируется.

---

### H-2. Запрет `git commit` с сырым `git add -A` — pre-commit

**Что проверяет:** анализирует `git status --porcelain` до и после `git add`. Если Координатор (или любой исполнитель) сделал `git add -A` / `git add .` **и** количество staged-файлов превышает 10, **и** среди них есть файлы, которые не были в задаче (эвристика: модифицированы менее чем за 30 минут другим процессом, детектируется через mtime + hint-файл от background-агентов) — выводит предупреждение с явным списком подозрительных файлов и запрашивает подтверждение.

**Что блокирует:** коммит не блокируется жёстко, но выводится warning и список; exit code 0 только если у скрипта есть явное согласие через переменную окружения `COMMIT_CONFIRMED=1` или интерактивный ответ `y`. В автономном режиме Claude (без TTY) — блокирует и требует явного списка файлов через `git add path1 path2` вместо `-A`.

**Технология:** bash + `git status --porcelain` + timestamp через `stat -c %Y`. Hint-файл `/root/coordinata56/.claude/active-workers.json` (формат — список background-субагентов со списком файлов, которые они держат под собой; пишется Координатором при запуске background-субагента, читается хуком).

**Почему важно:** прямое закрытие Б-04 (инцидент 15 апреля). Правило в CLAUDE.md есть, но срабатывает только на внимание Координатора. Хук убирает человеческий фактор.

**Обходное правило:** в автономном режиме — явный ENV `COORDINATOR_ALLOW_ADD_ALL=1` только в исключительных случаях (редактирование регламента, когда все 30 файлов правда нужны), с обязательным логированием в `/root/.claude/projects/-root/memory/git_add_all_log.md`.

### H-3. SubagentStop предупреждение для SendMessage к dormant — Claude Code hook

**Что проверяет:** при `PostToolUse` события `SendMessage` сравнивает получателя с реестром активных background-субагентов (файл `/root/.claude/teams/default/active-agents.json`, поддерживается hook-лайфциклом `SubagentStart` / `SubagentStop`). Если получатель **не в активных** — выводит предупреждение.

**Что блокирует:** ничего не блокирует. Только warning в stderr Координатора:
```
HOOK H-3 WARNING: SendMessage отправлен к <agent> (dormant с <timestamp>).
Сообщение попадёт в inbox (архив), но не запустит работу.
Для запуска новой сессии: Agent(subagent_type="<agent>", prompt=...).
```

**Технология:** Claude Code hook типа `PostToolUse` с фильтром по `tool_name == "SendMessage"`. Скрипт на Python (есть встроенная поддержка JSON payload в hook event). Реестр активных агентов ведётся двумя другими hooks: `SubagentStart` добавляет запись, `SubagentStop` удаляет.

**Почему важно:** 17 апреля потеряно ~7 SendMessage-инструкций к dormant-директорам (Design System, RFC-006, RFC-007, ADR 0017-0021, DB drafts) — вся работа не продвинулась за ночь. Правило в CLAUDE.md записано 18 апреля, но требует сознательной проверки Координатора перед каждым SendMessage. Хук делает это автоматически.

**Почему warning, не block:** бывают легитимные случаи SendMessage к dormant для архивирования (Координатор хочет оставить запись в inbox для будущей сессии агента). Блокировать — слишком грубо; предупреждения достаточно, оно немедленно видно в stream-output.

### H-4. UserPromptSubmit напоминание о `ultrathink` для Opus-субагентов — Claude Code hook

**Что проверяет:** при `PreToolUse` события `Agent` смотрит аргумент `subagent_type`, проверяет по справочнику (`docs/agents/opus-agents.yaml` — ведёт governance-director) относится ли он к Opus-модели. Если да — сканирует поле `prompt` на наличие ключевых слов `ultrathink`, `think harder`, `think hard`. Если ни одного нет — выводит warning.

**Что блокирует:** не блокирует. Warning в stderr:
```
HOOK H-4 WARNING: Agent-вызов к <subagent_type> (Opus) без ключевого слова extended thinking.
Правило CLAUDE.md (2026-04-18 msg 1355): для Opus-агентов вставлять `ultrathink` в начало промпта.
Продолжаем без thinking; если это намеренно — игнорируйте сообщение.
```

**Технология:** Claude Code hook `PreToolUse` с фильтром `tool_name == "Agent"`. Python-скрипт: YAML-парсинг справочника Opus-агентов + regex поиск ключевых слов в prompt.

**Почему важно:** правило действует два дня (с 18 апреля), применение пока дисциплинарное. Для Opus-агентов extended thinking критичен (качество рассуждения ×2–3 на сложных задачах), при забывчивости — мы платим за Opus, получаем работу уровня Sonnet.

**Почему warning, не block:** для тактических вопросов Opus-агенту (быстрое уточнение) thinking может быть избыточен. Правило — «по умолчанию да», но исключения допустимы. Блокировать — ломает гибкость.

### H-5. Pre-commit линт и fast-тесты на затронутых файлах — pre-commit

**Что проверяет:** при `git commit` определяет список staged-файлов, фильтрует Python-файлы `backend/app/**/*.py` и `backend/tests/**/*.py`, запускает:
1. `ruff check <files>` — блокирует при exit code != 0;
2. `ruff format --check <files>` — блокирует при exit code != 0;
3. `pytest -q --timeout=30 <test_files_related>` — запускает только тесты, затрагивающие изменённые модули (через `pytest --collect-only` + mapping модуль→тест-файл). Блокирует при провале.

**Что блокирует:** коммит падает с exit code 1 + краткий отчёт `ruff` / `pytest`.

**Технология:** bash-скрипт в `.git/hooks/pre-commit` + вызовы `ruff` и `pytest` из venv проекта. Mapping модуль→тест — простое правило: `backend/app/services/project.py` → `backend/tests/services/test_project.py` и `backend/tests/api/test_projects_api.py`. При отсутствии мэппинга — запускается `pytest -q` по всему каталогу `backend/tests/` с лимитом 60 секунд.

**Почему важно:** каждый round ревью начинается с `ruff check` reviewer'ом. Если бы на стороне backend-dev ruff уже падал до коммита — круг сокращается на 1 проход. По PR#1 и PR#2 это бы закрыло 2-3 замечания reviewer'a round-0 (минорные стиль-вопросы типа nit-1, nit-2 из pre-commit-review PR#2).

**Ограничение:** fast-тесты — не замена полного `pytest` в CI. Только smoke для изменённых модулей, таймаут 30 секунд на тест, 60 секунд на всю проверку. Полный CI никто не отменяет.

## 4. Acceptance criteria пилота

Пилот засчитан, если **одновременно** выполнены следующие условия:

### 4.1. Все 5 хуков работают на искусственных «минах»

ri-analyst готовит тест-стенд с 5 минимальными репозиториями-копиями (или 5 ветками в изолированном worktree), в каждую подкладывает ровно одну «мину»:
- **Мина 1:** `echo "JWT_SECRET_KEY=super_secret_12345" >> .env` + `git add .env` + `git commit`.
- **Мина 2:** симулированный `git add -A` с 15 staged-файлами, где 3 — из «чужого» background-агента.
- **Мина 3:** `SendMessage(to="design-director", body="test")` когда design-director dormant.
- **Мина 4:** `Agent(subagent_type="backend-director", prompt="review PR#42")` без `ultrathink`.
- **Мина 5:** ruff-нарушение (`unused import`) в staged Python-файле.

**Успех:** минимум **4 из 5** мин заблокированы/поймаются warning'ом. Одна допустимая осечка — любая на выбор пилота.

### 4.2. Ложно-положительные срабатывания ≤1 на 20 нормальных действий

ri-analyst прогоняет 20 легитимных операций (обычный коммит без секретов, обычный Agent-вызов с `ultrathink`, SendMessage к реально активному background-агенту, и т.д.). **Успех:** не более 1 ложно-положительного срабатывания (warning или block там, где не должен был).

### 4.3. Оверхед времени ≤2 секунд на коммит

`time git commit` до и после установки хуков на среднем коммите (5-10 файлов). **Успех:** разница ≤2 секунд. Если ruff + pytest на затронутых файлах дают больше — H-5 сокращается до только `ruff check` (без format и pytest), фиксируется в отчёте как deferred.

### 4.4. Governance-director утвердил DoD

governance-director читает отчёт пилота и подтверждает что:
- хуки задокументированы в `docs/agents/hooks/README.md` (создаётся в пилоте);
- правила обхода (hook-exception маркеры, ENV-переменные) не создают дыры;
- реестры (`active-agents.json`, `opus-agents.yaml`) согласованы с существующими документами.

### 4.5. Откат работает

ri-analyst документирует процедуру отката в отчёте:
```bash
rm .git/hooks/pre-commit
rm -rf /root/.claude/hooks/
rm /root/coordinata56/.claude/active-workers.json
rm /root/coordinata56/docs/agents/opus-agents.yaml
```
Проверено, что после отката `git commit` работает как раньше. **Успех:** процедура занимает ≤1 минуты, никаких остаточных артефактов не мешают работе.

## 5. Разделение работы

Общий бюджет: 2 рабочих дня. Подзадачи не пересекаются по времени, возможна параллельная работа (но backend-dev заканчивает каждый хук отдельно, ri-analyst включает его в стенд как только тот готов).

### backend-dev — 1.5 дня (≈12 часов)

День 1, утро (4 часа): H-1 + H-5.
- H-1 pre-commit с regex-паттернами для секретов, плюс файл `.gitsecrets-patterns.txt` как источник истины для паттернов.
- H-5 pre-commit запуск ruff + ruff format + целевой pytest. Module→test mapping как простая Python-функция.
- Документация в `docs/agents/hooks/README.md` для этих двух.

День 1, вечер (4 часа): H-2 + реестр active-workers.
- H-2 pre-commit со статусом staged-файлов и эвристикой «чужие файлы».
- Реестр активных background-субагентов (`.claude/active-workers.json`) с двумя командами инсталляции (register/unregister), которые Координатор будет вызывать при старте/остановке background-агента.
- Документация H-2.

День 2, утро (4 часа): H-3 + H-4 (Claude Code hooks).
- H-3 PostToolUse на SendMessage + подсистема `SubagentStart`/`SubagentStop` hooks, ведущая реестр активных агентов.
- H-4 PreToolUse на Agent с проверкой `ultrathink`.
- Справочник `docs/agents/opus-agents.yaml` (одна страница, 11 Opus-агентов по состоянию на 2026-04-18, ссылка на CLAUDE.md как источник).
- Документация H-3, H-4.

**Deliverables backend-dev:**
- `.git/hooks/pre-commit` + `.git/hooks/_common.sh` (общие функции).
- `/root/.claude/hooks/pre-send-message.py`, `pre-agent-call.py`, `subagent-lifecycle.py`.
- Конфигурационные файлы: `.gitsecrets-patterns.txt`, `docs/agents/opus-agents.yaml`, `.claude/active-workers.json` (пустой init).
- `docs/agents/hooks/README.md` — документация всех 5 хуков.
- Скрипт установки `scripts/install-hooks.sh` (копирует в `.git/hooks/` и `~/.claude/hooks/`).

### ri-analyst — 0.5 дня (≈4 часа)

Утро дня 2 (2 часа): подготовка стенда.
- 5 «минированных» веток/worktree (описаны в §4.1).
- 20 нормальных сценариев для ложно-положительного теста (§4.2).
- Скрипт измерения времени коммита (§4.3).

Вечер дня 2 (2 часа): прогон + отчёт.
- Прогон всех минных и чистых сценариев.
- Замеры оверхеда.
- Отчёт в `docs/research/pilots/2026-04-20-hooks-phase-0-report.md` (дата подставляется фактическая).
- Сводная таблица «мина → хук → результат (block/warn/miss)».

**Deliverables ri-analyst:**
- Отчёт пилота с таблицей результатов и рекомендацией (adopt / refine / reject).

## 6. DoD (Definition of Done)

Пилот считается завершённым, когда:

1. Все 5 хуков установлены и работают на реальной машине (`81.31.244.71`).
2. 4+ из 5 мин пойманы хуками (критерий §4.1).
3. Ложно-положительных ≤1/20 (критерий §4.2).
4. Оверхед коммита ≤2 с (критерий §4.3).
5. Вердикт governance-director — accept / accept-with-changes (не reject).
6. Отчёт ri-analyst опубликован в `docs/research/pilots/`.
7. `scripts/install-hooks.sh` + документация в `docs/agents/hooks/README.md` — в основной ветке через стандартный ревью-маршрут (reviewer → commit).
8. Откат проверен и задокументирован (критерий §4.5).
9. В `docs/research/findings.md` добавлена запись «Phase 0 hooks adopted <дата>» со статусом и ссылкой на отчёт.

## 8. Что НЕ делаем в Phase 0 (scope guard)

Этот список нужен, чтобы Phase 0 не превратилась в «напишем все 20 хуков сразу». Следующие идеи — **отложены** до результата пилота:

- H-6: проверка «write-эндпоинт без audit_service.log» — сложная, требует AST-парсинга Python.
- H-7: проверка «фильтр в Python после LIMIT» — то же, AST.
- H-8: проверка корректности миграций (уже есть CI job `lint-migrations`, переносить локально — оверкилл).
- H-9: автоматический запуск reviewer при `git commit` — это уже Вариант B' основного RFC-004, не Phase 0.
- H-10: dashboard интеграция (PreToolUse/PostToolUse → HTTP на 8765) — отдельный проект (см. находку 2026-04-17 claude-code-hooks-multi-agent-observability).
- Расширение ruff кастомными правилами через `ruff check --select` — требует написания плагина, не Phase 0.

Если пилот пройден — приоритет H-6…H-10 определяется отдельным RFC по результатам первого месяца эксплуатации Phase 0.
