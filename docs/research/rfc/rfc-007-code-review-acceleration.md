# RFC-2026-007: Ускорение code review — с 2-3 раундов к 1.5 на PR

## Расшифровка простым языком для Владельца (обязательный раздел)

### Что делаем

Сейчас каждый PR — это как стройплощадка с тремя приёмочными визитами инспектора. Плиточник уложил плитку, инспектор пришёл — «шов неровный, затирка не та, подрозетники не вырезаны». Плиточник переделал — инспектор второй раз: «теперь всё, но обнаружил, что розетка смещена». Третий визит — «готово». Каждый визит — полдня инспектора и день плиточника.

В коде у нас то же самое. Поставим цифры из последних двух недель: PR#1 Волны 1 прошёл 2 раунда ревью, PR#2 Волны 1 уже на третьем, Phase 3 Batch C — 3 раунда. Среднее — 2.3 раунда на PR. Если каждый раунд — это минимум 30-45 минут работы reviewer + час-два backend-dev, то на каждый PR уходит ~4 часа «пинг-понга» сверх фактической разработки.

Цель RFC — **свести среднее число раундов к 1.5** (половина PR проходит за 1 раунд, половина за 2). Для этого разобрали 8 зафиксированных раундов ревью и вытащили 5 повторяющихся причин. Против каждой — типовой инструмент.

### Чем это нам полезно

Два прямых эффекта:

1. **Быстрее до commit.** Средний PR сокращается с ~4 часов доработки до ~2. На 4-5 PR в неделю (темп M-OS-1) экономится ~8-10 часов reviewer и ~16-20 часов backend-dev.

2. **Меньше усталостных ошибок.** Третий раунд ревью у одного и того же человека — худшее по качеству место в процессе. reviewer уже устал, отличает ошибку от уже виденного с трудом, пропускает свежие дефекты. Все три P0 Phase 3 Batch A — всплыли не в первом раунде, а в третьем, когда фокус рассыпался.

Экономия: при текущем темпе 4-5 PR/нед × экономия 4-6 часов PR = ~20 часов reviewer в неделю. Один reviewer-субагент — это примерно 15-20% от бюджета Quality-департамента.

### Что предлагаем внедрить

Не один «большой скачок», а три дешёвые правки чек-листов + рекомендация про pilot-проверку перед PR:

1. **Уточнить чек-лист `departments/backend.md`** — 5 новых пунктов по самым частым замечаниям (литералы секретов, SQL вне репозитория, IDOR на non-list методах, ADR 0005 формат, `# type: ignore` без обоснования). Занимает час governance-director.

2. **Ввести «pre-PR self-check» — 15-минутный прогон backend-dev'ом** перед открытием PR: `ruff check`, grep на запрещённые паттерны, checklist из 8 вопросов «сам себе». Занимает 15 минут backend-dev, экономит 30-60 минут round-0 reviewer'а.

3. **Hook H-5 из Phase 0 (RFC-004)** — автоматическая часть pre-PR self-check через pre-commit. Ловит механику без дисциплины.

**Сам пилот этого RFC — не нужен.** Эффект каждой из трёх правок измеряется естественно по следующим 4-5 PR M-OS-1 (считаем раунды, сравниваем с baseline 2.3). Если через месяц среднее не упало до ≤1.7 — возвращаемся к RFC и ищем следующий уровень.

Срок внедрения правок 1-3: 1-2 часа governance-director + включение Phase 0 Hooks (отдельный пилот, 2 дня).

## 1. Проблема (с метриками)

### 1.1. Baseline — среднее число раундов

Выборка: все PR/Batch с зафиксированными ревью в `docs/reviews/` за последний месяц.

| PR / Batch | Раундов | P0 round-0 | P1 round-0 | P0 round-1+ | Ссылка |
|---|---|---|---|---|---|
| Phase 2 auth | 3 | ? | ? | 1 (BLOCKER-1, R2) | `docs/reviews/phase2-*` (3 файла) |
| Phase 3 Batch A | 3 | 4 | 5 | 0 | `docs/reviews/phase3-batch-a-*` |
| Phase 3 Batch B | 2 | ~1 | ~2 | 0 | `docs/reviews/phase3-batch-b-*` |
| Phase 3 Batch C | 3 | ~2 | ~3 | 0 | `docs/reviews/phase3-batch-c-*` (5 файлов) |
| PR#1 Волны 1 Multi-company | 2 | 2 | 1 | 0 | `pr1-wave1-multicompany-round-1-head-review.md` |
| PR#1 Волны 1 addon OpenAPI | 2 | 0 | 1 | 0 | `pr1-addon-openapi-*` |
| PR#2 Волны 1 RBAC + Consent | 2 (ожидаем 3-й) | 2 | 3 | 0 | `pr2-wave1-rbac-consent-round-1-review.md` + pre-commit-review |

**Среднее: 2.43 раунда на единицу работы** (медиана — 2). Разброс 2-3. Ни одного случая «approve с первого прогона».

Цель RFC: среднее **≤1.7**, минимум 40% PR проходят за 1 раунд.

### 1.2. Стоимость раунда

По оценке reviewer'а в pre-commit-review PR#2 (18 комментариев, 2 P0 + 3 P1 + 6 P2 + 3 nit за один раунд): один раунд — 30-60 минут концентрированной работы + 5-10 минут формирования отчёта. Backend-dev на исправление: 1-3 часа, в зависимости от числа замечаний. **Суммарно раунд — 2-4 часа**.

Для PR с 3 раундами это **6-12 часов** сверх фактической реализации. На баче M-OS-1 с 8-10 PR в месяц — около 60-120 часов в месяц «пинг-понг-налога».

### 1.3. Почему это важно именно сейчас

До M-OS-0 количество PR было ~1-2 в неделю, overhead переносился. На M-OS-1:
- 6 параллельных Директоров будут генерировать ~15-20 PR/нед;
- reviewer-субагентов — сейчас один (один «слот» в Quality-департаменте);
- даже при добавлении второго reviewer'а — 2 × 10 часов overhead = 20 часов/нед только на re-review.

Без снижения раундов до 1.5-1.7 — reviewer становится узким горлышком M-OS-1 через 2-3 недели после старта.

## 2. Первопричины (5 повторяющихся — по частоте)

Разбор по ревью-отчётам (primary-source: PR#1, PR#2, Batch A/B/C, interim-audit). Каждая причина — это **паттерн**, не единичный кейс. Частота = число раундов, где этот паттерн был причиной ≥1 замечания P0/P1/P2.

### Причина №1 — литералы секретов в тестах и seeds (частота: 3/8 раундов, ~38%)

**Где зафиксировано:**
- Phase 2 Round 2 BLOCKER-1 — литерал пароля в conftest.
- Phase 3 Batch A step 2 Round 1 P0-2 — то же в test_auth.
- PR#1 Волны 1 pre-commit — `change_me_on_first_login` в seeds.py (P0-2, устранено в round-1).

**Механика:** правило в CLAUDE.md существует с Phase 2, но запоминается только тем backend-dev, кто его уже обжёгся. Новый backend-dev или забывчивый повтор — снова литерал. Reviewer находит, блокирует, round +1.

**Почему сохраняется:** правило декларативно (текст в CLAUDE.md), не автоматизировано. Механическая проверка через grep `secrets.token_urlsafe` / `password\s*=\s*["'].{8,}["']` в staged занимает 0.3 секунды — но её нет.

**Доля в бюджете времени reviewer'a:** 5-10 минут на раунд, где это замечание присутствует, × 3 раунда = 15-30 минут за месяц. Кажется мало, но каждое такое замечание автоматически = +1 раунд.

### Причина №2 — SQL вне репозитория / architectural layer violations (частота: 2/8 раундов, 25%)

**Где зафиксировано:**
- Phase 3 Batch A step 4 — `from sqlalchemy import select` в сервисном слое (P0).
- PR#2 Волны 1 pre-commit — `select` в `deps.py`, `BaseRepository(db)` без специализации с hack `.model = UserModel` (P0-1, P1-2, закрыты в round-1).

**Механика:** ADR 0004 MUST #1a и Amendment 2026-04-18 MUST #1b — два нормативных документа, описывающих слои. backend-dev, реализуя новую ручку, видит: «нужен запрос к `User` по email» — и импортирует `select` прямо там, где он нужен (в auth.py или deps.py). Путь наименьшего сопротивления. Reviewer ловит.

**Почему сохраняется:** статического анализатора на запрет импорта нет. Взгляд backend-dev на чек-лист `departments/backend.md` — это 5 минут, а писать SQL в неправильном слое — 30 секунд. Разрыв стимулов.

**Доля в бюджете:** такое замечание часто P0 → обязательный re-review + часто архитектурный каскад (нужно создать `UserRepository`, вытащить `BaseRepository`). Типичная стоимость — 1.5-2 часа backend-dev на исправление + 20-30 минут re-review. **Один из самых дорогих классов.**

### Причина №3 — IDOR / забытая проверка scope на non-list методах (частота: 2/8 раундов, 25%)

**Где зафиксировано:**
- Phase 3 Batch A step 4 P1-1 — вложенный ресурс HouseConfiguration без `parent_id` check.
- PR#1 Волны 1 round-0 P0-1 — `require_role` без `user_context` в GET-by-id 4 сервисов (project, contract, contractor, payment).

**Механика:** backend-dev реализует `GET /resource/{id}` с `Depends(require_role(ROLE))` и забывает, что `require_role` не фильтрует company_scope — это задача сервисного слоя через `extra_conditions`. В голове «require_role есть — значит безопасно». Это системная иллюзия.

**Почему сохраняется:** декоратор `require_role` и `require_permission` выглядят как полный контракт безопасности — по имени. На деле они отвечают за проверку роли, но не за проверку принадлежности. Разделение ответственности не очевидно из имени.

**Доля в бюджете:** P0/P1 класс, 1-2 часа исправления на одну сущность × 4 сущности = 4-8 часов backend-dev за случай. Самый дорогой класс дефектов.

Отдельно: interim-audit 2026-04-18 нашёл **fail-open в `rbac.py:237-242`** (P1-A) — `can()` возвращает True для `resource.company_id is None`. В текущем коде не эксплуатируется (все модели NOT NULL), но латентный дефект той же природы: защита работает только если все звенья цепочки помнят про scope.

### Причина №4 — ADR 0005 ошибки и ADR 0007 порядок аудита (частота: 2/8 раундов, 25%)

**Где зафиксировано:**
- PR#2 Волны 1 pre-commit P1-3 — аудит записывается **до** физического удаления в `user_roles.py` и `role.py` (два места). Нарушение ADR 0007 «audit после операции».
- PR#2 Волны 1 pre-commit P2-1 — `raise ValueError` вместо `raise DomainValidationError` в `consent.py`. Нарушение ADR 0005 (клиент получит 500 вместо понятного кода).

**Механика:** ADR 0005 и ADR 0007 — нормативные, написаны раньше, чем появился текущий backend-dev. Новый исполнитель читает CLAUDE.md (которая ссылается на ADR), идёт в `docs/adr/0005-api-error-format.md` и `docs/adr/0007-audit-log.md` — но это два дополнительных чтения; в реальном коде backend-dev идёт от интуиции «raise ValueError — понятно же», «аудит раньше — гарантия что он запишется».

**Почему сохраняется:** чек-лист `departments/backend.md` имеет 8 пунктов, но не выделяет ADR 0005/0007 отдельной строкой с конкретным примером «как правильно / как неправильно».

**Доля в бюджете:** обычно P1, 30-60 минут backend-dev на исправление + 10-15 минут re-review.

### Причина №5 — `# type: ignore` / `# noqa` без корректного обоснования (частота: 2/8 раундов, 25%)

**Где зафиксировано:**
- PR#2 Волны 1 pre-commit P1-1 — `# type: ignore[arg-type]` с неверным обоснованием в consent.py.
- PR#2 Волны 1 pre-commit P1-2 — `# type: ignore[assignment]` при динамическом присвоении `.model` (архитектурный запах).
- Phase 3 Batch A — аналогичные минорные случаи (не блокеры, но собирались в P2).

**Механика:** mypy/pyright ругается на type mismatch. backend-dev под time pressure подавляет предупреждение, а не чинит контракт. CLAUDE.md явно запрещает такой паттерн («без комментария-обоснования»), но в реальности даже с комментарием обоснование часто неверное — как в P1-1 PR#2.

**Почему сохраняется:** дисциплинарное правило, механики блокирующей нет. `ruff check` не ловит type-ignore-с-плохим-обоснованием — это задача review-reader'а.

**Доля в бюджете:** P1-P2, 15-30 минут backend-dev.

## 3. Варианты решения (4 варианта)

### Вариант A — «Ничего не менять, наблюдать»

**Суть:** baseline 2.43 раунда фиксируется, реruns не делаем.

**Плюсы:** ноль трудозатрат.

**Минусы:** проблема линейно масштабируется с потоком PR. На M-OS-1 — гарантированное bottleneck.

**Вердикт:** отклонить. Включено для честности сравнения.

---

### Вариант B — «Только Hooks (Phase 0 Варианта B' RFC-004)»

**Суть:** внедряем 5 хуков из `rfc-004-hooks-phase-0-plan.md`. Причины 1 и 2 автоматизируются на 80%+. Причины 3, 4, 5 остаются.

**Эффект:** оценочно `-0.3` раунда к среднему (с 2.43 к ~2.1). По причине 1 автоматическая блокировка = 0 таких замечаний в новых раундах. По причине 2 ruff custom rule — снимает 80% случаев.

**Плюсы:** полностью механическое решение, ноль дисциплины после внедрения. Уже запланировано отдельным RFC.

**Минусы:** не решает причины 3, 4, 5 (50% оставшегося бюджета раундов).

**Вердикт:** **обязательно, но недостаточно**. Рассматриваем как **фундамент**, а не как законченное решение RFC-007.

---

### Вариант C — «Hooks + уточнить чек-лист backend.md + pre-PR self-check»

**Суть:** Вариант B + две дешёвые правки процесса:

1. **Расширить чек-лист `departments/backend.md`** с 8 пунктов до 13, добавив:
   - «IDOR на non-list: GET-by-id / PATCH / DELETE обязаны принимать `user_context` и пробрасывать `extra_conditions` через `get_or_404`. Пример: `service.get(id, user_context=ctx)`.»
   - «Формат ошибок домена: всегда `AppError`-подкласс (`DomainValidationError`, `NotFoundError`). Никогда `raise ValueError` в сервисах.»
   - «Аудит-лог после физической операции: `db.delete(obj)` → `db.flush()` → `audit_svc.log(...)`. НЕ наоборот.»
   - «Вложенные ресурсы: `parent.id == parent_id` проверка в сервисе → 404 при несовпадении.»
   - «`# type: ignore` / `# noqa`: только с **технически корректным** обоснованием в комментарии той же строки. Если не получается обосновать — чинить типизацию.»

2. **Ввести pre-PR self-check для backend-dev** — фикс-лист в `departments/backend.md` «перед открытием PR»:
   - `ruff check backend/app backend/tests` — 0 ошибок;
   - `grep -r "change_me\|password.*=.*\"[a-zA-Z0-9]\{8,\}\"" backend/tests` — 0 матчей;
   - `grep -rn "from sqlalchemy import select\|from sqlalchemy import update\|from sqlalchemy import delete" backend/app/services backend/app/api` — 0 матчей;
   - Все non-list методы (`GET /{id}`, `PATCH /{id}`, `DELETE /{id}`) принимают `user_context`;
   - Все new `raise` — это `AppError`-подкласс;
   - Все `audit.log()` — после `db.flush()`;
   - Все `# type: ignore` имеют осмысленный комментарий-обоснование;
   - Все вложенные ресурсы проверяют parent_id.

    Самопроверка занимает 10-15 минут, документируется в PR description: «pre-PR self-check пройден: ruff 0, grep 0, 8/8 чек-листа».

**Эффект:** оценочно `-0.6` раунда к среднему (с 2.43 к ~1.8). Причины 3, 4, 5 падают с 25% до 10% (не 0% — семантические дефекты всё равно иногда пропускают).

**Плюсы:** дешёвое внедрение (1-2 часа governance-director на правку чек-листа), высокое соотношение эффект/стоимость.

**Минусы:** часть работы переносится с reviewer'a на backend-dev. backend-dev потратит ~15 минут на self-check каждый PR (оценочно +60 минут на PR). В сумме экономия сохраняется: reviewer экономит 30-60 минут за раунд, backend-dev тратит 15 минут — net выигрыш остаётся.

**Вердикт:** **рекомендуемый вариант**. Объединяет механику (Hooks) и дисциплину (self-check + чек-лист).

---

### Вариант D — «Вариант C + обогащённый бриф backend-head и ранее smoke-ревью»

**Суть:** Вариант C + изменение на более ранних этапах:

1. **Обогащение брифа backend-head для backend-dev.** backend-head перед формированием task-брифа прогоняет этот же pre-PR чек-лист **в контексте задачи** (например, при IDOR-реализации — явно пишет в брифе: «не забудь user_context в сервисе; параграф IDOR — вот пример правильного кода»). Это front-loading знаний, снижает вероятность нарушения.

2. **Smoke-ревью после round-0 от самого backend-dev** — формально: ссылка на diff в task-отчёте + 3-минутный ответ «что я перечитал перед открытием PR». Даёт психологический toll против «ну, я же уже написал».

3. **Опционально — ранний консультативный review от reviewer'a в середине PR** (при больших PR >500 строк). reviewer смотрит 10-15 минут, указывает на системные проблемы до того, как они раскатятся по всем файлам.

**Эффект:** оценочно `-0.8` раунда к среднему (с 2.43 к ~1.6). Но с каскадом нагрузки на backend-head и reviewer'a (ранее вовлечение).

**Плюсы:** максимальное снижение числа раундов.

**Минусы:** значительная сложность внедрения. backend-head — уже bottleneck (один агент, два параллельных backend-dev). Добавление этапа brief-enrichment × каждая задача — это +15-30 минут backend-head на задачу. На большом потоке убивает его производительность. Early smoke-review от reviewer'a вводит двойной review — реально спорно, может наоборот тормозить.

**Вердикт:** **перебор для текущего масштаба**. Возврат к Варианту D — при росте потока до 10+ PR/нед **и** если Вариант C не добился ≤1.7.

## 4. Trade-offs (сравнение)

| Критерий | A (ничего) | B (Hooks) | C (Hooks + self-check) | D (Hooks + self-check + brief + early review) |
|---|---|---|---|---|
| Ожидаемое среднее раундов | 2.43 | ~2.1 | ~1.8 | ~1.6 |
| Вложение governance-director | 0 | 0 | 1-2 ч | 4-6 ч |
| Вложение backend-director | 0 | 0 | 1 ч | 3-4 ч (редизайн брифов) |
| Overhead backend-dev на PR | 0 | 0 | +15 мин | +20-30 мин |
| Overhead backend-head на задачу | 0 | 0 | 0 | +15-30 мин |
| Overhead reviewer на PR | −30-60 мин/раунд | −60 мин | −90 мин | −90 мин + возможный early-review +15 мин |
| Риск ложно-положительных блокировок | 0 | средний (настраивается H-1..H-5) | низкий | низкий |
| Возвратность (откат при неудаче) | — | 1 мин | 1 мин (удалить строки в чек-листе) | 1 день (редизайн брифов обратно) |
| Готовность к M-OS-1 (10+ PR/нед) | нет | частично | да | да |

**Пересечение с RFC-004 (Orchestration Layer):** Вариант D частично пересекается с планируемым автозапуском reviewer через hook SubagentStop (H-9 deferred). Реализация D без Orchestration Layer — ручная, с Orchestration Layer — автоматическая. Поэтому C — оптимум **до** появления Orchestration Layer; D — candidate **после**.

## 5. Рекомендация

**Вариант C — adopt без пилота как самостоятельного RFC.**

Обоснование:
1. **Pilot не требуется.** Каждый компонент Варианта C имеет самостоятельный формат проверки: Hooks уже пилотятся отдельно (RFC-004 phase-0-plan); правка чек-листа — это обновление документа, проверяемое на следующих 4-5 PR; pre-PR self-check — это изменение регламента backend-dev, метрика — естественный baseline «раунды/PR».
2. **Вложение минимальное.** Governance-director 1-2 часа + backend-director 1 час. Это меньше стоимости одного сэкономленного раунда (2-4 часа).
3. **Реверсируемо.** Если через месяц среднее не упадёт до ≤1.7 — откатываем правку чек-листа (или оставляем и переходим к D). Никаких технических долгов не накапливается.

### Последовательность действий (если Координатор примет рекомендацию)

1. **Phase 0 Hooks** (отдельный пилот, 2 дня) — см. `rfc-004-hooks-phase-0-plan.md`. **Предусловие**, без hooks часть эффекта RFC-007 недостижима.
2. **Правка `departments/backend.md` от governance-director** — 1-2 часа, через стандартный governance-request процесс. Вердикт утверждает quality-director.
3. **Правка `departments/quality.md` reviewer'а** — reviewer получает явное указание: **если** backend-dev в PR description указал «pre-PR self-check пройден, grep 0, ruff 0», **то** round-0 начинается с перепроверки только выборочных пунктов (spot-check), а не полной проверки. Сокращает время round-0 на 15-20 минут.
4. **Измерение** — следующие 4-5 PR M-OS-1 фиксируются в `docs/reviews/` как обычно; ri-director через месяц собирает статистику «раундов на PR» и сравнивает с baseline 2.43.
5. **Решение о Варианте D** — принимается по результатам измерения. Если среднее ≤1.7 — stop at C. Если >1.7 — запускаем RFC-007 v2 с анализом Варианта D.

### Что НЕ делаем в рамках RFC-007

- Не меняем роль reviewer'а (quality-director отдел работает как есть, только снижается нагрузка).
- Не вводим новых субагентов.
- Не трогаем CI/CD (это отдельный пилот RFC-001 Claude Code Routines + существующий `ci.yml`).
- Не делаем редизайн брифов backend-head (это Вариант D, отложен).

### Метрика успеха

Через 30 дней после внедрения Варианта C:
- **Среднее раундов на PR ≤1.7** (снижение с 2.43 на ≥0.7). Измеряется по `docs/reviews/*` за период.
- **Доля PR с approve в round-0 ≥40%** (с ~0% до ≥40%). Измеряется там же.
- **Ни одного повторного P0 из Причины 1 (литералы секретов)** после установки H-1 — это жёсткий тест, что Hooks работают.

Если цель не достигнута, возврат к RFC с Вариантом D.

## 6. Риски и ограничения

**Риск 1: pre-PR self-check превращается в формальность.** backend-dev пишет «пройден» без реальной проверки.

Митигация: reviewer в round-0 делает spot-check одного пункта из 8 (выбирается случайно); при явном ложно указанном «пройден» — публичное замечание и escalation в quality-director. Через 2-3 случая дисциплина восстанавливается.

**Риск 2: чек-лист растёт со временем и становится неуправляемым.** Сейчас 13 пунктов, через 6 месяцев — 30.

Митигация: правило «новое правило добавляется в чек-лист только после 2+ рецидивов одного класса дефекта», зафиксировано в governance.md. Каждые 90 дней — ревизия чек-листа с удалением правил, которые не срабатывают (уже автоматизированы или стали культурой).

**Риск 3: Вариант C не покрывает категорию «dropped requirements из брифа»** (см. interim-audit Б-02: было 2 случая в Batch C).

Признано. Эта категория — тема отдельного RFC о DoD-трассировке (R-03 из brief governance-state-2026-04-17.md). RFC-007 её не решает сознательно — другая природа причины.

**Риск 4: реальное поведение reviewer'а может отличаться от заявленного.** В Варианте C мы предполагаем, что reviewer «доверяет» pre-PR self-check и делает spot-check. На практике reviewer может продолжить полную проверку.

Митигация: изменение `departments/quality.md` формулируется явно — «при явном указании pre-PR self-check пройден reviewer **рекомендуется** spot-check, **не обязан**». Это опция backend-dev'а быстрее закрыть round-0, не навязанный сценарий.
