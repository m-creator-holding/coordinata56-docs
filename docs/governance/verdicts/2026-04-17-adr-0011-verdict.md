# Вердикт governance-комиссии: ADR 0011 — Foundation Multi-company, RBAC, Crypto Audit

- **Дата:** 2026-04-17
- **Документ:** `docs/adr/0011-foundation-multi-company-rbac-audit.md`
- **Автор ADR:** Архитектор (субагент `architect`)
- **Вердикт:** **approve с замечаниями** (5 замечаний уровня W, 0 блокеров)
- **Председатель:** governance-director
- **Статус аудитора:** отчёт аудитора не получен (предварительный вердикт, финальный — после отчёта)

---

## Проверка по 5 фокусам Координатора

### 1. ADR не ломает иерархию L0-L4, цепочку делегирования, регламенты

**Результат: OK.**

ADR 0011 затрагивает исключительно архитектуру данных и авторизации (Company, UserCompanyRole, RolePermission, AuditLog). Иерархия субагентов L0-L4 (CODE_OF_LAWS Книга I, статьи 9-13), цепочка делегирования (CLAUDE.md, v1.4 §5, v1.6) и регламенты по уровням (coordinator.md, director.md, head.md, worker.md) не затрагиваются.

ADR корректно помещает Company в `backend/app/core/master_data/` — это общее ядро по ADR 0009 §2. Модель UserCompanyRole описывает роли сотрудников холдинга, не субагентов.

### 2. ADR достаточно конкретен для декомпозиции backend-director

**Результат: OK с замечанием W-01.**

ADR содержит:
- Схемы всех 3 новых таблиц (Company, UserCompanyRole, RolePermission) с типами данных, ограничениями, индексами
- Схемы изменений 4 существующих таблиц (projects, contracts, contractors, payments) + 2 поля в AuditLog + 5 полей в Contract
- Алгоритм SHA-256 с конкатенацией через `"|"` и порядком полей
- Сигнатуру `can(user_context, action, resource)` с пошаговой логикой
- 4 шага реализации с зависимостями между ними
- Seed-скрипт миграции данных (4 пункта)
- Backfill-скрипт аудита

Backend-director может декомпозировать на задачи уровня Head/Worker.

**W-01:** Не указан формат `resource` — какие поля кроме `company_id` и `pod_id` содержит объект? Рекомендация: добавить минимальную Pydantic-схему ResourceContext.

### 3. Порядок реализации (multi-company -> RBAC -> crypto audit) корректен

**Результат: OK.**

Порядок соответствует зависимостям, зафиксированным в m-os-vision.md §11:
- RBAC зависит от multi-company (проверка company_id)
- Crypto audit покрывает оба предыдущих изменения
- Contract legal (шаг 3a) корректно параллелен шагу 3

Совпадает с m-os-vision.md §9: «Multi-company + fine-grained rights (одно должно быть до второго)» и «Аудит-цепочка -> только потом сложные процессы».

### 4. Scope не расплылся? Только Foundation, без M-OS-2 фич

**Результат: OK с замечаниями W-02, W-03.**

Раздел «Что явно не входит» перечисляет 8 исключений. Scope ограничен data model + service contracts + один endpoint верификации.

**W-02:** Поля `is_internal` и `counterparty_company_id` в Contract (§1.4) описываются как маркеры для «будущей консолидированной отчётности (elimination внутренних оборотов, ADR 0008 §8.9)». m-os-vision.md §8.9 отнесён к блоку «в ходе осмысления». Добавление полей в миграцию Foundation — допустимо (nullable, без бизнес-логики), но ADR должен явно зафиксировать: бизнес-логика консолидации НЕ входит в этот ADR. Формулировка «на M-OS-1 поля добавляются в модель» — присутствует, достаточна. Замечание минорное.

**W-03:** `file_id` (§4) ссылается на несуществующую таблицу files. ADR предлагает `DEFERRABLE INITIALLY DEFERRED` или TODO-комментарий. Для чистоты: FK на несуществующую таблицу не может быть DEFERRABLE — таблицы просто нет. Рекомендация: убрать FK, оставить поле как `int | None` с CHECK-комментарием. FK добавить при создании таблицы files (отдельный ADR).

### 5. Configuration-as-data (принцип 10) реализован для RBAC

**Результат: OK.**

Матрица прав хранится в таблице `role_permissions` (§2.2), не в коде. Начальные значения загружаются seed-скриптом. Редактирование — через будущий admin-UI. ADR явно ссылается на принцип 10 ADR 0008. Функция `can()` читает данные из таблицы, не из enum.

## Дополнительные замечания

**W-04:** ADR 0011 ссылается на «docs/legal/phase-3-legal-check.md» — фактический путь `docs/legal/phase-3-legal-check.md` — проверить при коммите, что путь совпадает. Проверено: файл существует по указанному пути.

**W-05:** Суперадмин (§1.3) — `is_holding_owner: bool` описан как «специальный флаг», но не указано, где именно он хранится. В JWT (§2.4) он есть. Предполагаемая логика: при login проверяется наличие UserCompanyRole с company_id=NULL (или иной маркер). Рекомендация: добавить явное поле `is_holding_owner: bool` в таблицу users или определить правило: «holding_owner = пользователь с role_template=owner и pod_id=NULL и company_id=NULL в UserCompanyRole». Без этого уточнения backend-dev будет додумывать.

## Итоговый вердикт

**APPROVE** — ADR 0011 принимается для реализации в M-OS-1 Спринт 1 Foundation Critical.

Замечания W-01..W-05 — рекомендательные, не блокирующие. Backend-director при декомпозиции должен учесть W-01 (ResourceContext schema), W-03 (file_id без FK), W-05 (is_holding_owner storage).

**Условие:** финальное утверждение — после получения отчёта governance-auditor. Если аудитор не выявит блокеров — вердикт становится финальным без повторного рассмотрения.
