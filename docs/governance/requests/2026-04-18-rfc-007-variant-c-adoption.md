## Что меняется

### Amendment A — `docs/agents/departments/backend.md`: 5 пунктов пред-PR чек-листа

Добавить в раздел «Чек-лист самопроверки backend-dev (перед сдачей)» пять явных уточнений — привязанных к конкретным ADR и к повторяющимся замечаниям из ревью. Существующие пункты чек-листа этим не отменяются; уточнения делают их операциональными (можно механически проверить одной командой или одним grep'ом).

**Пункт A.1 — литералы секретов.**

> Проверить, что в diff нет литералов секретов (пароли, токены, bearer, JWT). Все креды через `secrets.token_urlsafe()` в тестах, `os.environ.get()` в коде.

*Обоснование:* существующий пункт «Никаких литералов секретов» декларативен. Уточнение добавляет конкретный способ проверки (diff + запрещённые префиксы) и явно перечисляет классы секретов. Соответствует правилу CLAUDE.md раздел «Секреты и тесты». Частота дефекта по RFC-007 §2 причина №1 — 38% раундов.

**Пункт A.2 — SQL только через репозитории.**

> Проверить, что все SQL в сервисах — через репозитории. Никаких `db.execute(select(...))` в сервисах или `api/deps.py`. Паттерн ADR 0004 MUST #1a.

*Обоснование:* ADR 0004 MUST #1a после Amendment 2026-04-18 чётко разделяет запрещённые операции (запросы) и разрешённые (предикаты `ColumnElement[bool]`). Пункт формулирует запрет с явным перечислением запрещённых вызовов и явной ссылкой на ADR. Частота по RFC-007 §2 причина №2 — 25%.

**Пункт A.3 — RBAC на write-эндпоинтах.**

> Для всех write-эндпоинтов (POST/PATCH/DELETE) проверить: использован ли `require_permission` с `user_context`, пробрасывается ли `extra_conditions` в сервис. ADR 0004 MUST #1b.

*Обоснование:* ADR 0011 §2.4 заменил `require_role` на `require_permission`; `user_context` строится декоратором и используется в `CompanyScopedService._scoped_query_conditions` (ADR 0011 §1.3). Текущий чек-лист упоминает только «`require_role(...)` с правильными ролями» — устаревшая формулировка. Уточнение приводит чек-лист в соответствие с post-Wave-1 каноном.

**Пункт A.4 — формат ошибок и пагинации.**

> Для всех новых ошибок использовать формат ADR 0005 `{error: {code, message, details}}`. Для пагинации — envelope ADR 0006 `{items, total, offset, limit}`.

*Обоснование:* CLAUDE.md раздел «API» требует оба формата, но в текущем чек-листе backend-dev этого нет (есть только у reviewer в quality.md). Дублирование одной строкой делает проверку self-service.

**Пункт A.5 — аудит-лог write-операций.**

> Для всех write-операций вызван ли `audit_service.log()` в той же транзакции. ADR 0007.

*Обоснование:* существующий пункт «Все ли write-эндпоинты имеют `audit_service.log()`?» нужно уточнить про «в той же транзакции» — иначе можно вызвать логирование после `commit()`, что нарушает atomic-гарантию ADR 0007.

**Место вставки:** раздел «Чек-лист самопроверки backend-dev (перед сдачей)» в `departments/backend.md`, как расширение/уточнение существующих пунктов. Предпочтительно — развести: блок «ADR-gate» (A.1-A.5) отдельно от общих пунктов (`ruff`, `pytest`, `git status`).

### Amendment B — `docs/agents/departments/quality.md`: принцип доверия с spot-check

Добавить в раздел «Правила работы (выросшие из ошибок)» новый пункт:

> Если backend-dev в отчёте PR заявил self-check по списку (1-5), reviewer делает spot-check (2-3 случайных пункта), не полный повторный прогон. Цель — доверие с выборочной проверкой, не тотальный контроль.

*Обоснование:* сейчас reviewer по факту прогоняет весь чек-лист backend.md повторно, дублируя работу backend-dev. При наличии явного self-check-отчёта в PR это избыточно — каждый пункт проверяется дважды. Выборочная проверка 2-3 пунктов из 5 при каждом PR даёт статистически достаточное давление: если backend-dev начнёт халтурить, через 2-3 PR reviewer поймает расхождение и переведёт конкретного dev'а в «полный прогон». Это снижает стоимость round-0 на 10-20 минут при сохранении контроля.

**Место вставки:** раздел «Правила работы (выросшие из ошибок)», новый пункт 11 перед «Стандарты качества тестов». Параллельно — обновить чек-лист reviewer («Чек-лист reviewer (CRUD-эндпоинт)»): добавить заголовок «Spot-check (2-3 случайных пункта) если PR содержит self-check-отчёт; полный прогон если self-check-отчёта нет».

Добавить в раздел «Правила работы (выросшие из ошибок)» новый пункт:

> Перед стартом крупного PR Директор обязан сверить скоуп в формулировке Координатора с каноном (ADR + предыдущие PR-брифы); расхождения эскалируются до начала работы.

*Обоснование:* повторяющийся паттерн за 2026-04-17/18 — Координатор формулирует бриф по памяти, Директор принимает без сверки, на ревью выясняется что какие-то требования из ADR 0011 / 0013 / Amendment 2026-04-18 не были учтены в скоупе. Это не ошибка кода, а ошибка формулировки, и ловится она только в раунде ревью. Пункт переносит сверку на этап брифинга (по паттерну Координатор-транспорт v1.6 — Директор и так должен отказать при неверной формулировке, но сейчас это правило про роли, а не про скоуп).

**Место вставки:** раздел «Правила работы (выросшие из ошибок)» `departments/backend.md`, новый пункт 11 (после existing 1-10). Формулировка явно опирается на паттерн Координатор-транспорт v1.6 CLAUDE.md.

## Почему сейчас

1. **RFC-007 принят Владельцем в части Варианта C** (Координатор уведомлён). Реализация RFC на уровне регламента = внесение правок этой заявкой.
2. **M-OS-1 стартует в ближайшие дни** с 6 параллельными Директорами и ~15-20 PR/нед. Без снижения раундов до 1.5-1.7 reviewer становится узким горлышком (RFC-007 §1.3).
3. **Baseline 2.43 раунда** — цифра собрана ri-director по `docs/reviews/` за последний месяц; не гипотетика, а факт. Достаточно для обоснования правки регламента.

## Риск-оценка

- **Низкий риск для Amendment A:** уточнения существующих пунктов, прямые ссылки на ADR, импакт на код = 0.
- **Средний риск для Amendment B (spot-check):** ослабление контроля. Митигация: требование «если self-check-отчёта нет → полный прогон» + обязательный triple-check хотя бы раз в неделю на случайном PR (eyeball validation). Если через 2-4 недели метрики качества (% P0 в production-gate) ухудшатся — откат.

### Пункт A (Amendment в `backend.md` — 5 пунктов пред-PR чек-листа)

**Вердикт:** ACCEPT.

**Обоснование:**

- A.1 (литералы секретов) — конкретизирует правило CLAUDE.md раздел «Секреты и тесты», которое уже нормативно; добавление способа проверки не вводит нового правила, только операционализирует существующее. Подтверждается частотой дефекта (3/8 раундов, RFC-007).
- A.2 (SQL через репозитории) — цитирует ADR 0004 MUST #1a дословно; Amendment 2026-04-18 (ratify в CHANGELOG `2026-04-18`) разделил #1a/#1b, эта формулировка консистентна с обоими. Путь escape для легитимных случаев — `extra_conditions` — уже зафиксирован в backend.md v1.2 (правило №1, финальный абзац). Дубля не создаёт.
- A.3 (RBAC с `require_permission` + `user_context`) — **критически важно**: текущий чек-лист backend.md (строка 116) всё ещё упоминает `require_role(...)`, что устарело после ADR 0011 §2.4. Без этой правки backend-dev, читающий чек-лист буквально, применит `require_role` вместо `require_permission` — и ревью опять поймает. Правка устраняет противоречие между чек-листом и ADR 0011, а не создаёт новое правило. Приоритет коллизий governance.md: ADR (уровень 5) vs departments/*.md (уровень 4) — formally `departments` выше, но в данном случае ADR 0011 одобрен Владельцем (Telegram msg 1005 по Sync-2, далее применён в PR#2), а текущая строка чек-листа — bootstrap-редакция от 2026-04-15, до ADR 0011. Правка — синхронизация.
- A.4 (ADR 0005/0006) — дублирует правило из CLAUDE.md раздел «API» в операционный чек-лист. Не расширяет, только делает self-service. Acceptable (дублирование локальное, одна строка).
- A.5 (аудит в той же транзакции) — уточняет существующий пункт (строка 115), приводит в соответствие с ADR 0007. Без правки возможен интерпретационный разрыв («вызван» ≠ «в той же транзакции»).

**Уточнение при внесении:** оформить как новый явный блок «ADR-gate (перед коммитом)» **внутри** существующего чек-листа самопроверки, с подзаголовком и перечислением A.1-A.5 со ссылками на ADR. Не вынос в отдельный раздел — чтобы backend-dev видел весь чек-лист в одном месте. Существующие общие пункты (ruff, pytest, git status, миграции) сохраняются.

**Импакт на другие документы:** минимальный. `~/.claude/agents/backend-dev.md` может содержать устаревшие ссылки на `require_role` — проверить отдельным поручением `governance-auditor` в следующую сессию.

### Пункт B (Amendment в `quality.md` — spot-check при наличии self-check-отчёта)

**Вердикт:** ACCEPT WITH CHANGES.

**Обоснование:**

Сам принцип (доверие + выборочная проверка) обоснован через экономию времени reviewer (~10-20 мин/PR) и через статистическое давление (халтура ловится за 2-3 PR). Риск — реальное ослабление контроля. Два уточнения, без которых ACCEPT невозможен:

**Изменение B.1 — критерий «валидного self-check-отчёта».** В формулировку добавить: self-check-отчёт считается валидным, если (а) содержит явный список пунктов A.1-A.5 с галочкой pass/fail для каждого; (б) содержит ссылку на конкретный артефакт доказательства (grep-вывод, diff-строки, тест-id). Отсутствие артефактов = отчёт невалиден = полный прогон. Это закрывает атаку «backend-dev ставит 5 галочек не глядя».

**Изменение B.2 — обязательный random-full-audit.** Добавить: раз в неделю reviewer обязан по случайному PR выполнить **полный** прогон чек-листа независимо от наличия self-check-отчёта, чтобы получить baseline уровня корреляции между self-check и реальным состоянием кода. Результаты публикуются в `docs/governance/audits/weekly/spot-check-calibration-YYYY-MM-DD.md`. Если корреляция падает (≥2 расхождения за неделю) — принцип spot-check временно отключается и reviewer возвращается к полному прогону.

**Место вставки (финальное):** новый пункт 11 в разделе «Правила работы» `quality.md`, формулировка:

> Если PR содержит валидный self-check-отчёт backend-dev по пунктам A.1-A.5 (подтверждено артефактами), reviewer делает spot-check — 2-3 случайных пункта плюс полный проход остальных (форматы ошибок, RBAC-матрица, IDOR). Раз в неделю reviewer делает полный прогон на случайном PR для калибровки (отчёт — `docs/governance/audits/weekly/spot-check-calibration-YYYY-MM-DD.md`). При ≥2 расхождениях за неделю — режим отключается до восстановления корреляции.

Плюс строка в чек-листе reviewer: «Проверить наличие self-check-отчёта backend-dev; если есть — spot-check 2-3 пункта; если нет — полный прогон».

**Риск-монитор:** если через 4 недели (2026-05-16) метрика «P0 в production-gate» остаётся 0 и метрика «% approve round-0» выросла — подтверждение эффективности. Если P0 проскочил — немедленный откат и retro.

### Пункт C (Amendment в `backend.md` — сверка скоупа Директором)

**Вердикт:** ACCEPT.

**Обоснование:**

Формализует практику, уже требуемую `regulations_addendum_v1.6.md` (Координатор-транспорт). В v1.6 Директор обязан отказать при неверной формулировке («сделай <work>» vs «напиши бриф для <role>») — но правило касается **роли**, а не **скоупа**. Пропуск по скоупу (скажем, Координатор забыл упомянуть ADR 0011 §2.4) не покрывается v1.6, а на ревью ловится P1/P0. Новая формулировка закрывает эту дыру.

Формулировка из заявки корректна. Предлагаю добавить одно уточнение (инкорпорировать в текст):

> Перед стартом крупного PR (оценка Директора: ≥3 модели / ≥5 эндпоинтов / затрагивает RBAC/audit/миграции) Директор обязан сверить скоуп в формулировке Координатора с каноном (ADR + предыдущие PR-брифы за 7 дней + релевантные разделы CLAUDE.md). Расхождения эскалируются Координатору до начала работы Head/Worker. Сверка фиксируется строкой в PR-брифе «Scope-vs-ADR: verified | gaps: <list>».

Это добавляет: (1) триггер «крупный PR» (не каждая XS-задача), (2) окно предыдущих брифов (чтобы не тянуть весь архив), (3) явный артефакт (строка в брифе), чтобы проверка была видима в артефактах, а не декларативна.

**Место вставки:** раздел «Правила работы (выросшие из ошибок)» `backend.md`, новый пункт 11 (после существующих 1-10), до раздела «Правила для авторов миграций».
