**Статус:** pending (ожидает ADR 0009 + ADR 0010 + аудит)

## 1. Контекст

Владелец принял стратегическое решение переименовать проект в **M-OS (M Operating System)** — единое внутреннее AI-native ПО всего холдинга. Проект `coordinata56` становится первым pod-ом (`cottage-platform-pod`), а не самодостаточной ERP. Подробно — `docs/m-os-vision.md` (820 строк, 10 принципов, roadmap на 5 фаз).

Три документа задают новую рамку:
- **ADR 0008** — определение M-OS и её границ (статус: утверждён Владельцем по контексту стратегического разворота)
- **ADR 0009** — принципы pod-архитектуры: общее ядро, data contracts, event bus (в работе у architect, черновик ожидается в рамках M-OS-0)
- **ADR 0010** — таксономия субагентов: 4 типа — `core_department` / `domain_pod` / `governance` / `advisory` (в работе у architect)

Текущий Свод законов (`CODE_OF_LAWS.md` v1.0 от 2026-04-15) описывает `coordinata56` как «ERP коттеджного посёлка» (ст. 1) и иерархию субагентов без разделения на поды/ядро. Без правки Свод начнёт расходиться с реальностью уже в следующем коммите, и это создаст системное противоречие уровня 2 ↔ документы нижних уровней.

**Критерий готовности M-OS-0** (из vision §9): любой новый член команды (человек или субагент), прочитав Свод и vision, понимает что такое M-OS и как к ней подходить.

## 4. Ожидаемые последствия

### Положительные
- Единая терминологическая рамка для всех будущих разработок (M-OS, pod, общее ядро, таксономия субагентов)
- Новые поды (АЗС, металл, МКД, карьер) добавляются по шаблону без пересмотра архитектуры
- При добавлении нового pod'а активируются его субагенты (через таксономию ADR 0010) без создания с нуля
- Документация становится масштабируемой: каждый pod имеет свою папку, общее M-OS-wide — в корне `docs/`
- `CODE_OF_LAWS.md` перестаёт «врать» про scope проекта

### Отрицательные
- Крупная правка в один момент времени = высокий риск локальных неконсистентностей между 48 файлами агентов и yaml-картой
- Mermaid-диаграммы переделываются полностью, потребуется визуальная сверка
- Публичное зеркало `coordinata56-docs` должно быть синхронизировано в одно окно, иначе внешние консультанты видят расхождение

### Риски
- **Риск 1** (средний): ADR 0009 и 0010 будут доработаны быстрее, чем комиссия успеет их рассмотреть. Контрмера: параллельный review (см. §6).
- **Риск 2** (высокий): миграция `docs/phases/` → `docs/pods/cottage-platform/phases/` ломает ссылки в регламентах, Memory, past-коммитах. Контрмера: перед git mv — проход `grep -r "docs/phases/" docs/ ~/.claude/` и массовый sed-replace; после — smoke-check ссылок.
- **Риск 3** (средний): bulk-правка 48 файлов `~/.claude/agents/*.md` может повредить frontmatter YAML. Контрмера: добавление `taxonomy:` делается скриптом с yaml-парсером, не sed'ом; reviewer проверяет все 48 файлов валидаторным запуском.
- **Риск 4** (низкий): Батч C Phase 3 внутри coordinata56 параллелится с M-OS-0. Контрмера: M-OS-0 не трогает код, только docs + agents — пересечения нет.

## 5. План rollback

Если после внедрения обнаруживаем системный дефект (противоречие, которое нельзя исправить быстрой правкой):

**Откатываемое:**
1. Все `git mv` обратимы через `git mv` в обратную сторону — структура `docs/phases/` восстанавливается.
2. Коммит M-OS-0 — атомарный единый коммит (не по кускам): `git revert <hash>` откатывает всё разом.
3. Публичное зеркало синхронизируется ПОСЛЕ коммита в основном репо — если откатываем основной, зеркало не успело уйти вперёд.
4. `~/.claude/agents/*.md` — бэкап папки перед bulk-правкой: `cp -r ~/.claude/agents ~/.claude/agents.pre-m-os-0.bak`.

**Неоткатываемое** (и контрмеры):
- ADR 0008 уже утверждён — откат невозможен без нового решения Владельца. Если M-OS-0 откатываем, ADR 0008 остаётся как «принято, временно отложено внедрение».
- Разговор с Владельцем уже произошёл — репутационно неоткатываемо. Но это не блокер, это часть стратегического разворота.

**Критерий срочного rollback:**
- Батч C Phase 3 не проходит ревью из-за перемещения файлов (пути в импортах, фикстурах, скриптах) → немедленный revert миграции документации, ADR 0008/0009/0010 остаются, но `docs/pods/` не создаётся.
- Комиссия обнаруживает противоречие Свод ↔ ADR, которое нельзя исправить в рамках M-OS-0 → частичный rollback Книги VII Свода, ADR остаются в статусе «принято, внедрение отложено».

## 6. Состав комиссии и процедура утверждения ADR 0009 и 0010

### Обязательные участники (по текущему регламенту)

| Роль | Вход в комиссию | Основание |
|---|---|---|
| **governance-director** (председатель) | обязательно | `departments/governance.md` §«Процесс изменения регламента» |
| **governance-auditor** | обязательно | тот же §, пункт «Анализ» |
| **architect** (Советник) | обязательно | автор черновика ADR 0009/0010; ADR по определению в его компетенции |
| **Координатор** | обязательно | подписант финального approve (по CODE_OF_LAWS ст. 15, владеет метриками и памятью) |

### Опциональные участники (подключаются если затронуто их направление)

| Роль | Когда подключать |
|---|---|
| **backend-director** | ADR 0009 затрагивает структуру `backend/app/` (shared core vs pod-specific) |
| **quality-director** | ADR 0010 меняет правила ревью — нужна сверка с `departments/quality.md` |
| **frontend-director** / **design-director** | если ADR 0009 задаёт принципы композиции UI по подам |
| **infra-director** | если ADR 0009 затрагивает deployment topology (один процесс vs несколько) |
| **analyst** (Советник) | если таксономия ADR 0010 влияет на метрики и отчётность |

### Процедура review — параллельно, с синхронизацией

**Этап 1 (1 день):** architect публикует черновики ADR 0009 и 0010 одновременно в `docs/adr/` со статусом `draft`.

**Этап 2 (1-2 дня, параллельно):**
- governance-auditor проводит аудит обоих ADR на противоречия с текущими регламентами и ADR 0001-0007
- backend-director ревьюит ADR 0009 на предмет структурных следствий в коде
- quality-director ревьюит ADR 0010 на предмет следствий для правил ревью
- Координатор ревьюит оба ADR как «понятно ли новому члену команды»

Каждый ревьюер отдаёт либо `approve`, либо `request-changes` с конкретными замечаниями — письменно, в `docs/governance/reviews/2026-04-XX-adr-0009.md` и аналогично для 0010.

**Этап 3 (1 день):** governance-director сводит замечания, выносит вердикт:
- Все `approve` → ADR утверждается, запись в `CHANGELOG.md`
- Есть `request-changes` → возврат architect'у с консолидированным списком; новый цикл

**Этап 4 (непрерывно после utvержения ADR):** правки во вторичных документах (Свод, agents-map, diagrams, 48 файлов агентов) выполняются в рамках одного коммита — см. §7.

### Критерий утверждения

**Консенсус обязательных участников** (governance-director, governance-auditor, architect, Координатор) — 4 из 4 `approve`.

**По опциональным участникам** — большинство `approve` среди тех, кого подключили. Один `request-changes` от опционального не блокирует, но обязателен письменный ответ: либо правка, либо обоснованный отказ в `docs/governance/reviews/`.

**Почему консенсус, не большинство на обязательных:** ADR 0009 и 0010 — фундаментальные, их противоречие одному из 4 обязательных = системный риск. Лучше задержаться на раунд правок, чем протащить.

### SLA

По `departments/governance.md` §«SLA комиссии» — вердикт ≤ 2 рабочих дня с момента получения черновика. Для M-OS-0 это умещается в окно фазы (5-7 дней по vision, крайний лимит 10-14 дней).

## 8. Оценка риска срыва M-OS-0

### 8.1 Где наиболее критичная точка

**ADR 0010 (таксономия)** — самая рисковая часть, поскольку:
1. Мэппит 48 существующих файлов на 4 типа — любая ошибка классификации тянется в yaml, диаграммы, Свод.
2. Задаёт правила активации новых субагентов при добавлении нового pod'а — ошибка модели означает пересмотр когда будет второй pod.
3. Одновременно влияет на `agents-map.yaml`, `agents-diagrams.md`, 48 frontmatter'ов — большая bulk-правка.

**Менее рисковая:** ADR 0009 (pod-принципы) — в основном теория + ссылки на `m-os-vision.md §3`, коснётся кода только в Фазе M-OS-1.

**Наименее рисковая:** правка CODE_OF_LAWS — редактируется точечно, шаблон правок уже зафиксирован в §7.2 выше.

### 8.3 Как минимизировать

1. **Атомарный коммит** — все правки M-OS-0 в один git-commit, один rollback-прыжок.
2. **Dry-run bulk-правок** — сначала на 2-3 файлах, потом на всех 48.
3. **Pre-review ADR 0010** architect + governance-auditor + Координатор до публикации.
4. **Backup-snapshot** перед началом: копия `~/.claude/agents/`, `docs/phases/`, `docs/agents/`, `docs/stories/`.
5. **Публичное зеркало — в последнюю очередь**, после merge основного.
