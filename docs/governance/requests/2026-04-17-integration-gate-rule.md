**## Раздел V. Правило интеграционного шлюза**

**Статья 45а. Запрет живых интеграций до production-gate**

Живые внешние интеграции — реальные сетевые вызовы к сторонним системам (банки, Росреестр, 1С, ОФД, страховые и аналогичные) — запрещены до прохождения production-gate проекта.

До production-gate допускается:
- разработка кода адаптеров в режиме `enabled=False`;
- тестирование адаптеров на mock-данных в dev-среде.

Исключение в рамках M-OS-1: Telegram-бот. Telegram не работает с денежными транзакциями и персональными данными, поэтому разрешён как единственная живая интеграция до production-gate.

Критерии production-gate определяются в операционном плане соответствующей фазы.

**Статья 45б. Обязательный доклад Владельцу о предложениях новых интеграций**

Любое предложение о подключении стороннего сервиса (в том числе переводе существующего адаптера из `enabled=False` в `enabled=True`), поступившее от любого субагента, подлежит обязательному докладу Владельцу через Координатора.

Формат доклада (обязательный):
1. **Кто предложил** — роль и имя субагента
2. **Какой сервис** — наименование, поставщик, тип (финансовый/государственный/прочее)
3. **Зачем** — бизнес-обоснование: какую задачу решает, что не работает без него
4. **Стоимость и риски** — ожидаемая стоимость использования, правовые риски (ПДн, лицензирование), технические риски (зависимость, SLA поставщика)
5. **Рекомендация субагента** — approve / reject / отложить, с обоснованием

Доклад направляется Координатором Владельцу до начала любых технических работ по интеграции.

## 2. Почему (основание)

Прямое решение Владельца (Мартин, Telegram msg 1094, 2026-04-17):

> «Первая и единственная живая интеграция в M-OS-1 — Telegram. Никакие другие внешние сервисы (банки, Росреестр, 1С, ОФД, страховые) не подключаются к живым системам до production-gate. Можно писать код адаптеров "на полке" с флагом `enabled=False`, но не активировать. Исключение — Telegram: лёгкий сервис без денег/ПДн.»

Дополнительное решение Владельца (msg 1103, 2026-04-17):

> «Если любой субагент предлагает подключить сторонний сервис — это обязательно докладывается Владельцу через Координатора в формате: кто предложил / какой сервис / зачем / стоимость и риски / рекомендация субагента.»

Без закрепления в регламенте это правило существует только в памяти и плане фазы — субагенты, не загружающие m-os-1-plan.md, его не увидят.

## 4. Рекомендация governance-director

**Вердикт: APPROVE.**

Правило прямо следует из решения Владельца. Оно закрывает реальный пробел: без нормы в CODE_OF_LAWS субагент-исполнитель технически обязан следить за активными ADR и планом фазы, но не обязан читать Telegram-переписку. Норма на уровне CODE_OF_LAWS делает правило универсально обязательным.

Правило исключения для Telegram сформулировано достаточно конкретно (нет денежных транзакций, нет ПДн) — не создаёт прецедента для расширительного толкования.

Формат доклада по ст. 45б (5 пунктов) воспроизводит формат, прямо продиктованный Владельцем (msg 1103) — изменений не вносится.
