## Изменения по пунктам

### Пункт 1 — ADR Lifecycle со статусами Deprecated / Superseded

**Статья нормы (вносится в `docs/agents/departments/governance.md` новый раздел «ADR Lifecycle»):**

Жизненный цикл любого ADR строится вокруг четырёх терминальных статусов. Статус **`proposed`** присваивается при первой публикации черновика; ADR в этом статусе не является действующим источником правил. Статус **`accepted`** присваивается после вердикта комиссии governance (либо прямого утверждения Владельцем для ADR, принятых до бутстрапа комиссии). ADR-решение становится действующим именно в этот момент и фиксируется в CHANGELOG отдела. Статус **`deprecated`** применяется, когда решение более не используется, но замены нет (технология отпала, контекст исчез). Статус **`superseded`** применяется, когда принят новый ADR, заменяющий прежний; при этом обязательным становится двустороннее связывание — в старом ADR добавляется поле `superseded_by: NNNN`, а в новом — поле `supersedes: NNNN`. ADR в статусах `deprecated` и `superseded` **не удаляются** из репозитория и остаются как исторический слой решений.

**Frontmatter-контракт (минимальный набор полей для шапки каждого ADR):**

```yaml
---
id: ADR-NNNN
title: <одной строкой>
status: proposed | accepted | deprecated | superseded
date: YYYY-MM-DD
author: <роль или имя>
approved_by: <Владелец | governance-commission>      # опционально, при status=accepted
approved_date: YYYY-MM-DD                             # опционально, при status=accepted
supersedes: NNNN                                      # опционально, в преемнике
superseded_by: NNNN                                   # обязательно при status=superseded
---
```

**Объём правки существующих ADR (14 файлов).** Только шапка: приведение текущих разнородных формулировок («принято», «принято v1.1, редакция …», «утверждён (governance, …)», «черновик», «proposed (ожидает governance)») к единому enum. Разделы «Проблема», «Решение», «Последствия» **не трогаются** — это immutable-контент.

**Правило перехода между статусами:**
- `proposed → accepted` — исключительно по вердикту комиссии governance с соответствующей записью в CHANGELOG.
- `accepted → superseded` — только когда принят преемник; связывание двустороннее и происходит в одной заявке.
- `accepted → deprecated` — по решению комиссии без преемника.
- Обратные переходы (откаты) оформляются новым ADR с формулировкой «восстанавливает решение ADR-NNNN».

**Примечание по ADR 0004 Amendment (`routers` vs `api/`).** Amendment на ADR 0004 от 2026-04-15, зафиксированный в CLAUDE.md, фактически превратил исходную формулировку в устаревшую. После внедрения пункта 1 настоящей заявки Координатор может опциональным отдельным действием оформить превращение Amendment в полноценный ADR-преемник со статусом `superseded_by`, но эта работа **не входит** в DoD текущей заявки.

### Пункт 2 — RFC Naming Convention

**Статья нормы (вносится в `docs/agents/departments/governance.md`, перекрёстная ссылка из `departments/research.md`):**

Имя файла любого RFC подчиняется формату `rfc-YYYY-NNN-slug.md`, где `YYYY` — год подачи, `NNN` — трёхзначный сквозной номер в пределах года, `slug` — короткое kebab-case описание темы. Нумерация сквозная внутри года; пропущенные номера (зарезервированные, но не дошедшие до публикации) фиксируются в `docs/research/rfc/_skipped.md` и не переиспользуются. В 2027 году нумерация начинается с `rfc-2027-001`.

**Frontmatter-контракт RFC:**

```yaml
---
id: RFC-YYYY-NNN
title: <одной строкой>
status: draft | in-review | accepted | rejected | implemented
date: YYYY-MM-DD
author: <роль или имя>
owner_decision: <при status accepted/rejected/implemented — краткая запись>
---
```

**Переименование существующих 4 RFC (выполняет Координатор через `git mv` после approve):**
- `rfc-001-claude-code-routines.md` → `rfc-2026-001-claude-code-routines.md`
- `rfc-003-mem0-subagent-memory.md` → `rfc-2026-003-mem0-subagent-memory.md`

**Ссылки на старые имена.** Координатор по факту выполнения `git mv` обязан прогнать поиск подстрок `rfc-001-`, `rfc-003-`, `rfc-004-`, `rfc-005-` по всему репозиторию и заменить жёсткие ссылки. ### Пункт 3 — RFC vs ADR разграничение

**Статья нормы (вносится в `docs/agents/departments/governance.md` новым разделом «RFC vs ADR — границы»):**

**RFC (Request for Comments)** — живой документ этапа исследования. Живёт в `docs/research/rfc/`. Предназначен для обсуждения идеи, сбора контраргументов, выработки рекомендации. RFC может пересматриваться, переписываться, отзываться. Владелец нормы — R&I-department. Статусы RFC — `draft | in-review | accepted | rejected | implemented`.

**ADR (Architecture Decision Record)** — immutable-запись принятого архитектурного решения. Живёт в `docs/adr/`. Предназначен зафиксировать «что решили, почему, какие альтернативы были отвергнуты» на момент принятия. После статуса `accepted` контент ADR не правится — он либо действует, либо заменяется новым ADR с соответствующей парой `supersedes` / `superseded_by`. Владелец нормы — Governance-department (комиссия) совместно с автором-архитектором.

**Переход RFC → ADR.** Если RFC принят (`status: accepted`) и принятое решение требует архитектурной фиксации (а не просто organizational change) — Координатор поручает advisor `architect` или соответствующему Директору подготовить ADR. ADR ссылается на исходный RFC в поле `source_rfc:`. RFC остаётся в `docs/research/rfc/` как исторический документ, его статус переводится в `implemented`.

**RFC без ADR.** Не каждый RFC рождает ADR. Если принятое решение — процессное (изменение регламента, перераспределение обязанностей, новые шаблоны документов), оно оформляется правкой `CODE_OF_LAWS.md` либо `departments/*.md`, а не ADR. Настоящая заявка — пример такой ситуации: 3 из 4 её пунктов рождают процессные правила, не ADR.

**Директории:**
- `docs/research/rfc/` — R&I-RFC (внешние находки, cross-audit, новые практики).
- `docs/adr/` — только принятые архитектурные решения.
- Отдельная директория `docs/rfc/` для «внутренних архитектурных RFC» (как предлагалось в RFC-005 §3.6) **в рамках этой заявки не заводится** — откладывается до появления первого реального кейса, чтобы не плодить пустую структуру.

### Пункт 4 — Bandit + pip-audit в CI (governance-норматив)

**Статья нормы (вносится в `docs/agents/departments/governance.md` новым разделом «Security gate CI (норматив)»):**

Governance фиксирует следующий норматив для любого Python-кода в репозитории coordinata56, подлежащий инфраструктурной реализации через заявку к infra-director:

1. **Bandit** (`bandit -r backend/app/ -ll`) — статический security-анализ уровня medium и выше. Порог прохождения: severity ≥ medium, confidence ≥ medium → fail CI.
2. **pip-audit** — сканирование Python-зависимостей на известные CVE. Порог прохождения: CVE с CVSS ≥ 7.0 → fail CI; 4.0–6.9 → warn (не fail, но обязательная ссылка в PR-description).
3. Оба инструмента — open-source, включаются в CI рядом с `ruff check` и `pytest`.
4. При обнаружении P0-CVE (CVSS ≥ 9.0) в production-ветке — активируется процедура incident response через infra-director.

**Разграничение ответственности:**
- **Governance** — владелец самой нормы, её формулировки, порогов, правила изменения (через комиссию).
- **Infra-department** — владелец технической реализации (правка `.github/workflows/ci.yml`, Makefile-таргеты, документация шагов).

**Поэтому настоящая заявка фиксирует только норматив.** Техническая правка CI идёт отдельной мини-заявкой от Координатора к infra-director после активации последнего в соответствующем объёме. Эта мини-заявка трассируется в CHANGELOG governance как «RFC-005 Q-4 — техническая реализация», но не входит в DoD текущей заявки.

## Риски

Все риски — малой и средней степени, без P0.

**R1 — битые ссылки после переименования RFC.** Вероятность: средняя (жёсткие ссылки точно есть — минимум в `findings.md`, `CHANGELOG.md` governance и предыдущей заявке 2026-04-17). Последствие: дохлые ссылки в markdown. Митигация: аудитор составляет полный список до `git mv`, Координатор обновляет все вхождения в одном коммите вместе с переименованием.

**R2 — ADR-ссылки по номеру статуса.** Вероятность: низкая (ссылки обычно идут по номеру ADR, не по статусу). Последствие: если где-то условный лог проверяет строку `статус: принято` — после нормализации на `status: accepted` сломается. Митигация: аудитор в задаче 1 проверяет наличие таких мест.

**R3 — конфликт нормы Security gate CI с уже существующим разделом в `quality.md`.** Вероятность: низкая. Последствие: две нормы про SAST/CVE — у Governance и у Quality. Митигация: аудитор в задаче 4 проверяет; в случае существования — Governance фиксирует только мета-норму «Security gate CI — обязателен», Quality владеет конкретным списком чек-листов.

**R4 — частичная применимость ADR Lifecycle к ранним ADR.** Для ADR 0001–0007, принятых до бутстрапа комиссии governance, утверждающим записан Владелец напрямую. Это нормальный случай и не требует пересмотра — просто в frontmatter фиксируется `approved_by: Владелец` и `approved_date: <фактическая>`.

**R5 — увеличение attack surface CI.** Сам по себе пункт 4 рисков безопасности не добавляет (инструменты локальные, не сетевые). Но техническая мини-заявка к infra должна учесть: bandit/pip-audit запускаются на runner GitHub Actions, кэш зависимостей — на общем runner'е. Митигация — в объёме инфра-заявки, не текущей.

**R6 — задержка по срокам.** Оценка 3–4 дня складывается из: 14 ADR × ~10 минут на frontmatter = 2.5 часа; 4 RFC × 15 минут = 1 час; 3 раздела в governance.md = 2 часа; поиск и правка жёстких ссылок = 1–2 часа; мини-заявка к infra — вне этого скоупа. Суммарно ~6–8 часов чистой работы Координатора за 3 календарных дня. Риск срыва — низкий.

**Риска «конфликт с вышестоящим актом» не ожидается.** Все четыре пункта — нормы более низкого уровня (ADR-шаблон, RFC-naming, процессные разграничения, мета-норма для инфры), не затрагивают Конституцию, Процессуальный кодекс и CODE_OF_LAWS по существу. Максимум — добавление перекрёстных ссылок.

Вероятность низкая: пункты представляют собой либо новые нормы (2, 3, 4), либо косметическую унификацию существующей (1). Warnings уровня «битые ссылки» не являются блокирующим замечанием — обрабатываются Координатором в том же коммите что и git mv.

**Ключевой аргумент вердикта.** Все четыре пункта — нормативная гигиена: унификация статусов, единое имя, разграничение жанров, фиксация порога безопасности. Консистентность регламента растёт, полнота растёт, дублирование не создаётся, противоречий с вышестоящими актами не обнаружено по результату беглой проверки комиссией. Содержательный предмет — RFC-2026-005 — уже одобрен Владельцем, комиссии остаётся только оформить техническую фиксацию в регламент.
