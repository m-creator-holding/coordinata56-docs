## 2. Технологический тренд: российские Voice AI — обновления за неделю

По решению прошлой недели (Voice — в план M-OS-2, финалист Yandex SpeechKit), trend-scout и market-researcher продолжают отслеживать провайдеров. За неделю собрано:

| Провайдер | Что нового | Влияние на наш план |
|---|---|---|
| **Yandex SpeechKit** | Публично анонсирован бета-доступ к модели **`yandex-gpt-speech`** (совмещённый LLM + ASR для «понимания речи», не только распознавания). Сроки GA — Q3 2026. | Наблюдать. Для нашей M-OS-2 это избыточно — мы парсим транскрипт через Claude на своей стороне, дублирование. Но в M-OS-3 может упростить архитектуру. |
| **Sber SaluteSpeech** | Незначительное снижение цены на batch: с 0,35 до **0,30 руб/мин** (тариф обновлён на `developers.sber.ru/docs/ru/salutespeech/tariff` — подтвердить trend-scout). | Наш резервный вариант стал ещё дешевле. Усиливает позицию Sber как fallback, не меняет финалиста. |
| **Tinkoff VoiceKit** | Без релизных новостей за неделю. Статус "maintain mode" — продукт стабилен, обновлений нет. | Остаётся на 3-м месте. Из рассмотрения для cottage-pod — выводим. |
| **Whisper (OpenAI)** | Opensource-сообщество выпустило **`whisper-large-v3-turbo`** — в 8× быстрее large-v3, качество сопоставимое. | Снижает планку «Whisper неприменим на telephone». Тем не менее fine-tune на стройке — отдельный ML-проект (2–3 мес.). Переводим из Assess в «активный мониторинг», пересмотр при провале Yandex. |
| **Vosk (Alpha Cephei)** | Релиз Russian Model 0.54 — прирост точности на 2–3% (WER 14% → 11–12% на «чистой» речи). | Vosk = offline fallback для PWA при отсутствии сети. Пилот стоит запланировать в M-OS-3, не сейчас. |

**Вывод:** финалист остаётся **Yandex SpeechKit**. Новые данные не меняют решение, но снижают риск: Sber дешевле (резерв прочнее), Whisper-turbo смягчает сценарий «vendor lock-in» (быстрее сделать opensource fallback, если потребуется). Sber теперь занимает позицию «дешёвый массовый batch для не-критичных сценариев» — например, транскрибация видеосовещаний M-OS-3.

## 3. Пять рекомендаций Владельцу (по приоритетам)

Приоритет P0 = делаем немедленно в рамках M-OS-1. P1 = закладываем в план M-OS-2 сейчас. P2 = backlog с триггером. P3 = мониторим, не действуем.

### Рекомендация #1 — **P0 (в план, немедленно)** — собрать строительный глоссарий для Voice AI прямо сейчас

**Что.** До старта разработки voice-фичи в M-OS-2 собрать корпус из 200–300 строительных терминов + 5–10 фамилий бригадиров «Координаты 56» + 20–30 типовых фраз прорабов. Задача — construction-expert через Координатора, срок — неделя.

**Зачем P0.** Voice-фича запускается в M-OS-2 (через 4–8 недель). Глоссарий — вход в fine-tune модели Yandex. Если начнём собирать тогда, потеряем 2 недели старта. Задача дешёвая (1 человек, несколько часов), оттягивать нет смысла.

**Что делает Координатор.** Бриф construction-expert: «собрать глоссарий для Voice AI M-OS-2, формат — таблица термин/значение/частотность». Результат складывается в `docs/innovation/voice-glossary.md`.

---

### Рекомендация #2 — **P1 (в план M-OS-2)** — зафиксировать Sber SaluteSpeech как резерв, не искать дальше

**Что.** Остановить sensing по voice-провайдерам после 1 мая. Решение по финалисту (Yandex) и резерву (Sber) принято, дальнейший мониторинг — monthly, не weekly. Tinkoff из рассмотрения убираем.

**Зачем P1.** Избегаем «аналитического паралича». У нас есть рабочая пара вендор/резерв. Дальнейший sensing — расход времени без пользы до итогов пилота. После пилота M-OS-2 пересмотрим radar.

**Что делает Координатор.** Обновить Tech Radar: T1 зафиксировать с пометкой «финалист — Yandex SpeechKit, резерв — Sber SaluteSpeech, остальные — заморожены до результатов пилота». Сократить периодичность voice-sensing с weekly на monthly.

---

### Рекомендация #3 — **P1 (в план M-OS-2)** — изучить Android-offline-паттерн PlanRadar v6.12

**Что.** Заказать market-researcher один раздел анализа: как именно PlanRadar v6.12 реализует offline-cache чертежей (чертежи весят десятки МБ, синхронизация «всё подряд» убивает батарею и трафик). Выход — раздел в `docs/innovation/reports/planradar-offline-patterns.md`.

**Зачем P1.** Для нашей PWA в M-OS-2 (PlanRadar — прямой UX-референс по мобильному полю) эта находка применима напрямую: отдельно взятый паттерн сэкономит frontend-director 2–3 недели R&D.

**Что делает Координатор.** Бриф market-researcher, срок 3 дня, источник — публичные release notes + обзоры в Google Play. Живые установки не нужны.

---

### Рекомендация #4 — **P2 (backlog, триггер «провал Yandex»)** — Whisper-turbo на self-hosted как запасной путь

**Что.** Не запускать сейчас. Зафиксировать в Tech Radar (уже в Assess, AS2 рядом) как «активный backup-trigger»: если Yandex даст WER > 15% после fine-tune — инициируем отдельный R&D-трек на Whisper-turbo (8× ускорение = новая реальность).

**Зачем P2.** Новая модель Whisper-turbo меняет расклад — но не настолько, чтобы пересматривать решение по Yandex. Держим как «план Б».

**Что делает Координатор.** Обновить Tech Radar: AS9 (новая строка) — «Whisper-large-v3-turbo» с условием triggers. Ничего не запускать.

---

### Рекомендация #5 — **P3 (мониторинг, без действий)** — «AI-ассистенты к 1С» — отдельный сегмент

**Что.** Добавить в `competitor-watch.md` отдельную категорию «AI-ассистенты к 1С» с первыми 4 игроками из 1С:Маркет (Softline, First Bit, ещё 2 — уточняет market-researcher). Monthly scan, не weekly.

**Зачем P3.** Это не наш сегмент и не конкуренты (они работают ПОВЕРХ 1С, мы строим СВОЁ). Но тренд важен: если один из них начнёт отбирать долю «AI внутри 1С», наш аргумент «вы всё равно наткнётесь на limits 1С» ослабнет. Следим.

**Что делает Координатор.** Бриф market-researcher на monthly scan этого подсегмента, первая выгрузка — к следующему Digest.
