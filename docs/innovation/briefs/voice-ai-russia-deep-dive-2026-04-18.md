# Innovation Brief — Voice AI РФ, deep-dive

> **Дата:** 2026-04-18
> **Автор:** innovation-director (Департамент инноваций и развития)
> **Статус:** готов к решению Владельца
> **Одобрено ранее:** msg 1411 — Voice в план M-OS-2 (Trial T1 Technology Radar)
> **Формат:** deep-dive продолжение sensing, источники — live web search, публичные тарифы и документация
> **Потребитель результата:** Владелец (решение о провайдере) + Координатор (маршрутизация в backend-director для интеграции в M-OS-2)

## TL;DR (для занятых)

- **Рекомендуемый финалист:** **Yandex SpeechKit** (потоковое распознавание v3 + дообучение модели на строительном глоссарии).
- **Причина:** лучший компромисс «качество на русском + кастомизация терминов + 152-ФЗ + зрелость API» для полевого сценария стройки.
- **Резервный вариант:** **Sber SaluteSpeech** — дешевле на объёмах, но дообучение модели хуже документировано.
- **Альтернатива open-source Whisper:** **рассматривать только как офлайн-fallback**, не как основной канал (WER на русском хуже без fine-tune, а fine-tune — отдельный R&D-проект).
- **Оценка стоимости для 85 домов × 50 записей × ~1 мин ≈ 127 500 мин/мес** → порядка **60–90 тыс. руб/мес** при разговорах по 1 минуте, детали ниже.

## 2. Рекомендация — финалист

**Yandex SpeechKit (потоковое распознавание v3) + адаптация модели на строительный глоссарий.**

### Обоснование (почему именно он)

1. **Адаптация модели под домен задокументирована и доступна по API.** Для стройки это критично — без дообучения «ростверк», «арматурный каркас», «опалубка палубы», фамилии подрядчиков («Абдурахмонов», «Тохтамуродов») будут распознаваться с ошибками даже у топ-модели. У Yandex — готовый процесс: глоссарий → шаблоны → фразы для обучения.
2. **Зрелая документация и экосистема.** gRPC v3, WebSocket-примеры, PyPI-пакет `yandex-speechkit`, публичные примеры интеграции в Django/FastAPI — низкая стоимость внедрения командой.
3. **Hybrid-вариант (on-premise) существует.** Это страховка: если по 152-ФЗ в будущем потребуется хранить аудио внутри периметра холдинга, не придётся мигрировать платформу.
4. **Баланс цены.** Sber в абсолютных цифрах дешевле (0,60 руб/мин против ~1,00), но для M-OS-2 MVP важнее качество распознавания специфичной лексики, а не экономия 30–40% на потоке. Разница в деньгах — ~30 тыс. руб/мес на нашем объёме, легко перекрывается меньшим числом ошибок и переделок.
5. **Санкционная устойчивость = Sber и Tinkoff, то есть ничейная.** Все три игрока — российские, риск блокировки одинаков.

### Почему не Sber

- Минимум 600 руб/мес не блокер, но кастомизация модели под терминологию стройки менее прозрачна.
- Эмоции в реальном времени — фича крутая, но нерелевантная для сценария «прораб наговаривает дневной отчёт».
- Оставляем как **резервный вариант** — если по итогам пилота Yandex покажет WER > 15% на строительных терминах даже после fine-tune, мигрируем на Sber.

### Почему не Tinkoff

- Сильная сторона — телефония и call-центры (оттуда продукт вырос). Для полевого сценария «микрофон на строительной площадке» — не профильный.
- Цена онлайн-потока 0,72 руб/мин — между Yandex и Sber, без явных преимуществ.

## 3. Integration план для M-OS-2

### 3.1. Где подключаем

- **Pod:** `cottage-platform-pod` (первый domain pod).
- **Раздел PWA:** «Обход прораба» — новый экран в мобильном UI (Telegram WebApp / PWA). Прораб на объекте нажимает крупную кнопку «Записать отчёт», говорит, система распознаёт, показывает черновик формы.
- **Типы форм на старте пилота (минимальный scope):**
  1. Дневной отчёт по дому: «дом 12, выполнена заливка фундамента, ушло 22 куба бетона М300, бригада Абдурахмонова, 4 человека, с 8 до 14»
  2. Замечание по качеству: «дом 8, опалубка не вертикальна на восточной стене, фото прилагаю»
  3. Запрос материалов: «на завтра нужно 200 м арматуры диаметр 12, на дом 15»

### 3.2. User flow (happy path)

```
1. Прораб в PWA жмёт «Записать отчёт»
2. Браузер захватывает микрофон (WebAudio API)
3. Аудио-чанки по WebSocket/gRPC-web → backend-proxy → Yandex SpeechKit streaming
4. По мере распознавания — партиальный транскрипт в UI (live-превью)
5. Прораб заканчивает → финальный транскрипт
6. Backend: LLM-парсинг транскрипта в структурированный JSON (дом/дата/работа/объём/бригада/время)
7. UI показывает черновик формы с заполненными полями
8. Прораб правит/подтверждает → форма уходит в БД, срабатывает audit.log, уведомление мастеру участка в Telegram
```

### 3.3. Fallback при плохой связи

- **Офлайн-режим в PWA:** если сеть < 3G или потеряна, аудио записывается локально (IndexedDB).
- При восстановлении сети — асинхронная отправка в Yandex SpeechKit **batch**-режим (в 3× дешевле потокового: ~0,15 руб/мин vs 1,00).
- Прораб видит бейдж «3 записи ожидают отправки».
- Распознанный текст возвращается push-уведомлением в Telegram: «Отчёт распознан, откройте для проверки».

### 3.4. Ответственные роли

- **backend-director:** проектирование proxy-сервиса SpeechKit, LLM-парсер транскрипта в структуру, интеграция с cottage-pod.
- **frontend-director:** PWA-экран «Обход прораба», запись/отправка аудио, live-транскрипт, offline-queue.
- **innovation-director (я):** сбор корпуса из 1 000+ записей реальных прорабов для дообучения модели (как задача — передать construction-expert).
- **quality-director:** замер WER на контрольном наборе из 100 записей до и после fine-tune.
- **governance-director + legal:** DPA (data processing agreement) с Yandex.Cloud, согласие на запись в PWA (галочка «Аудио обрабатывается Yandex для распознавания речи, 152-ФЗ соблюдается»).

### 3.5. Критерии успеха пилота (exit-criteria)

- WER на строительной лексике ≤ 12% после fine-tune (на baseline-модели обычно 18–25% на редких терминах).
- Время «конец речи → черновик формы на экране» ≤ 3 секунды при хорошей сети.
- Доля записей, где прораб не редактирует форму вручную, ≥ 50% — тогда фича реально экономит время.
- Себестоимость ≤ 1500 руб / прораб / месяц.

## 4. Риски

### R1. Точность на шумной стройке (ветер, техника) — КРИТИЧЕСКИЙ

Vosk-бенчмарки и публичные тесты показывают: WER растёт в 2–3× при SNR < 15 дБ (рядом с экскаватором — SNR ~5–10 дБ).

**Митигация:**
- Специальная кнопка «Записать отчёт» побуждает прораба отойти в тихое место (вагончик, машина).
- Nоise suppression на клиенте — браузерный WebRTC-NS или библиотека rnnoise перед отправкой в SpeechKit.
- В батч-режиме — pre-processing аудио (полосовой фильтр 300–3400 Гц как в телефонии).

### R2. Жаргон и сленг прорабов — КРИТИЧЕСКИЙ

Примеры: «мужики залили плиту», «восьмёрка кладёт кирпич», «бадья» вместо «бункер», «пирог» (многослойный элемент), клички бригадиров.

**Митигация:**
- Глоссарий на старте — 200–300 терминов, собрать совместно с construction-expert и прорабами «Координаты 56».
- После 2 месяцев эксплуатации — ре-тренировка модели на собранном корпусе.
- LLM-постобработка (Claude) нормализует сленг в канонические формы перед записью в БД («мужики» → «бригада», «восьмёрка» → конкретная бригада N8).

### R3. Стоимость при масштабе — СРЕДНИЙ

На 85 домах, если каждый дом имеет закреплённого прораба с 50 записями в день × ~1 мин ≈ 4 250 мин/день × 30 = 127 500 мин/мес.

Но фактически прорабов меньше (1 прораб на 10–15 домов), реальный объём — **порядка 30 000–50 000 мин/мес** на всю стройку.

**Оценка стоимости (Yandex, потоковое + частично batch):**
- 70% в реальном времени × 1,00 руб/мин = 21 000–35 000 руб
- 30% в batch-режиме (офлайн-fallback) × 0,18 руб/мин = 1 620–2 700 руб
- Синтез речи (TTS для обратных уведомлений, опционально) — до 5 000 руб
- **Итого: 30 000–45 000 руб/мес** на ~50 000 мин
- Пессимистичный сценарий (все 127 500 мин в real-time): **~128 000 руб/мес**

**Митигация:**
- Batch-режим агрессивно (всё, что можно отложить на 5–10 минут, уходит в batch) — экономия ~5×.
- Локальный VAD (voice activity detection) отрезает паузы до отправки — экономия 20–30%.
- Лимит per-prorab: 100 мин/день жёсткий cap.

### R4. Зависимость от одного вендора (vendor lock-in) — СРЕДНИЙ

**Митигация:**
- Anti-corruption layer: сервис `voice_service` с интерфейсом `recognize(audio) -> text + metadata`, реализация Yandex-специфична, но за интерфейсом.
- При провале пилота — перейти на Sber меняя только адаптер, не затрагивая cottage-pod.
- Это прямая реализация принципа A1 из Technology Radar (Adopt).

### R5. Соответствие 152-ФЗ и согласие на запись — НИЗКИЙ

Аудио прораба = персональные данные (голосовой отпечаток как биометрия — спорно, но безопаснее считать ПДн).

**Митигация:**
- Согласие в трудовом договоре / доп. соглашении при найме.
- Отключаемая опция «не записывать» в PWA.
- Хранение транскрипта в БД, а не аудио (аудио удаляется после распознавания, кроме 5% выборки для QA).
- Хостинг распознавания в Yandex.Cloud = 152-ФЗ соблюдён.

## 5. Альтернатива — OpenAI Whisper (open-source, self-hosted)

### Плюсы
- **Бесплатно** по лицензии MIT. Платим только за GPU-хостинг (~15 000 руб/мес за 1 GPU-сервер A10/A100).
- **Полная приватность.** Аудио не покидает периметр холдинга. 152-ФЗ закрыт без DPA с вендором.
- **Офлайн** — работает без интернета, если модель развёрнута на edge-устройстве (теоретически даже на телефоне прораба, но модель large-v3 = 3GB, неприменимо).
- **Сообщество:** fine-tuned модель `antony66/whisper-large-v3-russian` даёт **WER 6.39%** на русском (на базе v3 — 9.84%).

### Минусы
- **Фактическая точность на русском хуже, чем у коммерческих РФ-провайдеров** в типовых задачах без fine-tune. OpenAI сами указывают русский в списке «лучше 50% WER», но детального benchmark против SpeechKit/SaluteSpeech нет.
- **Fine-tune под стройку — это полноценный ML-проект**: сбор данных, разметка, GPU для тренировки, валидация. 2–3 месяца работы ML-инженера, которого у нас нет.
- **Задержка real-time хуже.** Whisper изначально batch-ориентирован. Whisper.cpp и faster-whisper дают streaming, но с латентностью 1–2 секунды — на грани приемлемого для UX.
- **Операционная нагрузка:** свой GPU-сервер = мониторинг, апдейты, downtime. У Yandex это SaaS, downtime — их проблема.
- **Нет механизма глоссария «из коробки».** Нужно или fine-tune, или post-processing словарями.

### Вердикт по Whisper

**Не основной канал, но стратегический fallback для M-OS-3+:**
- Держим PoC на self-hosted Whisper в Tech Radar **Assess** (добавить).
- Если Yandex в пилоте даст WER > 15% после fine-tune **ИЛИ** ежемесячная стоимость уйдёт за 150 тыс. руб — инициируем отдельный R&D трек «M-OS Whisper Voice Engine» в M-OS-3.
- На M-OS-2 Whisper использовать **не будем** — слишком высокая операционная цена для пилотного сценария.

## 6. Решение и следующие шаги

### Решение (на утверждение Владельцу)

1. **Финалист:** Yandex SpeechKit (streaming v3 + fine-tune на строительной лексике).
2. **Резерв:** Sber SaluteSpeech (если Yandex провалит пилот).
3. **Shelved alternative:** Whisper self-hosted (возврат при триггерах — см. раздел 5).

### Действия (после одобрения Владельцем)

1. **Координатор → backend-director (бриф):** ТЗ на proxy-сервис SpeechKit (gRPC → internal REST), anti-corruption layer, интеграция в cottage-pod. Фаза M-OS-2, приоритет Trial T1.
2. **Координатор → frontend-director (бриф):** ТЗ на экран «Обход прораба» в PWA, запись аудио, offline-queue, live-транскрипт.
3. **Координатор → governance-director (бриф):** DPA с Yandex.Cloud, галочка согласия в PWA, обновление политики персональных данных.
4. **Innovation-director (я) → construction-expert (консультация):** сбор первичного глоссария 200–300 строительных терминов + 5–10 фамилий подрядчиков.
5. **Innovation-director → quality-director (через Координатора):** методика бенчмарка WER — контрольный набор 100 записей до и после fine-tune.
6. **Обновление Tech Radar:** T1 остаётся в Trial, внутри выбран вендор = Yandex. Whisper добавить в **Assess** (A9).

### Что НЕ делаем сейчас

- Не подписываем enterprise-контракт с Yandex — пилот на self-serve тарифе первые 2 месяца.
- Не трогаем on-premise Hybrid — это обсуждение уровня M-OS-3+ при росте объёма.
- Не разворачиваем Whisper-PoC — только когда/если Yandex провалится.
