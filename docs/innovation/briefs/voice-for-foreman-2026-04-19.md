## Проблема

### Что сегодня (as-is)

Прораб «Координаты 56» ведёт учёт работ, материалов и замечаний на бумаге (блокнот в кармане), потом:
1. Вечером / в конце недели передаёт записи бухгалтеру, мастеру участка, снабженцу.
2. Записи вбиваются вручную в 1С, Excel, переписку в Telegram.
3. Часть фактов теряется: забыл, неразборчиво, устно проговорил — не записал.

**Симптомы, которые видит Владелец:**
- Расхождение «бумажный отчёт ↔ 1С» в пределах 10–25% по материалам (оценка construction-expert).
- Запаздывание факта на 1–5 дней (решения принимаются на устаревших данных).
- Двойной/тройной ввод: прораб пишет → бухгалтер вбивает → отчёт верстает секретарь. Потери — ~20% рабочего времени офиса на «перекладывание данных из листа в экран».
- Замечания по качеству (кривая опалубка, брак арматуры) фиксируются только если прораб сфотографирует и отправит в чат. Чаще — не фиксируются вовсе.

### Что изменилось на рынке (почему сейчас)

- **Yandex SpeechKit v3** — зрелый gRPC-стриминг, документированный fine-tune под глоссарий, хостинг РФ, 152-ФЗ by design. До 2024 fine-tune под узкий домен был enterprise-only.
- **LLM structured extraction** (Claude tool use, function calling OpenAI) вышел в массовый продакшен в 2024–2025 — превращает «сырой транскрипт» в заполненную форму без отдельного NLP-инженера.
- **PWA + WebAudio API** стабильно работает на Android/iOS с 2023 года, включая background-запись и offline-queue через IndexedDB. Нативное приложение для MVP не требуется.
- **Конкуренты в РФ** (ПланРадар, Мобифорс, BuildDocs, самописные решения крупных застройщиков) — **voice-to-structured-form на русском не внедрили**. Есть voice memos (голосовое в чат без разбора), есть STT в западном Procore без русского fine-tune. Конкурентное окно открыто.

### Цена бездействия

- Если M-OS-2 выкатится **без** voice-ввода, прораб продолжит вести бумагу → данные в системе будут неполными и запоздалыми → весь блок аналитики М-OS-3 (план/факт, прогнозирование, KPI прорабов) получает «мусор на входе».
- Это прямой риск потерять ROI цифровизации: система красивая, а данных нет.

## Решение

### Функциональное ядро (что делает фича)

```
Прораб на объекте → крупная кнопка «Диктовать» в PWA →
  запись аудио (WebAudio) → streaming в бэкенд →
    Yandex SpeechKit (текст + таймкоды) →
      Claude (structured extract по tool-схеме домена) →
        черновик формы на экране прораба →
          подтверждение/правка →
            запись в cottage-pod (houses, materials, issues) + audit.log →
              уведомление ответственному (мастер участка / снабженец) в Telegram
```

### Три типа форм на старте (минимальный scope MVP)

1. **Дневной отчёт по дому** — пример фразы: *«дом 12, залили фундамент, 22 куба бетона М300, бригада Абдурахмонова, 4 человека, с 8 до 14»* → заполняет карточку WorkPerformed (house_id, work_type, volume, unit, material_id, brigade, shift).
2. **Замечание по качеству** — *«дом 8, опалубка кривая на восточной стене, фото приложу»* → создаёт Issue (house_id, category, description, severity) + открывает камеру для фото.
3. **Запрос материалов** — *«на завтра нужно 200 метров арматуры 12, на дом 15»* → создаёт MaterialRequest (house_id, material_id, quantity, needed_date).

### Почему именно такой scope на MVP

- Все три сценария **покрывают ~80% ежедневных записей прораба** по оценке construction-expert (см. `docs/knowledge/construction/06-roles-and-responsibilities.md`).
- Схема домена у нас уже есть (cottage-pod houses + materials + issues — строится в M-OS-1.1A), voice-слой становится **новым каналом ввода** к существующим моделям, а не новой подсистемой.
- Три tool-схемы для Claude extract — это ~1 день работы backend + qa, а не ML-проект.

### Ключевые архитектурные решения

- **Voice Service как core-level pod** (не внутри cottage-pod). Интерфейс `recognize(audio) -> transcript + metadata` + `extract(transcript, schema) -> structured_payload`. Cottage-pod — первый потребитель; при масштабировании на карьер/АЗС воспользуется тот же core-сервис. Соответствует ADR 0009 (pod-архитектура).
- **Anti-corruption layer к вендору** (Yandex ↔ Sber ↔ Whisper — подмена одной строкой в конфиге). Прямая реализация Tech Radar A1.
- **Glossary как config-as-data** — 200–300 строительных терминов + фамилии бригадиров хранятся в БД, редактируются админом через Admin UI без релиза кода. Соответствует принципу vision №10.
- **Audit ВСЕГДА** — каждая транскрипция пишется в `audit_log` с маскированием (`employee_name → ****`, см. CLAUDE.md §Данные/ПД). Аудио удаляется после распознавания (кроме 5% QA-выборки).

## Дельта

### Дельта vs декларация в vision §4.9/§5.1

| Ось | Vision (декларация) | Этот бриф (конкретика внедрения) |
|---|---|---|
| Сценарии | «голосовые заметки с транскрипцией» | 3 типа форм со схемой извлечения |
| Архитектура | нет | core-pod `voice_service` + ACL + glossary-as-data |
| Вендор | нет | Yandex SpeechKit (baseline) + Sber (резерв) + Whisper (shelved) |
| Offline | нет | PWA-очередь в IndexedDB + batch-режим |
| Стоимость | нет | 30–45 тыс. руб/мес на реальный объём (baseline оценка) |
| Критерии успеха | нет | WER ≤ 12% post-fine-tune, <3 сек до черновика, ≥50% форм не требуют правок, ≤1500 руб/прораб/мес |

### Дельта vs рынок (конкуренты)

| Конкурент | Что есть у них | Чего нет | Наше преимущество |
|---|---|---|---|
| Procore Mobile | photo upload, typed forms, западный STT | русский язык, fine-tune под стройку РФ, 152-ФЗ | язык + суверенитет данных |
| ПланРадар | photo + voice memo (audio в чат без разбора) | structured extraction в форму | LLM extract → готовая форма |
| BuildDocs РФ | typed forms, фото | voice | canonical voice-first UX |
| Мобифорс | мобильный ввод typed | voice, LLM extract | оба слоя |
| Крупные застройщики (самописное) | есть у 1–2 (ПИК, Самолёт) — по слухам, публичных кейсов нет | открытых API, масштабируемой схемы | пока открыто, окно 6–12 мес |

**Главная дельта:** мы делаем не «voice-заметку в чат», а **voice-to-structured-form** — речь превращается в заполненную карточку работы/материала/замечания, которую прораб только подтверждает. Это переводит голос из «канала мемо» в «канал оперативного учёта».

### Дельта vs 1С-интеграция (параллельная фича M-OS-2)

- 1С-интеграция решает проблему «данные уже есть в 1С, не вводить дважды».
- Voice решает проблему «данные НЕ доходят до системы вовсе, потому что ввод слишком трудоёмкий».
- Фичи **не конкурируют**, работают на разных участках воронки. Вместе дают замкнутый контур: прораб диктует → запись в M-OS → обмен с 1С → бухгалтер видит факт.

## Оценка внедрения

### Сложность: **L** (крупно)

Три слоя новой функциональности: (а) streaming-proxy с внешним вендором, (б) LLM extract с tool-схемой на 3 домена, (в) PWA с audio capture + offline-queue + live-транскриптом. Каждый — не тривиален в отдельности, вместе — интеграционный проект.

### Зависимости (что должно быть готово ДО старта voice)

| Зависимость | Статус | Блокер? |
|---|---|---|
| cottage-pod M-OS-1.1A (houses, materials, issues модели + API) | в работе (Sprint 1 Wave 8) | **ДА** — без моделей некуда писать |
| RBAC v2 (роль «прораб» с правами на write work-performed) | в работе M-OS-1.1A | ДА |
| Audit service v1 (маскирование ПД) | готов skeleton, живой — с M-OS-1.1B | ДА |
| Admin UI для редактирования глоссария | не начат | нет (fallback: seed из YAML) |
| DPA с Yandex.Cloud | не начат | **ДА перед live-вызовами** (production-gate) |
| Согласие сотрудника на запись голоса | не начат | ДА перед live-вызовами |
| Config-as-data layer (для glossary + fine-tune configs) | в M-OS-1 must-have (vision §8) | нет (можно без него — seed) |
| Event bus (для уведомлений мастеру) | в M-OS-1 must-have | нет (fallback: sync-call) |

### Оценка в человеко-днях (агент-днях)

Разбивка по департаментам (делаются параллельно там, где нет зависимостей):

| Работа | Департамент | Агент-дни |
|---|---|---|
| Proxy-сервис Yandex SpeechKit (gRPC → internal REST) + ACL | backend | 5–7 |
| LLM extract для 3 форм (Claude tool use) + validation | backend | 4–6 |
| Glossary store (read/write API, seed 200 терминов) | backend | 2–3 |
| Интеграция с cottage-pod (write work/issue/request + audit) | backend | 3–4 |
| PWA-экран «Обход прораба» (кнопка запись + live-транскрипт) | frontend | 5–7 |
| Offline-queue в IndexedDB + sync при coming back | frontend | 4–5 |
| Live-превью транскрипта и черновика формы | frontend | 3–4 |
| WER benchmark methodology + контрольный набор 100 записей | quality | 3–4 |
| DPA с Yandex.Cloud + политика ПД + согласие в PWA | governance + legal | 3–5 (live-юрист — позже, на production-gate) |
| Сбор стартового глоссария (200 терминов + 5–10 бригадиров) | construction-expert + innovation-analyst | 2–3 |
| Design research «большая кнопка на холоде» + wireframes | design | 2–3 |
| Собрать корпус 1000+ записей для fine-tune | innovation + construction-expert | 10–15 (календарно 2–3 недели) |
| Fine-tune модели Yandex + замер WER | innovation + quality | 3–5 |
| Code review + security review + интеграционные тесты | reviewer + qa | 4–6 |

**Суммарно:** ~53–77 агент-дней (при последовательном выполнении), календарно **5–7 недель** при параллельной работе трёх директоров.

### Поэтапность (чтобы не строить всё сразу)

- **Phase V1 (скелет, 2–3 недели):** форма #1 «дневной отчёт по дому», только online (без offline-queue), Yandex без fine-tune (baseline WER), без Admin UI для глоссария (seed YAML). Цель — end-to-end proof, замер реального WER на стройплощадке.
- **Phase V2 (расширение, 2–3 недели):** формы #2 и #3, offline-queue, glossary в Admin UI, fine-tune по первому корпусу.
- **Phase V3 (production-hardening, 1–2 недели):** DPA подписан, политика ПД обновлена, exit-criteria замерены, cutover с «пилота» на «по умолчанию для всех прорабов».

**Оценка verify-before-scale:** V1 — это верификация на одном прорабе и одном типе формы перед раскаткой на 85 домов. Соответствует правилу Владельца 2026-04-18 msg 1280.

### Риски (сжато, детали в baseline-брифе §4)

| Риск | Уровень | Митигация (ключевая) |
|---|---|---|
| WER на шумной стройке (ветер, техника) | **КРИТ** | кнопка побуждает отойти в тихое место + noise-suppression на клиенте |
| Жаргон/сленг прорабов | **КРИТ** | glossary + fine-tune + LLM-нормализация на втором этапе |
| Стоимость при масштабе | СРЕДН | batch-режим для всего, что можно отложить; жёсткий cap 100 мин/прораб/день |
| Vendor lock-in Yandex | СРЕДН | ACL-интерфейс, смена вендора — конфиг |
| 152-ФЗ и согласие на запись | НИЗК | хостинг РФ + DPA + согласие при найме + удаление аудио после транскрипции |
| Прораб откажется пользоваться (UX-риск) | **КРИТ** | design research до разработки + пилот на 1–3 лояльных прорабах «Координаты 56», обратная связь → итерации |
| Сбор корпуса для fine-tune замедлит проект | СРЕДН | V1 стартует БЕЗ fine-tune, fine-tune — в V2 параллельно |

### Что НЕ делаем в рамках MVP voice (out of scope)

- Синтез речи (TTS — обратные голосовые уведомления прорабу). Отложено в M-OS-3.
- Voice biometry как способ аутентификации прораба. Отдельный трек, vision §ГГ1.
- On-premise Hybrid Yandex SpeechKit. Обсуждаем только при росте объёма > 200 тыс. мин/мес.
- Self-hosted Whisper. Remain в Tech Radar Assess (A9), активируется при провале Yandex.
- Voice-ввод для других ролей (сметчик, бухгалтер, снабженец) — только после успеха на прорабе.

## Рекомендация

**Trial** — фича берётся в M-OS-2 как Phase V1 (скелет, 1 прораб «Координаты 56», 1 тип формы, baseline Yandex без fine-tune). Переход в **Adopt** — по exit-criteria Phase V3 (WER ≤ 12%, ≤1500 руб/прораб/мес, ≥50% форм без ручных правок, субъективное «прорабу удобно»).

### Основания для Trial (а не Adopt сразу)

- Решение Владельца 2026-04-18 msg 1411 уже поставило фичу в флагманский статус M-OS-2 — **статус-кво** принятия задекларирован.
- Экономика (30–45 тыс. руб/мес на пилот, 3 прораба) — посильна без enterprise-контракта.
- Вендор выбран в baseline-брифе, дельта за неделю (Round 2 sensing) финалиста не меняет (ожидание отчёта trend-scout 2026-04-22).
- Главные неизвестные — реальный WER на стройплощадке и UX-принятие прорабом — **решаются только пилотом**, не кабинетным анализом.

### Основания НЕ Adopt сразу

- Ни одного живого замера WER в полевых условиях «Координаты 56».
- Не собран корпус для fine-tune.
- Нет Admin UI для редактирования глоссария (сильно режет скорость итераций при плохой точности).
- DPA/согласие/политика ПД не оформлены — live-вызовы к Yandex заблокированы CODE_OF_LAWS «No live external integrations» до production-gate.

### Решения, которые нужны от Владельца (перед kick-off)

1. **Подтвердить старт Phase V1 после закрытия M-OS-1.1A.** Календарно — орientир 3–4 недели с момента зелёного регресса Wave 8/11.
2. **Выбрать 1–3 лояльных прорабов** «Координаты 56» для пилота. Innovation-director готов список кандидатов вместе с construction-expert.
3. **Согласовать бюджет пилота** 60–90 тыс. руб/мес на период V1+V2 (2–3 мес), потолок — 150 тыс. руб/мес.
4. **Дать OK на подготовку DPA с Yandex.Cloud** (governance-director запустит в работу; подписание — перед production-gate, не сейчас).

### Open questions для Владельца (2–3 критичных)

1. **Пилотные прорабы.** Готов ли Владелец назначить 1–3 прорабов «Координаты 56» пилотной группой на 2 месяца с правом фидбэка напрямую в Telegram Владельцу? Альтернатива — фокус-группа из снабженцев как более лояльных к цифре.
2. **Формат согласия на запись голоса.** Добавлять пункт в действующие трудовые договоры через допсоглашение сейчас, или подписывать только для пилотной группы с пометкой «эксперимент», полный rollout — после доработки ЛНА? Юридически чище вариант 2, кадрово — вариант 1.
3. **Приоритет voice vs 1С-интеграции в M-OS-2.** Vision ставит обе фичи флагманами фазы. Если одна из них задерживается — какую тащим первой? Предварительная рекомендация innovation-analyst: **1С первой** (устраняет саботаж бухгалтерии, блокирующий работу M-OS), **voice параллельно со второго месяца M-OS-2**.

## Приложение. Связь с регламентом

- **Tech Radar:** T1 Voice остаётся в **Trial**, внутри выбран вендор Yandex (baseline 2026-04-18). Whisper — в **Assess** (A9).
- **ADR 0008 (M-OS определение):** соответствует — Voice = AI-native канал ввода, принцип №1.
- **ADR 0009 (pod-архитектура):** voice_service как core-level pod, cottage-pod — первый потребитель.
- **Vision принцип 6 (152-ФЗ by design):** хостинг РФ, DPA, удаление аудио, маскирование ПД.
- **Vision принцип 10 (config-as-data):** glossary редактируется без релиза.
- **Vision принцип 11 (skeleton-first для ПД):** модели ПД строим сейчас, compliance (ЛНА, живой юрист) — перед production-gate.
- **CODE_OF_LAWS «No live external integrations»:** до production-gate код-адаптер и тесты на моках, живые вызовы Yandex SpeechKit — только после DPA + согласия + одобрения Владельца.
- **Verify-before-scale:** Phase V1 — пилот на 1 прорабе + 1 типе формы до раскатки.
- **Engineering principles:** скелет → итерации, не параллельное строительство V1+V2+V3.
