## 0. Уточнение фактов

Аудитор сравнил наш `m-os-1-frontend-plan.md` с «фактическим repo» и сделал вывод о Next 16.2.3 + React 19 + Supabase + `AGENTS.md`. Это некорректная формулировка. Фактическое состояние:

1. В репозитории `coordinata56` действительный фронтенд-каркас живёт по пути `/root/coordinata56/frontend/`. Это **Vite 5 + React 18.3 + TypeScript 5.5 + Tailwind + shadcn/ui + React Router 6 + TanStack Query 5 + Zustand + react-hook-form** (см. `frontend/package.json`, `frontend/vite.config.ts`). Nothing Next.js, nothing Supabase, nothing React 19, никакого App Router. Каркас 5 страниц (Dashboard / Houses / Finance / Schedule / NotFound), построен в Фазе 2–3 по ADR 0002 и соответствует плану.
2. То, что аудитор назвал «фактическим repo» — это **локальный прототип Владельца на MacBook** (Next 16.2.3 + React 19 + App Router + Supabase). В `coordinata56`-монорепо его нет, в git он не появлялся.
3. `AGENTS.md` в корне `/root/coordinata56/AGENTS.md` — это файл служебных заметок про субагентов M-OS, **не «AGENTS.md прототипа»** и не документация «нестандартного Next». Аудитор, вероятно, видел отдельный `AGENTS.md` внутри локального прототипа Владельца.

Значит обсуждение — не «мы строим план против живого кода», а «продолжаем ли мы использовать репо-каркас, который уже соответствует плану, или переходим на ветку локального прототипа Владельца». Это бизнес-решение Владельца о судьбе прототипа, а не техническая «миграция».

## 1. Архитектурный выбор: Next vs Vite

### Вариант A — остаёмся на плане (Vite + React 18 + SPA + PWA). Рекомендуется.

Плюсы:
- Репо-каркас уже такой (Vite, React 18, TS, shadcn/ui, Tailwind, RR6, TanStack Query, Zustand, RHF). Ничего не ломаем.
- ADR 0002 утверждён Владельцем и не противоречит фактическому коду. Amendment не требуется.
- 5 блокирующих вопросов плана уже закрыты согласованно: `bpmn-js` для BPM, собственный form-builder на RHF+zod, единый SPA с двумя роутинговыми треками `/admin` и `/field`, PWA через `vite-plugin-pwa`, lazy-chunks между треками.
- Уведомления у нас через Telegram (решение Владельца), SEO не нужен, весь UI — за авторизацией. SSR и edge-rendering не дают никакой бизнес-ценности.
- LLM-агенты генерируют чистый Vite+React стабильно; App Router у моделей даёт ошибки чаще (спутывание `use client` / Server Components, закладывание fetch вне RSC-контекста).
- Engineering principle Владельца: «сначала простота, только если есть бизнес-причина». Бизнес-причины для Next нет.

Минусы:
- Прототип Владельца на Next нельзя перенести в наш каркас «файл-в-файл». Любые идеи UX-решений из прототипа нужно переводить на наш стек вручную.
- Нет встроенных Server Actions / Server Components. Нам и не нужны, но психологически «выглядит не так современно».

Трудоёмкость: **каркас Vite+React уже есть**. Добавить в него маршруты `/admin`, `/field`, `vite-plugin-pwa`, auth, layout-selector, MSW-mock — это 0.5–1 неделя одного воркера в Волне 1 Code (Wave 1 M-OS-1.1).

### Вариант B — мигрируем план на Next 16.2.3 + React 19 + App Router.

Плюсы:
- Один стек с прототипом Владельца, если прототип признать боевым заделом.
- Server Components теоретически дают быстрый первый рендер (при медленной сети тяжёлых admin-экранов).
- App Router удобен, когда будет SSR-нужда (сейчас её нет).

Минусы:
- Next 16 — самая свежая мажорная версия, небольшие стабильные патчи ещё выкатываются. Для проекта из 17 субагентов стабильность критичнее, чем bleeding edge.
- Supabase в прототипе — **это альтернативный backend**. В M-OS backend уже зафиксирован: Python + FastAPI + SQLAlchemy + PostgreSQL (ADR 0002, 0011–0014). Если тащить Supabase — придётся ломать весь бэкенд-контур. Это уже не «миграция фронта», это смена backend-архитектуры.
- `bpmn-js` в Next App Router работает, но требует аккуратности: модуль использует `window`, должен быть `use client` и `dynamic({ ssr: false })`. Мы получим ту же функциональность, что и в Vite, плюс 3–5 рабочих дней отладки SSR-guard'ов.
- `vite-plugin-pwa` → `next-pwa` (или ручной Workbox через `@serwist/next`). `next-pwa` — на полу-поддержке одного контрибутора, в Next 14+ регулярно отстаёт от релизов. Это долгосрочный риск на обновлениях Next.
- Переписывание `m-os-1-frontend-plan.md` и связанных артефактов: структура `src/app/(admin)/...` вместо `src/admin/pages/...`, lazy-chunking через `dynamic()` вместо `React.lazy`, рассечение auth на middleware Next и client-hooks, замена `@tanstack/react-query` на TanStack + RSC-паттерны, отдельная стратегия cache tags. **Оценка переписывания плана — 2 рабочих дня фронт-директора.** Плюс потребуется **amendment ADR 0002** (layer 4, 5, 6) через Governance-комиссию — это ещё 3–5 дней процессуально.
- Потеря текущего `frontend/` каркаса или его прямой перевод на Next (снос `vite.config.ts`, установка `next`, переезд роутера, обновление tsconfig, eslint). Это 3–4 дня воркера, и за эти дни вся остальная работа по фронту замораживается.

Трудоёмкость переписывания плана + amendment ADR + миграция каркаса: **8–12 рабочих дней**, при этом ни одна бизнес-задача не закрывается, только меняется фреймворк.

5 блокирующих вопросов при варианте B:
- BPM: `bpmn-js` → работает, но через `use client` + `dynamic(..., {ssr:false})`.
- Form-builder: работает идентично.
- PWA: переключение на `@serwist/next` или `next-pwa`. Риск стабильности обновлений.
- Единый код vs два проекта: можно сделать, как и в Vite. Lazy-chunking через route groups.
- Реальное ускорение: **нулевое**. Для закрытых под auth интерфейсов SSR ничего не даёт.

### Вариант C — гибрид: admin на Next, field/PWA на Vite.

Плюсы:
- Теоретическая изоляция: admin-UI можно отдать «фулстек-платформе» типа Vercel/Next.
- Два трека независимо эволюционируют.

Минусы:
- Две кодовые базы. Дублируем `src/shared/`: auth, API-клиент, типы OpenAPI, shadcn-компоненты, query-кеш. **+30–40 % постоянной работы на синхронизацию.**
- Два деплоя, два бандла, два набора конфигов, две сессии логина (или SSO-контур, которого у нас нет).
- Два разных подхода к роутингу у одного backend'а. Риск, что LLM-агенты путают конвенции между `admin/` и `field/`.
- Ни одного бизнес-выигрыша: внутренний admin-UI не нуждается в SSR.

Вариант C — over-engineering без бизнес-основания. Отклонён.

### Вердикт направления

**Рекомендуется Вариант A.** Остаёмся на Vite + React 18 + SPA с PWA-слоем через `vite-plugin-pwa`. Локальный прототип Владельца — рабочий эскиз (prototype-to-throw-away), из него переносим **идеи UX**, а не файлы.

Если Владелец по бизнес-причинам настаивает на сохранении прототипа в проде — тогда вариант B с amendment ADR 0002 через Governance. Но от направления фронтенда бизнес-причины не видно.

## 2. Последствия Варианта A

### Судьба прототипа Владельца

Прототип — **эскиз**. Ценные артефакты:
- UX-решения, user flows, расположение фич, дерево навигации.
- Контент и названия разделов.
- Наглядные схемы, скриншоты — отдаём `designer` (L4) как material.
- Логика экранов переводится в user stories в `docs/pods/cottage-platform/stories/` через Координатора.

Код прототипа не берётся 1:1 ни в одной строке: Next-код не совместим с нашим каркасом, Supabase-вызовы не совместимы с нашим backend API. Перенос = переработка под наш стек.

Коммитить прототип в git **не надо** — по регламенту v1.6 любой код проходит Директор → Head → Worker → Reviewer.

### Минимальный каркас в Волне 1 Code (M-OS-1.1, первая рабочая неделя)

Уже сделано в репо:
- Vite + React 18 + TS + Tailwind + shadcn/ui + RR6 + TanStack Query + Zustand + react-hook-form.
- 5 рабочих страниц и RootLayout.

Нужно добавить (батч M-OS-1.1-fe-skeleton, ~0.5–1 неделя воркера):

1. Разделение роутинга на `/admin/*` и `/field/*` с lazy-chunked `AdminApp` / `FieldApp` (`React.lazy` + `<Suspense>`).
2. `AdminLayout` (левая нав-панель, верхняя шапка с компанией/юзером) и `FieldLayout` (нижняя навигация, sticky-хедер).
3. Auth-каркас под multi-company из ADR 0011: `AuthProvider`, `useCurrentUser`, `useCurrentCompany`, guard-обёртки.
4. `vite-plugin-pwa` — подключение с пустым manifest (иконки подтянем в 1.3). SW пока noop, но инфраструктура готова.
5. **MSW (Mock Service Worker)** — мок API по OpenAPI-спеке. До готовности backend-эндпоинтов каждой под-фазы фронт работает против MSW-моков. Это закрывает риск #3 из плана (позднее закрытие OpenAPI backend'ом).
6. Skeleton-страницы первых трёх разделов 1.1: Companies, Users, Roles. Без реальной логики, только таблица/форма-плейсхолдер и связь с mock-API.

Всё остальное (Company Settings, Integration Registry, System Config) — следующие батчи той же 1.1.

### MSW-mock до готовности API

Схема:
- Frontend генерирует TS-типы из OpenAPI backend'а через `openapi-typescript` в CI.
- До готовности настоящего endpoint'а backend-director (через Координатора) публикует zero-version OpenAPI — контракт без реализации.
- MSW-handlers пишутся по этому контракту. Mock-данные — `faker` + фикстуры.
- Как только backend выкатывает реальный endpoint — убираем handler из MSW (или оставляем в dev-режиме для оффлайн-разработки).

Это даёт параллельность backend/frontend и закрывает риск срыва старта под-фазы из-за ожидания API.

### Amendment к ADR 0002

Amendment по слоям 4 (frontend-фреймворк), 5 (UI-kit — остаётся), 6 (сборщик — замена Vite на Next):
- Подготовка amendment-документа — 0.5 дня.
- Governance-комиссия (`agents/governance/`) — согласование обычно занимает 2–4 дня.
- Утверждение Владельцем.
- Итого процессуально: **3–5 дней календарных** до момента «amendment принят».

Amendment обязательно затрагивает:
- Раздел «Вариант A vs Next.js» в ADR 0002 — меняется вердикт.
- Раздел «Последствия» — добавляются риски, связанные с App Router.
- Список файлов в конце ADR — меняется.

**Внимание**: новый ADR под номером 0018 «Admin UI» уже запланирован (см. `m-os-1-foundation-adr-plan.md`). При выборе Next он тоже частично перепишется: правила lazy-load и структура экранов зависят от фреймворка.

### Совместимость bpmn-js в Next App Router

- `bpmn-js` — чисто клиентская библиотека, использует `window`, `document`. В App Router обязаны:
  - Импортирующий компонент помечен `'use client'`.
  - Либо динамически: `const Modeler = dynamic(() => import('bpmn-js/lib/Modeler'), { ssr: false })`.
- Work risk: 3–5 рабочих дней отладки SSR-guard'ов при первой интеграции. В Vite такой работы нет вообще.

### Совместимость PWA с Next

- `vite-plugin-pwa` отпадает — это Vite-only.
- Замена: `next-pwa` (полу-заброшен, работает с ограничениями на Next 14+, для Next 16 нужно проверять актуальность), либо `@serwist/next` (активнее поддерживается, но менее известный контур).
- Service worker конфигурируется иначе: Next имеет свою runtime-cache-стратегию для статики. Придётся мержить.
- Work risk: 2–4 рабочих дня фронт-воркера на каркас PWA в Next против 0.5 дня в Vite.

### Сводная оценка варианта B

| Работа | Время |
|---|---|
| Переписывание frontend-плана | 2 дня фронт-директора |
| Amendment ADR 0002 + Governance | 3–5 дней календарных |
| Переезд каркаса с Vite на Next | 3–4 дня воркера |

И ни одного закрытого бизнес-требования за эти две недели.

## 4. Admin UI как Configuration-as-Data (ADR-0018 write-up)

Прототип Владельца использует `localStorage` для хранения настроек dashboard (`src/components/dashboard/use-dashboard-state.ts`). Это антипаттерн для нашего M-OS: конфигурация обязана жить на сервере в `configuration_entities` + child-tables, с версионированием, publish/rollback, аудитом. ADR-0017 это фиксирует.

Чтобы admin-UI стал настоящим Configuration-as-Data, **вне зависимости от Vite/Next**, нужны следующие экраны и backend-контракты. Это подготовка к ADR-0018.

### Экраны admin-UI (12 разделов, разбиты на под-фазы M-OS-1.1/1.2/1.3)

M-OS-1.1 (Foundation):
1. **Юрлица** — CRUD карточек компаний. Каждая карточка: ИНН, КПП, ОГРН, юр.адрес, банковские реквизиты, ФИО ген.директора, печать-подпись-файлы.
2. **Пользователи** — CRUD пользователей: ФИО, email, телефон, привязка к компаниям через `user_company_roles`.
3. **Роли** — редактор ролей: название, описание, принадлежность к компании (роль глобальная vs per-company).
4. **Матрица прав (Permissions Matrix)** — таблица «роль × permission»: admin может отметить checkbox для каждого действия на каждом ресурсе. Данные идут в `role_permissions` (ADR 0011).
5. **Company Settings** — форма 7 полей per-company: НДС-режим, валюта, часовой пояс, рабочая неделя, единицы измерения, логотип, цвет бренда (или что будет зафиксировано в ADR-0017 развилка В).
6. **Integration Registry** — каталог подключённых внешних систем. В M-OS-1 единственная запись — Telegram-бот с конфигом токена, chat_id алертов, списком разрешённых событий. API позволяет добавлять/выключать интеграции, но фактический адаптер — только Telegram (по ADR 0015).
7. **System Config** — глобальные системные настройки M-OS: URL приложения, лимиты, флаги фич.

M-OS-1.2 (BPM и нотификации):
8. **BPM Constructor** — bpmn-js canvas + properties panel + список процессов + кнопка Publish с preview-эффектом на запущенные экземпляры (migration rules из ADR-0017 Б2).
9. **Notification Constructor** — шаблоны уведомлений: текст, plugin-плейсхолдеры, правила когда отправлять (связь с BPM-событиями).
10. **Document Template Constructor** — шаблоны документов (счёт, акт, накладная), поля-плейсхолдеры.

M-OS-1.3 (Forms и Reports):
11. **Form Builder** — drag-drop конструктор форм, JSON-дескриптор в `form_definitions`.
12. **Report Builder** — конструктор таблиц-отчётов с фильтрами, сортировкой, агрегацией. JSON-дескриптор в `report_definitions`.

### Требуемые backend API-контракты

Для каждой из 9 категорий `configuration_entities` нужны единообразные эндпоинты:

- `GET /api/v1/config/{category}/` — список с пагинацией (ADR 0006).
- `GET /api/v1/config/{category}/{id}` — карточка.
- `POST /api/v1/config/{category}/` — создание (draft, не опубликовано).
- `PATCH /api/v1/config/{category}/{id}` — правка draft'а.
- `POST /api/v1/config/{category}/{id}/publish` — публикация с preview-эффектом (возвращает список затронутых объектов до применения).
- `POST /api/v1/config/{category}/{id}/rollback` — откат к предыдущей версии.
- `GET /api/v1/config/{category}/{id}/history` — история версий.
- `GET /api/v1/config/{category}/{id}/audit` — аудит изменений (из `audit_log` с фильтром).

Это общий контракт. Специализация per-category — в child-table, что показываем на UI.

### Правила versioning/rollback/audit в UI

- Каждая конфигурация имеет `version`, `published_at`, `created_by`. Виден в UI на карточке.
- Publish — всегда через диалог: preview эффекта + подтверждение + опциональный комментарий.
- Rollback — только из списка истории, с тем же диалогом.
- Audit-вкладка на каждой карточке — читает тот же `audit_log`, что и обычный ADR 0007 аудит, фильтр по `resource_type={category}`.

**Никаких `localStorage` для самой конфигурации.** Единственное уместное применение `localStorage` — UX-мелочи: свёрнута ли боковая панель, ширина колонок таблицы. Это не конфигурация, это personal UI-state.

Этот раздел — основа для write-up ADR-0018. Финальный документ пишет backend-director (автор ADR-плана) в координации с frontend-director'ом после одобрения плана Владельцем.

## 5. React 19 vs React 18

### Что меняется в React 19

- `use()` — хук для чтения promises и context прямо в render (суспенс-friendly).
- Actions — встроенная поддержка async-handler'ов в формах через `<form action={...}>` с Server Actions.
- `useOptimistic`, `useFormStatus`, `useActionState` — новые хуки.
- ref как prop (без `forwardRef`).
- Document metadata hoisting (`<title>`, `<meta>` прямо в компоненте).
- Removed legacy string refs, legacy context API.

### Применимо ли к прототипу Владельца

Без просмотра кода прототипа точно сказать нельзя, но с большой вероятностью прототип:
- Использует обычные client-components без Server Actions.
- Supabase-вызовы идут через клиентскую библиотеку Supabase, не через React 19 Actions.
- Новые хуки типа `useOptimistic` если и используются, то в паре мест.

Если прототип — обычная React-SPA на Next 16, то 90–95 % кода работает 1:1 на React 18. Миграция ценных UX-решений прототипа в наш Vite+React 18 стек **не блокируется React 19-фичами.**

### Наш выбор

В Варианте A остаёмся на React 18.3. **Это не критично**: всё, что даёт React 19, мы либо закрываем стандартными паттернами (TanStack Query-based optimistic updates вместо `useOptimistic`, обычные `forwardRef` вместо ref-as-prop), либо не нуждаемся (Server Actions нужны только в SSR-контексте, которого у нас нет).

Когда React 19 достигнет вторичного минора (19.1–19.2), а TanStack Query / RHF / react-router полностью стабилизируют совместимость (к концу 2026 скорее всего так и будет) — можно будет спокойно обновиться в M-OS-2.

## 6. Главные риски рекомендуемого выбора (Вариант A)

### Риск 1: Владелец ожидает «современный» Next и воспримет Vite как «недо-стек»

**Вероятность: средняя. Влияние: низкое.**

Vite + React 18 + TS + shadcn — полноценный production-стек на 2026 год, используется Shopify, Figma-частично, большинством внутренних SaaS. Для внутреннего ПО холдинга он идеален. Митигация: ясное бизнес-обоснование Владельцу через Координатора.

### Риск 2: Идеи UX-прототипа «теряются в переводе» на наш стек

**Вероятность: средняя. Влияние: среднее.**

Прототип содержит реальные пользовательские потоки, которые Владелец вручную выверил. Если кодом не пересажать, UX может разъехаться.

Митигация: перед стартом каждой под-фазы admin-UI (1.1 / 1.2 / 1.3) `designer` (L4 advisory) и frontend-director последовательно разбирают соответствующий раздел прототипа через Координатора, извлекают user flows и фиксируют в user stories. Код не копируется — поведение и расположение экранов переносятся.

### Риск 3: Backend опаздывает с OpenAPI, MSW-мок перерастает в теневой backend

**Вероятность: средняя. Влияние: высокое.**

Риск известный (риск #3 в плане). MSW должен быть только mock-инструментом, не «вторым источником правды». Если backend задерживается на >1 недели после старта под-фазы — эскалация Координатору, задержка под-фазы, но не «давайте мы пока сами напишем API в MSW».

Митигация: жёсткое правило — MSW-handler живёт не более 2 недель без эталонного backend-endpoint'а. После 2 недель handler удаляется, под-фаза ставится на hold до реального API.
