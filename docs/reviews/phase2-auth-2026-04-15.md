## P0 — Blocker

### P0-1. Timing attack через ранний выход при отсутствии пользователя

**Файл:** `backend/app/api/auth.py:65–74`

```python
result = await db.execute(select(User).where(User.email == body.email))
user = result.scalar_one_or_none()

if user is None or not verify_password(body.password, user.password_hash):
    ...
    raise HTTPException(...)
```

Выражение `user is None or not verify_password(...)` использует ленивое вычисление: если `user is None` — `verify_password` вообще не вызывается. Это означает, что запрос с несуществующим email обрабатывается значительно быстрее (нет bcrypt-операции ~100–300 мс), чем запрос с неверным паролем. Атакующий, измеряя время ответа, перечисляет существующие адреса — классическая timing-атака, нарушающая OWASP A07.

ADR §«Алгоритм» явно требует «предотвращение enumeration» — код это не реализует.

**Фикс:** всегда вызывать `verify_password` даже при отсутствии пользователя через dummy hash. Паттерн — в скиле `fastapi-auth-hardening` §1.

### P0-2. Небезопасный дефолт `jwt_secret_key`

**Файл:** `backend/app/core/config.py:46–49`

```python
jwt_secret_key: str = Field(
    default="change_me_to_random_32_char_secret",
    ...
)
```

Дефолт означает, что без `JWT_SECRET_KEY` в env приложение стартует с публично известным секретом. Любой может подписать JWT с ролью `owner`.

ADR явно требует: «`JWT_SECRET` — секрет подписи (обязательный, без дефолта)».

**Фикс:** убрать `default`, сделать обязательным (`Field(..., min_length=32)`), добавить `field_validator` с проверкой на словарные слова. Паттерн — `fastapi-auth-hardening` §2.

## P1 — Major

### P1-1. Разные сообщения 401 раскрывают состояние аккаунта

**Файл:** `backend/app/api/auth.py:68–82`

Сначала проверяется пароль, потом `is_active`. Деактивированный аккаунт с верным паролем → 401 «Аккаунт деактивирован», что отличается от «Неверный email или пароль». Атакующий узнаёт: аккаунт существует, пароль верный, но деактивирован.

**Фикс:** все три ветки 401 возвращают одинаковый detail.

### P1-2. plain-text пароля остаётся в памяти seed-миграции

**Файл:** `backend/alembic/versions/2026_04_15_0737_a1b2c3d4e5f6_seed_initial_owner.py:41–55`

ADR декларирует: «plain-text в памяти не сохраняется». Переменная `password` остаётся в scope `upgrade()` до конца. При утечке через traceback/дамп — виден.

**Фикс:** `del password` сразу после `password_hash = pwd_context.hash(password)`.

### P1-3. Нет теста на истёкший токен

**Файл:** `backend/tests/test_auth.py`

Ветка `jwt.ExpiredSignatureError` в `deps.py:74` не покрыта тестом. Покрытие 14/14 — но этот критический сценарий отсутствует.

**Фикс:** добавить тест с `jwt.encode({..., "exp": datetime.now() - timedelta(hours=1)})`. Аналогично — добавить тест на деактивированного юзера. Методика — скил `test-design-equivalence-classes`.

### P1-4. `register` раскрывает существование email

**Файл:** `backend/app/api/auth.py:122–126`

```python
detail=f"Пользователь с email '{body.email}' уже существует"
```

OWASP A07 — enumeration. Доступно только owner, но если токен скомпрометирован — атакующий перебирает.

**Фикс:** «Email уже занят» (без значения).
