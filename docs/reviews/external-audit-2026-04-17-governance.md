## 1. Анализ трёх противоречий

### Противоречие (а) — Admin-UI: под-фазы vs. «полный конструктор в M-OS-1.1»

**Статус после анализа:** ПОДТВЕРЖДЕНО. Не конфликт решений, а конфликт уровня детализации между планами.

**Источники:**
- m-os-1-plan.md v1.1/1.2: «Admin UI — полный конструктор с M-OS-1» без явного указания на под-фазы.
- m-os-1-foundation-adr-plan.md v3 (§ADR-0018): три чёткие под-фазы — M-OS-1.1 (6 разделов), M-OS-1.2 (BPM/notification/doc-template), M-OS-1.3 (form/report/rules).
- m-os-1-frontend-plan.md v1.0 §2.4: те же три под-фазы, оценки 3–4/3/3 нед.

**Разрешение:** Решение Владельца 2 (msg 1094) — «полный конструктор» — означает, что весь объём входит в M-OS-1, а не что вся реализация укладывается в M-OS-1.1. Разбивка на три под-фазы — операционная декомпозиция двух директоров (backend + frontend), не противоречащая Владельцу. Конфликт был в неполноте формулировки m-os-1-plan.md.

**Действие:** Внесена правка в m-os-1-plan.md v1.2 → v1.3: добавлено уточнение про поэтапную реализацию по под-фазам в соответствии с ADR-0018.

---

### Противоречие (б) — Терминология состояний адаптера

**Статус после анализа:** ПОДТВЕРЖДЕНО. Критически важно для кода.

**Источники:**
- m-os-1-foundation-adr-plan.md v3 §ADR-0014: `written` / `enabled` / `active_in_prod`
- ADR-0014 (docs/adr/0014-anti-corruption-layer.md, Enum `AdapterState`): `written` / `enabled_mock` / `enabled_live`

**Разрешение по приоритету документов (governance.md §«Приоритет коллизий»):** ADR-0014 является более специфичным и поздним документом. По иерархии (п. 5 — ADR высокая сила для технических вопросов) ADR-0014 является источником правды для технических терминов. Плановый документ использовал рабочие наброски.

Критичность: если разработчик читает только ADR-план v3 и пишет код с `enabled` / `active_in_prod` — seed-миграция, enum в Python, реестр и CI-тесты не совпадут. Это P0-риск.

**Действие:** Внесена правка в m-os-1-foundation-adr-plan.md v3 → v3.1: описание трёх состояний в §ADR-0014 заменено на финальные термины ADR-0014 с примечанием об исторических рабочих названиях.

### Противоречие (в) — PWA: «в начале M-OS-1.4» vs. «строгая последовательность»

**Статус после анализа:** ПОДТВЕРЖДЕНО, но не конфликт принципов — неточность формулировки.

**Источники:**
- m-os-1-plan.md v1.2 замечание к п. 7: «переносится в M-OS-2» при отставании frontend-director.
- m-os-1-frontend-plan.md v1.0 §3.6 вариант A: «PWA финиширует в первые 2 недели M-OS-1.4» (не M-OS-2).

Принцип «строгая последовательность» (п. 3 плана) не противоречит варианту A, поскольку: (1) M-OS-1.4 — бэкенд-only, не блокируется PWA; (2) п. 7 DoD м.б. закрыт в статусе «в работе» при закрытии 1.3 по 8/9 пунктов.

Формулировка «в M-OS-2» была избыточно жёсткой — она подходила лишь для крайнего случая (frontend-director вовсе не активирован).

**Действие:** Внесена правка в m-os-1-plan.md: замечание к п. 7 уточнено тремя сценариями (нормальный → триггерный перенос в 1.4 → крайний случай M-OS-2).

## 3. Оценка рекомендованного порядка действий аудитора

Аудитор предложил:
1. Зафиксировать governance-решение: Next.js+Prisma остаются или возврат к backend/Alembic/Vite
2. Срочно убрать hardcoded secrets, ротировать Anthropic key, закрыть /api/* auth guard
3. Минимальный Foundation PR
4. Admin-UI поверх API/OpenAPI

### Оценка governance-director

**Пункт 1 (стек-решение) — ПРАВИЛЬНЫЙ ПРИОРИТЕТ.**
Без ответа на вопрос о стеке любые правки кода — в пустоту. Если стек меняется — файлы с нарушениями удаляются целиком. Это решение Координатора → Владельца, не governance. Governance фиксирует: решение нужно до начала исправлений кода.

**Пункт 2 (ротация ключа) — ВЕРНО, но с уточнением.**
Ротация Anthropic key — немедленно и независимо от стека-решения. Это внешнее действие, не связанное с кодом. Очистка кода от hardcoded secrets — после решения по стеку (чистить файл, который будет удалён, бессмысленно). Порядок внутри пункта 2: сначала ротация credentials (Владелец), потом очистка кода (после стек-решения).

**Пункт 3 (минимальный Foundation PR) — ПРАВИЛЬНЫЙ ПРИНЦИП.**
Соответствует engineering principle «сначала простота» (Мартин, msg 817-818) и ADR-плану Волна 1 (ADR-0013 + ADR-0014 первыми, без всего лишнего). Governance добавляет: Foundation PR должен включать ADR-0014 (каркас адаптеров) как обязательный компонент, иначе повторение ст. 45а нарушений технически не заблокировано.

**Пункт 4 (Admin-UI поверх API/OpenAPI) — ПРАВИЛЬНЫЙ ПОРЯДОК, но недостаточно детализирован.**
Соответствует ADR-0018 (Admin UI строится поверх стабильного OpenAPI контракта). Нужно добавить: frontend не стартует под-фазу admin-UI без zero-version OpenAPI от backend — это правило уже зафиксировано в фронтенд-плане v1.0 §2.5 и требует соблюдения.

### Что добавить к рекомендованному порядку

5. **Реализовать ADR-0014 (ACL-каркас адаптеров) как первый PR Foundation.** Без этого повторение нарушения ст. 45а технически не заблокировано. ADR-0014 включает CI-защиту (pytest-socket), runtime-guard (AdapterDisabledError) и обязательный mock-режим.

6. **После выбора стека — подать ADR (или amendment) если Next.js остаётся.** Текущий ADR 0002 фиксирует Vite + React + FastAPI. Если Next.js остаётся — нужен Amendment ADR 0002 или новый ADR. Без этого команда работает в противоречии с утверждённым стеком.

7. **Активировать frontend-director немедленно** (обязательное условие запуска M-OS-1.1 по решению Владельца msg 1111). Это отдельное условие, не вытекающее из порядка аудитора, но блокирующее старт M-OS-1.1 без его выполнения.
