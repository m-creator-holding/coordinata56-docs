# Архитектурный аудит согласованности ADR 0001–0014 M-OS

- **Дата аудита**: 2026-04-18
- **Аудитор**: Архитектор (субагент `architect`, L4 Advisory)
- **Заказчик**: Координатор
- **Статус**: завершён
- **ADR в охвате**: 0001–0014 (14 документов)
- **ADR вне охвата**: 0015–0021 (черновики не переданы, анализируются только по упоминаниям в принятых ADR)

---

## Executive Summary

14 утверждённых ADR образуют логически связную архитектурную основу M-OS. Системных противоречий на уровне «фундаментальные принципы против принципов» нет. Большинство выявленных проблем — это несогласованности уровня P2/P3, возникшие при наращивании ADR без сквозной сверки: расхождение в именовании enum-значений, неявные предположения, неподтверждённые ссылки на будущие документы.

Обнаружена одна проблема уровня P1, требующая немедленного внимания до реализации: в ADR 0011 введён `CompanyScopedService`, который меняет поведение сервисного слоя — но ADR 0004 (контракт трёхслойной архитектуры) не обновлён и формально противоречит наследованию от базового класса `CompanyScopedService`. При реализации разработчик получит два взаимоисключающих референса. P0 не зафиксировано.

Ключевые структурные риски: предполагаемые ADR 0015–0021 упоминаются в принятых ADR как обязательные зависимости, но их черновиков нет — это откроет пробелы в реализации Волны 2.

## Детальный анализ по парам ADR

### C-01 — ADR 0001 ↔ ADR 0003: алгоритм хеширования паролей

**Вердикт: minor-inconsistency (P2)**

ADR 0001 §12 декларирует `argon2id` через `argon2-cffi` как выбранный алгоритм хеширования паролей. ADR 0003 §2 принимает решение использовать `bcrypt` через `passlib[bcrypt]` — с явной пометкой, что это отступление от рекомендации ADR 0001 по решению Владельца.

ADR 0003 содержит правильный footnote: «Примечание: ADR-0001 рекомендовал `argon2id`... Владелец принял решение использовать bcrypt в Фазе 2 ради меньшего числа зависимостей». Это корректное документирование. Однако ADR 0001 не получил обратного amendment с фиксацией: «§12 заменён решением ADR 0003». Читатель ADR 0001 получит устаревшее впечатление.

**Рекомендация**: Добавить amendment в ADR 0001 §12 с формулировкой: «Алгоритм переопределён в ADR 0003 §2 решением Владельца: `bcrypt` вместо `argon2id`. Переход на `argon2id` запланирован в security-фазе».

---

### C-02 — ADR 0003 ↔ ADR 0011: имя env-переменной JWT

**Вердикт: minor-inconsistency (P2)**

ADR 0003 в основном тексте использует `JWT_SECRET`, в Amendment 2026-04-15 фиксирует переименование в `JWT_SECRET_KEY`. ADR 0011 §2.4 расширяет JWT-клеймы (`company_ids`, `is_holding_owner`), но не ссылается явно на каноническое имя секрета и не подтверждает, что расширение клеймов не меняет алгоритм подписи.

Потенциальный риск: разработчик, реализующий RBAC v2 по ADR 0011, должен знать, что переменная называется `JWT_SECRET_KEY` (из amendment ADR 0003), а не `JWT_SECRET` (из основного текста ADR 0003). Ссылочная цепочка не очевидна.

**Рекомендация**: В ADR 0011 §2.4 добавить ссылку: «JWT-секрет: `JWT_SECRET_KEY` — см. ADR 0003 Amendment 2026-04-15».

---

### C-03 — ADR 0004 ↔ ADR 0011: контракт сервисного слоя (P1)

**Вердикт: major-inconsistency (P1) — требует amendment до реализации**

ADR 0004 фиксирует трёхслойную архитектуру Router → Service → Repository. Обязательное правило MUST №1: «SQLAlchemy-запросы пишутся только в `repositories/`. Ни роутер, ни сервис не импортируют `select`, `insert`, `update`». Сервис создаётся через конструктор с инъекцией зависимостей.

ADR 0011 §1.3 вводит базовый класс `CompanyScopedService` в ядре (`backend/app/core/`), от которого «наследуют все сервисы предметной логики». `CompanyScopedService` реализует `_scoped_query(user_context)` — т.е. сервис содержит логику формирования SQLAlchemy-запроса (WHERE-условие по `company_id`). Это нарушает MUST №1 ADR 0004: сервис начинает «знать» про SQLAlchemy.

Есть два пути разрешения:

**Путь A (рекомендуемый)**: `CompanyScopedService._scoped_query` делегирует построение WHERE-условия в `CompanyScopedRepository`, а не строит его сам. Сервис получает `company_id` и передаёт его в репозиторий как параметр. ADR 0004 остаётся без нарушений.

**Путь B**: ADR 0004 обновляется amendment с явной записью: «исключение для `CompanyScopedService` — допускается содержать логику формирования scope-параметров, передаваемых в репозиторий через API, не через прямой SQL».

Без явного решения в одном из документов два разработчика придут к разным реализациям.

**Рекомендация**: Эскалировать backend-director с задачей: выбрать путь и внести amendment либо в ADR 0004, либо в ADR 0011 §1.3 с уточнением, что `_scoped_query` строит только параметры для репозитория, а не SQL напрямую.

### C-04 — ADR 0007 ↔ ADR 0011: company_id в audit_log

**Вердикт: minor-inconsistency (P2)**

ADR 0011 §1.3 требует `company_id` во всех сущностях предметной логики: `projects`, `contracts`, `contractors`, `payments`. AuditLog — ядровая сущность, не предметная. Однако для корректной работы endpoint `GET /api/v1/audit/verify` с фильтрацией по компании и для вывода аудита бухгалтеру конкретной компании потребуется фильтрация по `company_id`. Текущий `audit_log` не имеет этого поля.

ADR 0011 §3.2 описывает `GET /api/v1/audit/verify?from=...&to=...` — без параметра `company_id`. Holding-owner видит всю цепочку, обычный пользователь — ??? Не определено.

**Рекомендация**: В amendment ADR 0011 или в отдельном ADR явно зафиксировать: является ли `audit_log.company_id` необходимым полем для фильтрации (добавить в миграцию Foundation) или endpoint `/audit/verify` доступен только `is_holding_owner=True` без фильтрации по компании.

### C-05 — ADR 0011 ↔ ADR 0014: enum AdapterState

**Вердикт: consistent (с замечанием P2)**

Задание аудита упоминает ранний конфликт именования enum в ADR 0014. Текущая редакция ADR 0014 содержит согласованный enum: `written` / `enabled_mock` / `enabled_live`. Это согласованный набор значений без конфликтов.

Замечание P2: ADR 0011 фиксирует, что переход `enabled=False → enabled=True` пишется в AuditLog. ADR 0014 фиксирует, что `AdapterDisabledError` пишется в AuditLog с `action=adapter_call_blocked`. Однако не указано, какой `action` enum-значение соответствует переходу состояния адаптера. В ADR 0001 enum `audit_log.action` содержит `create | update | delete | login | logout | access_denied` — `adapter_call_blocked` и `adapter_state_changed` отсутствуют.

**Рекомендация**: ADR 0014 должен содержать amendment к ADR 0001 §6 с добавлением новых значений в enum `audit_log.action`. Либо явно указать, что расширение enum документируется в Migrations Evolution Contract (ADR 0013 — расширение enum разрешено).

### C-06 — ADR 0009 ↔ ADR 0011: Configuration-as-data, схема role_permissions

**Вердикт: minor-inconsistency (P2)**

ADR 0009 §6 описывает принцип Configuration-as-data: роли, права, лимиты — в БД. ADR 0011 §2.2 конкретизирует таблицу `role_permissions`. Обе секции описывают одну концепцию, но по-разному:

- ADR 0009 §6 говорит: «любая бизнес-сущность, которая может меняться, хранится в БД». Это широкая декларация.
- ADR 0011 §2.2 задаёт конкретную схему `RolePermission (id, role_template, action, resource_type, pod_id, is_allowed)`.

Нет явной ссылки «ADR 0011 §2.2 реализует принцип ADR 0009 §6». Читатель ADR 0009 не знает, где искать реализацию. Читатель ADR 0011 не видит прямой ссылки на принцип, который он воплощает.

**Рекомендация**: В ADR 0011 §2.2 добавить ссылку: «Реализует принцип ADR 0009 §6 (Configuration-as-data layer)». В ADR 0009 §6 добавить ссылку: «Конкретная реализация матрицы прав — ADR 0011 §2.2».

### C-07 — ADR 0009 ↔ ADR 0012: event bus и task_events

**Вердикт: minor-inconsistency (P2)**

ADR 0009 §5 описывает шину событий: M-OS-1 — «таблица PostgreSQL, поды вызывают `EventBus.publish(event)`, конкретный транспорт скрыт за интерфейсом». ADR 0012 §3 вводит таблицы `tasks` и `task_events` как отдельную инфраструктуру оркестрации, публикует события через `EventBus.publish(event)`.

Проблема: не зафиксировано, является ли `task_events` (ADR 0012) частью той же шины событий, что описана в ADR 0009, или это отдельный механизм персистентности состояния задач. Если это разные таблицы — они должны быть чётко разделены именованием и документацией. Если одна — нужен единый контракт таблицы.

**Рекомендация**: В ADR 0012 добавить раздел «Соотношение с ADR 0009 event bus»: явно указать, что `task_events` — это хранилище состояния state machine оркестратора, а не шина domain-событий подов. Они используют одну инфраструктуру PostgreSQL, но разные таблицы с разной семантикой.

### C-08 — ADR 0012 ↔ ADR 0013: отсутствие ссылки на Migrations Contract

**Вердикт: minor-inconsistency (P3)**

ADR 0013 явно объявляет: «становится обязательной зависимостью для всех последующих ADR, содержащих миграции схемы: ADR 0014, ADR 0015, ADR 0016, ADR 0017». ADR 0012 также содержит DDL (таблицы `tasks`, `task_events`), но в его разделе «Связанные документы» нет ссылки на ADR 0013.

При этом статус ADR 0012 — черновик, не прошедший governance. При прохождении governance необходимо добавить ссылку.

**Рекомендация**: При утверждении ADR 0012 через governance добавить ADR 0013 в раздел «Связанные документы» и подтвердить, что DDL таблиц `tasks`/`task_events` проходит round-trip тест.

### C-09 — ADR 0001 ↔ ADR 0011: SET NULL для holding-owner в audit_log

**Вердикт: consistent (с замечанием P2)**

ADR 0001 фиксирует: `audit_log.user_id → SET NULL` при удалении пользователя. Это корректная политика анонимизации. ADR 0011 вводит `is_holding_owner` — пользователь с этим флагом не может быть удалён, пока в системе нет другого holding-owner. Это зафиксировано в ADR 0011 §1.3: «В системе должен всегда существовать хотя бы один holding-owner».

Формального противоречия нет: бизнес-правило (нельзя удалить единственного holding-owner) существует на уровне сервисного слоя, техническая политика SET NULL остаётся корректной. Однако нигде не зафиксировано: «удаление holding-owner блокируется сервисом пока он единственный» — это только подразумевается в ADR 0011.

**Рекомендация**: Уточнить в ADR 0011 §1.3 или §2.2 явную проверку: «сервис блокирует удаление/деактивацию пользователя если он — единственный `is_holding_owner=True` с ошибкой `LAST_HOLDING_OWNER`».

### C-10 — ADR 0013: amendment chain и force-majeure статус

**Вердикт: minor-inconsistency (P3)**

ADR 0013 принят в статусе `accepted (force-majeure)` с пометкой «ретроспективное ревью при восстановлении governance-director». Это корректная процедура при недоступности governance. Однако:

1. Amendment 2026-04-18 (разрешение `alembic.command` API) внесён в тот же ADR — это изменение уже принятого документа. По принципу immutability ADR-документов, amendment требует датировки и автора — они присутствуют в frontmatter, что соответствует стандарту.

2. ADR 0013 явно утверждён Координатором со ссылкой на «подтверждение Владельца Telegram msg 1271» — это достаточное основание. Immutability в части «тело ADR не меняется после утверждения» соблюдено: изменение оформлено как amendment в frontmatter, не как правка основного текста.

Нарушения immutability не обнаружено.

**Рекомендация**: При восстановлении governance-director запустить ретроспективное ревью ADR 0013 согласно процедуре. Если governance вносит изменения — оформить как amendment v2 с сохранением force-majeure версии.

### C-11 — ADR 0014 ↔ ADR 0015: зависимость без документа

**Вердикт: minor-inconsistency (P3)**

ADR 0014 DoD содержит: «Seed-миграция создаёт 7 записей в `integration_catalog`». `integration_catalog` является предметом ADR 0015, который не существует как документ. Это создаёт блокировку реализации: разработчик не может написать seed для таблицы, схема которой не утверждена.

**Рекомендация**: ADR 0015 должен быть приоритизирован и написан до начала реализации DoD ADR 0014. В ADR 0014 DoD добавить предусловие: «ADR 0015 принят governance».

### C-12 — ADR 0002 ↔ ADR 0001: упоминание python-jose

**Вердикт: minor-inconsistency (P3)**

ADR 0002 в разделе «Дополнительные библиотеки» содержит запись: `python-jose` / `python-jwt` (JWT-токены, Фаза 2)». ADR 0001 §12 явно фиксирует отказ от `python-jose` в пользу `PyJWT`. ADR 0002 имеет двусмысленную формулировку «`python-jose` / `python-jwt`» — это выглядит как «один из двух вариантов».

**Рекомендация**: Amendment к ADR 0002: заменить `python-jose` / `python-jwt` на `PyJWT` (зафиксировано в ADR 0001 §12, решение: `python-jose` отклонён из-за CVE).

### C-13 — ADR 0004 ↔ ADR 0007: пример конструктора в Amendment

**Вердикт: consistent (с замечанием P3)**

ADR 0007 §AuditService фиксирует: «AuditService создаётся в роутере вместе с основным сервисом и передаётся через конструктор». ADR 0004 Amendment `_make_service` соответствует этому: фабрика создаёт и сервис, и `AuditService(session)`. Паттерн согласован.

Замечание: в ADR 0004 Amendment пример `_make_service(db)` не включает `AuditService` в список параметров конструктора — это просто неполный иллюстративный пример, не противоречие. Читателю достаточно прочитать ADR 0007 для полной картины.

### C-14 — ADR 0006 ↔ ADR 0011: company_id в пагинации

**Вердикт: minor-inconsistency (P2)**

ADR 0006 описывает `list_paginated` с фильтрами soft-delete и плоскими query-параметрами. ADR 0011 вводит `CompanyScopedService`, который автоматически добавляет фильтр `WHERE company_id = ?`. Формально они не противоречат друг другу — scope-фильтр накладывается поверх.

Проблема: в ADR 0006 зафиксировано, что `total: int` возвращает число записей. Если `CompanyScopedService` добавляет фильтр по `company_id`, то `COUNT(*)` должен учитывать этот фильтр. Нигде не зафиксирован явный порядок применения фильтров: сначала company_scope, потом остальные. Разработчик без этого знания может реализовать `total` без company-фильтра — тогда пользователь увидит «всего 85 домов», хотя у его компании только 12.

**Рекомендация**: В ADR 0011 §1.3 добавить явное правило: «`CompanyScopedService._scoped_query` применяется как первый, базовый фильтр. Все дополнительные фильтры (soft-delete, query-параметры из ADR 0006) накладываются поверх него. `total` вычисляется после применения всех фильтров включая company-scope».

## Pod-изоляция vs Foundation: специальная проверка

**Вопрос из задания**: ADR 0009 pod-architecture + ADR 0021 Pod Patterns — проверить что pod'ы не могут обходить Foundation.

ADR 0021 не существует как документ. По тексту ADR 0009 §8 (структура директорий) и ADR 0011 §1.3:

- Поды располагаются в `backend/app/pods/<pod_name>/`
- Foundation (`Company`, `UserCompanyRole`, `RolePermission`, `AuditLog`) — в `backend/app/core/`
- ADR 0009 устанавливает запрет: «ни один под не импортирует напрямую модели другого»
- ADR 0009 о нарушении изоляции: «автоматическая проверка через линтер (запрет import между pod-папками)»

**Вердикт**: Правило изоляции зафиксировано текстуально. Механизм принуждения (линтер) задекларирован, но не реализован как DoD. ADR 0009 не содержит конкретного инструмента для запрета импортов поперёк pod-границ. Этот пробел должен закрыть ADR 0021, которого нет.

**Риск**: до появления ADR 0021 разработчик не получает автоматической защиты от нарушения pod-изоляции — только декларацию в документе.

## Severity уровни: сравнение шкал

**Вопрос из задания**: разные ли severity-шкалы в разных ADR?

Проверено: ADR 0005 (ошибки API), ADR 0004 (риски слоёв), ADR 0011 (риски Foundation). Все используют P0/P1/P2/P3 шкалу или Risk с Вероятность/Влияние. Единого формата таблиц рисков нет: ADR 0011 и 0012 используют таблицу с тремя колонками (Риск, Вероятность, Влияние, Контрмера), ADR 0004 и 0007 — с двумя (Риск, Митигация). Это не конфликт архитектуры, но снижает читаемость портфеля ADR для ревьюера.

**Рекомендация**: Зафиксировать единый шаблон таблицы рисков в ADR template. Не требует правки существующих ADR.
