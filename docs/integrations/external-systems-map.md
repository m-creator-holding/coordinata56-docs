# Карта внешних систем — coordinata56

**Версия**: 1.0
**Дата**: 2026-04-14
**Автор**: Интегратор (субагент `integrations`, Claude Code)
**Статус**: черновик — требует утверждения Владельца (Мартина)

---

## Преамбула

Настоящий документ фиксирует все внешние системы, с которыми будет взаимодействовать ПО `coordinata56`, определяет приоритет интеграций для MVP посёлка «Координата 56» и описывает архитектурные принципы, которыми будет руководствоваться каждая интеграция.

Согласно ROADMAP.md, интеграции начинаются в модуле M5 (после MVP1). Однако некоторые системы потребуются раньше в виде заглушек (mock-клиентов), чтобы архитектура ACL-слоя закладывалась с первой строки кода.

**Ни один боевой вызов платного API без явного согласия Мартина не производится.**

## Раздел 1. Реестр внешних систем (13 систем)

### Приоритеты (определения)

| Код | Значение |
|-----|----------|
| **P0** | Критично для MVP посёлка. Без этой интеграции система не выполняет своё ключевое назначение. |
| **P1** | Нужно в первых модулях после MVP1 (M2–M5). Без этого система работает, но с ручными заменителями. |
| **P2** | Желательно. Автоматизирует то, что сейчас делается вручную. Реализуется в M6–M8. |

### Сложность интеграции (шкала 1–5)

| Балл | Значение |
|------|----------|
| 1 | Простой REST-клиент, документация исчерпывающая, sandbox доступен бесплатно |
| 2 | REST с OAuth или сложной схемой аутентификации |
| 3 | REST + вебхуки, или нестандартный протокол (SOAP/XML), или слабая документация |
| 4 | Сложный протокол + юридические/регуляторные требования (КЭП, ЭДО) |
| 5 | Государственный API с непредсказуемым поведением, или интеграция через посредника |

### 1.1 Тинькофф — Эквайринг и РКО

| Поле | Значение |
|------|----------|
| **Назначение** | Приём онлайн-платежей от покупателей (карты, СБП), выписка РКО, автоматический импорт банковских операций в систему |
| **Тип API** | REST + HTTPS. Для эквайринга — платёжная форма + webhooks. Для РКО — OpenAPI (Tinkoff Business API) |
| **Аутентификация** | HMAC-SHA256 подпись запросов (эквайринг) + OAuth 2.0 client_credentials (РКО) |
| **Sandbox** | Да, бесплатный sandbox для эквайринга. РКО sandbox — по заявке |
| **Стоимость** | Эквайринг: 1,2–2,9% от оборота. РКО API: бесплатно при наличии счёта |
| **Сложность** | 3 / 5 — два разных API с разными схемами аутентификации |
| **Приоритет MVP** | **P1** — MVP работает с ручным вводом платежей; автоимпорт нужен в M5 |
| **ACL-изоляция** | `TinkoffPaymentGateway` (эквайринг) и `TinkoffBankStatement` (РКО) — два отдельных клиента. Внутри системы — единый доменный тип `Payment` и `BankTransaction`. При смене банка меняется только ACL-клиент, не доменная логика |
| **Директория** | `backend/integrations/tinkoff/` |

**Что изолирует ACL:**
- Формат подписи запросов (специфичен для Тинькофф)
- Структура webhook-событий (отличается от Сбера)
- Коды статусов и ошибок (Тинькофф имеет собственную схему кодов)

### 1.3 АТОЛ Онлайн — ОФД (оператор фискальных данных)

| Поле | Значение |
|------|----------|
| **Назначение** | Фискализация кассовых чеков согласно 54-ФЗ. Каждое поступление денег от покупателя — физического лица — требует выбивания кассового чека. При продаже квартир в рассрочку — чек на каждый платёж |
| **Тип API** | REST. АТОЛ Онлайн API v4 (JSON) |
| **Аутентификация** | Токен, обновляемый через логин/пароль + company_id |
| **Sandbox** | Да, тестовая касса предоставляется бесплатно |
| **Стоимость** | ~3 000–6 000 руб./мес. за кассу в облаке. Зависит от тарифа |
| **Сложность** | 3 / 5 — нужно понять схему очередей (чек принимается асинхронно, результат — через polling или callback) |
| **Приоритет MVP** | **P1** — фискализация нужна при первом реальном платеже от физлица. В MVP деньги не принимаем онлайн — можно отложить до M5 |
| **ACL-изоляция** | `AtolFiscalClient` реализует интерфейс `FiscalServicePort`. Изолируем: формат чека АТОЛ (собственный JSON-схема), polling-механизм статусов, коды ошибок |
| **Директория** | `backend/integrations/atol/` |

**Юридическая зависимость:** без регистрации кассы в ФНС API-интеграция невозможна. Кассу регистрирует Мартин как ИП или юрлицо.

### 1.4 1С:Бухгалтерия — синхронизация данных

| Поле | Значение |
|------|----------|
| **Назначение** | Передача первичных документов (договоры, акты, расходы, зарплатные начисления) в 1С для ведения официального учёта. Двусторонняя: из нашей системы в 1С и обратно (оплаты из 1С) |
| **Тип API** | HTTP-сервисы 1С (REST-подобные, через публикацию на Apache/IIS) или синхронизация через файловый обмен (CommerceML 2) в зависимости от версии 1С у Мартина |
| **Аутентификация** | HTTP Basic Auth (1С встроенная аутентификация) |
| **Sandbox** | Локальная установка 1С = sandbox |
| **Стоимость** | API бесплатно. Требует лицензии 1С (уже куплена в холдинге) |
| **Сложность** | 4 / 5 — 1С не имеет стандартизированного REST. Конкретная структура зависит от конфигурации у Мартина (1С:Строительство, 1С:Бухгалтерия 3.0, или другая). Необходимо уточнение |
| **Приоритет MVP** | **P1** — MVP ведёт собственный учёт расходов. Синхронизация с 1С нужна с первого реального платежа |
| **ACL-изоляция** | `OneCBuhClient` изолирует: XML-формат CommerceML или JSON-формат HTTP-сервисов 1С, кодировки (1С исторически использует cp1251), маппинг статей расходов нашей системы на план счетов 1С |
| **Директория** | `backend/integrations/one_c/` |

**Требуется уточнение от Мартина:** какая конфигурация 1С используется в холдинге, есть ли публикация HTTP-сервисов.

### 1.5 Росреестр — ЕГРН и ГПЗУ

| Поле | Значение |
|------|----------|
| **Назначение** | Получение выписок из ЕГРН (кто владелец участка, обременения), запрос ГПЗУ (градостроительный план земельного участка — какие параметры застройки разрешены). Проверка сделок перед подписанием договоров |
| **Тип API** | СМЭВ 3 (система межведомственного электронного взаимодействия) или REST Росреестра (API.Rosreestr.gov.ru). В 2024–2025 активно развивается REST-интерфейс |
| **Аутентификация** | ЕСИА (Госуслуги) + усиленная квалифицированная электронная подпись (УКЭП) для некоторых запросов |
| **Sandbox** | Тестовая среда есть, доступ — через заявку |
| **Стоимость** | Госпошлина: 750 руб. (физлицо) / 2 200 руб. (юрлицо) за выписку ЕГРН. Для ряда запросов — бесплатно через публичные сервисы |
| **Сложность** | 5 / 5 — государственная система, СМЭВ с XML-схемами, требует УКЭП, частые изменения форматов |
| **Приоритет MVP** | **P2** — в MVP проверки делаются вручную через сайт Росреестра. Автоматизация — в M2 (Документооборот) |
| **ACL-изоляция** | `RosreestrEgrnClient` изолирует: СМЭВ-протокол, XML-конверты, УКЭП-подпись запросов, формат ответа (XML → наши доменные объекты `PropertyRecord`, `LandPlanRecord`) |
| **Директория** | `backend/integrations/rosreestr/` |

**Предупреждение:** интеграция с Росреестром требует участия юриста для проверки правомерности запросов в автоматическом режиме.

### 1.11 Диадок (Контур) — ЭДО

| Поле | Значение |
|------|----------|
| **Назначение** | Юридически значимый электронный документооборот: договоры с покупателями (214-ФЗ), акты с подрядчиками, УПД (универсальные передаточные документы). Устраняет бумажный оборот |
| **Тип API** | REST. Диадок API. Есть SDK для Python (`diadoc`) |
| **Аутентификация** | OAuth 2.0 + УКЭП (усиленная квалифицированная электронная подпись) для подписания |
| **Sandbox** | Да, тестовая среда (Диадок.Тест) бесплатна |

| **Сложность** | 4 / 5 — OAuth + КЭП + форматы ФНС (XML UPD, ТОРГ-12), обязательная аккредитация |
| **Приоритет MVP** | **P1** — бумажный ЭДО работает, но при первом договоре 214-ФЗ лучше сразу работать в электронном виде |
| **ACL-изоляция** | `DiiadokEdoClient` изолирует: XML-форматы ФНС, подпись УКЭП, статусную машину документа Диадока (Отправлен → Доставлен → Подписан → Отказано). Внутри — `DocumentStatus` в наших терминах |
| **Директория** | `backend/integrations/diadok/` |

**Альтернатива**: СБИС (Тензор) — аналог Диадока. Та же сложность, другой API. Выбор — за Мартином (часто определяется тем, какой ЭДО уже используют контрагенты).

### 1.13 Банк-эскроу (214-ФЗ)

| Поле | Значение |
|------|----------|
| **Назначение** | Контроль эскроу-счетов при продажах по 214-ФЗ (если посёлок продаётся по ДДУ — договору долевого участия). Банк замораживает деньги покупателя до регистрации права собственности. Нужна сверка: сколько на счету, когда раскрывается, статус |
| **Тип API** | Зависит от банка-эскроу. Сбербанк и ДОМ.РФ Банк имеют API для застройщиков. Часто — выгрузка через личный кабинет, не API |
| **Аутентификация** | Банковские токены или личный кабинет |
| **Sandbox** | Как правило, нет публичного sandbox |
| **Стоимость** | API бесплатно. Комиссия банка — по договору эскроу |
| **Сложность** | 4 / 5 — API нестандартны, часто требуется интеграция через банковских партнёров |
| **Приоритет MVP** | **P1** — если продажи идут по 214-ФЗ, статус эскроу должен быть в системе. Уточнить у Мартина: используется ли 214-ФЗ или иная схема продаж |

## Раздел 3. Принципы интеграций

Все без исключения ACL-клиенты реализуют следующие принципы. Отступление от принципа возможно только с обоснованием в ADR.

### 3.1 Idempotency (идемпотентность)

**Что это (простым языком):** одна и та же операция, выполненная несколько раз, даёт тот же результат, что и один раз. Как кнопка лифта — жми сколько угодно, лифт едет один раз.

**Зачем:** сеть ненадёжна. Если запрос ушёл, но ответ не вернулся — мы не знаем, выполнился ли он. Без идемпотентности повторная отправка чека может списать деньги дважды.

**Реализация:**
- Каждый исходящий запрос на изменение состояния (создание платежа, отправка чека, отправка документа) сопровождается уникальным `idempotency_key` в заголовке или теле запроса
- `idempotency_key` формируется на нашей стороне как `UUID` от бизнес-события (например, `payment_id` из нашей БД)
- Сохраняем в локальной таблице `integration_requests`: `(external_system, idempotency_key, status, response_body, created_at)`
- Перед повторной отправкой — проверяем таблицу. Если запрос уже успешен — не отправляем повторно

### 3.2 Retry с exponential backoff (повторные попытки с нарастающей задержкой)

**Что это (простым языком):** если внешний сервис недоступен — пробуем снова, но каждый раз ждём дольше: 1 сек → 2 сек → 4 сек → 8 сек. Как звонок другу: если не берёт — ждёшь немного дольше перед каждым следующим звонком, чтобы не надоедать.

**Реализация:**
```
Попытка 1:  сразу
Попытка 2:  через 1 секунду
Попытка 3:  через 2 секунды
Попытка 4:  через 4 секунды
Попытка 5:  через 8 секунд (максимум)
После 5 попыток: событие в очередь DLQ (dead letter queue) + алерт
```

**Когда НЕ повторяем:**
- HTTP 400 (Bad Request) — ошибка в наших данных, повтор бесполезен
- HTTP 401 / 403 — проблема с авторизацией, нужно вмешательство человека
- HTTP 422 (Unprocessable Entity) — валидация на стороне сервера

**Когда повторяем:**
- HTTP 429 (Too Many Requests) — ждём `Retry-After` из заголовка
- HTTP 500 / 502 / 503 / 504 — временная недоступность
- `ConnectionTimeout` / `ReadTimeout`

**Библиотека:** `tenacity` (Python) — декларативные политики retry через декораторы.

### 3.3 Circuit Breaker (автоматический выключатель)

**Что это (простым языком):** если внешний сервис падает снова и снова — перестаём его дёргать на какое-то время, чтобы не перегружать его и не тратить наши ресурсы. Как пробки в электросети — при перегрузке выбивает автомат, чтобы не сжечь проводку.

**Состояния выключателя:**
```
CLOSED (нормально)  → ошибки > порога → OPEN (выключен)
OPEN (ждём)         → через N секунд → HALF-OPEN (проверяем)
HALF-OPEN           → 1 успешный запрос → CLOSED
HALF-OPEN           → 1 ошибка → OPEN (снова)
```

**Параметры по умолчанию:**
- Порог открытия: 5 ошибок из последних 10 запросов
- Время ожидания в OPEN: 60 секунд
- Таймаут одного запроса: 10 секунд (настраивается per-integration)

**Библиотека:** `pybreaker` или собственная реализация на базе Redis (состояние выключателя — в Redis, чтобы было общим для нескольких воркеров).

### 3.4 Webhook Security (безопасность входящих вебхуков)

**Что это:** когда внешний сервис сам «приходит» к нам с уведомлением (Тинькофф сообщает об успешном платеже, Диадок — о подписанном документе), мы должны убедиться, что запрос действительно от него, а не от злоумышленника.

**Реализация для каждого вебхука:**
1. **Проверка подписи** — каждый провайдер подписывает тело запроса своим ключом (HMAC-SHA256 или аналог). Мы вычисляем подпись сами и сравниваем с пришедшей. Несовпадение — HTTP 403, инцидент в лог
2. **Белый список IP** — принимаем вебхуки только с известных IP-адресов провайдера (документация каждого сервиса содержит список)
3. **Временная метка** — отклоняем вебхуки старше 5 минут (защита от replay-атак)
4. **Идемпотентность** — вебхук может прийти дважды (провайдер не уверен, что мы получили). Проверяем `event_id` в нашей таблице `webhook_events` перед обработкой

**Эндпоинт для вебхуков:** `POST /webhooks/{provider}/` — для каждого провайдера отдельный путь. Никогда не смешиваем провайдеров в одном обработчике.

### 3.5 Логирование (без секретов)

Каждый исходящий запрос логируется на уровне `INFO`:
```
[INTEGRATION] POST tinkoff/acquiring → payment_id=12345 | status=200 | latency=342ms
[INTEGRATION] POST atol/receipt → idempotency_key=abc123 | status=202 | attempt=1
```

Никогда не логируем:
- API-ключи, токены, пароли
- Номера карт, CVV, PANs
- Персональные данные (ФИО + паспорт в одном месте)

Каждый ошибочный ответ логируется на уровне `ERROR` с телом ответа (если нет чувствительных данных).

### 3.6 Таймауты

| Тип операции | Таймаут по умолчанию |
|-------------|----------------------|
| Запрос нормализации (Дадата) | 3 секунды |
| Запрос платёжного шлюза | 10 секунд |
| Запрос ОФД (фискализация) | 15 секунд |
| Запрос государственного API | 30 секунд |
| Запрос ИИ (Anthropic/OpenAI) | 60 секунд |

Таймаут — через параметр `timeout` в `httpx.AsyncClient`. Все ACL-клиенты обязательно устанавливают явный таймаут. Запросы «без таймаута» запрещены.

### 4.2 Принцип dependency injection (внедрение зависимостей)

В боевом окружении (`ENV=production`) FastAPI-приложение получает реальный клиент:
```
PAYMENT_GATEWAY=tinkoff  →  TinkoffPaymentGateway
```

В тестовом окружении (`ENV=test`) — мок:
```
PAYMENT_GATEWAY=mock  →  TinkoffPaymentGatewayMock
```

Переключение — через `.env`. Код бизнес-логики не знает, с кем именно работает. Он вызывает только `PaymentGatewayPort.create_payment()`.

### 4.3 Таблицы в БД для интеграций

```sql
-- Журнал исходящих запросов
integration_requests (
    id              bigserial PRIMARY KEY,
    external_system varchar(50) NOT NULL,   -- 'tinkoff', 'atol', 'diadok'...
    operation       varchar(100) NOT NULL,  -- 'create_payment', 'send_receipt'...
    idempotency_key uuid NOT NULL UNIQUE,
    status          varchar(20) NOT NULL,   -- 'pending', 'success', 'failed', 'retrying'
    attempt_count   smallint DEFAULT 1,
    request_body    jsonb,                  -- без секретов
    response_body   jsonb,
    created_at      timestamptz DEFAULT now(),
    updated_at      timestamptz DEFAULT now()
);

-- Журнал входящих вебхуков
webhook_events (
    id              bigserial PRIMARY KEY,
    provider        varchar(50) NOT NULL,
    event_id        varchar(255) NOT NULL,  -- ID события от провайдера
    event_type      varchar(100) NOT NULL,
    payload         jsonb NOT NULL,
    processed       boolean DEFAULT false,
    received_at     timestamptz DEFAULT now()
);
-- Уникальный индекс: (provider, event_id) — защита от двойной обработки
```
