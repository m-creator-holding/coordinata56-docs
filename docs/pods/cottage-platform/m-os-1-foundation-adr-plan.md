## 0. Что изменилось в v3 (относительно v2)

v2 закрывал 4 решения Владельца от 2026-04-17 (единственная живая интеграция — Telegram; полный admin-UI конструктор; два раздельных event bus; per-company настройки). Развилки A/Б/В оставались открытыми.

**v3 закрывает развилки A, Б, В по ответам Владельца (msg 1111, 1113):**

### Развилка А закрыта — принят A3 гибрид

Структура таблиц Configuration-as-Data: общая мета-таблица `configuration_entities` + специализированные дочерние таблицы на каждую из 9 категорий. Подробно — в §ADR-0017.

### Развилка Б закрыта — принята Б2 миграция

Версионирование BPM-процессов: правки через UI применяются к запущенным экземплярам с правилами совместимости. Подробно — в §ADR-0017, подраздел «Migration rules для BPM».

### Развилка В закрыта — 7 полей в M-OS-1.1, 3 отложены в M-OS-1.2

Финальный набор `company_settings` для M-OS-1.1 и перечень отложенных. Подробно — в §ADR-0017.

### Что осталось без изменений

- ADR-0013, ADR-0014, ADR-0015, ADR-0016, ADR-0019 — без изменений относительно v2.
- Таблица волн — обновлена (развилки сняты, волна 3 разблокирована).
- Добавлен новый раздел §5 «Dependencies на frontend-director» (ADR-0018).

## 1. Контекст (без изменений с v2)

### Что уже закрыто ADR 0011

ADR 0011 (утверждён 2026-04-17) закрывает три из восьми элементов Foundation-блока:

- **А1 (Multi-company)** — модель `Company`, таблица `user_company_roles`, `company_id` во всех объектах, `CompanyScopedService`
- **А2 (Fine-grained RBAC)** — таблица `role_permissions`, функция `can(user, action, resource)`, декоратор `require_permission`
- **Crypto Audit (Е1)** — поля `prev_hash` / `hash` в `AuditLog`, endpoint `/api/v1/audit/verify`

### Что остаётся без ADR

1. **А3** — эволюция схемы без breaking changes → ADR-0013
2. **А4** — anti-corruption layer к внешнему миру → ADR-0014
3. **Реестр интеграций** — каталог подключений, живой только Telegram → ADR-0015
4. **Event bus базовый (С2)** — две шины: бизнес и agent-control → ADR-0016
5. **Configuration-as-data layer** — полный scope, фундамент admin-UI → ADR-0017
6. **Admin UI** — полный конструктор, под-фазы M-OS-1.1/1.2/1.3 → ADR-0018
7. **Pluggability** — DI-контракт для всех внешних точек расширения → ADR-0019

Итого ADR-кандидатов: **7**.

## 2. Таблица ADR-кандидатов

### ADR-0013 — Migrations Evolution Contract

| Поле | Значение |
|---|---|
| **Номер** | ADR-0013 |
| **Название** | Migrations Evolution Contract: backward-compatible schema changes |
| **Проблема** | Нет формального соглашения о том, какие изменения схемы БД допустимы без breaking changes; при росте до 5 подов и сотни таблиц произвольные миграции сломают работающие клиенты и поды |
| **Приоритет** | P0 — критично для M-OS-1.1 |
| **Кто декомпозирует** | backend-director → db-head |
| **Изменения после решений Владельца** | нет |

**Варианты решения:**

A. Свод запретов + принудительный CI-чек (expand/contract pattern): запрещены DROP COLUMN, RENAME COLUMN без deprecation-периода; все ломающие изменения — в два шага. Инструмент: `squawk` или кастомный линтер миграций.

B. Версионирование API совместно с миграциями: каждое изменение схемы сопровождается версией API (`/api/v2/`), старая версия живёт 2 спринта. Избыточно для внутренней системы.

C. Event sourcing полностью: схема никогда не ломается, всё через события. Слишком сложно для текущей стадии.

**Рекомендация**: вариант **A** — expand/contract pattern с CI-линтером. Ключевые правила: нельзя DROP/RENAME без deprecation, нельзя добавлять NOT NULL без DEFAULT, обязательный round-trip в CI.

**Зависимости**: ADR 0001 (модель данных v1), ADR 0011 (foundation migration)

### ADR-0014 — Anti-Corruption Layer Foundation

| Поле | Значение |
|---|---|
| **Номер** | ADR-0014 |
| **Название** | Anti-Corruption Layer: каркас адаптеров с жизненным циклом и безопасностью по умолчанию |
| **Проблема** | M-OS-1 требует одной живой интеграции (Telegram) и нескольких «на полке» (банки, ОФД, 1С, Росреестр, КриптоПро). Без изолирующего слоя внешние API просачиваются в ядро. Без явного жизненного цикла адаптера есть риск случайного сетевого вызова к промышленному endpoint-у из dev/test/CI. Владелец прямо запретил живые вызовы к банкам/Росреестру/1С/ОФД до production-gate. |
| **Приоритет** | P0 — критично для M-OS-1.1 |
| **Кто декомпозирует** | backend-director → backend-head |
| **Изменения после решений Владельца** | решение 1: три состояния адаптера, обязательный mock, CI-защита |

**Ключевые требования к ADR:**

1. **Три состояния адаптера**, явно различимые в коде и реестре:
   - `written` — код адаптера написан, интерфейс реализован, mock работает. Ни в одной среде не подключён.
   - `enabled` — адаптер зарегистрирован для конкретной `company_id` и конкретной среды (dev/staging/prod), credentials-ref назначен. Может быть enabled только в dev (тестирование с live-sandbox контрагента) без активации в prod.
   - `active_in_prod` — переведён в production после явного production-gate (проверка юристом + Владельцем). До этого — только `enabled` в non-prod.
2. **Feature flag на каждый адаптер** (`adapter.enabled: bool`), по умолчанию `False`. Хранится в `integration_catalog` (ADR-0015), не в `.env`.
3. **Запрет на ссылки на живые endpoint'ы в коде адаптера вне prod-конфигурации.** Base-URL берётся только из реестра (ADR-0015), не литералится. Проверяется линтером.
4. **Обязательный mock-режим для тестов.** Каждый адаптер имеет `MockAdapter`, возвращающий детерминированные ответы.
5. **Безопасность по умолчанию — отказ в сетевом вызове при `enabled=False`.** Обёртка `IntegrationAdapter.__call__` выбрасывает `AdapterDisabledError` до открытия сокета.
6. **CI-инспекция**: `test_dormant_adapters_make_no_network_calls`, `test_no_hardcoded_external_urls`, `test_adapter_catalog_consistency`.

**Рекомендация**: вариант **A** (ABC + три состояния + mock + CI).

**Зависимости**: ADR 0009, ADR-0013 (схема `integration_catalog`), ADR-0015

### ADR-0015 — Integration Registry

| Поле | Значение |
|---|---|
| **Номер** | ADR-0015 |
| **Название** | Integration Registry: каталог подключений, единственная живая интеграция — Telegram |
| **Проблема** | Настройки интеграций хранятся в `.env` и коде; администратор не может включить или отключить адаптер без деплоя; нет единого каталога. На M-OS-1 реестр заполняется всеми известными адаптерами, но живой — только Telegram. |
| **Приоритет** | P1 — нужно в ходе M-OS-1 |
| **Кто декомпозирует** | backend-director → backend-head |
| **Изменения после решений Владельца** | решение 1: Telegram — canonical; все остальные со статусом `enabled=False` и пустыми credentials-refs в seed-миграции |

**Рекомендация**: таблица `integration_catalog` в PostgreSQL + CRUD через admin-UI. Поля: `id`, `company_id` (nullable), `adapter_name`, `adapter_version`, `base_url`, `credentials_ref`, `enabled` (bool), `last_healthcheck_at`, `last_healthcheck_status`, аудит-поля. Seed-миграция заводит все адаптеры Vision §3.4, Telegram — `enabled=True`, остальные — `enabled=False`.

**Зависимости**: ADR-0014, ADR 0011, ADR-0013

### ADR-0016 — Dual Event Bus

| Поле | Значение |
|---|---|
| **Номер** | ADR-0016 |
| **Название** | Dual Event Bus: раздельные бизнес-шина и agent-control-шина |
| **Проблема** | Поды не должны вызывать друг друга напрямую (ADR 0009 §1). Параллельно ADR 0012 требует шину управления субагентами. Объединение двух шин в одну создаёт риск случайной подписки модуля на чужие события. Владелец принял решение о раздельных шинах. |
| **Приоритет** | P0 — критично для M-OS-1.1 |
| **Кто декомпозирует** | backend-director → backend-head |
| **Изменения после решений Владельца** | решение 3: две шины, разные контракты, правила изоляции |

**Ключевые требования:**

1. `business_events_bus` — таблица `business_events` (append-only), публикация через `BusinessEventBus.publish(BusinessEvent)`.
2. `agent_control_bus` — таблица `agent_control_events` (append-only), публикация через `AgentControlBus.publish(AgentControlEvent)`.
3. Разные base-классы событий: `BusinessEvent` и `AgentControlEvent`; Pydantic-валидация на входе в `publish()` блокирует передачу события не в ту шину.
4. Правила изоляции: бизнес-модули (`backend/app/services/*`, `pods/*`) импортируют только `BusinessEventBus`; orchestration — только `AgentControlBus`. CI-grep-проверка.
5. Мониторинг обеих шин: отдельные метрики, dashboard субагентов читает agent-control.

**Рекомендация**: вариант A (две таблицы, два PG-notifier канала, два интерфейса, CI-инварианты).

**Зависимости**: ADR 0009, ADR-0013, ADR 0012

#### Финальное решение по развилке А — A3 гибрид (утверждено Владельцем)

Структура хранения Configuration-as-Data: **общая мета-таблица `configuration_entities` + специализированные дочерние таблицы** на каждую из 9 категорий.

**Мета-таблица `configuration_entities`** содержит только служебные поля (не definition-данные):

| Поле | Тип | Назначение |
|---|---|---|
| `id` | UUID PK | уникальный идентификатор записи |
| `company_id` | UUID FK nullable | привязка к юрлицу (null = холдинговый уровень) |
| `type` | enum | категория: role / company_setting / bpm_process / form / report / doc_template / notification / business_rule / integration |
| `key` | varchar | бизнес-ключ внутри типа (например, `payment_overrun_limit_pct`) |
| `version` | integer | монотонно растёт при каждой правке |
| `status` | enum | draft / published / archived |
| `created_by` | UUID FK | ссылка на `users.id` |
| `updated_at` | timestamptz | время последней правки мета-записи |

**Специализированные дочерние таблицы** (по одной на категорию) хранят типизированный definition. Каждая таблица:
- имеет FK на `configuration_entities.id` (NOT NULL, каскадное удаление запрещено — только soft archive);
- содержит поля, специфичные для категории (например, `bpm_process_steps` — JSON-схема шагов; `company_settings` — конкретные числовые/строковые поля; `form_fields` — JSON-массив дескрипторов поля);
- может иметь FK-связи между категориями (например, `bpm_process_definitions.form_id` FK → `configuration_entities.id WHERE type='form'`).

**Запрос «дай активную версию`** читает JOIN: `configuration_entities JOIN <child_table> ON id WHERE company_id=? AND key=? AND status='published'`. Производительность: составной индекс `(company_id, type, key, status)` на мета-таблице; child-таблицы индексируются по FK.

**Почему не A1 и не A2:**
- A1 (единая таблица с JSONB) — сложная валидация, нет FK между категориями, тяжело мигрировать definition-схему.
- A2 (только раздельные таблицы) — механизм версионирования и hot-reload дублируется 9 раз, нет единой точки audit-chain.
- A3 (гибрид) — единый механизм версий и аудита в мета-таблице + типизация в дочерних.

#### Финальное решение по развилке Б — Б2 миграция (утверждено Владельцем)

**Правки BPM-процесса через UI применяются к запущенным экземплярам** с правилами совместимости. Это сложнее в реализации, но правильнее для бизнеса (устранённая ошибка в процессе немедленно действует на все заявки в очереди).

##### Migration rules для BPM — обязательный подраздел ADR-0017

ADR-0017 должен содержать явный перечень правил совместимости. Базовая классификация (для команды — «что считается безопасной миграцией, а что требует решения»):

**Safe migration — применяется автоматически ко всем запущенным экземплярам:**

1. Изменение текстовых label/description шага — ни на что не влияет, применяется мгновенно.
2. Добавление нового необязательного шага в конец процесса — экземпляры, уже прошедшие предыдущий последний шаг, считаются завершёнными; новые экземпляры проходят новый шаг.
3. Изменение акторов (разрешённых ролей) на шаге — применяется к текущим активным шагам мгновенно (уже назначенные задачи переназначаются согласно новым ролям).
4. Изменение таймеров эскалации — применяется ко всем незакрытым шагам, которые ещё не вышли за таймер.
5. Изменение шаблона уведомления шага — применяется ко всем будущим уведомлениям этого шага.

**Requires user decision (merge-стратегия выбирается администратором в UI):**

6. Добавление обязательного нового шага в середину процесса — запущенные экземпляры, уже миновавшие точку вставки: три стратегии (пропустить новый шаг / вставить задачу немедленно в очередь / заблокировать до ручного решения). Администратор выбирает стратегию при публикации.
7. Удаление шага — экземпляры, находящиеся на удалённом шаге: три стратегии (перейти на следующий / завершить процесс / заблокировать).
8. Изменение условий ветвления decision gate — экземпляры, стоящие перед gate, перевычисляют условие по новой логике. Предупреждение в UI с preview-списком затронутых экземпляров.
9. Смена порядка шагов — требует явного указания mapping «старый шаг N → новый шаг M»; без mapping блокируется.

**Hard incompatible — запрещено через UI, только через code review + миграцию:**

10. Переименование ключевого идентификатора шага (`step_id`) — нарушает state-machine текущих экземпляров.
11. Полная замена структуры процесса (более 50% шагов) — UI предлагает создать новую версию как отдельный процесс, старый архивировать.

**Тесты на корректность миграции состояния** (DoD):
- `test_safe_migration_applies_to_running_instances` — smoke по каждому из 5 safe-правил.
- `test_user_decision_migration_blocks_without_strategy` — без выбора стратегии publish не проходит.
- `test_hard_incompatible_blocked_in_ui` — 7 из 7 hard cases возвращают ошибку.

#### Остальные разделы ADR-0017 (без изменений с v2)

**Версионирование конфигурации** — каждая правка = новая строка в child-таблице с `version = N+1`. Статус: draft / published / archived. Активна одна `published` на ключ + company. Rollback — сменить статус версий. AuditLog с hash-chain (ADR 0011) на каждую смену статуса и каждый publish.

**Hot-reload** — in-process кэш с инвалидацией через `business_events_bus` (событие `ConfigurationPublished`). TTL-фоллбек 60 сек.

**Защита от гонок** — optimistic lock по `version` (409 Conflict при несовпадении). Soft-lock на BPM draft-процессы (30 минут, не принудительный).

**Производительность** — `ConfigurationCache` (LRU in-memory, ~10k definitions, pluggable по ADR-0019).

**DoD для ADR-0017:**
- Все 9 категорий с Pydantic-схемами.
- Мета-таблица + 9 дочерних таблиц описаны с миграцией Alembic.

## 5. Dependencies на frontend-director (новый раздел, v3)

ADR-0018 (Admin UI Constructor) не может быть написан как завершённый документ без участия frontend-director. Ниже перечислены конкретные решения, которые блокируются до его активации.

### Блокирующие решения (ADR-0018 нельзя финализировать без них)

| Решение | Почему блокирует | Кто принимает |
|---|---|---|
| Выбор канвас-редактора для BPM-конструктора (M-OS-1.2) | `bpmn-js` vs альтернатива влияет на backend-контракт (формат JSON процесса, API импорта/экспорта BPMN-XML) | frontend-director + backend-director совместно |
| Выбор form-builder библиотеки (M-OS-1.3) | `formio` vs кастомный влияет на схему хранения form-definition в `configuration_entities` child-таблице | frontend-director + backend-director совместно |
| PWA или классический SPA | влияет на требования к API (offline support, service workers, push-уведомления через Telegram vs in-app) | frontend-director + Координатор |
| Оценка трудоёмкости под-фаз 1.1/1.2/1.3 со стороны фронтенда | нужна для согласования общего roadmap M-OS-1 | frontend-director |
| Порядок параллелизации backend/frontend в каждой под-фазе | какие под-фазы можно распараллелить, какие строго последовательны | frontend-director + backend-director |

### Неблокирующие решения (ADR-0018 можно писать без них, но надо зафиксировать как «TBD»)

| Решение | Когда нужно |
|---|---|
| UI-фреймворк (React vs Vue vs другой) | нужно до старта кода M-OS-1.1, не до ADR |
| UX-паттерны inbox, preview, publish (визуальный уровень) | согласуется с design-director отдельно, ADR-0018 фиксирует только функциональные требования к паттернам |
| Компонентная библиотека | до старта M-OS-1.1 frontend |

### Рекомендация

Активировать frontend-director параллельно с написанием ADR-0013 и ADR-0014 (Волна 1) — чтобы к началу Волны 4 (ADR-0018) у него был готов PWA-план. Без этого Волна 4 рискует растянуться: backend-director и backend-head не могут зафиксировать API-контракты admin-UI без знания PWA-требований.

| Seed-миграция ADR-0015 фиксирует адаптеров без согласования | Средняя | Среднее | Список адаптеров seed-а согласуется с Владельцем до миграции |

| **[новый v3]** frontend-director не активирован — Волна 4 заблокирована | Средняя | Высокое | Активировать параллельно с Волной 1 (рекомендация §5) |
