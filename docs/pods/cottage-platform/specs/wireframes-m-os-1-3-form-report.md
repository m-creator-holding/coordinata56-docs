## 0. Основание и источники

Этот бриф — третья дизайн-задача отдела в M-OS. Form Builder и Report Builder входят в скоуп M-OS-1.3 согласно `docs/pods/cottage-platform/m-os-1-plan.md` v1.3 §«Уточнение по разбивке» (цитата: «M-OS-1.3 — form builder, report builder, business rule engine»).

**Предшественник:** wireframes-m-os-1-2-bpm.md должны быть завершены до старта этой задачи. AdminLayout, навигационная карта и конвенции оформления — из M-OS-1.1 и M-OS-1.2, designer не переизобретает.

**Обязательные источники — designer читает до начала работы:**

1. `docs/pods/cottage-platform/m-os-1-plan.md` v1.3 — §M-OS-1.3 скоуп, список что НЕ входит, DoD.
2. `docs/pods/cottage-platform/specs/wireframes-m-os-1-1-admin.md` — конвенции AdminLayout.
3. `docs/pods/cottage-platform/specs/wireframes-m-os-1-2-bpm.md` — конвенции навигации, структура группы «Конструктор» в sidebar.
4. `docs/adr/0011-foundation-multi-company-rbac-audit.md` — multi-company контекст: формы и отчёты — per-company (изолированы по company_id).
5. `docs/agents/CODE_OF_LAWS.md` ст. 45а — живые интеграции в источниках данных отчётов запрещены (отчёт читает только внутреннюю БД, не внешние API банков/1С).
6. `/root/coordinata56/CLAUDE.md` секция «UX > визуальный дизайн» — структура и user flows, без полиша.

**Важная граница скоупа:**
- Form Builder на M-OS-1.3 — drag-and-drop конфигуратор полей формы (configuration-as-data). Полноценный рендеринг форм для конечных пользователей (не admin) — M-OS-2.
- Report Builder — конфигуратор отчётов: выбор источника данных, полей, группировки, фильтров. Визуализация (графики, диаграммы) — M-OS-2. На M-OS-1.3 — только табличный формат.
- Business Rule Engine (BRE) — отдельная задача вне этого брифа, не включать.

## 1. Формат выходного артефакта

Один файл: `docs/pods/cottage-platform/specs/wireframes-m-os-1-3-form-report.md`.

Для каждого из 5 экранов обязательны все шесть блоков (идентично M-OS-1.1 и M-OS-1.2):

```
### Экран N. <Название> — <путь>

#### Persona и сценарий
#### Text-based wireframe
#### Поля формы / колонки таблицы
#### Состояния UI
#### Связи с другими экранами
#### Ссылка на OpenAPI
```

В конце файла — раздел **«Навигационная карта»**: как 5 экранов встраиваются в sidebar AdminLayout, расширяя группу «Конструктор» из M-OS-1.2.

## 2. Ограничения и правила работы

| Правило | Детали |
|---|---|
| Только markdown | Wireframes — текстом, ASCII-схемы, таблицы. |
| Не визуальный полиш | Никаких цветов, брендинга, иконок. |
| Компоненты — shadcn/ui | `DragDropContext` (react-beautiful-dnd или аналог через shadcn), `Table`, `Sheet`, `Dialog`, `Form`, `Select`, `Switch`, `Badge`, `Card`, `Tabs`. Если компонент не в shadcn — описать текстом поведение, не изобретать visual-кастомный. |
| Tailwind, RR6 | Стек фиксирован ADR 0002. |
| 5 экранов и только они | Не включать: Business Rule Engine, BPM-движок, Notification Channels, Document Templates — они в других брифах. |
| Только внутренние источники данных | В Report Builder источники — только внутренние сущности M-OS (Payment, Contract, Project и т.д.), не внешние API. Ст. 45а CODE_OF_LAWS. |
| Не пишет код | Designer пишет описания, не TSX/HTML. |
| drag-and-drop — текстовое описание | Designer описывает drag-and-drop поведение текстом и ASCII-схемой, не ожидая, что это будет готовый figma-прототип. |

### Экран 1. Form Builder — Список форм — P0

**Путь:** `/admin/forms`

**Контекст:** список всех форм ввода данных, созданных администратором. Формы используются в BPM-процессах (как human_task с формой ввода) и как самостоятельные экраны ввода данных. На M-OS-1.3 — configuration-as-data: форма хранится в БД как JSON-схема полей.

**Что нужно спроектировать:**

**Список форм:**
- Таблица: название, описание, число полей, статус (черновик / активна / архив), дата изменения, связанные BPM-процессы (число).
- Системные формы (seed): «Согласование платежа», «Акт приёмки работ», «Создание договора» — badge «Системная».
- Кнопка «+ Создать форму».
- Строка кликабельна → редактор формы.
- Фильтр по статусу (chips или Select).

**Действия над строкой:**
- Редактировать, Дублировать, Архивировать, Предпросмотр.

**Связь с BPM:** в строке формы — ссылка «N процессов» → список шаблонов BPM, использующих эту форму (всплывающий Popover или Sheet со списком названий).

### Экран 2. Form Builder — Редактор формы — P0

**Путь:** `/admin/forms/:id/edit` (или `/admin/forms/new` при создании)

**Контекст:** главный UX этого брифа. Редактор формы — drag-and-drop конфигуратор полей. Администратор перетаскивает типы полей из палитры на холст, настраивает их свойства. На выходе — JSON-схема формы в БД.

**Что нужно спроектировать — трёхпанельный layout:**

```
┌───────────────┬──────────────────────────────────────┬───────────────────┐
│  Палитра      │  Холст (порядок полей)               │  Свойства поля    │
│  элементов    │                                      │  (правая панель)  │
│               │  [Перетащите элемент сюда]           │                   │
│  Текстовое    │                                      │  Выбранное поле:  │
│  поле         │  1. Название (Input)          [ ✎ ✕]│  Label: Название  │
│               │     обязательное                     │  Тип: text        │
│  Число        │                                      │  Обязательное: ☑  │
│               │  2. Сумма (Number)            [ ✎ ✕]│  Placeholder: ... │
│  Дата         │     руб., мин 0                      │  Валидация: ...   │
│               │                                      │                   │
│  Выпадающий   │  3. Статус (Select)           [ ✎ ✕]│  [Сохранить поле] │
│  список       │     Опции: активен / архив           │                   │
│               │                                      │                   │
│  Чекбокс      │  [+ Добавить раздел/группу]          │                   │
│               │                                      │                   │
│  Файл         │                                      │                   │
│               │                                      │                   │
│  Текстовый    │                                      │                   │
│  блок (info)  │                                      │                   │
└───────────────┴──────────────────────────────────────┴───────────────────┘
```

**Типы полей в палитре:**

| Тип | Описание | Параметры (в правой панели) |
|---|---|---|
| Текстовое поле | `Input` text | Label, Placeholder, Обязательное, Мин/Макс длина, Паттерн regex |
| Многострочный текст | `Textarea` | Label, Placeholder, Обязательное, Мин/Макс строк |
| Число | `Input` number | Label, Мин, Макс, Шаг, Единица измерения (руб./м²/шт.) |
| Дата | `DatePicker` | Label, Мин/Макс дата (абсолютная или «сегодня + N дней»), Обязательное |
| Выпадающий список | `Select` | Label, Варианты (список пар ключ-значение, редактируемый), Обязательное, Мульти-выбор |
| Чекбокс | `Checkbox` | Label, Текст чекбокса, Значение по умолчанию |
| Файл | `Input` type=file | Label, Допустимые типы (PDF/PNG/JPG), Макс размер (МБ). Заглушка на M-OS-1.3 — файловое хранилище в M-OS-2, кнопка загрузки активна, но показывает banner. |
| Информационный блок | `Alert` read-only | Текст для пользователя (инструкция, предупреждение) |

**Метаданные формы (шапка редактора):**

| Поле | Тип UI | Обязательность |
|---|---|---|
| Название формы | `Input` | Да |
| Описание | `Textarea` | Нет |
| Статус | `Select`: Черновик / Активна / Архив | Да |

**Действия (footer редактора):**
- «Сохранить как черновик» (без валидации — любое неполное состояние).
- «Опубликовать» (валидация: у каждого поля должен быть Label, хотя бы одно поле в форме).
- «Предпросмотр» → переход на Экран 5 в новой вкладке.
- «Отменить» → возврат в список (Dialog подтверждения, если есть несохранённые изменения).

**Группировка полей:** designer проектирует механизм «Добавить раздел» — именованный блок, внутри которого можно группировать поля. Пример: раздел «Банковские реквизиты» содержит поля BIK, счёт, банк.

**Важно:** designer описывает поведение drag-and-drop текстом: «пользователь перетаскивает тип поля из палитры на холст — поле появляется в конце списка или между двумя существующими полями, если бросить между ними; поля на холсте можно перетаскивать для изменения порядка».

### Экран 3. Report Builder — Список отчётов — P1

**Путь:** `/admin/reports`

**Контекст:** список конфигураций отчётов. Отчёт — это конфигурация (источник данных + поля + фильтры + группировка), хранящаяся в БД. Просмотр данных отчёта — через «Запустить отчёт» (переход на отдельный экран рендеринга). Экспорт (PDF, Excel) — M-OS-2.

**Что нужно спроектировать:**

**Список отчётов:**
- Таблица: название, источник данных (Payment / Contract / Project / ...), последний запуск, статус (черновик / активен), дата изменения.
- Системные отчёты (seed): «Платежи за период», «Договоры по объекту», «Задолженность по подрядчикам».
- Кнопка «+ Создать отчёт».
- Строка кликабельна → редактор отчёта.

**Действия над строкой:**
- Редактировать, Запустить (переход к просмотру данных), Дублировать, Архивировать.

---

### Экран 4. Report Builder — Редактор отчёта — P1

**Путь:** `/admin/reports/:id/edit`

**Контекст:** конфигуратор отчёта. Администратор выбирает источник данных, поля для показа, фильтры, группировку. На выходе — JSON-конфигурация в БД. Выполнение отчёта (SQL-запрос к БД) — бэкенд, не admin-UI.

**Что нужно спроектировать — четырёхшаговый wizard или single-page с секциями:**

Designer выбирает один из двух подходов и обосновывает:

**Вариант A (рекомендуется): single-page с секциями** — всё видно сразу, переходы между секциями без кнопок «Далее».

**Вариант B: wizard (Шаг 1 → Шаг 2 → Шаг 3 → Шаг 4)** — проще для новичка, но медленнее для опытного пользователя.

**Секции (независимо от выбора подхода):**

**Секция 1 — Источник данных:**

| Поле | Тип UI | Замечание |
|---|---|---|
| Основной источник | `Select`: Payment / Contract / Project / Contractor / AuditLog / ... | Каждый источник — внутренняя таблица M-OS |
| Компания | `Select`: «Все компании» / конкретная компания | Фильтр multi-company: owner видит все, accountant — только свою |
| Связанные данные | `Checkbox-группа` | Если источник Payment → можно добавить данные Contract (join); если Contract → данные Project и Contractor |

**Секция 2 — Поля отчёта:**

Список доступных полей источника (зависит от выбора в секции 1). Каждое поле — `Checkbox` + порядок (drag-and-drop или стрелки вверх/вниз).

```
Доступные поля:                   Выбранные поля:
─────────────────                  ──────────────────────────────────
[ ] amount_cents                   [x] paid_at            ↑ ↓
[x] paid_at                        [x] amount_cents       ↑ ↓
[ ] payment_method                 [x] contractor_name    ↑ ↓
[x] contractor_name                [x] status             ↑ ↓
[x] status
[ ] approved_by
```

Переименование колонки: рядом с каждым выбранным полем — `Input` для пользовательского названия (по умолчанию — техническое имя поля).

**Секция 3 — Фильтры:**

Список условий фильтрации. Каждое условие:

| Элемент | Тип UI |
|---|---|
| Поле | `Select` (из источника) |
| Оператор | `Select`: = / != / > / < / >= / <= / содержит / не содержит / в диапазоне |
| Значение | Динамически: `Input` / `DatePicker` / `Select` в зависимости от типа поля |
| Удалить условие | кнопка ✕ |

Кнопка «+ Добавить фильтр». Логика между условиями: `AND` (на M-OS-1.3 только AND; OR — M-OS-2).

**Секция 4 — Группировка и сортировка:**

| Поле | Тип UI | Замечание |
|---|---|---|
| Группировать по | `Select` (поле из источника, опционально) | Например: «группировать по contractor_name» → агрегация сумм по подрядчику |
| Агрегация | `Select`: Сумма / Среднее / Минимум / Максимум / Количество | Применяется к числовым полям при группировке |
| Сортировка | Список: поле + направление (ASC/DESC) | Порядок строк в результате |
| Лимит строк | `Input` number | Максимум строк (0 = все) |

**Метаданные (шапка):**

| Поле | Тип UI |
|---|---|
| Название отчёта | `Input` |
| Описание | `Textarea` |
| Статус | `Select`: Черновик / Активен |

**Действия (footer):**
- «Сохранить».
- «Запустить отчёт» → переход к просмотру результата (выполняет запрос к БД по текущей конфигурации).
- «Отменить».

**Просмотр результата отчёта** (inline preview в редакторе — кнопка «Предпросмотр»):

Таблица с выбранными колонками + первые 10 строк. Кнопка «Посмотреть полный отчёт» — открывает отдельную страницу `/admin/reports/:id/run`.

### Экран 5. Form Preview (Предпросмотр формы) — P2

**Путь:** `/admin/forms/:id/preview`

**Контекст:** admin-only просмотр того, как форма будет выглядеть для конечного пользователя. Данные в M-OS-БД не сохраняются (режим preview). Предпросмотр отрабатывает JSON-схему формы как рендер.

**Что нужно спроектировать:**

**Шапка предпросмотра:**
- Баннер: «Режим предпросмотра — данные не сохраняются».
- Название формы.
- Кнопка «Вернуться к редактору».

**Тело формы:**
- Рендер всех полей из JSON-схемы: в том порядке, в котором они расположены на холсте. С разделами (если есть).
- Поля с обязательностью — помечены звёздочкой.
- Placeholder-ы как настроены.

**Кнопки формы (dummy — не отправляют данные):**
- «Отправить» (disabled или при нажатии — toast «Это предпросмотр, данные не сохранены»).
- «Сбросить».

**Состояния:**
- Если форма пустая (нет полей) — empty-state: «Форма не содержит полей. Добавьте поля в редакторе.»

## 4. Навигационная карта

Предлагаемое расширение sidebar AdminLayout (дополняет M-OS-1.2):

```
M-OS Admin
──────────────────────────────────────────────────────
(существующие пункты M-OS-1.1 и M-OS-1.2)
Управление / Права и настройки / Система / Безопасность

Конструктор
  ├── Шаблоны процессов (BPM)    /admin/bpm/definitions     (из M-OS-1.2)
  ├── Каналы уведомлений          /admin/notifications        (из M-OS-1.2)
  ├── Шаблоны документов          /admin/document-templates   (из M-OS-1.2)
  ├── Формы ввода                 /admin/forms                (новое, M-OS-1.3)
  └── Отчёты                      /admin/reports              (новое, M-OS-1.3)
```

**Схема переходов:**

- Список форм → строка → Редактор формы
- Список форм → «+ Создать» → Редактор новой формы
- Редактор формы → «Предпросмотр» → Form Preview (новая вкладка)
- Form Preview → «Вернуться к редактору» → Редактор формы
- Редактор формы → поле «связанные процессы» → Список шаблонов BPM (M-OS-1.2 экран 1)
- Список отчётов → строка → Редактор отчёта
- Список отчётов → «Запустить» → `/admin/reports/:id/run` (страница просмотра данных)
- Редактор отчёта → «Запустить отчёт» → `/admin/reports/:id/run`

## 7. Definition of Done для designer

Wireframes считаются готовыми к передаче ux-head, когда:

1. Файл `docs/pods/cottage-platform/specs/wireframes-m-os-1-3-form-report.md` создан.
2. Все 5 экранов содержат все 6 обязательных блоков.
3. Раздел «Навигационная карта» заполнен: расширение sidebar + схема переходов.
4. Все принятые решения по выбору из вариантов — обоснованы текстом.
5. Не содержит Business Rule Engine (отдельная задача).
6. Form Builder: drag-and-drop описан текстом с ASCII-схемой, все типы полей из палитры описаны.
