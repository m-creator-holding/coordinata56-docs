## 0. Основание и источники

Этот бриф — вторая дизайн-задача отдела в M-OS. Является продолжением M-OS-1.1 admin wireframes. BPM Constructor входит в скоуп M-OS-1.2 согласно `docs/pods/cottage-platform/m-os-1-plan.md` v1.3 §M-OS-1.2 и §«Уточнение по разбивке».

**Предшественник:** wireframes-m-os-1-1-admin.md должны быть завершены до старта этой задачи (AdminLayout уже определён — designer переиспользует без изменений).

**Обязательные источники — designer читает до начала работы:**

1. `docs/pods/cottage-platform/m-os-1-plan.md` v1.3 — §M-OS-1.2 скоуп, §M-OS-1.3 (BPM-движок — что НЕ входит в admin-UI M-OS-1.2), §«Уточнение по разбивке».
2. `docs/pods/cottage-platform/specs/wireframes-m-os-1-1-admin.md` — готовые wireframes M-OS-1.1 для понимания навигационной структуры AdminLayout и конвенций оформления.
3. `docs/adr/0011-foundation-multi-company-rbac-audit.md` — модель AuditLog (crypto-chain), модель `company_settings` — designer понимает, что BPM работает внутри этих ограничений.
4. `docs/agents/CODE_OF_LAWS.md` ст. 45а — запрет живых интеграций. В разделе уведомлений (экран 2) живой UI — только для Telegram-канала. SMS, email — карточки со статусом `disabled`.
5. `/root/coordinata56/CLAUDE.md` секция «UX > визуальный дизайн» — MVP: структура и user flows, без полиша.

**Важная граница:** BPM-движок (исполняющий экземпляры) — это M-OS-1.3, не сейчас. В M-OS-1.2 designer проектирует **только конфигуратор шаблонов BPM** (работа с данными через admin-UI). Полноценный bpmn-js drag-and-drop canvas — M-OS-2. На M-OS-1.2 — форма или raw JSON-редактор в admin-панели.

## 1. Формат выходного артефакта

Один файл: `docs/pods/cottage-platform/specs/wireframes-m-os-1-2-bpm.md`.

Для каждого из 4 экранов обязательны все шесть блоков (идентично M-OS-1.1):

```
### Экран N. <Название> — <путь>

#### Persona и сценарий
#### Text-based wireframe
#### Поля формы / колонки таблицы
#### Состояния UI
#### Связи с другими экранами
#### Ссылка на OpenAPI
```

В конце файла — раздел **«Навигационная карта»**: как эти 4 экрана встраиваются в существующий sidebar AdminLayout из M-OS-1.1 (добавляются новые пункты в группу «Конструктор» или аналог — designer решает и обосновывает).

## 2. Ограничения и правила работы

| Правило | Детали |
|---|---|
| Только markdown | Wireframes — текстом, ASCII-схемы, таблицы. |
| Не визуальный полиш | Никаких цветов, брендинга, иконок — только структура. |
| Компоненты — shadcn/ui | `Table`, `Sheet`, `Dialog`, `Form`, `Select`, `Switch`, `Textarea`, `Badge`, `Card`, `Accordion`. |
| Tailwind, RR6 | Стек фиксирован ADR 0002. |
| 4 экрана и только они | Не включать: BPM-движок (запуск экземпляров процесса), Business Rule Engine, Form Builder, Report Builder — всё это отдельные задачи. |
| Telegram — единственная живая нотификация | В экране 2 (Notification Channels) живой UI — только для Telegram. |
| Не пишет код | Designer пишет описания, не TSX/HTML. |
| Без bpmn-js canvas | Полноценный drag-and-drop — M-OS-2. Здесь — форменный редактор или JSON-редактор. |

### Экран 1. ProcessDefinition (Шаблоны BPM) — P0

**Путь:** `/admin/bpm/definitions`

**Контекст:** `ProcessDefinition` — шаблон процесса как данные в БД. На M-OS-1.2 admin-UI позволяет просматривать, создавать, редактировать и версионировать шаблоны. Фактическое исполнение экземпляров — движок M-OS-1.3. Три обязательных шаблона (из m-os-1-plan.md §M-OS-1.3): Payment-approval, Work-acceptance, Contract-creation — должны быть представлены в списке как системные (seed-данные).

**Что нужно спроектировать — три режима:**

**Режим 1 — Список шаблонов:**
- Таблица: название, тип процесса, версия, статус (активен / черновик / архив), дата изменения.
- Системные шаблоны (seed) — помечены badge «Системный»: редактировать разрешено (название, описание, шаги), удалять — нельзя.
- Кнопка «+ Создать шаблон».
- Строка кликабельна → режим просмотра/редактирования.

**Режим 2 — Редактор шаблона:**

Designer выбирает ОДИН из двух подходов и обосновывает:

**Вариант A (рекомендуется для MVP):** структурная форма — шаги добавляются как список с полями:
```
Шаг 1: [Тип: human_task ▼] [Название: "Согласование бухгалтером"] [Роль: accountant ▼] [Timeout: 24 ч]
  [+ Добавить условие перехода]
  [+ Добавить шаг]
```

**Вариант B:** raw JSON-редактор (`Textarea` с моноширинным шрифтом) + кнопка «Валидировать схему». Проще реализовать, менее дружелюбен для бизнес-пользователя.

Обязательные поля шаблона:

| Поле | Тип UI | Обязательность | Замечание |
|---|---|---|---|
| Название | `Input` | Да | — |
| Описание | `Textarea` | Нет | Кратко: для чего этот процесс |
| Тип объекта | `Select`: Payment / Contract / WorkAct / Custom | Да | К какому объекту привязывается экземпляр |
| Версия | Только для чтения | — | Автоинкремент при сохранении нового варианта |
| Статус | `Select`: Черновик / Активен / Архив | Да | Только активный шаблон можно запустить |
| Шаги процесса | Список (Вариант A) или JSON (Вариант B) | Да | — |

Поля шага (`ProcessStep`):

| Поле | Тип UI | Замечание |
|---|---|---|
| Тип шага | `Select`: human_task / timer / auto | — |
| Название шага | `Input` | — |
| Роль-исполнитель | `Select` (список ролей из /admin/roles) | Только для human_task |
| Timeout (часы) | `Input` number | Только для human_task и timer; 0 = без ограничения |
| Действие при эскалации | `Select`: уведомить / завершить с ошибкой / передать следующей роли | Только при Timeout > 0 |

**Режим 3 — История версий шаблона:**

Таблица: версия, автор изменения, дата, комментарий. Кнопка «Восстановить» у каждой версии (копирует содержимое версии в новый черновик, не перезаписывает).

**Связь с M-OS-1.3:** в списке шаблонов (режим 1) — колонка «Активных экземпляров» (пока заглушка с 0, движок добавится в M-OS-1.3). Designer проектирует колонку, чтобы frontend не переделывал таблицу.

### Экран 2. Notification Channels (Каналы уведомлений) — P1

**Путь:** `/admin/notifications`

**Контекст:** настройка того, какие события через какие каналы уведомляют пользователей. На M-OS-1.2 живой канал — только Telegram (CODE_OF_LAWS ст. 45а). Email и SMS — карточки со статусом `disabled` («будет доступно после production-gate»).

**Что нужно спроектировать:**

**Блок 1 — Каталог каналов** (аналогично Integration Registry из M-OS-1.1):

```
┌─────────────────────┐  ┌─────────────────────┐  ┌─────────────────────┐
│  Telegram           │  │  Email (SMTP)         │  │  SMS                │
│  ● Активен          │  │  ○ Недоступно         │  │  ○ Недоступно       │
│  [Настроить]        │  │  production-gate      │  │  production-gate    │
└─────────────────────┘  └─────────────────────┘  └─────────────────────┘
```

**Блок 2 — Матрица событий × каналы** (основная UX этого экрана):

Таблица: строки — события системы, столбцы — каналы (Telegram, Email, SMS). Ячейка — `Switch` (включить/выключить уведомление по этому каналу для этого события). Неактивные каналы — disabled switches с tooltip «Канал не настроен».

Примеры событий (designer проектирует с учётом расширения):
- Платёж ожидает согласования
- Платёж одобрен / отклонён
- Договор создан / подписан
- Шаг BPM-процесса ожидает действия
- BPM-эскалация (timeout истёк)
- Ошибка системы (уровень critical)
- Новый пользователь добавлен

Группировка событий по категориям: «Финансы», «Документы», «Процессы BPM», «Система».

**Блок 3 — Настройка Telegram-канала (если не настроен — баннер):**

Если Telegram не настроен (не сконфигурирован в Integration Registry) — баннер: «Настройте Telegram-интеграцию в Реестре интеграций, затем вернитесь сюда». Ссылка на `/admin/integrations`.

Если Telegram настроен — краткая сводка: chat_id, статус бота, кнопка «Отправить тест».

### Экран 3. Document Templates (Шаблоны документов) — P2

**Путь:** `/admin/document-templates`

**Контекст:** шаблоны для генерации печатных/PDF документов: договор, акт КС-2, счёт-фактура. На M-OS-1.2 — configuration-as-data: шаблон хранится в БД как структура (не файл). Генерация PDF-файлов — M-OS-2.

**Что нужно спроектировать:**

**Список шаблонов:**
- Таблица: название, тип документа, версия, статус, дата изменения.
- Системные шаблоны (seed): «Договор подряда», «Акт приёмки КС-2», «Счёт на оплату» — badge «Системный».
- Кнопка «+ Создать шаблон».

**Редактор шаблона:**

Упрощённый — только метаданные на M-OS-1.2 (полноценный редактор тела шаблона — M-OS-2):

| Поле | Тип UI | Замечание |
|---|---|---|
| Название | `Input` | — |
| Тип документа | `Select`: Договор / Акт / Счёт / Другое | — |
| Описание | `Textarea` | — |
| Переменные шаблона | Список «ключ → описание» | Например: `{{company_name}}` → «Название компании». Designer проектирует простую таблицу для добавления пар. |
| Статус | `Select`: Черновик / Активен / Архив | — |

**Важный UX:** раздел «Тело шаблона» (rich text / HTML / Jinja2) — placeholder с баннером: «Редактор тела шаблона появится в M-OS-2. Сейчас вы можете определить метаданные и переменные.»

### Экран 4. Audit Log Viewer (Просмотр аудит-лога) — P2

**Путь:** `/admin/audit`

**Контекст:** просмотр записей AuditLog с крипто-цепочкой (ADR 0011). Только чтение — никаких действий с записями. Ключевой UX: пользователь должен понимать, что лог криптографически защищён и его нельзя подделать.

**Что нужно спроектировать:**

**Фильтры (над таблицей):**

| Фильтр | Тип UI | Замечание |
|---|---|---|
| Компания | `Select` | Фильтр по company_id |
| Пользователь | `Select` с поиском | — |
| Тип сущности | `Select`: Payment / Contract / User / Role / ... | — |
| Действие | `Select`: create / update / delete / approve / reject / ... | — |
| Период | `DateRangePicker` | — |

**Таблица аудит-лога:**

Колонки: дата/время, пользователь, компания, действие, тип сущности, ID сущности, IP-адрес, статус хэша.

**Колонка «Статус хэша»** — ключевая:
- `Badge` зелёный «OK» — хэш этой записи корректен относительно предыдущей.
- `Badge` красный «Нарушение» — хэш не совпадает (запись была изменена). Designer описывает, что при нажатии на «Нарушение» открывается `Dialog` с техническими деталями.

**Детальная запись (раскрывается по клику на строку):**
- `changes_json` — показывается как форматированный JSON (`pre` блок).
- `hash` — сокращённый (первые 12 символов + «...»), полный при наведении tooltip.
- Ссылка «Показать связанный объект» → переход на страницу сущности.

**Кнопка «Проверить целостность цепочки»:** запускает backend-валидацию всей цепочки (GET /api/v1/audit/verify). Прогресс-индикатор, результат: «Цепочка целостна — N записей проверено» / «Обнаружено нарушение на записи #M».

**Доступ:** только роли с `admin` правом на `audit` ресурс (владелец холдинга + роль с явно назначенным правом).

## 4. Навигационная карта

Ниже — предлагаемый скелет для включения новых экранов в sidebar AdminLayout M-OS-1.1. Designer **уточняет и изменяет** при финальной сборке, с обоснованием.

**Предлагаемое расширение левого sidebar:**

```
M-OS Admin
──────────────────────────────────────────
(существующие пункты из M-OS-1.1)
Управление
  ├── Юрлица
  ├── Пользователи
  └── Роли

Права и настройки
  ├── Матрица прав
  └── Настройки компании

Система
  ├── Реестр интеграций
  └── Системная конфигурация

(новые пункты M-OS-1.2)
Конструктор
  ├── Шаблоны процессов (BPM)    /admin/bpm/definitions
  ├── Каналы уведомлений          /admin/notifications
  └── Шаблоны документов          /admin/document-templates

Безопасность
  └── Аудит-лог                   /admin/audit
```

**Схема переходов между экранами M-OS-1.2:**

- Список шаблонов BPM → клик на строку → Редактор шаблона
- Редактор шаблона → вкладка «История версий» → История версий шаблона
- История версий → «Восстановить» → новый черновик шаблона
- Каналы уведомлений → баннер «Настроить Telegram» → /admin/integrations (из M-OS-1.1)
- Аудит-лог → клик на строку → детальная запись (inline expand или Sheet)
- Аудит-лог → ссылка «Показанный объект» → соответствующая страница сущности
- Системная конфигурация (M-OS-1.1) → ссылка «Аудит конфигурации» → Аудит-лог с предфильтром по entity_type=system_config

## 7. Definition of Done для designer

Wireframes считаются готовыми к передаче ux-head, когда:

1. Файл `docs/pods/cottage-platform/specs/wireframes-m-os-1-2-bpm.md` создан.
2. Все 4 экрана содержат все 6 обязательных блоков.
3. Раздел «Навигационная карта» заполнен: расширение sidebar + схема переходов.
4. Все принятые решения по выбору из вариантов — обоснованы текстом внутри файла.
5. Не содержит экранов M-OS-1.3 (Form Builder, Report Builder, Business Rule Engine).
6. Не содержит UI-форм активации email/SMS-каналов (только Telegram живой).
7. Экран BPM редактора не включает bpmn-js canvas (это M-OS-2).

## 8. Жёсткие ограничения

- Не пишет wireframes сам design-director.
