## Заключение по каждому экрану

### Экран 1. Companies (Юрлица)

**Статус: одобрен.**

Решение D1 (отдельный роут вместо Sheet) обосновано верно — 10 полей с условной видимостью (КПП, метка руководителя для ИП) в Sheet создали бы неприемлемую прокрутку. Четыре режима (список, карточка, создание, редактирование) разграничены чётко. Порядок вкладок по D10 логичен: реквизиты — первичный контекст, банковские реквизиты на третьей позиции — правильно, это не ежедневный запрос.

Маршрут `/admin/companies/:id/edit` присутствует и явно указан — директива соблюдена.

Банковские счета реализованы как sub-таблица с CRUD и кнопкой «+ Добавить счёт» через Dialog — директива о списке выполнена.

Один minor: форма банковского счёта (Dialog с полями) в wireframes описана в тексте, но самого wireframe Dialog-а нет — только его упоминание. Для handoff frontend-dev это пробел.

### Экран 2. Users (Пользователи)

**Статус: одобрен.**

Блок «Роли по компаниям» спроектирован точно по брифу: таблица привязок с колонками «Компания / Роль / Pod / Действия». Решение D3 (Dialog для добавления привязки) обосновано корректно — три поля не требуют Sheet, inline не масштабируется при 5–15 строках.

Wireframe Dialog-а добавления привязки присутствует полностью — это плюс.

Сброс пароля при редактировании: поле «Пароль» заменяется кнопкой «Сбросить пароль» с Dialog подтверждения — покрывает сценарий из брифа.

Состояние «Email занят» выделено отдельной строкой в таблице состояний — хорошая детализация.

---

### Экран 3. Roles (Роли)

**Статус: одобрен с minor-замечанием.**

Решение D4 (вкладка «Права» как навигационный переход в Экран 4, а не встроенный редактор) обосновано верно. Дублировать сложный экран матрицы внутри карточки роли — неоправданная сложность. Предфильтр по роли через `?role=<code>` — правильный паттерн.

Форма создания роли через Sheet — решение разумное, 5 полей умещаются без прокрутки.

Minor: в таблице списка ролей нет столбца «Описание» (краткое). При 10+ ролях пользователь не сможет различить их без перехода в карточку. Рекомендую добавить усечённую строку описания (до 60 символов) как опциональную колонку или tooltip при наведении на название роли. Не блокирует approve, но стоит зафиксировать для handoff.

### Экран 4. Permissions Matrix (Матрица прав)

**Статус: одобрен.**

Выбран Вариант A (директива ux-head). Таблица role × action с вкладками по resource_type реализована корректно — вкладки [contract] [payment] [project] [* (все)], строки — роли, колонки — действия.

Механизм ручного сохранения (директива ux-head): кнопки «Редактировать» → «Сохранить/Отменить» продублированы вверху и внизу — правильно с точки зрения accessibility, не заставляет прокручивать.

Состояние блокировки системных прав: иконка замка с tooltip — достаточный сигнал.

Предфильтр по роли через query param с выделением строки и кнопкой «Сбросить фильтр» — продуманное решение, не нарушает общую читаемость матрицы.

Единственный вопрос для FE handoff: при большом числе ресурсов (будущее расширение) вкладки могут не уместиться в строку. Рекомендую зафиксировать в потенциальных взаимосвязях — designer уже сделал это в разделе открытых вопросов, достаточно.

## Список minor-замечаний

### M-1 (ОБЯЗАТЕЛЬНО ДО HANDOFF): Отсутствует wireframe Dialog-а для банковского счёта

**Экран:** 1, вкладка «Банковские реквизиты».

**Проблема:** в wireframes упоминается, что «+ Добавить счёт» открывает Dialog с формой, но wireframe самого Dialog-а (ASCII-схема + таблица полей) отсутствует. Frontend-dev, которому передаётся документ, не получит контракт на форму банковского счёта.

**Что добавить:** ASCII-схема Dialog-а с 4 полями (Наименование банка, БИК, Расчётный счёт, Корреспондентский счёт) по аналогии с Dialog привязки роли в Экране 2.

**Таблица полей** уже есть в разделе «Поля формы / колонки таблицы» (sub-таблица банковских реквизитов) — нужно только добавить wireframe-схему.

---

### M-2 (ОБЯЗАТЕЛЬНО ДО HANDOFF): Переключатель компании в топбаре — уточнить на Combobox

**Экран:** все 7 (топбар присутствует везде).

**Проблема:** в wireframes топбар обозначен как «Переключатель компании ▼» без уточнения компонента. Координатор дал ответ на Q1: это **Combobox с поиском** (для 10+ компаний холдинга). Это отличается от обычного Select без поиска.

**Что добавить:** в разделе топбара (первый wireframe, Экран 1) добавить уточнение: «Переключатель компании — Combobox (shadcn/ui Combobox) с поисковой строкой. Для малого числа компаний (≤5) поиск скрыт.» Также отразить в Навигационной карте, раздел «Примечание по "Настройки компании" в sidebar».

---

### M-3 (РЕКОМЕНДАЦИЯ, не блокирует): Tooltip на описании роли в списке

**Экран:** 3, список ролей.

**Проблема:** таблица не содержит столбца «Описание». При 10+ ролях администратор не сможет различить их по названию без перехода в карточку.

**Рекомендация:** добавить tooltip при наведении на название роли, показывающий первые 80 символов описания. Это не требует нового столбца. Shadcn/ui Tooltip — компонент в стеке.

Не блокирует handoff — решение может быть принято frontend-dev самостоятельно, но стоит зафиксировать.
