## Экран 4 (детализация). Permissions Matrix — `/admin/permissions`

> Базовый wireframe экрана описан в `wireframes-m-os-1-1-admin.md` §Экран 4.
> Данный раздел добавляет: bulk-select, детали компонентного состава, data-testid конвенцию, расширенные состояния UI.

### Persona и сценарий (углублённый)

**Персона:** Владелец холдинга.

**Сценарий A (стандарт):** Владелец нажимает «Редактировать», в колонке «read» нажимает «Выделить всех» — все роли сразу получают read для текущего ресурса. Снимает отдельные исключения для конкретных ролей. Нажимает «Сохранить».

**Сценарий B (новая роль):** Создана роль «Кассир». В матрице по умолчанию у неё нет ни одного права. Владелец заходит в матрицу с предфильтром `?role=cashier`, строка «Кассир» выделена. Включает нужные права по одному чекбоксу или использует bulk по строке («Выделить всю строку»). Сохраняет.

**Детали bulk-механики:**
- Чекбокс «[☐ Вся строка]» в начале каждой строки: при активации ставит ☑ для всех редактируемых ячеек строки (пропускает 🔒). При частичной заполненности строки чекбокс в состоянии indeterminate.
- Кнопка «[∀ всех]» над каждой колонкой: при клике ставит ☑ для всех ролей в этой колонке (пропускает 🔒). Визуально: text-link «Выбрать всех» / «Снять всех» (toggle по состоянию большинства).
- Изменения накапливаются в памяти. PATCH-запрос отправляется только при «Сохранить».
- При «Отменить»: все изменения сессии редактирования сбрасываются без запроса к API.

### Поля формы / колонки таблицы (расширенная спецификация)

#### Матрица — компонентная схема

| Элемент | shadcn/ui компонент | Примечания |
|---|---|---|
| Вкладки ресурсов | `Tabs`, `TabsList`, `TabsTrigger` | Вкладка `* (все)` — агрегированный вид: показывает union всех resource_type. Только для чтения, без редактирования. |
| Строки таблицы | `TableRow` | Выделение предфильтра: `bg-[var(--color-info-bg)]`, левая граница `border-l-2 border-[var(--color-info)]` |
| Bulk-чекбокс строки | `Checkbox` с `indeterminate` prop | Первая ячейка строки; `aria-label="Выбрать все права для роли {role.name}"` |
| Bulk-кнопка колонки | `Button` variant=ghost, `text-xs` | В ячейке заголовка; `aria-label="Выбрать всех для действия {action}"` |
| Чекбокс ячейки | `Checkbox` | disabled + `Lock` 16px (Lucide) для системных прав |
| Tooltip на 🔒 | `Tooltip`, `TooltipContent` | Текст: «Системное право — не может быть изменено» |
| Баннер режима редактирования | `Alert` variant=warning | `AlertTriangle` 16px + текст «Режим редактирования» |
| Баннер предфильтра | `Alert` variant=info | `Info` 16px + текст + кнопка «Сбросить» (ghost, inline) |
| Кнопки Сохранить/Отменить | `Button` primary/secondary | Дублируются вверху и внизу (`sticky top` при скролле). |

#### Вкладка «* (все)» — агрегированный вид

Показывает все ресурсы как вложенные секции без переключения вкладок. Используется для быстрого обзора полной матрицы. Редактирование заблокировано (кнопка «Редактировать» скрыта). Рядом с вкладкой — `Badge` «Только просмотр».

### Состояния UI (полные 5)

| Состояние | Описание | Визуальный эффект |
|---|---|---|
| **loading** | Данные матрицы загружаются | Shimmer-таблица: заголовки отображены, ячейки — `animate-pulse` прямоугольники. Кнопка «Редактировать» disabled. |
| **empty** | Нет ролей или нет ресурсов для текущего resource_type | Центрированный блок: иконка `ShieldOff` 24px + «Для этого ресурса права не настроены» + ссылка «Управлять ролями» → `/admin/roles` |
| **error** | Ошибка загрузки матрицы | `Alert` variant=danger над таблицей: «Не удалось загрузить матрицу прав. [Повторить]». Таблица скрыта. |
| **success / save** | Успешное сохранение изменений | Режим редактирования выключается. Toast: «Матрица прав обновлена» (3 сек). Таблица обновляется без перезагрузки страницы. |
| **dialog — unsaved changes** | Попытка переключить вкладку ресурса с несохранёнными изменениями | `AlertDialog`: «Есть несохранённые изменения по ресурсу "{текущий}". Перейти без сохранения?» Кнопки: «Остаться» (secondary) и «Перейти» (danger). |

Дополнительно:

| Состояние | Описание |
|---|---|
| **error / save** | Toast danger: «Ошибка сохранения: {текст}». Режим редактирования остаётся активным. Изменения не сброшены. |
| **predfilter active** | Info-баннер над таблицей; строка выбранной роли выделена `bg-info-bg`. |
| **system lock hover** | Tooltip на `Lock`-иконке: «Системное право — не может быть изменено» |

## Экран 6 (детализация). Integration Registry — `/admin/integrations`

> Базовый wireframe экрана описан в `wireframes-m-os-1-1-admin.md` §Экран 6.
> Данный раздел добавляет: детальный Sheet с логами для Telegram, статусные состояния карточек, расширенные состояния UI, data-testid конвенцию.

### Persona и сценарий (углублённый)

**Персона:** Владелец холдинга.

**Сценарий A (настройка):** Впервые открывает Integration Registry. Telegram — единственная живая интеграция (Badge «Не подключён»). Нажимает «Настроить», вводит Bot Token и Chat ID. Нажимает «Проверить подключение» — бот отправляет тестовое сообщение. Сохраняет.

**Сценарий B (диагностика):** Получил Telegram-алерт о сбое. Открывает Integration Registry → «Настроить» на Telegram-карточке → секция «Последние события» → видит строку «Ошибка: upstream timeout» с временной меткой. Понимает причину сбоя без обращения к полному Audit Log.

**Сценарий C (ознакомление):** Новый администратор просматривает карточки «1С», «Сбербанк» со статусом «Недоступно» — понимает roadmap без лишних вопросов.

## Экран 7 (новый). Audit Log — `/admin/audit`

> Экран не описан в базовом документе — он упоминался как «переход по ссылке из других экранов» (потенциальные взаимосвязи п. 1).
> Данный раздел является первичным wireframe Audit Log как полноценного экрана.

### Persona и сценарий

**Персона:** Владелец холдинга (is_holding_owner=true). Единственный пользователь с кнопкой «Проверить цепочку». Другие роли могут просматривать лог, но не верифицировать.

**Сценарий A (мониторинг):** Владелец подозревает несанкционированное изменение матрицы прав. Открывает Audit Log, фильтрует по `entity_type=permission`, `action=update`. Видит запись: кто, когда, что именно изменил. Открывает детализацию строки.

**Сценарий B (верификация):** Владелец нажимает «Проверить цепочку». Система проверяет хэш-цепочку audit_log. Отображает результат: «OK» или список сломанных ID.

**Сценарий C (контекстный вход):** Из формы редактирования компании клик «История изменений» → Audit Log с предфильтром `entity_type=company, entity_id=5`.

### Ссылки на API endpoints

```
GET    /api/v1/audit/logs                         — список записей журнала с фильтрацией и пагинацией
  Query params:
    user_id          — фильтр по пользователю
    entity_type      — тип объекта (company / user / role / ...)
    action           — действие (create / update / delete / ...)
    date_from        — начало периода (ISO 8601)
    date_to          — конец периода (ISO 8601)
    company_id       — фильтр по компании
    entity_id        — ID конкретного объекта (для предфильтров)
    source           — источник (telegram, system и пр.)
    integrity        — broken (только нарушенные записи)
    offset           — пагинация
    limit            — пагинация (max 200, default 50)

GET    /api/v1/audit/logs/{id}                    — детализация одной записи (old_value, new_value, hash, prev_hash)
GET    /api/v1/audit/verify                       — верификация хэш-цепочки; возвращает {ok: bool, checked: int, broken: [{id, ts, user, entity}]}
GET    /api/v1/audit/export                       — экспорт в CSV с теми же query params что у /logs
```

### Схема переходов (экраны P2)

```
Экран 4 (Permissions Matrix)
  ├── Вход из Экрана 3 (карточка роли) → ?role=<code> (предфильтр + выделение строки)
  ├── Bulk-чекбокс строки → изменение всех разрешённых ячеек строки (в памяти)
  ├── Bulk-кнопка колонки → изменение всех ячеек колонки (в памяти)
  ├── «Сохранить» → PATCH /api/v1/roles/permissions
  ├── «Отменить» → сброс всех изменений без запроса
  └── Переключение вкладки с несохранёнными изменениями → AlertDialog подтверждения

Экран 6 (Integration Registry)
  ├── «Настроить» (Telegram) → Sheet настройки
  │     ├── «Отправить тест» → POST .../telegram/test (inline результат)
  │     ├── «Смотреть все в Audit Log» → /admin/audit?source=telegram
  │     └── «Сохранить» → PATCH .../telegram
  └── Клик на «Недоступно» → Dialog информационный

Экран 7 (Audit Log)
  ├── Вход из Экрана 1 (История компании) → ?entity_type=company&entity_id={id}
  ├── Вход из Экрана 5 (История настроек) → ?entity_type=company_settings&entity_id={id}
  ├── Вход из Экрана 6 (Telegram logs) → ?source=telegram
  ├── Клик по строке → Sheet детализации
  │     └── «Закрыть» / клик за Sheet → Sheet закрывается, позиция прокрутки сохраняется
  ├── «Проверить цепочку» → GET /api/v1/audit/verify → inline-результат
  │     └── Клик на broken ID → Sheet детализации соответствующей записи
  └── «Экспорт CSV» → GET /api/v1/audit/export → download
```
