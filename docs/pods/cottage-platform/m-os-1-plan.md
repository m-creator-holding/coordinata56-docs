## Принципы декомпозиции

1. Каждая под-фаза закрывается за 2–3 недели (правило Владельца, msg 817–818).
2. Каждая под-фаза — конкретный, проверяемый результат, не «работа над X».
3. Последовательность строгая: следующая не стартует, пока текущая не закрыта по DoD.
4. Исключение: M-OS-1.3 и M-OS-1.4 (BPM и Pod-проверка) могут частично перекрываться по времени, если Foundation (1.1) и Безопасность (1.2) закрыты.
5. Ответственный Директор — декомпозирует под-фазу дальше в батчи и задаёт задачи своим Head. Координатор взаимодействует только с Директором.

### M-OS-1.1 Foundation (Фундамент)

**Цель.** Заложить архитектурный каркас M-OS: multi-company данные, гранулярные права, два независимых event bus, полный configuration-as-data admin-UI, pluggability, anti-corruption layer, реестр интеграций, per-company бизнес-лимиты.

**Решения Владельца, расширяющие скоуп этой под-фазы (msg 1094/1101/1105):**

1. Admin UI реализуется как полный конструктор с M-OS-1: роли, юрлица, лимиты, BPM-процессы, формы ввода, отчёты, шаблоны документов, уведомления, бизнес-правила — всё настраивается через UI, не через код. Principle 10 соблюдается с M-OS-1.
2. Event Bus разделяется на два независимых транспорта: один для бизнес-событий (платежи, договоры, приёмки), второй — для управления ИИ-субагентами. Они не пересекаются.
3. Бизнес-лимиты (лимит платежа, порог согласования без директора, перерасход и т.д.) хранятся per-company в таблице `company_settings`, редактируются через UI.
4. Telegram — единственная живая внешняя интеграция в M-OS-1. Адаптеры банков, Росреестра, 1С, ОФД, страховых — разрабатываются «на полке» с флагом `enabled=False`, к живым системам не подключаются до production-gate.

**Скоуп — что входит:**

- Multi-company модель данных: добавить `company_id` (юрлицо) к ключевым сущностям, реализовать логику «один аккаунт — несколько юрлиц — разные видимости» (принцип А1, А2).
- Fine-grained права: модель «роль + объект + действие», расширение RBAC текущего cottage-platform (принцип А2).
- Бизнес Event Bus: внутренняя шина событий (pub/sub) для бизнес-событий — платежи, договоры, приёмки. Модули и поды общаются событиями, не прямыми вызовами (C2 из vision).
- AI Event Bus: отдельный транспорт для управляющих событий ИИ-субагентов. Физически и логически изолирован от бизнес-шины. ADR-кандидат по разделению (ADR-0016, готовит backend-director).
- Configuration-as-data layer: роли, права, лимиты, шаблоны документов, бизнес-правила — в БД как данные; код только читает и исполняет; аудит изменений конфигурации в том же AuditLog (принцип 10 vision).
- Per-company лимиты: таблица `company_settings` (лимит платежа, порог согласования без директора, флаги перерасхода и т.д.) — отдельные значения для каждого юрлица, редактируются через admin-UI.
- Admin UI — полный конструктор: настройка ролей, юрлиц, лимитов, BPM-процессов, форм ввода, отчётов, шаблонов документов, уведомлений, бизнес-правил. Всё — configuration-as-data; UI только читает БД и пишет в неё. Объём скоупа фиксируется в ADR-0017 (architecture) и ADR-0018 (data model), готовит backend-director. Раздувание скоупа за пределы перечисленного — только по новой заявке Владельцу.
- Pluggability (разомкнутая архитектура): dependency injection для всех внешних зависимостей (банки, ИИ-модели, форматы обмена, криптопровайдер); ни один внешний API не проникает в ядро напрямую (принцип 10 vision).
- Anti-corruption layer (каркас): обособленный слой-фасад для внешних систем; в M-OS-1 — каркас с интерфейсами; конкретные адаптеры (Сбер, 1С) — разрабатываются «на полке» с `enabled=False`, активируются только после production-gate в M-OS-2.
- Реестр интеграций: каталог подключений в БД (тип, версия API, статус, конфигурация, флаг `enabled`); управляется через admin-UI, не через код.
- Contract модель (tech-debt из Фазы 3): добавить `file_id` (P0-F-01) и `start_date`/`end_date` (P1-F-04) — без этого production gate не пройдём.

**Скоуп — что НЕ входит:**

- Конкретные внешние адаптеры (Сбер, 1С, ОФД, Росреестр, страховые) — только каркас интерфейсов с `enabled=False`. Активация — в M-OS-2, только после production-gate.
- MFA, crypto-chain аудита, staging/prod, feature flags — следующая под-фаза (M-OS-1.2).
- BPM-движок (исполняющий) — M-OS-1.3. Admin-UI в M-OS-1.1 содержит только конфигуратор шаблонов BPM в виде данных; движок, обрабатывающий экземпляры, — отдельно в M-OS-1.3.
- АЗС-pod-lite — M-OS-1.4.
- Drag-and-drop canvas для BPM — M-OS-2.
- Admin-UI категории, не перечисленные выше — только по решению Владельца.

**Definition of Done:**

1. `alembic upgrade head` + `alembic downgrade -1` + `alembic upgrade head` — чисто без ошибок (round-trip обязателен).
2. Все существующие 351 тест проходят без регрессий.
3. Новые тесты покрывают: company isolation (сотрудник из компании A не видит данные компании B), RBAC матрицу (роль + объект + действие — минимум по 2 позитивных + 2 негативных сценария на тип), оба event bus (событие публикуется → подписчик получает → порядок и идемпотентность; бизнес-шина не содержит AI-событий и наоборот).
4. Admin-UI реализован и покрывает: управление ролями, юрлицами, per-company лимитами (`company_settings`), шаблонами BPM (в виде данных), формами ввода, шаблонами документов, настройками уведомлений, бизнес-правилами. Каждая секция admin-UI задокументирована в Swagger.
5. Dependency injection: ни одного прямого импорта внешнего API в слое services или core — только через интерфейс. Проверяется ревью.
6. Реестр интеграций: CRUD-эндпоинты, схема в БД (поле `enabled`, дефолт `False` для всех адаптеров кроме Telegram), тесты.
7. Contract: `file_id` и `start_date`/`end_date` добавлены, миграции, тесты.
8. Reviewer (субагент) вынес вердикт approve по всем изменениям до коммита.
9. ADR-кандидаты (multi-company модель, два event bus, configuration-as-data admin-UI, per-company лимиты) — подготовлены backend-director и переданы Координатору для регистрации в Governance (кандидаты ADR-0013 / 0014 / 0015 / 0016 / 0017 / 0018 — нумерацию присваивает Координатор; не блокируют под-фазу, но параллельны).

**Зависимости:** M-OS-0 закрыта (выполнено, `06baf07`). Нет других предшественников.

**Оценка: 7-8 недель.**

Обоснование пересмотра с 3 до 7-8 недель:
- Полный admin-UI-конструктор (8 категорий настроек) — это не «пара admin-endpoint'ов», это самостоятельный подпроект с UX, API и тестами. Оценка прибавки: +2-3 нед.
- Два независимых Event Bus вместо одного — отдельная архитектура, разные транспортные слои, изоляция между ними, тесты на cross-загрязнение. Прибавка: +1 нед.
- Per-company лимиты (`company_settings`) — отдельная таблица, миграция, CRUD, UI-секция, тесты на изоляцию между юрлицами. Прибавка: +0,5-1 нед.
- Итого с учётом рисков multi-company migration (Риск 1): 7-8 нед. вместо 3.

**Ответственный Директор:** backend-director (декомпозирует на батчи по образцу Фазы 3; db-head и frontend-director задействуются под ним на admin-UI; оба — через стандартную цепочку).

### M-OS-1.2 Безопасность и аудит

**Цель.** Укрепить надёжность системы до уровня, при котором можно подключать боевые данные: crypto-chain аудита, PITR, MFA для критичных ролей, среды dev/staging/prod, feature flags.

**Предшественник:** M-OS-1.1 закрыта по DoD.

**Скоуп — что входит:**

- Крипто-цепочка хэшей аудита: каждая запись в AuditLog содержит `hash(prev_hash + payload)` — математически невозможно подделать задним числом (Е1 из vision). Реализуется на уровне сервиса аудита.
- Point-in-time recovery (PITR): настройка PostgreSQL WAL-архивирования, snapshot каждые 5 минут, документированная процедура восстановления (Е3 из vision).
- Первые DR-учения: восстановление из PITR на staging, задокументированный run-book (Е4 из vision).
- MFA для ролей owner и accountant: TOTP (Google Authenticator / Яндекс.Ключ), обязательность настраивается через configuration-as-data (ГГ3 из vision). Зависит от M-OS-1.1 (configuration-as-data layer).
- Среды staging и prod: разделение конфигурации, secrets через environment variables, README с инструкцией по развёртыванию в каждой среде (У1 из vision).
- Feature flags: механизм включения/выключения фич по ролям или юрлицам без деплоя, хранится в БД (У2 из vision, зависит от configuration-as-data из M-OS-1.1).
- Юр-действия (production gate): подготовка чек-листа для живого юриста (F-02, F-03, F-05, F-07 из `phase-3-status.md`). Governance-director отмечает как «требует заявки»: если юр-действия не выполнены до конца M-OS-1.2 — production gate остаётся закрытым.

**Скоуп — что НЕ входит:**

- SIEM, Zero Trust, PAM, Threat intelligence — это M-OS-2.
- Биометрия (voice biometry, YubiKey) — M-OS-4.
- Canary releases — M-OS-2 (У3).

**Definition of Done:**

1. `audit_log` содержит поле `hash`, каждая новая запись имеет корректный хэш от предыдущей. Тест: разрыв цепочки при подделке одной записи детектируется валидатором.
2. PITR: WAL-архив настроен, процедура восстановления задокументирована в `docs/ops/pitr-runbook.md`, восстановление отработано на staging (скриншот или лог в run-book).
3. MFA: owner и accountant при входе в staging обязаны ввести TOTP; тест на обход (без TOTP → 401/403) работает.
4. `staging` и `prod` конфигурации разделены; никаких хардкод-секретов в коде (проверяется grep + reviewer).
5. Feature flags: CRUD в admin, флаг «feature_X_enabled = false» блокирует эндпоинт для указанных ролей; тесты на включение/выключение.
6. DR run-book: `docs/ops/dr-runbook.md` создан, учения проведены хотя бы один раз на staging.
7. Reviewer approve до коммита.
8. Юр-чек-лист F-02/F-03/F-05/F-07 сформирован и передан Координатору для передачи Владельцу (не блокирует DoD этой под-фазы, но обязателен как артефакт).

**Зависимости:** M-OS-1.1 (configuration-as-data layer нужен для feature flags и MFA-политики).

**Оценка:** 2 недели.

**Ответственный Директор:** backend-director (security-advisor консультирует по crypto-chain и PITR, infra-director — по среде staging/prod; оба через Координатора).

### M-OS-1.3 Процессный движок BPM (базовый)

**Цель.** Реализовать минимально работающий BPM-движок с человеческими шагами, таймерами и тремя готовыми шаблонами процессов: согласование платежей, приёмка работ, создание договора.

**Предшественник:** M-OS-1.1 и M-OS-1.2 закрыты по DoD.

**Обоснование порядка:** BPM зависит от event bus (M-OS-1.1) — шаги BPM-процесса публикуются как события; зависит от feature flags (M-OS-1.2) — процессы можно включать/выключать; зависит от audit crypto-chain (M-OS-1.2) — каждый переход состояния процесса обязан писаться в защищённый аудит.

**Скоуп — что входит:**

- Модель данных BPM: `ProcessDefinition` (шаблон), `ProcessInstance` (запущенный процесс), `ProcessStep` (шаги с типами: human_task, timer, auto), `ProcessTransition` (переходы с условиями).
- Исполняющий движок: создание экземпляра процесса, переход по шагам, human_task (ожидает одобрения/отклонения от конкретной роли или набора ролей), timer (эскалация если не одобрено за N часов).
- Три готовых шаблона: согласование Payment (существующий approve/reject рефакторить на BPM), приёмка работ (акт КС-2 или аналог), создание договора (черновик → согласование → подписание) — Д1, Д3 из vision.
- Telegram-уведомления на шаге human_task: сотруднику приходит сообщение с кнопками «Одобрить / Отклонить» (Б4, В7 из vision). Зависит от конфигурации notification-channel в реестре интеграций (M-OS-1.1).
- Базовый визуальный редактор (admin-only, не production UI): JSON-схема процесса редактируется через форму или raw JSON в admin-панели. Полноценный drag-and-drop canvas — M-OS-2 (М3 из vision).
- Аудит каждого перехода состояния процесса — в AuditLog с crypto-chain (из M-OS-1.2).

**Скоуп — что НЕ входит:**

- Полный drag-and-drop визуальный редактор (canvas) — M-OS-2.
- Библиотека из 10+ шаблонов — M-OS-2. Здесь только 3 обязательных.
- Интеграция с внешними системами как шаги BPM (1С, банк) — M-OS-2.
- Conditional branching сложнее «сумма > N → путь А» — M-OS-2.

**Definition of Done:**

1. `ProcessDefinition` CRUD в admin; шаблон имеет версионирование (version field).
2. `ProcessInstance` запускается для конкретного объекта (Payment, Contract и т.д.); статус отслеживается.
3. Human task: сотруднику с нужной ролью приходит Telegram-уведомление с кнопками; кнопка «Одобрить» двигает процесс дальше; кнопка «Отклонить» завершает с причиной.
4. Timer: если human_task не выполнен за настраиваемый timeout → эскалация (уведомление следующей роли в цепочке или Владельцу). Тест с синтетически малым timeout (1 секунда).
5. Три шаблона: Payment-approval, Work-acceptance, Contract-creation — запускаются и проходят end-to-end в integration тестах.
6. Аудит: каждый переход записан в AuditLog, hash корректен.
7. Существующий endpoint Payment `/approve` и `/reject` — либо рефакторен под BPM, либо сосуществует с явной документацией в ADR-кандидате (решение принимает backend-director).
8. Reviewer approve до коммита.

**Зависимости:** M-OS-1.1 (event bus, configuration-as-data), M-OS-1.2 (audit crypto-chain, feature flags, Telegram-интеграция в реестре).

**Оценка:** 3 недели.

**Ответственный Директор:** backend-director.

### M-OS-1.4 Pod-проверка (АЗС-lite)

**Цель.** Добавить АЗС-pod-lite — минимальный второй pod — и доказать, что ядро M-OS не нужно переписывать при добавлении нового направления бизнеса.

**Предшественник:** M-OS-1.1, M-OS-1.2 и M-OS-1.3 закрыты по DoD.

**Обоснование порядка:** Pod-проверка идёт последней, потому что проверяет именно что foundation (1.1) устоял под нагрузкой второго потребителя. Если тест выявляет необходимость переделки ядра — это регресс M-OS-1.1, а не работа M-OS-1.4.

**Скоуп — что входит:**

- `gas-stations-pod`: минимальный набор сущностей, достаточный для доказательства изоляции. Состав: `GasStation` (АЗС как объект с company_id), `Shift` (смена, привязана к кассиру и АЗС), `FuelSale` (продажа, сумма, объём, тип топлива). Три сущности, не больше.
- Подключение к общему ядру: те же Users/RBAC (роль «кассир АЗС» видит только свою АЗС), тот же AuditLog с crypto-chain, тот же event bus.
- Shared master data: справочник `FuelType` как общий справочник через ядро.
- Pod-изоляция: кассир АЗС не видит cottage-platform данные — тест на cross-pod leak.
- Data contract (минимальный): схема `FuelSale` зафиксирована в `docs/pods/gas-stations/data-contract-v1.md`, подписчики ядра знают формат (С1 из vision).
- Документация: `docs/pods/gas-stations/README.md` — инструкция «как подключить новый pod», написана по итогам опыта АЗС-lite.

**Скоуп — что НЕ входит:**

- Реальная интеграция с ОФД — M-OS-2.
- Датчики уровней топлива — M-OS-4.
- Полная функциональность АЗС (смены, управление резервуарами, отчёты) — M-OS-2.
- Frontend для АЗС-pod — M-OS-2.

**Definition of Done:**

## Активация frontend-director — обязательное условие запуска M-OS-1.1

**Статус:** обязательное условие (v1.2; ранее в v1.1 — рекомендация).

Frontend-director должен быть активирован Координатором **до или одновременно** со стартом M-OS-1.1. Цель активации — дать frontend-director время подготовить декомпозицию PWA-работ к моменту начала M-OS-1.3, чтобы пункт 7 DoD M-OS-1 (мобильный PWA у прораба) не стал риском для закрытия всей фазы.

Порядок активации:
1. Координатор даёт frontend-director задачу на брифинг: «подготовь декомпозицию PWA-работ к M-OS-1.3».
2. Frontend-director работает параллельно с M-OS-1.1, не блокируя и не блокируясь бэкенд-фазой.
3. Если frontend-director не активирован к моменту старта M-OS-1.1 — M-OS-1.1 **не стартует**. Координатор обязан сначала активировать frontend-director, затем запускать под-фазу.

Последствие неисполнения: пункт 7 DoD M-OS-1 (PWA) автоматически переносится в M-OS-2 с явным примечанием в статус-документе.

Основание: решение Владельца (Telegram msg 1111, 2026-04-17).

## Риски и зависимости

### Риск 1 (высокий). Multi-company migration на живых данных

Добавление `company_id` к существующим сущностям cottage-platform — это обратно несовместимая миграция. Существующие 351 тест и все fixtures написаны без company_id. Потребуется либо bulk-backfill (все существующие записи получают дефолтный company_id = 1), либо nullable поле с постепенным переходом.

Митигация: backend-director обязан в брифинге M-OS-1.1 явно решить стратегию миграции и задокументировать в ADR-кандидате. Alembic round-trip обязателен. Регресс в тестах = блокер батча.

### Риск 2 (высокий). PWA / фронтенд — нет активного исполнителя

Пункт 7 DoD (PWA у прораба) требует frontend-director. Сейчас статус — active-supervising, но реальная фронтенд-работа не велась с начала проекта (бэкенд-фазы 0–3). Если frontend-director не получит задачу от Координатора до старта M-OS-1.3 — PWA станет риском для закрытия всей M-OS-1.

Митигация (v1.2 — теперь обязательное требование, не рекомендация): активация frontend-director является предварительным условием запуска M-OS-1.1. Координатор обязан активировать frontend-director до старта M-OS-1.1. Детали — в разделе «Активация frontend-director» выше. Основание: решение Владельца (Telegram msg 1111, 2026-04-17).

### Риск 3 (средний). Configuration-as-data scope creep

«Все бизнес-правила в БД» — потенциально бесконечный скоуп. Без явного ограничения backend-director будет переносить в конфигурацию всё подряд, что затянет M-OS-1.1.

Митигация: в брифинге M-OS-1.1 Координатор задаёт явный список того, что входит в scope configuration-as-data (роли, права, лимиты платежей, шаблоны процессов). Всё остальное — M-OS-2 или M-OS-3. Список фиксируется как ADR-кандидат.

### Риск 4 (средний). Полный admin-UI-конструктор может раздуться за границы скоупа

Admin-UI охватывает 8 категорий (роли, юрлица, лимиты, BPM, формы, отчёты, шаблоны, уведомления/бизнес-правила). Каждая категория — это отдельная UX и API-поверхность. Без жёстких границ backend-director и frontend-director начнут добавлять «пока делаем, заодно и X». Риск: затягивание M-OS-1.1 на 10-12 недель вместо 7-8.

Митигация: ADR-0017 (Configuration-as-Data architecture) и ADR-0018 (Admin-UI data model) должны фиксировать полный, закрытый список категорий admin-UI для M-OS-1. Всё, что не вошло в ADR-0017/0018 — автоматически M-OS-2. Подготавливает backend-director до старта кодирования.

### Риск 5 (средний). Два Event Bus увеличивают оперативную сложность

Два независимых транспорта (бизнес + AI) — это двойная операционная нагрузка: два набора конфигураций, два набора тестов, два набора SLA. Без чёткого разделителя субагенты начнут публиковать события в «неправильный» bus или создадут зависимость между транспортами.

Митигация: ADR-0016 (Event Bus split) должен зафиксировать: (а) физический или логический уровень изоляции, (б) правило маршрутизации («что считается бизнес-событием», «что — AI-событием»), (в) запрет на прямую подписку AI-шины на бизнес-шину без явной точки обработки. Подготавливает backend-director в рамках брифинга M-OS-1.1.

### Зависимость — production-gate (три столпа)

Инженерное закрытие M-OS-1 не требует прохождения production-gate. Однако подключение реальных персональных данных и боевых финансовых данных блокируется до прохождения всех трёх столпов (ст. 45а CODE_OF_LAWS v2.1):

1. **Юридический**: M-OS-1.2 генерирует чек-лист F-02/F-03/F-05/F-07. Координатор эскалирует Владельцу сразу после закрытия M-OS-1.2: «юр-чек-лист готов, требуется живой юрист».
2. **Security-аудит**: организуется Владельцем (внешний аудитор). Координатор информирует Владельца о готовности системы к аудиту после закрытия M-OS-1.2.
3. **Staging + симуляции**: детальный чек-лист симуляций разрабатывается как отдельная задача перед стартом M-OS-2. Координатор ставит эту задачу backend-director в конце M-OS-1.

Все три столпа — параллельные треки после закрытия M-OS-1. Снятие запрета на живые интеграции — только при одновременном прохождении всех трёх.

## Рекомендация по порядку запуска

**Запускать с M-OS-1.1 Foundation.**

Причины:
1. Все последующие под-фазы зависят от Foundation: event bus нужен BPM (1.3), configuration-as-data нужен feature flags и MFA (1.2), ACL-каркас нужен всем интеграциям.
2. Multi-company migration — наибольший технический риск всей фазы. Лучше встретить его первым, пока система ещё простая, чем натолкнуться на него в середине работы над BPM.
3. M-OS-1.1 — единственная под-фаза без предшественников: её можно запустить немедленно после M-OS-0 (уже закрыта).

**Обязательное условие (v1.2):** до или одновременно со стартом M-OS-1.1 Координатор активирует frontend-director. Это не рекомендация — это предварительное требование: M-OS-1.1 не стартует без активации frontend-director. Цель — к старту M-OS-1.3 иметь готовую декомпозицию PWA и не задерживать пункт 7 DoD.
