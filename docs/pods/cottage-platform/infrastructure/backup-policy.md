**Статус:** действует  
**Связанные документы:**

- Cron-запись: `/etc/cron.d/coordinata56-backup`

---

## 1. Расписание

| Параметр      | Значение                            |
|---------------|-------------------------------------|
| Частота       | Ежедневно                           |
| Время запуска | 03:00 UTC                           |
| Механизм      | `/etc/cron.d/coordinata56-backup`   |
| Запускает     | `root`                              |

Выбор 03:00 UTC: минимальная активность пользователей, снижает нагрузку на БД в момент дампа.

---

## 2. Хранение файлов

| Параметр          | Значение                                     |
|-------------------|----------------------------------------------|
| Директория        | `/var/backups/coordinata56/`                 |
| Права доступа     | `chmod 700`, owner `root`                    |
| Имя файла         | `pg_YYYYMMDD_HHMMSS.dump` (время UTC)        |
| Формат            | PostgreSQL custom (`-Fc`), zlib compress=6   |
| Retention         | 14 дней                                      |
| Защита от purge   | Удаление старых файлов происходит ТОЛЬКО если `.last_success` не старше 48 часов |

### Почему chmod 700

Файлы бэкапа содержат полный слепок БД, включая хешированные пароли и персональные данные. Доступ — только root.

---

## 3. Файлы состояния

| Файл                                       | Содержимое                                             |
|--------------------------------------------|--------------------------------------------------------|
| `/var/backups/coordinata56/.last_success`  | ISO-8601 UTC-метка последнего успешного бэкапа         |
| `/var/backups/coordinata56/.last_error`    | Текст ошибки последнего неудачного запуска (если был)  |

### Проверка свежести бэкапа вручную

```bash
cat /var/backups/coordinata56/.last_success
# Ожидаемый вывод: 2026-04-18T03:00:07Z
# Если файл отсутствует или дата старше 25ч — бэкап не прошёл, расследовать.
```

### Проверка через cron-лог

```bash
journalctl -t coordinata56-backup --since "24 hours ago"
# Успешная строка: OK: бэкап создан — /var/backups/coordinata56/pg_YYYYMMDD_HHMMSS.dump (NNKiB)
# Аварийная строка: ERROR: pg_dump завершился с ошибкой...
```

## 4. Ручной запуск

```bash
/root/coordinata56/infra/backups/pg_dump_daily.sh
```

Скрипт читает переменные `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB` из `/root/coordinata56/.env.dev`. Пароль передаётся через `-e PGPASSWORD` в `docker exec` — не виден в `ps aux` и системных логах.

## 5. Процедура восстановления

Полная процедура с шагами 0–6 и критериями корректности описана в:

```
docs/pods/cottage-platform/infrastructure/tasks/p0-2-pg-dump-format-spec.md
```

### Краткая схема

1. Определить нужный файл: `ls -lt /var/backups/coordinata56/pg_*.dump | head -5`
2. Создать временную целевую БД
3. Запустить `pg_restore --format=custom --no-owner --no-privileges`
4. Sanity-check: количество таблиц и row-count по 7 ключевым таблицам
5. Переключить `DATABASE_URL` на восстановленную БД (или переименовать БД)

## 6. RTO / RPO

| Метрика | Значение на старте | Условие улучшения          |
|---------|-------------------|----------------------------|
| RPO     | 24 часа           | При увеличении частоты бэкапов |
| RTO     | 4 часа            | При автоматизации failover  |

## 7. Базовые значения для верификации restore (2026-04-18)

- Таблиц в public-схеме: **23**
- Размер свежего дампа пустой dev-БД: **~93 KiB**
- Бэкап-файл должен быть **> 1 KiB**, иначе скрипт завершается с ошибкой

## 8. PITR — Point-in-Time Recovery (M-OS-1.1A)

Второй пояс защиты: непрерывная WAL-архивация в Яндекс Object Storage позволяет
восстановить базу на произвольный момент между базовым бэкапом и текущим временем.

| Параметр | Значение |
|----------|---------|
| Механизм | `archive_command` Postgres 16 + `wal_archive.sh` + wal-g / awscli |
| Хранение WAL | Яндекс Object Storage, бакет `coordinata56-wal` |
| Базовый бэкап | `pg_basebackup_weekly.sh`, еженедельно в воскресенье 02:00 UTC |
| Целевой RPO | < 5 минут (при непрерывной WAL-архивации) |
| Целевой RTO | < 60 минут |

**Полная процедура восстановления:** `docs/ops/pitr-runbook.md`

Drill (тестирование восстановления на dev-данных) — три сценария описаны в том же документе.
Отчёты drill хранятся в `docs/pods/cottage-platform/infrastructure/pitr-drill-YYYY-MM-DD.md`.

**Статус M-OS-1.1A:** теоретическая фаза. Активация после получения credentials Яндекс Cloud от Владельца.

## 9. Изменение политики

Любое изменение скрипта, расписания или retention требует:
1. Обновления этого документа
2. Ручного прогона для верификации
3. Проверки `.last_success` после первого автоматического запуска
