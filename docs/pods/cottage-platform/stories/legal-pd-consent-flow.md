# legal-pd-consent-flow — Flow согласия пользователя на обработку ПД

## Контекст и бизнес-обоснование

M-OS — внутренняя система холдинга. Пользователи создаются администратором, а не сами проходят регистрацию. По этой причине субъект ПД (сотрудник, подрядчик, контрагент) ещё не давал согласия на обработку данных в момент создания его учётной записи.

Статья 9 152-ФЗ требует, чтобы согласие давал сам субъект, а не оператор за него. Исключение — обработка в рамках трудового договора (ст. 86-88 ТК РФ), но оно не покрывает подрядчиков и агентов без ТД.

Данный flow закрывает P0-риск: без экрана согласия любая обработка ПД сотрудников, не являющихся работниками по ТД, юридически незаконна (ст. 13.11 КоАП: до 500 000 руб. при повторе).

**Решение Владельца (2026-04-18):** реализовать skeleton-first. Вся функциональность — сейчас. Compliance (ЛНА, штатный юрист, тексты в production) — отдельной фазой перед production-gate.

## US-01 — Редирект на экран согласия при первом входе

**Story**

Как **пользователь, впервые входящий в систему** (аккаунт создан администратором),
я хочу **автоматически попасть на экран согласия сразу после успешной аутентификации**,
чтобы **выразить свою явную волю на обработку персональных данных до начала использования системы**.

**Acceptance Criteria**

```
Сценарий 1 — Новый пользователь: редирект на согласие
Given  аккаунт пользователя создан администратором
And    поле pd_consent_at = NULL (согласие ещё не давалось)
When   пользователь вводит корректные email и пароль на экране входа
Then   система выдаёт временный JWT с клеймом consent_required: true
And    фронтенд перехватывает клейм и выполняет редирект на /consent
And    любые другие маршруты системы (кроме /consent и /logout) возвращают 403 до момента принятия согласия
And    после редиректа токен с consent_required: true автоматически не продлевается

Сценарий 2 — Пользователь, уже давший согласие: нет редиректа
Given  поле pd_consent_at IS NOT NULL
And    поле pd_consent_version совпадает с текущей активной версией политики
When   пользователь входит в систему
Then   система выдаёт JWT с клеймом consent_required: false
And    пользователь попадает на стандартный стартовый экран без редиректа

Сценарий 3 — Попытка обойти редирект (прямой переход по URL)
Given  пользователь получил JWT с consent_required: true
When   пользователь пытается обратиться к любому защищённому маршруту API кроме /auth/accept-consent и /auth/logout
Then   API возвращает 403 с кодом ошибки CONSENT_REQUIRED
And    фронтенд отображает сообщение «Для продолжения работы необходимо принять согласие на обработку персональных данных»

Сценарий 4 — Истёкший токен (expired) во время нахождения на экране согласия
Given  пользователь открыл экран согласия
And    токен успел истечь (exp < now) до нажатия «Подтверждаю»
When   пользователь нажимает «Подтверждаю»
Then   запрос к /api/v1/auth/accept-consent возвращает 401
And    фронтенд отображает сообщение «Сессия истекла. Войдите снова»
And    пользователь перенаправляется на экран входа
```

## US-02 — Просмотр текста согласия

**Story**

Как **пользователь на экране согласия**,
я хочу **прочитать полный текст согласия на обработку персональных данных**,
чтобы **понять, на что именно я соглашаюсь, какие данные собираются и на какой срок**.

**Acceptance Criteria**

```
Сценарий 1 — Отображение текста согласия
Given  пользователь перенаправлен на экран /consent
When   экран загружается
Then   отображается заголовок «Согласие на обработку персональных данных»
And    отображается полный текст согласия согласно разделу 4 документа ui-pd-labels-review-2026-04-18.md
And    текст содержит все обязательные элементы ст. 9 п. 4 152-ФЗ:
       — наименование оператора и его адрес
       — перечень персональных данных (ФИО, email, [телефон при наличии поля], должность, роль, журнал входов)
       — цель обработки
       — перечень действий с ПДн
       — способ обработки (автоматизированный)
       — срок действия согласия
       — порядок отзыва
       — ссылка на Политику обработки ПД

Сценарий 2 — Ненажатый чекбокс по умолчанию
Given  экран согласия отображается
When   страница загружается впервые
Then   чекбокс «Я ознакомлен(а) и даю согласие на обработку персональных данных» НЕ является предустановленным (unchecked)
And    кнопка «Подтверждаю» неактивна (disabled)
And    кнопка «Подтверждаю» становится активной только после отметки чекбокса

Сценарий 3 — Версия политики, отображаемая пользователю
Given  в системе зарегистрирована текущая версия политики (например, v1.0)
When   экран согласия загружается
Then   версия политики видна пользователю (например: «Версия документа: v1.0 от 2026-04-18»)
And    ссылка на Политику обработки ПД открывается в новой вкладке или модальном окне
```

## US-03 — Принятие согласия

**Story**

Как **пользователь, принявший условия**,
я хочу **подтвердить согласие нажатием кнопки «Подтверждаю»**,
чтобы **мой JWT обновился и я получил доступ к системе без повторного запроса согласия**.

**Acceptance Criteria**

```
Сценарий 1 — Успешное принятие согласия
Given  пользователь отметил чекбокс на экране согласия
And    токен с consent_required: true действителен
When   пользователь нажимает «Подтверждаю»
Then   фронтенд отправляет POST /api/v1/auth/accept-consent с текущим JWT в заголовке
And    бэкенд записывает pd_consent_at = текущий момент UTC в запись пользователя
And    бэкенд записывает pd_consent_version = текущая активная версия политики
And    бэкенд возвращает новый JWT с клеймом consent_required: false
And    бэкенд пишет запись в audit_log: action="pd_consent_granted", subject_id, consent_version, ip_address
And    фронтенд сохраняет новый токен и перенаправляет пользователя на стандартный стартовый экран
And    повторный вход в систему (при действительном согласии) не показывает экран согласия

Сценарий 2 — Двойное нажатие (double submit)
Given  пользователь нажал «Подтверждаю» первый раз
And    запрос уже отправлен и ещё не завершился
When   пользователь нажимает кнопку снова
Then   второй запрос не отправляется (кнопка переходит в состояние loading/disabled после первого клика)
And    итог — ровно одна запись в audit_log
And    pd_consent_at записывается один раз

Сценарий 3 — Сетевой сбой при отправке согласия
Given  пользователь нажал «Подтверждаю»
And    сеть недоступна в момент запроса
When   POST /api/v1/auth/accept-consent завершается с сетевой ошибкой (timeout или network error)
Then   фронтенд отображает сообщение об ошибке «Не удалось сохранить согласие. Проверьте подключение и попробуйте снова»
And    кнопка возвращается в активное состояние (пользователь может повторить)
And    pd_consent_at в базе НЕ записан (транзакция не зафиксирована)
And    пользователь остаётся на экране согласия, а не перенаправляется в систему
```

## US-04 — Отклонение согласия

**Story**

Как **пользователь, не желающий давать согласие**,
я хочу **отказаться от согласия нажатием кнопки «Отказаться»**,
чтобы **немедленно выйти из системы без сохранения каких-либо данных**.

**Acceptance Criteria**

```
Сценарий 1 — Отказ от согласия
Given  пользователь находится на экране согласия
When   пользователь нажимает «Отказаться» (или «Выйти»)
Then   фронтенд удаляет временный JWT (очищает хранилище токенов)
And    пользователь перенаправляется на экран входа
And    никакого запроса к /api/v1/auth/accept-consent не отправляется
And    pd_consent_at остаётся NULL в базе данных
And    бэкенд пишет в audit_log: action="pd_consent_declined", subject_id, ip_address

Сценарий 2 — Повторный вход после отказа
Given  пользователь ранее отказался от согласия
When   пользователь снова входит в систему с корректными учётными данными
Then   система снова перенаправляет на экран согласия
And    пользователь может принять согласие при следующей попытке

Сценарий 3 — Закрытие вкладки без выбора (без явного отказа)
Given  пользователь открыл экран согласия
When   пользователь закрывает вкладку браузера, не нажав ни одну кнопку
Then   pd_consent_at остаётся NULL
And    при следующем входе пользователь снова перенаправляется на экран согласия
```

## US-05 — Повторное согласие при обновлении политики

**Story**

Как **пользователь, уже давший согласие по прежней версии политики**,
я хочу **быть уведомлён о новой версии при следующем входе**,
чтобы **явно подтвердить своё согласие с обновлёнными условиями обработки ПД**.

**Acceptance Criteria**

```
Сценарий 1 — Обнаружение новой версии политики при входе
Given  в системе активирована новая версия политики (например, v1.1)
And    у пользователя pd_consent_version = "v1.0" (старая версия)
When   пользователь входит в систему
Then   система выдаёт JWT с клеймом consent_required: true
And    фронтенд перенаправляет на экран согласия
And    на экране отображается баннер «Политика обработки персональных данных была обновлена. Ознакомьтесь с новой версией»
And    текст согласия отражает новую версию политики (не старую)

Сценарий 2 — Принятие новой версии
Given  пользователь видит экран обновлённого согласия
When   пользователь отмечает чекбокс и нажимает «Подтверждаю»
Then   pd_consent_version обновляется до текущей версии (v1.1)
And    pd_consent_at обновляется новым timestamp
And    бэкенд пишет audit_log: action="pd_consent_updated", old_version, new_version, subject_id, ip_address

Сценарий 3 — Пользователь одновременно открыт в двух вкладках (повторное согласие)
Given  пользователь открыл экран согласия в двух вкладках браузера
When   пользователь принимает согласие в первой вкладке
And    нажимает «Подтверждаю» во второй вкладке (с уже устаревшим токеном)
Then   второй запрос к /api/v1/auth/accept-consent обрабатывается идемпотентно
And    pd_consent_version не меняется (уже актуальная), pd_consent_at не перезаписывается
And    бэкенд возвращает 200 с действительным JWT (без ошибки для пользователя)
```

## US-06 — Просмотр статуса согласия пользователя администратором

**Story**

Как **администратор системы**,
я хочу **видеть в карточке каждого пользователя статус его согласия (принято/не принято, дата, версия)**,
чтобы **контролировать, у кого ещё не получено согласие, и реагировать на юридические запросы**.

**Acceptance Criteria**

```
Сценарий 1 — Отображение статуса согласия в карточке пользователя
Given  администратор открывает карточку пользователя
When   страница пользователя загружается
Then   в разделе «Персональные данные» отображается блок «Согласие на обработку ПД»
And    блок содержит:
       — статус: «Принято» / «Не принято»
       — дата принятия (pd_consent_at в формате DD.MM.YYYY HH:MM UTC, или «—» если NULL)
       — версия политики (pd_consent_version, или «—» если NULL)
And    у пользователя с pd_consent_at = NULL статус отображается как «Не принято» с визуальным предупреждением

Сценарий 2 — Статус согласия для пользователя с устаревшей версией
Given  текущая активная версия политики — v1.1
And    у пользователя pd_consent_version = "v1.0"
When   администратор открывает карточку пользователя
Then   отображается статус «Требуется обновление согласия»
And    отображается: «Согласие дано по версии v1.0. Актуальная версия: v1.1»

Сценарий 3 — Администратор без права на просмотр ПД
Given  у администратора нет права consent:read
When   он открывает карточку пользователя
Then   блок «Согласие на обработку ПД» скрыт или отображается как «—» без деталей
And    прямой запрос к /api/v1/auth/consent-status/{user_id} возвращает 403
```

## US-07 — Экспорт списка согласий в CSV для юридического аудита

**Story**

Как **администратор системы**,
я хочу **выгрузить полный список пользователей со статусом их согласий в CSV-файл**,
чтобы **передать реестр согласий штатному юристу или предъявить при проверке РКН**.

**Acceptance Criteria**

```
Сценарий 1 — Запрос и получение CSV-выгрузки
Given  администратор находится в разделе управления пользователями
When   он нажимает «Экспортировать согласия (CSV)»
Then   формируется запрос к GET /api/v1/admin/users/consent-export
And    в ответе — CSV-файл с заголовком и строками
And    каждая строка содержит: user_id, full_name, email, pd_consent_at (UTC), pd_consent_version, статус (принято/не принято/устарело)
And    файл сохраняется с именем consent-export-YYYY-MM-DD.csv

Сценарий 2 — Выгрузка при большом числе пользователей (> 500)
Given  в системе зарегистрировано более 500 пользователей
When   администратор запрашивает экспорт
Then   запрос выполняется в фоновом режиме без таймаута HTTP
And    администратор получает сообщение «Файл формируется, будет доступен для скачивания через N секунд»
       (конкретный порог N — Open Question OQ-3)
And    по готовности — кнопка «Скачать» или уведомление

Сценарий 3 — Аудит факта экспорта
Given  администратор выполнил экспорт
When   операция завершена
Then   в audit_log записывается: action="consent_export", actor_id, timestamp, rows_count
And    повторная выгрузка создаёт новую audit-запись (каждый экспорт логируется)

Сценарий 4 — Отсутствие прав на экспорт
Given  у администратора нет права consent:export
When   он пытается скачать CSV
Then   API возвращает 403
And    кнопка «Экспортировать согласия» в UI скрыта или неактивна
```

## Open Questions — требуют ответа Владельца

**OQ-1. Кто является оператором ПД в тексте согласия?**
Юридический анализ выявил коллизию: в холдинге несколько юрлиц. В текст согласия должно вставляться наименование и адрес конкретного оператора. Вариант А — каждое юрлицо является отдельным оператором (каждый экземпляр системы использует реквизиты «своего» юрлица из настроек). Вариант Б — холдинг как единый оператор (одно юрлицо-головник обрабатывает данные за всех). Это бизнес-решение с правовыми последствиями. Влияет на: текст согласия в базе, поле `operator_name` в таблице настроек системы.

**OQ-2. Нужен ли экран согласия для пользователей, работающих только по трудовому договору?**
Юридический анализ (раздел 2.1 ревью) указывает: если все пользователи без исключения являются штатными сотрудниками по ТД — трудовое исключение работает, явное согласие заменяется ЛНА. Если есть хоть один подрядчик, агент, стажёр без ТД — согласие обязательно для него. Вопрос к Владельцу: планируются ли пользователи не-сотрудники? Влияет на: наличие или отсутствие упрощённого режима для ТД-пользователей.

**OQ-3. Порог синхронного vs асинхронного экспорта CSV?**
При каком количестве строк экспорт переключается с синхронного (ответ сразу) на асинхронный (фоновая задача + уведомление)? Предложение BA: порог 200 строк — синхронно, выше — задача очереди. Влияет на: архитектурное решение по фоновым задачам (нужна ли очередь уже в M-OS-1).

**OQ-4. Срок действия временного JWT (consent_required: true)?**
Какое время жизни у токена, выданного до принятия согласия? Текущий JWT живёт 60 минут. Для consent-токена можно установить более короткий срок (например, 15 минут) — чтобы нельзя было «подвесить» сессию без согласия. Влияет на: конфигурацию auth-сервиса.

**OQ-5. Нужно ли уведомлять пользователя по email при обновлении политики?**
Или достаточно баннера при следующем входе? Проактивное уведомление по email даёт юридически более сильную позицию (субъект был уведомлён, а не просто «увидел при входе»). Влияет на: наличие email-сервиса в M-OS-1.

## Допущения (Assumptions)

- **A1.** Поля `pd_consent_at` и `pd_consent_version` уже присутствуют в модели `User` (`backend/app/models/user.py`).
- **A2.** MSW-хендлеры для `/api/v1/auth/consent-status` и `/api/v1/auth/accept-consent` существуют на фронтенде — реальный бэкенд этих эндпоинтов ещё не реализован и создаётся в рамках данной US.
- **A3.** Версия политики — строковый семантический тег (например, `"v1.0"`), хранится как конфигурационное значение системы (не в БД пользователя).
- **A4.** Текст согласия версионируется и хранится в системе (не захардкожен во фронтенде).
- **A5.** Аудит-лог (`audit_log`) уже реализован (ADR 0007). Запись согласия — дополнительный action-тип.
- **A6.** Экспорт CSV в US-07 — роль с правом `consent:export` (отдельно от `consent:read`).
- **A7.** Текст согласия из раздела 4 legal-ревью используется как основа; финальная редакция утверждается штатным юристом перед production-gate.

- [ ] POST `/api/v1/auth/accept-consent` реализован: записывает `pd_consent_at`, `pd_consent_version`, возвращает новый JWT

- [ ] JWT содержит клейм `consent_required: true/false`
- [ ] Middleware/guard блокирует доступ к защищённым маршрутам при `consent_required: true`
- [ ] Фронтенд: экран `/consent` с текстом, чекбоксом (unchecked по умолчанию), кнопками «Подтверждаю» и «Отказаться»
- [ ] Фронтенд: кнопка «Подтверждаю» неактивна до отметки чекбокса

- [ ] Интеграционный тест: сценарий первого входа → редирект → принятие → полный доступ
- [ ] Интеграционный тест: сценарий отказа → выход → повторный вход → снова экран согласия

## Глоссарий (термины, использованные в этой US)

| Термин | Определение |
|--------|-------------|
| Согласие на обработку ПД | Явное волеизъявление субъекта ПД, выраженное в электронной форме через отметку чекбокса, на обработку его данных в целях, указанных в тексте согласия |
| consent_required | Клейм (поле) внутри JWT-токена. `true` — пользователь ещё не принял согласие и не имеет доступа к системе. `false` — согласие получено, доступ открыт |
| pd_consent_version | Версия политики обработки ПД, с которой пользователь согласился. Строка вида «v1.0». Хранится в записи пользователя |
| pd_consent_at | Дата и время UTC момента, когда пользователь принял согласие. NULL до первого принятия |
| Активная версия политики | Текущая версия текста согласия, зарегистрированная в системе. При несовпадении с pd_consent_version пользователя — требуется повторное согласие |
| Consent-экран | Страница /consent, отображаемая при consent_required: true. Содержит текст согласия, чекбокс и кнопки |
| ЛНА | Локальный нормативный акт — внутренний документ организации (Положение об обработке ПДн). Требуется 152-ФЗ ст. 18.1. Утверждается штатным юристом. В рамках skeleton — не блокирует разработку |
| РКН | Роскомнадзор — федеральный регулятор в сфере ПДн. Проводит плановые и внеплановые проверки |
