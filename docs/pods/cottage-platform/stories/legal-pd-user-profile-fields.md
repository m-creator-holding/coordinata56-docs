# legal-pd-user-profile-fields — ПД-поля в профиле пользователя и ИП-маркировка в Companies

## Контекст и бизнес-обоснование

Текущая модель пользователя хранит только `email`, `full_name`, `role`. Для оперативной работы холдинга (трудовые договоры, гражданско-правовые договоры, взаимодействие с контрагентами) необходимы расширенные данные: телефон, паспортные реквизиты, дата рождения.

Юридический анализ (документ `ui-pd-labels-review-2026-04-18.md`) выявил два P0-риска:
- P0-2: форма Users не маркирует ПД-поля (нарушение ст. 5 п. 2 152-ФЗ — принцип соответствия целям)
- P0-3: форма Companies не предупреждает о ПД-характере `full_name` для ИП

Данный документ формализует требования к расширению модели и к визуальной маркировке ПД-полей.

**Решение Владельца (2026-04-18):** реализовать skeleton-first. Функциональность — сейчас. Паспорт, телефон, дата рождения добавляются невзирая на отсутствие ЛНА. Compliance — отдельной фазой перед production-gate.

**Новые поля в таблице `users` (требуют Alembic-миграции):**
- `phone VARCHAR(20) NULL`
- `passport_series VARCHAR(4) NULL`
- `passport_number VARCHAR(6) NULL`
- `passport_issued_by VARCHAR(255) NULL`
- `passport_issued_at DATE NULL`
- `date_of_birth DATE NULL`

## US-01 — Форма User с расширенными ПД-полями

**Story**

Как **администратор системы**,
я хочу **видеть и заполнять все ПД-поля пользователя в одной форме** (ФИО, email, телефон, паспортные данные, дата рождения),
чтобы **иметь полный реестр сотрудников и контрагентов без обращения к внешним системам**.

**Acceptance Criteria**

```
Сценарий 1 — Отображение всех полей в форме создания/редактирования User
Given  администратор открывает форму создания нового пользователя ИЛИ форму редактирования существующего
When   форма загружается
Then   форма содержит следующие поля в разделе «Персональные данные»:
       — ФИО (full_name) — обязательное
       — Email — обязательное
       — Телефон (phone) — необязательное
       — Серия паспорта (passport_series) — необязательное
       — Номер паспорта (passport_number) — необязательное
       — Кем выдан (passport_issued_by) — необязательное
       — Дата выдачи паспорта (passport_issued_at) — необязательное
       — Дата рождения (date_of_birth) — необязательное
And    поля, не обязательные для заполнения, явно маркированы (например, «необязательно»)
And    раздел «Персональные данные» визуально отделён от служебных полей (Роль, Активность и т.д.)

Сценарий 2 — Форма редактирования: предзаполнение существующих данных
Given  администратор открывает форму редактирования пользователя, у которого уже заполнены поля
When   форма загружается
Then   все имеющиеся поля предзаполнены текущими значениями из базы
And    паспортные данные отображаются в замаскированном виде (правила маскирования — см. US-05)

Сценарий 3 — Сохранение частично заполненной формы
Given  администратор заполнил только обязательные поля (ФИО и email)
And    оставил все необязательные ПД-поля пустыми
When   администратор нажимает «Сохранить»
Then   пользователь создаётся / обновляется успешно
And    необязательные ПД-поля сохраняются как NULL
And    система не выдаёт ошибку валидации по пустым необязательным полям
```

## US-02 — Визуальная маркировка ПД-полей

**Story**

Как **администратор системы**,
я хочу **видеть рядом с каждым ПД-полем явную метку «ПД» с пояснением**,
чтобы **осознавать, какие данные являются персональными и обрабатываются в соответствии с 152-ФЗ**.

**Acceptance Criteria**

```
Сценарий 1 — Метка «ПД» рядом с каждым персональным полем
Given  администратор открывает форму User (создание или редактирование)
When   форма отображается
Then   рядом с каждым ПД-полем присутствует метка «ПД» (визуальный бейдж)
And    метка применяется к полям: ФИО, Email, Телефон, Серия паспорта, Номер паспорта, Кем выдан, Дата выдачи паспорта, Дата рождения
And    метка «ПД» НЕ применяется к служебным полям: Роль, Активность, Владелец холдинга

Сценарий 2 — Tooltip по клику/ховеру на метку «ПД»
Given  администратор видит метку «ПД» рядом с полем
When   администратор наводит курсор на метку ИЛИ нажимает на неё
Then   отображается всплывающая подсказка (tooltip) с текстом:
       «Это персональные данные. Обрабатываются в соответствии с Федеральным законом № 152-ФЗ "О персональных данных"»
And    в tooltip содержится ссылка на Политику обработки ПД (ведёт на тот же документ, что и в экране согласия)

Сценарий 3 — Цель сбора для каждого ПД-поля
Given  администратор просматривает форму User
When   форма отображается
Then   под каждым ПД-полем (или в tooltip) отображается краткое пояснение цели сбора:
       — ФИО: «Используется для идентификации в системе и в документах холдинга»
       — Email: «Используется для входа в систему и получения уведомлений»
       — Телефон: «Используется для оперативной связи»
       — Паспорт (серия, номер, кем выдан, дата): «Используется для оформления договорных документов»
       — Дата рождения: «Используется для кадровых документов»
And    цели отображаются в виде подписи серым текстом под полем ИЛИ в рамках tooltip
```

## US-03 — Валидация паспортных данных

**Story**

Как **администратор, вводящий паспортные данные пользователя**,
я хочу **получать мгновенную проверку формата серии и номера паспорта**,
чтобы **не допускать ввода заведомо некорректных данных и избежать ошибок при подготовке документов**.

**Acceptance Criteria**

```
Сценарий 1 — Корректный ввод серии паспорта
Given  администратор вводит серию паспорта
When   вводится ровно 4 цифры (например, «4510»)
Then   поле проходит валидацию без ошибок
And    пользователь может продолжить заполнение формы

Сценарий 2 — Некорректная серия паспорта (не 4 цифры)
Given  администратор вводит серию паспорта
When   вводится менее 4 цифр ИЛИ более 4 символов ИЛИ нецифровые символы
Then   под полем появляется сообщение об ошибке: «Серия паспорта — 4 цифры»
And    форму нельзя сохранить до исправления ошибки

Сценарий 3 — Корректный ввод номера паспорта
Given  администратор вводит номер паспорта
When   вводится ровно 6 цифр (например, «123456»)
Then   поле проходит валидацию без ошибок

Сценарий 4 — Некорректный номер паспорта (не 6 цифр)
Given  администратор вводит номер паспорта
When   вводится менее 6 цифр ИЛИ более 6 символов ИЛИ нецифровые символы
Then   под полем появляется сообщение об ошибке: «Номер паспорта — 6 цифр»
And    форму нельзя сохранить до исправления ошибки

Сценарий 5 — Частичное заполнение паспорта (только серия или только номер)
Given  администратор заполнил поле «Серия паспорта», но оставил «Номер паспорта» пустым (или наоборот)
When   администратор пытается сохранить форму
Then   система выдаёт предупреждение: «Паспортные данные заполнены частично. Рекомендуется указать и серию, и номер»
And    (Вопрос к Владельцу OQ-1) — предупреждение блокирует сохранение или только информирует?

Сценарий 6 — Невалидная дата рождения
Given  администратор вводит дату рождения
When   введена дата в будущем ИЛИ дата, при которой возраст менее 14 лет ИЛИ более 100 лет
Then   система отображает ошибку валидации с кратким пояснением
And    форму нельзя сохранить до исправления

Сценарий 7 — Невалидная дата выдачи паспорта
Given  администратор вводит дату выдачи паспорта
When   введена дата в будущем ИЛИ дата раньше 1997 года (год введения нового паспорта РФ)
Then   система отображает ошибку валидации: «Дата выдачи паспорта не может быть в будущем» / «Проверьте дату»
And    при вводе даты раньше 1997 года — предупреждение (не блокирующее), т.к. возможны исторические паспорта

Сценарий 8 — Все ПД-поля паспорта пустые (nullable)
Given  администратор оставил все паспортные поля пустыми
When   форма сохраняется
Then   все паспортные поля записываются как NULL без ошибки
And    это валидное состояние записи пользователя
```

## US-04 — ИП-маркировка в форме Companies

**Story**

Как **администратор, вводящий данные компании типа ИП**,
я хочу **видеть явное предупреждение, что поле «Полное наименование» содержит ПД физического лица**,
чтобы **понимать юридическую природу этих данных и обрабатывать их с соответствующей осторожностью**.

**Acceptance Criteria**

```
Сценарий 1 — Поведение формы Companies при выборе типа «ИП»
Given  администратор открывает форму создания или редактирования компании
When   администратор выбирает значение «ИП» в поле «Тип организации» (company_type)
Then   метка поля «Полное наименование» изменяется на «Полное наименование (ФИО предпринимателя)»
And    под полем отображается подсказка:
       «Для ИП это поле содержит ФИО физического лица — персональные данные. Обрабатываются в соответствии с 152-ФЗ.»
And    рядом с полем «Полное наименование» добавляется метка «ПД» (аналогично форме User)

Сценарий 2 — Поведение формы Companies при типе «ООО» / «АО» и других юрлицах
Given  администратор выбирает company_type != «ИП» (например, «ООО»)
When   форма отображается
Then   поле «Полное наименование» имеет стандартную метку без упоминания ПД
And    метка «ПД» рядом с полем НЕ отображается
And    подсказка об ИП НЕ отображается

Сценарий 3 — Смена типа с «ООО» на «ИП» без перезагрузки страницы
Given  администратор выбрал «ООО» и начал заполнять «Полное наименование»
When   администратор меняет тип на «ИП»
Then   метка и подсказка появляются без перезагрузки страницы (реактивно, без перехода на другой экран)
And    введённый текст в поле «Полное наименование» не сбрасывается

Сценарий 4 — Предупреждение при сохранении записи ИП
Given  администратор заполнил форму с типом «ИП»
When   администратор нажимает «Сохранить»
Then   система сохраняет запись без блокирующих предупреждений (пользователь уже предупреждён в форме)
And    в audit_log записывается создание/изменение компании типа ИП с пометкой о ПД (action="company_ip_pd_field_updated")
       (Вопрос к Владельцу OQ-2 — нужна ли отдельная audit-метка для ИП или достаточно стандартной записи об изменении)
```

## US-05 — Маскирование паспортных данных в карточке пользователя

**Story**

Как **администратор, просматривающий карточку пользователя**,
я хочу **видеть паспортные данные в маскированном виде по умолчанию**,
а при необходимости — **явно запросить показ полных данных** (с фиксацией в аудите),
чтобы **минимизировать случайное раскрытие чувствительной информации на экране**.

**Acceptance Criteria**

```
Сценарий 1 — Карточка пользователя: паспорт в маскированном виде
Given  администратор открывает карточку пользователя с заполненными паспортными данными
When   страница загружается
Then   серия паспорта отображается как «** **» (4 символа замаскированы)
And    номер паспорта отображается как «** ** XX» (первые 4 цифры замаскированы, последние 2 видны)
       (Вопрос к Владельцу OQ-3 — конкретный шаблон маскирования)
And    рядом с замаскированными полями присутствует кнопка или ссылка «Показать полностью»

Сценарий 2 — Показ полных паспортных данных (раскрытие)
Given  администратор видит замаскированные паспортные данные
When   администратор нажимает «Показать полностью»
Then   поля раскрываются и показывают полные значения
And    в audit_log записывается: action="passport_data_revealed", actor_id, subject_user_id, timestamp, ip_address
And    кнопка меняется на «Скрыть»
And    при уходе со страницы и возврате обратно данные снова маскируются (не кэшируются в открытом виде)

Сценарий 3 — Отсутствие паспортных данных
Given  у пользователя поля passport_series, passport_number = NULL
When   администратор открывает карточку
Then   поля паспорта отображаются как «—» (прочерк) без кнопки «Показать полностью»

Сценарий 4 — Администратор без права на просмотр паспортных данных
Given  у администратора нет права passport:read
When   он открывает карточку пользователя
Then   поля паспорта не отображаются вообще (скрыты, а не замаскированы)
And    кнопки «Показать полностью» нет
And    запрос к API с паспортными полями возвращает 403 или эти поля исключены из ответа
```

## US-06 — Экспорт данных пользователя по запросу субъекта (ст. 14 152-ФЗ)

**Story**

Как **администратор, исполняющий запрос субъекта ПД**,
я хочу **выгрузить все персональные данные конкретного пользователя в структурированном виде**,
чтобы **выполнить право субъекта на получение сведений об обрабатываемых данных (ст. 14 152-ФЗ) в срок не более 30 дней**.

**Acceptance Criteria**

```
Сценарий 1 — Запрос на экспорт данных конкретного пользователя
Given  администратор находится в карточке пользователя
When   он нажимает «Выгрузить данные (ст. 14 152-ФЗ)»
Then   формируется запрос к GET /api/v1/admin/users/{id}/pd-export
And    в ответе — структурированный файл (JSON или CSV — OQ-4) с полным набором ПД пользователя:
       id, full_name, email, phone, passport_series, passport_number, passport_issued_by,
       passport_issued_at, date_of_birth, pd_consent_at, pd_consent_version, created_at,
       роли и привязки к компаниям
And    файл скачивается с именем pd-export-{user_id}-YYYY-MM-DD.*

Сценарий 2 — Аудит факта экспорта
Given  администратор выполнил выгрузку ПД
When   операция завершена
Then   в audit_log: action="pd_subject_export", actor_id, subject_user_id, timestamp, file_hash

Сценарий 3 — Данные, которые НЕ включаются в экспорт
Given  администратор запрашивает экспорт ПД пользователя
When   файл формируется
Then   в файл НЕ включаются: хеш пароля, JWT-токены, системные технические поля (updated_at, deleted_at)
And    включаются только те поля, которые идентифицируют или описывают субъекта ПД

Сценарий 4 — Отсутствие прав на экспорт ПД
Given  у администратора нет права pd:export
When   он пытается выполнить выгрузку
Then   API возвращает 403
And    кнопка «Выгрузить данные» в карточке скрыта или неактивна
```

## US-07 — Удаление ПД пользователя по запросу субъекта (ст. 14 152-ФЗ)

**Story**

Как **администратор, исполняющий требование субъекта об удалении данных**,
я хочу **выполнить удаление всех персональных данных пользователя** (с сохранением технической записи о факте удаления),
чтобы **выполнить право субъекта на удаление данных (ст. 14 152-ФЗ) и одновременно сохранить доказательство факта исполнения требования**.

**Acceptance Criteria**

```
Сценарий 1 — Удаление ПД пользователя (soft delete + маскирование)
Given  администратор находится в карточке пользователя
When   он нажимает «Удалить персональные данные (ст. 14)» и подтверждает действие в диалоге
Then   POST /api/v1/admin/users/{id}/pd-erase выполняется
And    следующие поля обнуляются (NULL): full_name, email, phone, passport_series, passport_number,
       passport_issued_by, passport_issued_at, date_of_birth, pd_consent_at, pd_consent_version
And    запись пользователя НЕ удаляется физически (id, created_at, системные роли сохраняются)
And    поле is_active устанавливается в false
And    audit_log: action="pd_erased", actor_id, subject_user_id, erased_fields_list, timestamp, ip_address

Сценарий 2 — Отображение карточки после удаления ПД
Given  ПД пользователя были удалены
When   администратор открывает карточку этого пользователя
Then   ПД-поля отображаются как «[данные удалены]» (не пустая строка, а явная метка)
And    отображается информационный блок: «Персональные данные удалены по запросу субъекта [дата]»
And    кнопки «Редактировать ПД» и «Выгрузить данные» — скрыты или неактивны

Сценарий 3 — Блокировка повторного удаления
Given  ПД пользователя уже были удалены ранее
When   кто-то пытается повторно вызвать pd-erase
Then   API возвращает 409 с кодом PD_ALREADY_ERASED
And    бизнес-правило: повторное удаление невозможно без явного разблокирования Владельцем

Сценарий 4 — Попытка удаления при наличии активных договоров/обязательств
Given  пользователь связан с активными записями (например, активные договора, незакрытые платежи)
When   администратор пытается выполнить pd-erase
Then   система возвращает предупреждение: «Пользователь связан с активными записями. Удаление ПД может нарушить целостность данных»
And    (Вопрос к Владельцу OQ-5) — предупреждение блокирует операцию или оставляет выбор администратору?

Сценарий 5 — Отсутствие прав на удаление ПД
```

## Open Questions — требуют ответа Владельца

**OQ-1. Частичное заполнение паспорта — блокировать сохранение или предупреждать?**
Если введена серия, но не введён номер (или наоборот) — является ли это ошибкой, которая блокирует сохранение формы, или только предупреждением? С точки зрения бизнеса неполный паспорт бесполезен для документов, но возможны ситуации, когда данные добавляются поэтапно.

**OQ-2. Нужна ли отдельная audit-запись при сохранении данных компании типа ИП?**
Стандартная запись аудита при изменении компании уже создаётся. Вопрос: нужно ли маркировать её отдельным action-типом `company_ip_pd_field_updated`, или достаточно стандартной записи `company_updated`? Более детальная метка полезна при аудиторских запросах.

**OQ-3. Конкретный шаблон маскирования паспортных данных в карточке.**
Предложение BA: серия — «** **» (все 4 символа скрыты), номер — «** **XX» (первые 4 скрыты, последние 2 видны). Альтернатива: серия — «45 **», номер — «**3456». Какой уровень маскирования достаточен для операционных нужд (распознать при необходимости) и при этом не раскрывает данные при случайном наблюдении?

**OQ-4. Формат файла экспорта ПД по ст. 14 — JSON или CSV?**
JSON — более структурирован, удобен для передачи юристу или в госорган. CSV — удобен для открытия в Excel. Возможен вариант «оба формата по выбору». Влияет на реализацию эндпоинта и UI кнопки.

**OQ-5. Удаление ПД при наличии активных связанных записей — блокировать или предупреждать?**
Если у пользователя есть активные договора или незакрытые платежи — удаление его ПД нарушает связность данных (в договоре останется «пустой» контакт). Варианты: А — блокировать pd-erase до закрытия связанных записей; Б — предупреждать, но позволять выполнить; В — предупреждать и требовать дополнительного подтверждения с пометкой «я осознаю последствия». Юридически право на удаление имеет приоритет, но операционно вариант А безопаснее для целостности данных.

## Допущения (Assumptions)

- **A1.** Новые поля добавляются через Alembic-миграцию, все новые поля — nullable (expand-фаза паттерна expand/contract).
- **A2.** Маскирование паспортных данных реализуется на бэкенде: API по умолчанию возвращает маскированные значения, а полные данные — только при наличии права `passport:read`.
- **A3.** Право `pd:erase` — отдельное от стандартных прав администратора. По умолчанию не выдаётся рядовому администратору.
- **A4.** «Удаление» ПД в контексте US-07 — это обнуление полей (soft erasure), а не физическое удаление строки пользователя. Техническая запись (id, created_at, аудит) сохраняется для доказательства факта исполнения требования.
- **A5.** Визуальная метка «ПД» и tooltip реализуются как переиспользуемый UI-компонент (используется и в форме Users, и в форме Companies для ИП).
- **A6.** СНИЛС и ИНН физлица в модель Users в рамках данной US не добавляются (требуют отдельного решения Владельца — Open Question в companion-документе `legal-pd-consent-flow.md`).
- **A7.** Валидация форматов (серия: 4 цифры, номер: 6 цифр) реализуется и на фронтенде (UX), и на бэкенде (Pydantic) — двойная защита.

## Обновление глоссария

Следующие термины добавлены в `docs/knowledge/glossary.md` при публикации данной US:

| Термин | Определение |
|--------|-------------|
| ПД-маркировка | Визуальная метка «ПД» рядом с полем формы, сообщающая администратору о персональном характере данных. Сопровождается tooltip с пояснением цели сбора и ссылкой на 152-ФЗ |
| Soft erasure (мягкое обнуление) | Способ «удаления» ПД, при котором ПД-поля записи обнуляются (NULL), но сама строка сохраняется в базе. Применяется при исполнении требований ст. 14 152-ФЗ для сохранения доказательства факта обработки запроса |
| passport:read | Право доступа, позволяющее получить немаскированные паспортные данные пользователя через API. Не входит в стандартный набор прав администратора |
| pd:erase | Право доступа на выполнение операции обнуления ПД пользователя (pd-erase). Отдельное от прав удаления пользователя как учётной записи |
| pd:export | Право доступа на выгрузку полного набора ПД конкретного пользователя (ст. 14 152-ФЗ) |
| ИП | Индивидуальный предприниматель. В контексте M-OS — тип компании (`company_type = "ip"`), при котором поле `full_name` содержит ФИО физического лица и является персональными данными по 152-ФЗ |
