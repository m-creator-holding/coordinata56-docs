# FE-W1-3 — Управление ролями (Admin UI)

## Контекст

Экран «Роли» (`/admin/roles`) — третий экран администраторской панели, следующий за Users. Позволяет Владельцу холдинга просматривать, создавать и редактировать роли системы. Роль — именованный набор прав доступа, привязываемый к пользователю в контексте конкретной компании. Системные роли (owner, accountant, construction_manager, read_only) поставляются через seed-скрипт и не удаляются. Пользовательские роли создаются через этот экран.

---

## Глоссарий (к данным US)

| Термин | Определение |
|--------|-------------|
| Роль (Role) | Именованная сущность с кодом (snake_case) и набором прав; привязывается к пользователю через UserCompanyRole |
| Системная роль | Роль, созданная seed-скриптом; не может быть удалена; поле `is_system = true` |
| Пользовательская роль | Роль, созданная через UI администратором; может быть удалена, если не используется |
| Область действия (scope) | «Глобальная» — роль действует во всех компаниях холдинга; «Компания» — только в конкретной компании |
| Код роли (role_code) | Уникальный технический идентификатор роли в snake_case; после создания не изменяется |
| UserCompanyRole | Сущность-привязка: пользователь + компания + роль (+ опционально pod) |
| Permission | Право на конкретное действие над ресурсом (CRUD-операция × resource_type); управляется через матрицу прав (Экран 4) |

## US-01 — Admin просматривает список ролей

**Story:**
Как Владелец холдинга, я хочу видеть полный список ролей системы с их названиями, кодами, областью и типом (системная / пользовательская), чтобы понимать, какие роли существуют, и быстро находить нужную для дальнейшей настройки.

### Acceptance Criteria

**Сценарий 1.1 — Стандартная загрузка списка**

```
Given  администратор аутентифицирован и открывает /admin/roles
When   браузер выполняет GET /api/v1/roles
Then   отображается таблица с колонками: Название, Код, Область, Статус
And    каждая строка содержит корректные данные из API
And    системные роли (owner, accountant, construction_manager, read_only) отображаются с Badge «Системная»
And    пользовательские роли отображаются с Badge «Пользовательская»
And    при наведении на Название роли показывается Tooltip с первыми 80 символами описания роли
And    счётчик внизу таблицы показывает «Показано N из N»
```

**Сценарий 1.2 — Загрузка (shimmer-состояние)**

```
Given  браузер отправил GET /api/v1/roles и ожидает ответа
When   запрос ещё не завершён
Then   в таблице отображаются shimmer-строки вместо данных
And    кнопка «+ Создать роль» остаётся активной
```

**Сценарий 1.3 — Пустой список**

```
Given  GET /api/v1/roles вернул пустой массив
When   страница отрендерена
Then   таблица не отображается
And   на странице отображается сообщение «Ролей ещё нет. Проверьте, что seed-скрипт выполнен.»
```

**Сценарий 1.4 — Ошибка загрузки**

```
Given  GET /api/v1/roles вернул статус 5xx или сеть недоступна
When   запрос завершился с ошибкой
Then   отображается Banner «Не удалось загрузить роли. [Повторить]»
And    клик по «Повторить» повторяет GET /api/v1/roles
And    содержимое таблицы не показывает частичные/устаревшие данные
```

**Сценарий 1.5 — Навигация в карточку роли**

```
Given  список ролей загружен
When   администратор кликает на строку любой роли
Then   браузер переходит на /admin/roles/:id
And    URL содержит корректный id роли из API-ответа
```

### Edge Cases

- При 10+ ролях Tooltip становится критичным для идентификации — обязателен для всех ролей, включая те, у которых описание пустое (в этом случае Tooltip не отображается совсем, а не показывает пустую подсказку).
- Сортировка по колонке «Название» (сортировка предусмотрена в wireframe для данной колонки); колонки «Код», «Область», «Статус» — не сортируемые.

### API

- `GET /api/v1/roles` — список всех ролей

## US-02 — Admin создаёт роль с permissions

**Story:**
Как Владелец холдинга, я хочу создать новую роль с уникальным кодом, названием, описанием и областью действия, чтобы гибко разграничить доступ для нестандартных должностей (например, «Старший менеджер продаж»).

### Acceptance Criteria

**Сценарий 2.1 — Успешное создание роли**

```
Given  администратор находится на /admin/roles
When   он нажимает «+ Создать роль»
Then   открывается Sheet (боковая панель) с формой создания роли
And    форма содержит поля: Код роли, Отображаемое название, Описание (textarea), Область действия (select)
And    поле «Код роли» активно для ввода (в отличие от режима редактирования)
```

```
Given  форма создания роли открыта
And    администратор заполнил: Код роли = «senior_manager», Название = «Старший менеджер», Область = «Компания»
When   он нажимает «Создать роль»
Then   браузер выполняет POST /api/v1/roles с телом {"code": "senior_manager", "name": "Старший менеджер", "scope": "company"}
And    при статусе 201 Sheet закрывается
And    таблица ролей обновляется и содержит новую строку «Старший менеджер»
And    Toast «Роль создана» отображается 3–5 секунд
And    новая роль помечена Badge «Пользовательская» (не «Системная»)
```

**Сценарий 2.2 — Валидация поля «Код роли»**

```
Given  форма открыта
When   администратор вводит код, не соответствующий snake_case (например, «Senior Manager» или «seniorManager»)
Then   под полем отображается ошибка «Только латинские буквы и подчёркивание (snake_case). Пример: senior_manager»
And    кнопка «Создать роль» остаётся неактивной до исправления ошибки
```

**Сценарий 2.3 — Код роли уже занят**

```
Given  форма заполнена с кодом «accountant»
When   администратор нажимает «Создать роль»
And    POST /api/v1/roles возвращает 409 Conflict
Then   под полем «Код роли» отображается ошибка «Роль с таким кодом уже существует»
And    Sheet остаётся открытым
And    остальные поля не очищаются
```

**Сценарий 2.4 — Обязательные поля не заполнены**

```
Given  форма открыта и администратор оставил пустыми «Код роли» или «Название»
When   он нажимает «Создать роль»
Then   под незаполненными обязательными полями отображаются ошибки валидации
And    POST-запрос не отправляется
```

**Сценарий 2.5 — Отмена создания**

```
Given  форма открыта и частично заполнена
When   администратор нажимает «Отменить» или закрывает Sheet крестиком
Then   Sheet закрывается без сохранения данных
And    таблица ролей не изменяется
And    запрос POST /api/v1/roles не выполняется
```

### Edge Cases

- Поле «Описание» необязательно; если оставлено пустым — Tooltip для этой роли не отображается (см. US-01 edge case).
- Длина описания не более 512 символов — валидация на клиенте с счётчиком символов.
- Код роли фиксируется после создания: при следующем открытии карточки поле «Код роли» — read-only. Это должно быть явно отражено в help-тексте: «После сохранения не изменяется».

### API

- `POST /api/v1/roles` — создание роли

## US-03 — Admin редактирует роль (rename, описание)

**Story:**
Как Владелец холдинга, я хочу изменять отображаемое название и описание существующей роли, не затрагивая её код, чтобы корректировать формулировки без пересоздания роли и переназначения пользователей.

### Acceptance Criteria

**Сценарий 3.1 — Открытие формы редактирования**

```
Given  администратор находится на /admin/roles/:id (карточка роли)
When   он нажимает «Редактировать»
Then   открывается Sheet с формой редактирования
And    поле «Код роли» отображается как read-only (Input disabled)
And    поля «Название», «Описание», «Область действия» заполнены текущими значениями из API
And    кнопка «Создать роль» заменена кнопкой «Сохранить изменения»
```

**Сценарий 3.2 — Успешное сохранение изменений**

```
Given  форма редактирования открыта
And    администратор изменил Название с «Бухгалтер» на «Финансовый контролёр»
When   он нажимает «Сохранить изменения»
Then   браузер выполняет PATCH /api/v1/roles/:id с телом {"name": "Финансовый контролёр"}
And    при статусе 200 Sheet закрывается
And    на карточке роли отображается обновлённое название «Финансовый контролёр»
And    Toast «Изменения сохранены» отображается 3–5 секунд
```

**Сценарий 3.3 — Редактирование системной роли**

```
Given  администратор открыл карточку системной роли (например, «Прораб», is_system=true)
When   он открывает форму редактирования
Then   поля «Название» и «Описание» доступны для изменения
And    поле «Область действия» — read-only (системные роли имеют фиксированный scope)
And    Badge «Системная» отображается как информационный элемент без кнопки «Удалить» в интерфейсе
```

**Сценарий 3.4 — Пустое название при сохранении**

```
Given  форма редактирования открыта
When   администратор очищает поле «Название» и нажимает «Сохранить изменения»
Then   под полем «Название» отображается ошибка «Поле обязательно для заполнения»
And    PATCH-запрос не отправляется
```

### Edge Cases

- PATCH-запрос отправляется только с изменёнными полями (partial update); если администратор открыл Sheet и не изменил ничего — кнопка «Сохранить изменения» неактивна или запрос не отправляется.
- После успешного PATCH карточка роли должна показывать актуальные данные без ручного обновления страницы (refetch или optimistic update).

### API

- `GET /api/v1/roles/:id` — загрузка текущих данных для формы
- `PATCH /api/v1/roles/:id` — сохранение изменений

## US-04 — Admin удаляет роль (с проверкой «не используется»)

**Story:**
Как Владелец холдинга, я хочу удалить пользовательскую роль, которая больше не нужна, с защитой от случайного удаления роли, к которой привязаны активные пользователи, чтобы не нарушить чужой доступ к системе.

### Acceptance Criteria

**Сценарий 4.1 — Успешное удаление роли без пользователей**

```
Given  администратор открыл карточку пользовательской роли (is_system=false)
And    к данной роли не привязан ни один активный пользователь
When   он нажимает кнопку «Удалить»
Then   отображается Dialog подтверждения: «Вы уверены, что хотите удалить роль «Название»? Это действие необратимо.»
And    при нажатии «Удалить» в Dialog выполняется DELETE /api/v1/roles/:id
And    при статусе 204 Dialog закрывается
And    браузер перенаправляет на /admin/roles
And    удалённая роль отсутствует в таблице
And    Toast «Роль удалена» отображается 3–5 секунд
```

**Сценарий 4.2 — Попытка удаления роли с привязанными пользователями**

```
Given  администратор пытается удалить роль, к которой привязан хотя бы один активный пользователь
When   DELETE /api/v1/roles/:id возвращает 409 Conflict (тело: {"error": {"code": "ROLE_IN_USE", ...}})
Then   Dialog подтверждения закрывается
And    отображается Toast с ошибкой «Нельзя удалить роль: она назначена активным пользователям. Сначала снимите назначения.»
And    роль остаётся в системе без изменений
```

**Сценарий 4.3 — Попытка удаления системной роли**

```
Given  администратор открыл карточку системной роли (is_system=true)
When   страница отрендерена
Then   кнопка «Удалить» отсутствует на UI
And    Badge «Системная» имеет Tooltip «Системные роли созданы автоматически и не могут быть удалены»
And    прямой DELETE /api/v1/roles/:id для системной роли возвращает 403 Forbidden (защита на уровне API)
```

**Сценарий 4.4 — Отмена удаления**

```
Given  Dialog подтверждения удаления открыт
When   администратор нажимает «Отменить» или закрывает Dialog
Then   Dialog закрывается без выполнения DELETE-запроса
And    роль остаётся в системе
```

### Edge Cases

- Проверка занятости роли (ROLE_IN_USE) должна учитывать только активных пользователей (`is_active=true`). Пользователи с `is_active=false` не блокируют удаление.
- После успешного удаления и редиректа на `/admin/roles` таблица обновляется автоматически (не требует ручного F5).

### API

- `DELETE /api/v1/roles/:id` — удаление роли (возвращает 204 или 409 ROLE_IN_USE или 403 для системных)

## US-05 — Admin bulk-select permissions для роли

**Story:**
Как Владелец холдинга, я хочу переходить к матрице прав с предустановленным фильтром по выбранной роли, чтобы быстро видеть и редактировать именно её разрешения, не листая полную матрицу.

> Примечание: прямое редактирование permissions внутри экрана /admin/roles не предусмотрено wireframe'ом. Вкладка «Права» на карточке роли — навигационная точка входа. Bulk-select permissions происходит на Экране 4 (Permissions Matrix). Данная US описывает UX-связку «роль → матрица с предфильтром».

### Acceptance Criteria

**Сценарий 5.1 — Переход в матрицу прав из карточки роли**

```
Given  администратор находится на /admin/roles/:id
And    активна вкладка «Права»
When   он нажимает «Открыть матрицу прав для этой роли»
Then   браузер переходит на /admin/permissions?role=<code>
And    на экране Permissions Matrix строка данной роли выделена (highlight)
And    таблица отображает все роли, но строка <code> визуально выделена для ориентира
```

**Сценарий 5.2 — Вкладка «Права» на карточке роли**

```
Given  администратор открыл карточку роли /admin/roles/:id
When   он кликает на вкладку «Права»
Then   вкладка становится активной
And    отображается текст «Управление правами роли «Название». Перейдите в Матрицу прав для редактирования разрешений.»
And    присутствует кнопка «Открыть матрицу прав для этой роли»
And    кнопка содержит корректный code роли в целевом URL (/admin/permissions?role=<code>)
```

**Сценарий 5.3 — Вкладка «Права» для системной роли**

```
Given  администратор открыл карточку системной роли
When   он переходит на вкладку «Права»
Then   поведение идентично сценарию 5.2
And    кнопка перехода в матрицу прав доступна (просмотр прав системной роли разрешён)
```

### Edge Cases

- Если `code` роли содержит спецсимволы (не должен по валидации snake_case, но защита нужна) — URL encode при формировании query param.
- При переходе с предфильтром и отсутствии роли в матрице (несинхронизированный кэш) — Экран 4 показывает всю матрицу без выделения, без ошибки.

### API

- `GET /api/v1/roles/permissions` — получение матрицы прав (вызывается Экраном 4)
- `PATCH /api/v1/roles/permissions` — пакетное обновление прав (вызывается Экраном 4)

## US-06 — Admin назначает роль пользователю (через /user_roles)

**Story:**
Как Владелец холдинга, я хочу привязывать роль к пользователю в контексте конкретной компании (и опционально пода), чтобы точно контролировать, кто и где имеет доступ в рамках холдинга.

> Примечание: назначение роли пользователю технически выполняется с экрана карточки пользователя (/admin/users/:id), но данные о доступных ролях берутся из /api/v1/roles. US описывает полный сценарий назначения со стороны управления ролями.

### Acceptance Criteria

**Сценарий 6.1 — Открытие Dialog привязки роли**

```
Given  администратор находится на /admin/users/:id (карточка пользователя)
And    он находится на вкладке/секции с таблицей ролей пользователя
When   он нажимает «+ Добавить привязку»
Then   открывается Dialog «Добавить привязку роли»
And    Dialog содержит Select «Компания» (данные из /api/v1/companies)
And    Dialog содержит Select «Роль» (данные из /api/v1/roles)
And    Dialog содержит опциональный Select «Pod» с подсказкой «Оставьте пустым для доступа ко всем подам компании»
```

**Сценарий 6.2 — Успешное назначение роли**

```
Given  Dialog привязки роли открыт
And    администратор выбрал Компанию и Роль (Pod не заполнен)
When   он нажимает «Сохранить»
Then   браузер выполняет POST /api/v1/users/:id/roles с телом {"company_id": <id>, "role_id": <id>}
And    при статусе 201 Dialog закрывается
And    новая строка с Компанией, Ролью и Pod (пусто) появляется в таблице ролей пользователя
And    Toast «Привязка добавлена» отображается 3–5 секунд
```

**Сценарий 6.3 — Дублирующая привязка**

```
Given  у пользователя уже есть привязка «Компания A — Роль Бухгалтер»
When   администратор пытается добавить ещё раз «Компания A — Роль Бухгалтер»
And    POST /api/v1/users/:id/roles возвращает 409 Conflict
Then   Dialog остаётся открытым
And    отображается ошибка «Данная роль для этой компании уже назначена пользователю»
And    существующая привязка не дублируется
```

**Сценарий 6.4 — Удаление привязки роли**

```
Given  в таблице ролей пользователя есть хотя бы одна строка привязки
When   администратор нажимает «Удалить» в строке привязки
Then   открывается Dialog подтверждения «Снять роль «Название» для компании «Название»?»
And    при подтверждении выполняется DELETE /api/v1/users/:id/roles/:role_id
And    при статусе 204 строка удаляется из таблицы без перезагрузки страницы
And    Toast «Привязка удалена» отображается 3–5 секунд
```

**Сценарий 6.5 — Обязательные поля не выбраны**

```
Given  Dialog привязки открыт
When   администратор оставил пустым «Компания» или «Роль» и нажимает «Сохранить»
Then   под незаполненными полями отображаются ошибки «Выбор обязателен»
And    POST-запрос не отправляется
```

### Edge Cases

- Select «Роль» должен загружать актуальный список из /api/v1/roles в момент открытия Dialog (не из кэша списка ролей страницы /admin/roles).
- У одного пользователя может быть несколько привязок в разных компаниях с разными ролями — это нормально.
- Удаление привязки не влияет на саму роль в системе — удаляется только связка пользователь-компания-роль.
- Заблокированный пользователь (`is_active=false`) технически может иметь привязки, но не может войти в систему. Привязки при деактивации не удаляются автоматически.

### API

- `GET /api/v1/users/:id/roles` — список текущих привязок пользователя
- `POST /api/v1/users/:id/roles` — добавление привязки (тело: company_id, role_id, pod_id?)
- `PATCH /api/v1/users/:id/roles/:role_id` — изменение привязки
- `DELETE /api/v1/users/:id/roles/:role_id` — удаление привязки
- `GET /api/v1/roles` — список ролей для Select в Dialog

## Критерии готовности (Definition of Done)

- [ ] GET /api/v1/roles возвращает список с полями: id, code, name, description, scope, is_system
- [ ] POST /api/v1/roles создаёт роль, валидирует snake_case для code, возвращает 409 при дублировании code
- [ ] PATCH /api/v1/roles/:id принимает частичное обновление (name, description); code не изменяется
- [ ] DELETE /api/v1/roles/:id возвращает 403 для системных ролей, 409 ROLE_IN_USE при активных пользователях, 204 при успехе
- [ ] Страница /admin/roles отображает таблицу с shimmer, empty-state и error Banner
- [ ] Tooltip с описанием роли (до 80 символов) реализован через shadcn/ui Tooltip
- [ ] Sheet создания/редактирования роли корректно работает (валидация, submit, закрытие, Toast)
- [ ] Кнопка «Удалить» отсутствует для системных ролей; Badge «Системная» имеет Tooltip с пояснением
- [ ] Вкладка «Права» на карточке роли содержит корректную ссылку /admin/permissions?role=<code>
- [ ] Dialog привязки роли к пользователю работает через /api/v1/users/:id/roles (создание и удаление)
- [ ] Все write-эндпоинты пишут запись в audit_log в рамках той же транзакции (ADR 0007)
- [ ] Ошибки API соответствуют формату ADR 0005: {"error": {"code", "message", "details"}}
- [ ] Неаутентифицированный запрос к любому /api/v1/roles* или /api/v1/users/:id/roles возвращает 401

## Допущения (Assumptions)

- **A1.** Системные роли определяются полем `is_system=true` в ответе API; фронтенд не хардкодит список системных кодов.
- **A2.** Код роли после создания не изменяется — поле `code` отсутствует в теле PATCH-запроса; backend игнорирует или отклоняет его при попытке передать.
- **A3.** Проверка ROLE_IN_USE учитывает только пользователей с `is_active=true`.
- **A4.** Редактирование permissions (прав) происходит на Экране 4 (Permissions Matrix); Экран 3 только navigates туда с предфильтром.
- **A5.** Pod в привязке UserCompanyRole — опциональное поле; отсутствие означает «все поды компании».
- **A6.** Область действия (scope) существующей роли не изменяется через форму редактирования (только Название и Описание); для системных ролей scope read-only явно.

## Open Questions

**OQ-1. Можно ли изменить scope («Область действия») при редактировании пользовательской роли?**
Wireframe не описывает это явно. Если пользовательская роль создана со scope «Компания», может ли администратор изменить его на «Глобальная»? Влияет на: состав полей в PATCH-запросе, логику UserCompanyRole.

**OQ-2. Нужна ли сортировка по колонке «Статус» или «Область» в таблице ролей?**
Wireframe указывает сортировку только для «Название». Системные роли отображаются сверху или вперемешку с пользовательскими?
