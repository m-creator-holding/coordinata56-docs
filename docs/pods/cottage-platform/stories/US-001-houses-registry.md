# US-001 — Учёт домов посёлка

## Story

Как **владелец посёлка (Мартин)**,
я хочу **видеть полный список всех 85 домов с их типом, текущей стадией строительства и ключевыми характеристиками, а также открывать детальную карточку каждого дома**,
чтобы **в любой момент иметь актуальную картину состояния проекта и принимать управленческие решения на основе факта, а не со слов сотрудников**.

---

## Контекст и бизнес-обоснование

Посёлок «Координата 56» — 85 домов, 4 типовых проекта (A, B, C, D) с набором опций. Строительство ведётся параллельно по нескольким участкам. Без единого реестра владелец вынужден собирать статусы вручную через прорабов. Данная User Story формирует фундаментальный экран всей системы — «реестр домов» — от которого отходят все остальные модули (бюджет, материалы, подрядчики, аудит).

---

## Acceptance Criteria

### Сценарий 1 — Просмотр полного списка домов

```
Given  пользователь с ролью owner вошёл в систему
And    в базе данных присутствуют 85 домов с заполненными полями
When   пользователь открывает раздел «Дома»
Then   система отображает список всех 85 домов (без пагинации или с пагинацией — решить в Open Q1)
And    каждая строка содержит: номер участка, тип дома (A/B/C/D), текущую стадию строительства, статус дома (см. Open Q2)
And    список отсортирован по умолчанию по номеру участка по возрастанию
And    количество записей в списке совпадает с реальным числом домов в базе
And    время загрузки списка не превышает 3 секунд при стандартном соединении
```

### Сценарий 2 — Фильтрация списка по типу дома

```
Given  пользователь находится на странице «Дома»
And    в списке отображаются дома всех типов
When   пользователь выбирает фильтр по типу «B»
Then   система отображает только дома типа B (ожидаемо 30 штук согласно seeds)
And    счётчик записей обновляется и показывает количество отфильтрованных домов
And    при сбросе фильтра список возвращается к полному составу из 85 домов
```

### Сценарий 3 — Просмотр карточки отдельного дома

```
Given  пользователь находится на странице «Дома»
When   пользователь кликает на любой дом в списке
Then   система открывает карточку этого дома
And    карточка содержит: номер участка, тип дома, текущую стадию строительства, список выбранных опций (если есть), историю смены стадий (дата и стадия)
And    на карточке присутствуют ссылки/вкладки для перехода к данным о бюджете и материалах этого дома (даже если разделы ещё не реализованы — кнопки неактивны с меткой «скоро»)
And    карточка открывается за не более чем 2 секунды
```

### Сценарий 4 — Пустое состояние (нет домов в базе)

```
Given  пользователь вошёл в систему
And    в базе данных нет ни одного дома (например, на чистом окружении до seeds)
When   пользователь открывает раздел «Дома»
Then   система не падает с ошибкой и не показывает пустую белую страницу
And    система отображает информативное сообщение: «Дома ещё не добавлены. Обратитесь к администратору»
And    кнопка «Добавить дом» отображается, если роль пользователя это позволяет (см. Open Q3)
```

### Сценарий 5 — Фильтрация по стадии строительства

```
Given  пользователь находится на странице «Дома»
When   пользователь применяет фильтр по стадии «Фундамент»
Then   список отображает только дома, у которых текущая стадия — «Фундамент»
And    дома на других стадиях не отображаются
And    при одновременном применении фильтра по типу и фильтра по стадии список показывает пересечение обоих условий (логика AND)
```

### Сценарий 6 — Доступ с ролью read_only

```
Given  пользователь вошёл в систему с ролью read_only
When   пользователь открывает раздел «Дома» и карточку любого дома
Then   данные отображаются в полном объёме (аналогично owner)
And    кнопки редактирования, изменения статуса, добавления дома — скрыты или неактивны
And    система не возвращает ошибку доступа при GET-запросах
```

## Open Questions — требуют ответа Владельца

**OQ-1. Пагинация или бесконечный скролл?**
85 домов — относительно небольшой список, но он будет расти при масштабировании холдинга. Нужно ли разбивать список на страницы (например, по 20 домов), применять бесконечный скролл, или отображать все 85 записей единым списком?
Влияет на: проектирование API (параметры limit/offset), UX-решение.

**OQ-2. Какие статусы дома использовать в реестре?**
ADR-0001 определяет 11 стадий строительства (земля → сервис) как прогресс-флоу. Но в реестре может быть нужен отдельный бизнес-статус дома как объекта продажи или управления: например, «Свободен», «Зарезервирован», «Продан», «В стройке», «Сдан». Эти два понятия (стадия строительства и статус продажи/распоряжения домом) — разные вещи или Владелец имеет в виду только стадию строительства?
Влияет на: модель данных (нужна ли колонка `sale_status` в таблице `houses`), фильтры реестра.

**OQ-3. Кто может добавлять/редактировать дома в реестре?**
Сейчас 85 домов зафиксированы через seeds. Может ли кто-то через интерфейс добавить новый дом (например, при расширении посёлка)? Или реестр домов считается неизменным справочником в рамках MVP, и добавление идёт только через разработчика?
Влияет на: наличие формы «Добавить дом» в UI, проверки прав на POST /houses, объём работ фазы 3.

**OQ-4. Какие поля дома самые важные для быстрого взгляда в списке?**
Кроме номера участка, типа и стадии — что Владелец хочет видеть в строке списка без входа в карточку? Например: площадь дома, цена продажи, имя покупателя/контрагента, процент готовности, флаг «есть просроченные задачи»?
Влияет на: состав колонок таблицы, производительность (SELECT полей).

**OQ-5. Нужна ли карта посёлка с визуальным генпланом?**
Роадмап явно исключает «визуальный генплан с кликабельной картой» из MVP1. Подтверждает ли Владелец, что в рамках данной User Story реестр — это только табличный/карточный список без карты? Или нужна хотя бы статичная схема расположения участков?
Влияет на: объём работ и технологический выбор (SVG/карта vs таблица).

## Допущения (Assumptions)

Следующие допущения приняты при написании истории. Если они неверны — US-001 требует пересмотра.

- **A1.** Все 85 домов уже присутствуют в БД (засеяны через `seeds.py`). Ручное добавление домов через UI в MVP не является обязательным.
- **A2.** Удалённых (`deleted_at IS NOT NULL`) домов в реестре не отображается без специального фильтра.
- **A3.** Стадии строительства берутся из таблицы `stages` (11 стадий по ADR-0001); текущая стадия дома — это поле `current_stage_id` в таблице `houses`.
- **A4.** Опции дома (навесы, интерьерные пакеты, спа и т.д.) отображаются в карточке, но не в строке списка.
- **A5.** История смены стадий (`house_stage_history`) отображается в карточке в виде хронологического лога.
- **A6.** Фильтрация реализуется на стороне бэкенда (query params), не на клиенте через JS-фильтр по загруженному массиву.

## Критерии готовности (Definition of Done)

- [ ] API-эндпоинт `GET /houses` возвращает список домов с поддержкой фильтров `?type=` и `?stage=`
- [ ] API-эндпоинт `GET /houses/{id}` возвращает полную карточку дома с опциями и историей стадий
- [ ] Pydantic-схемы для ответов задокументированы в Swagger
- [ ] Права доступа проверены: owner и read_only видят данные; неаутентифицированный запрос возвращает 401
- [ ] Фронтенд-страница «Дома» отображает список с фильтрами по типу и стадии
- [ ] Фронтенд-карточка дома отображает все поля из Acceptance Criteria сценария 3
- [ ] Пустое состояние (нет домов) обработано на UI
- [ ] Время ответа `GET /houses` не превышает 500 мс при 85 записях (без N+1 запросов)
- [ ] Код покрыт минимум одним интеграционным тестом на список и одним на карточку
- [ ] Audit_log пишет запись при каждом изменении данных дома (не при GET)

- `houses` — основная таблица, 85 записей

## Глоссарий (термины, использованные в этой US)

| Термин | Определение |
|--------|-------------|
| Реестр домов | Единый список всех домов посёлка с актуальными атрибутами — основной навигационный экран системы |
| Тип дома | Один из 4 базовых типовых проектов (A, B, C, D), определяющих площадь и планировку |
| Стадия строительства | Текущий этап возведения дома из 11 предопределённых (земля → сервис) |
| Опция | Дополнительный элемент комплектации дома (навес, спа, интерьерный пакет и т.п.) |
| Карточка дома | Детальный экран одного дома со всеми атрибутами, историей и ссылками на смежные разделы |
| Пустое состояние | UI-паттерн: что показывает экран, когда данных ещё нет (вместо ошибки или пустой страницы) |
| Seeds | Скрипт первичного наполнения базы данных тестовыми/начальными данными |
