# FE-W1-4 — Матрица прав доступа (Admin UI)

## Контекст

Экран «Матрица прав» (`/admin/permissions`) — четвёртый экран администраторской панели, следующий за Ролями. Является точкой управления правами доступа: какая роль может выполнять какое действие над каким ресурсом. Представляет права в виде матрицы «ресурс × действие» с строками по ролям.

Экран вызывается двумя путями:
- прямая навигация через меню `/admin/permissions`;
- переход из карточки роли (Экран 3) по кнопке «Открыть матрицу прав для этой роли» с query-параметром `?role=<code>` — роль при этом выделяется в матрице.

Системные роли (`is_system=true`) отображаются в матрице в режиме read-only: права видны, но не могут быть изменены или отозваны через интерфейс.

---

## Глоссарий (к данным US)

| Термин | Определение |
|--------|-------------|
| Permission (право) | Атомарная единица доступа: действие (`action`) над ресурсом (`resource`). Пример: `houses:write`, `payments:approve`. Хранится в справочнике `/api/v1/permissions`. |
| Матрица прав | Таблица «роль × permission», в ячейках которой отображается наличие или отсутствие права для данной роли. |
| Ресурс (resource) | Сущность системы, доступом к которой управляют через матрицу. Примеры: `houses`, `payments`, `contractors`, `users`. |
| Действие (action) | Тип операции над ресурсом. Стандартный набор: `read`, `write`, `delete`, `approve`. У ресурсов могут быть специфичные действия. |
| Системный permission | Право, закреплённое за системной ролью (`is_system=true`) на уровне seed-скрипта. Не может быть отозвано через UI. |
| Bulk-select (групповой выбор) | Выбор нескольких permissions одновременно по строке (ресурс) или столбцу (действие) в матрице для последующего назначения или отзыва. |
| Audit log (журнал аудита) | Автоматически ведущийся журнал всех изменений в системе. Каждое изменение прав записывается в него в рамках той же транзакции. |
| Диф-история (history diff) | Представление истории изменений прав роли в виде «было → стало» по каждой операции: дата, автор, какие права добавлены/убраны. |

## US-01 — Admin просматривает матрицу прав для всех ролей

**Story:**
Как Владелец холдинга, я хочу видеть матрицу прав в формате «ресурс × действие» с отметками по каждой роли, чтобы одним взглядом понимать, у кого какой доступ настроен, и выявлять неожиданные разрывы или избыточные права.

### Acceptance Criteria

**Сценарий 1.1 — Стандартная загрузка матрицы**

```
Given  администратор аутентифицирован и открывает /admin/permissions
When   браузер выполняет GET /api/v1/permissions (справочник) и GET /api/v1/role_permissions (матрица)
Then   отображается таблица, где строки — пары «ресурс × действие», столбцы — все роли системы
And    в каждой ячейке отображается Toggle/Checkbox: заполнен — право есть, пуст — права нет
And    заголовок столбца для системных ролей содержит значок-замок (lock icon)
And    счётчик над таблицей показывает «N ресурсов, M действий, K ролей»
```

**Сценарий 1.2 — Загрузка (shimmer-состояние)**

```
Given  браузер отправил запросы к API и ожидает ответа
When   хотя бы один запрос ещё не завершён
Then   в таблице отображаются shimmer-строки вместо данных
And    элементы управления (фильтры, кнопки) остаются доступны для взаимодействия
```

**Сценарий 1.3 — Пустая матрица (справочник permissions пуст)**

```
Given  GET /api/v1/permissions вернул пустой массив
When   страница отрендерена
Then   таблица не отображается
And    отображается empty-state: «Справочник прав пуст. Проверьте, что seed-скрипт выполнен.»
And    кнопка экспорта в CSV неактивна
```

**Сценарий 1.4 — Ошибка загрузки**

```
Given  любой из запросов к API вернул 5xx или сеть недоступна
When   запрос завершился с ошибкой
Then   отображается Banner «Не удалось загрузить матрицу прав. [Повторить]»
And    клик по «Повторить» повторяет оба запроса
And    частичные данные (только один из запросов успешен) не отображаются
```

**Сценарий 1.5 — Переход с предфильтром из Экрана 3 (FE-W1-3)**

```
Given  администратор перешёл с /admin/roles/:id по ссылке /admin/permissions?role=<code>
When   страница загрузилась
Then   матрица отображает все роли
And    строка с колонкой роли <code> визуально выделена (highlight цветом или рамкой)
And    при отсутствии роли <code> в матрице — выделение не применяется, без ошибки
```

### Edge Cases

- При большом количестве ресурсов (20+) и ролей (10+) таблица горизонтально прокручивается; заголовки строк (ресурс × действие) фиксированы при горизонтальном скролле.
- Группировка строк по ресурсу: все действия одного ресурса идут блоком с визуальным разделителем и возможностью свернуть/развернуть группу.

### API

- `GET /api/v1/permissions` — справочник всех permissions
- `GET /api/v1/role_permissions` — полная матрица: все роли × все permissions

## US-02 — Admin фильтрует матрицу по роли

**Story:**
Как Владелец холдинга, я хочу отфильтровать матрицу по конкретной роли и видеть только её права, чтобы быстро проверять или редактировать доступ одной роли, не отвлекаясь на остальные столбцы.

### Acceptance Criteria

**Сценарий 2.1 — Выбор роли из фильтра**

```
Given  матрица загружена
When   администратор выбирает роль из Select-фильтра «Показать роль»
Then   в таблице остаётся только один столбец — выбранная роль
And    URL обновляется до /admin/permissions?role=<code> без перезагрузки страницы
And    кнопка «Сбросить фильтр» становится активной
```

**Сценарий 2.2 — Сброс фильтра**

```
Given  в фильтре выбрана роль, матрица отображает один столбец
When   администратор нажимает «Сбросить фильтр» или выбирает пустое значение в Select
Then   матрица возвращается к полному виду (все роли)
And    URL обновляется до /admin/permissions (без query-параметра)
```

**Сценарий 2.3 — Фильтр через query-параметр при загрузке**

```
Given  администратор открывает /admin/permissions?role=accountant напрямую
When   страница загрузилась
Then   Select-фильтр показывает «accountant»
And    матрица отображает только столбец роли «accountant»
And    если роль «accountant» не найдена в данных API — фильтр сбрасывается, матрица показывает все роли, без ошибки
```

**Сценарий 2.4 — Дополнительный фильтр по ресурсу**

```
Given  матрица загружена (с фильтром роли или без)
When   администратор выбирает значение из Select-фильтра «Ресурс»
Then   строки таблицы фильтруются: отображаются только пары с выбранным ресурсом
And    оба фильтра (роль и ресурс) работают одновременно
```

### Edge Cases

- Фильтр не инициирует новый API-запрос: фильтрация выполняется на фронтенде по уже загруженным данным (матрица загружается один раз полностью).
- При фильтре по роли кнопка bulk-select по столбцу (US-03) становится ещё удобнее — это сценарий, который нужно поддержать визуально.

### API

- Новых API-запросов при фильтрации нет (работа на клиенте с данными от первичной загрузки).

## US-03 — Admin назначает permission на роль (bulk-select)

**Story:**
Как Владелец холдинга, я хочу назначать rights роли оптом — выбрав все действия по ресурсу или все ресурсы по действию — чтобы настраивать доступ быстро, не кликая по каждой ячейке отдельно.

### Acceptance Criteria

**Сценарий 3.1 — Bulk-select всех действий по ресурсу**

```
Given  матрица загружена, выбрана конкретная роль (фильтр или вся матрица)
When   администратор кликает чекбокс-«выбрать строку» в заголовке строки ресурса (например, «houses»)
Then   все ячейки этой строки для данной роли переходят в состояние «выбрано»
And    в плавающей панели действий отображается: «Выбрано N permissions»
And    кнопка «Назначить выбранное» активна
```

**Сценарий 3.2 — Bulk-select всех ресурсов по действию**

```
Given  матрица загружена
When   администратор кликает чекбокс-«выбрать столбец» в заголовке столбца действия (например, «read»)
Then   все ячейки этого столбца для данной роли переходят в состояние «выбрано»
And    в плавающей панели действий отображается: «Выбрано N permissions»
```

**Сценарий 3.3 — Применение bulk-назначения**

```
Given  администратор выбрал одно или несколько permissions через bulk-select или индивидуально
When   он нажимает «Назначить выбранное»
Then   браузер выполняет PATCH /api/v1/role_permissions с телом {"role_id": <id>, "add": [<permission_ids>]}
And    при статусе 200 все выбранные ячейки переходят в состояние «заполнена» (право есть)
And    Toast «Права назначены» отображается 3–5 секунд
And    плавающая панель выбора скрывается
And    выделение ячеек снимается
```

**Сценарий 3.4 — Попытка назначить permission системной роли**

```
Given  администратор открыл матрицу
When   он пытается кликнуть на ячейку столбца системной роли (is_system=true)
Then   ячейка не реагирует на клик (disabled)
And    при наведении отображается Tooltip: «Права системных ролей нельзя изменить»
And    чекбокс-«выбрать столбец» для системной роли недоступен (disabled)
```

**Сценарий 3.5 — Ошибка при назначении**

```
Given  администратор нажал «Назначить выбранное»
And    PATCH /api/v1/role_permissions вернул 4xx или 5xx
Then   отображается Toast с ошибкой «Не удалось назначить права. Попробуйте ещё раз.»
And    состояние матрицы не изменяется (оптимистичный update откатывается)
And    выбор ячеек сохраняется, чтобы администратор мог повторить
```

### Edge Cases

- Если выбранные permissions уже назначены роли — PATCH идёт с этими ids в `add`, но сервер их игнорирует (идемпотентность). Ячейки остаются заполненными. Ошибка не возникает.
- При одновременном выборе по строке и столбцу пересекающаяся ячейка выбирается однократно.

### API

- `PATCH /api/v1/role_permissions` — частичное обновление матрицы (тело: role_id, add: [permission_ids])

## US-04 — Admin отзывает permission с роли (с подтверждением)

**Story:**
Как Владелец холдинга, я хочу отзывать конкретное право у роли через одиночное действие с обязательным шагом подтверждения, чтобы не допустить случайного лишения доступа сотрудников.

### Acceptance Criteria

**Сценарий 4.1 — Отзыв единичного permission**

```
Given  матрица загружена
And    ячейка на пересечении permission и пользовательской роли заполнена (право есть)
When   администратор кликает на неё (или кнопку «Отозвать» при hover)
Then   открывается Dialog подтверждения: «Отозвать право «<permission>» у роли «<role>»? Пользователи с этой ролью потеряют доступ к данному действию.»
And    кнопка «Подтвердить» выделена деструктивным цветом (красный)
```

**Сценарий 4.2 — Подтверждение отзыва**

```
Given  Dialog подтверждения открыт
When   администратор нажимает «Подтвердить»
Then   браузер выполняет DELETE /api/v1/role_permissions с телом {"role_id": <id>, "permission_id": <id>}
And    при статусе 200 или 204 Dialog закрывается
And    ячейка переходит в состояние «пуста» (право отсутствует)
And    Toast «Право отозвано» отображается 3–5 секунд
```

**Сценарий 4.3 — Отзыв у роли с активными пользователями**

```
Given  Dialog подтверждения открыт
And    у роли есть хотя бы один пользователь с is_active=true
When   Dialog отображается
Then   под основным текстом отображается Warning: «У этой роли есть активные пользователи. Они немедленно потеряют данное право.»
And    Dialog не блокирует отзыв — кнопка «Подтвердить» доступна
```

**Сценарий 4.4 — Попытка отзыва у системной роли (прямой запрос)**

```
Given  клиентская защита обойдена (прямой API-запрос)
When   DELETE /api/v1/role_permissions отправлен для permission системной роли
Then   API возвращает 403 Forbidden
And    тело ответа: {"error": {"code": "SYSTEM_ROLE_IMMUTABLE", "message": "...", "details": {}}}
```

**Сценарий 4.5 — Отмена отзыва**

```
Given  Dialog подтверждения открыт
When   администратор нажимает «Отмена» или закрывает Dialog крестиком
Then   Dialog закрывается без отправки DELETE-запроса
And    ячейка остаётся заполненной (право не изменилось)
```

### Edge Cases

- Одиночный отзыв (данная US) не поддерживает bulk-отзыв — для массового снятия прав требуется отдельное UX-решение (см. OQ-2).
- Количество активных пользователей роли: число отображается в тексте Warning, если API возвращает это поле (уточнить в OQ-3).

### API

- `DELETE /api/v1/role_permissions` — отзыв конкретного permission у роли

## US-05 — Admin видит системные permissions (readonly)

**Story:**
Как Владелец холдинга, я хочу видеть права системных ролей в матрице в режиме только для чтения, чтобы понимать, какие базовые права зашиты в систему, но случайно не изменить их через интерфейс.

### Acceptance Criteria

**Сценарий 5.1 — Визуальная индикация системных ролей**

```
Given  матрица загружена и содержит системные роли (is_system=true)
When   страница отрендерена
Then   заголовки столбцов системных ролей содержат значок-замок
And    ячейки системных ролей отображают права (заполнены/пусты) как read-only
And    hover на ячейку системной роли не вызывает кнопку «Отозвать» и не меняет курсор
And    чекбокс bulk-select для заголовка столбца системной роли disabled
```

**Сценарий 5.2 — Tooltip при попытке взаимодействия с системным permission**

```
Given  администратор наводит курсор на ячейку столбца системной роли
When   hover-состояние активно
Then   отображается Tooltip: «Права системной роли «<name>» зафиксированы. Редактирование через UI недоступно.»
```

**Сценарий 5.3 — Фильтр по системной роли (view-only)**

```
Given  администратор выбирает системную роль в фильтре «Показать роль»
When   матрица отфильтрована
Then   отображается один столбец с правами системной роли
And    все ячейки disabled
And    над таблицей отображается информационный Banner: «Отображаются права системной роли. Изменение недоступно.»
And    кнопки bulk-select скрыты или disabled
```

**Сценарий 5.4 — Защита на уровне API**

```
Given  клиентская защита обойдена
When   PATCH или DELETE /api/v1/role_permissions отправлен для системной роли
Then   API возвращает 403 Forbidden с кодом SYSTEM_ROLE_IMMUTABLE
```

### Edge Cases

- Список системных ролей определяется полем `is_system=true` из API — фронтенд не хардкодит их список (аналогично принципу A1 из FE-W1-3).
- Экспорт CSV (US-06) включает системные права в файл — они read-only только в интерфейсе.

### API

- `GET /api/v1/role_permissions` — содержит флаг `is_system` для каждой роли
- `GET /api/v1/permissions` — содержит флаг `is_system` для каждого permission (если применимо, уточнить в OQ-4)

## US-06 — Admin экспортирует матрицу прав в CSV

**Story:**
Как Владелец холдинга, я хочу выгружать текущую матрицу прав в CSV-файл, чтобы передавать её аудитору или юристу, сохранять снимок на дату и сравнивать с предыдущими версиями вне системы.

### Acceptance Criteria

**Сценарий 6.1 — Экспорт полной матрицы**

```
Given  матрица загружена (данные в браузере есть)
When   администратор нажимает «Экспорт CSV»
Then   браузер инициирует скачивание файла
And    имя файла: permissions-matrix-YYYY-MM-DD.csv
And    файл содержит строки: «Ресурс», «Действие», а затем по одному столбцу на каждую роль
And    значения в ячейках: «Да» / «Нет» (или «1» / «0» — уточнить в OQ-5)
And    первая строка — заголовки: Resource, Action, <role_name_1>, <role_name_2>, ...
And    кодировка файла: UTF-8 с BOM (для корректного открытия в Excel)
```

**Сценарий 6.2 — Экспорт отфильтрованной матрицы**

```
Given  применён фильтр по роли или ресурсу
When   администратор нажимает «Экспорт CSV»
Then   в CSV выгружается только то, что видно в таблице (с учётом фильтров)
And    имя файла: permissions-matrix-<role_code>-YYYY-MM-DD.csv (если выбрана роль)
```

**Сценарий 6.3 — Экспорт при пустой матрице**

```
Given  матрица пуста (справочник permissions пуст)
When   администратор видит empty-state
Then   кнопка «Экспорт CSV» неактивна (disabled)
And    Tooltip на кнопке: «Нет данных для экспорта»
```

### Edge Cases

- Экспорт выполняется на клиенте из данных, уже загруженных в браузер (без дополнительного API-запроса). Это даёт актуальность данных на момент последней загрузки страницы, но не гарантирует синхронность с сервером после действий другого администратора. Допущение принято.
- Системные права включаются в экспорт и помечаются столбцом «Системная» (значение: «Да» / «Нет») или иным способом — уточнить в OQ-5.

### API

- Новых API-запросов нет (клиентская генерация CSV).

## US-07 — Admin просматривает историю изменений прав роли (audit log)

**Story:**
Как Владелец холдинга, я хочу видеть хронологию изменений прав конкретной роли с указанием «что изменилось, кто изменил, когда», чтобы расследовать инциденты с доступом и подтверждать аудиторам соответствие изменений регламентам.

### Acceptance Criteria

**Сценарий 7.1 — Открытие панели истории для роли**

```
Given  матрица загружена
And    применён фильтр по конкретной роли (или администратор выбрал роль из матрицы)
When   он нажимает «История изменений» (кнопка в заголовке столбца или в фильтр-панели)
Then   открывается Drawer (боковая панель) с заголовком «История прав роли «<name>»»
And    браузер выполняет GET /api/v1/roles/{id}/permissions с параметром ?include=history
And    в Drawer отображается хронологический список записей аудита
```

**Сценарий 7.2 — Формат записи диф-истории**

```
Given  Drawer истории открыт и содержит записи
When   запись отрендерена
Then   каждая запись содержит: дату и время (UTC+3), имя автора, действие («Назначено» / «Отозвано»), название permission
And    записи отсортированы по убыванию даты (новые — сверху)
And    визуально: «Назначено» отображается зелёным индикатором, «Отозвано» — красным
```

**Сценарий 7.3 — Пустая история (новая роль)**

```
Given  Drawer открыт для роли, у которой не было изменений прав
When   GET-запрос вернул пустой массив истории
Then   в Drawer отображается: «Изменений прав ещё не было. История формируется с момента первого назначения.»
```

**Сценарий 7.4 — Пагинация истории**

```
Given  у роли более 50 записей истории
When   Drawer прокручен до конца
Then   подгружаются следующие 50 записей (infinite scroll или кнопка «Показать ещё»)
And    индикатор загрузки отображается в нижней части Drawer во время подгрузки
```

**Сценарий 7.5 — Закрытие Drawer**

```
Given  Drawer истории открыт
When   администратор нажимает крестик или кликает вне Drawer
Then   Drawer закрывается
And    матрица и её состояние (фильтры, выбор) остаются неизменными
```

### Edge Cases

- Drawer истории доступен для системных ролей в режиме read-only — история есть, но кнопки действий не отображаются.
- История ограничена рамками audit_log, который ведётся только с момента внедрения системы. Более ранние изменения (до запуска) не отображаются.

### API

- `GET /api/v1/roles/{id}/permissions` — права роли (основная загрузка)
- `GET /api/v1/roles/{id}/permissions?include=history` или отдельный эндпоинт — уточнить структуру в OQ-1

---

## Критерии готовности (Definition of Done)

- [ ] `GET /api/v1/permissions` возвращает справочник прав с полями: id, resource, action, description, is_system
- [ ] `GET /api/v1/role_permissions` возвращает полную матрицу: все роли × все permissions; содержит `is_system` для каждой роли
- [ ] `GET /api/v1/roles/{id}/permissions` возвращает список permissions конкретной роли
- [ ] `PATCH /api/v1/role_permissions` принимает role_id + массив permission_ids для назначения; возвращает 403 для системных ролей
- [ ] `DELETE /api/v1/role_permissions` принимает role_id + permission_id; возвращает 403 для системных ролей, 200/204 при успехе
- [ ] Все write-эндпоинты записывают в audit_log в рамках той же транзакции (ADR 0007)
- [ ] Ошибки API соответствуют формату ADR 0005: `{"error": {"code", "message", "details"}}`
- [ ] Страница /admin/permissions отображает таблицу с shimmer, empty-state, error Banner
- [ ] Горизонтальная прокрутка с фиксированными заголовками строк реализована
- [ ] Фильтр по роли и ресурсу работает на клиенте без повторного API-запроса
- [ ] Ячейки системных ролей disabled (нет hover-кнопки «Отозвать», нет bulk-select по столбцу)
- [ ] Dialog подтверждения при единичном отзыве отображает Warning для роли с активными пользователями
- [ ] Экспорт в CSV работает на клиенте, файл корректно открывается в Excel (UTF-8 с BOM)
- [ ] Drawer истории изменений открывается, показывает диф-записи в хронологии
- [ ] Неаутентифицированный запрос к любому `/api/v1/role_permissions*` или `/api/v1/permissions*` возвращает 401

---

## Допущения (Assumptions)

- **A1.** Статус системной роли определяется полем `is_system=true` в ответе API; фронтенд не хардкодит список системных ролей.
- **A2.** Матрица загружается полностью при первом открытии страницы; фильтрация выполняется на клиенте без повторных запросов.
- **A3.** Отзыв permission у роли с активными пользователями — предупреждение, не блокировка. Владелец принимает решение осознанно.
- **A4.** Экспорт CSV формируется на клиенте из данных, уже загруженных в браузер. Данные актуальны на момент последней загрузки.
- **A5.** История изменений permissions ведётся через стандартный audit_log (ADR 0007) и доступна через `/api/v1/roles/{id}/permissions` с include-параметром либо отдельный эндпоинт.
- **A6.** Права системных ролей не редактируются через UI — это защита на двух уровнях: отключённые элементы на фронтенде и 403 на бэкенде.
- **A7.** Количество активных пользователей у роли доступно в ответе API при вызове информации о роли или рассчитывается отдельным запросом при открытии Dialog.

## Открытые вопросы (Open Questions)

**OQ-1. Структура API для истории изменений permissions роли.**
Используется ли параметр `?include=history` для `GET /api/v1/roles/{id}/permissions`, или это отдельный эндпоинт (например, `/api/v1/roles/{id}/permissions/history`)? От этого зависит поведение Drawer и формат данных. Вопрос к Архитектору или Бэкенд-директору.

**OQ-2. Нужен ли bulk-отзыв прав?**
Текущая US-04 описывает только единичный отзыв с подтверждением. Нужен ли администратору способ bulk-отзыва (снять сразу несколько прав одним действием, аналогично bulk-назначению в US-03)? Если да — нужен отдельный сценарий и решение по UX: один общий Dialog подтверждения для всего набора или отдельный для каждого? Вопрос к Владельцу.

**OQ-3. Отображать ли количество активных пользователей у роли в Dialog подтверждения?**
Сценарий 4.3 требует Warning при наличии активных пользователей. Должна ли система показывать конкретное число («5 активных пользователей потеряют право»)? Если да — нужен либо дополнительный API-поле в ответе при вызове роли, либо отдельный запрос перед открытием Dialog. Вопрос к Владельцу.

**OQ-4. Существует ли флаг `is_system` у самих permissions (не только у ролей)?**
Текущая модель: системность определяется через флаг роли. Но может существовать ситуация, когда пользовательской роли нельзя назначить определённые «системные» permissions (например, `system:admin`). Нужно ли это ограничение? Вопрос к Архитектору.

**OQ-5. Формат значений в CSV: «Да»/«Нет» или «1»/«0», или «✓»/«–»?**
Влияет на читаемость в Excel и обработку внешними инструментами аудита. Вопрос к Владельцу или аудитору (если есть требования к формату).
