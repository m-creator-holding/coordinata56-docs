## 1. Контекст

Frontend-director впервые активирован в боевом режиме. До этого статус direction был `active-supervising` без реальных задач: бэкенд-фазы 0–3 шли без изменений в UI. В `frontend/` уже лежит стартовый скелет: React 18 + TS + Vite + Tailwind + shadcn/ui + React Router + TanStack Query + Zustand + react-hook-form (судя по `package.json`). Пять рабочих страниц (Dashboard / Houses / Finance / Schedule / NotFound). Это важная отправная точка: базовый выбор стека (ADR 0002) подтверждается живым кодом.

Задача фронтенда в M-OS-1 разбита на два трека:

1. **Admin-UI конструктор** (часть ADR-0018) — три под-фазы (M-OS-1.1 / 1.2 / 1.3), 12 функциональных разделов.
2. **PWA прораба** (DoD п. 7) — мобильное веб-приложение для стройплощадки «Координата 56» в M-OS-1.3.

Оба трека живут **в одном репозитории** и **разделяют один код-стек**. Никаких отдельных фронт-проектов. Обоснование — §4.

## 2. Ответы на 5 блокирующих вопросов из §5 ADR-плана v3

### 2.1 Canvas-редактор для BPM-конструктора

**Рекомендация**: `bpmn-js` (bpmn.io, Camunda).

Обоснование:
- Де-факто стандарт для BPMN 2.0 в Web — более 10 лет в production, используется в Camunda, Flowable, Zeebe.
- **Экспорт и импорт BPMN-XML «из коробки»** — это важный контракт с backend'ом: процесс в `configuration_entities` child-table `bpm_process_definitions` хранится как BPMN-XML, а не как наш внутренний JSON. Значит, backend не изобретает свой формат, backend парсит стандарт.
- Open-source (MIT), не требует сервера, работает чисто на клиенте.
- Расширяемый (palette, properties panel), можно ограничить набор элементов, чтобы админ не наделал хаоса (только те задачи, таймеры и шлюзы, которые умеет исполнять наш движок в M-OS-1).
- Большая база примеров, LLM-агенты генерируют интеграцию с `bpmn-js` надёжнее, чем с альтернативами (важно для нашей ИИ-команды).

Почему не альтернативы:
- `reactflow` (он же xyflow) — универсальный canvas, но BPMN-семантика пишется руками. 2–3 дополнительные недели без необходимости.
- `jointjs` + `bpmn2` plugin — платная версия для production-BPMN, бесплатная урезана.
- Кастом на SVG — over-engineering, запрещено engineering-принципами Владельца.

**Контракт с backend-director**: процесс хранится как BPMN-XML (`text` в child-table). В admin-UI используется `bpmn-js Modeler`. Migration rules для BPM (ADR-0017 Б2) валидируются на стороне backend при publish — frontend только показывает список «затронутых запущенных экземпляров» через preview-API.

**Появление в фазе**: M-OS-1.2 (по разбиению ADR-0018). В M-OS-1.1 BPM-раздела в admin-UI ещё нет.

### 2.2 Form-builder

**Рекомендация**: собственный лёгкий form-builder поверх `react-hook-form` + `zod` (уже в стеке) + JSON-дескриптор полей.

Обоснование:
- `react-hook-form` уже в стеке. `zod` — стандарт валидации в React-мире; backend'у удобно получить ту же `zod`-схему сериализованной в JSON-Schema (`zod-to-json-schema`), которая затем валидируется на backend'е через `pydantic` (есть адаптер `json-schema` → Pydantic).
- Form-builder = драг-дроп список полей слева, preview посередине, properties panel справа. Внутри всё это — обычный React, без тяжёлой библиотеки.
- Конфигурация формы хранится в child-table `form_definitions` как JSON-массив дескрипторов (`{type, name, label, required, options, validation}`). Runtime-рендер формы — наш же компонент `DynamicForm` на `react-hook-form`.
- Полей M-OS-1 хватает штук 10 (text, textarea, number, money, date, select, multiselect, checkbox, file, user-picker). Можно собрать за 2–3 недели.

Почему не `formio` (он же form.io):
- Это полноценная платформа с own backend, own API, own admin. Интегрировать в наш backend — жизнь на противоположном конце от «configuration-as-data в нашей БД».
- Лицензия: OSS-версия у `formio.js` либеральная, но серверная часть коммерческая. Мы бы подключали только клиент — тогда зачем?
- Over-engineering на старте.

Почему не `react-jsonschema-form` (rjsf):
- Хорош как runtime-рендер формы из JSON Schema. Но builder (визуальный редактор) у rjsf слабый — собственный нужно всё равно писать.
- Если бы нам нужен был только рендер без builder — был бы отличный вариант. У нас задача именно builder.

**Контракт с backend-director**: форма хранится как JSON-массив дескрипторов (явная схема согласуется backend + frontend совместно в ADR-0018 финализации). Валидация: `zod` на фронте и Pydantic-модель из JSON Schema на бэке. Одна точка правды — JSON-дескриптор в БД.

**Появление в фазе**: M-OS-1.3 (по разбиению ADR-0018).

### 2.3 Архитектура PWA vs SPA

**Рекомендация**: **единый SPA на React + Vite**, из которого PWA прораба получается добавлением `vite-plugin-pwa` + манифест + service worker, но **без превращения в отдельный репозиторий** и **без SSR**.

Обоснование:
- Вся M-OS-1 — внутренняя для сотрудников холдинга, под авторизацией, SEO не нужен. SSR (Next.js) ничего не даёт — только добавляет сложности.
- Единая кодовая база экономит 30–40% времени: shadcn/ui-компоненты, API-клиент, auth, query-кеш, i18n — всё общее.
- PWA = веб-манифест + service worker. Это не отдельный фреймворк, а конфигурация плагина.
- Разные entry-points достигаются на уровне роутинга: `/admin/*` — desktop-лейаут для офиса, `/field/*` — mobile-first-лейаут для прораба. Lazy-chunked через `React.lazy`, прораб не тянет код admin-UI и наоборот.
- Push-уведомления через браузер **не нужны** — уведомления идут через Telegram (решение Владельца). PWA на iOS Safari всё равно не умеет web push в полном объёме, так что это ещё и снимает риск.

Почему не отдельное Next.js-приложение:
- SSR не нужен. App Router добавляет кривую обучения. LLM-агенты генерируют Next.js App Router с ошибками чаще, чем чистый Vite+React (engineering principle: сначала простота).
- Два приложения = два набора deploy-конфигов, два бандла, две auth-сессии. Ненужная сложность.

Почему не нативное (React Native / Flutter):
- Vision §12 прямо запрещает нативные мобильные приложения на MVP: «Начинаем с PWA, нативные — позже, если потребуется».

**Ключевые параметры PWA**:
- `display: standalone` в manifest — прораб «устанавливает» на домашний экран, выглядит как приложение.
- `theme_color`, `background_color`, иконки 192/512.
- Service Worker через `vite-plugin-pwa` в режиме `autoUpdate`.
- Кеш-стратегии: см. §3.4.

### 2.4 Оценка трудоёмкости Admin UI

Admin-UI разбит ADR-0018 на три под-фазы. Оценки **чисто фронтенда** (backend-часть — по оценке backend-director отдельно):

| Под-фаза | Разделы admin-UI | Оценка фронтенда |
|---|---|---|
| **M-OS-1.1** | Юрлица, Пользователи и роли, Матрица прав, Company settings (7 полей), Integration Registry, System Config | **3–4 недели** |
| **M-OS-1.2** | BPM Constructor (bpmn-js canvas + properties + migration preview), Notification Constructor, Document Template Constructor | **3 недели** |
| **M-OS-1.3** | Form Builder (drag-drop), Report Builder (таблица + фильтры + агрегация), Business Rule Engine (условия if/then в UI) | **3 недели** |

**Итого чисто admin-UI: 9–10 фронтенд-недель.**

Параллелизм с backend:
- 1.1: **backend идёт на 1–1.5 недели впереди фронтенда** (нужно время на схему БД + миграции + CRUD API). Фронт стартует, когда API-контракт зафиксирован и mock-сервер на основе OpenAPI работает.
- 1.2 и 1.3: благодаря жёсткой фиксации ADR-0017 (migration rules, 9 категорий) — backend и frontend могут идти в параллели после первого спринта-согласования API.

### 2.5 Порядок параллелизации backend/frontend

| Под-фаза | Что должно быть готово у backend до старта frontend | Что фронт может делать параллельно |
|---|---|---|
| 1.1 | Alembic-миграции `configuration_entities` + 6 child-tables; CRUD-эндпоинты `Companies`, `Users`, `Roles`, `Permissions`, `CompanySettings`, `IntegrationCatalog`; OpenAPI-спека стабильна | Начиная со 2-й недели backend'а: frontend-каркас admin-UI, роутинг, Auth-сессия, layout; skeleton-страницы на mock-данных |
| 1.2 | Схема `bpm_process_definitions` + endpoint publish с валидацией migration rules; preview-API «затронутые экземпляры» | Интеграция bpmn-js, UI notification/document templates (можно начать в конце 1.1 против mock-endpoint) |
| 1.3 | Schema `form_definitions` / `report_definitions` / `business_rule_definitions`; runtime-эндпоинты рендера | Параллельно PWA прораба (другой трек) |

**Главная развилка**: frontend **не стартует** кодирование новой под-фазы admin-UI, пока backend не дал хотя бы **zero-version OpenAPI** на соответствующие ресурсы. Это прямо в Definition of Done каждой под-фазы.

**Критическая зависимость от backend-director**: две части не решаются без двустороннего договора и оформляются как требования к ADR-0018:
1. Формат хранения BPMN — BPMN-XML-текст, не собственный JSON.
2. Формат хранения формы — JSON-массив дескрипторов (согласованная схема).

Координатор организует согласование (не фронт-директор → бэк-директор напрямую — запрещено msg 665).

## 3. Декомпозиция PWA прораба (M-OS-1.3)

### 3.1 Назначение и бизнес-контекст

PWA прораба — мобильное веб-приложение для сотрудников на стройке посёлка «Координата 56». Контекст работы:
- Полевая работа, руки в перчатках, песок, холод, дождь. Интерфейс должен быть «крупные кнопки, минимум тапов».
- Интернет на площадке нестабилен: 3G/4G, периодические потери сигнала.
- Уведомления и действия приходят через **Telegram-бот** (решение Владельца): «Одобрить / Отклонить / Принять» — кнопка в Telegram открывает PWA на нужном экране.
- Камера для фото-фиксации — обязательно.

### 3.2 Технологический стек

| Слой | Выбор | Обоснование |
|---|---|---|
| Фреймворк | React 18 + TypeScript + Vite | ADR 0002, единая база с admin-UI, не удваиваем стек |
| UI-kit | shadcn/ui + Tailwind | Адаптируется под mobile через Tailwind-брейкпоинты; кастомная mobile-layout-обёртка |
| PWA | `vite-plugin-pwa` (на Workbox под капотом) | Стандарт для Vite, авто-регистрация service worker |
| Offline-cache | Workbox strategies | См. §3.4 |
| Локальное хранилище | IndexedDB через `idb-keyval` (4KB gzip) | Для очереди черновиков отметок, локального кэша задач; **НЕ Dexie** (30KB gzip) — engineering principle «простота»|
| Камера | Нативный `<input type="file" accept="image/*" capture="environment">` | Работает на iOS Safari и Android Chrome без отдельных библиотек. Никаких тяжёлых camera-SDK |
| Сжатие фото на клиенте | `browser-image-compression` (15KB gzip) | Уменьшает 4MB→300KB перед загрузкой, экономит мобильный трафик |
| Sync | TanStack Query + ручная очередь мутаций в IndexedDB | Online → обычный запрос; offline → mutation лежит в IDB до восстановления |

**Что намеренно НЕ взято**:
- `Capacitor` / `Ionic` — нативная обёртка, нам это 3 недели лишнего времени и публикация в сторы. Противоречит решению vision §12.
- `Dexie.js` — тяжёлая IndexedDB-обёртка, избыточна. `idb-keyval` хватает.
- Redux / Redux Toolkit — не нужен, Zustand в стеке достаточно для UI-стейта, TanStack Query — для серверного.

### 3.3 Структура экранов (8 экранов для M-OS-1.3)

| № | Экран | Назначение | Offline |
|---|---|---|---|
| 1 | `/field/login` | Аутентификация прораба (логин + пароль, позже — QR из Telegram) | ❌ нужен интернет |
| 2 | `/field/tasks` | **Список задач** на сегодня и завтра: назначенные прорабу приёмки, проверки, фото-фиксация | ✅ из кэша |
| 3 | `/field/tasks/:id` | **Детали задачи**: описание, объект (дом #N, этап), чеклист, срок | ✅ из кэша |
| 4 | `/field/tasks/:id/photo` | **Фото-фиксация**: открывается камера, делаем фото, сжимаем, привязываем к задаче | ✅ фото → IndexedDB |
| 5 | `/field/tasks/:id/accept` | **Приёмка работ**: чек-лист критериев, отметка «принято/не принято», комментарий, подпись (canvas-скетч) | ✅ мутация → IndexedDB |
| 6 | `/field/approvals` | **Входящая очередь одобрений**: платежи и согласования, которые нужны от прораба (Б7 из vision) | ✅ из кэша, одобрение online |
| 7 | `/field/bpm/:instanceId/:stepId` | **Целевая точка из Telegram-кнопки**: открывается при нажатии кнопки в Telegram-уведомлении, показывает шаг BPM-процесса и доступные действия | ✅ показ из кэша, действие online |
| 8 | `/field/profile` | Профиль: смена пароля, выход, статус синхронизации, список отложенных (pending) действий | ✅ полностью локально |

Дополнительно: общий `FieldLayout` с нижней навигацией (Tasks / Approvals / Profile), sticky-хедер с именем и статусом сети, offline-индикатор.

### 3.4 Offline-стратегия

**Уровень 1 (обязательно в M-OS-1.3) — просмотр и постановка отметок в очередь:**

1. **App shell (HTML/CSS/JS бандл)** — стратегия `CacheFirst`, TTL 7 дней. Prerobic запустил один раз при наличии связи → приложение работает без сети.
2. **Список задач** — при логине подтягиваются задачи на горизонт 7 дней → сохраняются в IndexedDB (key `tasks_v1`). Фон-обновление через TanStack Query `staleTime: 5min`. Offline — читаем из IDB.
3. **Детали задачи + чек-листы** — кэшируются на первом открытии.
4. **Фото и акты приёмки** — мутации сохраняются в IDB-очередь (`pending_mutations`) с уникальным локальным `client_id`. После восстановления сети — очередь последовательно дотсылается. Сервер на стороне backend возвращает 409 при дубликате, фронт убирает мутацию из очереди.
5. **Синхронизация**: сервис-воркер слушает `online` событие → пробуждает очередь. Пользователь видит в `/field/profile` счётчик «5 действий в очереди» и кнопку «Синхронизировать сейчас».

**Уровень 2 (вне скоупа M-OS-1.3, но архитектура готова)** — full-offline редактирование сложных сущностей с CRDT. На M-OS-1 не нужно: для прораба критичны приёмка и фото, не редактирование договоров.

**Сервер-требования** (оформляются в брифе backend-director через Координатора):
- Идемпотентность мутаций по `client_id` (деталь, не нова — уже в ADR 0007 для аудита).
- Endpoint `GET /field/tasks?assignee_id=me&window_days=7` — пачкой, с `updated_at` для инкремента.
- Endpoint `POST /field/tasks/:id/acceptance` принимает `client_id` в теле.

### 3.5 Интеграция с BPM-кнопками Telegram

Сценарий: backend публикует human_task в BPM → `telegram-bot` шлёт прорабу сообщение «Требуется приёмка по задаче X» + кнопка «Открыть». Кнопка — `inline URL-button` со ссылкой вида:

```
https://<app-host>/field/bpm/<instance_id>/<step_id>?token=<short_jwt>
```

Логика фронта:
1. По переходу проверяется токен — короткоживущий JWT от `telegram-bot`, подписывается таким же секретом, что и наш API.
2. Если прораб уже залогинен и `user_id` в токене совпадает — сразу показываем экран шага.
3. Если не залогинен — редирект на `/field/login?next=<encoded-url>`, после логина восстанавливаем навигацию.
4. Действие (Одобрить/Отклонить) — POST к `/api/v1/bpm/instances/<id>/steps/<stepId>/decide`.

Никаких browser push notifications — весь нотификационный контур через Telegram (решение Владельца).

### 3.6 Оценка PWA

| Этап | Работа | Время |
|---|---|---|
| Каркас PWA | vite-plugin-pwa, manifest, иконки, сервис-воркер, FieldLayout | 0.5 нед |
| Auth flow | Login + обработка Telegram deep-links + восстановление сессии | 0.5 нед |
| Список и детали задач | Tasks list + Task details + Checklist rendering | 1 нед |
| Фото-фиксация | Camera input + compression + IDB-очередь + upload | 0.5 нед |
| Приёмка работ | Acceptance form + подпись-canvas + мутация | 0.5 нед |
| Inbox одобрений | Approvals list + детали + approve/reject | 0.5 нед |
| BPM deep-link | Экран шага + decide | 0.5 нед |
| Offline-синхронизация | Sync queue + background sync + UX статуса | 1 нед |
| Профиль + тесты + полировка | Profile + smoke E2E + fixes | 0.5 нед |

**Итого PWA: 5–5.5 недель одного фронт-разработчика.**

M-OS-1.3 по плану — 3 недели. Значит, либо:
- **вариант A** — PWA идёт параллельно BPM-батчу backend'а на протяжении всей 1.3, и заканчивается на 2 недели **после** её инженерного закрытия. DoD п. 7 закрывается первый же PR, где все 8 экранов работают end-to-end — формально PWA может быть закрыта в течение M-OS-1.3 + первые 2 недели M-OS-1.4. Это не блокирует M-OS-1.4 (pod-проверка бэкенд-only).
- **вариант B** — активировать `frontend-dev-2` (вторая позиция фронт-разработчика) параллельно первому. Первый делает admin-UI M-OS-1.3, второй — PWA. Оба завершают в 3 недели. Требует Volume-решения Координатора по росту ИИ-команды.

**Рекомендация**: вариант A без дополнительных людей. Сроки фазы уже 14–15 недель, 2 недели переноса PWA на старт 1.4 терпимы. Формально DoD п. 7 закрыт к **концу M-OS-1**, не к концу M-OS-1.3.

## 4. Стратегия фронтенда всей фазы M-OS-1

### 4.1 Единый стек

**Один проект, один бандл, разделение на уровне роутинга.**

```
frontend/
├── src/
│   ├── app/                    общий root, auth, layout selector
│   ├── admin/                  всё admin-UI (M-OS-1.1/1.2/1.3)
│   │   ├── pages/
│   │   ├── features/
│   │   │   ├── companies/
│   │   │   ├── users-roles/
│   │   │   ├── permissions/
│   │   │   ├── company-settings/
│   │   │   ├── integrations/
│   │   │   ├── bpm/            bpmn-js интеграция
│   │   │   ├── forms/          form-builder
│   │   │   ├── reports/
│   │   │   ├── notifications/
│   │   │   ├── doc-templates/
│   │   │   └── business-rules/
│   │   └── layout/
│   ├── field/                  PWA прораба (M-OS-1.3)
│   │   ├── pages/
│   │   ├── features/
│   │   │   ├── tasks/
│   │   │   ├── photo/
│   │   │   ├── acceptance/
│   │   │   ├── approvals/
│   │   │   └── bpm-deeplink/
│   │   ├── offline/            IDB-очередь, sync
│   │   └── layout/
│   ├── shared/                 общее: компоненты, хуки, api-клиент, типы
│   │   ├── ui/                 shadcn-компоненты
│   │   ├── api/                генерированные типы из OpenAPI
│   │   ├── hooks/
│   │   ├── auth/
│   │   └── query/
│   └── main.tsx
├── public/
│   └── pwa-icons/
└── vite.config.ts              + vite-plugin-pwa
```

Админ-часть и PWA-часть **lazy-loaded**: `const FieldApp = lazy(() => import('./field/FieldApp'))`. Прораб, открыв `/field/*`, не тянет ~300KB admin-кода.

### 4.2 Дизайн-система

**Оставляем shadcn/ui + Tailwind** (уже в проекте и в ADR 0002). Ничего не меняем. Обоснование:
- shadcn/ui копируется в `src/shared/ui/`, мы владеем кодом, кастомизация тривиальна.
- Tailwind-утилиты дают mobile-first responsive без CSS-файлов.
- Engineering principle «простота»: ставить Ant Design или Mantine рядом с shadcn'ом — двойная зависимость, двойная библиотека бандла, путаница. Нет.

Визуальный полиш (брендбук, типографика, цветовая палитра M-OS) — **отдельная фаза** (v1.0 того же vision § UX > визуальный дизайн). Активация `design-director` отложена до закрытия M-OS-1.1.

### 4.3 Стандарты (жёсткие)

| Область | Стандарт | Обоснование |
|---|---|---|
| Типобезопасность | TS `strict: true`, `noUncheckedIndexedAccess: true`; `any` запрещён (ESLint правило) | Catch-ошибки LLM-агентов на этапе IDE, не рантайма |
| API-клиент | Типы генерируются из backend OpenAPI через `openapi-typescript` в CI | Никаких ручных interface-описаний ответов — контракт один: OpenAPI |
| Серверный стейт | **TanStack Query** — всё, что живёт на сервере. Никаких redux/zustand для server-data | Кеш, инвалидация, optimistic-updates — из коробки |
| UI-стейт | **Zustand** для глобального (sidebar open, current company); `useState` для локального | Простота, 1KB |
| Формы | **react-hook-form + zod** | Единый формат валидации; zod → JSON Schema → Pydantic на backend |
| Роутинг | **react-router v6**, lazy chunks для `/admin` и `/field` | Уже в стеке, SSR не нужен |
| Тесты | Unit: **vitest** (каркас в 1.1); E2E: **Playwright** (стартуем в 1.3, когда PWA есть) | Стандарт для Vite-проектов |
| Линт и форматы | ESLint + Prettier уже настроены (см. package.json) | — |
| Иконки | `lucide-react` | Уже в стеке |
| i18n | Не делаем в M-OS-1. Всё по-русски. | Принцип «от цели»: многоязычие пока не задача |
| Accessibility | Базовый (semantic HTML, alt-тексты, focus-visible) через shadcn/ui-primitives | Audit (WCAG-AA) — отдельная задача M-OS-2 |

### 4.4 Порядок активации фронтенд-исполнителей

| Момент | Кого активировать | Почему |
|---|---|---|
| **Немедленно (с M-OS-1.1 стартом)** | `frontend-head` (сейчас dormant) | По новому регламенту Директор→Head→Worker frontend-head распределяет задачи frontend-dev |
| **Немедленно** | `frontend-dev` (первый воркер) | Берёт Admin-UI M-OS-1.1 (6 разделов) |
| **Конец M-OS-1.1 / начало 1.2** | Решение Координатора: оставить одного `frontend-dev` или активировать `frontend-dev-2` | Зависит от реальной скорости. Если первый закрывает 1.1 за 4 недели вместо 3 — второго не нужно. Если за 3 недели + PWA требует параллелизма — нужен |
| **M-OS-1.3 (по необходимости)** | `design-director` advisory — **только для консультаций** по mobile UX паттернам PWA | Визуальный полиш не делаем, но UX-ревью полезно |

Пока fields-frontend ставим в `active-supervising` режим, не параллельный `field-frontend-dev` — один и тот же `frontend-dev` переключается между треками через frontend-head'а.

Совет с `designer` (L4 advisory) — **ad-hoc**, через Координатора, по конкретным UX-вопросам (например, схема drag-drop формы в form-builder; mobile-лейаут приёмки).

### 4.5 Обновление регламента отдела

`docs/agents/departments/frontend.md` сейчас в версии 0.1 (скелет, dormant). Frontend-director готовит заявку в Governance-комиссию на повышение до v1.0 со следующим содержанием:
- Зафиксировать стек (как в §4.1–4.3 этого документа);
- Зафиксировать правило «один проект — два трека — общий shared» (§4.1);
- Зафиксировать стандарт OpenAPI → TS-типы (§4.3);
- Добавить раздел «Работа через frontend-head» (цепочка делегирования);
- Добавить правило «zero-version OpenAPI до старта кодирования под-фазы».

Заявка оформляется **после** одобрения Владельцем текущего плана — чтобы не гонять комиссию два раза.

## 6. Открытые вопросы к Владельцу

Вопросы, на которые не могу ответить сам и которые нужно решить до закрытия 1.1:

1. **Язык UI**: M-OS-1 делаем только русский, без i18n-инфраструктуры. Подтверждение Владельца нужно, потому что vision §8.10 упоминает white-label долгосрочно — если серьёзно рассматривается, стоит с самого начала закладывать ключи-переводы. Моя рекомендация: **без i18n в M-OS-1**, ключи добавим при белом лейбле (M-OS-5+).
2. **Брендинг и название в интерфейсе**: «M-OS», «Координата 56», что-то ещё? В заголовке, в манифесте PWA, в стрингах. Нужно определение хотя бы на уровне «рабочее название, потом поменяем».
3. **Активация `frontend-dev-2`**: готов ли Владелец активировать второго фронт-исполнителя в конце M-OS-1.1 для параллельной работы Admin-UI и PWA? Это единственный способ уложить PWA ровно в M-OS-1.3 без переноса на 1.4.
