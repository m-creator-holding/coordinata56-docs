## 1. Цель задачи

Закрыть открытый вопрос ADR 0014 §«Egress-блокировка на уровне инфраструктуры (iptables)» — обеспечить **второй эшелон защиты** против случайного живого сетевого вызова к продуктивному endpoint-у в обход слоя адаптеров.

Первый эшелон (runtime-guard `AdapterDisabledError` + `pytest-socket`) реализуется backend-отделом в PR #4 Волны 1. Эта задача — сетевой периметр, который технически блокирует исходящие HTTP-соединения к любому endpoint-у, не попавшему в явный whitelist, даже если в код попадёт hardcoded URL/ключ (повторение инцидента INC-2026-04-17-001).

Формула защиты: **default-deny + explicit allowlist + страховка от самоблокировки**.

Задача закрывает:
- открытый вопрос ADR 0014 §«Открытый вопрос: egress-блокировка через iptables» (строки 209–215 ADR);
- финальный пункт трека INC-2026-04-17-001 (инцидент остаётся PARTIALLY-RESOLVED до реализации этой задачи);
- эскалацию quality-director §3.7 и §7 пункт 2 ответа внешнему аудитору.

### Пункт 2. Дизайн политики egress-блокировки

**Что:** devops выбирает и обосновывает **технический механизм** блокировки и формат правил. В рамках ADR 0014 §«Egress-блокировка» требование сформулировано на уровне «iptables default-deny», но конкретный механизм — решение devops-level, возвращается Директору → Координатору на утверждение.

**Кандидаты, которые devops обязан рассмотреть и сравнить:**

1. **Чистый iptables + `ipset` с динамическим обновлением IP** — cron/systemd-timer резолвит whitelisted hostnames каждые N минут, добавляет в `ipset`. Плюсы: стандартный стек ядра Linux, без доп. процессов. Минусы: гонка «резолв устарел ↔ запрос пошёл по новому IP».
2. **nftables + set** — современная замена iptables, тот же принцип с set-ами. Плюсы: более выразительный язык, родной для 22.04+ Ubuntu. Минусы: меньше материалов по troubleshooting, часть legacy-тулов по-прежнему работает через iptables.
3. **Forward proxy (squid/tinyproxy) + iptables REDIRECT** — все исходящие HTTP(S) проходят через локальный прокси с whitelist по hostname, iptables блокирует прямой egress. Плюсы: hostname-based, не зависит от смены IP. Минусы: HTTPS-whitelist возможен только через SNI-inspection или MITM — MITM недопустим для Telegram/Anthropic (ломает сертификаты); SNI-based работает, но требует разбора TLS ClientHello.
4. **iptables + модуль `string` по SNI в TLS ClientHello** — грязный хак, нестабильно, не рекомендуется.
5. **ufw как фронт к iptables** — удобно для простых кейсов, но хуже для сложных allowlist с динамическими IP; рассматривать как обёртку, не как самостоятельное решение.

**Обязательные элементы дизайна (любой выбранный механизм):**

- **DEFAULT POLICY на цепочке OUTPUT: DROP.** Явно, не `REJECT` (чтобы злоумышленник не мог по ICMP понять, что порт закрыт сетевым фильтром; и чтобы ошибка для приложения выглядела как timeout, а не как connection refused — это честнее отражает «сеть недоступна»).
- **Цепочка INPUT — не трогать** в рамках этой задачи (она отвечает за входящие; сейчас там уже что-то настроено для web-dashboard 8765 и ssh 22; ломать не наш скоуп).
- **Цепочка FORWARD: DROP** (обычно и так DROP; зафиксировать).
- **Loopback (`lo`) — явный ACCEPT на OUTPUT и INPUT** (без этого сломаются localhost-соединения).
- **Established/related на OUTPUT — ACCEPT** (`-m conntrack --ctstate ESTABLISHED,RELATED`) — иначе ответы на входящие запросы не пройдут.
- **Логирование DROP** с rate-limit (`-m limit --limit 5/min`) в отдельную цепочку `OUTPUT_DROP_LOG`, вывод в syslog с префиксом `EGRESS_DROP:`. Без rate-limit лог забьёт диск при первой же ошибке в коде.
- **Persistent rules**: `iptables-persistent` (Debian/Ubuntu пакет) ИЛИ systemd service `netfilter-persistent`, ИЛИ свой systemd unit с `iptables-restore` при старте. Выбрать и обосновать в артефакте.
- **Именованные цепочки** для читаемости: `EGRESS_ALLOW_TELEGRAM`, `EGRESS_ALLOW_ANTHROPIC`, `EGRESS_ALLOW_SYSTEM` (DNS+NTP+APT). При добавлении нового адаптера — новая цепочка, не правка «универсального» списка.

**Acceptance:**
- `docs/operations/egress-design.md` — документ с:
  - сравнительной таблицей 4 кандидатов по 5 критериям (надёжность, сложность, hostname vs IP, production-готовность, обратимость);
  - **рекомендацией devops** одного варианта с обоснованием;
  - структурой цепочек iptables/nftables (в псевдокоде, не готовые скрипты);
  - обработкой динамических IP (`api.anthropic.com`, NTP-пулы): механизм обновления, периодичность, fallback при недоступности DNS;
  - обработкой первичного запуска (cold start): что происходит, если при старте сервиса правила ещё не применены;
- явно помечено: **выбор одного варианта остаётся за Координатором** — devops даёт рекомендацию, Координатор подтверждает или просит пересмотреть; девops **не применяет никаких правил** на этом пункте.

### Пункт 3. Скрипт установки + откат (код)

**Что:** два bash-скрипта, идемпотентные (повторный запуск не ломает состояние), с явным `--dry-run` режимом и sanity-check-ами.

**Файл 1: `scripts/egress-apply.sh`** — установка/обновление правил.
- Режимы: `--dry-run` (печатает команды, ничего не применяет) и `--apply` (реально запускает). По умолчанию — `--dry-run`. Опция `--apply` требует явного подтверждения флагом `--i-understand-this-may-break-networking` (длинный флаг намеренно — чтобы не ввести случайно).
- Читает whitelist из структурированного источника (см. ниже, §Пункт 5 про структуру данных); не хардкодит хосты в коде скрипта.
- Перед применением: проверяет наличие sudo-прав, существование `/sbin/iptables` (или `/usr/sbin/nft`), корректность whitelist-файла (линт YAML/JSON).
- Страховка от самоблокировки (критично): перед `iptables -P OUTPUT DROP` скрипт **планирует автоматический rollback через `at now + 5 minutes`** — если через 5 минут администратор не подтвердил «всё работает» командой `scripts/egress-confirm.sh`, назначенная задача `at` автоматически запускает `egress-rollback.sh` и снимает правила. Это защита от сценария «применил правила, случайно заблокировал сам себе ssh, остался без доступа к серверу».
- Идемпотентность: перед применением нового набора — вычищает все цепочки `EGRESS_*` (наши, не чужие!), пересоздаёт заново. Не трогает цепочки, имя которых не начинается с `EGRESS_`.
- Бэкап текущих правил: `iptables-save > /var/backups/iptables/iptables.YYYYMMDD-HHMMSS.rules` перед любым изменением. Директория создаётся, если её нет. Сохраняются 10 последних бэкапов, старее — удаляются.
- Вывод: человекочитаемый (какие правила добавлены, какие удалены), с цветным ок/fail.

**Файл 2: `scripts/egress-rollback.sh`** — откат в «всё разрешено».
- `iptables -P OUTPUT ACCEPT`, `iptables -F OUTPUT`, удаление наших цепочек `EGRESS_*`;
- восстановление из последнего бэкапа опционально (флаг `--restore-last`);
- не требует `--i-understand` флага (наоборот — упростить сценарий «всё сломалось, откатываем срочно»);
- идемпотентен (повторный запуск безопасен).

**Файл 3: `scripts/egress-confirm.sh`** — подтверждение, что правила работают.
- Снимает запланированный `at`-job автоматического rollback;
- печатает текущие счётчики DROP (чтобы сразу видеть — что-то блокируется или нет);
- сохраняет timestamp последнего подтверждения в `/var/log/egress-confirm.log`.

**Acceptance:**
- все 3 скрипта созданы, с shebang `#!/usr/bin/env bash`, `set -euo pipefail`, trap на EXIT/ERR;
- `shellcheck scripts/egress-*.sh` — чисто;
- `scripts/egress-apply.sh --dry-run` на локальном тестовом хосте (docker-контейнере, см. Пункт 4) выводит корректный план правил;
- `scripts/egress-apply.sh --apply --i-understand-this-may-break-networking` — срабатывает, но перед применением выводит предупреждение на 10 секунд с countdown и возможностью Ctrl+C;
- `scripts/egress-rollback.sh` откатывает состояние к «default ACCEPT»;
- никаких секретов в скриптах (grep: пусто);
- никаких хардкоженных IP (только хосты, резолв в runtime).

### Пункт 4. Тест-план и его исполнение в изолированной среде

**Что:** до применения правил на coordinata56 server — полная прогонка в изолированном docker-контейнере (Ubuntu 22.04, та же версия что на сервере). Сервер не трогаем, пока тест-контур не зелёный.

*Positive (должно работать):*
1. `curl -sS -o /dev/null -w '%{http_code}' --max-time 10 https://api.anthropic.com/v1/messages -X POST -H "anthropic-version: 2023-06-01"` → ответ HTTP (401/400 — нас интересует, что TLS поднялся и ответ пришёл; не timeout).
2. `curl -sS -o /dev/null -w '%{http_code}' --max-time 10 https://api.telegram.org/bot0:fake/getMe` → 404 от Telegram (ключ невалидный, но соединение есть).
3. `getent hosts example.com` → возвращает IP (DNS работает).
4. `chronyc tracking` или `timedatectl timesync-status` → NTP синхронизация успешна.
5. `apt update` (на тестовом контейнере) → работает.
6. `curl --max-time 5 https://raw.githubusercontent.com/torvalds/linux/master/README` → ответ приходит (если GitHub в whitelist — в зависимости от решения Пункта 1).

*Negative (должно блокироваться):*
7. `curl -sS --max-time 5 https://api.openai.com` → timeout или `Couldn't connect to server`, exit code ≠ 0.
8. `curl -sS --max-time 5 https://huggingface.co` → timeout.
9. `curl -sS --max-time 5 https://example.com` → timeout.
10. `curl -sS --max-time 5 https://8.8.4.4` (прямой IP, не домен) → timeout.
11. `nc -vz api.openai.com 443 -w 5` → failure.
12. Проверка логов: `journalctl | grep EGRESS_DROP` содержит записи для п. 7–11.

*Resilience:*
13. После `systemctl restart networking` (или аналога) — правила сохраняются (persistence работает).
14. После перезагрузки ОС (`reboot` в контейнере) — правила восстанавливаются при старте.
15. Whitelist обновляется при добавлении нового хоста в источник данных без ручного вмешательства (если выбран подход с cron-обновлением ipset).
16. Rollback-скрипт возвращает состояние к исходному: `iptables-save` до и после rollback идентичны (diff = пусто).

**Acceptance:**
- `docs/operations/egress-test-plan.md` с полным списком кейсов, ожиданиями, фактическими результатами;
- `scripts/egress-test.sh` — автоматизированный прогон кейсов 1–12, вывод pass/fail;
- на тестовом контейнере — все 16 кейсов зелёные;
- лог прогона приложен к отчёту Head'а Директору;
- **отдельный smoke для рабочих сервисов сервера (позже, при реальном применении, не сейчас):** проверить, что `claude-telegram.service` продолжает принимать сообщения; dashboard 8765 отвечает внешним клиентам; субагенты Claude Code работают (тестовый `Agent`-вызов проходит).

## 6. Риски и зависимости

### 6.1 Главный риск: самоблокировка сервера

**Сценарий:** неверное правило блокирует ssh/Telegram/dashboard/субагентов → сервер становится недоступен.

**Митигации:**
1. **Страховка `at now + 5 minutes && iptables -F OUTPUT` при любом `--apply`.** Если администратор не подтвердил `egress-confirm.sh` в течение 5 минут — автоматический откат. Реализовано в Пункте 3.
2. **ssh-сессия держится открытой keep-alive'ом** при первом применении. Рекомендация: одновременно два терминала ssh (один выполняет apply, второй — мониторит).
3. **Изолированный docker-контейнер для отладки** — настоящий сервер трогается только после зелёных тестов в контейнере. Пункт 4.
4. **Rollback в один клик** — `scripts/egress-rollback.sh`. Пункт 7.
5. **Бэкап текущих iptables** перед любым изменением. Пункт 3.

### 6.2 Риск динамических IP (`api.anthropic.com`, `api.telegram.org`)

Cloudflare и подобные CDN меняют IP-адреса в ответах DNS без предупреждения. Правило по IP устареет через часы.

**Митигация:** механизм периодического пере-резолва DNS → обновления `ipset` (Пункт 2, рекомендация devops). Периодичность — не реже раза в 15 минут, желательно раз в 5 минут. Плюс кэширование в `systemd-resolved` с разумным TTL.

**Остаточный риск:** если DNS-резолв упал, и старый IP устарел — первый же вызов упадёт. Митигация: алерт в будущем на рост DROP-счётчика (M-OS-2).

### 6.3 Риск конфликта с существующими правилами на сервере

На coordinata56 уже настроены какие-то правила (минимум — открытые порты 22 ssh и 8765 dashboard). Полная перезапись через `iptables -F` снесёт их.

**Митигация:**
- скрипт `egress-apply.sh` **трогает только цепочку OUTPUT**, не INPUT/FORWARD;
- **наши правила — в отдельных именованных цепочках `EGRESS_*`**, основная цепочка OUTPUT только ссылается на них. При пересборке очищаются только `EGRESS_*`;
- inventory текущих iptables-правил на сервере (Пункт 1 ч. 2) — обязательный шаг перед первым apply.

### 6.4 Риск поломки Telegram-бота

`claude-telegram.service` принимает сообщения через Telegram long polling (исходящее HTTPS к `api.telegram.org`). Если whitelist для Telegram работает криво — Владелец перестанет получать/отправлять сообщения, канал связи обрывается.

**Митигация:**
- Telegram — первый в whitelist, высший приоритет;
- в тестах-Positive (Пункт 4, кейс 2) — Telegram проверяется обязательно;
- перед применением на сервере — Координатор уведомляет Владельца в Telegram: «применяю правила, если бот перестанет отвечать на 5+ минут — ssh на сервер и запусти `scripts/egress-rollback.sh`».

### 6.5 Зависимость от backend-отдела (нет)

Эта задача не требует координации с backend в реалтайме. Backend реализует runtime-guard в PR #4 Волны 1 параллельно; iptables — второй независимый эшелон. Интеграция между ними (подписка на событие `AdapterActivated` из `business_events_bus`) — отложена на M-OS-2+, сейчас только зафиксировано в документации.

### 6.6 Зависимость от Владельца

Применение на production-сервере coordinata56 (81.31.244.71) требует явного го-ахед Владельца. Он же — собственник сервера, он же — единственный, кто пользуется Telegram-каналом. Без его подтверждения — apply на live-сервере запрещён (см. §11).

## 11. Ограничения (жёсткие)

- **Первое реальное применение на сервере coordinata56 (81.31.244.71) — только после явного go-ahead Владельца** через Координатора. Правило CLAUDE.md: «Производственные операции — только с явного разрешения Владельца». Сервер несёт рабочие сервисы (Telegram, dashboard, субагенты) — любая ошибка прерывает работу Владельца.
- **Никаких секретов-литералов** в скриптах, yaml, документах. Только `$ENV_VAR` или чтение из конфига.
- **Никаких хардкоженных IP.** Только hostnames в yaml, IP-резолв — в runtime через DNS.
- **`git add -A` запрещён.** Только перечисление конкретных файлов (правило CLAUDE.md §«Git»).
- **Никаких изменений в `INPUT` и `FORWARD` цепочках iptables.** Только `OUTPUT` и именованные цепочки `EGRESS_*`.

- **`set -euo pipefail`** в каждом bash-скрипте. `shellcheck` — обязательно чистый.
- **Idempotency** — каждый скрипт безопасен при повторном запуске.

Можно выделить в `scripts/lib/safety-net.sh` и переиспользовать в будущих задачах (firewall-INPUT, systemd-переключения). Отложить на M-OS-2, если Координатор не сочтёт иначе.
2. **Мониторинг DROP-событий** — простой алерт «если >100 DROP/минуту — уведомление в Telegram». Не в скоупе этой задачи; предлагаю добавить в отдельный мини-PR после появления live-мониторинга (M-OS-2). Сейчас — только логирование.
3. **Полноценный forward-proxy (вариант 3 из Пункта 2)** — более мощное решение на будущее. Сейчас избыточно, но при росте количества внешних интеграций в M-OS-2+ может стать удобнее управлять allowlist на уровне hostname, а не IP. Предлагаю: если devops в Пункте 2 в сравнительной таблице показывает ощутимые преимущества proxy-подхода — рассмотреть проводить рефакторинг в M-OS-2 как плановая работа. В M-OS-1 — iptables+ipset достаточно.
4. **Пакет `iptables-persistent`** ставится `apt install` — это системный пакет. По CLAUDE.md глобальному: перед `apt install` предупредить Владельца что именно ставится и зачем. Предложение: Координатор в бриф Владельцу явно вставляет пункт «установка пакета `iptables-persistent` на coordinata56» вместе с запросом go-ahead на применение правил.
