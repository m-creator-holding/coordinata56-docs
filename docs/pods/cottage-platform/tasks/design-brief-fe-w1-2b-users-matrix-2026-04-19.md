---

## 0. Назначение

Текущая страница `/admin/users` (FE-W1-2) даёт базовый CRUD пользователей.
Нужно спроектировать **матрицу назначений user × company × role** —
вложенный экран/вкладка, где Владелец видит полную раскладку «какой
пользователь в каком юрлице какую роль имеет» и может массово назначать
роли.

Wireframes v0.3 явно не описали этот экран (остановились на list + detail
базового users), но из решений Владельца (C-1 consent, row-isolation
бухгалтера, per-company roles ADR 0011) ясно: матрица обязательна.
Design-director заполняет пробел.

---

## 1. Бизнес-цель

Владелец быстро отвечает на вопросы:
- «Кто в ООО „Координата 56" прораб?»
- «У Иванова какие роли в каких юрлицах?»
- «Какой бухгалтер привязан к АЗС-Холдингу?»

Без матрицы сейчас надо открывать каждого пользователя и смотреть вкладку
«Роли» — это медленно и не даёт общей картины.

---

## 2. Что спроектировать (3 экрана)

### 2.1 List View `/admin/users` — добавить колонки

Существующая таблица показывает full_name / email / phone / is_active.
Добавить:
- **Юрлица** — список Badge-ов названий компаний, к которым привязан user
  (≤3 Badge, если > 3 — «+N»)
- **Роли** — список Badge-ов ролей (de-duplicated по всем компаниям)
- **Holding-owner** — иконка «корона» если is_holding_owner=true

**Фильтры в toolbar:**
- Select «Компания» — показать только users, у которых есть привязка к выбранной
- Select «Роль» — показать только users с этой ролью хотя бы в одной компании
- Input «Поиск» — по full_name / email

Row-level для бухгалтера: видит только своих коллег по одному юрлицу;
фильтр «Компания» ему заблокирован (выбрано своё юрлицо).

### 2.2 Detail View — новая вкладка «Привязки» (user_company_role matrix)

Существующие вкладки user: Основное / Роли. **Заменить вкладку «Роли» на
«Привязки»** — матрица company × role для этого пользователя.

Layout:
```
┌─────────────────────────────────────────────────────────────┐
│  Привязки: 3 активных                    [+ Добавить]       │
├─────────────────────────────────────────────────────────────┤
│  Компания              │ Роль            │ Назначен │ ⚙     │
│  ООО «Координата 56»   │ Старший прораб  │ 01.04.26 │[X]    │
│  ООО «Координата 56»   │ Финансовый      │ 10.04.26 │[X]    │
│  ООО «АЗС Холдинг»     │ read_only       │ 01.04.26 │[X]    │
└─────────────────────────────────────────────────────────────┘
```

- Диалог «+ Добавить привязку»: Select компания (все, к которым есть доступ),
  Select роль (из fixtures/roles), кнопки «Отмена» / «Назначить»
- Кнопка «X» в строке — AlertDialog «Снять привязку? Пользователь потеряет
  доступ к данным этого юрлица.»
- Если user is_holding_owner=true — вкладка скрыта, banner «Владелец холдинга
  имеет доступ ко всем юрлицам автоматически»

### 2.3 Standalone matrix page `/admin/users/matrix` (опционально)

Полноэкранная матрица users × companies, ячейки = роли (Badge-список).
Владелец видит всю холдинговую картину.

Layout:
```
            │ ООО «К56» │ ООО «АЗС» │ ООО «Карьер» │ …
Иванов П.С. │ прораб    │ —         │ —            │
Сидоров Д.В.│ фин       │ read_only │ —            │
Петров А.И. │ бух       │ бух       │ бух          │
```

Клик по ячейке — открывает `UserDetailsPage` на вкладке «Привязки».

**Для sprint 1.1B достаточно 2.1 и 2.2.** 2.3 — nice-to-have.

---

## 3. Что НЕ рисовать

- Массовое назначение роли сразу N пользователям (bulk) — M-OS-2
- Импорт пользователей из CSV — M-OS-2
- SSO / OAuth — out of scope M-OS-1

## 5. Acceptance Criteria (≥3 тестовых сценария)

AC-1. **List users — фильтр по компании.**
Given MSW с 5 users, каждый привязан к разному набору компаний.
When Владелец выбирает фильтр «Компания = ООО „Координата 56"».
Then таблица показывает только users с привязкой к этой компании; колонка
«Юрлица» у каждой строки содержит Badge «Координата 56».

AC-2. **Добавление привязки.**
Given user Иванов, вкладка «Привязки» на его detail.
When клик «+ Добавить», выбрана компания ООО «АЗС Холдинг» + роль
`read_only`, «Назначить».
Then Dialog закрывается; новая строка появляется в таблице привязок;
Toast «Привязка назначена»; POST `/api/v1/users/:id/roles` вызван.

AC-3. **Бухгалтер — row-isolation.**
Given бухгалтер авторизован, привязан к ООО «Координата 56».
When открывает `/admin/users`.
Then таблица показывает только пользователей с привязкой к той же компании;
фильтр «Компания» заблокирован на своём юрлице; кнопки «+ Добавить
пользователя» и «X» (снять привязку) отсутствуют.

AC-4. **Holding-owner — специальный баннер.**
Given Мартин (is_holding_owner=true), вкладка «Привязки».
Then вкладка показывает только banner «Владелец холдинга имеет доступ ко
всем юрлицам автоматически»; таблица привязок и кнопка «+ Добавить»
отсутствуют.

## 7. Открытые вопросы к design-director

DQ-1. Нужна ли standalone matrix `/admin/users/matrix` в sprint 1.1B или
откладываем? Рекомендация: откладываем (nice-to-have).

DQ-2. Диалог добавления привязки — Dialog (modal) или Sheet (drawer)?
Рекомендация: Dialog, т.к. форма из 2 полей, не требует длинной прокрутки.

DQ-3. Отображение истории «кто назначил эту привязку и когда» — нужно
сейчас или это M-OS-1.1C аудит? Рекомендация: сейчас только `assigned_at`
в колонке таблицы, кто назначил — в M-OS-1.1C.
