## 0. Основание и источники

Этот бриф — первая дизайн-задача отдела в M-OS. Артефакт (текстовые wireframes)
является содержательным контрактом для frontend-dev в батчах M-OS-1.1-batch-2
и последующих. FE-skeleton строится frontend-отделом параллельно
(`docs/pods/cottage-platform/tasks/fe-skeleton-m-os-1-1.md`); wireframes передаются
им на handoff как содержательный контент для наполнения skeleton-страниц.

**Обязательные источники — designer читает до начала работы:**

1. `docs/adr/0011-foundation-multi-company-rbac-audit.md` — модели данных:
   `Company`, `UserCompanyRole`, `RolePermission`, `AuditLog` (крипто-цепочка).
   Из ADR 0011 берутся: список полей сущностей, логика multi-company, семантика
   матрицы прав (роль + action + resource_type).
2. `docs/pods/cottage-platform/m-os-1-plan.md` v1.3 — скоуп M-OS-1.1 (§«Под-фазы»):
   что входит в M-OS-1.1, что сдвинуто в M-OS-1.2 и M-OS-1.3.
3. `docs/pods/cottage-platform/tasks/fe-skeleton-m-os-1-1.md` — контракт с
   frontend: структура AdminLayout (левый sidebar, топбар с переключателем
   компании), routing `/admin/*`, shadcn/ui-компоненты, TanStack Query,
   ADR 0002 стек. Designer не выходит за пределы этого стека при описании
   wireframes.
4. `docs/agents/CODE_OF_LAWS.md` ст. 45а — запрет живых интеграций:
   в разделе Integration Registry (экран 6) designer не проектирует UI-формы
   для подключения банков, 1С и прочих — только Telegram.
5. `/root/coordinata56/CLAUDE.md` секция «UX > визуальный дизайн»:
   на MVP фокус — удобная архитектура UX, user flows, читаемые формы;
   визуальный полиш — M-OS-1.4, не сейчас.

**Прототип Владельца (Next.js + Supabase, MacBook).**
Прямого доступа к коду прототипа у designer нет — он в отдельной локальной среде
Владельца и не перенесён в репозиторий (решение A, принятое Координатором).
Если designer встретит неоднозначность в UX (порядок полей, иерархия действий,
формулировки) — эскалировать через ux-head Координатору с конкретным вопросом.
Координатор запросит у Владельца скриншот или walkthrough нужного экрана.
Без ответа — designer принимает решение самостоятельно, обоснование фиксирует
в wireframe-файле.

## 1. Формат выходного артефакта

Один файл: `docs/pods/cottage-platform/specs/wireframes-m-os-1-1-admin.md`.

Для каждого из 7 экранов обязательны все шесть блоков:

```
### Экран N. <Название> — <путь>

#### Persona и сценарий
<Кто входит на этот экран и с какой целью. 2–4 предложения.>

#### Text-based wireframe
<ASCII/символьный layout: шапка, sidebar, main content area, actions.
Стрелки, боксы, колонки — любым способом, лишь бы структура была читаема.>

#### Поля формы / колонки таблицы
<Таблица: Поле | Тип UI-элемента | Обязательность | Валидация | Help-текст>

#### Состояния UI
<Список: loading, empty, error, success — для каждого: что видит пользователь.>

#### Связи с другими экранами
<Какие действия куда ведут. Список: «Кнопка X → Экран Y», «Клик по строке → Детальная карточка».>

#### Ссылка на OpenAPI
<Список endpoint-заглушек из zero-version OpenAPI, которые обслуживают этот экран.
Формат: METHOD /api/v1/path — назначение.>
```

В конце файла — раздел **«Навигационная карта»**:
- Структура левого sidebar AdminLayout: порядок пунктов, вложенность.
- Схема связей между 7 экранами: как они соединяются (не диаграмма — текстом,
  список переходов).

## 2. Ограничения и правила работы

| Правило | Детали |
|---|---|
| Только markdown | Figma не используется. Wireframes — текстом, ASCII-схемы, таблицы. |
| Не визуальный полиш | Никаких цветов, брендинга, иконок — только структура и контент. Цвет бренда, логотип, типографика — M-OS-1.4. |
| Компоненты — shadcn/ui | При описании UI-элементов ссылаться на shadcn/ui-компоненты: `Table`, `Sheet`, `Dialog`, `Form`, `Select`, `Switch`, `Checkbox`, `Badge`, `Card`. Не изобретать собственные. |
| Tailwind, RR6 | Стек фиксирован ADR 0002. Designer не предлагает другие библиотеки. |
| 7 экранов и только они | М-OS-1.2 (BPM Constructor, Notification Constructor, Document Templates) и M-OS-1.3 (Form Builder, Report Builder) — отдельные задачи. Не включать. |
| Телеграм — единственная живая интеграция | В экране 6 (Integration Registry) живой UI — только для Telegram-бота. Все прочие интеграции: карточки в «каталоге» со статусом `disabled`, без форм активации (ст. 45а CODE_OF_LAWS). |
| Не пишет код | Designer пишет описания, не TSX/HTML. |

### Экран 1. Companies (Юрлица) — P0

**Путь:** `/admin/companies`

**Что нужно спроектировать:** три режима одного раздела:
- **Список** — таблица компаний с поиском и фильтрацией.
- **Форма создания / редактирования** — карточка с полями компании.
- **Детальная карточка** — расширенный просмотр с вкладками.

**Поля компании (из ADR 0011, §1.1):**

| Поле | Тип данных | Замечание для UI |
|---|---|---|
| `inn` | строка 10 или 12 цифр | ИП — 12, ООО/АО — 10. Валидация обязательна. Nullable на уровне БД, но UI для production-записи требует заполнения. |
| `kpp` | строка 9 цифр | Только для юрлиц (ООО, АО). Для ИП — скрывать поле. |
| `full_name` | строка | Полное юридическое наименование |
| `short_name` | строка | Краткое (показывается в топбаре, списках) |
| `company_type` | enum: ООО, АО, ИП, ДРУГОЕ | Влияет на набор полей (КПП) |
| `is_active` | boolean | Переключатель. Неактивная — выделяется в списке. |

**Дополнительные поля реквизитов** (не в ADR 0011, но описаны в задаче Координатора как обязательные для M-OS-1.1):

| Поле | Тип данных | Замечание для UI |
|---|---|---|
| ОГРН | строка 13 или 15 цифр | 13 — ЮЛ, 15 — ИП |
| Юридический адрес | строка | Полный адрес |
| Банковские реквизиты | сгруппированный блок | БИК, расчётный счёт, наименование банка, корр. счёт. Возможно несколько счётов — designer решает, один или список. |
| ФИО генерального директора | строка | Для ИП — «Генеральный директор» заменить на «Индивидуальный предприниматель». |
| Печати и подписи | файловый блок | На M-OS-1 — placeholder с кнопкой «Загрузить» (файловое хранилище появится в M-OS-2, ADR 0011 §4 file_id-заглушка). Designer проектирует UI с учётом этого: кнопка активна, но ведёт в empty-state с пояснением. |

**Логика UX:**
- Список — таблица с колонками: краткое название, ИНН, тип, статус, дата создания. Строка кликабельна → детальная карточка.
- В правом верхнем углу кнопка «+ Добавить компанию».
- Форма создания/редактирования — `Sheet` (боковая панель) или отдельный роут `/admin/companies/new` и `/admin/companies/:id/edit`. Designer выбирает и обосновывает.
- Детальная карточка `/admin/companies/:id` — с вкладками: «Реквизиты», «Сотрудники» (список пользователей с ролями в этой компании), «Настройки» (ссылка на экран 5).

### Экран 2. Users (Пользователи) — P0

**Путь:** `/admin/users`

**Что нужно спроектировать:**
- **Список** — таблица пользователей.
- **Форма создания / редактирования.**
- **Детальная карточка** — включает блок «Роли по компаниям».

**Поля пользователя:**

| Поле | Замечание для UI |
|---|---|
| ФИО (`full_name`) | Обязательно |
| Email | Обязательно, уникально. Валидация формата. |
| Телефон | Необязательно. Маска: +7 (___) ___-__-__. |
| `is_active` | Переключатель. Деактивированный пользователь не может войти. |
| Пароль | Только при создании. При редактировании — отдельная кнопка «Сбросить пароль». |

**Многокомпанийная логика (критически важно):**

Один пользователь может иметь роль в нескольких компаниях (модель `UserCompanyRole`).
Поле «Роли по компаниям» — это не одно поле, а список привязок:

```
Компания          Роль                    Pod (опционально)    Действия
─────────────────────────────────────────────────────────────────────
ООО «Металл»      Бухгалтер               —                    [Изменить] [Удалить]
ООО «АЗС»         Прораб                  gas-stations         [Изменить] [Удалить]
[+ Добавить привязку]
```

Designer проектирует этот блок отдельно и описывает, как добавляется новая привязка
(inline-форма в таблице? модальный `Dialog`? боковой `Sheet`?).

**Поля формы привязки** (UserCompanyRole):

| Поле | Тип UI | Обязательность |
|---|---|---|
| Компания | `Select` со списком компаний | Обязательно |
| Роль | `Select` со списком ролей | Обязательно |
| Pod (ограничить доступ) | `Select` или текстовое поле, с пометкой «Оставьте пустым для доступа ко всем подам» | Необязательно |

### Экран 3. Roles (Роли) — P0

**Путь:** `/admin/roles`

**Что нужно спроектировать:**
- **Список ролей** — таблица с колонками.
- **Форма создания / редактирования роли.**

**Контекст:** роли в ADR 0011 — `role_template` со значениями `owner`, `accountant`,
`construction_manager`, `read_only`. На M-OS-1 это фиксированный набор (seed-скриптом),
но admin-UI позволяет редактировать название, описание и область действия.
Designer проектирует UI без предположения о том, что роли нельзя будет добавлять
в будущем — форма должна допускать это расширение.

**Поля роли:**

| Поле | Тип UI | Замечание |
|---|---|---|
| Код (`role_template`) | Текст (только для чтения при редактировании) | При создании — текстовое поле с валидацией `snake_case`. |
| Название (отображаемое) | `Input` | Пример: «Бухгалтер», «Прораб», «Владелец» |
| Описание | `Textarea` | Краткое — что за роль, какие типичные обязанности |
| Область действия | `Select`: «Для конкретной компании» / «Глобальная (все компании)» | Соответствует логике per-company vs is_holding_owner в ADR 0011 |
| Статус | `Badge`: Активна / Системная (не удаляется) | Системные роли из seed — помечаются иначе |

**Связь с экраном 4 (Permissions Matrix):** на детальной карточке роли — кнопка
«Управлять правами» → переход в экран 4, предфильтрованный по этой роли.

### Экран 4. Permissions Matrix (Матрица прав) — P1

**Путь:** `/admin/permissions`

**Что нужно спроектировать:**
- **Таблица-редактор** «роль × action × resource_type».
- **Механизм сохранения** изменений (построчно или пакетно).

**Контекст:** модель `RolePermission` (ADR 0011, §2.2):

```
role_template | action     | resource_type | pod_id | is_allowed
──────────────────────────────────────────────────────────────
owner         | read       | contract      | NULL   | true
accountant    | approve    | payment       | NULL   | false
```

Действия (`action`): `read`, `write`, `approve`, `delete`, `admin`.
Ресурсы (`resource_type`): `contract`, `payment`, `project`, `*` (все).

**UX этого экрана — самый сложный из семи.** Designer должен принять решение о подходе
и явно обосновать его в wireframe-файле. Два варианта на рассмотрение:

**Вариант A (рекомендуется для MVP):** таблица role × action с группировкой по
resource_type. Колонки — действия, строки — роли, ячейки — чекбоксы. Если ресурсов
несколько — вкладки или accordion по resource_type.

```
                  contract               payment
                  ───────────────────────────────
Роль              read  write  approve   read  write  approve  delete
──────────────────────────────────────────────────────────────────────
Владелец          [x]   [x]    [x]       [x]   [x]    [x]      [x]
Бухгалтер         [x]   [x]    [ ]       [x]   [x]    [ ]      [ ]
Прораб            [x]   [ ]    [ ]       [x]   [ ]    [ ]      [ ]
Только просмотр   [ ]   [ ]    [ ]       [x]   [ ]    [ ]      [ ]
```

**Вариант B (альтернатива):** выбор роли из dropdown → список строк permission
с чекбоксами. Менее компактно, зато проще реализовать при большом числе ресурсов.

Designer выбирает один вариант и описывает оба состояния: «режим просмотра» и
«режим редактирования» (inline или через кнопку «Редактировать»).

**Механизм сохранения:** выбрать один из двух и обосновать:
- Авто-сохранение при изменении каждого чекбокса (с индикатором «Сохранено»).
- «Редактировать» → изменения → «Сохранить» / «Отменить».

**Состояние блокировки:** системные права (например, `owner` + `admin` + `*`) —
не редактируются. Designer описывает, как они выглядят (disabled чекбокс, tooltip).

### Экран 5. Company Settings (Настройки компании) — P1

**Путь:** `/admin/companies/:id/settings` (или `/admin/settings` с переключателем компании)

**Контекст:** таблица `company_settings` — per-company значения. Семь полей из
решения Владельца (Telegram msg 1094):

| Поле | Тип UI | Значения / Валидация | Help-текст |
|---|---|---|---|
| НДС-режим | `Select` | Без НДС, НДС 20%, НДС 10%, Упрощённая система | Влияет на расчёт договоров и платежей |
| Валюта | `Select` | RUB (основная), USD, EUR | На M-OS-1 — справочно; мультивалютный учёт — M-OS-2 |
| Часовой пояс | `Select` | Список IANA-зон (Europe/Moscow и т.д.) | Влияет на отображение времени в аудит-логе |
| Рабочая неделя | `Checkbox-группа` | Пн/Вт/Ср/Чт/Пт/Сб/Вс | Влияет на расчёт сроков в BPM-процессах (M-OS-1.2) |
| Единицы измерения | `Select` | метрические (м, кг, м²) / имперские | Для строительных объёмов |
| Логотип | Файловый upload | PNG/SVG, до 2 МБ | Отображается в шапке при печати документов. На M-OS-1 — placeholder (файловое хранилище M-OS-2) |
| Цвет бренда | `Input` type color или HEX-строка | Hex (#RRGGBB), с превью | Используется в шаблонах документов (M-OS-1.2) |

**Важно:** файловое хранилище появляется в M-OS-2. На M-OS-1 поля «Логотип» и
«Печати/подписи» (экран 1) — UI-заглушки: кнопка «Загрузить» активна, но при нажатии
показывает информационный banner: «Загрузка файлов будет доступна после настройки
файлового хранилища. Свяжитесь с администратором M-OS.»

**UX:** форма разбивается на группы («Бухгалтерия», «Региональные настройки»,
«Внешний вид»). Кнопки «Сохранить» / «Отменить изменения» — внизу формы.
Audit-trail: под формой — ссылка «История изменений» → отфильтрованный аудит-лог
по этой компании и entity_type = `company_settings`.

### Экран 6. Integration Registry (Реестр интеграций) — P2

**Путь:** `/admin/integrations`

**Контекст (CODE_OF_LAWS ст. 45а):** на M-OS-1 — одна живая интеграция, Telegram-бот.
Все остальные адаптеры (банки, Росреестр, 1С, ОФД) существуют в коде с
`enabled=False` и не подключаются до production-gate.

**Что нужно спроектировать:**

**Каталог интеграций** — список карточек (`Card`) каждой потенциальной интеграции:

```
┌──────────────────────┐   ┌──────────────────────┐   ┌──────────────────────┐
│  Telegram-бот        │   │  Сбербанк             │   │  1С: Предприятие     │
│  ● Подключён         │   │  ○ Недоступно         │   │  ○ Недоступно         │
│  Алерты и уведомл.  │   │  Откроется после      │   │  Откроется после     │
│  [Настроить]         │   │  production-gate      │   │  production-gate     │
└──────────────────────┘   └──────────────────────┘   └──────────────────────┘
```

Статус «Недоступно» — не кнопка, а информационный текст. Клик на карточку
«Недоступно» открывает modal с пояснением: «Интеграция будет активирована
после прохождения production-gate (юридическая проверка + security-аудит).»

**Форма настройки Telegram-бота** (Sheet или `/admin/integrations/telegram`):

| Поле | Тип UI | Валидация | Help-текст |
|---|---|---|---|
| Bot Token | `Input` type password | Не пустой, формат `\d{10}:[\w-]{35}` | Получите в @BotFather. Токен хранится в зашифрованном виде. |
| Chat ID для алертов | `Input` | Числовое | Telegram chat_id, куда бот отправляет системные оповещения |
| Разрешённые события | `Checkbox-группа` | — | Выберите события, о которых бот уведомляет (платёж принят, договор подписан, ошибка системы, ...) |
| Статус | `Switch` + `Badge` | — | Включить / Выключить бота без удаления конфигурации |

**Кнопка «Проверить подключение»** — отправляет тестовое сообщение в указанный chat_id.
Ответ: «Тестовое сообщение отправлено» / «Ошибка: <текст ошибки от Telegram API>».

### Экран 7. System Config (Системная конфигурация) — P2

**Путь:** `/admin/system`

**Доступ:** только пользователям с `is_holding_owner=true` (суперадмин холдинга).
Все остальные видят экран 403 с текстом «Этот раздел доступен только владельцу холдинга.»

**Что нужно спроектировать:**
- Форма глобальных настроек M-OS.
- Блок просмотра системной информации (read-only).

**Поля (глобальные, не per-company):**

| Поле | Тип UI | Замечание |
|---|---|---|
| URL приложения | `Input` | Базовый URL (используется в генерации ссылок в письмах, уведомлениях) |
| Название системы | `Input` | По умолчанию «M-OS». Отображается в шапке и письмах. |
| Максимальный размер файла (МБ) | `Input` number | Глобальный лимит загружаемых файлов |
| Флаги фич (Feature Flags) | Список с `Switch` на каждый флаг | На M-OS-1 — минимальный набор. Designer проектирует expandable-список с поиском. |
| Сессия: максимальное время неактивности (мин) | `Input` number | После N минут — принудительный logout |

**Системная информация (read-only блок):**

| Параметр | Откуда |
|---|---|
| Версия M-OS | Константа из backend |
| Дата последнего деплоя | Метка из build |
| Статус audit-цепочки | GET /api/v1/audit/verify — показать «OK» / «Нарушение цепочки» |
| Число компаний в системе | GET /api/v1/companies?count=true |

**UX-замечание:** экран системного конфига должен визуально сигнализировать о
высокой ответственности действий — зона только для владельца. Designer описывает,
как это достигается структурой и текстом (без цвета — цвет M-OS-1.4).

## 4. Навигационная карта (designer заполняет этот раздел в итоговом файле)

Ниже — скелет, который designer **дополняет и уточняет** при финальной сборке
всех семи wireframes. Этот раздел не является директивным — designer может
изменить структуру, если нашёл более логичный порядок, с обоснованием.

**Предлагаемая структура левого sidebar AdminLayout:**

```
M-OS Admin
──────────────────────────────
Управление
  ├── Юрлица               /admin/companies
  ├── Пользователи         /admin/users
  └── Роли                 /admin/roles

Права и настройки
  ├── Матрица прав         /admin/permissions
  └── Настройки компании   /admin/companies/:id/settings
                           (или /admin/settings — designer выбирает)

Система
  ├── Реестр интеграций    /admin/integrations
  └── Системная конфигур.  /admin/system
```

**Схема переходов между экранами** (designer детализирует):

- Список компаний → клик на строку → Детальная карточка компании
- Детальная карточка компании → вкладка «Настройки» → Экран 5 (Company Settings)
- Список компаний → «+ Добавить» → Форма создания компании
- Список пользователей → клик на строку → Детальная карточка пользователя
- Детальная карточка пользователя → блок «Роли» → Форма привязки роли
- Список ролей → клик на строку → Детальная карточка роли
- Детальная карточка роли → «Управлять правами» → Матрица прав (предфильтр по роли)
- Матрица прав → (без обратного перехода, только через sidebar)
- Реестр интеграций → карточка Telegram → Форма настройки Telegram
- Системная конфигурация → не ведёт никуда дополнительно

## 7. Definition of Done для designer

Wireframes считаются готовыми к передаче ux-head, когда:

1. Файл `docs/pods/cottage-platform/specs/wireframes-m-os-1-1-admin.md` создан.
2. Все 7 экранов содержат все 6 обязательных блоков (persona + wireframe + поля +
   состояния + связи + OpenAPI endpoint-ссылки).
3. Раздел «Навигационная карта» заполнен: структура sidebar + схема переходов.
4. Все принятые решения по выбору из вариантов — обоснованы текстом внутри файла.
5. Не содержит экранов M-OS-1.2 и M-OS-1.3 (BPM, Form Builder и т.д.).
6. Не содержит UI-форм активации внешних интеграций кроме Telegram.
7. В разделе Integration Registry статус неактивных интеграций — «Недоступно»
   (информационный), не «Подключить» (кнопка действия).

## 8. Жёсткие ограничения (повторение)

- **Не пишет wireframes сам design-director.** Это задача designer. Данный файл —
  только бриф.
- **Не выходит за 7 экранов M-OS-1.1.** При сомнении — секция «Вопросы к
  Координатору» в wireframe-файле, не самостоятельное расширение скоупа.
- **Не принимает решений о визуальном языке.** Цвета, типографика, иконки —
  M-OS-1.4. В wireframes — только структура.
- **Живые интеграции кроме Telegram запрещены.** Ст. 45а CODE_OF_LAWS.
- **Shadcn/ui-компоненты.** Не изобретать компоненты вне библиотеки.

## 9. Вопросы к Координатору (от design-director)

Ниже — пункты, которые требуют решения Владельца или подтверждения
backend-director'а. Designer не блокируется на них — продолжает работу
с разумными предположениями, фиксируя их в wireframe-файле. При получении
ответа Координатора — уточняет.

1. **Путь к Company Settings.** Предлагаю `/admin/companies/:id/settings` (настройки
   привязаны к компании в URL). Альтернатива — `/admin/settings` с переключателем
   компании в шапке (как в Stripe). Какой подход ближе к видению Владельца?
   Влияет на структуру sidebar и маршрутизацию.

2. **Количество банковских счётов на одну компанию.** ADR 0011 не описывает
   таблицу банковских реквизитов. Это один счёт (простые поля на форме)
   или список (sub-таблица с CRUD)? Влияет на сложность wireframe экрана 1.

3. **Feature Flags для System Config.** Какие флаги фич планируются на M-OS-1?
   Если набор уже известен backend-director'у из ADR-0018 — нужен список, чтобы
   designer спроектировал реальную таблицу, а не placeholder.

4. **Скриншоты прототипа Владельца.** Если Владелец готов сделать walkthrough
   admin-разделов своего прототипа (Next.js + Supabase) — это сэкономит designer'у
   1–2 дня на принятие UX-решений по формам и порядку полей. Прошу Координатора
   уточнить у Владельца.
