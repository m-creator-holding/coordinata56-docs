## 0. Назначение

Спроектировать новую страницу `/admin/rules` — управление per-company
бизнес-правилами (payment_overrun_limit_pct, approval_amount_threshold
и будущие) с обязательным версионированием через `configuration_entities`
+ снапшоты `payment_rule_snapshots` (ADR-0023 / Решение Владельца Q5
msg 1480).

Wireframe v0.3 §3 детально описал list view, detail/edit view и UX
снапшотов. Design-director финализирует и утверждает детали, которые
в wireframe заданы текстом (banner-readonly, AlertDialog warnings,
snapshot drawer vs page).

---

## 1. Бизнес-цель

Владелец централизованно управляет лимитами/порогами/правилами по всем
5 юрлицам холдинга из одного экрана. При изменении правила:
- старые заявки сохраняют ссылку на версию правила, по которой были одобрены
  (через `payment_rule_snapshots`)
- новые заявки используют актуальную версию
- изменение требует обязательного комментария «почему»
- история версий — прозрачна, откат возможен одним кликом

Бухгалтер видит тот же экран в режиме readonly, может проверять какие
лимиты действуют.

## 2. Что спроектировать (3 экрана + 1 drawer)

### 2.1 List View `/admin/rules` — реестр правил

Таблица (wireframe §3.1):
| Правило (rule_key_human) | Тип | Юрлицо | Значение | Версия |
| Порог перерасхода платежа | Процент | ООО «К56» | 15% | v2 |
| Порог перерасхода платежа | Процент | ООО «АЗС» | 8% | v1 |
| Сумма на подпись директора | Рубли | ООО «К56» | 500 000 | v3 |

Одно правило может быть представлено N строками (по одной на каждое
юрлицо), т.к. значения per-company (Решение 4).

**Фильтры в toolbar:**
- Select «Тип правила» (Все / Процент / Рубли / Дни)
- Select «Юрлицо» (все юрлица Владельца / собственное у бухгалтера)
- Input «Поиск по названию»

**Кнопка `+ Добавить правило`** — видна только Владельцу.

**Для бухгалтера:**
- Фильтр «Юрлицо» заблокирован на своём
- Кнопка «+ Добавить» скрыта
- Столбец «⚙» (контекстное меню) скрыт

### 2.2 Detail/Edit View `/admin/rules/:id` — редактор правила

Layout с правой панелью версий (wireframe §3.2):

**Левая часть:**
- Заголовок: название правила + Badge v2/Активна
- Read-only строка: `rule_key`, `Применяется к: ООО «Координата 56»`
- Input «Значение» с единицей (Input + прицепленный Select: % / ₽ / дни)
- Диапазон допустимых значений подсказкой
- Textarea «Описание правила» (readonly — из catalog)
- Блок «Влияет на BPM-процессы»:
  - Список процессов (маркированный список)
  - Alert (variant=warning): «Изменение применится к N текущим
    заявкам на согласование» (счётчик из API)
- **Textarea «Комментарий к изменению»** — обязательное поле (Zod required)
- Кнопки «Отмена» / «Сохранить изменения»

**Правая панель «История версий»** (~280 px):
- Таймлайн rule_v3 / rule_v2 / rule_v1 с датой, автором, коротким
  комментарием из поля commit
- Активная помечена точкой и текстом «Текущая»
- У неактивных — кнопка «Откатить»

### 2.3 Readonly-mode для бухгалтера

- Banner info: «У вас режим просмотра. Для изменений — обратитесь к
  директору.» — фиксированный сверху формы
- Input значения — disabled + `bg-muted`
- Textarea комментария — **скрыт полностью** (не disabled, а display:none)
- Кнопки «Сохранить изменения» / «Отмена» — скрыты
- Правая панель истории — видна (readonly), кнопки «Откатить» скрыты
- Badge версии правила в detail view платежа — кликабельный, снапшот
  доступен

### 2.4 Drawer/Dialog снапшота правила (при клике на badge в detail платежа)

Sheet (правый drawer, ширина ~480 px) — reuse паттерна из
`RolePermissionsHistoryDrawer`:

```
Снапшот правила: snap #4521 (rule_v1, 15.03.2026)
──────────────────────────────────────────────────
Правило: Порог перерасхода платежа
rule_key: payment_overrun_limit_pct
Применялось к: ООО «Координата 56»
Одобрено: 15.03.2026, Мартин
Статус правила сейчас: [Архивная] (актуальная — rule_v2)

Полный JSON правила:
┌──────────────────────────────────────────────┐
│ { "rule_key": "...", "value": 10,            │
│   "unit": "percent", "company_id": "...",    │
│   "comment": "Начальное значение", ... }     │
└──────────────────────────────────────────────┘
                                      [Закрыть]
```

Route: `/rules/snapshots/:snap_id` (deep-link, открывает тот же Sheet).

## 3. Что НЕ рисовать

- Симуляция «что будет если изменить лимит» — M-OS-2 симуляционная фаза
- Флаг `apply_to_running` для перезапуска активных BPM — отложено Владельцем
- Кросс-компанийные reports по правилам — M-OS-2
- Создание новой категории правил (rule_key catalog editor) — M-OS-1.1C

## 5. Acceptance Criteria (≥3 тестовых сценария)

AC-1. **List View — фильтр по юрлицу.**
Given 5 правил в MSW: 3 для К56, 2 для АЗС.
When Владелец выбирает фильтр «Юрлицо = ООО „АЗС Холдинг"».
Then таблица показывает 2 строки; обе — для АЗС; кнопка «+ Добавить
правило» присутствует.

AC-2. **Сохранение с BPM-warning.**
Given Владелец на `/admin/rules/42` (порог перерасхода 10% для К56).
And в MSW `active_approvals_count=3`.
When меняет значение с 10 на 15 и не вводит комментарий.
Then Zod-валидация: «Комментарий обязателен».
When вводит «Ужесточение по аудиту» и кликает «Сохранить».
Then Alert warning «Изменение применится к 3 текущим заявкам»
отображается ДО подтверждения; после Confirm — создаётся rule_v3; Toast
«Лимит обновлён. Версия v3 активна.»

AC-3. **Readonly для бухгалтера.**
Given бухгалтер на `/admin/rules/42`.
Then banner «У вас режим просмотра» виден; Input значения disabled;
Textarea «Комментарий к изменению» отсутствует в DOM; кнопки «Сохранить
изменения» / «Отмена» отсутствуют; правая панель истории видна без кнопок
«Откатить»; `payment_rule_snapshot` badges в detail платежа кликабельны
и открывают Sheet.

AC-4. **Снапшот правила — доступ из платежа.**
Given detail view платежа #PAY-0042, одобрен по rule_v1 (snap #4521).
When кликает Badge «Одобрено по rule_v1 (snap #4521) →».
Then открывается Sheet с полным JSON правила; заголовок «Снапшот правила:
snap #4521 (rule_v1, 15.03.2026)»; Badge «Архивная» + надпись «актуальная
— rule_v2»; кнопка «Закрыть» возвращает к detail платежа.

## 7. Открытые вопросы к design-director

DQ-1. Снапшот правила — **Sheet** (drawer) или **отдельная страница**
`/rules/snapshots/:id`? Рекомендация: Sheet для UX компактности, deep-link
тоже работает через query-param `?snapshot=4521`.

DQ-2. Если у правила нет BPM-процессов под ним — блок «Влияет на
BPM-процессы» скрывать полностью или показывать «Не используется в BPM»?
Рекомендация: скрывать полностью, без пустого блока.

DQ-3. **Цветовая кодировка юрлиц в таблице правил.** При 5+ юрлицах
Владельцу хочется визуально отделять карьер от АЗС от коттеджа. Рисовать
Badge с цветом per-company? Рекомендация: да, но на этапе sprint 1.1B
достаточно sequential palette (5 цветов tailwind), без семантики —
M-OS-2 разработает design-token цветов.

DQ-4. **Добавление нового правила (кнопка `+ Добавить`).** Wireframe не
показал form. Design-director рисует отдельный Dialog/Page: выбор
rule_key из catalog + юрлицо + начальное значение + описание.
