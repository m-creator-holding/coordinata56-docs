## 3. Скоуп PR #1 — 8 пунктов с acceptance criteria

Head обязан декомпозировать в задачи для backend-dev в указанном порядке. Каждый пункт — самостоятельный acceptance-критерий для ревью.

### Пункт 1. `backend/tools/lint_migrations.py` с шестью проверками

**Что:** Python-скрипт, парсирующий файлы `backend/alembic/versions/*.py` и ищущий запрещённые операции. Парсинг — через `ast` (рекомендация, не прибивать gvoздями) по вызовам `op.*`. Вход: путь до директории или список файлов; выход: код возврата 0 при чистоте, ≠0 при нарушениях с текстовым отчётом в stderr по формату `FILE:LINE — RULE — MESSAGE`.

**Шесть обязательных проверок (строго по ADR 0013 §«Запрещено в одном шаге»):**

| № | Правило | Что искать в AST |
|---|---|---|
| 1 | Запрет `DROP COLUMN` | Вызов `op.drop_column(...)` |
| 2 | Запрет `RENAME COLUMN` | `op.alter_column(..., new_column_name=...)` с изменённым именем |
| 3 | Запрет `RENAME TABLE` | `op.rename_table(...)` |
| 4 | Запрет `ALTER COLUMN ... NOT NULL` без `server_default` | `op.alter_column(..., nullable=False, ...)` — и в том же вызове нет `server_default=` с непустым значением |
| 5 | Запрет `DROP TABLE` | `op.drop_table(...)` |
| 6 | Запрет изменения типа колонки с потерей данных | `op.alter_column(..., type_=...)` — детекция по списку «сужающих» переходов: `Integer→String` без `using`, `String(N)→String(M<N)`, `Numeric(p,s)→Numeric(p',s'<s)`, `Text→String(N)`. Всё, что не в whitelist расширения, — ошибка |

**Исключения (по ADR 0013 §«Исключения из правил»):** если в docstring файла миграции встречается маркер `# migration-exception: <правило> — <обоснование>`, конкретное правило для этого файла подавляется. Линтер печатает warning, но не fail-ит.

**Acceptance:**
- файл `backend/tools/lint_migrations.py` создан, импортируется как модуль и работает как CLI (`python -m tools.lint_migrations backend/alembic/versions/`);
- все 6 правил реализованы и независимо включаемы (`--rule=drop_column` и т. п. — nice-to-have, но не обязательно в PR #1);
- на текущих 8 миграциях линтер возвращает 0 (они все legal);
- `ruff check backend/tools/` чисто;
- тайп-хинты на публичных функциях (см. §«Стандарты качества» регламента).

**Acceptance:**

- визуально проходит на текущем состоянии main (0 нарушений).

### Пункт 3. CI-шаг `round-trip`

**Что:** отдельный job `round-trip` в том же `ci.yml`, поднимает Postgres так же, как существующий `test`-job, и выполняет:

```
python -m alembic upgrade head
python -m alembic downgrade -1
python -m alembic upgrade head
```

Любой ненулевой код — job падает. Переменные окружения те же, что у `test`-job (`TEST_DATABASE_URL`, `JWT_SECRET_KEY`), с генерацией секретов через GitHub Actions secrets или `secrets.token_urlsafe` в CI-шаге (литералов — нет, см. пункт «Секреты и тесты» в CLAUDE.md).

**Acceptance:**
- job `round-trip` добавлен, блокирующий;
- **триггер: только `pull_request`** (не `push` в main) — решение Координатора от 2026-04-17 для экономии минут GitHub Actions. На push в main достаточно `test`-job, а round-trip мигрций важен именно при ревью PR;
- все три команды отрабатывают на текущих 8 миграциях без ошибок;
- job независим от `test`-job (падение одного не блокирует другой), но оба — required checks для merge PR.

### Пункт 8. `op.execute` — warning-проверка (7-я проверка линтера)

**Что:** добавить в `lint_migrations.py` седьмую проверку, которая ищет вызовы `op.execute(...)` с произвольным SQL. Это **warning**, не error — линтер печатает сообщение, но exit-code остаётся 0, если это единственное нарушение.

**Обоснование:** без этой проверки разработчик может обойти все 6 правил, написав `op.execute("ALTER TABLE ... DROP COLUMN ...")`. ADR 0013 не покрывает `raw SQL`. Warning требует, чтобы автор миграции явно отметил свой выбор маркером `# migration-exception: op_execute — <обоснование>` в docstring или комментарии рядом с вызовом.

**Правило подавления:** если непосредственно перед вызовом `op.execute(...)` стоит комментарий `# migration-exception: op_execute — <любой текст>` — warning подавляется. Если комментария нет — warning печатается с форматом `FILE:LINE — op_execute_without_exception — op.execute() requires explicit exception marker`.

**Acceptance:**
- 7-я проверка реализована в `lint_migrations.py`, именуется `op_execute_without_exception`;
- **exit-code не меняется**: даже при наличии таких warning линтер возвращает 0, если нет ошибок по правилам 1–6;
- warning виден в stderr отдельной секцией `WARNINGS:` после секции ошибок (или на отдельной строке с префиксом `WARNING:`);
- в `backend/tests/test_lint_migrations.py` — 2 теста: positive (миграция с `op.execute` без маркера — печатается warning), negative (миграция с `op.execute` и маркером — молчание);
- тест на exit-code: при наличии только warning (без ошибок) линтер возвращает 0;
- подраздел в `docs/agents/departments/backend.md` «Правила для авторов миграций» расширяется абзацем о маркере `# migration-exception: op_execute`.

**Решение Координатора от 2026-04-17:** включить в PR #1 (20 минут работы, закрывает реальную дыру обхода 6 правил). Это пункт из §12.3 предложений Директора, теперь переведён в обязательный скоуп.

## 6. Риски и зависимости

- **Frontend-зависимостей нет.** PR #1 не меняет OpenAPI-схему, не добавляет эндпоинтов, не трогает модели. Frontend-director и frontend-head в этом PR не участвуют.
- **Зависимость от db-director:** только в одном сценарии — если round-trip падает на какой-то из существующих 8 миграций из-за реального дефекта миграции (не линтера). Тогда Head приостанавливает PR, эскалирует Директору, тот принимает решение — либо исправить миграцию (задача db-head), либо задокументировать исключение. В нормальном ходе — зависимости нет.
- **Риск ложных срабатываний линтера на реальных миграциях.** Митигация: smoke-тест из Пункта 4 — линтер на реальных 8 миграциях должен вернуть 0. Если возвращает ≠0 — баг линтера, не миграций.
- **Риск хрупкости AST-парсинга.** Alembic-ревизии пишутся по шаблону, но `op.alter_column` имеет много форм вызова (keyword vs positional). Митигация: фикстурные тесты покрывают разные формы вызова (positional, keyword, mixed).
- **Риск конфликта с параллельным PR #2 (RBAC).** PR #2 добавит новую миграцию `role_permissions`. Если PR #2 уходит в работу до мержа PR #1 — его миграция должна пройти линтер. Митигация: PR #1 обязан замержиться первым; backend-head координирует с Директором.

## 8. Ревью-маршрут

1. **backend-dev → backend-head.** Head делает ревью уровня файлов: каждый файл, каждое правило линтера, каждая фикстура. Проверяет чек-лист самопроверки отдела.
2. **backend-head → review-head → reviewer.** Reviewer проверяет на соответствие CLAUDE.md и ADR 0013. Особое внимание: не попадают ли секреты, не поломан ли существующий CI, корректен ли parametrize в round-trip.
3. **Reviewer approve → backend-head → backend-director.** Я принимаю работу на уровне DoD: состав PR, скриншоты/логи CI, логи round-trip.
4. **Backend-director approve → Координатору.** Координатор делает git commit — правило «Reviewer — до git commit» (CLAUDE.md §«Процесс»).

## 9. Что Head возвращает Директору на приёмку

Head оформляет отчёт одним сообщением (не файлом, не отдельным документом) со следующими разделами:

1. **Состав PR.** Список всех изменённых/созданных файлов с путями.
2. **Результаты тестов.** Вывод `pytest backend/tests/test_lint_migrations.py backend/tests/test_round_trip.py -q --tb=short` (копипаст последних строк).
3. **Результаты round-trip.** Вывод трёх команд `alembic upgrade head && alembic downgrade -1 && alembic upgrade head` (успех/ошибки).
4. **Результаты CI.** Ссылка на pull request (если уже открыт) или логи `gh run list` / снимок экрана с тремя зелёными jobs.
5. **Замечания ревьюеров и их закрытие.** Краткая сводка (кто какие P0/P1/P2 дал, как закрыто, с ссылками на коммиты исправлений).
6. **Отклонения от брифа.** Если Head принял какое-то решение, не описанное в брифе (например, выбрал subprocess вместо `alembic.command` API в round-trip-тесте) — фиксируется явно, с обоснованием.
7. **Метрики.** Время на задачу (план 3–4 дня vs факт), количество раундов ревью, наличие request-changes.
