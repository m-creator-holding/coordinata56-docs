## Ключевые решения, принятые в подготовке

**Архитектура (ADR 0004–0007):**
- Router → Service → Repository (3 слоя, тестируемо без HTTP и БД).
- Формат ошибок — свой JSON `{error: {code, message, details}}` с 7 фиксированными кодами.
- Пагинация — offset/limit + конверт `{items, total, offset, limit}`, лимит 200.
- Аудит — явный `audit_service.log()` в service-слое, diff `before/after`, маскировка секретов через Read-схемы.

**Скоуп разбит на 3 батча** (A → B → C, строгий порядок):
- **A** — Каталог и справочники (Project, Stage, HouseType, OptionCatalog, House, HouseConfiguration, …). 3–4 дня.
- **B** — Финансы-план (BudgetCategory, BudgetPlan + bulk upload). 2–3 дня.
- **C** — Финансы-факт (Contractor, Contract, Payment, MaterialPurchase). 3–4 дня.

**Процесс** (из v1.3): reviewer на staged diff до коммита; отклонения от ADR — стоп и запрос к Координатору; секреты не литералом нигде; assert detail в anti-enumeration тестах; границы файлов при параллельном запуске субагентов.

## Открытые вопросы — требуют решения до старта кода

Analyst выделил 3 блокирующих вопроса:

### Q12 ⛔ (Батч C) — Механизм иммутабельности Payment
Модель `Payment` не имеет поля «утверждено». Скоуп обещает: approved-платёж нельзя менять. Нужно выбрать механизм:
- (a) поле `is_approved: bool` + timestamp
- (b) `PaymentStatus` enum (draft / pending / approved / rejected)
- (c) отдельный эндпоинт `POST /payments/{id}/approve`

**Моя рекомендация**: (b) + (c) — enum для состояния, отдельный эндпоинт для перехода. Расширяется, аудитируется.

### Q9 ⚠️ (Батч B) — Bulk-загрузка бюджета: upsert или insert?
Если accountant повторно загружает план — перезаписываем старые строки или получаем дубли и ошибку?
- (a) upsert по ключу (проект, статья, стадия, дом) — перезапись, «последняя загрузка побеждает»
- (b) insert only, конфликт → 409 — нужна явная очистка перед повторной загрузкой
- (c) версионирование плана (план v1, v2, v3) — сложно, но аудитируемо

**Моя рекомендация**: (a) на MVP. Версионирование — после MVP.

### Q6 ⚠️ (Батч A) — Смена стадии: пропуск или последовательность?
Можно ли перевести дом с 1-й стадии сразу на 5-ю (в реальности бывает — стадии идут нелинейно)?
- (a) только соседние стадии (строго +1)
- (b) любой переход вперёд (можно 1→5)
- (c) любой переход, включая назад (откат)

**Моя рекомендация**: (b) — реальная стройка не линейна, но откаты маловероятны и подозрительны.

### Q1 (желательно прояснить до Батча A) — Bulk создания домов?
85 домов. Создавать по одному через UI (долго) или bulk-эндпоинт `POST /houses/bulk`?
**Моя рекомендация**: оба. Bulk на этапе первичной загрузки, единичное — для редких правок.
