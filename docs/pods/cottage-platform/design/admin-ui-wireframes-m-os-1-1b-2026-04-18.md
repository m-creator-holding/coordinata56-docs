## Обзор и принципы проектирования

### Паттерн Developer Mode (по образцу Odoo Studio)

Каждая категория Admin UI строится по одной UX-модели:

1. **Список** — таблица конфигурационных записей с метаданными версии
2. **Детальная форма** — редактирование, с версионным журналом справа
3. **Управление версиями** — просмотр истории, сравнение, откат

Ключевое правило: изменение вступает в силу **сразу после сохранения** — без перезапуска и деплоя (Принцип 10 M-OS).

### Персонажи

| Персонаж | Роль в системе | Доступные действия |
|---|---|---|
| **Владелец** (Мартин) | `owner` — главный admin | Все: создать / изменить / откатить / удалить |
| **Бухгалтер** | `accountant` — readonly admin | Только просмотр (R) всех объектов + экспорт отчётов (Exp). Ни одно поле не редактирует. На странице лимитов — все поля disabled, banner «У вас режим просмотра. Для изменений — обратитесь к директору.» |

### Версионирование (Решение 5 + Решение 6 — BPM B2)

- Каждое сохранение создаёт новую запись в `configuration_entities` (version +1, status `active`)
- Предыдущая версия переходит в статус `archived`
- Откат = создание новой версии с данными из выбранной archived-версии (не мутация)
- Для BPM-процессов: изменение применяется и к **запущенным экземплярам** по правилам совместимости (Решение 6)

#### Поведение дерева групп

- **Expand/collapse** — плавная анимация (CSS transition height). Состояние каждой группы сохраняется в localStorage сессии.
- **Badge `[N]`** на заголовке группы — количество сущностей внутри. При поиске: число совпавших / общее.
- **Чекбокс «все в группе»** (header-level) — bulk-назначение. Примеры:
  - `[✓ все в группе]` — включает **R=1** на все сущности группы (минимальный bulk). Диалог уточняет: «Выбрать действия для всей группы: [C][R][U][D][Exp][Appr][Arc]» — пользователь выбирает какие именно.
  - Если хоть один чекбокс снят в группе — badge показывает промежуточное состояние (`−`, indeterminate).
- **Поиск по названию сущности** (верхняя панель):
  - Группы с совпавшими сущностями раскрываются автоматически.
  - Группы без совпадений схлопываются и затеняются (opacity 0.4).
  - При очистке поиска — восстанавливается предыдущее состояние expand/collapse.
- **Appr (Согласовать)** — при попытке снять: `AlertDialog` «Это действие используется в BPM-процессах. Снятие немедленно повлияет на запущенные маршруты. Продолжить?» (отдельный для каждой ячейки).
- **Arc (Архивировать)** — отдельное от D. Без Arc пользователь не может переводить записи в `archived`.
- Системные объекты из группы «Настройки» — у роли `owner` всегда `[✓]` и нередактируемы (поля disabled, tooltip «Системное право — нельзя снять»).
- Изменение применяется немедленно для всех активных сессий с этой ролью (через сброс permission-cache).

**Компоненты shadcn/ui**: `Collapsible` / `Accordion` (группы-папки с анимацией), `Checkbox` (ячейки матрицы + bulk header), `Table`, `Input` + `Search` icon (поиск сущности), `Badge`, `Button`, `Textarea`, `AlertDialog` (Appr), `Tooltip` (системные права), `Card` + `ScrollArea` (правая панель).

**API-паттерн матрицы**: при сохранении — один `PATCH /roles/{id}/permissions` с полной матрицей (`{entity: string, actions: {C,R,U,D,Exp,Appr,Arc}: bool}[]`). Не построчный PATCH — иначе N запросов на N строк. Backend атомарно заменяет всю матрицу роли.

### 3.2а Версионирование бизнес-правил

**Решение Владельца 2026-04-19 msg 1480 (Q5)**: хранение снапшота правила на момент согласования — в **отдельной таблице `payment_rule_snapshots`**, не в колонке таблицы `Payment`. Второй вариант гибче: один платёж может быть привязан к нескольким правилам (например, лимит перерасхода + порог подписи директора).

**Модель версий**: каждое изменение лимита создаёт новую строку в `configuration_entities` с инкрементальным суффиксом `rule_v1`, `rule_v2`, `rule_v3` и т.д. Предыдущая версия переходит в статус `archived`.

**Архитектурный паттерн `payment_rule_snapshots`**:

```
payment_rule_snapshots
───────────────────────────────────────────────
id              UUID PK
payment_id      FK → payments.id
rule_key        VARCHAR  (напр. "payment_overrun_limit_pct")
rule_version    VARCHAR  (напр. "rule_v1")
snapshot_id     INTEGER  (номер снапшота, напр. #4521)
rule_json       JSONB    (полный JSON правила на момент одобрения)
applied_at      TIMESTAMP
applied_by      FK → users.id
```

Снапшот создаётся BPM-движком в момент перехода платежа в статус `approved` — в той же транзакции. При каждом следующем платеже создаётся новая строка (не мутируется существующая).

**Универсальный паттерн**: по аналогии создаются `contract_rule_snapshots`, `stage_rule_snapshots` и т.д. при появлении соответствующих BPM-процессов. Паттерн передаётся `backend-director` для ADR (Future: отдельная таблица для каждого бизнес-объекта с BPM-одобрением).

**Detail View платежа — badge версии правила**:

```

│                                                                             │
│  Применённые правила:                                                       │
│  [Одобрено по rule_v1 (snap #4521) →]   ← кликабельный badge               │
```

## Общие UX-паттерны всех категорий

### Toast-уведомления

| Событие | Текст | Тип |
|---|---|---|
| Запись сохранена | «[Объект] обновлён. Версия vN активна.» | success |
| Ошибка валидации | «Заполните обязательные поля: [список]» | destructive |
| Откат выполнен | «Откат к версии vN выполнен. Текущая версия: vM.» | success |
| BPM-предупреждение | «Изменение применится к N активным процессам.» | warning |

### Dialog подтверждения откатов

Любой откат — через `AlertDialog` (shadcn/ui), не через простой клик. Текст диалога включает:
- Название объекта
- Версию, к которой откатываем
- Дату и автора той версии
- Количество затронутых пользователей / активных BPM-экземпляров
- Кнопки: «Отмена» и «Да, откатить»

### Запись аудита

Каждое действие в Admin UI (создание / изменение / откат) пишется в `audit_log` через `audit_service.log()` в той же транзакции (ADR 0007). В поле `action` — тип операции, в `metadata` — JSON с diff предыдущей и новой версии.
