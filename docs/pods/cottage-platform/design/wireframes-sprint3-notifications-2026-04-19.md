## Принятые решения

| # | Точка выбора | Выбрано | Обоснование |
|---|---|---|---|
| N1 | Notification Center: отдельная страница /notifications vs Drawer/Popover | Отдельная страница /notifications | BPM approvals требуют полноценных действий (форма + кнопки) — Popover слишком мал; страница даёт место для деталей |
| N2 | Иконка в топбаре: колокольчик + Badge-счётчик | Badge с числом непрочитанных на иконке Bell | Стандартный UX-паттерн; цифра исчезает при «Отметить все прочитанными» |
| N3 | Approval actions: inline в списке vs отдельная страница | Отдельная страница /notifications/{id} для approvals | Форма комментария + 2 кнопки + AlertDialog — слишком много для inline-карточки |
| N4 | Матрица каналов: отдельный /admin/notifications vs вкладка /admin/system-config | Отдельный роут /admin/notifications | Логически отдельная функциональность; sidebar уже содержит раздел «Конструктор»; Telegram-настройка требует своего пространства |
| N5 | Telegram в топбаре: прямая ссылка на Telegram-чат | Не добавляем | Дублирует Telegram-бот без добавления ценности; пользователь сам знает чат; UX-сложность без пользы на MVP |
| N6 | Email/SMS каналы: hidden vs disabled | Disabled с tooltip | Пользователь должен знать, что каналы БУДУТ — disabled + tooltip «доступно после production-gate» лучше чем отсутствие информации |

## Role Matrix

| Роль | /notifications (центр) | /notifications/{id} (детали) | /admin/notifications (настройка) |
|---|---|---|---|
| owner | Все уведомления. Может approve/reject BPM. | Полный просмотр + approve/reject | Полный доступ. Редактирование матрицы. |
| director | Уведомления своей компании. Может approve/reject где назначен исполнителем. | Просмотр + approve/reject (если назначен) | Только просмотр. Баннер readonly. |
| accountant | Только финансовые уведомления (BPM платежи). | Просмотр, нет approve/reject (нет прав BPM approval) | 403 Forbidden |
| foreman | Только уведомления своих задач и актов. | Просмотр, нет approve/reject | 403 Forbidden |

## Навигационная карта

```
ТОПБАР (глобальный)
  └── [Bell иконка с Badge-числом]  → /notifications    (для всех авторизованных ролей)

SIDEBAR — раздел КОНСТРУКТОР (новый, M-OS-1.2)
  ├── Шаблоны BPM          /admin/bpm/definitions
  ├── ● Каналы уведомлений /admin/notifications      ← Экран 3 (Admin)
  └── Шаблоны документов   /admin/document-templates

ПЕРЕХОДЫ МЕЖДУ ЭКРАНАМИ:
/notifications
  ├── клик на карточку BPM Approval              → /notifications/{id}  (Экран 2)
  └── информационные уведомления                 → inline expand (Accordion)

/notifications/{id}
  ├── [← Назад]                                  → /notifications
  ├── [Одобрить] / [Отклонить] + AlertDialog     → POST /api/v1/bpm/instances/{id}/approve|reject
  └── [Перейти к объекту] (ссылка)               → /payments/{id} или /contracts/{id}

/admin/notifications
  ├── Блок 3: баннер «Настроить Telegram»        → /admin/integrations
  └── сохранение матрицы                         → PATCH /api/v1/notification-settings
```

## Экран 1. `/notifications` — Центр уведомлений

### Persona и сценарий

**Персона A — Владелец (owner):**
Кликает на иконку Bell в топбаре — видит Badge «3». Открывает /notifications. Видит таб «Ожидают действия» с 2 BPM approvals (платёж и договор). Кликает на approval платежа — переходит на /notifications/{id} для принятия решения.

**Персона B — Бухгалтер (accountant):**
Получает уведомление «Платёж #234 отклонён». Открывает /notifications, фильтрует таб «Все», видит карточку с деталями. Информационное уведомление — клик раскрывает детали inline (Accordion).

**Персона C — Прораб (foreman):**
Видит в топбаре Bell с «1». Открывает /notifications, видит уведомление «Акт №12 ожидает вашей подписи». Переходит на детальное уведомление.

### Поля формы / структура карточки уведомления

**Карточка типа BPM Approval:**

| Элемент | Тип UI | Примечание |
|---|---|---|
| Иконка типа | иконка ⚙ (Settings из lucide) | BPM = Settings; System = AlertCircle; Info = Info |
| Подпись типа | текст «BPM СОГЛАСОВАНИЕ» | uppercase, muted, small |
| Badge статуса | Badge «Требует действия» — variant=destructive | «Прочитано» — variant=secondary; «Информационное» — variant=outline |
| Заголовок | текст жирный | Название BPM события |
| Краткие детали | текст muted | Компания · сумма / объект · инициатор |
| Timestamp | текст muted right-align | Относительное («2 ч назад») если < суток; дата если > суток |
| Кнопка действия | Button variant=default | «Перейти к согласованию →» только для BPM approvals |

**Карточка типа Информационное:**

- Нет кнопки «Перейти»
- Клик на карточку → Accordion раскрывает inline-детали
- Если прочитано → фон карточки без выделения (bg-background), если непрочитано → bg-accent/10

**Компоненты shadcn/ui:** `Tabs`, `TabsList`, `TabsTrigger`, `TabsContent`, `Select` (фильтры), `Button`, `Card`, `CardContent`, `Badge`, `Accordion`, `AccordionItem`, `AccordionTrigger`, `AccordionContent`.

## Экран 2. `/notifications/{id}` — Детальное уведомление (BPM Approval)

### Persona и сценарий

**Персона A — Владелец:**
Открывает согласование платежа #456 (250 000 ₽). Читает детали: сумма превышает порог подписи директора. Пишет комментарий «Одобрено — в рамках бюджета Q2». Нажимает [Одобрить]. AlertDialog: «Подтвердить одобрение платежа #456 на сумму 250 000 ₽?» → [Да, одобрить] → Toast success → уведомление уходит в «Архив».

**Персона B — Директор:**
Открывает то же уведомление — видит те же данные. Если директор назначен исполнителем шага — кнопки approve/reject активны. Если нет — видит только просмотр.

**Персона C — Бухгалтер:**
Получает уведомление о том же платеже (информационное: «Платёж отправлен на согласование»). Открывает /notifications/{id} — видит данные но кнопок approve/reject нет (нет прав).

**Секция «Ваше решение» (conditional):**

| Поле | Тип UI | Условие |
|---|---|---|
| Комментарий | Textarea | Всегда в секции |
| [Отклонить] | Button variant=destructive | Только если пользователь — исполнитель шага |
| [Одобрить] | Button variant=default | Только если пользователь — исполнитель шага |

**Валидация:**
- Кнопка «Отклонить» disabled если поле «Комментарий» пустое
- Кнопка «Одобрить» активна даже без комментария (комментарий опционален при одобрении)

### Поля формы / структура матрицы

**Блок 1 — Каталог каналов:**

| Канал | Состояние | Тип UI |
|---|---|---|
| Telegram | Активен | Card: Badge «Активен» (success), текст chat_id, Button «Настроить» |
| Email (SMTP) | Недоступно | Card: Badge «Недоступно» (secondary), текст «production-gate», Button «Настроить» disabled |
| SMS | Недоступно | Card: Badge «Недоступно» (secondary), текст «production-gate», Button «Настроить» disabled |

**Блок 2 — Матрица событий:**

| Элемент | Тип UI |
|---|---|
| Группировка (Финансы/Документы/Процессы BPM/Система) | Table section header, полная ширина, bg-muted |
| Telegram Switch | Switch: активный (включаем/выключаем); сохраняется по «Сохранить настройки» |
| Email Switch | Switch: disabled=true; Tooltip «Доступно после production-gate» |
| SMS Switch | Switch: disabled=true; Tooltip «Доступно после production-gate» |

**Компоненты shadcn/ui:** `Table`, `TableHeader`, `TableRow`, `TableCell`, `Switch`, `Tooltip`, `TooltipTrigger`, `TooltipContent`, `Card`, `CardContent`, `Badge`, `Button`, `Alert`, `Separator`.

**Сохранение матрицы:** PATCH /api/v1/notification-settings отправляет полный объект матрицы — все события и их Switch-состояния для каждого канала. Одна кнопка «Сохранить настройки» для всей матрицы.

### Состояния UI

| Состояние | Описание |
|---|---|
| Loading | Skeleton матрицы (8 строк) + skeleton карточек каналов |
| Сохранение | Кнопка «Сохранить настройки» → spinner + disabled. Toast success: «Настройки уведомлений сохранены.» |
| Ошибка сохранения | Toast destructive: «Не удалось сохранить настройки.» Кнопка возвращается в активное состояние. |
| Тест-сообщение Telegram | Кнопка «Отправить тест» → spinner → Toast success: «Тест-сообщение отправлено в Telegram» / Toast destructive при ошибке. |
| Telegram не настроен | Блок 3 показывает Alert warning с CTA «Перейти к реестру интеграций». Telegram Switch в матрице — disabled + tooltip «Сначала настройте Telegram». |
