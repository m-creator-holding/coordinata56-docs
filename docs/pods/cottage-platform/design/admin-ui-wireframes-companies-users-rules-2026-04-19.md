## Принципы этого батча

- **Стиль**: tree слева (группировка по направлениям бизнеса) + table/form справа — единый с v0.3.
- **Header**: «M-OS | Коттеджи» (Решение 10) в топбаре каждого экрана.
- **Версионирование**: каждое сохранение = новая запись в `configuration_entities`; откат = новая версия с данными archived-версии.
- **Права**: Владелец — полный CRUD; Бухгалтер — readonly с баннером во всех Admin UI разделах.
- **Живые интеграции**: не проектируются (CODE_OF_LAWS ст. 45а). Банковские карточки — только реквизиты р/с, не API-интеграция.

## Страница 1. `/admin/companies` — Юрлица холдинга

### 1.0 Обзор

5 компаний Владельца организованы по бизнес-направлениям. Левое дерево группирует компании по сегменту. Правая панель — список или детальная форма выбранной компании.

**7 полей company_settings (Решение Q7)**:

| Поле | Тип | Обязательное |
|---|---|---|
| `payment_overrun_limit_pct` | Number (%) | Да |
| `approval_amount_threshold` | Number (руб.) | Да |
| `bank_primary_account` | String (р/с) | Нет |
| `vat_regime` | Enum: ОСН / УСН 6% / УСН 15% / Патент | Да |
| `company_director_id` | FK → users | Да |
| `business_segment` | Enum: Карьер / Металлообработка / АЗС / МКД / Коттеджи | Да |
| `default_currency` | Enum: RUB | Да (сейчас всегда RUB) |

**Права**:
- Владелец: все поля редактируемы. Кнопка «Откатить» в правой панели — видна.
- Бухгалтер: все поля `disabled` (фон `muted`), баннер ℹ, кнопки «Сохранить» и «Откатить» — скрыты (не disabled, а `display:none`). Вкладка «Аудит» — доступна (только просмотр).

**Поведение «Сохранить»**:
1. Валидация обязательных полей (`*`) + комментария.
2. Создаётся запись `configuration_entities` version +1.
3. Предыдущая версия → статус `archived`.
4. Toast: «Юрлицо обновлено. Версия v4 активна.»
5. Правая панель истории обновляется без перезагрузки.

### 1.5 User Flow — Юрлица

```
mermaid
flowchart TD
    A[Открывает /admin/companies] --> B{Роль?}
    B -- Владелец --> C[Видит дерево + список 5 компаний]
    B -- Бухгалтер --> D[Видит только своё юрлицо, кнопки CRUD скрыты]

    E -- Клик на компанию --> F[Detail View]

    F --> I{Роль?}
    I -- Владелец --> J[Редактирует поля]
    I -- Бухгалтер --> K[Только просмотр + баннер]

    J --> L[Сохранить]
    L --> M[Валидация OK?]
    M -- Нет --> N[Toast error: список незаполненных полей]
    M -- Да --> O[version +1 в configuration_entities]
    O --> P[Toast success: Версия vN активна]

    F --> Q[История версий — кнопка Откатить]
    Q --> R[AlertDialog: объект + версия + дата + N пользователей]
    R -- Да, откатить --> S[Новая версия с данными archived]
    S --> P

    H --> T{Подтверждение}
    T -- Да --> U[company.status = archived]
    T -- Нет --> C
```

### 1.6 Acceptance Criteria — `/admin/companies`

| # | Критерий | Приоритет |
|---|---|---|
| AC-C1 | Дерево компаний слева группирует по полю `business_segment`. При клике на группу — разворачивает список. При клике на компанию — правая панель показывает Detail View. | P0 |
| AC-C2 | Бухгалтер видит в списке только юрлицо своей компании (row-level RBAC через `user_company_role.company_id`). Кнопка «+ Добавить» и иконки `...` — скрыты (`display:none`). | P0 |
| AC-C3 | Detail View отображает все 7 полей `company_settings` с корректными типами UI (RadioGroup для НДС-режима, `disabled` для `default_currency=RUB`). | P0 |
| AC-C4 | Кнопка «Сохранить» — недоступна пока поле «Комментарий к изменению» пустое. При сохранении создаётся запись `configuration_entities` с `version+1`, предыдущая — переходит в `archived`. | P0 |
| AC-C5 | Правая панель «История версий» обновляется без перезагрузки страницы после каждого сохранения. Кнопка «Откатить» — только у Владельца; открывает `AlertDialog` с указанием версии, даты и автора. | P1 |
| AC-C6 | Баннер «У вас режим просмотра» отображается для роли `accountant` на всех вкладках Detail View. Все `Input`, `Select`, `RadioGroup` — `disabled`. Кнопки «Сохранить» и «Откатить» — `display:none`. | P0 |
| AC-C7 | Деактивация юрлица (мягкое архивирование) запрашивает `AlertDialog` с текстом: «Юрлицо будет помечено архивным. Новые заявки создать нельзя. Старые видны только для чтения. Удаление данных — отдельная процедура через юриста (7 лет хранения ПД).» После подтверждения — `company.deactivated = true`, строка в списке помечается `Badge` variant=`secondary` «Архивировано». В Detail View отображается баннер `Alert` variant=`warning` «Юрлицо архивировано DD.MM.YYYY». Физическое удаление записи — недоступно через Admin UI. | P1 |
| AC-C8 | Маршрут `/admin/companies/new` возвращает `403` для роли `accountant`. Форма создания доступна только Владельцу. | P0 |

```

## Страница 2. `/admin/users` — Пользователи

### 2.0 Обзор

CRUD пользователей + назначение роли в каждой компании через матрицу `user_company_role`. Фильтрация по роли и компании. Одна роль на компанию для каждого пользователя; один и тот же пользователь может иметь разные роли в разных компаниях.

**Системный баннер (отображается вверху страницы для всех ролей)**:

```
ℹ Один пользователь — один логин, одна роль в каждой компании.
  Пользователь может работать в нескольких компаниях с разными ролями.
```

---

### 2.1 Layout-схема

```
┌────────────────────────────────────────────────────────────────────────────────────┐
│ M-OS | Коттеджи                                           [Мартин ▾]  [?]  [⚙]   │
├──────────────────────────────────────────────────────────────────────────────────  │
│  SIDEBAR — стандартный (см. §0 навигационная карта)                               │
├────────────────────────────────────────────────────────────────────────────────────┤
│  MAIN CONTENT (без дерева — пользователи не группируются по сегменту)             │
│                                                                                    │
│  Пользователи                                            [+ Добавить пользователя] │
│  ─────────────────────────────────────────────────────────────────────────────    │
│                                                                                    │
│  Поиск: [________________________]  Роль: [Все ▾]  Компания: [Все ▾]              │

### 2.7 Acceptance Criteria — `/admin/users`

| # | Критерий | Приоритет |
|---|---|---|
| AC-U1 | Фильтр «Роль» фильтрует пользователей по `user_company_role.role_id`. Фильтр «Компания» — по `user_company_role.company_id`. Оба фильтра работают одновременно. | P0 |
| AC-U2 | Колонка «Компании (роли)» в таблице показывает все привязки пользователя (`user_company_role`). Если привязок нет — прочерк `—`. | P0 |
| AC-U3 | Вкладка «Привязки» позволяет изменить роль пользователя в конкретной компании через inline `Select` в строке таблицы. Уникальный индекс `(user_id, company_id)` — одна запись на компанию. | P0 |
| AC-U4 | Dialog «Добавить компанию» — одиночный выбор компании. В списке отсутствуют компании, к которым привязка уже есть. При дублировании — inline-ошибка. | P1 |
| AC-U5 | Деактивация пользователя (`Switch` → off) запрашивает `AlertDialog` и при подтверждении немедленно инвалидирует все активные сессии пользователя (endpoint: `POST /users/{id}/deactivate`). | P0 |
| AC-U6 | Удаление привязки компания+роль запрашивает `AlertDialog` с указанием компании и роли, которые теряются. | P1 |
| AC-U7 | Бухгалтер видит в таблице только себя. Маршруты `/admin/users/new` и `/admin/users/{id}/edit` для чужих пользователей возвращают `403`. | P0 |
| AC-U8 | Поле «Email (логин — неизменяем)» — `Input` с `disabled`, иконка замка. Отдельное поле «Контактный email» — редактируемое через кнопку `✏ Редактировать`. Логин (email при создании) хранится в `users.email`; контактный email — в `users.contact_email`. | P1 |

### 3.5 User Flow — Бизнес-правила

```
mermaid
flowchart TD

    G --> V[Правая панель: кнопка Откатить на archived-версии]

    W -- Да, откатить --> X[Новая версия с данными из archived]
    X --> S

    D --> G
    G -- Бухгалтер --> Y[Detail View: все disabled + баннер]
    Y --> T
```

### 3.6 Acceptance Criteria — `/admin/rules`

| # | Критерий | Приоритет |
|---|---|---|
| AC-R1 | Дерево слева группирует правила по полю `rule_category` (Финансы / Кадры / Процессы). При клике на категорию — таблица справа фильтруется. При клике на конкретное правило в дереве — сразу открывается Detail View. | P0 |
| AC-R2 | При наличии N > 0 активных BPM-заявок, связанных с данным правилом, отображается `Alert` variant=`warning` с точным числом N. При N = 0 — Alert не показывается. Число N запрашивается с бэкенда при открытии Detail View. | P0 |
| AC-R3 | Поле «Комментарий к изменению» — обязательное. Кнопка «Сохранить» неактивна (`disabled`), пока поле пустое. | P0 |
| AC-R4 | Каждое сохранение создаёт новую версию `rule_vN+1` в `configuration_entities`. Изменения применяются только к новым заявкам. Одобренные хранят ссылку на версию в `payment_rule_snapshots`. | P0 |
| AC-R5 | Кнопка «Просмотр» на archived-версии в панели истории открывает `Sheet` с полным snapshot JSON (readonly). JSON отображается как форматированный блок `<pre><code>`. | P1 |
| AC-R6 | Кнопка «Откатить» открывает `AlertDialog` с данными: название правила, номер версии, дата, автор, количество N активных заявок. После подтверждения создаётся новая версия с данными archived-записи. | P1 |
| AC-R7 | Бухгалтер: все поля `disabled`, баннер ℹ, кнопки «Сохранить» и «Откатить» — `display:none`. Кнопка «Просмотр» (snapshot JSON) — видна и работает. | P0 |
| AC-R8 | Откат и сохранение пишут запись в `audit_log` через `audit_service.log()` в той же транзакции (ADR 0007). В `metadata` — JSON diff предыдущей и новой версии. | P0 |
| AC-R9 | Кнопка «+ Создать тип правила» отображается только при `role.code === 'owner'`; для `accountant`, `director`, `foreman` — `display:none`. Роли `director`/`foreman` видят в Detail View заметку «Вы можете изменить значение правила. Создание нового типа — только Владелец.» Поле `rule_key` для них — `disabled`. | P0 |

### 3.7 API endpoints

```
GET    /api/v1/rules                             — список правил (?category=, ?company_id=, поиск, пагинация)
GET    /api/v1/rules/{id}                        — детальная карточка правила
POST   /api/v1/rules                             — создание (только owner)
PATCH  /api/v1/rules/{id}                        — обновление + создание новой версии
GET    /api/v1/rules/{id}/versions               — история версий configuration_entities
GET    /api/v1/rules/{id}/versions/{version_id}  — snapshot конкретной версии (полный JSON)
POST   /api/v1/rules/{id}/rollback               — откат (тело: {version_id})
GET    /api/v1/rules/{id}/active-requests-count  — количество активных BPM-заявок для этого правила
```

## Общие UX-паттерны трёх страниц

### Toast-уведомления

| Событие | Текст | Тип |
|---|---|---|
| Запись сохранена | «[Объект] обновлён. Версия vN активна.» | success |
| Ошибка валидации | «Заполните обязательные поля: [список]» | destructive |
| Откат выполнен | «Откат к версии vN выполнен. Текущая версия: vM.» | success |
| BPM-предупреждение | «Изменение применится к N активным процессам.» | warning |
| Архивирование | «[Объект] архивирован.» | success |
| Деактивация пользователя | «Пользователь деактивирован. Сессии завершены.» | success |

### AlertDialog подтверждения (обязателен для)

- Откат любой версии (правило, юрлицо, правило RBAC)
- Архивирование юрлица
- Удаление привязки user_company_role
- Деактивация пользователя

Каждый AlertDialog содержит: объект воздействия + количество затронутых пользователей / заявок. Кнопки: «Отмена» и «Да, [действие]».

### Баннер readonly (бухгалтер)

```
shadcn/ui Alert variant=info
  AlertDescription: «У вас режим просмотра. Для изменений — обратитесь к директору.»
```

Отображается вверху любой Detail View для роли `accountant`. Позиция: sticky под header формы, над первым полем.

### Аудит (ADR 0007)

Каждое действие в Admin UI (создание / изменение / откат / архивирование / деактивация) пишется в `audit_log` через `audit_service.log()` в той же транзакции. Поле `action` — тип операции, `metadata` — JSON diff.
