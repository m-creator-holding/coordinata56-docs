# Онбординг: что прочитать в первые три дня

**Для кого:** новый сотрудник — бухгалтер, прораб, менеджер по продажам.
**Цель:** за три дня стать уверенным пользователем системы «Координата 56».

Здесь нет ничего технического. Вам не нужно знать программирование. Система — это инструмент, как Excel или 1С. Только проще.

## Для разработчиков: сущности Фазы 3 (Батч C)

Фаза 3 добавила 4 сущности для учёта финансового факта: **Contractor**, **Contract**, **Payment**, **MaterialPurchase**. Все 22 эндпоинта следуют единому паттерну из ADR 0004.

Интерактивная документация всех эндпоинтов — в Swagger: `http://127.0.0.1:8000/docs`.

---

### Contractor — подрядчики

Компания или ИП, выполняющие работы на объекте. Уникальность определяется ИНН.

**Бизнес-правила:**
- ИНН уникален среди активных (не удалённых) подрядчиков. Мягко удалённый подрядчик не блокирует создание нового с тем же ИНН.
- Удалить подрядчика нельзя, если у него есть активные договоры (статус `draft` или `active`).
- Мягкое удаление: запись остаётся в базе, выставляется `deleted_at`.

**Права доступа:**

| Операция | owner | accountant | construction_manager | read_only |
|----------|-------|------------|----------------------|-----------|
| Чтение   | да    | да         | да                   | да        |
| Создание | да    | да         | нет                  | нет       |
| Изменение| да    | да         | нет                  | нет       |
| Удаление | да    | да         | нет                  | нет       |

**Эндпоинты:**

```
GET    /contractors          — список с фильтрами (category, search)
GET    /contractors/{id}     — один подрядчик
POST   /contractors          — создать
PATCH  /contractors/{id}     — обновить
DELETE /contractors/{id}     — мягко удалить
```

**Пример — создать подрядчика через curl:**

```bash
curl -X POST http://127.0.0.1:8000/contractors \
  -H "Authorization: Bearer <токен>" \
  -H "Content-Type: application/json" \
  -d '{
    "short_name": "ООО Стройресурс",
    "inn": "7701234567",
    "kpp": "770101001",
    "category": "Монолит"
  }'
```

Ответ (201 Created):

```json
{
  "id": 1,
  "short_name": "ООО Стройресурс",
  "inn": "7701234567",
  "kpp": "770101001",
  "category": "Монолит",
  "deleted_at": null
}
```

**Пример — список через Swagger:**
Откройте `http://127.0.0.1:8000/docs`, найдите тег `contractors`, нажмите `GET /contractors` → «Try it out» → «Execute».

### Contract — договоры

Договор связывает подрядчика, проект и (опционально) конкретный дом. Номер договора уникален в рамках одного подрядчика.

**Бизнес-правила:**
- Пара `(contractor_id, number)` уникальна среди действующих (не удалённых) договоров.
- Переходы статуса: `draft → active → completed` или `draft → cancelled` или `active → cancelled`. Обратные переходы запрещены (ответ 409).
- Удалить договор нельзя, если по нему есть платежи.
- Поле `house_id` (если указано) должно принадлежать тому же `project_id`.

**Права доступа:**

| Операция | owner | accountant | construction_manager | read_only |
|----------|-------|------------|----------------------|-----------|
| Чтение   | да    | да         | да                   | нет       |
| Создание | да    | да         | нет                  | нет       |
| Изменение| да    | да         | нет                  | нет       |
| Удаление | да    | да         | нет                  | нет       |

Роль `read_only` не имеет доступа к договорам — они содержат коммерческую информацию.

**Эндпоинты:**

```
GET    /contracts            — список с фильтрами (contractor_id, project_id, house_id, status)
GET    /contracts/{id}       — один договор
POST   /contracts            — создать
PATCH  /contracts/{id}       — обновить (включая смену статуса)
DELETE /contracts/{id}       — мягко удалить
```

**Пример — перевести договор в статус «действующий»:**

```bash
curl -X PATCH http://127.0.0.1:8000/contracts/5 \
  -H "Authorization: Bearer <токен>" \
  -H "Content-Type: application/json" \
  -d '{"status": "active"}'
```

Если переход недопустим (например, договор уже `completed`), ответ:

```json
{
  "error": {
    "code": "BUSINESS_RULE_VIOLATION",
    "message": "Недопустимый переход статуса completed → active",
    "details": {}
  }
}
```

### Payment — платежи

Фактический платёж по договору. Жизненный цикл управляется через статусы и специальные действия.

**Жизненный цикл статусов:**

```
draft ──► pending ──► approved
  │                      (иммутабелен)
  └──────────────────► rejected
                         (иммутабелен)
```

Переходы `draft ↔ pending` — через обычный PATCH.
Переходы в `approved` или `rejected` — только через отдельные эндпоинты `/approve` и `/reject`. Эти действия доступны только роли `owner`.

**Бизнес-правила:**
- Платёж в статусе `approved` или `rejected` нельзя изменить или удалить (ошибка `PAYMENT_IMMUTABLE`).
- Физическое удаление разрешено только из статуса `draft`.
- При согласовании система проверяет: сумма всех одобренных платежей по договору + текущий не должна превышать 120% суммы договора. При превышении — ответ 409 `PAYMENT_EXCEEDS_CONTRACT`.
- Договор должен быть в статусе `active` или `completed` для создания платежа.

**Права доступа:**

| Операция           | owner | accountant | construction_manager | read_only |
|--------------------|-------|------------|----------------------|-----------|
| Чтение             | да    | да         | да                   | да        |
| Создание / PATCH   | да    | да         | нет                  | нет       |
| Удаление           | да    | да         | нет                  | нет       |
| Approve / Reject   | да    | нет        | нет                  | нет       |

**Эндпоинты:**

```
GET    /payments               — список с фильтрами (contract_id, status, payment_method, даты)
GET    /payments/{id}          — один платёж
POST   /payments               — создать (статус draft)
PATCH  /payments/{id}          — обновить (draft ↔ pending)
DELETE /payments/{id}          — удалить (только из draft)
POST   /payments/{id}/approve  — согласовать (только owner)
POST   /payments/{id}/reject   — отклонить с причиной (только owner)
```

**Пример — согласовать платёж:**

```bash
curl -X POST http://127.0.0.1:8000/payments/12/approve \
  -H "Authorization: Bearer <токен_владельца>"
```

Тело запроса не нужно. Ответ (200 OK) — объект платежа с `status: "approved"`.

**Пример — отклонить платёж с указанием причины:**

```bash
curl -X POST http://127.0.0.1:8000/payments/12/reject \
  -H "Authorization: Bearer <токен_владельца>" \
  -H "Content-Type: application/json" \
  -d '{"reason": "Неверные реквизиты получателя"}'
```

Поле `reason` обязательно, минимум 3 символа. Ответ — платёж с `status: "rejected"`.

### MaterialPurchase — закупки материалов

Фактическая закупка материала, привязанная к дому и стадии строительства.

**Бизнес-правила:**
- `total_price_cents` вычисляется автоматически: `round(quantity × unit_price_cents)`. Если передать значение вручную, система проверит совпадение с вычисленным — при расхождении ответ 409 `TOTAL_PRICE_MISMATCH`.
- `house_id` должен принадлежать тому же `project_id`.
- Физическое удаление (без мягкого): запись удаляется полностью. Восстановить нельзя.
- `purchased_at` не может быть датой в будущем.

**Права доступа:**

| Операция | owner | accountant | construction_manager | read_only |
|----------|-------|------------|----------------------|-----------|
| Чтение   | да    | да         | да                   | да        |
| Запись   | да    | да         | да                   | нет       |

**Эндпоинты:**

```
GET    /material-purchases          — список с фильтрами (project_id, house_id, stage_id, material_name, даты)
GET    /material-purchases/{id}     — одна запись
POST   /material-purchases          — создать
PATCH  /material-purchases/{id}     — обновить (пересчёт total_price_cents)
DELETE /material-purchases/{id}     — физически удалить
```

**Пример — создать запись о закупке:**

```bash
curl -X POST http://127.0.0.1:8000/material-purchases \
  -H "Authorization: Bearer <токен>" \
  -H "Content-Type: application/json" \
  -d '{
    "project_id": 1,
    "house_id": 7,
    "stage_id": 3,
    "material_name": "Цемент М500, мешок 50 кг",
    "quantity": 40,
    "unit_price_cents": 45000,
    "purchased_at": "2026-04-15T10:00:00"
  }'
```

`total_price_cents` в теле не нужен — система вычислит сама (40 × 45000 = 1 800 000 копеек = 18 000 руб.).

## Для разработчиков: общие паттерны API

Все эндпоинты Фазы 3 следуют единым правилам (ADR 0004–0007).

**Формат ошибок** (ADR 0005):

```json
{
  "error": {
    "code": "CONTRACTOR_INN_DUPLICATE",
    "message": "Подрядчик с таким ИНН уже существует",
    "details": {}
  }
}
```

**Формат списка** (ADR 0006):

```json
{
  "items": [...],
  "total": 42,
  "offset": 0,
  "limit": 20
}
```

**Пагинация:** параметры `offset` (по умолчанию 0) и `limit` (по умолчанию 20, максимум 200).

**Аудит-лог** (ADR 0007): каждое создание, изменение и удаление фиксируется в таблице `audit_log` с diff до/после. Для действий смены статуса (approve/reject) структура записи:

```json
{
  "transition": "approved",
  "from_status": "pending",
  "reason": "..."
}
```

**Коды HTTP-ответов:**
- `201` — успешное создание
- `200` — успешное чтение/обновление
- `204` — успешное удаление (тело пустое)
- `403` — недостаточно прав
- `404` — запись не найдена
- `409` — нарушение бизнес-правила или дублирование уникального поля
- `422` — ошибка валидации входных данных
