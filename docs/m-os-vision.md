> **Статус:** рабочий черновик, обновляется по мере обсуждения с Владельцем.
> **Автор черновика:** Координатор (Claude Code) по указанию Мартина.
> **Дата старта:** 2026-04-16.
> **Источник:** диалог с Мартином в Telegram msg 808-835 + ADR 0008 + память проекта.

> Этот документ — не нормативный (ADR и регламенты — отдельно). Это **полная картина того, что мы хотим построить**, чтобы любой новый член команды (человек или субагент) мог прочитать и понять замысел.

## 1. Что такое M-OS

**M-OS (M Operating System)** — единое внутреннее программное обеспечение холдинга Мартина. Аналог корпоративной ERP (1С), но построенный современно и AI-native. Предназначен исключительно для сотрудников холдинга.

Ключевая идея: **один корпоративный организм — одна программа**. Все направления бизнеса (коттеджи, МКД, АЗС, металл, карьер) работают внутри одной системы как модули (поды). Сотрудник холдинга входит в M-OS по одному аккаунту и видит то, что положено ему по роли.

ИИ внутри M-OS берёт на себя рутинные операции. Важные действия — платежи, подписания, согласования — требуют явного одобрения человека.

Клиентские программы для внешних покупателей создаются отдельно и **на базе** M-OS, но **после** того как внутренняя система готова.

## 2. Принципы (зафиксированы в ADR 0008)

1. **Всё записывается.** Каждое действие — человеческое или ИИ-автоматическое — фиксируется в неизменяемом журнале с указанием кто/что/когда/на основании чего.
2. **Данные холдинга принадлежат холдингу.** Своя инфра, никаких SaaS, шифрование бэкапов, требование ФЗ-152.
3. **ИИ делает рутину, человек одобряет важное.** Ни одно финансово или юридически значимое действие не происходит без подтверждения человека.
4. **Сотрудник не меняет привычное поведение.** Интерфейс подстраивается под роль, не заставляет «учиться новой программе».

### Дополнительные принципы (в работе, для будущих ADR)

5. **Multi-company с первого дня.** Холдинг — несколько юрлиц, у сотрудника могут быть роли в разных. Сделки между юрлицами — внутренние проводки.
6. **Fine-grained права.** Доступ = роль + объект + действие. «Бухгалтер АЗС» видит финансы АЗС и не видит коттеджи, даже если «бухгалтер» — общее название роли.
7. **Эволюция без breaking changes.** Каждое изменение модели/API — через ADR и миграцию. Старые данные всегда читаемы новыми версиями.
8. **Anti-corruption layer к внешнему миру.** M-OS общается с банками, ОФД, Росреестром, 1С, криптопровайдерами через обособленный слой. Внешние API не просачиваются в ядро.
9. **Устройства и оффлайн.** Мобильные (стройка, АЗС) + десктоп (офис), частичный оффлайн с синхронизацией.

10. **Разомкнутая архитектура (pluggability) + редактируемость.** Любая интеграция и функциональный модуль могут быть добавлены, заменены или удалены без правки ядра M-OS — **через интерфейс администратора, не через код**. Это касается всего: банков (завтра добавили Альфа-банк), подов (новое направление появилось — подключили pod), ИИ-моделей (переехали с Claude на другую), систем подписи (КриптоПро → другой криптопровайдер), нотификаций (SMS → push), форматов обмена с 1С/бухгалтерией, **а также всех бизнес-правил, процессов, ролей, лимитов, шаблонов документов, форм, отчётов, уведомлений**. Обязательное требование Владельца: «в любой момент можем что-то добавить, поменять или убрать» — без привлечения разработчика.

11. **Skeleton-first для персональных данных (ПД).** Функциональность, затрагивающая персональные данные (ФИО, паспорта, контакты, биометрия), строится сначала как скелет — структуры данных, маски, заглушки. Полноценный compliance (уведомление Роскомнадзора, согласия, политики хранения) реализуется как отдельный трек **перед production-gate**, не откладывается «до конца». Источник: решение Владельца 2026-04-18 msg 1409.

## 3. Архитектура

### 3.1 Пользователи

Один сотрудник = один аккаунт. Видимость и действия определяются ролью + правами на конкретный объект + юрлицом. Владелец холдинга видит всё.

Сотрудники не обязаны быть технически подкованными. M-OS должна работать для прораба на стройке, бухгалтера, продавца АЗС, мастера на металле одинаково просто.

### 3.2 Поды (модули по направлениям)

| Под | Направление | Статус |
|---|---|---|
| `cottage-platform-pod` | Направление коттеджного девелопмента; пилотный посёлок «Координата 56» (85 домов) внутри `cottage-platform-pod` | Активный (пилот) |
| `gas-stations-pod` | Сеть из 8 АЗС | Запланирован |
| `metal-pod` | Завод металлообработки | Запланирован |
| `mkd-pod` | Девелопер МКД | Запланирован |
| `quarry-pod` | Карьер | Запланирован |
| `core-pod` | HR, финансы холдинга, CRM, общие функции | Запланирован |

Поды изолированы по предметной логике, но разделяют общее ядро.

### 3.3 Общее ядро

- Пользователи и аутентификация
- Роли и права (включая межподовые)
- Журнал действий (аудит)
- Справочники: организации, контрагенты, банки, статьи затрат
- Уведомления
- Процессный движок (BPM)
- Интеграционный слой
- Чат и комментарии к объектам

### 3.4 Интеграционный слой (anti-corruption)

Все внешние системы подключаются через обособленный слой, изолирующий ядро от их изменений.

**Обязательные интеграции:**
- Банки (Сбер, Тинькофф, ВТБ и др.) — выписки, платёжные поручения
- ОФД (онлайн-касса АЗС) — все продажи топлива в реальном времени
- 1С — двусторонний обмен документами/проводками (переходный период)
- Криптопровайдер (КриптоПро, ФЗ-63) — УКЭП, подпись документов
- Росреестр, ЕГРЗ — кадастр, разрешения (для девелопера)
- ДаДата — справочник контрагентов по ИНН
- Telegram — уведомления и кнопки действий для сотрудников

### 3.5 AI-фабрика (способ разработки, не продукт)

Субагенты (Директора, начальники отделов, исполнители) — это способ **строить и сопровождать** M-OS. Для сотрудника холдинга в интерфейсе M-OS субагентов не видно.

## 4. Функциональные возможности

### 4.1 Аудит и неподдельность (блок Е)

- Каждое действие — запись в журнале (уже реализовано в `AuditLog`)
- **Криптографическая цепочка хэшей** — подделать запись задним числом математически невозможно
- **Автоматическая проверка договоров на типовые риски** — штрафы, сроки, форс-мажор, ЧЗ, нестандартные условия
- **Резервное копирование с восстановлением на любую точку во времени** (point-in-time recovery, снимок каждые 5 минут)
- **Регулярные учения disaster recovery** — раз в квартал искусственно ломаем и восстанавливаем

### 4.2 Корпоративная память (блок Ж)

- **«Мозг компании»** — ИИ как старший сотрудник, который всё помнит. Не теряются знания при уходе людей.
- **Исторический explorer решений** — «почему мы выбрали этого подрядчика 3 года назад» → ответ с контекстом, альтернативами, аргументами, кто одобрил.
- **Policy-bot** — «могу ли я взять командировку 3 дня в Москву» → ответ по регламенту с цитатой.
- **Внутренняя вики** — база знаний, индексируется ИИ для поиска.

### 4.3 Физический мир через IoT и видео (блок З)

- **Видеоаналитика стройки** — распознавание касок, стадий работ, неправомерного входа, краж.
- **GPS-трекинг техники и материалов** — самосвалы, RFID на дорогих материалах (металл), устраняет «потерянные» партии.
- **Подключение касс АЗС и датчиков резервуаров** — реальные остатки топлива, автоматический флаг расхождений (признак воровства).
- **Дроны на стройке** (долгосрочно) — автоматическое сравнение факта с планом, видеоархив.

### 4.4 CEO-инструмент: сценарии и раннее предупреждение (блок И)

- **Scenario planning** — «что если бензин подорожает на 10%» → прогноз выручки и прибыли с доверительным интервалом.
- **Что-если калькулятор инвестиций** — «куда вложить 20 млн» → сравнение альтернатив по NPV/IRR/срок окупаемости.
- **Competitor intelligence** — мониторинг рынка: цены конкурентов АЗС, коттеджей у соседей, тендеры по металлу.
- **Раннее предупреждение о рисках** — ключевая ставка, курс, регулирование → связь с бизнес-метриками холдинга.

### 4.5 Самоулучшающаяся система (блок К)

- **Система замечает повторы и предлагает автоматизации** — «вы 5 раз одобрили этот платёж, автоматизировать?»
- **Обучение на собственных данных** — fine-tuning маленькой модели на корпоративных документах.
- **LLM-router** — для каждой задачи выбирается оптимальная модель (Haiku/Sonnet/Opus), экономия 40-60%.
- **Self-healing** — стандартные проблемы (деградация бэкапа, зависший воркер) решаются автоматически, эскалируются только сложные.

### 4.6 Коммуникация с привязкой к объектам (блок Л)

- **Встроенный чат для каждого объекта** — дом, договор, платёж. Переписка не теряется в WhatsApp.
- **Голосовые заметки с транскрипцией** — прораб на объекте говорит, речь → текст → привязка к объекту + уведомления.
- **Комментарии везде с упоминаниями** — любое поле/объект можно прокомментировать, @имя, всё в аудит-журнале.
- **Async видео-сообщения** (Loom-style) — записал 2 минуты, прикрепил к объекту, команда смотрит когда удобно.

### 4.7 Low-code и самообслуживание (блок М)

- **Конструктор форм** для новых справочников — бухгалтер создаёт справочник без разработчика.
- **Конструктор отчётов** — руководитель собирает отчёт drag-drop, сохраняет как шаблон.
- **Визуальный редактор процессов** — BPM-движок с canvas, рисуешь процесс, система исполняет.

## 5. Интерфейс и UX-ноу-хау (блок Б)

### 5.1 Голосовой ввод
Прораб на стройке в перчатках диктует вместо набора. Whisper → распознавание → автозаполнение форм.

### 5.2 Фото → данные
Фото счёта/накладной/договора → ИИ извлекает реквизиты, суммы, даты. Убирает главный блокер стройки: «некогда вводить вручную».

### 5.3 Чат на естественном языке
«Сколько потратили на фундамент дома 23 в марте» → ИИ ходит в БД и отвечает. Не SQL, не отчёты — вопрос-ответ. Role-aware: что видит роль, про то и отвечает.

### 5.4 Проактивный ассистент
ИИ сам напоминает: «счёт на 240k без оплаты 5 дней». Пушит в Telegram с кнопками «Оплатить/Отложить».

### 5.5 Генерация документов
Черновик договора по шаблону + данным конкретной сделки. Человек читает и подписывает.

### 5.6 Anomaly detection
ИИ видит «платёж в 3x среднего» или «материалы в 2x нормы на объём» → флажок владельцу.

### 5.7 Единая очередь одобрений
Вместо 10 уведомлений — inbox: «3 платежа на сегодня, 1 срочный». Сортировка по критичности. Кнопки действий.

### 5.8 Устройства
- Мобильный (стройка, АЗС) — PWA, оффлайн-first, камера, микрофон, биометрия
- Десктоп (офис) — полная функциональность
- Планшет (точки продаж) — упрощённый интерфейс

## 6. Процессный движок (блок Д)

BPM как first-class. Не хардкод процессов в код.

- **Визуальный редактор** — рисуешь процесс, система исполняет
- **Готовые шаблоны** — согласование счёта, приёмка материалов, сдача этапа работ
- **Человеческие шаги** — «требуется одобрение от N человек» / «от любого из директоров»
- **Таймеры и эскалации** — «если не одобрено за 48 часов — эскалация владельцу»
- **Условные ветвления** — «если сумма > 100k — путь через владельца»

---

## 7. Аналитика и управление (блок Г)

- **Дашборд владельца** — одно окно: выручка, маржа, cash, проекты, риски. Realtime.
- **Сравнение направлений** — «АЗС №5 маржа 12%, №3 — 7%. Почему?» ИИ ищет причины.
- **KPI по ролям** — каждая роль получает свои метрики автоматически.
- **Cash-flow прогноз** — ML-прогноз движения денег на 3 месяца вперёд.

## 8. Дополнительные направления (в ходе осмысления)

Эти блоки пока в стадии обсуждения, детализация позже.

### 8.1 Финансовый контроллинг (новое)

- Бюджетирование (частично реализовано в Фазе 3)
- Прогнозирование налогов (НДС, НДФЛ, прибыль)
- Контроль дебиторской и кредиторской задолженности со старением
- Консолидированная отчётность по холдингу (все юрлица)
- Управление ликвидностью — «когда возникнет кассовый разрыв»

### 8.2 Supply chain и тендеры (новое)

- Каталог типовых материалов с историей цен по каждому поставщику
- Внутренняя тендерная платформа — объявил потребность, собрали 3+ предложений, ИИ сравнил и рекомендовал
- Предиктивный заказ — «цемент кончится через 4 дня, заказ формируется автоматически»
- Reliability score поставщиков — срывы сроков, качество

### 8.3 Реестр рисков (новое)

- Каждый риск — объект в системе: тип, вероятность, последствия, меры
- По направлениям: стройка (аварии, срывы), АЗС (пожары, простои), металл (травмы, простои станков), карьер (экология)
- Связь со страховыми системами
- Регулярный health-check по чек-листам

### 8.4 HR и employee experience (новое)

- Профиль сотрудника: зарплата, отпуска, график, обучение
- Онбординг для новых — интерактивные туры
- Learning management — курсы, аттестации
- Feedback loops — сотрудник предлагает улучшение прямо в M-OS
- Анонимные жалобы в HR

### 8.5 Уведомительный хаб (новое)

- Единая система уведомлений: Telegram, email, push, SMS
- Настройка per-type: что пушить немедленно, что в inbox, что в daily digest
- Правила «не беспокоить» в нерабочее время
- Эскалация если не прочитано в срок

### 8.6 Календарь и планирование (новое)

- Корпоративный календарь всех встреч холдинга
- Автоматическое планирование — ИИ предлагает время встреч по занятости участников
- Синхронизация с личным Google/Outlook сотрудника

### 8.7 AR для стройки (долгосрочно)

- Наведи планшет на участок → увидь как будет выглядеть дом (проект 3D)
- Сравнение факта с проектом наложением AR
- Для демонстрации покупателям перед продажей коттеджа

### 8.8 Предиктивное обслуживание оборудования (новое)

- АЗС и металл — оборудование ломается, предсказать когда
- Данные с датчиков → ML → рекомендация замены до поломки
- Снижает простои и аварийные ремонты

### 8.9 Консолидация юрлиц (новое)

- Модель холдинга: материнская + дочерние + связанные
- Внутригрупповые сделки = проводки, не внешние
- Консолидированная отчётность (elimination внутренних оборотов)

### 8.10 Долгосрочно: внешний контур (когда внутренняя готова)

- Клиентские программы на базе M-OS — покупатель коттеджа, клиент АЗС
- Партнёрский портал — подрядчики видят статус своих договоров
- API для внешних интеграторов
- White-label — возможно, M-OS или отдельные поды продаются другим холдингам

### 8.11 Научный подход к управлению (блок Н)

- **Decision journal** — важное решение + прогноз → через N месяцев автоматическая сверка с фактом. Накапливается объективная картина точности решений.
- **OKR-каскад** — цели Владельца → директоров → руководителей → KPI сотрудников в дереве. Каждый видит связь своей работы с общими целями.
- **A/B-тестирование бизнес-гипотез** — изменили цену на 3 АЗС, сравниваем с 3 контрольными, стат-значимость считает ИИ.
- **Lessons learned автоматически** — после завершения проекта ИИ собирает уроки. Корпоративная «книга граблей» пополняется без усилий.

### 8.12 Продвинутый ИИ (блок О)

- **Explainability** — когда ИИ рекомендует, он объясняет почему (оценки, история, цена, риски). Не чёрный ящик.
- **Multi-modal** — голос + фото + текст в одной задаче. Прораб фотографирует, описывает голосом, отмечает на фото — ИИ понимает всё вместе.
- **Collaboration агентов на пользовательских задачах** — «подготовь презентацию для банка» → цепочка ИИ-агентов (data → analyst → designer → legal).
- **Guardrails** — настраиваемые запреты: не предлагать сокращать персонал, не обсуждать конкурентов, не генерировать юр. формулировки без проверки юристом.

### 8.13 Enterprise-безопасность (блок П)

- **SIEM** — мониторинг событий безопасности, ИИ флажит аномалии (массовая выгрузка, нетипичное время, чужие объекты).
- **Zero Trust** — каждый запрос авторизуется, нет «доверенных» сегментов.
- **PAM** — привилегированный доступ на ограниченное время под задачу, с аудитом.
- **Threat intelligence** — подписка на CVE/IOC, ИИ сопоставляет с нашим стеком.
- **Honeypots** — фальшивые «приманки» для обнаружения инсайдеров.

### 8.14 Compliance-инструменты (блок Р)

- **Compliance-календарь** — все регуляторные сроки (ФНС, Росстат, Росприроднадзор, ФЗ-115) в одном месте с напоминаниями и эскалацией.
- **Contract lifecycle management** — договор как объект: черновик → согласование → исполнение → архив. Авто-триггеры на продление.
- **Отслеживание регуляторных изменений** — ИИ следит за изменениями в 214-ФЗ, 152-ФЗ, ГК, связывает с нашей деятельностью.
- **Litigation management** — иски как объект, история споров для аналитики.
- **Privacy engineering** — минимизация ПД, право на удаление, автосчёт сроков хранения по ТК.

### 8.15 Межмодульные контракты (блок С)

- **Data contracts между pods** — схема «Contractor v1», подписчики знают формат, миграция при v2.
- **Event bus** — pods общаются событиями, не прямыми API-вызовами.
- **Semantic consistency** — глоссарий M-OS: «Контрагент» в стройке и на АЗС — один объект, не два.
- **Shared master data** — справочники организаций, банков, статей затрат — единые для всех pods.

### 8.16 Производительность и масштаб (блок Т)

- **Read replicas** — аналитические запросы идут на копию, основная БД не тормозит.
- **Кэширование (Redis)** — часто запрашиваемые данные, быстрее интерфейс.
- **Живое vs архивное хранилище** — 2 года на SSD, старше — дешёвое холодное.
- **Multi-region (долгосрочно)** — синхронная репликация в 2 ДЦ для катастрофоустойчивости.
- **Rate limiting** — защита от перегрузок (сотрудник, ИИ-агент, внешний бот).

### 8.17 Среды и безопасный выкат (блок У)

- **Sandbox / staging / production** — три контура, эксперименты не затрагивают боевых данных.
- **Feature flags** — включение/выключение фич по ролям/юрлицам без деплоя.
- **Canary releases** — сначала 5% сотрудников, мониторим метрики, потом раскатка.
- **Shadow mode для ИИ** — новая ИИ-функция считает, но не решает; сравниваем с человеком; после достижения точности — включаем.

### 8.18 EHS: охрана труда и окружающая среда (блок Ф)

Законодательно требуется для стройки, АЗС, металла, карьера.

- **Реестр инцидентов** — травмы, пожары, ДТП, экологические выбросы. Архив 10 лет.
- **Near-miss tracking** — «чуть не случилось». 90% аварий имеют предшественников.
- **Обязательные инструктажи и аттестации** — кто прошёл, когда истекает; просрочено → недопуск.
- **Пропускная система с интеграцией** — кто сейчас на объекте; при пожаре знаем кого эвакуировать.
- **Отчётность для Ростехнадзора, ФНС, Росприроднадзора** — автогенерация.

### 8.19 Управление активами и автопарком (блок Х)

- **Реестр активов** — техника, оборудование, транспорт, недвижимость. Амортизация, инвентарные номера, ответственные.
- **Планы технического обслуживания** — регламенты ТО, напоминания по моточасам/пробегу.
- **Учёт простоев** — упущенная выручка при поломках.
- **Автопарк** — GPS, топливо, водители; расход выше нормы → флажок возможного слива.
- **Аренда vs собственность** — финансовая модель, обновляемая квартально.

### 8.20 CRM для продаж коттеджей (блок Ц)

Живёт внутри cottage-platform pod, используется продавцами и менеджерами.

- **Lead pipeline** — от заявки до передачи ключей, каждый лид как объект с историей.
- **Lead scoring ИИ** — горячие vs холодные, фокус продавца на приоритетных.
- **Воронка и конверсия** — аналитика по типам домов, месяцам, менеджерам.
- **Единые коммуникации** — звонки, мессенджеры, встречи — привязаны к лиду.
- **Документооборот сделки** — ДДУ, акты, гарантии — автогенерация + УКЭП.
- **Постпродажа** — клиент остаётся в системе через внешний портал (гарантии, ЖКХ, обслуживание).

### 8.21 Работа со временем (блок Ч)

- **Effective-dated data** — цена / тариф / регламент с периодом действия, версии во времени.
- **Future-dated транзакции** — запланировать на дату, пока не исполнится, можно менять.
- **Historical reconstruction** — состояние системы на любую дату в прошлом.
- **Timeline view** — любой объект имеет временную шкалу событий.

### 8.22 Одновременная работа нескольких людей (блок Ш)

- **Real-time co-editing** — как Google Docs, без конфликтов.
- **Presence indicators** — кто сейчас смотрит/редактирует.
- **Multi-user approval** — N из M подписей, пока не все — не исполняется.
- **Delegation на отпуск** — «за меня одобряет Иванов с 1 по 15 мая, лимит 500k».
- **Уведомление следующего в цепочке** — сразу после одобрения.

### 8.23 Документооборот и OCR-конвейер (блок Щ)

- **Полнотекстовый поиск в PDF** — по содержимому, не по имени файла.
- **Версионирование документов** — diff между версиями договора, подсветка изменений.
- **Watermarking / DRM** — имя скачавшего в водяном знаке, отслеживание утечек.
- **Autoclassify входящих** — тип документа + извлечение полей + маршрутизация по процессу.
- **OCR + structured extraction** — фото счёта → ИНН, сумма, реквизиты в БД.

### 8.24 Геолокация и физическое присутствие (блок Ы)

- **Check-in на объектах** — геозона или ручная отметка, табель ведётся сам.
- **Геозоны для триггеров** — «самосвал въехал на склад» → автонакладная на приём.
- **Оптимизация маршрутов** — для автопарка и выездных бригад.
- **Map view** — карта всех объектов, техники, персонала холдинга.

### 8.25 Мотивация и геймификация (блок Э)

- **Прозрачный расчёт ЗП и бонусов** — формула видна сотруднику, сумма в реальном времени.
- **Leaderboards для продавцов** — АЗС и менеджеры коттеджей; настраиваемая анонимность.
- **Бейджи и вехи для бригад** — «дом в срок на 5% ниже бюджета» → бейдж + премия, в истории сотрудника.
- **Peer recognition** — коллега отмечает коллегу, накапливается в профиле, раннее обнаружение лидеров.
- **Internal marketplace задач** — сложные непрофильные задачи с премией, любой в холдинге может взять.

### 8.26 Управленческий учёт CFO-уровня (блок Ю)

- **Трансфертные цены между юрлицами** — внутренние продажи по рыночным ценам, налогово корректно.
- **Аллокация косвенных затрат** — зарплаты центрального офиса по направлениям (по выручке, штату, часам).
- **Бюджетирование на нескольких уровнях** — холдинг → направление → проект → команда.
- **Tax optimization suggestions** — ИИ предлагает легальные способы оптимизации, решение за владельцем + юристом.
- **Dividend planning** — прогноз выплат дивидендов по юрлицам с учётом налогов и оборотного капитала.

### 8.27 Производственное планирование (блок Я)

Для металла и карьера.

- **План выпуска** — тонны металла, кубы щебня; квартал и год с учётом заказов и ремонтов.
- **MRP — планирование материалов** — какое сырьё нужно, когда закупить.
- **Capacity planning** — загрузка оборудования, узкие места, резервы.
- **Себестоимость единицы продукции** — фактическая в реальном времени, сравнение с плановой.
- **Управление заказами** — от заказа клиента до отгрузки по этапам.

### 8.28 Складской учёт и логистика (блок АА)

- **Multi-warehouse** — стройка, металл, карьер, промежуточные площадки.
- **FIFO / LIFO / FEFO** — для материалов со сроком или партионным учётом.
- **Циклические инвентаризации** — ежедневно по частям, приоритет по риску.
- **Тракинг материалов** — приёмка → хранение → выдача → использование → списание.
- **Минимальные запасы и автозаказ** — правила с одобрением человека на сумме выше лимита.

### 8.29 Управление знаниями холдинга (блок ББ)

- **Knowledge graph** — связи «сотрудник ↔ объект ↔ подрядчик ↔ тренинг» как граф.
- **Data lineage** — каждая цифра в отчёте имеет источник, клик → до исходного документа.
- **IP management** — патенты, ноу-хау, уникальные проекты, рецептуры.
- **Succession planning** — кто заменит ключевого сотрудника, карта single point of failure.
- **Инциденты → уроки → регламенты → тренинги** — замкнутый цикл в одном процессе.

### 8.30 Расширяемость системы (блок ВВ)

- **Plugin architecture** — специфические модули как плагины, не правка ядра.
- **Internal app store** — каталог плагинов для владельцев pods.
- **Webhooks и triggers** — простые автоматизации Zapier-like для бизнес-пользователей.
- **Innovation pipeline** — идеи сотрудников → воронка → реализация, с бейджами и премиями авторам.

### 8.31 Биометрия и критичные подтверждения (блок ГГ)

- **Voice biometry для одобрений** — владелец говорит «одобряю», голос сверяется с эталоном.
- **Аппаратные токены** — YubiKey для критичных подписаний.
- **MFA обязателен для owner/accountant** — TOTP + биометрия, политика.
- **Second-device confirmation** — вход с ноутбука → подтверждение с телефона.

### 8.32 Корпоративные коммуникации (блок ДД)

- **Корпоративный newsroom** — объявления от руководства, не теряются в чатах.
- **Опросы (pulse-checks)** — анонимные, агрегированные для HR и владельца.
- **Daily digest** — утреннее письмо с задачами дня, ночными событиями, новостями.
- **Exit interviews автоматизированные** — системные причины увольнений, не разовые.
- **Анонимный канал к владельцу** — защита от токсичного среднего менеджмента.

## 9. Структурированный план по фазам

Все 32 тематических блока (~150 функций) распределены по 5 фазам. Каждая фаза имеет чёткий критерий завершения. Параллельность указана явно.

### Фаза M-OS-0 «Reframing» — 5-7 дней

**Цель:** зафиксировать архитектуру и термины, чтобы вся дальнейшая разработка шла по единой рамке.

**Результаты (deliverables):**
- ADR 0008 — определение M-OS и границ (готов)
- Этот документ m-os-vision.md — полное видение (готов)
- ADR 0009 — принципы pod-архитектуры: общее ядро, data contracts, event bus
- ADR 0010 — таксономия субагентов: 4 типа (core_department / domain_pod / governance / advisory)
- Обновление CODE_OF_LAWS под новую терминологию
- Обновление agents-system-map.yaml и Mermaid-диаграмм
- Миграция документации: `docs/phases/` → `docs/pods/cottage-platform/phases/`
- Мэппинг 48 существующих субагентов на новую таксономию
- Утверждение через governance-director (комиссия Свода законов)
- Синхронизация публичного зеркала `coordinata56-docs`

**Критерий завершения:** любой новый член команды (человек или субагент), прочитав документы, понимает что такое M-OS и как к ней подходить.

### Фаза M-OS-1 «Скелет» — 2-3 месяца

**Цель:** cottage-platform-pod как полноценный пилот + foundation-фичи, проверка модульности через добавление второго pod.

**Обязательное (must-have для MVP M-OS):**

**Foundation архитектура:**
- Multi-company модель данных (А1)
- Fine-grained права: роль + объект + действие (А2)
- Принципы эволюции: миграции без breaking changes (А3)
- Anti-corruption layer — каркас для внешних интеграций (А4)
- **Разомкнутая архитектура (pluggability)** — принцип 10: интеграционные адаптеры, провайдеры ИИ-моделей, форматы обмена — всё подключаемо через интерфейсы, без правки ядра. Dependency injection везде, где есть внешний мир. (критично, без этого M-OS-2/3/4 превратятся в переделку)
- **Реестр интеграций** — каталог всех подключений (какой банк, какая версия API, статус), настройки через админ-интерфейс, не через код
- **Event bus базовый (С2)** — поды и модули общаются событиями, не прямыми вызовами
- **Configuration-as-data layer** — все редактируемые бизнес-сущности (роли, права, лимиты, процессы, шаблоны документов, формы, отчёты, уведомления, бизнес-правила) хранятся в БД как данные, не в коде. Администратор редактирует через UI. Код только читает конфигурацию и исполняет. Аудит каждого изменения конфигурации — тот же журнал.

**Legal PD skeleton (параллельный трек):**
- Структуры данных для хранения персональных данных (ФИО, паспорт, контакты) с масками
- Заглушки для согласий, политики хранения, права на удаление
- Маскирование ПД в аудит-журнале и логах (паспорт, номер телефона — всегда маскируется, не только в production)
- Полноценный compliance-трек (уведомление Роскомнадзора, согласия по ФЗ-152) — выполняется перед production-gate, не позже

**Безопасность и аудит (базовый):**
- Аудит со всех действий — уже есть AuditLog, усилить крипто-цепочкой хэшей (принцип 1, Е1)
- Point-in-time recovery (Е3)
- DR drills ежеквартально (Е4)
- MFA для owner/accountant (ГГ3)
- Sandbox / staging / prod (У1)
- Feature flags по ролям (У2)

**Процессы:**
- Процессный движок BPM — базовый, с человеческими шагами и таймерами (Д1, Д3)
- Визуальный редактор процессов упрощённый (М3)

**Pod-архитектура:**
- Data contracts между pods (С1) — минимальный уровень
- Event bus базовый (С2)
- Shared master data — справочники организаций, банков, статей (С4)

**Cottage-platform-pod полный:**
- Завершение Фазы 3 coordinata56 (OWASP, legal, tech-writer, согласование Владельца)
- Фаза 4-9 по существующему roadmap, но внутри pod-структуры
- Первичный frontend MVP (Vite+React, уже есть скелет)
- UI composition по ролям (принцип 4 + А2)
- Единая inbox одобрений (Б7)
- Telegram-уведомления для всех сотрудников с кнопками (Б4, В7)
- PWA для мобильных + частичный оффлайн

**Проверка модульности:**
- АЗС-pod-lite в параллели — минимальный модуль (справочники АЗС, продавцы, смены), подключён к M-OS
- Критерий: при добавлении АЗС-pod-lite ядро НЕ пришлось переписывать

**Критерий завершения:** сотрудник входит в M-OS, видит свой pod по роли, может работать ежедневно; архитектура выдержала проверку добавлением второго pod.

## 10. Дефиниция MVP (M-OS-1 считается готовым, когда)

1. Сотрудник холдинга заходит по одному аккаунту и видит только то, что положено по роли и юрлицу
2. Cottage-platform-pod полностью функционирует для ежедневной работы на посёлке «Координата 56»
3. Все действия (человеческие + ИИ-автоматические) фиксируются в крипто-защищённом аудите
4. Backup работает, DR-учения пройдены хотя бы один раз
5. Процессный движок обрабатывает минимум: согласование платежей, приёмка работ, создание договора
6. Telegram-уведомления доходят до всех участников процессов с кнопками действий
7. Мобильное PWA работает у прораба на стройке (пусть с базовым набором)
8. Хотя бы одна интеграция с внешним миром подключена через anti-corruption layer (рекомендую: Сбер-API для выписок)
9. Второй pod (АЗС-pod-lite) добавлен без переделки ядра — модульность подтверждена

## 11. Критические зависимости

Порядок нельзя нарушать без риска переделок:

**M-OS-0 Reframing** →
- ADR 0008, 0009, 0010 зафиксированы
- Таксономия субагентов обновлена
- Документация реорганизована в pod-структуру

↓

**M-OS-1 Foundation** (параллельно внутри фазы):
- Multi-company + fine-grained rights (одно должно быть до второго)
- Аудит-цепочка → только потом сложные процессы
- Anti-corruption layer → до любых внешних интеграций
- Pod-архитектура → до добавления второго pod
- **Legal PD skeleton** → параллельно с foundation, compliance-трек завершается перед production-gate

↓

**M-OS-2 Интеграции** (порядок по бизнес-ценности):
- **1С — первая интеграция** (решение Владельца 2026-04-18) > Банки > ОФД > КриптоПро > Росреестр
- **Voice AI для прораба** ПАРАЛЛЕЛЬНО с первыми интеграциями (флагман UX)
- EHS и security КРИТИЧНО с начала фазы (не откладывать)

↓

**M-OS-3 Интеллект** зависит от:
- Накопленного корпуса данных (минимум 3-6 месяцев работы M-OS-2)
- Готовой крипто-цепочки аудита (для decision explorer и lineage)
- Explainability → только до guardrails
- Knowledge graph → до self-improving

↓

**M-OS-4 Физический мир** зависит от:
- Стабильного core (не в режиме активных изменений)
- Data pipelines для работы с потоковыми IoT-данными
- Успешного опыта нескольких pods
- **CV-датасет**: камеры установлены с 2026-04-18, dataset накапливается с первого дня; обучение CV-модели начинается когда объём позволяет (ориентир — 6+ месяцев записей)

↓

**M-OS-5 Внешний контур** только после валидации внутренней.

## 12. Что не входит в M-OS (явно)

- Внешние клиенты (покупатели коттеджей, клиенты АЗС) — отдельные программы на базе M-OS, после готовности внутренней
- Мобильные нативные приложения (iOS/Android) — начинаем с PWA, нативные — позже, если потребуется
- Замена 1С с первого дня — переход постепенный
- Криптовалютные аспекты (blockchain-аудит внешних транзакций) — рассмотрим после M-OS-2, скорее всего не требуется
- **CV-анализ видеопотока** — только в M-OS-4; в M-OS-1/2/3 камеры пишут и хранят, но автоматический анализ не запускается
- **Живые юридические флоу с ПД** (согласия, уведомления Роскомнадзора, активные политики хранения) — только перед production-gate, не раньше; до этого — skeleton и заглушки
