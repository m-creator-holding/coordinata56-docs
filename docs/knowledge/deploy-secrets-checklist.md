# Чек-лист ротации секретов при первом деплое

**Контекст**: в git-истории остались дефолтные значения JWT_SECRET_KEY и seed-пароля owner'а. Это «заглушки», не боевые секреты, утечки реальных данных нет — но при деплое их обязательно заменить.

## Перед первым запуском в prod

- [ ] Сгенерировать JWT-секрет: `openssl rand -hex 32`
- [ ] Записать в `.env` на сервере (НЕ в git): `JWT_SECRET_KEY=<сгенерированное>`
- [ ] Сгенерировать DB-пароль: `openssl rand -base64 24`
- [ ] Записать в `.env`: `DATABASE_URL=postgresql+asyncpg://coordinata:<пароль>@db:5432/coordinata56`
- [ ] Убедиться, что `.env` в `.gitignore` (проверить `git check-ignore .env`)
- [ ] Проверить, что `config.py` падает при старте без env vars (default= удалены — задача security)
- [ ] Первый owner: при первом логине принудительная смена пароля (seed-пароль `change_me_on_first_login` действует только для dev)

## При каждой ротации (раз в полгода минимум)

- [ ] Сгенерировать новый JWT_SECRET_KEY
- [ ] Обновить `.env` на сервере
- [ ] Перезапустить backend — старые токены станут невалидными, пользователи перелогинятся
- [ ] Занести ротацию в журнал (дата, кто, зачем)

## Хранение секретов

**Dev**: `.env.dev` локально, под `.gitignore`.
**Prod**: ENV-переменные systemd unit или Docker secrets. НЕ файлы на диске без ограничения прав.
**Резервные копии секретов**: зашифрованный менеджер паролей (1Password, Bitwarden, KeePassXC) на стороне Мартина. НЕ в git, НЕ в мессенджерах.
