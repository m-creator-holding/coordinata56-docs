## Обзор

Все запросы к защищённым эндпоинтам требуют двух вещей:

1. **JWT-токен** в заголовке `Authorization: Bearer <токен>` — удостоверяет личность пользователя
2. **Заголовок `X-Company-ID`** — указывает, с данными какого юрлица холдинга работает запрос (обязателен, если у пользователя несколько компаний)

## Эндпоинты авторизации

### POST /api/v1/auth/login

> **Как получить JWT-токен**

Принимает логин и пароль, возвращает токен.

**Тело запроса**

```json
{
  "email": "ivan@holding.ru",
  "password": "..."
}
```

**Успешный ответ** `200 OK`

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "bearer"
}
```

**Что внутри токена** (payload после декодирования)

```json
{
  "sub": "42",
  "company_ids": [1, 3],
  "is_holding_owner": false,
  "exp": 1745200000
}
```

| Поле | Что означает |
|------|-------------|
| `sub` | ID пользователя |
| `company_ids` | Список ID компаний, к которым у пользователя есть доступ |
| `is_holding_owner` | `true` — суперадмин холдинга, видит все компании без ограничений |
| `exp` | Время истечения токена (Unix timestamp) |

**Пример curl**

```bash
curl -X POST http://127.0.0.1:8000/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email": "ivan@holding.ru", "password": "..."}'
```

### GET /api/v1/auth/me

> **Как проверить текущего пользователя и его компании**

Возвращает профиль и список компаний пользователя из токена.

**Заголовки**

```
Authorization: Bearer <токен>
```

**Успешный ответ** `200 OK`

```json
{
  "id": 42,
  "email": "ivan@holding.ru",
  "full_name": "Иван Петров",
  "company_ids": [1, 3],
  "is_holding_owner": false
}
```

**Пример curl**

```bash
curl http://127.0.0.1:8000/api/v1/auth/me \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
```

## Как передавать X-Company-ID

> **How-to** — передача заголовка компании в запросах

Заголовок `X-Company-ID` указывает системе, данные какой компании должны использоваться в запросе.

### Когда заголовок нужен обязательно

Если у пользователя **больше одной компании** (поле `company_ids` в токене содержит 2 и более ID) — заголовок обязателен. Без него вернётся ошибка `400`.

### Когда заголовок можно не передавать

Если у пользователя **ровно одна компания** — система автоматически подставит её. Заголовок необязателен.

### Пример запроса с заголовком

```bash
curl http://127.0.0.1:8000/api/v1/projects \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." \
  -H "X-Company-ID: 3"
```

Этот запрос вернёт только проекты компании с ID 3.

## Защищённый эндпоинт: пример

> **How-to** — создание договора с авторизацией

```bash
curl -X POST http://127.0.0.1:8000/api/v1/contracts \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." \
  -H "X-Company-ID: 1" \
  -H "Content-Type: application/json" \
  -d '{
    "number": "КП-2026-001",
    "amount_cents": 5000000,
    "subject": "Монолитные работы",
    "project_id": 7,
    "contractor_id": 3
  }'
```

Для выполнения этого запроса пользователь должен иметь право `contract.write` в компании 1. Если прав нет — вернётся `403 Forbidden`.

## Справочник ошибок

> **Reference** — что означает каждый код ошибки

Все ошибки возвращаются в едином формате (ADR 0005):

```json
{
  "error": {
    "code": "COMPANY_ID_REQUIRED",
    "message": "Необходимо указать X-Company-ID",
    "details": {}
  }
}
```

### 400 Bad Request

| Код ошибки | Причина | Что сделать |
|-----------|---------|-------------|
| `COMPANY_ID_REQUIRED` | У пользователя несколько компаний, но заголовок `X-Company-ID` не передан | Добавить заголовок `X-Company-ID: <id>` |
| `INVALID_REQUEST` | Тело запроса не прошло валидацию | Проверить поля тела по документации |

### 401 Unauthorized

| Код ошибки | Причина | Что сделать |
|-----------|---------|-------------|
| `TOKEN_MISSING` | Заголовок `Authorization` не передан | Добавить `Authorization: Bearer <токен>` |
| `TOKEN_EXPIRED` | Токен истёк | Войти заново через `/auth/login` |
| `TOKEN_INVALID` | Токен повреждён или подделан | Войти заново |

### 403 Forbidden

| Код ошибки | Причина | Что сделать |
|-----------|---------|-------------|
| `PERMISSION_DENIED` | Пользователь не имеет права на это действие (проверьте матрицу прав) | Обратиться к администратору для получения нужной роли |
| `COMPANY_ACCESS_DENIED` | `X-Company-ID` указывает на компанию, не входящую в `company_ids` токена | Проверить токен — у пользователя нет доступа к этой компании |

> **Обратите внимание**: попытка получить ресурс чужой компании возвращает `404 Not Found`, а не `403`. Это намеренно — система не раскрывает существование чужих данных.

## Проверка прав: как это работает изнутри

> **Explanation** — логика проверки `require_permission`

Декоратор `require_permission(action, resource_type)` на каждом write-эндпоинте выполняет следующую цепочку:

```
Запрос → JWT валидация → X-Company-ID валидация
→ загрузка UserCompanyRole из БД (с кешем)
→ can(user_context, action, resource_type)
  ├─ is_holding_owner? → разрешить
  └─ есть роль для этой компании?
       └─ роль разрешает action+resource_type в role_permissions?
            ├─ да → разрешить
            └─ нет → 403
```

Права настраиваются через данные в таблице `role_permissions`, а не через код. Чтобы изменить права роли — нужна строка в таблице, а не деплой.
