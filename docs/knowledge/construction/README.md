## Назначение базы знаний

Эта папка — **отраслевая основа** для:
1. **BPM-процессов** системы coordinata56 (смена стадий, документооборот, согласования).
2. **Справочников** системы (стадии, роли, подрядчики, статьи бюджета).
3. **Обучающего материала** для новых сотрудников холдинга и разработчиков M-OS.
4. **Внешней коммуникации** — консультанты, юристы, инвесторы читают отсюда.

База знаний **не дублирует** проектную документацию coordinata56 (ADR, модели БД, регламенты субагентов). Здесь — доменные знания о стройке, а не системные решения.

## Как пользоваться базой — по типам задач

### Я новый сотрудник, впервые на стройке
1. **Старт:** `04-glossary.md` — прочитать целиком, чтобы понимать речь вокруг.
2. **Потом:** `01-cottage-build-stages.md` — что происходит на стройке последовательно.
3. **Затем:** `06-roles-and-responsibilities.md` — кто тут за что отвечает, включая мою роль.
4. **На потом:** `05-best-practices.md` — чтобы не повторять чужих ошибок.

### Я разработчик, делаю новую модель / процесс / форму
1. **Сначала:** `01-cottage-build-stages.md` — понять бизнес-процесс, который оцифровываем.
2. **Документы и акты:** смотри чек-листы приёмки в `01-*` — это основа для моделирования документооборота.
3. **Роли в системе:** `06-roles-and-responsibilities.md` — кто будет использовать функциональность, RBAC.
4. **Метрики и показатели:** `03-kpi-catalog.md` — какие отчёты / дашборды понадобятся.
5. **Термины в коде:** `04-glossary.md` — чтобы не изобретать названия.

### Я владелец или директор, принимаю решение по проекту
1. **Ритм и сроки:** `01-cottage-build-stages.md` (коттедж) или `02-mkd-build-stages.md` (МКД).
2. **Что контролировать:** `03-kpi-catalog.md`, секция «TOP-10 владельца».
3. **Что не надо делать:** `05-best-practices.md`, раздел «10 типовых ошибок».
4. **Кого нанимать:** `06-roles-and-responsibilities.md`, матрица RACI.

### Я бизнес-аналитик, пишу User Story
1. **Доменная логика:** `01-cottage-build-stages.md` — структура стадий.
2. **Роли:** `06-roles-and-responsibilities.md` — Actors для US.
3. **Документы в бизнес-процессе:** чек-листы приёмки в `01-*`.
4. **Нормативная основа:** `02-mkd-build-stages.md` для 214-ФЗ, экспертизы и т.д.

### Я делаю регламент / инструкцию подрядчикам
1. **Глоссарий:** `04-glossary.md` — единая терминология.
2. **Входной контроль:** `05-best-practices.md` раздел 3.
3. **Форма взаимодействия:** `06-roles-and-responsibilities.md` раздел «Подрядчики».

### Я юрист, проверяю договор
1. **Термины:** `04-glossary.md` — сверка формулировок.
2. **Правовая база для МКД:** `02-mkd-build-stages.md` раздел про 214-ФЗ.
3. **Что должно быть в договоре:** `05-best-practices.md` раздел 1.3 (чек-лист проверки подрядчика).

## Нормативная база — куда обращаться

**Федеральные законы:**
- Градостроительный кодекс РФ (190-ФЗ от 29.12.2004, ред. 2026) — основа стройки.
- 214-ФЗ от 30.12.2004 — долевое строительство МКД.
- 152-ФЗ от 27.07.2006 — персональные данные.
- 384-ФЗ от 30.12.2009 — безопасность зданий.
- 123-ФЗ от 22.07.2008 — пожарная безопасность.

**Постановления Правительства:**
- ПП РФ № 87 от 16.02.2008 — состав разделов ПД.
- ПП РФ № 478 от 10.04.2020 — проектная декларация.

**Своды правил (СП) — основные для коттеджа и МКД:**
- СП 54.13330.2022 — здания жилые многоквартирные.
- СП 55.13330.2016 — дома жилые одноквартирные.
- СП 50.13330.2017 — тепловая защита зданий.
- СП 20.13330.2017 — нагрузки и воздействия.
- СП 22.13330.2016 — основания зданий.
- СП 70.13330.2012 — несущие и ограждающие конструкции.
- СП 47.13330.2016 — инженерные изыскания.

**Полезные публичные ресурсы:**
- **ЕИСЖС (наш.дом.рф)** — раскрытие информации застройщиков МКД.
- **ЕРЗ.РФ** — рейтинги застройщиков, аналитика рынка.
- **Росреестр (rosreestr.gov.ru)** — выписки ЕГРН.
- **Реестр СРО (reestr-sro.ru)** — членство подрядчиков и проектировщиков.
- **egrul.nalog.ru** — проверка юрлиц.
- **kad.arbitr.ru** — судебные дела.
- **bankrot.fedresurs.ru** — банкротства.

## Где публиковать дополнения и поправки

Если вы нашли:
- **Новый термин** — дополнить `04-glossary.md`.
- **Ошибку в сроках / нормах** — поправить соответствующий файл + пометить «проверено на проекте X».
- **Новую best practice** — дополнить `05-best-practices.md`.
- **Новую роль** — дополнить `06-roles-and-responsibilities.md`.
- **Новый KPI** — дополнить `03-kpi-catalog.md` с полным набором полей.

Процедура:
1. Открыть pull request с изменением.
2. Тэгнуть `construction-expert` на ревью.
3. После ревью — merge с комментарием о дате и источнике.
