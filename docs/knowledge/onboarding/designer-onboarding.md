## Контекст: что здесь строим

M-OS — внутреннее корпоративное ПО холдинга. Не публичный сайт. Не мобильное приложение для покупателей. Аудитория — сотрудники: бухгалтеры, прорабы, менеджеры, владелец холдинга.

Первый модуль — cottage-platform-pod: управление строительством 85 коттеджей.

Если вы ещё не читали общий онбординг: [`docs/ONBOARDING.md`](../../ONBOARDING.md)

---

## Главный принцип: UX важнее внешнего вида

На текущей фазе (M-OS-1) задача дизайна — **удобная архитектура экранов и user flows**, а не визуальный полиш. Подбор цветов бренда, типографика, иконки, анимации — это M-OS-1.4, отдельная фаза позже.

Что оценивается сейчас: логика навигации, полнота форм, ясность статусов, отсутствие лишних шагов для пользователя.

Источник правила: `feedback_ux_over_visual_design.md` (зафиксировано Владельцем).

## UI-библиотека: только shadcn/ui

Стек зафиксирован в ADR 0002 и не меняется до M-OS-2. Для описания интерфейса используйте компоненты из shadcn/ui:

- `Table` — таблицы данных
- `Sheet` — боковая панель
- `Dialog` — модальные окна
- `Form`, `Input`, `Select`, `Checkbox`, `Switch` — формы
- `Badge` — статусы и метки
- `Card` — карточки объектов
- `Accordion`, `Tabs` — группировка контента

Не изобретайте компоненты, которых нет в библиотеке, без явного обоснования. Уход с Radix primitives (основа shadcn/ui) — это +30% времени разработки и +2 дня QA.

Три сценария, где кастомное решение обосновано: BPMN-canvas (конструктор процессов), DnD Form Builder (конструктор форм), Permissions Matrix (матрица прав). Всё остальное — через shadcn.

## Формат wireframes

Wireframe в этом проекте — контракт для frontend-разработчика. Не картинка для вдохновения. Контракт.

Каждый wireframe должен содержать:

**1. Описание персон**
Кто смотрит этот экран. Роль (owner, accountant, construction_manager, read_only) и типичная задача.

**2. Текстовая схема layout**
ASCII-схема или markdown-описание расположения блоков. Figma не обязательна на MVP — достаточно точного текстового описания.

Пример:
```
[Шапка: название экрана | фильтры | кнопка "Добавить"]
[Таблица:
  Колонки: Номер дома | Тип | Стадия | Бюджет план | Бюджет факт | Отклонение
  Строки: 20 штук, пагинация внизу
  Пустое состояние: "Дома не найдены. Измените фильтры."
]
[Сайдбар (Sheet): открывается при клике на строку → детали дома]
```

**3. Поля форм**
Для каждого поля: тип (Input / Select / DatePicker / Textarea), обязательность, подсказка (placeholder), правило валидации (если не очевидно из названия).

**4. Состояния**
- loading: что показывается пока данные грузятся (Skeleton)
- empty: что показывается если данных нет
- error: что показывается при ошибке API
- success: состояние после успешного действия

**5. Связи между экранами**
Какой переход происходит при нажатии на кнопку или ссылку.

**6. Ссылки на API endpoint**
Какой endpoint поставляет данные. Можно взять из Swagger: http://localhost:8000/docs

**7. Открытые вопросы**
Если что-то неясно — зафиксировать явно, не додумывать самостоятельно.

## Ограничения (важно)

**Интеграции с банками, 1С, ОФД, Росреестром не проектируются в UI.**
По CODE_OF_LAWS ст. 45а, живые внешние интеграции запрещены до production-gate. В wireframes для таких интеграций — карточка со статусом `disabled` и подписью «Будет активировано после юридического согласования».

Исключение: Telegram — разрешён. Уведомления через Telegram, кнопки действий в Telegram — проектировать можно.

**Scope строго по брифу.**
Не расширяйте задачу самостоятельно. Если в брифе «экран списка домов», не добавляйте «заодно и карточку дома» без согласования. Замеченное — в раздел «Открытые вопросы» wireframe, не в реализацию.

## Design System Initiative

Параллельно с разработкой ведётся Design System Initiative — единая система компонентов, токенов и UX-паттернов для всей M-OS.

Текущее состояние: [`docs/design/design-system-initiative.md`](../../design/design-system-initiative.md)

Что уже определено:
- Стек: React 18 + TypeScript + Vite + Tailwind v3.4 + shadcn/ui + Radix primitives. Не меняется.
- shadcn/ui покрывает ~80% компонентов. Кастомные — только для 3 сценариев (BPMN, DnD Form Builder, Permissions Matrix).
- Доступность: WCAG 2.2 AA — обязательно, проверяется в CI.
- Performance budget: Admin UI initial bundle ≤ 200 KB gzip, Telegram WebApp ≤ 150 KB gzip.
- Storybook — отложен до Фазы 7. Документация компонентов — Markdown в репозитории.

Что пока открыто (решается в RFC-006):
- Dark mode в M-OS-1 или M-OS-2
- Единый или раздельный дизайн-язык для Admin UI и Telegram WebApp
- Количество brand-токенов на компанию

---

## Доступность (WCAG 2.2 AA)

Обязательный стандарт, не опциональный. Базовые требования:
- Любой статус (ошибка, предупреждение, успех) = цвет + иконка + текст. Только цвет — недостаточно (8% мужчин дальтоники).
- Клавиатурная навигация: все интерактивные элементы достижимы через Tab.
- Aria-атрибуты: описать роли для скринридеров (shadcn/ui делает это за вас через Radix, не ломайте).
- Контрастность текста: минимум 4.5:1 для обычного текста, 3:1 для крупного.

## Процесс согласования

```
Designer готовит wireframe
    ↓
UX-head проверяет: полнота состояний, соответствие брифу
    ↓
design-director утверждает: соответствие Design System Initiative
    ↓
frontend-director подтверждает: техническая реализуемость
    ↓
Wireframe передаётся в разработку
```

Документы wireframes хранятся в `docs/pods/cottage-platform/wireframes/` (или аналогичной папке по согласованию с design-director).
