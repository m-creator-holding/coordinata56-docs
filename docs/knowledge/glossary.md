# Глоссарий бизнес-терминов проекта coordinata56

Пополняется субагентом `business-analyst` по мере формализации User Stories. Термины — строго из лексикона Владельца или устоявшиеся в домене.

---

## А

**АЗС** — автозаправочная станция. В контексте холдинга Мартина — одно из 5 бизнес-направлений (8 станций).

**Action-endpoint** — специальный эндпоинт для выполнения действия над объектом, которое меняет его статус. Не путать с обычным PATCH-обновлением. Пример: `POST /payments/{id}/approve` и `POST /payments/{id}/reject`. Введён в ADR 0004 Amendment, чтобы явно отделить бизнес-действие от редактирования полей.

---

## Б

**Bulk-select (групповой выбор)** — одновременный выбор нескольких элементов в матрице прав через отметку строки (все действия одного ресурса) или столбца (все ресурсы для одного действия). Используется на экране Permissions Matrix для массового назначения прав без поочерёдного клика по ячейкам.

---

## В

**Владелец** — Мартин, принимает все финальные решения, имеет право вето на любое действие системы и субагентов.

---

## Г

**Генплан** — генеральный план посёлка, схема расположения участков с домами. В MVP1 — не реализуется (только табличный реестр).

---

## Д

**Дом** — единица учёта в посёлке «Координата 56». Один из 85 объектов, каждый привязан к типовому проекту и имеет набор опций, стадию строительства, историю изменений.

**Договор** (`Contract`) — соглашение между холдингом и подрядчиком на выполнение работ. Привязан к подрядчику, проекту и (опционально) конкретному дому. Имеет жёсткий жизненный цикл статусов: черновик → действующий → завершён / отменён. Обратные переходы запрещены системой.

**Действие (action)** — тип операции над ресурсом системы в контексте управления правами. Стандартный набор: `read` (просмотр), `write` (создание и редактирование), `delete` (удаление), `approve` (согласование). У отдельных ресурсов могут быть специфичные действия. Совместно с ресурсом образует Permission.

**Диф-история (history diff)** — представление журнала изменений прав роли в формате «до → после» по каждой операции назначения или отзыва. Каждая запись содержит: дату, автора, тип изменения («Назначено» / «Отозвано»), конкретное право. Хранится в audit_log, доступна через Drawer истории на экране Permissions Matrix.

---

## З

**Закупка материалов** (`MaterialPurchase`) — фактическая запись о закупке строительного материала для конкретного дома на определённой стадии. Итоговая сумма вычисляется автоматически как произведение количества на цену за единицу. Физически удаляется (без возможности восстановить).

---

## И

**Иммутабельность платежа** — бизнес-правило: платёж в статусе «утверждён» (`approved`) или «отклонён» (`rejected`) не может быть изменён или удалён. Обеспечивает достоверность финансовой истории.

**ИП** — индивидуальный предприниматель. В контексте M-OS — тип компании (`company_type = "ip"`), при котором поле `full_name` содержит ФИО физического лица и является персональными данными по 152-ФЗ. При выборе этого типа в форме Companies — обязательно отображать ПД-маркировку и предупреждение.

---

## К

**Карточка дома** — детальный экран одного дома: все атрибуты, история стадий, опции, ссылки на бюджет и материалы.

**Карьер** — добыча нерудных материалов. Одно из 5 бизнес-направлений холдинга.

**Конфигуратор** — модуль выбора опций для дома покупателем (планируется в M4, вне MVP1).

---

## Л

**Лимит перерасхода договора** — бизнес-ограничение: сумма всех согласованных платежей по договору не может превышать 120% суммы договора. При попытке согласовать платёж, который выведет итог за лимит, система отвечает ошибкой `PAYMENT_EXCEEDS_CONTRACT`. Порог настраивается через переменную `PAYMENT_OVERRUN_LIMIT_PCT` без перезапуска системы.

**ЛНА** — локальный нормативный акт. В контексте ПД: Положение об обработке персональных данных — внутренний документ организации, обязательный по ст. 18.1 152-ФЗ. Утверждается штатным юристом. В рамках skeleton-фазы M-OS — не блокирует разработку; обязателен перед production-gate.

---

## М

**МКД** — многоквартирный дом. Одно из направлений девелопмента в холдинге.

**Мягкое удаление** (`soft delete`) — способ удаления записи, при котором данные не стираются физически, а выставляется отметка `deleted_at`. Запись становится невидимой для обычных запросов, но сохраняется в базе. Применяется для подрядчиков и договоров.

**Матрица прав** — экран Admin UI (`/admin/permissions`), отображающий все права доступа в виде таблицы «ресурс × действие» с колонками по ролям. В каждой ячейке — наличие или отсутствие конкретного права у данной роли. Позволяет назначать и отзывать права через интерфейс.

---

## О

**Опция** — дополнительный элемент комплектации дома, выбирается поверх базового типа. Примеры: навес, спа, сауна, интерьерный пакет, ландшафт. Фиксируется с ценой на момент выбора.

**Область действия роли** (scope) — атрибут роли, определяющий, в каком контексте она применяется: «Глобальная» — роль действует во всех компаниях холдинга; «Компания» — только в конкретной компании при привязке через UserCompanyRole.

---

## П

**Партиальный уникальный индекс** — индекс базы данных, обеспечивающий уникальность только среди записей, отвечающих условию. В проекте используется вид: `UNIQUE (inn) WHERE deleted_at IS NULL` — то есть ИНН уникален только среди активных подрядчиков. Удалённый подрядчик не блокирует регистрацию нового с тем же ИНН.

**Пилот** — первый проект для обкатки системы ERP холдинга. Пилот — посёлок «Координата 56».

**Платёж** (`Payment`) — фактическая выплата по договору. Создаётся в статусе «черновик»; переход в «согласован» или «отклонён» выполняется только через action-endpoints и доступен только Владельцу. После согласования или отклонения платёж становится иммутабельным.

**Плановый бюджет** — совокупность плановых сумм в разрезе проект × статья бюджета × стадия строительства × дом. Загружается до начала работ; является эталоном для сравнения с фактом.

**Под ключ** — полный цикл девелопмента: от покупки земли до сдачи готового дома клиенту, включая все стадии стройки.

**Подрядчик** (`Contractor`) — компания или ИП, выполняющая работы на объекте. Идентифицируется по ИНН. Может быть мягко удалён; удаление блокируется при наличии активных договоров.

**Прораб** — руководитель строительства на объекте. В системе соответствует роли `construction_manager`.

**ПД (персональные данные)** — любая информация, относящаяся к прямо или косвенно определённому физическому лицу (субъекту ПД). В M-OS к ПД относятся: ФИО, email, телефон, паспортные данные, дата рождения. Регулируются Федеральным законом № 152-ФЗ «О персональных данных».

**ПД-маркировка** — визуальная метка «ПД» рядом с полем формы, сообщающая администратору о персональном характере данных. Сопровождается tooltip с пояснением цели сбора и ссылкой на 152-ФЗ. Реализуется как переиспользуемый UI-компонент.

**Permission (право доступа)** — атомарная единица разграничения доступа в M-OS: конкретное действие (`action`) над конкретным ресурсом (`resource`). Примеры: `houses:read`, `payments:approve`, `users:write`. Хранится в справочнике и назначается ролям через матрицу прав. Системные permissions, закреплённые за системными ролями, не могут быть отозваны через UI.

---

## Р

**Реестр домов** — основной экран системы: единый список всех 85 домов с типом, стадией и ключевыми полями.

**Разрез бюджета** — комбинация измерений для планирования/учёта затрат: проект + статья + стадия + дом. Любой из последних трёх может быть null (план «общий по проекту»).

**Роль** (`Role`) — именованная сущность с уникальным кодом (snake_case) и привязанным набором прав доступа. Определяет, что пользователь может делать в системе. Назначается пользователю в контексте конкретной компании. Системные роли (owner, accountant, construction_manager, read_only) поставляются через seed-скрипт и не удаляются через интерфейс.

**РКН** — Роскомнадзор, федеральный регулятор в сфере персональных данных. Проводит плановые и внеплановые проверки соответствия 152-ФЗ. Штраф по ст. 13.11 КоАП за обработку ПД без согласия — до 300 000 руб., при повторе — до 500 000 руб.

**Ресурс (resource)** — сущность системы, доступом к операциям над которой управляют через матрицу прав. Примеры: `houses` (дома), `payments` (платежи), `contractors` (подрядчики), `users` (пользователи). Совместно с действием образует Permission.

---

## С

**Смена стадии** — действие прораба (`construction_manager`) по переводу дома с текущей стадии строительства на следующую. Автоматически создаёт запись в `HouseStageHistory`. Обратный переход (регресс) по умолчанию запрещён.

**Статья бюджета** (`BudgetCategory`) — справочная единица классификации затрат: фундамент, кровля, инженерные сети и т.д. Используется в плановом и фактическом бюджете.

**Стадия строительства** — текущий этап возведения дома из 11 предопределённых: земля → изыскания → фундамент → стены → кровля → инженерка → отделка → ландшафт → интерьер → сдача → сервис.

**Согласие на обработку ПД** — явное волеизъявление субъекта ПД, выраженное в электронной форме через отметку чекбокса (не предустановленного), на обработку его данных в целях и в объёме, указанных в тексте согласия. Требуется ст. 9 п. 1 152-ФЗ. В M-OS фиксируется в полях `pd_consent_at` и `pd_consent_version` модели User.

**Soft erasure (мягкое обнуление)** — способ «удаления» ПД, при котором ПД-поля записи обнуляются (NULL), но сама строка сохраняется в базе. Применяется при исполнении требований ст. 14 152-ФЗ (право субъекта на удаление данных) для сохранения доказательства факта обработки запроса.

---

## Т

**Тип дома** — один из 4 базовых типовых проектов (A, B, C, D), определяющих планировку и площадь. Является основой для формирования стоимости и набора совместимых опций.

**Переход статуса в аудит-записи** (`audit.meta transition`) — стандартная структура записи аудита при смене статуса через action-endpoint: `{"transition": "<новый_статус>", "from_status": "<прежний_статус>"}`. При отклонении добавляется поле `"reason"`. Ключи фиксированы — не допускаются альтернативные варианты вроде `old_status` / `new_status`.

---

## У

**Участок** (`plot_number`) — уникальный номер земельного участка в посёлке, идентифицирует дом внутри проекта. Два дома в одном проекте не могут иметь одинаковый `plot_number`.

**UserCompanyRole** — сущность-привязка: соединяет конкретного пользователя, конкретную компанию и конкретную роль. Один пользователь может иметь несколько привязок (разные роли в разных компаниях холдинга). Опционально содержит поле `pod_id` — для ограничения доступа в рамках одного пода компании.

## Х

**Холдинг** — группа компаний Мартина: 5 направлений (карьер, металлообработка, 8 АЗС, девелопер МКД, девелопер коттеджей).

## Ц

**Цифровизация** — перевод управленческих процессов холдинга в единую информационную систему с прозрачностью план/факт по каждому объекту.

## Технические термины (Блок 2 — Архитектура Backend)

**API** (Application Programming Interface — интерфейс программного взаимодействия) — набор «окошек», через которые один программный компонент обращается к другому. В M-OS: браузер (фронтенд) общается с сервером (backend) через REST API. Каждое «окошко» — эндпоинт.

**Alembic** — инструмент управления миграциями базы данных для Python/SQLAlchemy. Хранит историю всех изменений схемы в виде Python-файлов. Умеет применять изменения вперёд (`upgrade`) и откатывать назад (`downgrade`).

**Декоратор** — конструкция Python вида `@что-то`, которая «помечает» функцию и изменяет её поведение. Пример: `@router.get("/")` говорит FastAPI: «вот эта функция обрабатывает GET-запросы на адрес `/`».

**Depends** — механизм внедрения зависимостей в FastAPI. Говорит фреймворку: «перед вызовом функции выполни вот эту другую функцию и передай результат». Используется для открытия сессии БД, проверки токена, проверки роли.

**Endpoint (эндпоинт)** — конкретное «окошко» API. Определяется URL-адресом и HTTP-методом. Пример: `GET /projects/` — один эндпоинт, `POST /projects/` — другой эндпоинт на том же адресе.

**expand/contract** — безопасный паттерн изменения схемы базы данных с живыми данными: 1) добавить структуру как необязательную (expand), 2) заполнить данные (backfill), 3) сделать обязательной (contract). Позволяет изменять базу без остановки системы.

**HTTP-методы** — стандартные команды протокола HTTP, определяющие намерение запроса: GET (получить), POST (создать), PATCH (частично изменить), PUT (полностью заменить), DELETE (удалить).

**JSON** (JavaScript Object Notation) — текстовый формат для передачи данных. Выглядит как словарь: `{"ключ": "значение"}`. Универсален: используется и в Python, и в браузере, и в мобильных приложениях.

**Миграция** — файл с инструкциями по изменению структуры (схемы) базы данных: добавить таблицу, добавить колонку, создать индекс. Управляется Alembic. Каждая миграция содержит функции `upgrade()` и `downgrade()`.

**Mixin (миксин)** — вспомогательный класс Python, который «подмешивает» набор готовых полей или методов в другой класс. В M-OS: `TimestampMixin` добавляет `created_at`/`updated_at`, `SoftDeleteMixin` добавляет `deleted_at`.

**ORM** (Object-Relational Mapper — объектно-реляционное отображение) — прослойка между Python-кодом и реляционной базой данных. Переводит Python-объекты в SQL-запросы и обратно. В M-OS используется SQLAlchemy.

**Pool (пул соединений)** — запас заранее открытых соединений с базой данных. Вместо того чтобы открывать новое соединение для каждого запроса (медленно) — ORM берёт готовое из пула. В M-OS: `pool_size=5`, `max_overflow=10`.

**REST** (Representational State Transfer) — набор договорённостей о построении API: адрес описывает «что», метод описывает «как», данные передаются в JSON. Все современные веб-сервисы используют этот подход.

**round-trip (миграций)** — обязательная проверка: `upgrade → downgrade → upgrade` без ошибок. Гарантирует, что откат миграции возможен. В нашем проекте — обязательное требование перед принятием миграции в main.

**Router (роутер)** — файл в `backend/app/api/`, который определяет набор эндпоинтов для конкретной сущности. Только транспорт: принять запрос, проверить формат, передать в сервис. Бизнес-логики не содержит.

**Сессия (SQLAlchemy session)** — единица работы с базой данных в рамках одного HTTP-запроса. Открывается при входе в запрос, закрывается при выходе. Все изменения в рамках сессии либо фиксируются вместе (`commit`), либо откатываются вместе (`rollback`).

**SQL** (Structured Query Language — язык структурированных запросов) — язык, на котором разговаривают с реляционными базами данных. ORM автоматически генерирует SQL из Python-кода.

**SQLAlchemy** — библиотека Python для работы с реляционными базами данных. В M-OS используется версия 2.0 с асинхронным API (async/await).

**Транзакция** — группа операций с базой данных, которые выполняются по принципу «всё или ничего». Если любая операция внутри транзакции завершилась с ошибкой — все изменения откатываются. Фиксируется командой `commit()`.

**ForeignKey (внешний ключ)** — ограничение базы данных: значение колонки должно существовать в другой таблице. Пример: `company_id` в таблице `projects` должен совпадать с `id` в таблице `companies`. Обеспечивает целостность данных.

**mapped_column / Mapped** — конструкции SQLAlchemy 2.0 для описания колонок в Python-классе. `Mapped[int]` — подсказка типа, `mapped_column(...)` — описание параметров колонки в базе данных.

## Технические термины (Блок 3 — Безопасность и качество)

**Аутентификация** (authentication) — процесс проверки личности: «Ты действительно тот, за кого себя выдаёшь?» В M-OS: ввод email и пароля при входе. Отличается от авторизации (что тебе разрешено делать).

**Авторизация** (authorization) — процесс проверки прав: «Тебе разрешено это делать?» В M-OS: проверка роли (owner, accountant, construction_manager) перед выполнением действия.

**bcrypt** — алгоритм хеширования паролей. Намеренно медленный (~100-300 мс на одну проверку) для защиты от брутфорса. Автоматически добавляет соль. Результат содержит и хеш, и соль в одной строке формата `$2b$12$...`. В M-OS — единственный алгоритм хеширования паролей.

**Брутфорс** (brute force — атака грубой силой) — метод взлома перебором всех возможных вариантов пароля. Защита: медленное хеширование (bcrypt) + rate-limiting + MFA.

**Переменная окружения** (environment variable) — настройка, которую программа читает из операционной системы, а не из собственного кода. Задаётся до запуска программы. В M-OS хранится в файле `.env` и читается через `pydantic-settings`.

**CVE** (Common Vulnerabilities and Exposures — общие уязвимости и подверженности) — международный реестр уязвимостей в программном обеспечении. Каждая уязвимость имеет номер `CVE-YYYY-NNNNN`, описание, затронутые версии и оценку опасности (CVSS — от 0 до 10). В M-OS: при аудите зависимостей ищем CVE с HIGH/CRITICAL оценкой.

**CVSS** (Common Vulnerability Scoring System) — система оценки опасности уязвимости по шкале 0-10: 0-3 низкая, 4-6 средняя, 7-8 высокая, 9-10 критическая.

**gitleaks** — инструмент поиска секретов (паролей, ключей, токенов) в коде и в истории Git-репозитория. Знает паттерны тысяч сервисов. В M-OS: входит в чек-лист задачи S-2 аудита безопасности.

**detect-secrets** — альтернативный инструмент поиска секретов в коде. Умеет создавать «базовую линию» ложных срабатываний.

**HS256** (HMAC with SHA-256 — HMAC с SHA-256) — симметричный алгоритм подписи JWT. «Симметричный» означает: один и тот же ключ используется и для подписания, и для проверки. В M-OS: используется для JWT-токенов, достаточен для внутреннего ПО.

**IDOR** (Insecure Direct Object Reference — небезопасная прямая ссылка на объект) — уязвимость, при которой пользователь может получить доступ к чужим данным, перебирая идентификаторы (ID). В M-OS: устраняется через `company_id`-фильтры в `CompanyScopedService`. OWASP категория A01.

**JWT** (JSON Web Token — токен в формате JSON) — стандарт для передачи информации в виде подписанной строки. Состоит из трёх частей: header (алгоритм), payload (данные о пользователе), signature (криптографическая подпись). Не шифруется — только подписывается. В M-OS: живёт 60 минут, содержит email, роль, company_ids.

**Клейм** (claim) — отдельное поле внутри JWT payload. Стандартные клеймы: `sub` (субъект / пользователь), `exp` (время истечения), `iat` (время выдачи). Произвольные: `role`, `company_ids`, `is_holding_owner`. В Legal PD track добавлен клейм `consent_required`.

**Литерал** (literal) — значение, написанное прямо в коде. `"secret_key_123"` прямо в Python-файле — литерал. Запрещено для секретов: секрет должен читаться из `os.environ.get()` или `Settings`.

**MFA** (Multi-Factor Authentication — многофакторная аутентификация) — второй способ подтверждения личности помимо пароля. Примеры: SMS-код, TOTP-приложение (Google Authenticator), аппаратный ключ. В M-OS: обязателен для ролей `owner` и `accountant` (ст. 81 Конституции M-OS), блокер production-gate.

**OWASP** (Open Web Application Security Project — открытый проект безопасности веб-приложений) — международная некоммерческая организация, публикующая стандарты и чек-листы по безопасности. OWASP Top 10 — список 10 наиболее опасных уязвимостей веб-приложений, обновляется раз в несколько лет. Текущая версия: 2021.

**passport:read** — право доступа, позволяющее получить немаскированные паспортные данные пользователя через API. Не входит в стандартный набор прав администратора. Каждый факт раскрытия логируется в audit_log.

**pd:erase** — право доступа на выполнение операции мягкого обнуления ПД пользователя. Отдельное от прав удаления пользователя как учётной записи. Предназначено для исполнения требований ст. 14 152-ФЗ.

**pd:export** — право доступа на выгрузку полного набора ПД конкретного пользователя в структурированном файле. Используется при исполнении права субъекта на получение информации об обрабатываемых данных (ст. 14 152-ФЗ).

**pip-audit** — инструмент проверки Python-зависимостей на наличие CVE. Читает `pyproject.toml` или `requirements.txt`, запрашивает базу уязвимостей, выводит список находок. В M-OS: входит в задачу S-3 аудита.

**pre-commit hooks** — скрипты, которые автоматически выполняются перед каждым `git commit`. Могут проверять код на секреты, форматирование, ошибки. Если проверка не прошла — коммит блокируется.

**Rainbow table** (радужная таблица) — заранее вычисленная таблица «пароль → хеш» для миллионов популярных паролей. Позволяет мгновенно найти пароль по хешу. Защита: соль в хеше делает радужные таблицы бесполезными.

**Rate-limiting** (ограничение частоты запросов) — механизм, ограничивающий количество запросов от одного источника в единицу времени. Пример: не более 5 попыток входа с одного IP за 1 минуту. Защита от брутфорса. В M-OS: отсутствует, запланирован в PR #2.

**Salt** (соль) — случайная строка, добавляемая к паролю перед хешированием. Уникальна для каждого пользователя. Обеспечивает уникальность хешей даже при одинаковых паролях. В bcrypt встроена автоматически.

**Security headers** (заголовки безопасности) — HTTP-заголовки, которые сервер отправляет браузеру с инструкциями по безопасности: откуда загружать ресурсы, использовать ли только HTTPS, запрещать ли встраивание в iframe. В M-OS: отсутствуют, запланированы в PR #3.

**SSRF** (Server-Side Request Forgery — подделка запроса на стороне сервера) — уязвимость, при которой злоумышленник заставляет сервер делать HTTP-запросы к произвольным адресам. В M-OS: нет нарушений, все исходящие запросы изолированы адаптерами. OWASP категория A10.

**SQL-инъекция** (SQL injection) — атака подстановки вредоносного SQL-кода через пользовательский ввод. Защита: использование ORM с параметризованными запросами (никогда не подставлять ввод напрямую в SQL-строку). В M-OS: нет нарушений, весь код через SQLAlchemy ORM. OWASP категория A03.

**Timing-атака** (timing attack) — атака, при которой злоумышленник определяет информацию (например, существует ли email) по времени ответа сервера. Защита в M-OS: `dummy_verify` выравнивает время ответа на `/auth/login` для случаев «нет пользователя» и «неверный пароль».

**User enumeration** (перечисление пользователей) — утечка информации, позволяющая определить, зарегистрирован ли конкретный email в системе. Защита: одинаковое сообщение об ошибке и одинаковое время ответа для любой ошибки входа.

**Хеш** (hash, хеш-функция) — результат одностороннего преобразования данных. По исходным данным легко вычислить хеш. По хешу — невозможно восстановить исходные данные. Используется для хранения паролей: в базе хранится хеш, не сам пароль.

**`.env`** — файл с переменными окружения для конкретного сервера или окружения разработки. Никогда не попадает в Git (добавлен в `.gitignore`). В репозитории хранится только `.env.example` с названиями переменных без значений.

**`.gitignore`** — файл, определяющий, какие файлы и папки Git должен игнорировать (не отслеживать, не коммитить). Обязательно включает `.env`, `*.key`, файлы с реальными секретами.
