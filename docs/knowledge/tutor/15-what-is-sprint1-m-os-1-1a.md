## Коротко о главном

Sprint 1 M-OS-1.1A — первый спринт основного скелета M-OS. Он закладывает четыре фундаментальных механизма, без которых система холдинга из 5 компаний невозможна. Ни один из них не виден конечному пользователю напрямую — это невидимые «несущие стены» здания.

---

## 1. Multi-company (мульти-компанийность)

### Зачем

У Вас 5 юридических лиц. Карьер, металл, АЗС, девелопер МКД, девелопер коттеджей — каждое ведёт свои деньги, объекты, персонал. Если все данные лежат в одной «куче», бухгалтер карьера случайно увидит платежи АЗС, отчёт смешает цифры двух компаний, а аудитор не сможет проверить одно направление, не задев другое.

### Аналогия

Это как разные бухгалтерские книги для каждого юрлица. Несмотря на то что все они стоят на одной полке у главного бухгалтера холдинга, каждая книга принадлежит только своему ООО. Можно смотреть любую — но всегда знаешь, чья она.

### Конкретика

Каждая строка в базе данных содержит поле `company_id` — идентификатор (уникальный номер) компании. Запрос на «показать платежи» автоматически добавляет условие «только для company_id = текущей компании». Данные физически в одной базе, но логически разделены.

```
Платёж №1001  company_id=3  (карьер)   сумма=500 000 ₽
Платёж №1002  company_id=5  (АЗС)      сумма=120 000 ₽
Платёж №1003  company_id=3  (карьер)   сумма=80 000 ₽

Запрос бухгалтера карьера → возвращает только #1001 и #1003
```

### Риски без этого

- Финансовый отчёт смешивает выручку АЗС и карьера — цифры холдинга верны, цифры направлений — нет.
- Сотрудник одной компании видит зарплатную ведомость другой. Конфликт, утечка, юридический риск.

## 2. RBAC-матрица (7 действий × 22 сущности)

**RBAC** (Role-Based Access Control — управление доступом по ролям) — система, которая отвечает на вопрос «кто с чем что может делать».

### Зачем

Не все сотрудники должны всё видеть и всё менять. Бухгалтер читает платежи, но не одобряет их сам. Прораб видит график работ, но не видит зарплаты. Без такой системы либо все видят всё (риск), либо никто ничего не видит (бесполезная система).

### Аналогия

Это как пропускная система на строительной площадке с пропусками разного цвета. Рабочий (синий пропуск) — заходит только на свой участок. Прораб (красный пропуск) — заходит в любой блок, но не в бытовку охраны. Генеральный директор (золотой пропуск) — везде. Охранник не запоминает каждого человека — он просто смотрит на цвет пропуска.

### Конкретика

Матрица — это таблица: строки — роли (бухгалтер, прораб, директор, аудитор), столбцы — сущности (платёж, объект, договор, сотрудник). На пересечении — разрешённые действия: читать, создавать, редактировать, удалять, одобрять, экспортировать, видеть стоимость.

```
Роль          | Платёж        | Зарплата      | График работ
Бухгалтер     | читать        | читать        | —
Прораб        | —             | —             | читать, редактировать
Директор      | все действия  | все действия  | все действия
```

### Риски без этого

- Рядовой сотрудник случайно удаляет утверждённый договор — откат невозможен.
- Прораб видит ставки других сотрудников — демотивация, конфликты, нарушение трудового законодательства.

## 3. JWT-токен с claims (цифровой пропуск)

**JWT** (JSON Web Token — токен в формате JSON) — зашифрованная строка, которую сервер выдаёт пользователю при входе в систему. Она хранится в браузере и отправляется с каждым запросом.

**Claims** (клеймы — утверждения) — данные, зашитые внутрь токена: кто этот пользователь, в каких компаниях работает, какая роль.

### Зачем

Сервер не помнит пользователей между запросами — он отвечает на тысячи обращений в минуту. Вместо того чтобы при каждом клике лезть в базу и проверять «а кто это вообще такой», он читает токен — и мгновенно знает всё нужное.

### Аналогия

Это как ламинированный бейдж сотрудника. На нём написано: имя, должность, список объектов, к которым есть допуск. Охранник на КПП не звонит в отдел кадров — он смотрит на бейдж. Бейдж подписан директором (сервер подписывает токен своим ключом), подделать его нельзя — подпись будет недействительной.

### Конкретика

После входа сервер создаёт токен с зашитыми данными:

```
{
  "user_id": 42,
  "companies": [3, 5],        ← в каких компаниях работает
  "current_company": 3,       ← какая компания выбрана сейчас
  "role": "accountant",       ← роль в текущей компании
  "cross_company": false      ← запрещено переключаться в чужую компанию
}
```

Эта строка шифруется и отправляется пользователю. При следующем запросе браузер отправляет её обратно — сервер расшифровывает и сразу знает контекст.

### Риски без этого

- Без токена каждый запрос требует обращения к базе данных за профилем пользователя — система работает в 10 раз медленнее.
- Без подписи злоумышленник может подменить `"role": "accountant"` на `"role": "admin"` — и получить полный доступ.

## 4. Middleware (охранники на входе)

**Middleware** (мидлвэр — промежуточный слой) — код, который выполняется автоматически при каждом запросе, до того как запрос дойдёт до бизнес-логики.

### Зачем

Без middleware каждый отдельный обработчик запросов (получить список платежей, создать договор, удалить объект) должен был бы сам: проверить токен, вытащить текущую компанию, убедиться, что у пользователя есть право. Это сотни одинаковых проверок — и при любой ошибке в одном месте дыра в безопасности.

### Аналогия

Это как КПП при въезде на территорию комбината. Неважно, едет ли машина к прорабу, в бухгалтерию или на склад — сначала КПП: документы, пропуск, журнал. Никто не может объехать КПП и попасть сразу к нужному цеху. Охранник в каждом цеху не нужен — есть один централизованный пост.

### Конкретика

Middleware — это три «поста», через которые проходит каждый запрос:

```
Запрос от браузера
    ↓
[Пост 1] Проверить токен — он действителен? Не истёк?
    ↓
[Пост 2] Вытащить из токена текущую компанию и роль
    ↓
[Пост 3] Проверить: у этой роли есть право на это действие?
    ↓
Бизнес-логика (список платежей, создание договора и т.д.)
```

Если любой пост «не пускает» — запрос останавливается, бизнес-логика не выполняется.

### Риски без этого

- Разработчик добавляет новый endpoint (точку доступа), забывает вставить проверку токена — дыра, через которую любой может читать данные без авторизации.
- При изменении логики проверки нужно поменять код в 50 местах, а не в одном — высокий риск пропустить.

---

## Зачем Вам знать это

Когда система заработает, Вы будете принимать решения о доступах: кто видит зарплаты, кто может одобрять платежи, может ли директор одного направления видеть данные другого. Эти решения напрямую ложатся в RBAC-матрицу — Вы будете её заказчиком, а не техническим исполнителем. Понимание того, как устроены токены и мульти-компанийность, поможет Вам ставить задачи точно: «прораб видит только объекты компании 3» — это и есть claims + company_id в одной фразе.
