## Коротко о главном

**Аутентификация** (authentication, от греч. «подлинный») — это процесс проверки: «Ты действительно тот, за кого себя выдаёшь?» Сервер должен знать, кто делает каждый запрос, и убеждаться, что это не самозванец. В M-OS аутентификация строится на двух механизмах: **bcrypt** для проверки паролей при входе и **JWT** для подтверждения личности при каждом последующем запросе.

---

## Зачем это нужно

В строительстве есть понятие контрольно-пропускного пункта (КПП). На закрытый объект так просто не войдёшь: охранник проверяет удостоверение, фиксирует в журнале, выдаёт временный пропуск. Этот пропуск действует ограниченное время и даёт доступ только в определённые зоны.

В M-OS то же самое:
- **Удостоверение** = email + пароль (то, что Вы знаете)
- **Проверка охраной** = сервер сверяет пароль с bcrypt-хешем в базе
- **Временный пропуск** = JWT-токен (живёт 60 минут)
- **Зоны доступа** = роли (owner видит всё, accountant — только финансы)

## Шаг 1: Вход — эндпоинт `/auth/login`

Файл: `backend/app/api/auth.py`

Когда Вы вводите email и пароль в форму входа, браузер отправляет POST-запрос на `/api/v1/auth/login` с таким телом:

```json
{
    "email": "martin@coordinata.ru",
    "password": "Ваш_пароль"
}
```

Сервер выполняет следующие шаги:

### Шаг 1.1 — Найти пользователя в базе данных

```python
result = await db.execute(select(User).where(User.email == body.email))
user = result.scalar_one_or_none()
```

Ищем пользователя по email. Если не нашли — переходим к шагу 1.2 (специальная обработка).

### Шаг 1.2 — Если пользователь не найден: `dummy_verify`

```python
if user is None:
    dummy_verify(body.password)   # <- зачем это?
    raise HTTPException(401, "Неверный email или пароль")
```

Зачем вызывать `dummy_verify`, если пользователь не найден? Это защита от **timing-атаки**. Объяснение:

Проверка пароля через bcrypt занимает ~100-300 миллисекунд — это намеренно медленная операция. Если пользователь не найден и мы сразу вернём ошибку, ответ придёт за 1-2 мс. Если пользователь найден, но пароль неверный — ответ придёт за 200 мс. Злоумышленник может замерять время ответов и понять: «этот email существует в системе, этот — нет». Это называется **перечисление пользователей (user enumeration)** — утечка информации через время ответа.

`dummy_verify` выполняет настоящую bcrypt-проверку против заранее вычисленного фиктивного хеша. Результат игнорируется — важно лишь, что на это уйдут те же 200 мс. Теперь злоумышленник не может отличить «нет такого email» от «неверный пароль» по времени ответа.

### Шаг 1.3 — Проверить пароль через `verify_password`

```python
if not verify_password(body.password, user.password_hash):
    raise HTTPException(401, "Неверный email или пароль")
```

```python
# security.py
def verify_password(plain: str, hashed: str) -> bool:
    return bool(_pwd_context.verify(plain, hashed))
```

Bcrypt извлекает соль из хеша и проверяет, соответствует ли введённый пароль. Возвращает `True` или `False`. Если `False` — возвращаем ту же ошибку `"Неверный email или пароль"`. Заметьте: сообщение одинаковое для «нет пользователя» и «неверный пароль» — это намеренно. Не говорим злоумышленнику, существует ли email в системе.

### Шаг 1.4 — Создать JWT-токен

```python
token = create_access_token(
    sub=user.email,
    role=user.role.value,
    company_ids=company_ids,
    is_holding_owner=is_holding_owner,
)
return TokenResponse(access_token=token, token_type="bearer")
```

Сервер создаёт токен и возвращает его. Браузер сохраняет этот токен (обычно в памяти или в localStorage) и будет использовать его в каждом следующем запросе.

## Шаг 2: JWT-токен — что это и как работает

**JWT** (JSON Web Token — токен в формате JSON) — это строка, которая содержит информацию о пользователе и подтверждает, что её выдал наш сервер, а не кто-то посторонний.

### Структура JWT

JWT состоит из трёх частей, разделённых точками:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
.
eyJzdWIiOiJtYXJ0aW5AY29vcmRpbmF0YS5ydSIsInJvbGUiOiJvd25lciIsImNvbXBhbnlfaWRzIjpbXSwiaXNfaG9sZGluZ19vd25lciI6dHJ1ZSwiZXhwIjoxNzE2MDAwMDAwLCJpYXQiOjE3MTU5OTY0MDB9
.
K7mVqNpR2xWzYoEt8bJcUsFiG3kQ9nL0dRyM1l4A
```

**Часть 1 — Header (заголовок)**:
```json
{
    "alg": "HS256",
    "typ": "JWT"
}
```
Говорит: «Это JWT-токен, подписан алгоритмом HS256».

**Часть 2 — Payload (содержимое, «клеймы»)**:
```json
{
    "sub": "martin@coordinata.ru",
    "role": "owner",
    "company_ids": [],
    "is_holding_owner": true,
    "exp": 1716000000,
    "iat": 1715996400
}
```

Это читаемая информация о пользователе. `sub` — субъект (subject), то есть пользователь. `exp` — время истечения (expiration), Unix-время. `iat` — время выдачи (issued at).

**Важно**: payload не шифруется — его может прочитать любой, кто получит токен. Поэтому в payload нельзя класть секреты (пароли, ключи).

**Часть 3 — Signature (подпись)**:
```
HMAC_SHA256(
    base64(header) + "." + base64(payload),
    JWT_SECRET_KEY
)
```

Это криптографическая подпись. Сервер берёт header и payload, объединяет их, и вычисляет «цифровую печать» с помощью секретного ключа (`JWT_SECRET_KEY`). Без знания секретного ключа невозможно создать правильную подпись. Это и есть гарантия подлинности.

### Что такое HS256

**HS256** (HMAC with SHA-256) — симметричный алгоритм подписи. «Симметричный» значит: один и тот же ключ используется и для создания подписи, и для её проверки.

Это как обычный ключ от замка — он и открывает, и закрывает. Подходит для внутреннего ПО (M-OS), где и тот, кто выдаёт токены (сервер), и тот, кто их проверяет (тот же сервер), — это одно и то же.

Альтернатива — RS256 (асимметричный): два ключа, публичный и приватный. Нужен, когда токены выдаёт один сервис, а проверяет другой. В M-OS нам это не нужно — HS256 достаточно и проще.

## Шаг 3: Проверка токена при каждом запросе — `deps.py`

После входа у браузера есть токен. При каждом следующем запросе браузер добавляет его в заголовок:

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

Сервер должен проверить этот токен. Это делает функция `get_current_user` в файле `backend/app/api/deps.py`:

```python
async def get_current_user(
    token: str = Depends(oauth2_scheme),
    db: AsyncSession = Depends(get_db),
    x_company_id: int | None = Header(default=None, alias="X-Company-ID"),
) -> tuple[User, UserContext]:
```

`Depends(oauth2_scheme)` — FastAPI автоматически извлекает токен из заголовка `Authorization: Bearer ...`.

### Алгоритм проверки

**1. Декодировать и проверить подпись:**

```python
try:
    payload = decode_token(token)
except jwt.ExpiredSignatureError:
    raise credentials_exception   # токен просрочен
except jwt.InvalidTokenError:
    raise credentials_exception   # подпись не совпадает
```

`decode_token` в `security.py` вызывает PyJWT, который:
- Проверяет подпись: вычисляет ожидаемую подпись по тому же алгоритму HS256 и сравнивает с той, что в токене
- Проверяет `exp`: если текущее время позже времени истечения — выбрасывает `ExpiredSignatureError`
- Если всё верно — возвращает payload (словарь с данными пользователя)

**2. Найти пользователя в базе данных:**

```python
sub = payload.get("sub")  # email из токена
result = await db.execute(select(User).where(User.email == sub))
user = result.scalar_one_or_none()
if user is None:
    raise credentials_exception
```

Даже если подпись токена верна, мы всё равно проверяем пользователя в базе. Зачем? Если пользователя удалили или деактивировали после выдачи токена — токен ещё «живой» (не истёк), но пользователь уже не должен иметь доступа. DB-проверка это поймает.

**3. Собрать контекст пользователя:**

```python
user_context = UserContext(
    user_id=user.id,
    company_id=active_company_id,
    company_ids=company_ids,
    is_holding_owner=is_holding_owner,
)
return user, user_context
```

Возвращается пара: сам пользователь + контекст (в какой компании он сейчас работает). Это используется для фильтрации данных — пользователь видит только данные своей компании.

## Как это выглядит в эндпоинте

Любой защищённый эндпоинт использует `Depends(get_current_user)`:

```python
@router.get("/me")
async def me(
    current_user: User = Depends(get_current_user_only),
) -> UserRead:
    return UserRead.model_validate(current_user)
```

`Depends(get_current_user_only)` — FastAPI автоматически вызывает функцию проверки токена перед вызовом функции эндпоинта. Если токен невалидный — эндпоинт вообще не запускается, сразу возвращается 401. Если токен верный — `current_user` содержит объект пользователя из базы.

## Создание токена в `create_access_token`

Посмотрим на полный код в `security.py`:

```python
def create_access_token(
    sub: str,
    role: str,
    company_ids: list[int] | None = None,
    is_holding_owner: bool = False,
) -> str:
    settings = get_settings()
    secret = settings.jwt_secret_key

    now = datetime.now(tz=UTC)
    expire = now + timedelta(minutes=settings.jwt_expire_minutes)  # 60 минут

    payload: dict[str, object] = {
        "sub": sub,                         # email пользователя
        "role": role,                       # его роль
        "company_ids": company_ids or [],   # к каким компаниям имеет доступ
        "is_holding_owner": is_holding_owner,
        "iat": now,                         # когда выдан
        "exp": expire,                      # когда истекает
    }

    token: str = jwt.encode(payload, secret, algorithm=settings.jwt_algorithm)
    return token
```

`jwt.encode` — создаёт строку JWT: кодирует header и payload в base64, вычисляет подпись HMAC-SHA256, склеивает через точки.

Время жизни токена — 60 минут. После этого нужно войти снова. В планах M-OS-1 — добавить refresh-токены (токены обновления, которые живут дольше и позволяют получить новый access-токен без повторного ввода пароля).

## Риск: утечка JWT_SECRET_KEY

Если кто-то узнает `JWT_SECRET_KEY`, он может:

1. Взять любой существующий JWT-токен (например, из трафика или логов)
2. Изменить payload: заменить `"role": "accountant"` на `"role": "owner"`, или поменять `"sub"` на email администратора, или установить `"is_holding_owner": true`
3. Вычислить новую правильную подпись, используя украденный ключ
4. Предъявить этот токен серверу — сервер примет его как настоящий

Последствие: полный доступ к системе с любыми правами, без знания паролей.

Именно поэтому:
- `JWT_SECRET_KEY` — минимум 32 случайных символа
- Хранится только в `.env`, никогда в коде
- Блокируется словарные значения валидатором в `config.py`

## Полная схема: от входа до запроса

```
Браузер                                    Сервер M-OS
   |                                           |
   |-- POST /auth/login {email, password} ---> |
   |                                           | 1. Ищет User в БД по email
   |                                           | 2. verify_password(password, hash)
   |                                           | 3. create_access_token(...)
   |<---------- {access_token: "eyJ..."} ------| 
   |                                           |
   | [сохранил токен]                          |
   |                                           |
   |-- GET /projects                           |
   |   Authorization: Bearer eyJ... ---------->|
   |                                           | 1. decode_token(token) → payload
   |                                           | 2. Находит User в БД
   |                                           | 3. Проверяет is_active
   |                                           | 4. Собирает UserContext
   |                                           | 5. Выполняет эндпоинт
   |<---------- {items: [...]} ----------------| 
```

## Как объяснить на совещании

«Когда сотрудник входит в систему — он называет email и пароль. Сервер проверяет пароль через bcrypt, без хранения самого пароля. Если всё верно — выдаёт временный пропуск на 60 минут (JWT-токен). Каждый следующий запрос в систему сопровождается этим пропуском. Сервер проверяет подпись пропуска и узнаёт: кто это, его роль, к каким компаниям есть доступ. Без пропуска — нельзя ничего. С чужим пропуском — тоже нельзя, потому что подпись проверяется по секретному ключу, который знает только сервер.»
