## Коротко о главном

**React** — это библиотека (готовый инструментарий) для построения пользовательского интерфейса. Написана компанией Meta (Facebook) в 2013 году, сегодня является стандартом индустрии. В M-OS весь интерфейс администратора — таблицы юридических лиц, формы, дашборды — написан на React.

Главная идея React: интерфейс собирается из **компонентов** (component) — независимых кирпичей, каждый из которых отвечает за свой кусок экрана.

---

## Зачем это нужно

До React веб-страницы писали иначе: один большой HTML-файл, внутри которого JavaScript напрямую находил нужные элементы и менял их. Если пользователь нажал кнопку — JavaScript ищет нужную строку таблицы по идентификатору, меняет текст, обновляет счётчик. При росте проекта это превращалось в хаос: любое изменение ломало что-то в другом месте, и понять почему — почти невозможно.

React решает эту проблему. Вместо «найди элемент и измени его» — «опиши, как должен выглядеть экран при данных условиях, а React сам разберётся что обновить».

Для M-OS это критично: у нас будет десятки экранов — объекты, договоры, платежи, сотрудники. Без системного подхода поддерживать это нереально.

## Аналогия: строительные модули вместо монолита

Представьте два способа строить коттеджный посёлок.

**Способ 1 (до React):** каждый дом строится с нуля, полностью уникальный. Хотите изменить стандарт окон — переделываете каждый дом вручную. 85 домов — 85 отдельных переделок.

**Способ 2 (React):** вы описываете типовой модуль «окно», «дверь», «кровля». Каждый дом собирается из этих модулей. Хотите изменить стандарт окон — меняете описание одного модуля, и все дома обновляются автоматически.

React работает по второму принципу. `StatusBadge` — модуль «значок статуса». Он используется в таблице, в карточке компании, в отчёте. Описан один раз, работает везде.

## Конкретика: что есть в нашем проекте

Откройте файл `frontend/src/pages/admin/companies/CompaniesListPage.tsx`. Это страница списка юридических лиц холдинга.

### Компонент (Component)

**Компонент** — это функция, которая возвращает описание того, что нужно показать на экране.

```tsx
function StatusBadge({ isActive }: { isActive: boolean }) {
  return isActive ? (
    <Badge className="bg-green-100 text-green-800">
      Активна
    </Badge>
  ) : (
    <Badge variant="secondary">
      Неактивна
    </Badge>
  )
}
```

Это компонент `StatusBadge`. Он принимает одно значение — `isActive` (активна ли компания) — и возвращает зелёный или серый значок. Используется в каждой строке таблицы.

Обратите внимание: функция возвращает что-то похожее на HTML — с тегами `<Badge>`. Это и есть **JSX**.

### JSX

**JSX** (JavaScript XML — смесь JavaScript и разметки) — это специальный синтаксис, который позволяет писать описание интерфейса прямо внутри кода. Браузер JSX не понимает — перед запуском специальная программа (компилятор) переводит JSX в обычный JavaScript.

Аналогия: JSX — это как чертёж в AutoCAD. Архитектор рисует в удобном для себя формате, а программа строит из этого реальную документацию.

```tsx
// JSX — то, что пишет разработчик
<Badge className="bg-green-100">Активна</Badge>

// То, во что это превращается после компиляции (упрощённо)
React.createElement(Badge, { className: "bg-green-100" }, "Активна")
```

Первый вариант читать человеку гораздо проще.

### TypeScript

Вы заметили запись `{ isActive }: { isActive: boolean }`. Это **TypeScript** — расширение JavaScript с указанием типов данных.

`isActive: boolean` означает: «параметр `isActive` должен быть булевым значением (да/нет)». Если кто-то передаст туда строку или число — программа сразу сообщит об ошибке ещё до запуска. Это как контракт: оба участника точно знают, что передаётся и что ожидается.

Весь M-OS написан на TypeScript, а не на чистом JavaScript — именно для такой защиты.

### Props (свойства)

**Props** (от properties — свойства) — это данные, которые компонент получает снаружи. Аналогия: параметры задания, которые вы передаёте подрядчику.

```tsx
// Родительский компонент передаёт props
<StatusBadge isActive={company.is_active} />

// StatusBadge получает props и использует
function StatusBadge({ isActive }) {
  // isActive — это то, что нам передали
}
```

Props текут в одну сторону: от родителя к ребёнку. Это называется **однонаправленный поток данных** (one-way data flow). Компонент не может изменить props, которые ему передали — только использовать их.

### State (состояние)

**State** (состояние) — это данные, которые компонент хранит и может менять сам. Аналогия: внутренняя бухгалтерия отдела. Заказчику она не видна, но влияет на то, что отдел выдаёт наружу.

В нашей странице:

```tsx
const [searchInput, setSearchInput] = useState(search)
```

`searchInput` — это state: текст, который пользователь вводит в поле поиска прямо сейчас, до нажатия Enter. Пока он набирает, state меняется при каждом символе. Это локальное дело компонента.

`useState` — это хук (hook, специальная функция React), который создаёт state и возвращает два значения:
- текущее значение (`searchInput`)
- функцию для его изменения (`setSearchInput`)

Когда `setSearchInput` вызывается — React автоматически перерисовывает компонент с новым значением. Разработчик не занимается обновлением DOM (Document Object Model — дерево элементов страницы) вручную.

### Виртуальный DOM

**DOM** — это внутреннее представление HTML-страницы в браузере. Дерево элементов: `<html>` → `<body>` → `<table>` → `<tr>` → `<td>`.

**Виртуальный DOM** — это копия этого дерева, которую React хранит в памяти. Когда что-то меняется:

1. React строит новый виртуальный DOM
2. Сравнивает его со старым (алгоритм называется **reconciliation** — согласование)
3. Находит только изменившиеся части
4. Обновляет в реальном DOM только их

Аналогия: вместо того чтобы переделать весь отчёт при исправлении одной цифры — выделить только изменённую ячейку и исправить её. React делает то же самое с интерфейсом — это делает его быстрым даже при больших таблицах.

### Пример целого компонента: SkeletonRows

```tsx
function SkeletonRows() {
  return (
    <>
      {Array.from({ length: 4 }).map((_, i) => (
        <TableRow key={i} aria-hidden="true">
          <TableCell><Skeleton className="h-4 w-40" /></TableCell>
          <TableCell><Skeleton className="h-4 w-28" /></TableCell>
          <TableCell><Skeleton className="h-4 w-12" /></TableCell>
          <TableCell><Skeleton className="h-4 w-16" /></TableCell>
          <TableCell><Skeleton className="h-4 w-16" /></TableCell>
        </TableRow>
      ))}
    </>
  )
}
```

Это компонент-заглушка: он показывает серые «кости» таблицы, пока данные ещё грузятся. `Array.from({ length: 4 })` создаёт массив из 4 пустых элементов, `.map()` превращает их в 4 строки-скелетона. Пользователь видит структуру страницы сразу, без пустого экрана.

### Как это объяснить на совещании

> «У нас интерфейс собран из независимых блоков — компонентов. Значок статуса компании, строка таблицы, кнопка — каждый блок живёт отдельно. Меняем один блок — меняется везде, где он используется. Это даёт скорость разработки и защиту от ошибок: если блок сломан, сразу понятно где именно, а не в каком-то общем "котле"».

## Риски

### Over-rendering (избыточная перерисовка)

Если state меняется слишком часто — React перерисовывает компонент слишком часто. При большой таблице это замедляет интерфейс. В нашей странице поле поиска обновляет `searchInput` при каждом нажатии клавиши — но реальный запрос к серверу идёт только по Enter. Это правильное решение.

Признак проблемы: интерфейс «подтормаживает» при быстром вводе или скролле.

### Prop drilling (сквозная передача свойств)

Если компонент A хочет передать данные компоненту D, который вложен глубоко (A → B → C → D) — приходится передавать props через B и C, которые эти данные не используют. Это называется prop drilling — «протаскивание props» через слои.

Решение — специальные механизмы для «глобального» состояния (Context API, Zustand). В M-OS это будет актуально при реализации профиля пользователя и текущей компании.

### Компонент без ключа (key)

В списках React требует атрибут `key` у каждого элемента — уникальный идентификатор. Без него React не понимает, какой элемент изменился при обновлении, и перерисовывает весь список:

```tsx
// Правильно: key по id из базы данных
data?.items.map((company) => (
  <TableRow key={company.id}>...</TableRow>
))

// Неправильно: key по индексу массива (порядковый номер)
// При сортировке или удалении React запутается
items.map((item, i) => <Row key={i} />)
```

В нашей таблице компаний ключ — `company.id`. Это правильно.
