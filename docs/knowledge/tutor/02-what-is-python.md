## Зачем это важно для M-OS

Каждый технологический выбор в M-OS зафиксирован в документе ADR 0002 (Architecture Decision Record — «запись об архитектурном решении»). Язык программирования — это самый фундаментальный выбор: он определяет, насколько просто читать код, насколько быстро команда ИИ-субагентов генерирует качественный код, и насколько легко будет передать проект другой команде через год. Понимание этого выбора помогает Вам участвовать в стратегических решениях по технологиям.

---

## Аналогия: инструмент под задачу

У прораба на стройке разные инструменты для разных задач. Перфоратор — для бетона. Лобзик — для дерева. Болгарка — для металла.

Никто не пытается забить гвоздь болгаркой — инструмент должен подходить задаче. Точно так же языки программирования выбираются под задачу:

| Инструмент | Язык | Лучше всего для |
|---|---|---|
| Универсальный шуруповёрт | Python | Бизнес-логика, скрипты, ИИ, веб-API |
| Прецизионный токарный станок | C++ / Rust | Драйверы, игры, высокая производительность |
| Кран (тяжело, надёжно) | Java | Банки, корпоративные системы с тысячами серверов |
| Быстрый пневмоинструмент | Go | Микросервисы, высоконагруженные сети |
| Широкий шпатель (везде понемногу) | PHP | Старые сайты, WordPress |

Python — это как хороший универсальный шуруповёрт: подходит для 90% задач, прост в обращении, и новый сотрудник разберётся без месяца обучения.

---

## Что такое язык программирования

Язык программирования — это набор правил, по которым пишется код. Точно так же, как русский язык имеет грамматику, пунктуацию и словарный запас — Python имеет свои правила написания команд, которые компьютер понимает.

Разные языки — разный синтаксис (внешний вид кода), разные возможности, разные экосистемы (готовые библиотеки инструментов).

## Конкретика: Python в нашем проекте

Вот реальный файл из проекта — `backend/app/api/health.py` (весь файл, 32 строки):

```python
"""Роутер проверки живости сервиса (health check).

Эндпоинт /api/v1/health используется:
- балансировщиком нагрузки / Docker health check для проверки, что сервис запущен;
- мониторингом (Uptime Robot, Grafana и т.п.);
- разработчиком для быстрой проверки, что backend отвечает.
"""

from fastapi import APIRouter

from app import __version__

router = APIRouter(tags=["health"])

@router.get("/health")
async def health_check() -> dict[str, str]:
    """Возвращает статус «ok» и версию приложения."""
    return {"status": "ok", "version": __version__}
```

Прочитайте его как обычный текст. Вы уже понимаете большую часть:
- Первый блок в тройных кавычках — это документация (комментарий для людей, не для компьютера)
- `from fastapi import APIRouter` — «возьми инструмент APIRouter из библиотеки fastapi»
- `def health_check()` — «объяви функцию с именем health_check»
- `return {"status": "ok", ...}` — «верни ответ: статус ок, версия такая-то»

Это весь код, который отвечает на запрос «жив ли сервер?». Никакого «программистского шаманства» — обычный читаемый текст.

## Почему Python, а не другие языки

Из ADR 0002 (официальное решение по стеку) — три главные причины:

### 1. Читаемость

Python специально проектировался так, чтобы код читался почти как человеческий язык. Сравните:

**Python:**
```python
if user.is_active and user.role == "OWNER":
    allow_access()
```

**Java (то же самое):**
```java
if (user.getIsActive() == true && user.getRole().equals(UserRole.OWNER)) {
    allowAccess();
}
```

**C++ (то же самое, упрощённо):**
```cpp
if (user->isActive && user->role == UserRole::OWNER) {
    allowAccess();
}
```

Python-код ближе всего к тому, как Вы бы описали логику на совещании словами.

### 2. Качество ИИ-генерации

Наша команда — 17 ИИ-субагентов. Качество кода, который они генерируют, прямо зависит от того, сколько примеров на этом языке было в их обучающих данных. Python — самый популярный язык в мире по индексу TIOBE 2024-2025. Субагенты генерируют Python-код лучше всего, с наименьшим числом ошибок. Это прямо влияет на скорость и стоимость разработки.

### 3. Экосистема для нашего типа задач

Python имеет готовые библиотеки (наборы инструментов) для всего, что нужно M-OS:
- Веб-API — FastAPI
- База данных — SQLAlchemy
- Валидация данных — Pydantic
- Тесты — pytest
- В будущем: ИИ и аналитика — pandas, scikit-learn, OpenAI SDK

## Что такое FastAPI

FastAPI — это фреймворк (каркас, рамочная конструкция) поверх Python. Аналогия: Python — это бетон и арматура, а FastAPI — это готовая форма-опалубка, которая позволяет отлить правильную конструкцию без изобретения формы заново.

FastAPI автоматически делает три вещи, которые без него пришлось бы писать вручную:

**1. Маршрутизация** — решает, какой код вызвать при каком запросе. Пришёл запрос `GET /health` → вызвать `health_check()`.

**2. Валидация** — проверяет входящие данные. Кто-то отправил данные без обязательного поля `email`? FastAPI автоматически вернёт ошибку с понятным описанием.

**3. Документация** — генерирует интерактивный справочник по всем API прямо из кода. Откройте `http://localhost:8000/docs` — и увидите все функции системы с описанием, можно тестировать прямо в браузере. Это доступно уже сейчас в нашем проекте.

## Что такое библиотека

Библиотека — это готовый набор кода, написанный кем-то другим, который можно использовать в своём проекте. Аналог: купить готовое окно у завода, а не точить его самому из стекла.

В нашем проекте используется более 15 библиотек. Посмотрите на начало любого Python-файла — строки `from ... import ...` — это подключение библиотек:

```python
# Из backend/app/services/audit.py, строки 1-4
from typing import Any
from sqlalchemy.ext.asyncio import AsyncSession
from app.models.audit import AuditLog
from app.models.enums import AuditAction
```

- `typing` — стандартная библиотека Python для описания типов данных
- `sqlalchemy.ext.asyncio` — инструменты для работы с базой данных в асинхронном режиме
- `app.models.audit` — наш собственный код из другого файла проекта

## Почему не PHP

PHP — старый язык, доминировавший в вебе в 2000-х (WordPress, VKontakte писался на PHP). Для нас его минусы критичны:

- Устаревший синтаксис — код читается хуже
- Слабая типизация — ИИ-субагенты генерируют PHP с большим числом ошибок
- Нет нативной асинхронности — для нашего стека это ограничение
- Экосистема для ИИ/аналитики отсутствует — а M-OS это потребует в фазах M-OS-2 и далее

---

## Почему не Java

Java — надёжный язык для корпоративного ПО. Но:

- Многословный: «Hello, World» на Java — 5 строк, на Python — 1 строка
- Высокий порог входа: концепции классов, интерфейсов, сборщиков проектов (Maven/Gradle) — тяжело для Владельца без опыта
- Медленная разработка: Java требует «компиляции» перед запуском, итерации дольше
- Для нашего объёма данных производительность Java не даёт ощутимого преимущества

---

## Почему не Go

Go — современный язык от Google, очень быстрый. Но:

- Строгая минималистичная философия: многие вещи приходится писать «руками», нет готовых ORM и инструментов уровня SQLAlchemy
- Меньше примеров в обучающих данных ИИ — субагенты генерируют Go-код хуже
- Нет экосистемы для ИИ/аналитики
- Читаемость хуже Python для новичка

## Риски и типичные ошибки

**1. «Python медленный — надо на Go/Rust».**
Python медленнее Java и Go по сырой вычислительной мощности. Но для нашего объёма данных (85 домов, тысячи транзакций, 10 пользователей) это несущественно. Узкое место системы — запросы к базе данных, а не скорость Python. Переход на другой язык сейчас — это месяц работы ради нулевого прироста в продуктивности.

**2. «Давайте добавим ещё одну библиотеку».**
Каждая новая библиотека — зависимость. Она может выпустить несовместимое обновление, перестать поддерживаться, содержать уязвимость. Принцип M-OS: новая библиотека добавляется только если нет способа решить задачу имеющимися средствами. Это архитектурное решение, а не рутина.

**3. «Мы перейдём на другой язык позже».**
Смена языка программирования — это фактически переписывание проекта с нуля. Решение, принятое в ADR 0002, определяет весь стек на горизонте минимум 3-5 лет. Пересматривается только при появлении критических, а не вкусовых причин.

## Проверка понимания

**Вопрос 1.** Коллега говорит: «В другом проекте у нас был PHP — давайте добавим модуль на PHP для отчётности». Почему это предложение требует отдельного архитектурного решения (ADR), а не просто реализации?

> Добавление второго языка — это не просто «ещё код». Это новая технологическая зависимость, новые требования к команде, отдельная среда исполнения, отдельная система деплоя. Смешивание языков в одном проекте резко увеличивает сложность поддержки. В M-OS Python выбран именно для того, чтобы весь backend-код был в единой технологической рамке.

**Вопрос 2.** Что даёт FastAPI помимо возможности написать веб-API?

> FastAPI автоматически генерирует интерактивную документацию (Swagger UI) — Владелец видит все функции системы в браузере. Также встроенная валидация входящих данных через Pydantic: если кто-то отправил запрос с ошибкой, FastAPI автоматически вернёт описание проблемы, а не упадёт с непонятной ошибкой.

**Вопрос 3.** В чём разница между «языком программирования» и «фреймворком»?

> Язык — это правила написания кода (Python). Фреймворк — это готовый каркас на этом языке, решающий конкретный класс задач (FastAPI — для веб-API). Аналогия: язык — это бетон, фреймворк — это опалубка нужной формы.
