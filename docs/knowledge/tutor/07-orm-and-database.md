# Урок 7. ORM и база данных: как Python разговаривает с PostgreSQL

**Блок 2 — Архитектура Backend** | Время чтения: ~16 минут

---

## Что Вы поймёте после этого урока

После этого урока Вы будете знать: что такое ORM (Object-Relational Mapper), зачем он нужен, чем отличается от прямых SQL-запросов, как SQLAlchemy (наш ORM) соединяет Python-код с базой данных PostgreSQL, и почему изменение Python-класса не означает автоматического изменения таблицы в базе.

---

## Зачем это важно для M-OS

Всё, что хранится в M-OS — проекты, платежи, договоры, пользователи — живёт в базе данных PostgreSQL. Python-код работает с этими данными через SQLAlchemy. Каждый раз, когда в отчёте написано «сервис создал запись» или «запрос к базе вернул ошибку» — это работа ORM. Понимание ORM позволяет Вам задавать правильные вопросы: почему запрос медленный, почему данные не обновились, откуда взялся дубликат.

---

## Аналогия: переводчик на переговорах

Представьте переговоры между российской и японской компаниями. Российский директор говорит по-русски. Японский партнёр понимает только японский. Между ними — **переводчик**.

Директор не учит японский. Он говорит переводчику: «Скажи им, что мы готовы поставить 500 тонн щебня в июне». Переводчик знает оба языка и передаёт смысл на японском.

Именно так работает ORM:

| Переговоры | ORM |
|---|---|
| Российский директор | Python-код |
| Японский партнёр | База данных PostgreSQL |
| Переводчик | SQLAlchemy (ORM) |
| Русская речь | Python-объекты и методы |
| Японская речь | SQL-запросы |

Python-разработчик не пишет SQL вручную — он работает с Python-объектами. ORM переводит это в SQL, отправляет в базу, получает результат и переводит обратно в Python-объекты.

---

## Что такое SQL и зачем он вообще нужен

**SQL** (Structured Query Language — язык структурированных запросов) — это язык, на котором разговаривают с базами данных. PostgreSQL его понимает напрямую.

Вот как выглядит «дайте мне список активных проектов» на SQL:

```sql
SELECT id, code, name, status
FROM projects
WHERE deleted_at IS NULL
  AND status != 'archived'
ORDER BY created_at DESC
LIMIT 20 OFFSET 0;
```

А вот как это же выглядит в Python-коде с SQLAlchemy:

```python
projects = await session.execute(
    select(Project)
    .where(Project.deleted_at.is_(None))
    .where(Project.status != "archived")
    .order_by(Project.created_at.desc())
    .limit(20)
    .offset(0)
)
```

Результат одинаковый. Разница — в языке. Первый вариант — SQL. Второй — Python через ORM.

---

## Зачем ORM, если SQL и так работает

Можно писать SQL-запросы прямо в Python — это называется «raw SQL» (сырой SQL). Это работает. Но у ORM есть существенные преимущества:

**1. Безопасность от SQL-инъекций.**
SQL-инъекция — это атака, когда злоумышленник подсовывает вместо значения кусок SQL-команды. Например, вместо email вводит: `'; DROP TABLE users; --`. Если вставить это прямо в строку SQL — таблица удалится.

ORM автоматически «экранирует» (обезвреживает) все значения. Вам не нужно думать об этом — защита встроена.

**2. Читаемость.**
Python-код с ORM читается как описание намерения: «найди пользователей с ролью owner, у которых is_active = True». SQL-код — более низкоуровневый, его труднее читать неспециалисту.

**3. Переносимость.**
Если завтра решим сменить PostgreSQL на другую базу данных — большая часть кода не изменится, поменяется только драйвер подключения. С raw SQL пришлось бы переписывать запросы.

**4. Автоматическое управление соединениями.**
SQLAlchemy управляет «пулом соединений» с базой (pool — запас заранее открытых соединений). Открывать новое соединение для каждого запроса — дорого по времени. ORM держит пул и выдаёт готовое соединение по требованию.

## Реальный файл: `backend/app/db/session.py`

Этот файл — точка входа в базу данных для всего приложения.

```python
engine = create_async_engine(
    _settings.database_url,
    pool_pre_ping=True,
    pool_size=5,
    max_overflow=10,
)
```

Разбор:

- `create_async_engine` — создаёт «движок» подключения. Движок знает адрес базы, логин, пароль. Само соединение ещё не открыто — только описание, как подключаться.
- `database_url` — строка вида `postgresql+psycopg://user:password@host:5432/dbname`. Берётся из переменных окружения (`.env`-файл), не из кода — это требование безопасности.
- `pool_size=5` — держать 5 постоянно открытых соединений. Вместо того чтобы каждый раз открывать новое (медленно) — берём готовое из «запаса».
- `max_overflow=10` — если нагрузка выросла, можно открыть ещё до 10 временных соединений сверх основных.
- `pool_pre_ping=True` — перед отдачей соединения из пула проверить, что оно живое. Защита от ситуации «база перезапустилась, а соединение в пуле устарело».

```python
async def get_db() -> AsyncGenerator[AsyncSession, None]:
    async with AsyncSessionLocal() as session:
        yield session
```

Это функция-зависимость, которую Вы уже видели в уроке 6. Она открывает «сессию» (session — сеанс работы с базой в рамках одного HTTP-запроса) и закрывает её по завершении. `yield` — это специальная конструкция Python, которая говорит: «отдай сессию вызывающему коду, а после его завершения продолжи и закрой».

**Сессия** — это единица работы. В пределах одной сессии можно сделать несколько запросов, и они будут частью одной «транзакции». Если что-то пошло не так — транзакция откатывается целиком, ни одно изменение не сохраняется.

## Реальный файл: `backend/app/services/project.py`

Посмотрим, как сервис использует ORM для создания проекта:

```python
async def create(self, data: ProjectCreate, actor_id: int, ...) -> Project:
    existing = await self.repo.get_by_code(data.code)
    if existing is not None:
        raise ConflictError(...)

    project = await self.repo.create(data.model_dump())

    await self.audit.log(...)

    return project
```

Здесь `self.repo` — это репозиторий (repository — слой доступа к данным). Репозиторий — ещё один «переводчик», но уже между сервисом и ORM. Сервис говорит: «создай проект с такими данными» — репозиторий делает вызов SQLAlchemy.

Внутри `self.repo.create(data.model_dump())` происходит примерно следующее:

```python
# Внутри репозитория (упрощённо):
obj = Project(**data)        # создаём Python-объект
session.add(obj)             # говорим SQLAlchemy: «добавь это в сессию»
await session.flush()        # SQLAlchemy переводит в SQL: INSERT INTO projects (...)
return obj                   # возвращаем объект с id, который назначила база
```

На уровне базы данных PostgreSQL получает:

```sql
INSERT INTO projects (code, name, status, company_id, created_at, updated_at)
VALUES ('C56', 'Координата 56', 'active', 1, now(), now())
RETURNING id, code, name, status, company_id, created_at, updated_at;
```

ORM сгенерировал этот SQL сам. Разработчик его не писал.

## Транзакция: всё или ничего

В уроке 6 Вы видели в роутере такую строку:

```python
await db.commit()
```

Это «фиксация транзакции». До этой строки все изменения существуют только в памяти — в «сессии». Если произошла ошибка до `commit()` — ничего не сохранится. Если `commit()` прошёл — всё зафиксировано в базе.

Аналогия из строительства: смета и приказ о начале работ подготовлены (это flush/add), но стройка официально начата только после подписи директора (это commit). Если директор не подпишет — всё отменяется.

В M-OS commit делается **в роутере**, а не в сервисе. Сервис только выполняет бизнес-логику — он не знает, будет ли commit. Это позволяет роутеру откатить всё, если что-то пошло не так между несколькими вызовами сервисов.

---

## Что ORM НЕ делает автоматически

Это важно понять, чтобы не ошибиться.

**ORM не синхронизирует Python-класс с таблицей в БД автоматически.**

Если разработчик добавил новое поле в Python-класс `Project`:

```python
class Project(Base, TimestampMixin):
    ...
    budget_limit: Mapped[int | None] = mapped_column(Integer, nullable=True)  # новое поле
```

В базе данных колонка `budget_limit` **не появится сама**. Для этого нужна **миграция** — специальный файл с инструкциями для Alembic (инструмент управления схемой БД). Миграции — тема следующего урока 10.

Если поле добавить в класс, но не создать миграцию — при попытке записать данные в это поле будет ошибка: «колонки не существует».

## Пример своими словами

На совещании Вы можете сказать так:

> «SQLAlchemy — это переводчик между нашим Python-кодом и базой данных. Разработчик описывает действие на Python: «найди проекты компании 1». ORM переводит это в SQL и отправляет в PostgreSQL. Это удобнее, безопаснее и быстрее, чем писать SQL вручную. Но важно понимать: изменение Python-класса без миграции не меняет реальную таблицу в базе. Эти вещи нужно делать синхронно.»

## Риски и типичные ошибки

**1. Забыть commit.**
Данные созданы в сессии, но `commit()` не вызван. При закрытии сессии всё потеряется. В нашем проекте это правило зафиксировано: `commit()` — в роутере, после всех вызовов сервисов.

**2. N+1 запросов.**
Классическая проблема ORM: получили список из 50 проектов, и потом для каждого отдельно запрашиваем компанию. Итого 51 запрос вместо одного с JOIN. Это называется «проблема N+1» — резко замедляет работу. Решение: указывать SQLAlchemy загружать связанные объекты вместе (`joinedload`, `selectinload`).

**3. Держать сессию слишком долго.**
Сессия занимает соединение из пула. Если долго держать открытую сессию (например, из-за медленного кода после запроса к БД) — пул исчерпается и новые запросы встанут в очередь. В нашем проекте сессия открывается только на время одного HTTP-запроса.

**4. Изменить Python-класс без миграции.**
Как описано выше: класс и таблица разойдутся. Приложение будет работать с устаревшей схемой, запросы начнут падать с ошибками типа «column does not exist».

## Проверка понимания

**Вопрос 1.** Разработчик говорит: «Я добавил поле `region` в модель Project, но при сохранении нового проекта получаю ошибку от базы данных». Что, скорее всего, не сделано?

> Не создана и не применена миграция Alembic. Python-класс описывает новое поле, но в реальной таблице PostgreSQL колонки `region` ещё нет.

**Вопрос 2.** Чем `session.flush()` отличается от `session.commit()`?

> `flush()` переводит изменения в SQL и отправляет в базу в рамках текущей транзакции, но не фиксирует их — данные ещё можно откатить. `commit()` окончательно фиксирует транзакцию: данные сохранены в базе навсегда и видны другим запросам.

**Вопрос 3.** Почему в M-OS разработчики используют SQLAlchemy, а не пишут SQL-запросы напрямую?

> Три причины: автоматическая защита от SQL-инъекций, более читаемый Python-код, управление пулом соединений. Кроме того, при смене базы данных бо́льшая часть кода останется неизменной.
