## Зачем это важно для M-OS

Весь backend M-OS построен из функций. Каждый раз, когда прораб создаёт закупку материала, система вызывает функцию `create()` в сервисе. Каждый раз, когда нужно записать действие в журнал аудита — вызывается функция `log()`. Понимая, как работают переменные и функции, Вы понимаете, как устроена любая операция в системе.

## Аналогия: бригадир и наряд на работу

Представьте бригадира на стройке.

**Бригадир** — это функция. Он существует независимо от конкретной задачи. Его можно вызвать много раз.

**Наряд на работу** — это параметры (входные данные функции). В нём написано: «Дом №12, Фундамент, 5 кубов бетона».

**Акт выполненных работ** — это возвращаемое значение. Бригадир сделал работу и вернул документ: «Готово, уложено 4,8 куба».

Можно вызвать одного и того же бригадира на разных объектах с разными нарядами — и каждый раз получить другой акт. Бригадир один, задачи разные, результаты разные.

```
ВЫЗОВ БРИГАДИРА (функции):

Прораб:  "Бригадир! Дом №12, Фундамент, 5 кубов бетона."
         ↓ (параметры: дом=12, этап=Фундамент, объём=5)

Бригадир: [выполняет работу]

Бригадир: "Готово. Уложено 4.8 куба. Акт №47."
          ↑ (возвращаемое значение)
```

## Что такое переменная

**Переменная** — это именованная ячейка памяти. Вы даёте данным имя, и под этим именем можете к ним обращаться.

Аналогия из бизнеса: ячейка в Excel. Ячейка B3 содержит значение «1 500 000». Вы называете её «бюджет_фундамента» — и теперь можно написать формулу, которая ссылается на «бюджет_фундамента» вместо «B3». Если значение изменилось — меняете в одном месте, а не во всех формулах.

```python
# Переменная: имя слева, значение справа
budget = 1500000        # число
stage_name = "Фундамент"  # текст
is_completed = False    # да/нет

# Обращение к переменной по имени:
print(budget)           # выведет: 1500000
print(stage_name)       # выведет: Фундамент
```

Переменная — не постоянная. Её можно изменить:

```python
budget = 1500000
budget = budget + 200000  # прибавили 200 тысяч
# теперь budget = 1700000
```

## Конкретика: переменные в нашем коде

Откроем `backend/app/services/audit.py`, строки 20-21:

```python
AUDIT_EXCLUDED_FIELDS: frozenset[str] = frozenset(
    {"password_hash", "password", "token", "secret", "key", "jwt"}
)
```

Здесь:
- `AUDIT_EXCLUDED_FIELDS` — имя переменной (заглавные буквы означают «константа» — не менять)
- `frozenset(...)` — неизменяемое множество значений (как список, но нельзя добавить/удалить элемент)
- `{"password_hash", "password", "token", "secret", "key", "jwt"}` — сами значения: имена полей, которые нельзя писать в журнал аудита

**Зачем эта переменная?** Если бы вместо неё везде в коде писали `{"password_hash", "password", ...}` напрямую — это нужно было бы менять в 10 местах при добавлении нового запрещённого поля. Теперь меняем в одном месте.

## Что такое функция

**Функция** — это именованный блок кода, который можно вызвать по имени. Функция:
1. Принимает входные данные (параметры)
2. Делает что-то внутри
3. Возвращает результат (или не возвращает — просто делает действие)

Синтаксис Python:

```python
def имя_функции(параметр1, параметр2):
    # тело функции
    результат = параметр1 + параметр2
    return результат
```

Слово `def` — сокращение от «define» (объявить). `return` — «вернуть результат».

## Конкретика: функция `_sanitize` из нашего кода

Файл `backend/app/services/audit.py`, строки 25-34:

```python
def _sanitize(data: dict[str, Any]) -> dict[str, Any]:
    """Удаляет чувствительные поля из словаря перед записью в audit_log."""
    return {k: v for k, v in data.items() if k not in AUDIT_EXCLUDED_FIELDS}
```

Разберём по частям:

- `def _sanitize(` — объявляем функцию с именем `_sanitize` (подчёркивание в начале — сигнал: «только для внутреннего использования, не вызывать снаружи»)
- `data: dict[str, Any]` — параметр `data`, тип: словарь (пары ключ-значение). Подробнее о словарях — в уроке 4.
- `-> dict[str, Any]` — функция вернёт словарь (аннотация типа результата, для читаемости)
- Тело функции: берёт каждую пару `k: v` из `data`, оставляет только те, где ключ `k` не находится в запрещённом списке
- `return` — возвращает очищенный словарь

**Зачем эта функция?** Перед тем как записать в журнал аудита изменение данных пользователя — нужно удалить из записи пароль и токены. Эта функция делает именно это. Вместо того чтобы писать проверку в каждом месте — один раз описали функцию, вызываем везде.

## Параметры и аргументы

**Параметр** — имя слота во входных данных функции (при объявлении).
**Аргумент** — конкретное значение, которое передали при вызове.

Аналогия: параметр — это «слот Дом №» в наряде. Аргумент — конкретное значение «12» в этом слоте.

```python
# Объявление (параметры: data):
def _sanitize(data):
    ...

# Вызов (аргумент: конкретный словарь):
result = _sanitize({"email": "ivan@mail.ru", "password": "secret123"})
```

После вызова `result` будет содержать `{"email": "ivan@mail.ru"}` — пароль удалён.

## Конкретика: функция с несколькими параметрами

Откроем `backend/app/services/audit.py`, строки 50-58, метод `log`:

```python
async def log(
    self,
    user_id: int | None,
    action: AuditAction,
    entity_type: str,
    entity_id: int | None,
    changes: dict[str, Any] | None,
    ip_address: str | None = None,
    user_agent: str | None = None,
) -> None:
```

Параметров много — разберём:
- `self` — особый параметр, означает «эта функция принадлежит классу» (подробнее в уроке 8)
- `user_id: int | None` — ID пользователя: число или пусто (None = «ничего»)
- `action: AuditAction` — тип действия: CREATE, UPDATE, DELETE и т.д.
- `entity_type: str` — что меняли: "User", "BudgetPlan", "MaterialPurchase"
- `entity_id: int | None` — ID записи, которую меняли
- `changes: dict[str, Any] | None` — что именно изменилось (до и после)
- `ip_address: str | None = None` — IP-адрес; `= None` означает параметр необязательный, по умолчанию пусто
- `-> None` — функция ничего не возвращает, только записывает в базу

Реальный вызов этой функции из `backend/app/services/budget_plan.py`, строки 152-160:

```python
await self.audit.log(
    user_id=actor_id,
    action=AuditAction.CREATE,
    entity_type="BudgetPlan",
    entity_id=plan.id,
    changes={"after": _plan_to_dict(plan)},
    ip_address=ip,
    user_agent=user_agent,
)
```

Слово `await` перед вызовом означает «подожди, пока функция завершится» — это связано с асинхронностью (тема отдельного урока). Пока просто знайте, что оно там есть.

## Функция возвращающая результат и функция без результата

Есть два типа функций:

**Функция-вычислитель** — принимает данные, возвращает результат.

```python
# Из backend/app/services/material_purchase.py, строки 40-47
def _compute_total(quantity, unit_price_cents):
    """Вычисляет итоговую цену = количество × цена_за_единицу."""
    return int(round(quantity * unit_price_cents))
```

Передали количество и цену — получили итог. Функция не меняет базу данных, не пишет в журнал. Просто считает.

**Функция-действие** — делает что-то в мире (пишет в базу, отправляет письмо, записывает файл), ничего не возвращает.

```python
async def log(...) -> None:
    # пишет запись в таблицу audit_log
    self.session.add(entry)
    await self.session.flush()
    # нет return
```

## Почему функции — это важно для управления проектом

Каждая функция — это единица работы с именем. Когда в ревью написано «в функции `create()` сервиса `BudgetPlanService` нет проверки принадлежности дома проекту» — Вы знаете, о чём речь. Это конкретная функция в конкретном файле.

Функции также позволяют команде работать параллельно: один разработчик пишет функцию `create()`, другой — функцию `delete()`. Они не мешают друг другу, потому что функции независимы.

На совещании Вы можете сказать так:

## Риски и типичные ошибки

**1. Функция делает слишком много.**
Принцип M-OS (ADR 0004): каждая функция делает одну вещь. Функция, которая одновременно считает бюджет, отправляет уведомление и пишет в лог — это «бригадир, который сам копает, сам заливает и сам принимает работу». Если что-то сломалось — непонятно где. Поэтому в нашем коде `_sanitize()`, `_compute_total()`, `log()` — отдельные функции с одной ответственностью каждая.

**2. Забыть вызвать функцию.**
Из CLAUDE.md проекта: «Write-эндпоинт без `audit_service.log()` — это дефект». Это именно тот случай: забыли вызвать функцию аудита — и действие не записалось. Поэтому в код-ревью проверяется, что после каждой операции изменения данных вызывается `log()`.

**3. Дублировать логику вместо создания функции.**
Если в 5 местах одинаково вычисляется цена — это 5 мест, которые нужно обновить при изменении формулы. Функция `_compute_total()` решает эту проблему: меняем в одном месте.

## Проверка понимания

**Вопрос 1.** В чём разница между определением функции и её вызовом?

> Определение (`def log(...): ...`) — это объявление бригадира: «он существует, вот что он умеет». Происходит один раз при загрузке файла. Вызов (`await self.audit.log(...)`) — это конкретное задание бригадиру. Происходит каждый раз, когда нужно выполнить действие.

**Вопрос 2.** Функция `_sanitize` принимает словарь с данными пользователя, включая `password_hash`. Что она вернёт?

> Вернёт тот же словарь, но без поля `password_hash` — оно входит в `AUDIT_EXCLUDED_FIELDS`. В журнал аудита пароль не попадёт.

**Вопрос 3.** Почему в параметрах функции `log()` написано `ip_address: str | None = None`, а не просто `ip_address: str`?

> `| None` означает, что IP может отсутствовать (например, при системной операции без запроса от пользователя). `= None` — значение по умолчанию: если IP не передали явно, функция ведёт себя как будто его нет. Это делает вызов гибким — не нужно всегда передавать IP, только когда он есть.
