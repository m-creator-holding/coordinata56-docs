## Зачем это важно для M-OS

Пока система работает — в базе накапливаются реальные данные: проекты, платежи, договоры, пользователи. Структуру этой базы нельзя просто «стереть и пересоздать» при каждом обновлении кода — данные пропадут. При этом код постоянно развивается: добавляются поля, создаются новые таблицы, меняются связи. Миграции — это инструмент, который позволяет эволюционировать структуру базы данных безопасно, не теряя данных и не останавливая работу системы.

---

## Аналогия: переезд склада без остановки производства

Представьте металлообрабатывающий цех. Склад работает круглосуточно, производство не останавливается. Вы решаете расширить склад: добавить новый стеллаж для дорогостоящего инструмента, перенести часть номенклатуры в другую зону, изменить систему маркировки.

Нельзя просто закрыть склад на неделю — производство встанет. Поэтому:

1. Сначала строят новый стеллаж рядом со старым (не ломая ничего)
2. Постепенно перекладывают инструмент с маркировкой по новой системе
3. После полного переноса — убирают старый стеллаж

Каждый шаг можно отменить: если новый стеллаж оказался неудобным — вернули инструмент обратно, разобрали. Вся история изменений — в журнале.

Именно так работают миграции базы данных.

---

## Что такое Alembic

**Alembic** — инструмент управления миграциями для SQLAlchemy. Это отдельная программа (не часть Python-кода приложения), которая:

- Отслеживает текущую версию схемы базы данных
- Хранит историю всех изменений в виде Python-файлов
- Умеет применять изменения вперёд (`upgrade`) и откатывать назад (`downgrade`)
- Умеет сравнивать Python-модели с реальной схемой и генерировать файл миграции автоматически

Каждый файл миграции — это пара функций: `upgrade()` и `downgrade()`. Первая применяет изменения, вторая их отменяет.

## Реальные файлы: история миграций проекта

В нашем проекте миграции живут в `backend/alembic/versions/`. Вот их хронология:

```
2026-04-11  f80b758cadef  initial_schema              — первая схема (все базовые таблицы)
2026-04-15  a1b2c3d4e5f6  seed_initial_owner          — начальные данные: первый пользователь
2026-04-15  48b652e20e99  payment_status_enum         — добавлен статус платежа
2026-04-15  c9d1e2f3a4b5  budget_soft_delete          — мягкое удаление для бюджета
2026-04-15  d1e2f3a4b5c6  payment_approve_reject_audit — поля согласования платежей
2026-04-15  e1f2a3b4c5d6  contractor_inn_partial_unique — уникальность ИНН подрядчика
2026-04-16  ...           contract_contractor_number   — уникальность номера договора
2026-04-17  f7e8d9c0b1a2  multi_company_foundation    — мультикомпанийная архитектура
```

Каждая запись — один шаг в развитии системы. Alembic знает, на какой версии сейчас находится база, и применяет только те миграции, которые ещё не применялись.

Аналогия: журнал версий чертежей в архиве. «Версия 1.0 — исходный чертёж. Версия 1.1 — добавлена веранда. Версия 1.2 — изменён фундамент. Объект сейчас на версии 1.2».

## Детальный разбор: `f7e8d9c0b1a2_multi_company_foundation.py`

Это самая сложная миграция в нашем проекте. Она добавила поддержку нескольких юридических лиц (компаний) в систему. Разберём её по шагам.

### Что делала эта миграция

До неё: одна безымянная система, один контекст.
После: несколько юридических лиц (ООО, АО, ИП), каждый пользователь привязан к конкретной компании с конкретной ролью в ней.

### Шаг 1: создание новой таблицы `companies`

```python
op.create_table(
    "companies",
    sa.Column("id", sa.Integer(), autoincrement=True, nullable=False),
    sa.Column("inn", sa.String(12), nullable=True),
    sa.Column("full_name", sa.String(512), nullable=False),
    sa.Column("company_type", sa.String(20), nullable=False),
    sa.Column("is_active", sa.Boolean(), nullable=False, server_default=sa.text("TRUE")),
    sa.Column("created_at", sa.DateTime(timezone=True), nullable=False, server_default=sa.text("now()")),
    sa.PrimaryKeyConstraint("id", name="pk_companies"),
    sa.CheckConstraint("company_type IN ('ooo', 'ao', 'ip', 'other')", name="ck_companies_company_type"),
)
```

Новая таблица — безопасная операция. Существующие таблицы не затронуты.

### Шаг 2: создание таблицы `user_company_roles`

Таблица связей: какой пользователь в какой компании имеет какую роль.

Хитрость с уникальностью: стандартный UNIQUE-индекс в PostgreSQL пропускает несколько NULL-значений (NULL != NULL в SQL). Поэтому два частичных индекса:

```python
# Когда pod_id заполнен — уникальность по 4 полям
op.create_index("uq_ucr_user_company_role_pod", ..., postgresql_where=sa.text("pod_id IS NOT NULL"))

# Когда pod_id пустой — уникальность по 3 полям
op.create_index("uq_ucr_user_company_role_nopod", ..., postgresql_where=sa.text("pod_id IS NULL"))
```

### Шаг 3: seed-данные

```python
op.execute(
    sa.text("""
        INSERT INTO companies (id, inn, full_name, short_name, company_type, is_active)
        VALUES (1, NULL, 'Холдинг (по умолчанию)', 'Холдинг', 'ooo', TRUE)
    """)
)
```

Сразу после создания таблицы — заполняем её начальными данными. Компания с id=1 — это «Холдинг по умолчанию». Без этого следующий шаг (backfill) не сработает.

### Шаг 4: safe-migration для добавления `company_id`

Это ключевой момент урока. Нужно добавить колонку `company_id` в таблицы `projects`, `contracts`, `contractors`, `payments`. Но в этих таблицах уже есть данные. Если сразу добавить `company_id NOT NULL` — база откажет: «в существующих строках нет значения для этого поля».

Решение — три шага:

```python
# Шаг A: добавить как nullable (пустое разрешено)
op.add_column("projects",
    sa.Column("company_id", sa.Integer(),
              sa.ForeignKey("companies.id", ondelete="RESTRICT"),
              nullable=True))   # ← пока разрешаем NULL

# Шаг B: заполнить все существующие строки
op.execute(sa.text("UPDATE projects SET company_id = 1"))   # ← ставим холдинг

# Шаг C: сделать обязательным
op.alter_column("projects", "company_id", nullable=False)   # ← теперь NOT NULL
```

Это и есть паттерн **expand/contract** в применении к одной колонке:

- **Expand** (расширить): добавляем nullable колонку — система продолжает работать
- **Fill** (заполнить): применяем данные по умолчанию
- **Contract** (сжать): делаем колонку обязательной

### Шаг 5: дополнительные поля в `contracts`

```python
op.add_column("contracts", sa.Column("is_internal", sa.Boolean(), nullable=False, server_default=sa.text("FALSE")))
op.add_column("contracts", sa.Column("start_date", sa.Date(), nullable=True))
op.add_column("contracts", sa.Column("end_date", sa.Date(), nullable=True))
```

`is_internal` добавляется как NOT NULL, но с `server_default=FALSE` — PostgreSQL сам заполняет существующие строки значением `FALSE`. Это безопасная операция.

### Шаг 6: seed пользователей в новую таблицу ролей

```python
op.execute(sa.text("""
    INSERT INTO user_company_roles (user_id, company_id, role_template)
    SELECT id, 1, role
    FROM users
    WHERE deleted_at IS NULL
"""))
```

Переносим существующих пользователей в новую таблицу связей. Все получают роль в компании «Холдинг».

### Функция `downgrade`: откат

```python
def downgrade() -> None:
    # Порядок обратный upgrade — сначала данные, потом таблицы
    op.drop_column("contracts", "end_date")
    op.drop_column("contracts", "start_date")
    op.drop_column("contracts", "is_internal")

    op.drop_index("ix_payments_company_id", table_name="payments")
    op.drop_column("payments", "company_id")
    # ... аналогично для других таблиц

    op.drop_table("user_company_roles")
    op.drop_index("uq_companies_inn_active", table_name="companies")
    op.drop_table("companies")
```

Команды в `downgrade` идут в **обратном** порядке к `upgrade`. Нельзя удалить таблицу `companies`, пока на неё есть внешние ключи из других таблиц.

## Паттерн expand/contract: полная версия

В сложных случаях — когда система работает и нельзя допустить даже секунды простоя — используется расширенный вариант:

```
Шаг 1. Expand (расширение)
  — Добавить новую колонку/таблицу как nullable
  — Развернуть новый код, который умеет писать в ОБА места (старое и новое)
  — Данные накапливаются в новом месте

Шаг 2. Fill (заполнение)
  — Перенести исторические данные в новое место (backfill)
  — Верифицировать корректность

Шаг 3. Contract (сжатие)
  — Переключить чтение на новое место
  — Сделать новую структуру обязательной
  — Удалить старую структуру

Шаг 4. Cleanup (уборка)
  — Убрать код совместимости
```

Для нашего MVP мы используем упрощённый вариант: останавливаем сервис, накатываем миграцию, запускаем снова. Полный паттерн нужен для систем без окна обслуживания (production 24/7 без даунтайма).

## Что такое round-trip и почему он обязателен

В нашем проекте есть правило (зафиксировано в CLAUDE.md):

```
Миграции — обязательный round-trip:
alembic downgrade -1 && alembic upgrade head
чисто, без ошибок. Без round-trip миграция не считается готовой.
```

**Round-trip** (буквально «туда и обратно») — это проверка: применить миграцию вперёд, откатить назад, снова применить вперёд. Всё должно пройти без ошибок.

Зачем? Потому что однажды может понадобиться откатить версию. Если `downgrade` написан с ошибкой или пропущен — откат невозможен. Обнаруживать это в момент аварии — слишком поздно.

Аналогия: страховка. Вы платите за неё каждый месяц и надеетесь никогда не использовать. Но когда нужна — она должна работать с первого раза.

## Команды Alembic: краткий справочник

```bash
# Посмотреть текущую версию базы
alembic current

# Посмотреть историю миграций
alembic history --verbose

# Применить все непримнёненные миграции
alembic upgrade head

# Откатить последнюю миграцию
alembic downgrade -1

# Откатить до конкретной версии
alembic downgrade f80b758cadef

# Сгенерировать новую миграцию на основе изменений в моделях
alembic revision --autogenerate -m "краткое описание"
```

Разработчики запускают эти команды в папке `backend/` (там лежит файл `alembic.ini` с настройками).

## Когда нужна миграция

Миграция нужна при любом изменении структуры базы данных:

| Что меняется | Нужна миграция |
|---|---|
| Добавить новую таблицу | Да |
| Добавить колонку в таблицу | Да |
| Убрать колонку | Да |
| Изменить тип колонки (String → Integer) | Да |
| Добавить/убрать индекс | Да |
| Добавить/убрать внешний ключ | Да |
| Изменить значение в Python-классе | Нет |
| Изменить метод в сервисе | Нет |
| Добавить новый роутер | Нет |

## Пример своими словами

На совещании Вы можете сказать так:

> «Когда мы добавляем новое поле в систему — например, поле "дата последней проверки объекта" — мы не можем просто взять и добавить колонку в базу вручную. Это сломает синхронизацию с кодом, история изменений пропадёт, и другие разработчики не узнают, что поле существует. Вместо этого разработчик создаёт файл миграции: там написано "добавить такую-то колонку с такими-то параметрами". Этот файл попадает в git, проходит проверку, и применяется на все стенды — тестовый, стейджинг, production — в одинаковом порядке. Если что-то пошло не так — откатываем командой downgrade. Вся история изменений структуры базы прозрачна и воспроизводима.»

## Риски и типичные ошибки

**1. Изменить базу вручную, минуя миграции.**
«Быстро добавил колонку через pgAdmin» — в git этого нет, другой стенд не знает, следующий разработчик не знает. При следующем деплое — конфликт или потеря изменения.

**2. Не проверить `downgrade`.**
Написать `upgrade`, но не написать обратный `downgrade`. Когда нужно откатить — ошибка. Поэтому в нашем проекте обязателен round-trip.

**3. Автогенерация без проверки.**
Alembic умеет автоматически генерировать файл миграции (`--autogenerate`), сравнивая модели с базой. Но автогенерация не всегда точна: может пропустить частичные индексы, неверно обработать некоторые типы данных. Каждый сгенерированный файл нужно читать и проверять.

**4. Не делать backfill для NOT NULL колонок.**
Добавили `nullable=False` без шага заполнения существующих строк — база откажет: «в существующих строках нет значения». Паттерн: nullable → backfill → not null.

**5. Миграции с данными production в порядке.**
Некоторые операции безопасны на пустой базе, но опасны на базе с миллионом строк. `ALTER TABLE` с переиндексацией может занять минуты и заблокировать таблицу. На production — обязательно тестировать на копии данных.

## Проверка понимания

**Вопрос 1.** Команда написала новую миграцию, применила на тестовом стенде — всё работает. Но не проверили `downgrade`. Почему Вы не принимаете эту миграцию в main?

> Потому что откат не проверен. Если завтра на production обнаружится критическая ошибка и потребуется откат — `downgrade` может сломаться. Обнаружить это в момент аварии значительно хуже, чем на этапе проверки. Round-trip обязателен регламентом.

**Вопрос 2.** Разработчик спрашивает: «Можно я просто через pgAdmin добавлю колонку в production-базу, не создавая миграцию? Так быстрее». Какие последствия?

> Минимум три проблемы: 1) следующий `alembic upgrade head` на другом стенде не добавит колонку — стенды разойдутся; 2) в git нет записи об изменении — история потеряна; 3) при следующем деплое Alembic увидит расхождение и поведёт себя непредсказуемо. Все изменения схемы — только через миграции.

**Вопрос 3.** Почему при добавлении колонки `company_id NOT NULL` в таблицы с существующими данными нельзя просто сделать один `ADD COLUMN company_id INTEGER NOT NULL`?

> Потому что в существующих строках нет значения для этой колонки. PostgreSQL не знает, что туда поставить, и откажет с ошибкой. Нужен трёхшаговый паттерн: ADD COLUMN nullable → UPDATE (backfill) → ALTER COLUMN NOT NULL. Именно так сделано в миграции `f7e8d9c0b1a2`.
