## Коротко о главном

**OWASP Top 10** — это список десяти наиболее опасных уязвимостей в веб-приложениях, который публикует организация OWASP (Open Web Application Security Project — открытый проект безопасности веб-приложений). Обновляется раз в несколько лет. Текущая версия — 2021 год. Используется как стандартный чек-лист при аудите безопасности любого ПО.

2026-04-17 в M-OS был проведён полный аудит безопасности. Ниже — разбор каждой из 10 категорий с тем, что нашли именно в нашем проекте.

---

## Аналогия: чек-лист техники безопасности на стройке

На строительном объекте перед началом работ проверяют стандартный чек-лист: каски, страховочные ремни, пожарная сигнализация, наличие аптечки, ограждение котлована, инструктаж. Не потому что это перестраховка — а потому что статистика несчастных случаев выявила именно эти пункты как источники 80% инцидентов.

OWASP Top 10 — тот же принцип. Это не теоретические угрозы — это реальная статистика того, через какие уязвимости взламывают большинство систем. Если все 10 категорий закрыты — вероятность взлома снижается многократно.

## A01 — Нарушение контроля доступа (Broken Access Control)

### Что это

Пользователь получает доступ к данным или действиям, которые ему не положены. Самый распространённый вид атаки — **IDOR** (Insecure Direct Object Reference — небезопасная прямая ссылка на объект).

**Пример**: Пользователь компании «Альфа» делает запрос:
```
GET /api/v1/projects/42
```
Если сервер возвращает проект №42 без проверки, принадлежит ли он компании пользователя — это IDOR. Пользователь может перебирать ID (42, 43, 44...) и читать чужие данные.

### Что нашли в M-OS

**P1: 6 API-модулей без `company_id` фильтра.** На момент аудита часть эндпоинтов возвращала данные без проверки принадлежности к компании пользователя. То есть сотрудник компании «Альфа» теоретически мог читать данные компании «Бета».

**Что делаем**: PR #2 — внедрение полноценного RBAC (Role-Based Access Control — контроль доступа на основе ролей). Каждый сервис наследует `CompanyScopedService`, который автоматически добавляет фильтр `WHERE company_id = ?` к каждому запросу. Реализован `UserContext` с `active_company_id` — текущая компания пользователя.

Код фильтра в `backend/app/services/company_scoped.py`:
```python
class CompanyScopedService:
    def _scoped_query(self, query):
        # Добавляет WHERE company_id = активная_компания_пользователя
        return query.where(Model.company_id == self.context.company_id)
```

## A02 — Криптографические ошибки (Cryptographic Failures)

### Что это

Слабая или отсутствующая криптография: слабые пароли, слабые ключи, передача данных без шифрования, хранение паролей в открытом виде.

### Что нашли в M-OS

**Без критических нарушений.** Пароли хранятся через bcrypt (современный алгоритм, устойчив к брутфорсу). JWT подписывается HS256. Длина `JWT_SECRET_KEY` — минимум 32 символа, контролируется валидатором при старте.

**Что планируется**: в рамках ADR 0011 Часть 3 — криптографическая цепочка аудит-лога. Каждая запись в `audit_log` будет содержать хеш предыдущей записи (`prev_hash`). Это гарантирует, что записи нельзя тихо удалить или изменить — нарушение цепочки обнаружится при верификации. Это похоже на технологию блокчейн в миниатюре.

## A03 — Инъекции (Injection)

### Что это

Злоумышленник подставляет вредоносный код в запросы, и система его выполняет. Самая известная — **SQL-инъекция**.

**Пример SQL-инъекции**: если система напрямую вставляет ввод пользователя в SQL-запрос:
```python
# ОПАСНО — прямая подстановка строки в SQL
query = f"SELECT * FROM users WHERE email = '{body.email}'"
```

Злоумышленник вводит email: `' OR '1'='1`. Запрос превращается в:
```sql
SELECT * FROM users WHERE email = '' OR '1'='1'
```
Условие `'1'='1'` всегда истинно — запрос вернёт всех пользователей. Полный обход аутентификации.

### Что нашли в M-OS

**Нет нарушений.** Весь код работает через ORM (SQLAlchemy). ORM никогда не вставляет пользовательский ввод напрямую в SQL — он использует параметризованные запросы (placeholders). Например:

```python
# БЕЗОПАСНО — ORM использует параметр, не подстановку строки
await db.execute(select(User).where(User.email == body.email))
```

Под капотом SQLAlchemy генерирует:
```sql
SELECT * FROM users WHERE email = $1  -- PostgreSQL параметр
```
И передаёт значение `body.email` отдельно. Что бы ни ввёл пользователь — это будет значением параметра, а не частью SQL-кода.

## A04 — Небезопасный дизайн (Insecure Design)

### Что это

Архитектурные решения, которые делают систему уязвимой по своей природе. Не баги в коде, а неправильные решения при проектировании.

### Что нашли в M-OS

**P1: Нет rate-limiting на эндпоинте `/auth/login`.** Rate-limiting — ограничение количества запросов в единицу времени. Без него злоумышленник может делать тысячи попыток входа в секунду, перебирая пароли (brute force — атака грубой силой). Bcrypt замедляет каждую проверку до ~200 мс, но без ограничения числа попыток это лишь снижает скорость атаки, не блокирует её.

**Что делаем**: PR #2 — добавить middleware (промежуточный слой) для ограничения запросов. Например: не более 5 попыток входа с одного IP за 1 минуту. При превышении — 429 Too Many Requests.

**P1: Нет MFA (Multi-Factor Authentication — многофакторная аутентификация)** для роли `owner` и `accountant`. MFA — это второй способ подтверждения личности помимо пароля (например, код из SMS или приложения-аутентификатора). Статья 81 нашей Конституции M-OS требует MFA для расширенных прав. Это блокер production-gate — до реального развёртывания систему с доступом к финансовым данным без MFA запускать нельзя.

## A05 — Неправильная конфигурация безопасности (Security Misconfiguration)

### Что это

Сервер или приложение запущено с небезопасными настройками по умолчанию: открытые порты, отключённые заголовки безопасности, публичный доступ к внутренним эндпоинтам, режим отладки в продакшене.

### Что нашли в M-OS

**P2: Нет security headers (заголовков безопасности).** Браузер и сервер обмениваются HTTP-заголовками. Существуют специальные заголовки, которые говорят браузеру: «не выполняй скрипты из сторонних источников», «всегда используй HTTPS», «не позволяй этой странице загружаться в iframe чужого сайта». Их отсутствие — не критично для внутреннего ПО, но является hygiene-вопросом.

Примеры заголовков:
- `Content-Security-Policy` — ограничивает, откуда браузер может загружать ресурсы
- `X-Frame-Options: DENY` — запрещает встраивание в iframe (защита от clickjacking — кликджекинга, атаки подменой кнопок)
- `Strict-Transport-Security` — заставляет браузер всегда использовать HTTPS

**Что делаем**: backlog на PR #3 — добавить middleware с security headers. Не блокирует PR #1 и #2.

## A06 — Уязвимые компоненты (Vulnerable and Outdated Components)

### Что это

Использование библиотек и зависимостей с известными уязвимостями (CVE — Common Vulnerabilities and Exposures, реестр уязвимостей). Злоумышленник знает, что версия X библиотеки Y имеет уязвимость Z, и использует это против всех, кто не обновился.

**CVE** — это идентификатор конкретной уязвимости в конкретной версии программного обеспечения. Например, `CVE-2023-12345` — нумерованная запись в публичном реестре, описывающая проблему, затронутые версии и исправление.

### Что нашли в M-OS — **сегодняшний инцидент**

**P0 (закрыт hotfix!): `python-jose` с CVE.** Библиотека `python-jose` использовалась для работы с JWT. В ней была обнаружена уязвимость. Более того — сегодня (2026-04-17) при старте приложения возникла ошибка:

```
ModuleNotFoundError: No module named 'jwt'
```

Это произошло потому, что `python-jose` и `PyJWT` — две разные библиотеки для работы с JWT. Они конфликтуют. `python-jose` импортирует модуль `jose`, а `PyJWT` — модуль `jwt`. Когда одна была удалена и заменена другой, код, написанный под `jose`, перестал работать.

**Решение**: сегодня в hotfix `python-jose` была удалена и заменена на `PyJWT` (современная, активно поддерживаемая библиотека). Весь код в `security.py` и `deps.py` переписан под `import jwt` от PyJWT. Аналогичная история с `passlib` — заброшенная с 2020 года, заменена на прямой вызов `bcrypt`.

**Что делаем на будущее**: регулярный запуск `pip-audit` — инструмента, который проверяет все Python-зависимости против базы CVE и сообщает о найденных уязвимостях. Входит в чек-лист задачи S-3 аудита.

## A07 — Ошибки аутентификации и идентификации (Identification and Authentication Failures)

### Что это

Слабости в механизмах аутентификации: нет ограничения попыток входа, слабые пароли разрешены, сессии не закрываются корректно, токены не инвалидируются при выходе.

### Что нашли в M-OS

**P1: `is_holding_owner` — флаг суперадмина без DB-перепроверки при каждом запросе.** Посмотрите на код в `deps.py`:

```python
# is_holding_owner читаем из JWT-клейма; при его отсутствии — False.
is_holding_owner = bool(payload.get("is_holding_owner", False))
```

Флаг `is_holding_owner` берётся из JWT-токена, а не из базы данных. JWT выдаётся при входе и живёт 60 минут. Если за эти 60 минут роль пользователя была снижена (например, он перестал быть владельцем холдинга), система об этом не узнает до истечения токена.

Для большинства случаев это приемлемо. Но флаг `is_holding_owner = True` означает обход всех company-фильтров — полный доступ ко всем данным. Доверять такому критичному флагу только из токена без актуальной проверки в базе — риск.

**Что делаем**: в PR #2 — при значении `is_holding_owner = True` из токена делать дополнительный запрос в базу для подтверждения. Это одна дополнительная строка запроса на каждый запрос от суперадмина — незначительная нагрузка за значительное улучшение безопасности.

**P1 (из A04): Нет rate-limiting** — уже упомянуто выше, но относится и к этой категории.

## A08 — Нарушение целостности данных и программного обеспечения (Software and Data Integrity Failures)

### Что это

Обновления ПО, CI/CD пайплайны или данные, поступающие из ненадёжных источников без проверки подписи или целостности. В частности — библиотеки без фиксированных версий.

### Что нашли в M-OS

**Частично закрыто.** `pyproject.toml` фиксирует версии зависимостей. Есть `poetry.lock` (файл блокировки версий). Аудит-лог (`audit_log`) создаётся при каждой записи — но без криптографической цепочки (плановая задача ADR 0011 Часть 3).

**Риск без crypto-chain**: записи в `audit_log` можно удалить или изменить напрямую в базе данных, и это не будет обнаружено. После внедрения цепочки хешей — любое изменение или удаление нарушит цепочку, что будет обнаружено при верификации эндпоинтом `/audit/verify`.

## A09 — Недостатки логирования и мониторинга (Security Logging and Monitoring Failures)

### Что это

Критические события (попытки взлома, ошибки авторизации, изменения прав) не фиксируются или фиксируются так, что расследовать инцидент невозможно.

### Что нашли в M-OS

**Хорошее состояние.** Посмотрите на `auth.py` — каждая попытка входа логируется с IP-адресом:

```python
logger.warning(
    "Неудачная попытка входа (нет пользователя): email=%s ip=%s",
    body.email, ip,
)
logger.info("Успешный вход: user_id=%s email=%s ip=%s", user.id, user.email, ip)
```

`audit_log` фиксирует каждую write-операцию с информацией о пользователе, типе действия, ID объекта. Это выполняет требование ADR 0007.

**Что остаётся улучшить**: централизованный мониторинг тревог (alerting). Сейчас логи пишутся в файлы. Для production нужна система, которая при N неудачных попытках входа с одного IP за минуту автоматически уведомляет ответственного. Это backlog — не блокер MVP.

## A10 — SSRF (Server-Side Request Forgery — подделка запроса на стороне сервера)

### Что это

Злоумышленник заставляет сервер делать HTTP-запросы к адресам, которые он задаёт. Например, к внутренним сервисам, недоступным снаружи (базам данных, облачным metadata-сервисам).

**Пример**: если система принимает URL от пользователя и делает запрос по нему:
```python
# ОПАСНО
url = request.body.get("webhook_url")
response = httpx.get(url)  # пользователь задал http://internal-db:5432/
```

Через `webhook_url` можно заставить сервер читать данные с любого внутреннего ресурса.

### Что нашли в M-OS

**Нет нарушений.** В кодовой базе M-OS нет пользовательских URL в исходящих запросах. Единственный разрешённый живой внешний запрос — Telegram API (для уведомлений). Он изолирован отдельным адаптером, и URL задан статически, не пользователем.

Это соответствует CODE_OF_LAWS ст. 45а: «Любой живой исходящий вызов вне `_live_transport` адаптера — P0». Когда в будущем добавятся интеграции (банки, Росреестр, 1С) — они тоже будут изолированы по тому же принципу (ADR 0014).

## Инцидент дня: ModuleNotFoundError jwt

Сегодня при старте приложения произошло:

```
ModuleNotFoundError: No module named 'jwt'
```

Разберём, как это связано с OWASP:

1. **A06**: `python-jose` имела CVE и была удалена. Это правильное решение.
2. Но `python-jose` использует `from jose import jwt`, а `PyJWT` — `import jwt`. Это разные модули с похожими именами.
3. После удаления `python-jose` код, использующий `jose`, перестал работать — `ModuleNotFoundError`.
4. **Решение**: переписать все импорты с `from jose import jwt` на `import jwt` (PyJWT).

Этот инцидент — хорошая иллюстрация А06: управление зависимостями — это не «поставить один раз и забыть». Уязвимые компоненты надо регулярно находить и заменять. При замене — тщательно проверять совместимость API.

«OWASP — это как строительный СНиП по безопасности. Не наши правила — международный стандарт по результатам тысяч взломов. Мы провели полный аудит системы и нашли: одна критическая проблема с устаревшей библиотекой — устранена сегодня. Четыре серьёзных — устраняются в следующем этапе разработки. Одна несрочная — в плане. Три категории из десяти — полностью чисты. Для внутреннего ПО холдинга это хороший результат для MVP.»
