# Урок 1. Что такое код, файл и программа

**Блок 1 — Основы** | Время чтения: ~15 минут

---

## Что Вы поймёте после этого урока

После этого урока Вы будете знать, что такое исходный код, почему он хранится в обычных текстовых файлах, чем отличается «код» от «программы», и почему Python-файлы из нашего проекта можно читать в блокноте, а Windows-программы — нет.

---

## Зачем это важно для M-OS

M-OS — это программное обеспечение. Оно живёт в файлах с кодом. Когда Вы видите строку вроде `backend/app/models/user.py` в отчёте от команды — это не абстракция, это конкретный текстовый файл на диске сервера. Понимание того, что такое «файл с кодом», даёт Вам возможность ориентироваться в структуре проекта и задавать правильные вопросы команде.

---

## Аналогия: чертёж и построенный дом

Представьте процесс строительства коттеджа.

**Чертёж** — это набор листов с линиями, размерами, спецификациями. Сам по себе чертёж не является домом. Он написан на профессиональном языке (ГОСТы, условные обозначения), его может прочитать и проверить любой аттестованный инженер.

**Построенный дом** — это результат исполнения чертежа. Жить в чертеже нельзя. Изменить дом без изменения чертежа — значит создать рассогласование между документацией и реальностью.

Именно так работает программное обеспечение:

| Строительство | Программирование |
|---|---|
| Чертёж (листы с текстом и линиями) | Исходный код (текстовые файлы) |
| Построенный дом | Работающая программа в памяти компьютера |
| Строитель читает чертёж и строит | Компьютер читает код и исполняет |
| Типовые чертежи для разных домов | Один код — много запусков |
| Архив чертежей | Репозиторий (хранилище кода) |

## Конкретика: что такое исходный код

**Исходный код** (или просто «код») — это обычный текст, записанный в файл. Откройте любой файл из нашего проекта в текстовом редакторе — и увидите буквы, цифры, скобки. Никакой «тайнописи».

Вот реальная строка из нашего проекта (`backend/app/models/user.py`, строка 19):

```python
email: Mapped[str] = mapped_column(String(255), unique=True, nullable=False, index=True)
```

Это просто текст. Он описывает, что в таблице `users` есть колонка `email`, которая хранит строку максимум 255 символов, уникальная для каждого пользователя, обязательная к заполнению, и по ней создан индекс для быстрого поиска.

## Компилятор и интерпретатор: два способа «прочитать» код

Разные языки программирования обрабатываются по-разному.

### Компилятор (переводчик, работающий заранее)

Компилятор — это программа, которая **заранее** переводит весь исходный код в машинный язык (последовательность команд процессора — нули и единицы). Результат — бинарный файл (`.exe` на Windows, бинарник на Linux). Этот файл уже не является текстом: открыть его в блокноте и прочитать невозможно.

Примеры компилируемых языков: C, C++, Go, Rust.

Аналогия: перевели весь договор с русского на японский ещё до встречи с партнёром. На встрече выдали готовый перевод — быстро, но изменить уже нельзя.

### Интерпретатор (переводчик, работающий «на лету»)

Интерпретатор читает исходный код и исполняет его строка за строкой прямо сейчас, без предварительного перевода. Файл с кодом остаётся текстом — его можно открыть и прочитать в любой момент.

Примеры интерпретируемых языков: **Python**, JavaScript, Ruby.

Аналогия: переводчик-синхронист на переговорах. Слышит фразу — сразу переводит. Чуть медленнее, но можно поменять контекст прямо в ходе разговора.

### Почему Python — интерпретатор, и что это значит для нас

Python-файл (`.py`) всегда остаётся читаемым текстом. Это означает:
- Вы в любой момент можете открыть файл и прочитать, что делает код
- Команда не может «спрятать» логику в бинарнике — всё прозрачно
- Изменение кода вступает в силу при следующем запуске — не нужно «пересобирать» проект часами

## Файл, папка, модуль, пакет

Разберём структуру нашего проекта на конкретном примере.

```
coordinata56/
  backend/
    app/
      models/
        user.py        ← файл с кодом (модуль)
        audit.py       ← файл с кодом (модуль)
      services/
        audit.py       ← другой файл с тем же именем, в другой папке
      api/
        health.py
```

- **Файл** — единица хранения кода. Содержит текст на Python.
- **Папка** (директория) — организует файлы. Как папка с документами в архиве.
- **Модуль** — Python-файл, который можно «подключить» к другому файлу. Это способ сказать: «возьми код из того файла и используй здесь».
- **Пакет** — папка с Python-файлами, которую Python воспринимает как единый блок кода.

---

## Исходник vs бинарник: практическое следствие

В нашем проекте:

- `backend/app/models/user.py` — исходник. Текст. Читается. Хранится в git (система контроля версий). Это «чертёж».
- Когда сервер запускает `python backend/app/main.py` — Python читает файлы и начинает их исполнять. Это «стройка». Результат живёт в памяти сервера, пока сервер работает.
- Если сервер упал — код в файлах никуда не делся. Достаточно запустить снова.

---

## Что значит «версия кода» и зачем Git

Git — это система, которая запоминает все изменения в файлах с кодом. Аналог: журнал правок в чертёжном архиве, где каждая версия чертежа пронумерована, подписана, и известно, кто и когда что изменил.

В нашем проекте каждое изменение кода фиксируется как «коммит» (запись в журнале). Можно вернуться к любой предыдущей версии — как к старому чертежу.

## Пример своими словами

На совещании Вы можете сказать так:

> «У нас есть файл `user.py` — это описание пользователя системы. В нём написано, какие данные мы храним о каждом сотруднике: email, имя, роль, активен ли аккаунт. Когда сервер запускается, Python читает этот файл и создаёт соответствующую таблицу в базе данных. Сам файл — это как чертёж, а база данных — как реальный дом по этому чертежу.»

## Риски и типичные ошибки

**1. Запутать «файл с кодом» и «работающую программу».**
Код изменили — программа не обновилась сама. Нужно перезапустить сервер. Это принципиально важно при деплое (развёртывании) изменений.

**2. Думать, что «бинарник» надёжнее исходника.**
Некоторые поставщики ПО дают только бинарные файлы, скрывая исходный код. Это означает: Вы не можете проверить, что программа делает на самом деле, не можете исправить баг без участия поставщика, не можете адаптировать под свои нужды. Именно поэтому M-OS пишется на открытом коде — всё прозрачно.

**3. Потерять исходный код.**
Если файлы с кодом удалены и нет резервной копии (Git-репозитория) — программа «умерла». Бинарник продолжает работать, но починить ничего нельзя. Поэтому Git-репозиторий coordinata56 — это стратегически важный актив, не менее важный, чем база данных.

## Проверка понимания

**Вопрос 1.** Сотрудник говорит: «Я поменял код в файле `user.py`, теперь пользователи смогут видеть новый раздел». Вы уточняете: «Сервер перезапускали после изменения?» — Почему этот вопрос уместен?

> Потому что Python-интерпретатор читает файлы при запуске, а не «вживую». Изменение файла не означает автоматического обновления работающей программы в памяти сервера.

**Вопрос 2.** Чем файл `backend/app/models/user.py` отличается от таблицы `users` в базе данных PostgreSQL?

> Файл — это описание (чертёж), текст на Python. Таблица в БД — реальные данные (построенное), которые возникли по этому описанию при первой миграции. Файл можно поменять, не трогая данные. Но если хочешь изменить структуру таблицы — нужно создать специальный файл-миграцию и запустить его.

**Вопрос 3.** Почему Python называют интерпретируемым языком?

> Потому что специальная программа — Python-интерпретатор — читает текстовый файл с кодом и исполняет его строка за строкой в момент запуска, без предварительного перевода в бинарный формат.
