## Что Вы поймёте после этого урока

После этого урока Вы будете знать: что такое SQLAlchemy-модель, как Python-класс описывает структуру таблицы в базе данных, что означают строки с `Mapped`, `mapped_column`, `ForeignKey` в наших файлах `models/*.py`, и почему `Company` и `User` устроены именно так, а не иначе.

---

## Зачем это важно для M-OS

Каждая сущность M-OS — компания, пользователь, проект, платёж, договор, подрядчик — живёт в двух местах одновременно: как **Python-класс** в папке `backend/app/models/` и как **таблица** в PostgreSQL. Понимание того, как класс соотносится с таблицей, позволяет Вам читать архитектурные решения команды, оценивать корректность модели данных и задавать вопрос «а это поле вообще есть в базе?» — не как гипотезу, а как осознанное утверждение.

## Аналогия: чертёж дома и реальный дом в посёлке

В Вашем девелоперском бизнесе есть **типовой чертёж** коттеджа класса А. Этот чертёж описывает:
- какие у дома комнаты (спальня, кухня, гостиная)
- их размеры и расположение
- из каких материалов строить
- какие коммуникации предусмотрены

По этому чертежу построены 20 одинаковых домов на участке. Каждый дом — реальный объект. Чертёж — описание.

**Модель SQLAlchemy — это чертёж. Строки в таблице базы данных — это дома.**

| Строительство | SQLAlchemy |
|---|---|
| Типовой чертёж дома | Python-класс модели (например, `class Company`) |
| Спецификация комнат | Поля класса (`id`, `inn`, `full_name`, ...) |
| Один конкретный дом | Одна строка в таблице `companies` |
| 85 домов в посёлке | 85 строк в таблице |
| Материал стен (кирпич/газобетон) | Тип колонки (`String`, `Integer`, `Boolean`) |
| Ограничение «высота потолков минимум 2.7 м» | Ограничение в БД (`nullable=False`, `unique=True`) |

## Реальный файл: `backend/app/models/company.py`

```python
class Company(Base, TimestampMixin):
    """Юридическое лицо холдинга."""

    __tablename__ = "companies"

    id: Mapped[int] = mapped_column(primary_key=True, autoincrement=True)
    inn: Mapped[str | None] = mapped_column(String(12), nullable=True)
    kpp: Mapped[str | None] = mapped_column(String(9), nullable=True)
    full_name: Mapped[str] = mapped_column(String(512), nullable=False)
    short_name: Mapped[str] = mapped_column(String(255), nullable=False)
    company_type: Mapped[CompanyType] = mapped_column(
        Enum(CompanyType, ..., native_enum=False, length=16, ...),
        nullable=False,
    )
    is_active: Mapped[bool] = mapped_column(Boolean, nullable=False, default=True)
```

Разберём каждый элемент:

### `class Company(Base, TimestampMixin)`

Класс `Company` наследует от двух родителей:

- **`Base`** — базовый класс SQLAlchemy. Он говорит: «этот класс — это модель базы данных, SQLAlchemy должен за ним следить». Без `Base` класс был бы просто Python-классом, никак не связанным с базой.
- **`TimestampMixin`** — подмешиваемый класс (mixin — от английского «смесь»), который добавляет два поля: `created_at` (когда создали) и `updated_at` (когда последний раз изменили). Вместо того чтобы копировать эти поля в каждую модель — написали один раз в `mixins.py` и подключаем везде.

Аналогия с чертежом: `Base` — это ГОСТ на оформление чертежей. `TimestampMixin` — типовая секция «Журнал изменений», которую вставляют в каждый чертёж.

### `__tablename__ = "companies"`

Это имя таблицы в базе данных. SQLAlchemy создаёт/ищет таблицу именно с этим именем. Если написать `"company"` вместо `"companies"` — класс будет искать таблицу `company`, а её нет, и всё сломается.

### `id: Mapped[int] = mapped_column(primary_key=True, autoincrement=True)`

Разбор:

- `id` — имя поля в Python и имя колонки в таблице
- `Mapped[int]` — подсказка типа: это целое число. `Mapped` — специальный контейнер SQLAlchemy 2.0, который говорит «это поле модели, а не просто атрибут класса»
- `mapped_column(...)` — описание колонки
- `primary_key=True` — это первичный ключ (уникальный идентификатор строки). В каждой таблице должен быть первичный ключ
- `autoincrement=True` — база данных сама присваивает следующий номер при каждой новой записи: 1, 2, 3, ...

Аналогия: кадастровый номер участка. Уникален, назначается реестром автоматически, никогда не повторяется.

### `inn: Mapped[str | None] = mapped_column(String(12), nullable=True)`

- `str | None` — поле может содержать строку или быть пустым (None в Python = NULL в SQL)
- `String(12)` — строка максимум 12 символов (ИНН — ровно 10 или 12 цифр)
- `nullable=True` — в базе данных колонка может быть NULL (пустой)

Почему ИНН nullable? Потому что при первом запуске системы создаётся «seed-запись» холдинга с `inn=NULL`. Реальный ИНН вносит администратор позже при первичной настройке. Это осознанное архитектурное решение, зафиксированное в комментарии к коду.

### `full_name: Mapped[str] = mapped_column(String(512), nullable=False)`

- `nullable=False` — значение обязательно. В базе данных на уровне ограничения: если попытаться сохранить строку без `full_name` — PostgreSQL вернёт ошибку.

### `company_type: Mapped[CompanyType] = mapped_column(Enum(...), nullable=False)`

- `CompanyType` — Python-перечисление (enum): `ooo`, `ao`, `ip`, `other`
- `Enum(CompanyType, native_enum=False)` — хранить в базе как строку с CHECK-ограничением, а не как нативный PostgreSQL-тип. `native_enum=False` — осознанное решение для простоты миграций
- SQLAlchemy проверяет: при попытке записать `company_type="zao"` (которого нет в перечислении) — ошибка ещё на уровне Python, до запроса к базе

## Реальный файл: `backend/app/models/user.py`

```python
class User(Base, TimestampMixin, SoftDeleteMixin):
    __tablename__ = "users"

    id: Mapped[int] = mapped_column(primary_key=True, autoincrement=True)
    email: Mapped[str] = mapped_column(String(255), unique=True, nullable=False, index=True)
    password_hash: Mapped[str] = mapped_column(String(255), nullable=False)
    full_name: Mapped[str] = mapped_column(String(255), nullable=False)
    role: Mapped[UserRole] = mapped_column(
        Enum(UserRole, name="user_role", native_enum=False, length=32),
        nullable=False,
    )
    is_active: Mapped[bool] = mapped_column(Boolean, nullable=False, default=True)
    last_login_at: Mapped[datetime | None] = mapped_column(
        DateTime(timezone=True), nullable=True, default=None
    )
```

Что нового по сравнению с `Company`:

**`SoftDeleteMixin`** — подмешивает поле `deleted_at`. Когда пользователя «удаляют» — не стирают строку из базы, а заполняют `deleted_at` текущим временем. При выборке фильтруем: `WHERE deleted_at IS NULL`. Это называется «мягкое удаление» (soft delete) — данные сохраняются для аудита, но система их не показывает.

Почему это важно с бизнес-точки зрения? Если сотрудник был уволен, а потом возник судебный спор о его действиях — данные нужны. Физическое удаление их уничтожает. Мягкое удаление — сохраняет.

**`unique=True`** в email — в таблице нельзя будет двух пользователей с одинаковым email. PostgreSQL создаст уникальный индекс. При попытке — ошибка.

**`index=True`** в email — создаёт индекс для быстрого поиска. Без индекса поиск пользователя по email проходит по всей таблице строка за строкой. С индексом — мгновенно, как поиск по оглавлению книги.

**`password_hash`** — хранится не сам пароль, а его хеш (необратимое преобразование). Если база данных утечёт — пароли из неё не извлечь. Это стандартная практика безопасности.

**`last_login_at: Mapped[datetime | None]`** — дата и время последнего входа. `timezone=True` — хранить с часовым поясом (UTC). Это важно для аудита: без часового пояса «2026-04-18 09:00» — это Москва или Владивосток?

## Реальный файл: `backend/app/models/project.py`

```python
class Project(Base, TimestampMixin, SoftDeleteMixin):
    __tablename__ = "projects"

    id: Mapped[int] = mapped_column(primary_key=True, autoincrement=True)
    company_id: Mapped[int] = mapped_column(
        ForeignKey("companies.id", ondelete="RESTRICT"),
        nullable=False,
        index=True,
    )
    code: Mapped[str] = mapped_column(String(64), unique=True, nullable=False, index=True)
    name: Mapped[str] = mapped_column(String(255), nullable=False)
    description: Mapped[str | None] = mapped_column(Text, nullable=True)
    status: Mapped[str] = mapped_column(String(32), nullable=False, default="active")
```

Новый элемент: **`ForeignKey`** (внешний ключ).

`ForeignKey("companies.id", ondelete="RESTRICT")` означает: значение `company_id` в таблице `projects` должно существовать как `id` в таблице `companies`. Нельзя создать проект с `company_id=999`, если компании с id=999 не существует.

`ondelete="RESTRICT"` — что делать при попытке удалить компанию, у которой есть проекты: **запрещено**. PostgreSQL вернёт ошибку. Это защита целостности данных: нельзя «потерять» проекты, удалив их родительскую компанию.

Аналогия: нельзя ликвидировать ООО, пока у него есть незакрытые договоры. Реестр запретит это сделать.

## Как класс превращается в таблицу

**Класс не создаёт таблицу автоматически.** Это распространённое заблуждение.

Последовательность:

1. Разработчик пишет или изменяет Python-класс (модель)
2. Разработчик запускает `alembic revision --autogenerate -m "описание"` — Alembic сравнивает текущий класс с тем, что есть в базе, и генерирует файл миграции
3. Разработчик проверяет и корректирует файл миграции (автогенерация не всегда точна)
4. Разработчик запускает `alembic upgrade head` — Alembic применяет миграцию к реальной базе

Только после шага 4 таблица появляется или изменяется в PostgreSQL. Подробно о миграциях — в уроке 10.

## Мixin-ы: повторное использование полей

В нашем проекте несколько моделей содержат одинаковые поля. Вместо копирования — используем mixin-ы:

**`TimestampMixin`** добавляет:
- `created_at` — когда создана запись (заполняется автоматически при INSERT)
- `updated_at` — когда обновлена (обновляется при каждом UPDATE)

**`SoftDeleteMixin`** добавляет:
- `deleted_at` — когда «удалена» (NULL = не удалена, дата = удалена)

Какие модели используют SoftDelete: `User`, `Project`, `Contractor`, `Stage`, `House`, `Material`. Платёж (`Payment`) — нет: у него другая логика жизненного цикла через статусы.

На совещании Вы можете сказать так:

> «Наша модель `Company` — это чертёж таблицы компаний в базе данных. В ней описано: какие у компании есть атрибуты (ИНН, название, тип), какие из них обязательны, какие уникальны, и как компания связана с другими таблицами. Когда мы создаём новую компанию через API — SQLAlchemy читает этот чертёж, составляет SQL-команду и отправляет в PostgreSQL. Чертёж не меняется при добавлении данных — меняются только строки в таблице.»

## Риски и типичные ошибки

**1. Изменить модель без миграции.**
Добавили поле в класс — но не создали миграцию. Приложение запустится, но при попытке работать с новым полем — ошибка «колонки нет в таблице».

**2. `nullable=False` без `default`.**
Если поле обязательное и нет значения по умолчанию — при добавлении колонки в существующую таблицу с данными PostgreSQL откажется: «в существующих строках нет значения для этого поля». Нужно либо указать `server_default`, либо сначала добавить nullable, заполнить данные, потом сделать NOT NULL. Именно так сделано в миграции `f7e8d9c0b1a2` для `company_id`.

**3. Неверный `ondelete`.**
Если поставить `ondelete="CASCADE"` вместо `"RESTRICT"` — удаление компании автоматически удалит все её проекты. Это может быть нежелательным. В нашем проекте выбрано RESTRICT: удалять компанию с данными запрещено.

**4. Пропустить `index=True` на часто фильтруемых полях.**
`company_id` используется в фильтрах почти везде. Без индекса каждый запрос «дайте проекты компании 1» будет читать всю таблицу. С индексом — мгновенно.

## Проверка понимания

**Вопрос 1.** Команда добавила в модель `Project` поле `budget_limit: Mapped[int | None] = mapped_column(Integer, nullable=True)`. При тестировании возникает ошибка: «column budget_limit of relation projects does not exist». Что не сделано?

> Не создана и не применена миграция Alembic. Класс описывает новое поле, но в реальной таблице PostgreSQL эта колонка ещё отсутствует. Нужно выполнить `alembic revision --autogenerate` и `alembic upgrade head`.

**Вопрос 2.** Почему поле `email` в модели `User` имеет `index=True`, а поле `full_name` — нет?

> По `email` часто ищут пользователя при логине — это частый запрос, нужен индекс для скорости. По `full_name` поиск редкий, а индекс занимает дополнительное место на диске и замедляет запись. Индексы ставятся только там, где они реально нужны.

**Вопрос 3.** Чем `SoftDeleteMixin` отличается от физического удаления и почему это важно для холдинга?

> `SoftDeleteMixin` добавляет поле `deleted_at`. При «удалении» запись не стирается, а получает дату удаления. Данные остаются в базе — для аудита, для истории, для возможного восстановления. Это важно при возникновении споров, проверок или ошибочных удалений.
