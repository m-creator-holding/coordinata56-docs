## Коротко о главном

Секреты — пароли, ключи, токены — нельзя писать прямо в код. Почему? Потому что код хранится в системе контроля версий (Git), и любой, кто видит историю проекта, видит секрет — навсегда. Даже если Вы удалили файл из кода, Git хранит всю историю изменений. Секрет хранится в специальном файле `.env`, который в Git не попадает никогда.

---

## Зачем это нужно

Представьте: у Вас на АЗС есть сейф с дневной выручкой. Комбинация от сейфа — это секрет. Вы можете хранить комбинацию двумя способами:

**Способ 1 (плохой)**: написать комбинацию на доске объявлений в офисе. Все сотрудники видят её каждый день. Бывший сотрудник, уволенный год назад, до сих пор её помнит. Клиент, зашедший в офис, тоже увидел.

**Способ 2 (правильный)**: комбинация записана в закрытом личном ежедневнике директора, который сдаётся в хранилище и недоступен посторонним.

В программировании:
- **Код** = доска объявлений в офисе (видят все, кто работает с проектом, + вся история сохраняется навсегда)
- **`.env` файл** = закрытый ежедневник директора (не попадает в историю, есть только на конкретном сервере)

## Что такое секрет в программировании

Секрет — любое значение, знание которого даёт несанкционированный доступ к чему-либо:

- **JWT_SECRET_KEY** — ключ, которым сервер «подписывает» токены входа. Зная его, можно сделать фальшивый токен и войти в систему как любой пользователь
- **DATABASE_URL с паролем** — строка подключения к базе данных. Зная её, можно подключиться к базе напрямую и читать или изменять любые данные
- **API-ключ Anthropic** — доступ к ИИ, за который платит Ваш аккаунт. Украли — платите чужие счета
- **GitHub Personal Access Token (PAT)** — доступ к репозиторию кода. Украли — могут изменить или удалить весь код

## Три реальных инцидента нашего проекта

### Инцидент 1: JWT_SECRET_KEY в `.env.dev` — placeholder блокирован

Когда 2026-04-17 мы настраивали окружение разработки, в файле `.env.dev` стояло значение:

```
JWT_SECRET_KEY=change_me
```

Приложение не запустилось. Не потому что сломалось — а намеренно. Посмотрите на код в `backend/app/core/config.py`:

```python
forbidden = {"change_me", "secret", "test", "default"}
lower = v.lower()
if any(word in lower for word in forbidden):
    raise ValueError(
        "JWT_SECRET_KEY содержит словарное слово — задайте случайные 32+ символа"
    )
```

Система проверяет, не используется ли «словарное» значение как секрет. `change_me`, `secret`, `test`, `default` — это слова, которые разработчики часто пишут как заглушки. Система их блокирует. Это называется **валидатор конфигурации** — приложение лучше упадёт при старте с понятной ошибкой, чем будет работать с небезопасным секретом.

Правильное значение JWT_SECRET_KEY — случайная строка длиной 32+ символа. Например, генерируется командой:

```bash
python -c "import secrets; print(secrets.token_urlsafe(32))"
```

Результат: что-то вроде `K9mPvXqLwNrZ4dYoEh7bJcUsFtGiA2kQ8nV0xRyM1l`. Это нельзя угадать или подобрать словарём.

### Инцидент 2: Anthropic API Key в прототипе

Ранее в прототипе системы был hardcoded (встроенный прямо в код) ключ Anthropic. Это выглядело примерно так:

```python
# ПЛОХО — никогда так не делать
client = anthropic.Anthropic(api_key="sk-ant-api03-РЕАЛЬНЫЙ_КЛЮЧ_ЗДЕСЬ...")
```

Если бы такой код попал в Git — ключ был бы виден всем, кто имеет доступ к репозиторию. И был бы виден навсегда — в истории Git даже после удаления файла.

Правильный способ:

```python
# ХОРОШО — читаем из переменной окружения
import os
client = anthropic.Anthropic(api_key=os.environ.get("ANTHROPIC_API_KEY"))
```

### Инцидент 3: GitHub PAT в `.mcp.json` — только `.gitignore` спас

В файле `.mcp.json` хранятся настройки инструментов разработки. Туда попал Personal Access Token (персональный токен доступа) к GitHub. Только потому, что этот файл был перечислен в `.gitignore` (список файлов, которые Git игнорирует), токен не попал в публичную историю репозитория. Спасло правило, добавленное заранее. Без него — инцидент.

## Как работает `.env`

Файл `.env` — это текстовый файл с переменными окружения (environment variables — настройки, которые программа читает из операционной системы, а не из своего кода).

Выглядит так:

```bash
# Файл: backend/.env  (НИКОГДА не попадает в Git)
JWT_SECRET_KEY=K9mPvXqLwNrZ4dYoEh7bJcUsFtGiA2kQ8nV0xRyM1l
DATABASE_URL=postgresql+psycopg://coordinata:реальный_пароль@localhost:5432/coordinata56
INITIAL_OWNER_EMAIL=martin@coordinata.ru
INITIAL_OWNER_PASSWORD=реальный_пароль_владельца
```

Программа читает значения из этого файла при запуске. Код при этом выглядит так:

```python
# Файл: backend/app/core/config.py
class Settings(BaseSettings):
    jwt_secret_key: str = Field(...)        # читается из JWT_SECRET_KEY
    database_url: str = Field(default=...)  # читается из DATABASE_URL
```

`BaseSettings` — это механизм, который автоматически берёт значения из `.env` файла или из переменных окружения операционной системы. Разработчик пишет только имя настройки — значение подставляется из окружения, не из кода.

В репозитории Git лежит только файл-пример:

```bash
# Файл: backend/.env.example  (попадает в Git — содержит только названия, не значения)
JWT_SECRET_KEY=             # обязательный, минимум 32 символа
DATABASE_URL=postgresql+psycopg://user:password@localhost:5432/dbname
INITIAL_OWNER_EMAIL=
INITIAL_OWNER_PASSWORD=
```

## Git никогда не забывает

Это фундаментальный факт о системах контроля версий. Git хранит полную историю всех изменений. Если Вы сегодня добавили секрет в файл и через час его удалили — в истории Git секрет останется навсегда.

Представьте это как бухгалтерский журнал с невозможностью исправлений. Каждая запись — навсегда. Вы можете добавить новую запись («удалено»), но старая останется читаемой.

Именно поэтому `.gitignore` — это не опциональная мера предосторожности, а обязательное правило. Файл `.gitignore` содержит список файлов и папок, которые Git полностью игнорирует:

```
# Файл: .gitignore (фрагмент нашего проекта)
.env
*.env
backend/.env
backend/.env.dev
.mcp.json
```

Если файл добавлен в `.gitignore` до первого коммита — он никогда не попадёт в историю. Если не добавлен, а секрет уже закоммичен — даже удаление файла не помогает. Требуется полная перезапись истории Git (`git filter-branch` или `git filter-repo`) и немедленная ротация (смена) всех скомпрометированных секретов.

## Хеширование паролей: почему нельзя хранить пароль в открытом виде

Пароли пользователей — особый вид секрета. Они хранятся в базе данных, но хранить их в открытом виде нельзя ни при каких обстоятельствах.

**Аналогия**: представьте, что все пароли сотрудников от корпоративной системы хранятся в таблице Excel на сервере. Если злоумышленник взломает сервер и получит доступ к базе — он мгновенно узнает пароли всех пользователей. Причём многие люди используют одинаковые пароли на разных сайтах — значит, взломав вашу систему, можно войти в их личные банковские кабинеты.

Решение — хранить не пароль, а его **хеш** (цифровой отпечаток).

**Хеш** — это результат одностороннего преобразования. По паролю легко вычислить хеш. По хешу — невозможно вычислить пароль обратно. Как отпечаток пальца: по человеку можно снять отпечаток, но по отпечатку нельзя восстановить человека.

В нашем проекте используется алгоритм **bcrypt** — один из самых надёжных для хеширования паролей. Посмотрите на код в `backend/app/core/security.py`:

```python
_pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

def hash_password(plain: str) -> str:
    return str(_pwd_context.hash(plain))
```

Функция `hash_password` принимает пароль в открытом виде (`plain`) и возвращает хеш. Результат выглядит примерно так:

```
$2b$12$EixZaYVK1fsbw1ZfbX3OXePaWxn96p36WQoeG6Lruj3vjPGga31lW
```

Это bcrypt-хеш пароля `secret`. Восстановить `secret` из этой строки — невозможно.

### Что такое «соль» (salt)

Есть атака на пароли, которая называется **«атака радужными таблицами» (rainbow table attack)**. Злоумышленник заранее вычисляет хеши для миллионов популярных паролей и хранит таблицу `пароль → хеш`. Если он получил хеш из базы — просто ищет его в своей таблице и находит пароль за секунды.

Защита — **соль** (salt). При хешировании к паролю добавляется случайная строка (соль), которая генерируется заново для каждого пользователя. Даже если два пользователя используют одинаковый пароль — их хеши будут разными, потому что соли разные. Радужная таблица против такого защитника бессильна.

В bcrypt соль генерируется автоматически и встраивается в сам хеш. Поэтому одинаковые пароли дают разные хеши при каждом вызове:

```python
hash_password("qwerty123")  # → $2b$12$abc...xyz  (результат 1)
hash_password("qwerty123")  # → $2b$12$def...uvw  (результат 2, другой!)
```

Оба хеша верны для пароля `qwerty123`, просто соли разные. При проверке пароля bcrypt сам извлекает соль из хеша и использует её.

## Правило: `os.environ.get()`, никогда литерал

**Литерал** — значение, написанное прямо в коде. Например, строка `"K9mPvXqLwNrZ4d"` прямо в Python-файле — это литерал.

Правило проекта (из `CLAUDE.md` и `CODE_OF_LAWS`): любой секрет читается через `os.environ.get()` или механизм конфигурации. Никогда — литерал.

```python
# ЗАПРЕЩЕНО — литерал секрета в коде
API_KEY = "sk-ant-api03-реальный-ключ"

# ПРАВИЛЬНО — читаем из окружения
import os
API_KEY = os.environ.get("ANTHROPIC_API_KEY")

# ЕЩЁ ЛУЧШЕ — через pydantic-settings (как в нашем проекте)
class Settings(BaseSettings):
    anthropic_api_key: str = Field(...)  # обязательно, нет дефолта
```

## Инструменты для обнаружения секретов

Проблема: человек может случайно написать секрет в код. Инструменты помогают это поймать автоматически.

**gitleaks** — сканирует репозиторий Git и находит секреты в коде и в истории коммитов. Знает паттерны для API-ключей разных сервисов (AWS, Anthropic, GitHub и сотен других).

**detect-secrets** — аналог от Yelp. Создаёт «базовую линию» (baseline) известных ложных срабатываний, чтобы не мешать работе.

**pre-commit hooks** — автоматические проверки, которые запускаются перед каждым коммитом. Если находят секрет — коммит блокируется, пока разработчик не уберёт секрет.

В плане нашего аудита (задача S-2 от quality-director) — запуск `gitleaks detect` по всему репозиторию входит в обязательный чек-лист перед production.

## Как объяснить на совещании

«Секреты — это как ключи от сейфа. Ключи хранятся в сейфе (`.env` файл на сервере), а не в публичном журнале (код в Git). Git хранит всю историю вечно — если ключ однажды попал туда, считайте, что замок скомпрометирован и его нужно менять. Пароли пользователей мы не храним — мы храним цифровые отпечатки пальцев (bcrypt-хеши), из которых невозможно восстановить оригинал.»

## Риски, если правило нарушено

| Что нарушено | Что происходит |
|---|---|
| JWT_SECRET_KEY в коде | Злоумышленник создаёт фальшивые токены и входит в систему как любой пользователь |
| DATABASE_URL с паролем в коде | Прямой доступ к базе данных: чтение, изменение, удаление всех данных |
| Пароли в открытом виде в базе | Взлом базы = мгновенный доступ к паролям всех пользователей |
| API-ключ в коде | Злоумышленник использует Ваш оплаченный аккаунт; Вы платите по его счетам |
| `.env` файл закоммичен в Git | Все перечисленные выше последствия сразу |

В нашем `CODE_OF_LAWS` (Книга IV, ст. 40 и 79): нарушение — P0, блокер, немедленная эскалация Владельцу.
