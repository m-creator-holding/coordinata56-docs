# Урок 9. Слой сервисов: где живёт бизнес-логика

**Блок 2 — Архитектура Backend** | Время чтения: ~17 минут

---

## Что Вы поймёте после этого урока

После этого урока Вы будете знать: что такое «слой сервисов», зачем бизнес-логику выносят отдельно от роутеров и от базы данных, как устроен `PaymentService` — один из ключевых сервисов M-OS — и что именно происходит при нажатии кнопки «Утвердить платёж».

---

## Зачем это важно для M-OS

Каждое бизнес-правило M-OS — «платёж нельзя согласовать дважды», «сумма не должна превысить 120% от договора», «удалить можно только черновик» — живёт в слое сервисов. Это «мозг» системы. Понимание этого слоя позволяет Вам оценивать полноту реализации: посмотрел на сервис — понял, какие правила реально работают, а каких ещё нет.

---

## Аналогия: производственный отдел на заводе

Представьте металлообрабатывающее производство. Там есть несколько отделов:

**Отдел приёма заявок** — принимает заявки от клиентов, проверяет, правильно ли оформлены документы, и передаёт в производство. Он не знает, как точить металл — только как принять и оформить.

**Производственный отдел** — знает все технологические правила: какой металл при какой температуре обрабатывать, какие допуски допустимы, как провести контроль качества. Именно здесь принимаются технологические решения.

**Склад** — хранит заготовки и готовые детали. Принимает и выдаёт. Не принимает решений о том, что делать с металлом.

В M-OS:

| Завод | M-OS |
|---|---|
| Отдел приёма заявок | Роутер (`api/payments.py`) |
| Производственный отдел | Сервис (`services/payment.py`) |
| Склад | Репозиторий (`repositories/payment.py`) |
| Склад + база хранения | База данных PostgreSQL |

Роутер принимает HTTP-запрос и передаёт данные в сервис. Сервис применяет бизнес-правила и обращается к репозиторию. Репозиторий работает с базой данных через SQLAlchemy.

Каждый отдел знает только своё дело. Это называется «разделение ответственности».

---

## Почему не писать логику прямо в роутере

Можно было бы написать всю проверку прямо в функции роутера. Многие начинающие команды так и делают. Но это создаёт проблемы:

**1. Дублирование.**
Если правило «платёж нельзя утвердить дважды» нужно и в API, и в Telegram-боте — придётся писать одно и то же в двух местах. Изменится правило — менять в двух местах, легко пропустить одно.

**2. Тестирование.**
Протестировать чистую бизнес-логику без HTTP-сервера гораздо проще. Сервис можно вызвать напрямую из тестов, без имитации HTTP-запросов.

**3. Читаемость.**
Роутер с 300 строками бизнес-логики невозможно читать. Чёткое разделение: роутер — транспорт, сервис — логика.

В нашем проекте это правило зафиксировано в комментарии к `payment.py`:

```
Слой не знает про FastAPI (Request, Response, status) — только бизнес-логика.
```

## Реальный файл: `backend/app/services/payment.py`

Разберём по частям наиболее важный сервис — платежей.

### Константы в начале файла

```python
_IMMUTABLE_STATUSES: frozenset[PaymentStatus] = frozenset(
    {PaymentStatus.APPROVED, PaymentStatus.REJECTED}
)

_APPROVABLE_STATUSES: frozenset[PaymentStatus] = frozenset(
    {PaymentStatus.DRAFT, PaymentStatus.PENDING}
)

_PAYABLE_CONTRACT_STATUSES: frozenset[str] = frozenset(
    {ContractStatus.ACTIVE.value, ContractStatus.COMPLETED.value}
)
```

Это не код как таковой — это **документация бизнес-правил в виде кода**. Посмотрев на эти три строки, любой член команды сразу понимает:

- Статусы `APPROVED` и `REJECTED` — терминальные: платёж нельзя изменить
- Одобрить/отклонить можно только из `DRAFT` или `PENDING`
- Создать платёж можно только к договору в статусе `ACTIVE` или `COMPLETED`

`frozenset` — неизменяемое множество. Как список, но нельзя добавить или удалить элемент после создания. Используется специально: правила не должны меняться в процессе работы программы.

### Метод `create`: создание платежа

```python
async def create(self, data: PaymentCreate, actor_id: int, ip: str | None, ...) -> Payment:
    contract_info = await self.repo.get_contract_amount_and_status(data.contract_id)

    if contract_info is None:
        raise NotFoundError("Contract", data.contract_id)

    contract_amount, contract_status = contract_info

    if contract_status not in _PAYABLE_CONTRACT_STATUSES:
        raise DomainValidationError(
            f"Нельзя создать платёж: договор id={data.contract_id} "
            f"находится в статусе '{contract_status}'. "
            "Разрешены только статусы: active, completed.",
            code="CONTRACT_NOT_PAYABLE",
        )

    payment = await self.repo.create(create_data)

    await self.audit.log(
        user_id=actor_id,
        action=AuditAction.CREATE,
        entity_type="Payment",
        entity_id=payment.id,
        changes={"after": _payment_to_dict(payment)},
        ip_address=ip,
    )

    return payment
```

Последовательность действий:

1. Проверить, существует ли договор
2. Проверить статус договора — нельзя создать платёж к договору в статусе «черновик» или «расторгнут»
3. Создать платёж в статусе DRAFT
4. Записать в журнал аудита: кто, когда, что создал
5. Вернуть созданный платёж

Обратите внимание: шаг 4 (аудит) — обязательный. В нашем проекте это правило зафиксировано в CLAUDE.md: «write-эндпоинт без `audit_service.log()` — это дефект».

### Метод `approve`: согласование платежа

```python
async def approve(self, payment_id: int, actor_id: int, ip: str | None, ...) -> Payment:
    payment = await self.get(payment_id)
    before = _payment_to_dict(payment)

    if payment.status not in _APPROVABLE_STATUSES:
        raise ConflictError(
            f"Платёж id={payment_id} в статусе '{payment.status.value}' "
            "нельзя согласовать. Допустимые статусы: draft, pending.",
            code="INVALID_STATUS_TRANSITION",
        )

    approved_total = await self.repo.sum_approved_amount_for_contract(
        payment.contract_id
    )

    contract_info = await self.repo.get_contract_amount_and_status(payment.contract_id)
    contract_amount = contract_info[0] if contract_info is not None else 0

    limit_pct = get_settings().payment_overrun_limit_pct
    if (approved_total + payment.amount_cents) * 100 > contract_amount * (100 + limit_pct):
        raise ConflictError(
            f"Одобрение платежа id={payment_id} превысит лимит "
            f"{100 + limit_pct}% суммы договора.",
            code="PAYMENT_EXCEEDS_CONTRACT",
        )

    update_data = {
        "status": PaymentStatus.APPROVED,
        "approved_at": datetime.now(UTC),
        "approved_by_user_id": actor_id,
    }

    payment = await self.repo.update(payment, update_data)

    await self.audit.log(
        user_id=actor_id,
        action=AuditAction.UPDATE,
        entity_type="Payment",
        entity_id=payment_id,
        changes={
            "before": before,
            "after": _payment_to_dict(payment),
            "meta": {"transition": "approved", "from_status": before["status"]},
        },
        ...
    )

    return payment
```

Здесь реализовано несколько бизнес-правил:

**Правило 1. Нельзя согласовать уже согласованный платёж.**
`if payment.status not in _APPROVABLE_STATUSES` — проверяет статус перед действием.

**Правило 2. Нельзя превысить 120% суммы договора.**
```python
if (approved_total + payment.amount_cents) * 100 > contract_amount * (100 + limit_pct)
```
Это точная арифметика с целыми числами (копейки, не рубли). Важная деталь: используем `amount_cents` (копейки), не рубли с копейками. Дробные числа в вычислениях с деньгами могут накапливать погрешность. Целые числа (копейки) — точны всегда. `limit_pct` берётся из настроек (`get_settings()`) — это значит, что лимит можно изменить без правки кода.

**Правило 3. Аудит должен содержать `before` и `after`.**
```python
changes={"before": before, "after": _payment_to_dict(payment), "meta": {...}}
```
Для операций изменения в журнал записывается состояние ДО и ПОСЛЕ. Это позволяет потом ответить на вопрос: «какие именно поля изменились, и какими они были раньше?».

### Метод `reject`: отклонение с причиной

```python
async def reject(self, payment_id: int, reason: str, actor_id: int, ...) -> Payment:
    payment = await self.get(payment_id)

    if payment.status not in _APPROVABLE_STATUSES:
        raise ConflictError(...)

    update_data = {
        "status": PaymentStatus.REJECTED,
        "rejected_at": datetime.now(UTC),
        "rejected_by_user_id": actor_id,
        "rejection_reason": reason,
    }

    payment = await self.repo.update(payment, update_data)

    await self.audit.log(
        ...,
        changes={
            "before": before,
            "after": after,
            "meta": {
                "transition": "rejected",
                "from_status": before["status"],
                "reason": reason,
            },
        },
    )

    return payment
```

Причина отклонения (`reason`) — обязательный параметр на уровне метода. Попытка вызвать `reject` без `reason` приведёт к ошибке Python ещё до отправки запроса в базу. Это программная гарантия бизнес-правила.

## Слой сервисов: что знает и что не знает

Сервис **знает**:
- Бизнес-правила предметной области
- Модели данных (Payment, Contract, User)
- Репозиторий и сервис аудита

Сервис **не знает**:
- Как пришёл запрос (HTTP, Telegram, CLI)
- IP-адрес пользователя (берёт только то, что ему передали как параметр)
- Как отобразить ошибку пользователю (бросает исключение — роутер его перехватит)
- `status_code` HTTP-ответа

Это означает: один и тот же `PaymentService.approve()` можно вызвать и из HTTP-роутера, и из Telegram-бота, и из административного скрипта. Бизнес-правила одинаковы везде.

## Архитектура: три слоя в стеке

```
HTTP-запрос
     ↓
Роутер (api/payments.py)       ← «окошко МФЦ»: принять, проверить формат, передать
     ↓
Сервис (services/payment.py)   ← «производственный отдел»: применить правила
     ↓
Репозиторий (repositories/)    ← «склад»: положить/взять из базы
     ↓
База данных (PostgreSQL)        ← «физическое хранилище»
```

Каждый слой может зависеть только от слоя ниже, не выше. Роутер знает про сервис. Сервис знает про репозиторий. Репозиторий знает про SQLAlchemy. Никто не смотрит «вверх».

## Пример своими словами

На совещании Вы можете сказать так:

> «Когда финансовый директор нажимает "Утвердить платёж", запрос проходит три слоя. Первый — принимает запрос и проверяет, что пользователь вообще имеет право нажимать эту кнопку. Второй — проверяет бизнес-правила: нет ли двойного утверждения, не превышает ли сумма 120% договора. Третий — записывает результат в базу данных. Если любое правило нарушено — операция не выполняется, и в журнале остаётся запись о попытке. Все бизнес-правила живут в одном месте — в сервисе. Поэтому добавить новое правило можно в одном файле, а не искать по всему коду.»

## Риски и типичные ошибки

**1. Бизнес-логика в роутере.**
Разработчик «для скорости» написал проверку прямо в функции роутера. Через месяц добавили Telegram-бот — правило не работает там, потому что его нет в сервисе. Стандарт нашего проекта: роутер — только транспорт.

**2. Бизнес-логика в репозитории.**
Другая крайность: разработчик добавил в метод `repo.create()` проверку уникальности. Репозиторий становится «умным», его сложнее тестировать. Правила предметной области — в сервисе, доступ к данным — в репозитории.

**3. Пропущенный аудит.**
Добавили новый метод (например, `transfer_to_counterparty`) и забыли вызвать `audit.log()`. Действие исполнено, но в журнале — пусто. По регламенту нашего проекта это дефект уровня P0.

**4. Работа с деньгами в рублях (float).**
```python
# Неправильно:
amount = 1500.50  # float, накапливает погрешность
# Правильно:
amount_cents = 150050  # целое число копеек
```
В нашем проекте все суммы хранятся в копейках (`amount_cents`). Это не случайность — это осознанное архитектурное решение.

**5. Не проверить статус перед изменением.**
Если пропустить проверку `if payment.status not in _APPROVABLE_STATUSES` — можно дважды согласовать платёж. Машина не знает, что это «неправильно» — она делает то, что написано. Проверки статусов обязательны.

## Проверка понимания

**Вопрос 1.** Финансист жалуется: «Я нажал «Утвердить», кнопка стала неактивной, но через несколько секунд состояние не изменилось, и в журнале ничего нет». Где искать проблему — в роутере, в сервисе или в репозитории?

> Скорее всего, в сервисе: бизнес-правило проверки отработало и выбросило исключение (например, платёж уже в терминальном статусе, или сумма превышает лимит). Нужно смотреть логи сервиса — там будет конкретный код ошибки. В журнале аудита попытка не записывается, если операция не прошла до вызова `audit.log()`.

**Вопрос 2.** Команда говорит: «Мы хотим добавить правило: если контрагент имеет просроченные платежи по другим договорам, новый платёж нельзя создать». Куда именно в коде должно попасть это правило?

> В метод `create` сервиса `PaymentService`. Это бизнес-правило, не транспортное правило и не правило хранения данных. Роутер только принял запрос — он не должен знать про просроченные платежи.

**Вопрос 3.** Почему суммы платежей в M-OS хранятся в копейках (`amount_cents`) как целые числа, а не в рублях как дробные (`amount = 1500.50`)?

> Дробные числа (float) в компьютере хранятся приблизительно — могут накапливаться ошибки округления. В финансовых расчётах это недопустимо: 1500.50 * 3 может дать не 4501.50, а 4501.4999999999. Целые числа (копейки) точны абсолютно. `150050 * 3 = 450150` — без погрешностей.
