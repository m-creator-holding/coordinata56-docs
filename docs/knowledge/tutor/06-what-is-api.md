# Урок 6. Что такое API и как работает REST

**Блок 2 — Архитектура Backend** | Время чтения: ~16 минут

---

## Что Вы поймёте после этого урока

После этого урока Вы будете знать: что такое API, почему он устроен как «окошки», что означают слова GET / POST / PATCH / DELETE в коде нашего проекта, как M-OS принимает запросы от браузера и возвращает данные, и что происходит внутри файла `backend/app/api/projects.py`, который команда упоминает в каждом отчёте.

---

## Зачем это важно для M-OS

M-OS состоит из двух частей: **backend** (серверная сторона, Python) и **frontend** (то, что видит пользователь в браузере). Эти две части общаются через API. Каждый раз, когда сотрудник открывает список проектов в браузере — браузер делает запрос к API. Каждый раз, когда бухгалтер утверждает платёж — это тоже запрос к API.

Понимание API — это понимание «переговорного стола» между интерфейсом и данными.

---

## Аналогия: окошки в МФЦ

Представьте МФЦ. Внутри — огромный архив документов, базы данных, сотрудники. Снаружи — граждане. Гражданин не заходит в архив сам и не копается в базах. Он подходит к **конкретному окошку** с **конкретным запросом**.

Окошка — разные. Одно выдаёт справки (только читает, ничего не меняет). Другое принимает заявления (создаёт новые записи). Третье вносит изменения в существующие записи. Четвёртое аннулирует документы.

**API — это набор таких окошек для программ.**

| МФЦ | API |
|---|---|
| Окошко «Выдача справок» | GET-эндпоинт (получить данные) |
| Окошко «Приём заявлений» | POST-эндпоинт (создать запись) |
| Окошко «Внесение изменений» | PATCH-эндпоинт (изменить часть записи) |
| Окошко «Аннулирование» | DELETE-эндпоинт (удалить запись) |
| Номер окошка и кабинет | URL-адрес эндпоинта |
| Бланк заявления | JSON-тело запроса |
| Выданный документ | JSON-ответ |

## Конкретика: REST API

**REST** (от английского Representational State Transfer) — это набор договорённостей о том, как строить API. Не технология, не язык — именно договорённость. Все современные веб-сервисы используют её, потому что она предсказуема.

Суть в трёх правилах:

**1. Адрес (URL) — это «что».**
`/projects` — это «проекты». `/projects/42` — это «проект с номером 42». `/payments` — это «платежи».

**2. Метод HTTP — это «как».**
Один и тот же адрес может делать разное в зависимости от метода:
- `GET /projects` — получить список проектов
- `POST /projects` — создать новый проект
- `PATCH /projects/42` — изменить проект 42
- `DELETE /projects/42` — удалить проект 42

**3. Данные — в JSON.**
JSON (JavaScript Object Notation — формат записи объектов) — это текстовый формат для передачи данных. Выглядит как словарь:

```json
{
  "code": "C56",
  "name": "Координата 56",
  "status": "active"
}
```

Это не код на Python и не на JavaScript — это просто универсальный формат. Браузер отправляет такой текст серверу, сервер отвечает таким же текстом.

---

## HTTP-методы: таблица

| Метод | Смысл | Аналогия | Что делает в M-OS |
|---|---|---|---|
| GET | Получить данные | «Покажи мне» | Список проектов, детали платежа |
| POST | Создать новое | «Создай новое» | Новый проект, новый договор |
| PATCH | Изменить частично | «Поправь вот это» | Сменить статус, обновить название |
| PUT | Заменить полностью | «Перепиши целиком» | Редко используется, PATCH предпочтительнее |
| DELETE | Удалить | «Убери это» | Мягкое удаление проекта |

В M-OS мы почти не используем PUT — только PATCH. Это осознанное решение: PATCH требует передавать только изменяемые поля, а не весь объект целиком. Меньше данных в запросе — меньше риск случайно затереть поле, которое не трогали.

## Реальный файл: `backend/app/api/projects.py`

Откройте файл `backend/app/api/projects.py`. Это роутер (router) — файл, который определяет все «окошки» для работы с проектами. Разберём его построчно по смыслу.

### Строка 31: объявление роутера

```python
router = APIRouter(prefix="/projects", tags=["projects"])
```

`prefix="/projects"` означает: все окошки в этом файле начинаются с адреса `/projects`. Команда FastAPI (фреймворк, на котором работает наш backend) автоматически добавляет этот префикс ко всем адресам внутри файла.

`tags=["projects"]` — это метка для автоматической документации. FastAPI генерирует интерактивную документацию API (по адресу `/docs`), и там все наши окошки будут сгруппированы под заголовком «projects».

### Строки 46–85: первое окошко, GET /projects/

```python
@router.get(
    "/",
    response_model=ListEnvelope[ProjectRead],
    summary="Список проектов",
)
async def list_projects(
    is_archived: bool | None = None,
    pagination: PaginationParams = Depends(),
    db: AsyncSession = Depends(get_db),
    user_pair: tuple[User, UserContext] = Depends(get_current_user),
) -> ListEnvelope[ProjectRead]:
```

Разбор по частям:

- `@router.get("/")` — это «декоратор» (пометка над функцией). Говорит FastAPI: «Когда браузер делает GET-запрос на адрес `/projects/` — запусти вот эту функцию».
- `response_model=ListEnvelope[ProjectRead]` — описание формата ответа. FastAPI проверяет, что функция возвращает именно такой формат, и автоматически описывает его в документации.
- `async def list_projects(...)` — сама функция-обработчик. `async` означает «асинхронная» — она умеет не блокировать сервер во время ожидания ответа от базы данных.
- `Depends(get_db)` — зависимость (Depends — буквально «зависит от»). FastAPI автоматически создаёт соединение с базой данных и передаёт его в функцию. Когда функция заканчивает работу — соединение закрывается. Это как «кредитная карта»: берёшь на время, возвращаешь по окончании.
- `Depends(get_current_user)` — проверяет JWT-токен (токен авторизации) пользователя. Если токен недействителен — вернёт ошибку 401 ещё до входа в функцию.

### Строки 137–166: POST /projects/ (создать проект)

```python
@router.post(
    "/",
    response_model=ProjectRead,
    status_code=status.HTTP_201_CREATED,
)
async def create_project(
    body: ProjectCreate,
    request: Request,
    db: AsyncSession = Depends(get_db),
    current_user: User = Depends(require_role(UserRole.OWNER)),
) -> ProjectRead:
```

Ключевое отличие от GET:

- `status_code=status.HTTP_201_CREATED` — при успехе сервер вернёт код 201 (вместо 200). HTTP-коды — это стандартный язык: 200 «ОК», 201 «создано», 404 «не найдено», 403 «нет прав», 409 «конфликт» (например, дубликат).
- `body: ProjectCreate` — FastAPI автоматически прочитает JSON из тела запроса и проверит его на соответствие схеме `ProjectCreate`. Если поле обязательное — отсутствие вернёт ошибку 422 (неверный формат).
- `Depends(require_role(UserRole.OWNER))` — проверка роли. Только пользователь с ролью «owner» может создавать проекты. Все остальные получат 403.

### PATCH и DELETE

```python
@router.patch("/{project_id}", ...)  # изменить проект
@router.delete("/{project_id}", ...)  # удалить проект
```

`{project_id}` в фигурных скобках — это «параметр пути». Когда приходит запрос на `/projects/42` — FastAPI автоматически извлекает `42` и передаёт его в функцию как `project_id=42`.

## Что происходит при одном запросе: цепочка

Когда сотрудник нажимает «Создать проект» в браузере, происходит следующее:

```
Браузер
  → отправляет POST /projects/ с JSON {"code": "C56", "name": "Координата 56"}
  → с заголовком Authorization: Bearer <токен>

FastAPI (наш сервер)
  → находит нужное окошко (функцию create_project)
  → проверяет токен через get_current_user
  → проверяет роль через require_role(OWNER)
  → парсит JSON через схему ProjectCreate
  → открывает соединение с БД через get_db
  → вызывает функцию create_project

create_project
  → вызывает ProjectService.create(data, actor_id, ...)

ProjectService
  → проверяет, не занят ли code
  → создаёт запись в БД
  → записывает в audit_log (кто, когда, что создал)
  → возвращает созданный объект

FastAPI
  → сериализует объект в JSON через ProjectRead
  → возвращает HTTP 201 с JSON

Браузер
  → получает ответ {"id": 1, "code": "C56", "name": "Координата 56", ...}
  → отображает новый проект в списке
```

Заметьте: роутер (файл `api/projects.py`) не занимается бизнес-логикой — только принимает запросы и передаёт дальше. Проверки «можно ли создать» — в сервисе. Это называется «разделение ответственности» — каждый слой знает только своё дело.

## Пример своими словами

На совещании Вы можете сказать так:

> «Наш API — это набор окошек, через которые браузер общается с базой данных. Когда бухгалтер утверждает платёж, браузер отправляет запрос PATCH с номером платежа. Сервер проверяет права, применяет бизнес-правила и записывает результат. Браузер не лезет в базу напрямую — только через эти окошки. Поэтому мы контролируем каждое действие: можно проверять права, писать журнал аудита, валидировать данные.»

## Риски и типичные ошибки

**1. Перепутать метод.**
Если фронтенд отправляет GET вместо POST — данные не будут переданы (GET не имеет тела запроса). Сервер либо вернёт ошибку 405 (метод не разрешён), либо проигнорирует. Это частая ошибка при первой интеграции.

**2. Открыть лишнее окошко без проверки прав.**
Если добавить эндпоинт и забыть поставить `Depends(require_role(...))` — любой аутентифицированный пользователь сможет делать то, что не должен. В нашем проекте это правило зафиксировано в CLAUDE.md: каждый write-эндпоинт обязан иметь проверку роли.

**3. Вернуть лишние данные.**
Если в ответе вернуть объект с полем `password_hash` — это утечка. Поэтому мы используем отдельные схемы ответа (`ProjectRead`, `PaymentRead`), которые явно перечисляют, какие поля включить в ответ.

**4. Неверные HTTP-статусы.**
Возвращать 200 вместо 404 или 500 вместо 400 — значит давать фронтенду неверную информацию. Хороший API говорит точно: «не нашёл» (404), «нет прав» (403), «конфликт данных» (409), «ваши данные неверны» (422).

## Проверка понимания

**Вопрос 1.** Сотрудник жалуется: «Я нажал «Редактировать проект», но изменения не сохранились, хотя ошибки не было». Какой вопрос Вы зададите разработчику?

> Какой метод HTTP использует кнопка «Редактировать» — PATCH или GET? Если GET — данные не отправляются в теле запроса, изменений быть не может.

**Вопрос 2.** Почему файл `api/projects.py` не содержит никакой проверки уникальности кода проекта, хотя в документации написано, что код должен быть уникальным?

> Потому что проверка уникальности — это бизнес-правило, оно живёт в `services/project.py`. Роутер только принимает запрос и передаёт данные в сервис. Разделение ответственности.

**Вопрос 3.** Команда говорит: «Мы добавили новый эндпоинт для экспорта отчёта». Какой HTTP-метод, скорее всего, используется, и почему?

> GET — потому что экспорт только читает данные и ничего не меняет. Использование POST для чтения было бы нарушением REST-договорённостей и сбивало бы с толку.
