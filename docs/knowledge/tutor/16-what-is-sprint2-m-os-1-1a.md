## Коротко о главном

Sprint 2 M-OS-1.1A — второй спринт скелета системы. Если Sprint 1 строил стены и пропуска, Sprint 2 строит внутреннюю связь между блоками и правила общения с внешним миром. Четыре механизма — невидимые, но именно они не дают системе превратиться в «лапшу» из прямых соединений.

---

## 1. Шина событий (Event Bus)

### Зачем

Когда в системе что-то происходит — платёж одобряется, договор подписывается, объект сдаётся — об этом должны узнать несколько модулей: бухгалтерия, отчёты, журнал аудита, уведомления. Самый очевидный способ — каждый модуль напрямую вызывает другой. Но если модулей 20 — это 400 возможных прямых соединений. Поменяй один — сломаешь ещё пять.

### Аналогия

Стройплощадка без рации. Водитель привёз бетон. Ему нужно оповестить прораба, бетонщиков, охранника на въезде и складского. Если он лично обходит каждого — теряет 20 минут, может кого-то не застать, забыть. Но если на площадке есть общий канал связи — он один раз говорит в рацию «бетон на въезде», и каждый, кому это нужно, слышит сам и реагирует. Водитель не знает, кто и как отреагирует — его дело объявить.

### Конкретика

Шина событий — это внутренний «почтамт»: один модуль публикует событие с названием и данными, другие модули подписаны на нужные им события и реагируют автоматически.

```
Модуль «Платежи»
    └── публикует: payment.approved { id: 1001, amount: 500000 }

Подписчики:
    Бухгалтерия     ← слышит payment.approved → делает проводку
    Отчёты          ← слышит payment.approved → обновляет дашборд
    Аудит           ← слышит payment.approved → записывает в журнал
```

Модуль «Платежи» не знает и не заботится о том, кто подписан. Добавить нового подписчика — не нужно трогать модуль «Платежи».

### Риски без этого

- Прямые вызовы между модулями — «лапша»: один модуль упал, потянул за собой три других.
- Добавление нового реагирующего модуля требует правки уже работающего кода вместо просто новой подписки.

## 2. Двойная шина (Dual Bus)

### Зачем

В системе есть два принципиально разных типа сообщений: бизнес-события (платёж одобрен, договор подписан) и команды ИИ-субагентам (запустить анализ, сформировать отчёт, проверить аномалию). Смешивать их в одном канале опасно: бизнес-модули могут случайно «услышать» команды ИИ и среагировать, а команды ИИ могут засорять бизнес-логику непредвиденными побочными эффектами.

### Аналогия

На большой стройке два отдельных канала рации: один — для бригад (бетон, арматура, кровля), другой — для охраны и видеонаблюдения. Бригадир не слышит переговоры охраны о подозрительном человеке у забора — это не его задача и только мешает работе. Охрана не слышит «бетон на въезде» — ей это не нужно. Смешай каналы — и оба потока превратятся в шум.

### Конкретика

Два изолированных потока:

```
Шина 1 — бизнес-события:
    payment.approved, contract.signed, object.completed
    → слушают: бухгалтерия, отчёты, аудит, уведомления

Шина 2 — команды ИИ-субагентам:
    agent.analyze_cashflow, agent.detect_anomaly, agent.generate_report
    → слушают: ИИ-субагенты, оркестратор
```

Бизнес-модули физически не подключены к шине 2 — даже если захотят, не смогут получить команды ИИ.

### Риски без этого

- ИИ-субагент ошибочно публикует событие в бизнес-шину → бухгалтерия делает фантомную проводку.
- Отладка превращается в поиск иголки в стоге: непонятно, кто опубликовал и кто среагировал.

## 3. Каркас защиты от внешних систем (Anti-Corruption Layer, ACL)

**ACL (Anti-Corruption Layer, каркас защиты)** — слой-«переводчик» между ядром системы и внешними источниками данных.

### Зачем

Банк присылает данные в своём формате. 1С — в своём. Росреестр — в третьем. Если ядро системы напрямую работает с этими форматами, оно «заражается» чужой логикой: поменял банк формат выписки — падает половина системы. Каркас защиты принимает удар на себя: снаружи — любой формат, внутри — всегда наш стандарт.

### Аналогия

Иностранный поставщик арматуры говорит только по-немецки, присылает документы в своём стандарте с немецкими единицами измерения. Если прораб начнёт сам разбираться с немецкими документами — потеряет время, сделает ошибку в расчётах. Но если нанять переводчика-нормировщика: он принимает немецкий документ, переводит в наш формат (рублей, метров, ГОСТов), и прорабу передаёт уже привычный наряд. Прораб не знает и не должен знать, что поставщик немецкий.

### Конкретика

```
Внешний банк присылает:
    { "trx_id": "A1B2", "amt": 500000, "ccy": "RUB", "dt": "18-04-2026" }

ACL принимает и нормализует:
    { "id": "A1B2", "amount": 500000, "currency": "RUB", "date": "2026-04-18" }

Ядро системы получает всегда один и тот же контракт.
```

Если завтра банк поменяет формат — меняем только ACL, ядро не трогаем.

### Риски без этого

- Банк меняет поле `dt` на `transaction_date` — падают все запросы к платёжному модулю.
- В ядро проникает «кривая» логика внешней системы: например, банк считает дату по UTC, ядро — по Москве, и везде незаметное расхождение.

## 4. Подключаемые поды (Pluggability)

**Pod (под, модульный блок)** — самостоятельный блок системы для одного направления бизнеса: коттеджи, МКД, АЗС, карьер, металл. Каждый под знает только своё.

### Зачем

Холдинг из 5 направлений — это 5 разных бизнесов. Коттеджный под нужен для «Координаты 56», АЗС-под — для восьми заправок. Если поды «вшиты» друг в друга, подключение нового направления — хирургическая операция на работающей системе. Если поды подключаются и отключаются как модули — добавление нового направления не затрагивает работающие.

### Аналогия

Пять отдельных строительных участков на одной территории холдинга. У каждого — свои ворота, свой журнал, своя бригада, свои нормы. Генеральный директор видит все участки через единый дашборд, но происходящее на участке «МКД» никак не влияет на работу участка «АЗС». Открыть новый участок «Металл» — значит поставить ворота, назначить прораба и подключить к общей системе отчётности. Не перекладывать фундамент уже работающих участков.

### Конкретика

Между подами — явные контракты (data contracts): чётко описанный список того, что один под может запросить у другого и в каком формате. Не «смотрит напрямую в базу» — а «запрашивает через контракт».

```
Под «Коттеджи» хочет проверить платёж → 
    запрашивает через контракт: get_payment(id, company_id)
    Получает: { id, amount, status, date }

Под «АЗС» отключён на обслуживание → 
    под «Коттеджи» продолжает работать без изменений
```

Поды не вызывают друг друга напрямую — только через ядро и явные контракты.

### Риски без этого

- Ошибка в поде «МКД» роняет систему всего холдинга, включая АЗС и карьер.
- Добавление нового направления требует тестирования всей системы с нуля — даже тех модулей, которых оно не касается.

## Зачем Вам знать это

Четыре механизма Sprint 2 — это архитектурные решения с прямыми бизнес-последствиями.
