## Зачем это важно для M-OS

Вся бизнес-логика M-OS — это условия. Принадлежит ли дом проекту? Может ли пользователь выполнить это действие? Нужно ли пересчитать цену при обновлении? Каждый раз система проходит через ветки `if`/`else`. Понимание этих конструкций — это понимание того, как система принимает решения.

## Аналогия: BPM-процесс в строительстве

Представьте регламент приёмки платежа:


Это и есть `if`/`elif`/`else` и `for` в форме бизнес-регламента. Python просто записывает тот же регламент в синтаксисе языка.

## Конструкция `if` — условие

Синтаксис:
```python
if условие:
    # что делать, если условие истинно
```

**Важно**: отступ (4 пробела) обязателен. Python определяет принадлежность строки к блоку по отступу, а не по скобкам. Это одна из особенностей Python — код выглядит как настоящий регламент.

Простой пример:
```python
if budget > 1000000:
    print("Требуется согласование директора")
```

---

## Конструкция `if`/`elif`/`else` — разветвление

`elif` — сокращение от «else if» (иначе если). Позволяет проверить несколько условий по порядку.

```python
if сумма > 5000000:
    approver = "Директор"
elif сумма > 1000000:
    approver = "Прораб"
else:
    approver = "Бухгалтер"
```

Python проверяет условия сверху вниз и входит в первое истинное. Остальные не проверяются.

## Конкретика: `if` в нашем коде сервиса

Файл `backend/app/services/budget_plan.py`, строки 95-108:

```python
extra_conditions = []

if project_id is not None:
    extra_conditions.append(BudgetPlan.project_id == project_id)
if category_id is not None:
    extra_conditions.append(BudgetPlan.category_id == category_id)
if stage_id is not None:
    extra_conditions.append(BudgetPlan.stage_id == stage_id)
if house_id is not None:
    extra_conditions.append(BudgetPlan.house_id == house_id)
```

Это функция фильтрации плановых строк бюджета. Логика:

- Начинаем с пустым списком условий
- Для каждого параметра фильтрации: **если** пользователь его передал (не `None`) — добавляем условие в список
- В итоге список содержит только те условия, которые реально нужны

Например, если пользователь хочет посмотреть бюджет только по `project_id=5` — передаёт только `project_id=5`, остальные `None`. В список попадёт одно условие. Запрос к базе данных будет отфильтрован только по проекту.

## Конкретика: `if`/`else` в проверке бизнес-правила

Файл `backend/app/services/budget_plan.py`, строки 64-69:

```python
house = await self.repo.get_house_by_id(house_id)
if house is None or house.project_id != project_id:
    raise ConflictError(
        f"Дом с id={house_id} не принадлежит проекту id={project_id}",
        code="BUSINESS_RULE_VIOLATION",
    )
```

Это проверка: «дом принадлежит этому проекту?»

- `house is None` — дома с таким ID не существует вообще
- `or` — логическое «или»: достаточно одного из условий быть истинным
- `house.project_id != project_id` — дом существует, но принадлежит другому проекту
- Если любое из условий выполнилось — `raise ConflictError(...)` — бросить ошибку (прервать выполнение с описанием проблемы)

**Зачем эта проверка?** Без неё злоумышленник или ошибившийся пользователь мог бы добавить строку бюджета к дому из чужого проекта. Это называется IDOR (несанкционированный доступ к ресурсу по ID). Проверка блокирует такое.

## Конструкция `for` — цикл по элементам

`for` повторяет блок кода для каждого элемента из последовательности.

```python
for элемент in последовательность:
    # что делать с каждым элементом
```

Простой пример:
```python
stages = ["Фундамент", "Коробка", "Кровля"]
for stage in stages:
    print(f"Стадия: {stage}")

# Выведет:
# Стадия: Фундамент
# Стадия: Коробка
# Стадия: Кровля
```

## Конкретика: `for` в проверке массовой загрузки

Файл `backend/app/services/budget_plan.py`, строки 268-275:

```python
for item in data.items:
    if item.house_id is not None:
        house = await self.repo.get_house_by_id(item.house_id)
        if house is None or house.project_id != data.project_id:
            raise ConflictError(
                f"Дом с id={item.house_id} не принадлежит проекту id={data.project_id}",
                code="BUSINESS_RULE_VIOLATION",
            )
```

Это валидация при массовой загрузке (`bulk_upsert`) строк бюджета.

Логика: для **каждой** строки в пакете `data.items`:
- Если у строки указан `house_id` — проверить принадлежность дома проекту
- Если хотя бы одна строка не прошла проверку — весь пакет отклоняется с ошибкой

Комбинация `for` + `if` — одна из самых частых в нашем коде. Перебираем список, для каждого элемента проверяем условие.

## Конкретика: `for` в очистке данных

Файл `backend/app/services/audit.py`, строки 79-85:

```python
if changes is not None:
    sanitized_changes = {}
    for key, val in changes.items():
        if isinstance(val, dict):
            sanitized_changes[key] = _sanitize(val)
        else:
            sanitized_changes[key] = val
```

Здесь `changes.items()` возвращает пары (ключ, значение) из словаря. Цикл проходит по каждой паре:
- Если значение само является словарём — очистить его через `_sanitize()`
- Иначе — оставить как есть

Результат собирается в `sanitized_changes` — новый словарь без чувствительных данных.

## Конструкция `while` — цикл с условием

`while` повторяет блок, пока условие истинно. В нашем backend-коде встречается редко (чаще в административных скриптах), но важно понимать принцип.

```python
attempts = 0
while attempts < 3:
    result = try_connect_to_database()
    if result.success:
        break  # выйти из цикла
    attempts += 1  # увеличить счётчик на 1
```

Аналогия: охранник на проходной пускает попытки входа, пока счётчик не достиг 3. После третьей неудачной — блокирует.

`break` — команда «выйти из цикла немедленно». Используется когда условие выхода стало истинным внутри цикла, а не в заголовке `while`.

## Логические операторы в условиях

В условиях часто используются три оператора:

| Оператор | Значение | Пример |
|---|---|---|
| `and` | оба должны быть истинны | `is_active and role == "OWNER"` |
| `or` | хотя бы одно истинно | `house is None or house.project_id != project_id` |
| `not` | обратное значение | `not is_active` (значит: неактивен) |

Из нашего кода — классический `or` для защитной проверки:
```python
if house is None or house.project_id != project_id:
    raise ConflictError(...)
```

«Дома нет» ИЛИ «дом есть, но не тот» — в обоих случаях ошибка.

---

## Как вложенность отступов структурирует логику

Python использует отступы для обозначения вложенности. Это ключевое отличие от большинства языков (где используются фигурные скобки `{}`):

```python
for item in data.items:           # уровень 1 — для каждой строки
    if item.house_id is not None: # уровень 2 — если есть дом
        house = await ...         # уровень 3 — получить дом
        if house is None:         # уровень 3 — если дом не найден
            raise ConflictError() # уровень 4 — ошибка
```

Каждый отступ — это «вложенность» в предыдущий блок. Код сам показывает свою структуру визуально.

## Риски и типичные ошибки

**1. Фильтрация после выборки, а не в запросе.**
Ошибка: сначала загрузить все 85 домов из базы, потом в Python через `if` отфильтровать нужные. Правильно: условие `if` нужно отправить в базу данных через `WHERE` — тогда из базы придут уже только нужные записи. В нашем проекте это правило зафиксировано в CLAUDE.md: «Никаких фильтров коллекций постобработкой в Python после LIMIT».

**2. Забыть про `None` в условии.**
Если написать `if house.project_id != project_id` без предварительной проверки `if house is None` — система упадёт с ошибкой, когда дом не найден (нельзя обращаться к полю `project_id` у значения `None`). Поэтому всегда `house is None or house.project_id != project_id` — сначала проверяем существование.

**3. Бесконечный цикл `while`.**
Если условие выхода из `while` никогда не становится ложным — программа зависает. Защита: всегда предусматривать счётчик попыток или таймаут.

## Проверка понимания

**Вопрос 1.** Почему в проверке `if house is None or house.project_id != project_id` первым стоит именно `house is None`?

> Python вычисляет условия слева направо. Если `house is None` истинно — правая часть `house.project_id != project_id` не вычисляется вообще (потому что `or` — «хватит одного»). Это защита: обращение к `house.project_id` когда `house` равен `None` вызвало бы ошибку. Порядок важен.

**Вопрос 2.** Что произойдёт в функции `bulk_upsert`, если из 50 строк бюджета 49 корректны, а 1 указывает на дом из другого проекта?

> Весь пакет будет отклонён. Функция `raise ConflictError(...)` внутри цикла `for` немедленно прерывает выполнение — никакие строки не сохраняются. Это транзакционное поведение: либо всё, либо ничего.

**Вопрос 3.** Чем `for` отличается от `while` в практическом применении?

> `for` используют когда заранее известна последовательность (список строк, список элементов). `while` — когда неизвестно заранее, сколько раз нужно повторить, и выход зависит от условия (например, ждать успешного соединения с базой, пробуя не более 3 раз).
