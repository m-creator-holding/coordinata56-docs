## Зачем это важно для M-OS

Типы данных — это фундамент. Каждое поле в базе данных имеет тип. Когда Вы видите в отчёте «поле `amount_cents` — BigInteger» или «поле `is_active` — Boolean» — это не случайный выбор. Неправильный тип данных может привести к потере денег (число с плавающей точкой для суммы), к невозможности искать по полю, или к переполнению (число слишком большое для выбранного типа). Понимание типов — это понимание того, как M-OS хранит информацию о холдинге.

---

## Аналогия: складские ячейки разного формата

Представьте складской комплекс с ячейками хранения. У каждой ячейки — жёсткий формат:

| Ячейка | Что хранит | Пример |
|---|---|---|
| Ячейка «Целое число» | Только целые числа | 85, -3, 1000000 |
| Ячейка «Текст» | Строки символов | "Иван Петров", "ivanov@mail.ru" |
| Ячейка «Да/Нет» | Только два значения | True / False |
| Ячейка «Список» | Упорядоченный набор | [1, 2, 3] или ["дом", "квартира"] |
| Ячейка «Справочник» | Пары ключ-значение | {"имя": "Иван", "роль": "Прораб"} |
| Ячейка «Пусто» | Отсутствие данных | None |

Если попытаться положить текст в ячейку «Целое число» — система выдаст ошибку. Это не баг, а защита от ошибки оператора.

## Тип 1: `int` — целое число

**Что это**: целое число без дробной части. Может быть положительным, отрицательным, нулём.

**В нашем проекте**: идентификаторы (ID), суммы в копейках, количество домов.

Из `backend/app/models/user.py`, строка 18:
```python
id: Mapped[int] = mapped_column(primary_key=True, autoincrement=True)
```

- `id` — идентификатор пользователя. Целое число: 1, 2, 3, ...
- `autoincrement=True` — база данных автоматически присваивает следующий номер при создании нового пользователя

**Почему суммы в копейках, а не в рублях?**

Из начальной миграции `backend/alembic/versions/2026_04_11_1911_f80b758cadef_initial_schema.py`:
```python
sa.Column('base_price_cents', sa.BigInteger(), nullable=False),
```

Суммы хранятся в **копейках** как целые числа (`BigInteger` — очень большое целое). Почему?

Числа с дробной частью (float) в компьютере хранятся неточно. Например:
```python
0.1 + 0.2  # = 0.30000000000000004 (не 0.3!)
```

Для финансов это катастрофа: при 1000 транзакциях накапливается погрешность. Поэтому всё в копейках (целые числа), а отображение «в рублях» — задача интерфейса.

1 500 000 рублей → 150 000 000 копеек → хранится как `150000000` (целое число, без ошибок).

## Тип 2: `str` — строка (текст)

**Что это**: последовательность символов. Буквы, цифры, знаки препинания — всё, что заключено в кавычки.

**В нашем проекте**: email, имена, роли пользователей, коды объектов.

Из `backend/app/models/user.py`, строки 19-21:
```python
email: Mapped[str] = mapped_column(String(255), unique=True, nullable=False, index=True)
password_hash: Mapped[str] = mapped_column(String(255), nullable=False)
full_name: Mapped[str] = mapped_column(String(255), nullable=False)
```

- `String(255)` — строка максимум 255 символов
- `unique=True` у email — нет двух пользователей с одинаковым email
- `index=True` у email — база данных создаёт «индекс» для быстрого поиска по email (как алфавитный указатель в книге)

**Важно**: `password_hash` — это не сам пароль, а его хеш (необратимое преобразование). Даже если базу данных взломают — пароли не украдут. Об этом подробно в уроке 12.

---

## Тип 3: `bool` — логическое значение (да/нет)

**Что это**: только два возможных значения: `True` (истина) или `False` (ложь). По-русски: «да» или «нет».

**В нашем проекте**: признаки состояния.

Из `backend/app/models/user.py`, строка 26:
```python
is_active: Mapped[bool] = mapped_column(Boolean, nullable=False, default=True)
```

- `is_active` — активен ли пользователь: `True` (да) или `False` (нет)
- `default=True` — при создании нового пользователя он сразу активен, если явно не указано иное

Когда пользователя «удаляют» (например, уволился прораб) — не удаляем запись из БД, а ставим `is_active = False`. Это называется «мягкое удаление» — данные сохраняются для аудита, доступ заблокирован.

## Тип 4: `list` — список

**Что это**: упорядоченный набор значений. Как нумерованный список в документе — у каждого элемента есть позиция (индекс), начиная с 0.

**В нашем проекте**: списки условий фильтрации, списки результатов из базы данных.

Из `backend/app/services/budget_plan.py`, строки 95-104:
```python
extra_conditions = []  # пустой список

if project_id is not None:
    extra_conditions.append(BudgetPlan.project_id == project_id)
if category_id is not None:
    extra_conditions.append(BudgetPlan.category_id == category_id)
```

Здесь строится список условий фильтрации. Начинаем с пустого списка `[]`, добавляем условие `append()` если параметр передан. В итоге `extra_conditions` — список из 0, 1, 2 или более условий, которые потом передадим в запрос к базе данных.

Работа со списком:
```python
stages = ["Фундамент", "Коробка", "Кровля", "Отделка"]
stages[0]   # "Фундамент" — первый элемент (индекс 0)
stages[2]   # "Кровля" — третий элемент (индекс 2)
len(stages) # 4 — количество элементов
```

## Тип 5: `dict` — словарь (справочник)

**Что это**: набор пар «ключ: значение». Как телефонный справочник — по имени (ключу) найти номер (значение). Порядок не важен, главное — уникальность ключей.

**В нашем проекте**: данные об изменениях в audit_log, параметры конфигурации, JSON-поля.

Из `backend/app/services/audit.py`, строка 78-96:
```python
changes={"after": _plan_to_dict(plan)}
```

А при обновлении:
```python
changes={"before": before, "after": after, "diff": diff}
```

Здесь `changes` — это словарь с тремя ключами:
- `"before"` → снимок данных до изменения
- `"after"` → снимок данных после изменения
- `"diff"` → список полей, которые изменились

Работа со словарём:
```python
user = {
    "email": "ivan@mail.ru",
    "full_name": "Иван Петров",
    "role": "OWNER",
    "is_active": True
}

user["email"]     # "ivan@mail.ru"
user["role"]      # "OWNER"
"email" in user   # True — ключ "email" существует
```

Из нашего кода (`backend/app/api/health.py`):
```python
return {"status": "ok", "version": __version__}
```

Это словарь. API возвращает его как JSON — стандартный формат обмена данными в вебе.

---

## Тип 6: `None` — отсутствие значения

**Что это**: специальное значение «ничего нет», «данных нет», «поле не заполнено». Не ноль, не пустая строка — именно «отсутствие».

**В нашем проекте**: необязательные поля, которые могут быть не заполнены.

Из `backend/app/models/user.py`, строки 27-29:
```python
last_login_at: Mapped[datetime | None] = mapped_column(
    DateTime(timezone=True), nullable=True, default=None
)
```

- `last_login_at` — дата последнего входа. Для нового пользователя, который ещё не входил — `None` (он никогда не входил)
- `nullable=True` — разрешено хранить `None` в базе данных (в SQL это называется `NULL`)
- `datetime | None` — тип: либо дата-время, либо None

Разница между None, нулём и пустой строкой:
```python
None   # нет данных вообще
0      # есть данные, и это ноль
""     # есть данные, и это пустая строка
```

Для `last_login_at`:
- `None` — пользователь никогда не входил (не то же самое, что «вошёл в нулевой момент времени»)

## Бонус: `frozenset` — неизменяемое множество

В нашем коде встречается ещё один тип — `frozenset`. Это как список, но без порядка и без дубликатов, и нельзя изменить после создания.

Из `backend/app/services/audit.py`, строки 20-22:
```python
AUDIT_EXCLUDED_FIELDS: frozenset[str] = frozenset(
    {"password_hash", "password", "token", "secret", "key", "jwt"}
)
```

Проверка «есть ли элемент в множестве» работает быстрее, чем в списке, — поэтому для запрещённых полей используется `frozenset`, а не `list`.

## Как типы связаны с базой данных

Python-тип и тип в PostgreSQL — это разные вещи, но они соответствуют друг другу:

| Python-тип | PostgreSQL-тип | Пример использования |
|---|---|---|
| `int` | `INTEGER` | ID, количество |
| `int` | `BIGINT` | Суммы в копейках (большие числа) |
| `str` | `VARCHAR(255)` | Email, имя, код |
| `str` | `TEXT` | Длинные примечания |
| `bool` | `BOOLEAN` | is_active |
| `datetime` | `TIMESTAMPTZ` | Даты и время с часовым поясом |
| `dict` | `JSONB` | Произвольные структуры (audit changes) |
| `None` | `NULL` | Поле не заполнено |

SQLAlchemy (ORM) берёт на себя перевод между Python-типами и SQL-типами. Вы описываете поле в Python — SQLAlchemy знает, как создать соответствующую колонку в базе.

## Риски и типичные ошибки

**1. Хранить деньги как float.**
`float` даёт погрешность при арифметике. Для финансов — всегда целое число (копейки, центы). В нашем проекте все суммы — `amount_cents`, `price_cents`, `cost_cents` — именно поэтому.

**2. Путать None и False.**
`is_active = None` — «неизвестно, активен ли». `is_active = False` — «точно неактивен». Это разные ситуации. Фильтр `WHERE is_active = False` не поймает записи с `is_active IS NULL`.

**3. Использовать список вместо словаря для структурированных данных.**
Список `["Иван", "ivanov@mail.ru", "OWNER"]` — непонятно, что есть что по позиции. Словарь `{"full_name": "Иван", "email": "ivanov@mail.ru", "role": "OWNER"}` — самодокументируем, порядок не важен.

## Проверка понимания

**Вопрос 1.** В базе данных хранится `amount_cents = 150000000`. Какую сумму это означает в рублях и почему используется такой формат?

> 150 000 000 копеек = 1 500 000 рублей. Целые числа используются вместо дробных для точности: компьютер не умеет хранить дробные числа абсолютно точно, и при финансовых расчётах накапливается погрешность. В копейках — всегда точно.

**Вопрос 2.** Пользователь только что зарегистрирован в системе. Каким будет значение поля `last_login_at`?

> `None` — пользователь ещё не входил. Это не ноль и не пустая строка, а специальное значение «данных нет». В базе данных хранится как `NULL`.

**Вопрос 3.** Почему словарь `{"before": {...}, "after": {...}, "diff": [...]}` удобнее, чем список `[{...}, {...}, [...]]` для хранения изменений в audit_log?

> В словаре каждое значение имеет имя (`"before"`, `"after"`, `"diff"`). Можно обратиться по имени: `changes["before"]`. В списке нужно помнить порядок: `changes[0]` — это before или after? При чтении кода через год коллега не поймёт без комментария.
