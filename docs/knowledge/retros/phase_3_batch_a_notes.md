# Ретро-заметки — Фаза 3, Батч A

**Дата закрытия:** 2026-04-15
**Длительность:** 1 рабочий день (план — 3-4 дня)
**Координатор:** Claude Opus 4.6 (1M)
**Владелец:** Мартин

## Итог

Реализован полный CRUD для 8 сущностей (Project, Stage, HouseType, OptionCatalog, HouseTypeOptionCompat, House, HouseConfiguration, HouseStageHistory) + специальные действия (`/bulk`, `/stage`, `PUT options`, `/configurations`). Тестов 211/211 зелёных. Базовый паттерн заморожен.

## Что сработало

1. **Эталон + тиражирование**. Сначала вложили глубину в Project (2 раунда ревью), затем 7 сущностей по замороженному паттерну прошли быстрее. Эталонный подход экономит время на тиражируемых задачах.
2. **CLAUDE.md как живой документ** (новое в этом батче). Каждая ошибка ревью добавляла строку правила. К концу батча — 13 правил, которые ловят повторяющиеся дефекты на старте следующих задач.
3. **Action-endpoints вместо generic REST**. `PATCH /houses/{id}/stage`, `PUT /house-types/{id}/options` — отдельные смысловые операции читаются понятнее, чем generic update со скрытой бизнес-логикой.
4. **extra_conditions как механизм фильтрации**. Сделано в Round 2 эталона — переиспользуется во всех 7 последующих сущностях. Total в пагинации корректен на любом фильтре.
5. **Аудит в одной транзакции с записью**. ADR 0007 + явный `audit_service.log()` в сервисе — нет ни одной write-операции без аудита (qa проверил).

## Что не сработало

1. **Реc-курсив одних и тех же ошибок**. Литералы паролей в тестах ловились трижды (Phase 2 R2, Phase 3 step 2 R1, Phase 3 step 4 R1) — несмотря на v1.3 §3 и CLAUDE.md. Причина: backend-dev не читал CLAUDE.md перед тестами. Мера: с Батча B — обязательное «прочти CLAUDE.md» в каждом промпте, проверка в самопроверке.
2. **`git add -A` Координатора подобрал чужую работу**. Коммит регламента v1.4 случайно прихватил qa-файлы, которые ещё не прошли ревью. Пришлось делать post-hoc ревью. Правило добавлено в CLAUDE.md.
3. **Postобработка фильтров в Python**. Эталон Project в Round 1 имел фильтр `is_archived` через Python list-comprehension после LIMIT — total в пагинации ломался. Поймано Round 2. Урок: фильтры всегда в SQL, всегда через extra_conditions. Правило в CLAUDE.md.
4. **IDOR во вложенных ресурсах**. Эндпоинты `/houses/{house_id}/configurations/{cfg_id}` не проверяли принадлежность cfg_id к house_id — можно было бы менять чужую конфигурацию. Поймано в step 4 Round 1. Правило в CLAUDE.md.

## Что попробовать

1. **Параллелизация subagents в Батче B** — пилот регламента v1.4. Backend-direktор + backend-head + 2-3 параллельных backend-dev. Ожидаемое ускорение ~50%.
2. **Pre-commit хуки** (devops-задача): gitleaks на секреты, кастомный grep на «фильтр после LIMIT», ast-проверка «write-эндпоинт без audit_service.log». Это снимет с reviewer механические замечания.
3. **Скелет-генератор** для CRUD (если в Батче B/C снова появится массовое тиражирование — пока скоуп B/C меньше, может не понадобиться).

## Что удивило

1. **Скорость v1.4-обсуждения**. Вся новая иерархия (49 ролей, 6 направлений, правила коммуникации, модели) — согласована за 30 минут переписки. До регламентов это занимало бы дни.
2. **Принцип Бориса Черни в действии**. CLAUDE.md из 70 строк — реально поменял качество промптов (внутренних), бэкенд начал писать по правилам без напоминаний.
3. **Post-hoc ревью qa оказалось OK**. Несмотря на процессный сбой Координатора — qa выполнил работу качественно (212 тестов, 0 багов, 3 nit). Это говорит, что субагенты надёжны даже когда Координатор ошибается.
