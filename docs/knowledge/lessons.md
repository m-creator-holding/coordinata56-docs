# Извлечённые уроки

## Phase 0 — Инфраструктура (2026-04-11)

### L0.1 — Docker Hub rate limit для неавторизованных
**Что произошло.** При первом `docker compose pull` Docker Hub отказал — бесплатный лимит на pull образов для неавторизованных IP-адресов превышен.

**Как проявилось.** `Error: You have reached your unauthenticated pull rate limit`.

**Решение.** Сменили источник на `mirror.gcr.io/library/postgres:16-alpine` — зеркало Google, которое не требует аутентификации и не лимитирует публичные образы.

**Урок на будущее.** Для production — обязательно завести Docker Hub Pro аккаунт или настроить приватный registry. Для всех публичных образов в dev-окружении — по умолчанию использовать `mirror.gcr.io/library/<image>`.

### L0.2 — Host-порт 5432 занят системным postgres
**Что произошло.** Установили postgresql-16 через `apt install` на хост (на всякий случай), а потом запустили postgres в Docker. Контейнер не стартовал — порт 5432 занят.

**Как проявилось.** `Error: failed to bind host port 127.0.0.1:5432/tcp: address already in use`.

**Решение.** Перенесли внешний маппинг Docker на `127.0.0.1:5433:5432`. Внутри Docker-сети postgres всё равно доступен на 5432, поэтому backend-контейнер ничего не заметил.

**Урок на будущее.** Не ставить сервисы на хост, если они будут работать в Docker. Либо отключать host-сервис, либо использовать альтернативные порты. На следующем сервере — ставить ТОЛЬКО в контейнер.

### L0.3 — Permissions cache в сессии Claude Code
**Что произошло.** Изменения в `~/.claude/settings.local.json` (defaultMode: bypassPermissions) не применились в текущей сессии — продолжали приходить подтверждения в Telegram.

**Как проявилось.** Мартин видел «Permission: Bash / Allowed» запросы даже после записи настроек.

**Решение.** Понял, что Claude Code кэширует разрешения при старте сессии. Полное применение требует перезапуска `claude --channels plugin:telegram@claude-plugins-official`. В текущей сессии пришлось продолжать с подтверждениями.

**Урок на будущее.** Подобные настройки менять ДО старта работ, а не посреди. Или перезапускать сессию. Также — subagent-файлы в `~/.claude/agents/` подхватываются только при старте сессии, не применимы мгновенно.

## Phase 1 — Модель данных (2026-04-11)

### L1.1 — passlib + bcrypt 4.x несовместимость
**Что произошло.** Использовал `passlib[bcrypt]` для хеширования паролей в seeds. При первом запуске `bcrypt.hashpw` упал с `ValueError: password cannot be longer than 72 bytes`. Причина — passlib при инициализации пытается захешировать тестовую строку длиннее 72 байт для определения старого бага, но в bcrypt 4.x эта попытка кидает ValueError вместо обрезания.

**Как проявилось.** Exception прямо в `_upsert_owner(db)`, не при первом импорте.

**Решение.** Убрал passlib из зависимостей, использую библиотеку `bcrypt` напрямую. Функция `_hash_password` — 2 строки кода, без лишнего слоя абстракции.

**Урок на будущее.** passlib заброшен (последний релиз 2020). Для новых проектов — сразу `bcrypt` или `argon2` напрямую. passlib только в legacy-миграциях.

### L1.2 — Integer vs BigInteger для PK
**Что произошло.** В ADR 0001 заявлены bigint PK, а по факту SQLAlchemy `Mapped[int]` с `mapped_column(primary_key=True)` даёт обычный `INTEGER`. Увидел это, когда проверил созданную таблицу через `\d houses`.

**Оценка.** Integer = до 2.147 млрд значений. Для наших 85 домов + платежей + материалов за 5-10 лет — с огромным запасом хватит. BigInteger избыточен.

**Решение.** Оставить Integer. ADR поправить — не обманывать будущего себя заявлениями, которые не реализованы. Денежные поля остаются BigInteger (копейки), как и было запланировано.

**Урок на будущее.** Проверять генерированную миграцию против ADR до применения. Честный минимализм лучше незаметного расхождения между документацией и кодом.

### L1.3 — Отсутствие глобального фильтра soft delete
**Что произошло.** Реализовал `deleted_at` через `SoftDeleteMixin`, но не добавил глобального фильтра «WHERE deleted_at IS NULL» на уровне ORM. Это значит, что каждый запрос к `User.query` или `Contractor.query` должен явно это учитывать, иначе в выборку попадут удалённые.

**Оценка.** Это известный риск в пути. В v1 мягкого удаления у нас ещё нет — это появится в Phase 3+ когда будут реальные CRUD-операции. До этого момента — терпимо.

**Решение.** Отложил. В Phase 3 добавить `with_loader_criteria` на уровне `Query` или обернуть всё в Repository-паттерн, который фильтрует автоматически.

**Урок на будущее.** Добавить TODO в model base для будущей имплементации. Перед первым реальным soft-delete-use-case — обязательно проверить, что выборки корректно фильтруют.

### L1.4 — Alembic ruff post-write hook ломается в контейнере
**Что произошло.** По умолчанию alembic.ini содержал хук `ruff` через `console_scripts`, но внутри контейнера entrypoint не находился: `Could not find entrypoint console_scripts.ruff`. Миграция генерировалась корректно, но hook писал ошибку в конец.

**Решение.** Убрал блок `[post_write_hooks]` из alembic.ini. Ruff запускается вручную на генерированной миграции, когда нужно.

**Урок на будущее.** Post-write hooks в Alembic — полезная идея, но требует проверки работоспособности внутри того же окружения, где запускается Alembic. Либо настраивать через `exec` подход, либо отключить.
