## Обзор

В проекте добавлены два инструмента автоматической проверки документации.
Оба запускаются как в CI (GitHub Actions), так и локально через `make`.

| Инструмент | Что проверяет | Режим CI |
|---|---|---|
| markdownlint-cli2 | Структура и стиль markdown | errors — fail |
| lychee | Битые ссылки в docs/ | внутренние — fail, внешние — warning |

## markdownlint

### Что проверяет

Конфиг: `/root/coordinata56/.markdownlint.yaml`

Включены только структурно важные правила (Trial, minimal ruleset):

| Правило | Описание |
|---|---|
| MD001 | Заголовки идут по порядку (h1 → h2 → h3, без прыжков) |
| MD003 | Единый стиль заголовков — ATX (`#`, `##`, `###`) |
| MD022 | Пустые строки вокруг заголовков |
| MD024 | Нет дублирующихся заголовков одного уровня |
| MD025 | Не более одного H1 на файл |
| MD031 | Пустые строки вокруг блоков кода |
| MD040 | Блоки кода должны иметь указание языка |
| MD041 | Первая строка файла — H1 |

Намеренно отключены: MD013 (длина строки — нереалистично для русского текста),
MD032 (blank вокруг списков — слишком много ложных срабатываний),
MD036 (жирный текст как заголовок — паттерн `**Поле**: значение` в ADR — норма),
MD060 (стиль таблиц — смешанный стиль в docs — норма до рефакторинга).

### Запуск локально

```bash
# Требует Node.js (уже есть в проекте)
# Установка (один раз):
npm install -g markdownlint-cli2

# Проверка всей документации:
make lint-docs
# или напрямую:
npx markdownlint-cli2 "docs/**/*.md"

# Проверка одного файла:
npx markdownlint-cli2 docs/adr/0001-data-model-v1.md
```

### Как добавить исключение для конкретного файла

Добавить в начало файла (после frontmatter) строку:

```markdown
<!-- markdownlint-disable MD041 -->
```

Или для одной строки:

```markdown
<!-- markdownlint-disable-next-line MD022 -->
```

### Как добавить правило в конфиг

Отредактировать `/root/coordinata56/.markdownlint.yaml`, добавить правило по образцу.
Полный список правил: https://github.com/DavidAnson/markdownlint/blob/main/doc/Rules.md

## lychee

### Что проверяет

Конфиг: `/root/coordinata56/lychee.toml`
Ignore-list: `/root/coordinata56/.lycheeignore`

Проверяет ссылки в файлах `docs/**/*.md`:
- В CI: только внутренние файловые ссылки (`--offline`). Битая ссылка = fail.
- Внешние HTTP-ссылки: отдельный job с `continue-on-error: true` — результат виден, но не блокирует PR.

### Запуск локально

```bash
# Установка lychee (один раз):
# macOS:   brew install lychee
# Linux:   cargo install lychee  ИЛИ скачать бинарник с GitHub Releases
# https://github.com/lycheeverse/lychee/releases

# Только внутренние ссылки (быстро, без интернета):
make check-links
# или напрямую:
lychee --offline --config lychee.toml docs/

# Внутренние + внешние ссылки (медленнее, нужен интернет):
lychee --config lychee.toml docs/
```

### Как добавить исключение для ссылки

Добавить паттерн в `/root/coordinata56/.lycheeignore` — одна строка, glob или точная ссылка.

Пример:
```
https://внутренний-сервер.local
http://test.example.com
```

## GitHub Actions workflow

Файл: `.github/workflows/docs-validation.yml`

Три jobs:
1. `markdownlint` — запускается при изменениях в `docs/`. Errors → fail.
2. `lychee` (internal) — проверяет только файловые ссылки (`--offline`). Fail при битой ссылке.
3. `lychee-external` — проверяет HTTP-ссылки. `continue-on-error: true` — всегда warning, никогда не блокирует PR.

Workflow запускается только при изменениях в `docs/**` или конфигах валидации —
не мешает CI бэкенда при правке кода.

## Предсуществующие нарушения (baseline 2026-04-17)

При первом прогоне на 295 файлах с включённым ruleset выявлено **1179 ошибок**.

Топ-5 по количеству (все предсуществующие, не блокируют работу):

| # | Правило | Количество | Описание |
|---|---|---|---|
| 1 | MD022 | 557 | Нет пустой строки вокруг заголовка |
| 2 | MD031 | 335 | Нет пустой строки вокруг блока кода |
| 3 | MD041 | 153 | Файл не начинается с H1 |
| 4 | MD040 | 126 | Блок кода без указания языка |
| 5 | MD025 | 11 | Более одного H1 в файле |

Эти нарушения — следствие органического роста документации без линтера.
Они фиксируются как известный baseline и не должны блокировать текущую работу.

**Стратегия устранения**: исправлять инкрементально при редактировании файлов,
не делать отдельного «mega-fix» PR (риск конфликтов слияния).

Lychee (внутренние ссылки, offline): проверка требует установки бинарника lychee.
В CI будет работать через `lycheeverse/lychee-action@v2`.
