# Notification ПДн Review

**Дата:** 2026-04-19  
**Статус:** Legal analysis — не юридическое заключение  
**Автор:** Legal adviser (субагент)

---

## Предмет

Модель `Notification`: поля `title` (String 255), `body` (Text), `payload` (JSONB nullable).  
Контекст: BPM-движок генерирует уведомления вида «Иван Иванов одобрил оплату 50 000 ₽».

---

## Правовой анализ: title / body как ПДн

### Квалификация

`title` и `body` могут содержать ФИО (ст. 3 п. 1 152-ФЗ — прямые ПДн). Сочетание ФИО + финансовое действие + сумма — ПДн с расширенным составом.

**Ключевой вопрос: является ли уведомление в системе «обработкой» ПДн?**

Ответ: **да**. Запись в БД — это хранение, т.е. обработка по ст. 3 п. 3 152-ФЗ.

### Правовое основание обработки

Обработка ФИО сотрудника в BPM-уведомлении:
- **Основание:** ст. 6 ч. 1 п. 5 152-ФЗ — обработка необходима для исполнения договора, стороной которого является субъект (трудовой договор), ИЛИ
- **Основание:** ст. 6 ч. 1 п. 1 — обработка по согласию субъекта

**Вывод:** При условии трудовых отношений — основание есть. При обработке данных внешних контрагентов-физлиц (не стороны трудового договора) — нужен анализ договорных отношений.

### Маскирование в title/body: нужно или нет?

**Позиция:** для получателя уведомления — маскирование НЕ требуется при следующих условиях:
1. Получатель (receiver) — сотрудник, имеющий правомерный доступ к этой информации по RBAC
2. Уведомление отображается только своему receiver (FK user_id — соблюдается в модели)
3. Передача через защищённый канал (HTTPS / Telegram Bot с авторизацией)

**Вывод:** plaintext в title/body для receiver допустим. Это не нарушение — это передача ПДн лицу, которому она адресована, при наличии правового основания.

## Критический риск: AuditLog + Notification

### Проблема

`audit_log.changes_json` фиксирует изменения записей. При изменении Notification (например, status: pending → read) ORM-хук может записать в `changes_json` всё содержимое объекта, включая:
- `title` с ФИО
- `body` с суммами
- `payload` с BPM-контекстом

**Это означает:** ПДн из Notification попадает в AuditLog и хранится там по audit retention (3 года) — возможно дольше, чем retention самого уведомления (1 год).

### Риск по 152-ФЗ

Ст. 5 ч. 4: хранение не дольше, чем требуют цели. Если уведомление удалено через 1 год, а его ПДн-содержимое живёт в audit_log 3 года — цель хранения в audit_log должна быть документально обоснована.

Ст. 18.1: оператор обязан задокументировать все цели обработки. «Аудит» — законная цель, но требует фиксации в ЛНА.

### Рекомендации backend-dev

**Обязательно реализовать** в audit_service при логировании Notification:

```python
# В audit_service.log() — при entity_type == "Notification"
# Маскировать ПДн-поля перед записью в changes_json

NOTIFICATION_PD_FIELDS = {"title", "body", "payload"}

def _mask_notification_changes(changes: dict) -> dict:
    """Маскирование ПДн в audit_log для Notification.
    
    title и body — заменять на хеш или '[CONTENT MASKED]'.
    payload — если содержит user-идентифицирующие данные — маскировать.
    
    Обоснование: 152-ФЗ ст. 5 ч. 4, ПП РФ 1119 п. 15.
    """
    masked = {}
    for key, value in changes.items():
        if key in NOTIFICATION_PD_FIELDS:
            masked[key] = "[PD MASKED]"
        else:
            masked[key] = value
    return masked
```

**Альтернатива (предпочтительная):** в audit_log для Notification фиксировать только изменения статуса, не содержимого:

```python
# Вместо полного diff — только:
# {"status": {"old": "pending", "new": "read"}, "read_at": "..."}
# title/body/payload — не логировать совсем.
```

## Notification.payload (JSONB)

### Текущее состояние

`payload` — «произвольный JSON-контекст (BPM step, link, params)». Без схемы.

### Риск

BPM может поместить в payload:
- `{"approver_name": "Иванов И.И.", "amount": 50000}` — ПДн + финансы
- `{"user_email": "ivanov@company.ru"}` — ПДн

### Рекомендация

До BPM-интеграции (US-11+) зафиксировать допустимый состав payload в ADR или task-описании. Запрет на передачу в payload полного ФИО и email субъекта — вместо них использовать user_id (обезличенный идентификатор).

```python
# Рекомендуемая структура Notification.payload:
{
    "bpm_step": "payment_approval",
    "bpm_instance_id": 123,
    "amount": 50000,
    "approver_user_id": 42,       # ID, не ФИО
    "action_url": "/payments/789"
}
# НЕ: {"approver_name": "Иванов И.И.", "approver_email": "..."}
```

## Итоговая матрица рисков

| Поле | Риск | Severity | Рекомендация |
|---|---|---|---|
| title / body → receiver | Низкий при RBAC | Low | Plaintext допустим. Контроль RBAC обязателен |
| title / body → AuditLog | ВЫСОКИЙ — retention mismatch | High | Маскировать в audit_service до записи |
| payload с ФИО | Средний | Medium | Запретить ФИО в payload; использовать user_id |
| payload → AuditLog | Высокий при наличии ФИО | High | Masked payload policy |
