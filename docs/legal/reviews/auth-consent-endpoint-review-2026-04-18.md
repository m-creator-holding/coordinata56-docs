## Краткое резюме

API в текущем дизайне содержит три критических несоответствия 152-ФЗ ст.9 и одно недостаточно продуманное место по ст.14. Все они устранимы на уровне дизайна схемы БД и контрактов API до написания основного кода.

## Анализ по шести вопросам

### 1. Запись о согласии — достаточно ли version + timestamp?

**Вывод: недостаточно. Риск высокий.**

152-ФЗ ст.9 ч.4 устанавливает исчерпывающий перечень реквизитов, которые должно содержать письменное согласие (или электронный документ, равнозначный ему):

- ФИО, адрес субъекта, реквизиты документа, удостоверяющего личность
- ФИО/наименование и адрес оператора
- Цель обработки
- Перечень персональных данных
- Перечень действий с ПДн и описание способов обработки
- Срок действия согласия и способ его отзыва
- Подпись субъекта

С 01.09.2025 (изм. к 152-ФЗ) добавлены дополнительные обязательные элементы: разъяснение о добровольности предоставления данных и последствиях отказа.

**Что нужно сохранить в БД:**

Текущая модель (`pd_consent_at`, `pd_consent_version`) фиксирует лишь факт согласия и версию политики. Закон требует доказательства того, *с чем именно* согласился субъект и *в какой форме*. Хранить только `version` — значит не иметь доказательной базы при проверке РКН или судебном споре.

**Рекомендуемое решение:** отдельная таблица `pd_consents` вместо двух полей в `users`:

```sql
CREATE TABLE pd_consents (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(id),
    version         VARCHAR(16) NOT NULL,
    accepted_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    revoked_at      TIMESTAMPTZ NULL,
    ip_address      INET NULL,              -- подтверждение канала получения
    user_agent      TEXT NULL,              -- подтверждение канала получения
    policy_text_hash VARCHAR(64) NOT NULL,  -- SHA-256 текста политики на момент согласия
    -- Поля ниже заполняются из профиля пользователя на момент согласия:
    subject_name    TEXT NULL,              -- ФИО субъекта на момент согласия
    operator_name   TEXT NOT NULL DEFAULT 'ООО «[Оператор]»',
    purpose         TEXT NOT NULL,
    data_categories TEXT[] NOT NULL,
    processing_actions TEXT NOT NULL,
    validity_period  TEXT NOT NULL,
    revocation_method TEXT NOT NULL
);
```

`policy_text_hash` — критически важен: позволяет доказать, что субъект видел именно этот текст, а не другой. Без него при смене политики доказательная база рушится.

В таблице `users` поля `pd_consent_at` и `pd_consent_version` можно оставить как денормализованный кэш для быстрых проверок, но источником правды является `pd_consents`.

### 2. Текст политики — immutable или файл в репо?

**Вывод: текст каждой версии политики должен быть immutable и храниться в БД. Хранение только в файле репо — недостаточно.**

**Обоснование:**

Закон не содержит прямого требования о способе хранения текста политики, однако ст.9 ч.4 требует доказательства того, что субъект дал «конкретное, предметное, информированное, осознанное и однозначное» согласие. Если текст политики — файл в git-репо, то:

- Файл может быть перезаписан без изменения номера версии
- При проверке РКН нельзя доказать, что субъект видел именно этот текст
- Если политика изменится в файле — все предыдущие согласия технически становятся несогласиями с новым текстом, а пересогласование не было запрошено

**Рекомендуемое решение:** таблица `pd_policy_versions`:

```sql
CREATE TABLE pd_policy_versions (
    version     VARCHAR(16) PRIMARY KEY,
    text        TEXT NOT NULL,
    text_hash   VARCHAR(64) NOT NULL GENERATED ALWAYS AS (encode(sha256(text::bytea), 'hex')) STORED,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    created_by  UUID REFERENCES users(id),
    is_active   BOOLEAN NOT NULL DEFAULT false
);
```

Запись добавляется только INSERT — никакого UPDATE текста существующей версии. Это и есть immutability. Файл в репо можно держать как исходник для деплоя, но при деплое он копируется в БД и больше не изменяется.

Endpoint `GET /api/v1/auth/consent-policy/v1.0` должен читать из этой таблицы, а не из файловой системы.

### 3. Отзыв согласия — нужен endpoint revoke-consent?

**Вывод: endpoint обязателен. Это прямое требование ст.9 ч.5 152-ФЗ.**

Ст.9 ч.5: субъект вправе отозвать согласие. Оператор обязан прекратить обработку ПДн в течение **30 дней** с момента получения отзыва (ст.21 ч.3). Если у субъекта нет механизма отзыва внутри системы — он всё равно может направить заявление по почте или лично, и оператор обязан его исполнить. Но отсутствие API-механизма означает: субъект не может реализовать своё право в системе, что фиксируется как нарушение при проверке РКН.

**Что должен делать endpoint после revoke:**

Здесь важно разграничить два случая:

а) Если единственное основание для обработки ПДн пользователя — его согласие по ст.9, то после отзыва оператор **обязан прекратить обработку и уничтожить данные** в 30-дневный срок. Аккаунт фактически удаляется.

б) Если есть иные основания обработки (ст.6 ч.1 пп.2-11) — например, заключённый договор с покупателем — оператор вправе продолжить обработку данных в объёме, необходимом для исполнения договора. В этом случае согласие отзывается, но полного удаления не происходит.

**Для системы coordinata56 (девелопер, покупатели ДДУ):** скорее всего применим случай (б) — у покупателя есть договор, поэтому обработка продолжается в части договорных данных. Но это нужно явно прописать в политике и ЛНА.

**Минимальный дизайн endpoint:**

```
POST /api/v1/auth/revoke-consent
Body: {"reason": "string" | null}
Response: {
    "revoked_at": "timestamp",
    "processing_continues_until": "date" | null,  -- если есть иные основания
    "data_destruction_deadline": "date"             -- 30 дней с revoked_at
}
```

После revoke: пометить запись в `pd_consents.revoked_at`, создать задачу в очереди на уничтожение данных через 30 дней, залогировать в audit_log, уведомить субъекта по email.

Немедленный logout при revoke — технически оправдан, но не обязателен законом. Если доступ к системе основан исключительно на согласии — выход логичен. Если есть договорное основание — блокировку определяет бизнес-логика, не закон напрямую.

### 4. Несовершеннолетние пользователи

**Вывод: нужна отдельная логика. Риск средний, но при отсутствии — нарушение ст.9 ч.1 152-ФЗ.**

По 152-ФЗ и ГК РФ:

- До 14 лет: согласие даёт только законный представитель (родитель, опекун). Самостоятельное согласие ребёнка юридически ничтожно.
- 14-18 лет: по разъяснениям РКН (письмо от 30.03.2023) — подросток может давать согласие самостоятельно в части правоотношений, в которых он участвует непосредственно. Однако для финансовых, договорных и иных значимых правоотношений рекомендуется согласие законного представителя.
- С 18 лет / после эмансипации / после вступления в брак: полная самостоятельность.

**Для контекста coordinata56 (покупатели недвижимости, сотрудники):** несовершеннолетний покупатель ДДУ — редкий, но возможный случай. Сотрудник до 18 лет — возможен (ТК допускает с 14-16 лет с ограничениями).

**Минимальный дизайн:**

В форме регистрации / профиле — поле `date_of_birth`. При `accept-consent`:

```
если возраст < 14 → отказ с кодом MINOR_REQUIRES_GUARDIAN
если 14 ≤ возраст < 18 → принять согласие, но флаг is_minor=true + запросить/сохранить данные законного представителя
если возраст ≥ 18 → стандартный флоу
```

В таблице `pd_consents` добавить поля: `is_minor BOOLEAN`, `guardian_name TEXT NULL`, `guardian_contact TEXT NULL`.

### 5. Экспорт по ст.14 — что включать?

**Вывод: закон требует предоставить конкретный перечень сведений, не «все данные системы».**

Ст.14 ч.7 152-ФЗ — перечень сведений, которые оператор обязан предоставить по запросу субъекта:

1. Подтверждение факта обработки ПДн и её правовое основание
2. Цели и применяемые методы обработки
3. Наименование и местонахождение оператора; сведения о лицах, имеющих доступ или которым могут быть переданы данные
4. Обрабатываемые ПДн, источник их получения (если закон не запрещает раскрывать)
5. Сроки обработки, в том числе сроки хранения
6. Порядок реализации субъектом прав по 152-ФЗ
7. Информация об осуществлённой или предполагаемой трансграничной передаче
8. Наименование или ФИО и адрес лица, осуществляющего обработку по поручению оператора

Срок предоставления — **10 рабочих дней** с момента обращения (ст.14 ч.4).

**Что включать в экспорт (минимум для coordinata56):**

```json
{
  "subject": {
    "id": "...",
    "name": "...",
    "email": "...",
    "phone": "...",
    "created_at": "..."
  },
  "consent_history": [...],  -- из pd_consents
  "audit_log": [...],         -- действия пользователя в системе
  "processing_info": {
    "operator": "ООО «...»",
    "purposes": [...],
    "legal_basis": "...",
    "retention_period": "...",
    "third_parties": [...]
  }
}
```

Технические внутренние поля (хеши паролей, внутренние ID связей, системные флаги) — не включать. Данные о других субъектах — не включать (даже если они связаны, например, члены семьи в сделке).

**Endpoint:** `GET /api/v1/auth/data-export` — ответ асинхронный (задача в очереди) или синхронный JSON. При больших объёмах — ссылка на скачивание файла с TTL.

### 6. Удаление по ст.14 — soft или hard? Сроки хранения?

**Вывод: необходим комбинированный подход. Ни чистый soft, ни немедленный hard не соответствуют всей совокупности требований.**

**Что говорит закон:**

- Ст.21 ч.3 152-ФЗ: при отзыве согласия — уничтожить данные в **30 дней**
- Ст.21 ч.1: при незаконной обработке или недостижении цели — уничтожить в **10 дней**
- Ст.21 ч.4: уничтожение подтверждается актом или иным документом
- 402-ФЗ «О бухгалтерском учёте» ст.29: первичные учётные документы — **5 лет**
- НК РФ ст.23 п.8: документы, необходимые для налогового учёта — **4 года**
- 214-ФЗ «Об участии в долевом строительстве»: документы по сделке — срок не менее **5 лет** после исполнения договора

**Рекомендуемый подход — трёхуровневое удаление:**

Уровень 1 — «Soft delete» с немедленным эффектом: при запросе на удаление — поле `deleted_at TIMESTAMPTZ` в таблице `users`, блокировка входа, удаление из всех активных выборок. Субъект перестаёт получать сервис.

Уровень 2 — «Pseudonymisation» (обезличивание) в течение 30 дней: ПДн, обработка которых основана исключительно на согласии (email, телефон, имя в маркетинговых списках), — заменяются на хеши или токены. Оригиналы уничтожаются. Это «уничтожение» в смысле 152-ФЗ.

Уровень 3 — Хранение «скелета» для юридических нужд: записи о транзакциях, договорах, бухгалтерских проводках — хранятся в обезличенном виде до истечения сроков по 402-ФЗ / НК РФ / 214-ФЗ (5 лет). Связь с конкретным физлицом разорвана.

**Сводная таблица сроков хранения:**

| Тип данных | Основание | Срок |
|---|---|---|
| Согласие на ПДн (запись) | 152-ФЗ + ПП РФ 687 | 3 года после прекращения обработки |
| Данные по ДДУ | 214-ФЗ | 5 лет после исполнения договора |
| Бухгалтерские документы | 402-ФЗ | 5 лет |
| Налоговые документы | НК РФ ст.23 | 4 года |
| Audit log (действия в системе) | ПП РФ 1119 | минимум 1 год, рекомендуется 3 года |

В таблице `pd_consents` запись о согласии (включая revoked_at) — **не удалять никогда**. Это доказательство выполнения требований закона.

## Топ-3 исправления дизайна API

### Исправление 1. Таблица pd_consents вместо двух полей в users

**Проблема:** `pd_consent_at + pd_consent_version` в таблице `users` — это не доказательная база, а технический флаг. При проверке РКН оператор не сможет подтвердить, что субъект видел конкретный текст политики.

**Решение:** вынести в отдельную таблицу `pd_consents` со следующими обязательными полями: `user_id`, `version`, `accepted_at`, `revoked_at`, `ip_address`, `user_agent`, `policy_text_hash`. В `users` оставить `pd_consent_version` как денормализованный кэш для middleware-проверки при входе, но запись правды — только в `pd_consents`.

**Закон:** 152-ФЗ ст.9 ч.3 (бремя доказывания получения согласия лежит на операторе).

### Исправление 2. Таблица pd_policy_versions + policy_text_hash

**Проблема:** `policy_text_url: "/api/v1/auth/consent-policy/v1.0"` — если URL возвращает текст из файла в репо, то текст можно изменить без создания новой версии. Субъект давал согласие с одним текстом, а хранится другой.

**Решение:** таблица `pd_policy_versions` с immutable-записями (только INSERT). Endpoint `GET /consent-policy/{version}` читает из БД. При записи согласия в `pd_consents` сохранять `policy_text_hash = SHA-256(policy_text)`.

**Закон:** 152-ФЗ ст.9 ч.4 — согласие должно быть «конкретным, предметным, информированным».

### Исправление 3. Endpoint revoke-consent + очередь на уничтожение данных

**Проблема:** в текущем дизайне API нет механизма отзыва. Субъект не может реализовать право из ст.9 ч.5 внутри системы.

**Решение:** `POST /api/v1/auth/revoke-consent`. После вызова: (а) пометить `pd_consents.revoked_at`, (б) поставить задачу в очередь на уничтожение ПДн через 30 дней (ст.21 ч.3), (в) записать в `audit_log`, (г) отправить уведомление субъекту. Ответ включать `data_destruction_deadline` — дату, до которой данные будут уничтожены.

**Закон:** 152-ФЗ ст.9 ч.5, ст.21 ч.3.

## 3 открытых вопроса для живого юриста на production-gate

**Вопрос 1. Форма согласия: достаточно ли «любой формы» или нужна квалифицированная электронная подпись?**

152-ФЗ ст.9 ч.1 допускает «любую форму, позволяющую подтвердить факт получения согласия». Однако для ряда категорий данных (специальные — ст.10, биометрические — ст.11) требуется письменная форма, равнозначная которой является только ЭП по 63-ФЗ. Вопрос: какие именно категории данных будет обрабатывать coordinata56 в разрезе покупателей ДДУ и сотрудников? Если среди них — паспортные данные, СНИЛС, сведения о здоровье (например, инвалидность при оформлении льгот) — нужна квалифицированная ЭП, а не просто «нажал кнопку».

**Вопрос 2. Основания обработки ПДн без согласия по ст.6 ч.1 пп.2-5: какой перечень применим к координата56?**

При отзыве согласия оператор вправе продолжить обработку, если есть иные основания (договор, законная обязанность, жизненно важные интересы и т.д.). Для покупателей ДДУ — явно применимо основание «исполнение договора» (пп.5). Для сотрудников — ТК РФ. Но конкретный объём данных, который можно хранить после отзыва согласия по каждому основанию — должен определить живой юрист с учётом всех бизнес-процессов холдинга. Без этого `revoke-consent` нельзя корректно реализовать.

**Вопрос 3. Уведомление РКН об обработке ПДн: статус регистрации оператора?**

152-ФЗ ст.22 обязывает оператора уведомить уполномоченный орган (РКН) до начала обработки ПДн, если нет исключений по ст.22 ч.2. Вопрос: подана ли уведомительная форма в РКН? Если нет — систему нельзя выводить в production даже для тестовых пользователей, если их данные обрабатываются реально. Регистрация — не технический вопрос, это обязательный административный шаг до запуска.
