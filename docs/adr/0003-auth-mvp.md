# ADR 0003 — Аутентификация и авторизация MVP

- **Статус**: принято
- **Дата**: 2026-04-15
- **Автор**: Архитектор (Claude Code, субагент `architect`)
- **Утверждающий**: Владелец (Мартин)
- **Контекст фазы**: Phase 2 — аутентификация, пользователи, роли
- **Связанные ADR**: ADR-0001 (таблица `users`, роли, `password_hash`), ADR-0002 (стек: FastAPI, Python 3.12, PostgreSQL 16)

---

## Контекст

Фазы 0 (инфраструктура) и 1 (модель данных, 17 таблиц) закрыты. Фаза 2 вводит в работу модуль аутентификации и авторизации. Система работает в закрытом контуре (несколько доверенных пользователей), прямого выхода в интернет на этапе MVP нет. Глубокая security-hardening запланирована отдельной фазой перед подключением к боевым данным.

ADR-0001 уже зафиксировал структуру таблицы `users` (колонка `password_hash text not null`, enum `role`) и отказ от `python-jose` в пользу `PyJWT`.

---

## Проблема

Нужно определить минимально достаточный способ выдачи и проверки идентичности пользователей и контроля доступа к эндпоинтам FastAPI для Фазы 2. Решение должно:

1. Не требовать внешней инфраструктуры (сервис идентификации, база сессий).
2. Быть понятным без опыта программирования — стандартные паттерны, минимум нестандартных механизмов.
3. Оставлять явные точки расширения для security-фазы (ротация ключей, 2FA, rate-limiting).

---

## Решение

Все пункты ниже утверждены Владельцем и фиксируются без дальнейшего обсуждения альтернатив.

### 1. Собственная аутентификация на FastAPI

Используем стандартный OAuth2PasswordBearer (форма `username`/`password`) из `fastapi.security`. Никаких внешних сервисов идентификации (Keycloak, Authentik, Auth0).

### 2. Хранение паролей — bcrypt через passlib

Колонка `users.password_hash text not null` (зафиксировано в ADR-0001).
Библиотека: `passlib[bcrypt]`. Хеш хранится в PHC-формате, соль генерируется автоматически.

> Примечание: ADR-0001 рекомендовал `argon2id` как технически более сильный выбор. Владелец принял решение использовать bcrypt в Фазе 2 ради меньшего числа зависимостей. Переход на argon2id возможен миграцией при повторном входе пользователей (re-hash on login) — зафиксировать в security-фазе.

### 3. JWT access-токен без refresh-ротации

- Алгоритм подписи: `HS256`.
- Библиотека: `PyJWT` (зафиксировано в ADR-0001).
- Переменные окружения:
  - `JWT_SECRET` — секрет подписи (обязательный, без дефолта).
  - `JWT_ACCESS_TTL_MINUTES` — время жизни токена (дефолт: `60`).
- Refresh-токены не вводятся. Истёкший токен требует повторного входа.
- Токены в БД не хранятся.

### 4. Роли и контроль доступа

Enum в БД и Python-коде: `owner | accountant | construction_manager | read_only`.

Проверка доступа — FastAPI dependency:

```
require_role(role: UserRole) -> Callable
```

Dependency внедряется в сигнатуру маршрута. Логика: декодировать токен → получить пользователя из БД → проверить `user.role >= required_role`.

Иерархия доступа (от полного к ограниченному):
`owner` > `accountant` / `construction_manager` > `read_only`

`accountant` и `construction_manager` — одного уровня, доступ разграничивается по конкретным маршрутам, не иерархически.

### 5. 2FA не внедряется

Двухфакторная аутентификация в Фазе 2 отсутствует. Зафиксировано как осознанный выбор (см. риски ниже).

### 6. Первый owner — сид-миграция Alembic

При запуске миграций: если таблица `users` пуста — создаётся одна запись `owner`.

Переменные окружения (обязательные, без дефолтов):
- `INITIAL_OWNER_EMAIL`
- `INITIAL_OWNER_PASSWORD` — читается, хешируется bcrypt, plain-text в памяти не сохраняется.

Сид-скрипт выполняется идемпотентно: повторный запуск миграции не создаёт дубль.

### 7. Аудит входов

В Фазе 2 — только стандартное логирование приложения (`structlog` / `logging`): событие входа, `user_id`, IP, статус (успех/неудача).

Полный audit-log (таблица `audit_log`, покрытие CRUD-эндпоинтов) — Фаза 3.

## Последствия

**Положительные:**
- Нет внешних зависимостей инфраструктуры — сервис полностью автономен.
- Стандартный OAuth2PasswordBearer читается в документации FastAPI напрямую.
- Роли в одном enum — легко расширить без структурных миграций (добавить значение).
- Сид-миграция исключает ручное создание первого пользователя вне кода.

**Отрицательные:**
- Без refresh-токенов пользователи будут выходить из системы каждые `JWT_ACCESS_TTL_MINUTES` минут.
- bcrypt ограничен 72 байтами входного пароля — на Фазе 2 это приемлемо, при смене на argon2id потребуется re-hash.
- `JWT_SECRET` в env без vault — единственная точка компрометации всех токенов.

---

## Known risks deferred to security phase

Следующие риски осознанно отложены и **должны быть устранены перед подключением боевых данных**:

- **Нет 2FA** — компрометация пароля = полный доступ к аккаунту.
- **Нет защиты от brute-force** — эндпоинт `/auth/token` не ограничен по числу попыток.
- **Нет rate-limiting** — любой эндпоинт API открыт для флуда.
- **Нет ротации JWT-секрета** — утечка `JWT_SECRET` компрометирует все выданные токены без возможности отзыва.
- **Нет refresh-токенов** — нет механизма тихого обновления сессии без повторного ввода пароля.
- **JWT_SECRET хранится в env без vault** — при компрометации окружения все токены скомпрометированы.

## Альтернативы (кратко)

### A. Keycloak / Authentik как внешний IdP

- **Плюсы**: 2FA, ротация ключей, управление сессиями из коробки.
- **Минусы**: отдельный сервис, конфигурация, оперативная поддержка, избыточность для 4–10 пользователей.
- **Почему отклонено**: противоречит принципу минимальной достаточности Фазы 2; отложено на security-фазу как опция.

### B. Сессии в БД вместо JWT

- **Плюсы**: мгновенный отзыв сессии, нет проблемы утечки секрета подписи.
- **Минусы**: каждый запрос = запрос к БД за сессией; больше кода; нестандартный паттерн для FastAPI.
- **Почему отклонено**: JWT — стандарт для FastAPI-экосистемы, проще в обучении и обслуживании на MVP.

## Amendments

- **2026-04-15** (коммиты 15a7479, 312dd50): `JWT_SECRET` → `JWT_SECRET_KEY` (имя env-переменной секрета подписи).
  Причина: совместимость с уже существующим `.env` Фазы 1, где переменная называлась `JWT_SECRET_KEY`.
  Влияние на безопасность: отсутствует — переименование не затрагивает само значение секрета и логику его валидации.
  Применимость: MVP-этап. При выходе на staging/production — оценить целесообразность возврата к каноническому имени `JWT_SECRET`.

- **2026-04-15** (Фаза 2, решение Координатора): в схемах Pydantic используется `str` вместо `EmailStr` для полей email.
  Причина: зависимость `email-validator` (требуется для `EmailStr`) отложена до security-фазы, чтобы не раздувать зависимости MVP.
  Влияние на безопасность: валидация формата email на уровне схемы отсутствует — принято осознанно для MVP с доверенным кругом пользователей.
  Применимость: MVP-этап. При добавлении `email-validator` в зависимости — заменить `str` на `EmailStr` во всех схемах, где фигурирует email.
