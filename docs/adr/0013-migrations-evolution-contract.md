# ADR 0013 — Migrations Evolution Contract: backward-compatible schema changes

- **Статус**: proposed (ожидает governance)
- **Дата**: 2026-04-17
- **Автор**: backend-director (субагент L2)
- **Утверждающий**: governance-director, затем Владелец (Мартин) — отдельная заявка
- **Контекст фазы**: M-OS-1 «Скелет», Волна 1 Foundation
- **Связанные документы**:
  - ADR 0001 (модель данных v1) — не нарушать
  - ADR 0011 (Foundation migration) — прецедент правил round-trip
  - `docs/pods/cottage-platform/m-os-1-foundation-adr-plan.md` v3 — план Волны 1
  - `CLAUDE.md` — правило round-trip (`alembic downgrade -1 && alembic upgrade head`)

---

## Проблема

На текущем этапе (ADR 0001–0012) отсутствует формальное соглашение о том, какие изменения схемы БД считаются обратно совместимыми (backward-compatible), а какие являются ломающими (breaking changes).

При росте системы до 5 подов и сотни таблиц разрозненные миграции без единого регламента создают нарастающий риск:

1. **Поломка работающих клиентов.** Код пода, который читает колонку, может получить ошибку, если другой под в той же транзакции уже применил `RENAME COLUMN` или `DROP COLUMN` без предварительного периода совместимости.
2. **Нарушение контракта между слоями.** SQLAlchemy-модели в `backend/app/models/` описывают ожидаемую схему. Если миграция меняет схему «втихую», модели и реальная таблица расходятся.
3. **Невозможность rollback без потери данных.** Миграция с `DROP COLUMN` без deprecation-периода делает downgrade разрушительным: данные в колонке теряются безвозвратно.
4. **Скрытые ошибки в CI.** Без автоматического линтера миграций ломающие изменения попадают в main в составе PR, которые на первый взгляд выглядят корректно.

Проблема становится P0 именно сейчас: ADR 0011 уже добавил первую крупную Foundation-миграцию с multi-company. Последующие ADR (0015–0017) добавят ещё десятки таблиц. Без регламента риск экспоненциально растёт с каждой следующей волной.

---

## Контекст

Текущее состояние:
- Миграции ведутся через Alembic.
- Правило round-trip (`alembic downgrade -1 && alembic upgrade head`) уже зафиксировано в `CLAUDE.md`, но не автоматизировано в CI.
- ADR 0011 содержит явную downgrade-стратегию — это правильный прецедент, но он не обязателен для следующих миграций по регламенту.
- Нет инструмента, который автоматически запрещает `DROP COLUMN`, `RENAME COLUMN`, `NOT NULL без DEFAULT` в Pull Request.

Данный ADR фиксирует три составляющих контракта: перечень запретов, процессную модель (expand/contract), инструментальную CI-защиту.

---

## Рассмотренные варианты

### Вариант A — Expand/contract pattern + CI-линтер (принятый)

Свод правил: что запрещено делать в одном шаге, что требует двухшагового процесса (expand → migrate data → contract), что разрешено без ограничений. Каждое правило проверяется линтером миграций в CI. Нарушение правила — блокировка PR.

Плюсы: минимальный операционный overhead; правила понятны как документ и как автоматизация; не меняет архитектуру системы.

Минусы: требует поддержки линтера при добавлении новых правил; разработчик должен осознанно применять двухшаговый процесс.

### Вариант B — Версионирование API вместе с миграциями

Каждое изменение схемы сопровождается новой версией API (`/api/v2/`). Старая версия живёт 2 спринта — за это время клиенты переходят.

Плюсы: полная гарантия совместимости через изоляцию версий.

Минусы: избыточно для внутренней системы с контролируемыми клиентами; два раза больше кода эндпоинтов; операционный overhead несоразмерен задаче.

**Отклонён.**

### Вариант C — Event sourcing полностью

Схема БД не меняется вообще: все изменения состояния — только через события. Проблема ломающих миграций снимается как класс.

Плюсы: принципиально решает проблему.

Минусы: требует полной переработки текущей реляционной модели; слишком высокая сложность для M-OS-1; противоречит решению по dual-event-bus (ADR-0016), где события — транспорт, а не источник истины для всех сущностей.

**Отклонён.**

## Решение

**Принят вариант A — expand/contract pattern с обязательным CI-линтером.**

### Правила допустимых изменений схемы

#### Запрещено в одном шаге (требует двухшагового процесса или deprecation)

| Операция | Статус | Обязательный путь |
|---|---|---|
| `DROP COLUMN` | Запрещено напрямую | Deprecation-период 2 спринта (6 рабочих недель), затем отдельная миграция удаления |
| `RENAME COLUMN` | Запрещено напрямую | Шаг 1: добавить новую колонку + копировать данные. Шаг 2: через 2 спринта удалить старую |
| `RENAME TABLE` | Запрещено напрямую | Аналогично: представление совместимости или переименование в два шага |
| `ALTER COLUMN ... NOT NULL` (без DEFAULT) | Запрещено | Только с `DEFAULT` или после заполнения существующих строк |
| `DROP TABLE` | Запрещено напрямую | Таблица переводится в «архивную» (переименование `_deprecated_<name>`), через 2 спринта — удаление отдельной миграцией |
| Изменение типа колонки с потерей данных | Запрещено | Только через expand/contract |

#### Требует явного обоснования в комментарии к миграции

| Операция | Требование |
|---|---|
| Изменение типа без потери данных (например, `varchar(64) → varchar(256)`) | Комментарий: «данные не теряются, обоснование: …» |
| Добавление NOT NULL с DEFAULT | Комментарий с указанием значения DEFAULT и его смысла |
| Добавление CHECK-ограничения на непустую таблицу | Комментарий: «существующие данные проверены, нарушений нет» |
| Изменение индексов (удаление, переименование) | Комментарий: «влияние на производительность оценено» |

#### Разрешено без ограничений

- Добавление новой таблицы
- Добавление новой nullable-колонки
- Добавление новой колонки с DEFAULT
- Добавление индекса (без конкурентного создания — `CREATE INDEX` без `CONCURRENTLY` блокирует таблицу; на больших таблицах обязательно `CONCURRENTLY`)
- Добавление внешнего ключа с `DEFERRABLE`
- Расширение enum (добавление нового значения)

### Двухшаговый процесс (expand/contract)

Для любой операции из раздела «Запрещено в одном шаге» применяется обязательный двухшаговый процесс:

**Шаг 1 — Expand.** Добавляется новый объект (колонка, таблица) рядом со старым. Старый объект остаётся. Приложение начинает писать в оба, читать из нового. Помечается комментарий в описании Alembic-ревизии: `# DEPRECATION: <old_column> will be removed in sprint N+2`.

**Шаг 2 — Contract.** Через 2 спринта (не раньше), после подтверждения что весь код перешёл на новый объект: отдельная миграция удаляет старый объект. Перед применением — повторный round-trip тест.

Между шагами 1 и 2 может быть промежуточный шаг переноса данных, если это необходимо (data migration).

### Обязательный round-trip тест в CI

Каждая миграция обязана проходить round-trip в CI:

```
alembic upgrade head
alembic downgrade -1
alembic upgrade head
```

Все три команды должны завершаться без ошибок. Это уже зафиксировано в `CLAUDE.md` как правило, данный ADR делает его обязательным CI-шагом, а не просто рекомендацией.

### Инструмент линтера миграций

CI-линтер проверяет каждый файл миграции Alembic при открытии PR. Два варианта реализации (выбирается при DoD):

**Вариант 1 — `squawk`.** Готовый PostgreSQL-линтер (open source), понимает SQL DDL, имеет встроенные правила для ряда ломающих изменений. Подключается как GitHub Action или pre-commit hook.

**Вариант 2 — кастомный линтер миграций.** Python-скрипт, парсирующий `op.*` вызовы в файлах Alembic-ревизий и проверяющий список запрещённых операций. Проще в поддержке, не требует внешнего инструмента, правила можно добавлять напрямую.

Рекомендация: начать с кастомного Python-линтера (`tools/lint_migrations.py`) — он проще в развёртывании и напрямую понимает Alembic API. При необходимости расширить `squawk`-ом для глубокого SQL-анализа.

Линтер интегрируется в CI-пайплайн как обязательный шаг до ревью: `pytest --co -q tools/test_lint_migrations.py` или отдельный GitHub Action.

### Исключения из правил

Исключение допускается только при соблюдении всех трёх условий одновременно:

1. Явное обоснование в тексте миграции: почему правило неприменимо.
2. Согласование с backend-director через commit-сообщение с пометкой `[migration-exception]`.
3. Ревью миграции backend-head отдельным проходом до мержа в main.

## Последствия

### Положительные

- Ломающие миграции блокируются автоматически в CI до попадания в main.
- Разработчики получают чёткий документ-ответ на вопрос «можно ли так менять схему».
- Downgrade любой миграции становится безопасным: двухшаговый процесс гарантирует, что данные сохранены до удаления объекта.
- Операционный риск при росте до 5 подов управляем: правила не зависят от количества подов.

### Отрицательные

- Некоторые изменения схемы теперь требуют двух PR вместо одного. Это замедляет рефакторинг модели данных.
- Линтер нужно поддерживать: при добавлении новых правил — обновление инструмента.
- Deprecation-период 2 спринта означает, что «технический долг по удалению» накапливается в реестре задач.

### Нейтральные

- Правило round-trip уже было зафиксировано в `CLAUDE.md`. Данный ADR формализует его как CI-требование, но не меняет существующую практику.
- Количество Alembic-ревизий вырастет (expand и contract — отдельные файлы), но это нормально для проекта с историей.

## Влияние на существующие ADR

**ADR 0001** (модель данных v1) — правила данного ADR применяются к любым будущим изменениям таблиц, описанных в ADR 0001.

**ADR 0011** (Foundation migration) — downgrade-стратегия в ADR 0011 уже соответствует принципам данного ADR. Ретроактивных изменений не требуется. ADR 0011 остаётся без изменений.

**ADR 0013 становится обязательной зависимостью для всех последующих ADR**, содержащих миграции схемы: ADR-0014, ADR-0015, ADR-0016, ADR-0017. Каждый из них обязан ссылаться на ADR-0013 в разделе «Связанные документы».
