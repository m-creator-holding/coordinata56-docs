# ADR 0024 — Verification Gate for Live Activations

- **Статус**: proposed (черновик — ожидает ревью Координатора перед governance approval; НЕ утверждать самостоятельно)
- **Дата**: 2026-04-19
- **Автор**: architect (субагент-советник)
- **Утверждающий**: governance-director, затем Владелец (Мартин)
- **Контекст фазы**: M-OS горизонтальный — действует поверх всех фаз и pods
- **Связанные документы**:
  - ADR 0015 (Integration Registry) — каталог подключений; live-активация интеграции подпадает под этот gate
  - ADR 0017 (Hooks as Defense-in-Depth) — данный ADR добавляет процессный слой защиты поверх механического
  - ADR 0018 (Production-Gate Definition) — production-gate как финальная граница; verification gate — обязательный чек-пункт внутри фазы
  - `CLAUDE.md` §«Процесс (runtime)» — правило verify-before-scale, инцидент 2026-04-19

## Контекст

M-OS-1 вводит live-инфраструктуру: WAL archiving в Yandex Object Storage, Sentry error tracking, SMTP-email, S3 backup. Каждая из этих систем активируется один раз (точка «включить»), а затем работает автономно — без постоянного внимания агента.

Специфика live-активации: результат — не отображение в UI, а событие, которое система должна сгенерировать самостоятельно при наступлении триггера. Человек или агент не наблюдает процесс в реальном времени — он видит только артефакт (файл в бакете, событие в Sentry, письмо в inbox).

Отсюда возникает класс ошибок «silent broken activation»: система технически «включена» по конфигурации, но в runtime не работает. Ошибка может оставаться незамеченной неделями.

---

## Проблема

**Инцидент 2026-04-19.** При live-активации WAL archiving:

1. DevOps вручную загрузил тестовый файл в Yandex Object Storage — убедившись, что доступ к бакету работает.
2. Этот факт был принят как доказательство работы archiving.
3. На следующий день `pg_switch_wal()` + проверка бакета выявили: `archive_mode=off` в runtime PostgreSQL — авто-архивация не работала более суток.

**Root cause 1 — расщепление env.** `.env.dev` содержит `POSTGRES_ARCHIVE_MODE=on`, но `docker-compose.yml` читает переменную из корневого `/root/coordinata56/.env` при подстановке в блок `command:`. В корневом `.env` переменная отсутствовала → fallback `off`. Ошибка не выявлялась, так как `docker compose config` не запускался.

**Root cause 2 — подмена доказательств.** Ручной upload в бакет доказывает только сетевую доступность хранилища. Он не доказывает, что PostgreSQL autoarchiver выполняет команду `archive_command` при заполнении WAL-сегмента.

Эти два root cause независимы и могут воспроизводиться в любой будущей live-активации (Sentry DSN, SMTP relay, backup cron, replication slot) — если не зафиксировать явный Definition of Done.

## Рассмотренные альтернативы

### Альтернатива A — Status quo: ручная проверка по усмотрению исполнителя

Доказательством работы считается любая позитивная обратная связь от инфраструктуры: ping прошёл, файл загрузился, конфиг выглядит правильно.

**Отклонена.** Именно этот подход привёл к инциденту 2026-04-19. Ручная проверка не отличает «доступ работает» от «автоматика работает». Риск: broken archiving незамеченным сутки и более.

### Альтернатива B — CI-gate: автоматизированный E2E smoke-test в pipeline

При каждом деплое pipeline прогоняет полный triggered-event test: `pg_switch_wal()` → ожидание → `aws s3 ls` проверка нового ключа.

**Принята как follow-up (отдельная задача для devops).** Корректное долгосрочное решение. Не блокирует текущий ADR: CI-gate требует инвестиций в инфраструктуру тестирования, которой пока нет. До его готовности действует ручной gate по данному ADR.

### Альтернатива C — Monitoring alert как единственная защита

Prometheus/Grafana/alerting на метрики archiving lag. Если alert не пришёл — всё работает.

**Принята как defense-in-depth, но не замещает gate.** Monitoring срабатывает постфактум: alert приходит после того, как пауза уже произошла. Verification gate — проактивная проверка в момент активации. Оба нужны одновременно.

### Альтернатива D — Явный E2E verification gate (ВЫБРАНА)

Перед закрытием задачи live-активации исполнитель обязан пройти три шага: runtime-check → triggered-event → artifact-verify. Без всех трёх DoD не считается выполненным.

---

## Решение

**Любая live-активация внешней инфраструктуры требует прохождения E2E Verification Gate перед закрытием задачи. Ручной upload, ping, curl к endpoint или визуальная проверка конфига — не являются доказательством автоматической работы системы.**

«Live-активация» — это первое включение любой из следующих категорий:
- Backup / archiving / replication (WAL, pg_dump cron, S3 backup)
- Monitoring / alerting (Sentry, Prometheus, uptime checks)
- Внешние уведомления (SMTP, Telegram webhook, Push)
- Интеграции с внешними системами (любой адаптер из Integration Registry, ADR 0015)
- Replication slots, logical decoding, CDC

## Правила (обязательные, статус MUST)

### Правило 1 — Трёхшаговый DoD live-активации

Каждая live-активация закрывается только при наличии доказательств всех трёх шагов:

**Шаг 1. Runtime-check.**
Выполнить проверку параметра внутри живого контейнера / процесса — не на localhost дев-машины, не по документации конфига:
- PostgreSQL: `docker exec <container> psql -U <user> -c "SHOW archive_mode; SHOW archive_command;"`
- Sentry: `curl -s <sentry_dsn_health_endpoint>` или проверка DSN через SDK
- SMTP: `telnet <host> 25` или отправка тестового письма с проверкой delivery log
- Общий случай: `curl /health` на реальном сервисе с проверкой ответа

**Шаг 2. Triggered-event.**
Искусственно спровоцировать автоматическое событие от системы — не создать артефакт вручную в обход системы:
- WAL archiving: `SELECT pg_switch_wal();` — принудительная архивация текущего WAL-сегмента
- Sentry: `sentry-cli send-event --message "verification gate test"` или `raise Exception("verification gate test")` в контексте SDK
- Backup cron: `docker exec <container> <backup-command>` (не копирование файла в бакет вручную)
- SMTP: отправка письма через SMTP relay, а не через API или прямую загрузку в inbox

**Шаг 3. Artifact-verify.**
Убедиться, что артефакт появился у цели с временной меткой менее 60 секунд после triggered-event:
- S3/YOS: `aws s3 ls s3://<bucket>/<prefix>/ --recursive` → проверить `LastModified` нового файла
- Sentry: открыть Issues → убедиться, что test-event видно в течение 60 секунд
- Email inbox: проверить письмо с корректными заголовками `Received:` от нашего relay, не от внешнего сервиса
- Мониторинг: alert должен появиться в системе уведомлений (не только в метриках)

Все три шага фиксируются в задаче (тикете/PR-описании) как скриншот или вывод команды.

---

### Правило 2 — docker compose config dry-run при изменении env

Любое изменение, которое добавляет, переименовывает или удаляет переменную окружения, участвующую в подстановке `docker-compose.yml` (блоки `environment:`, `command:`, `args:`), требует перед применением:

```
docker compose --env-file <path-to-env> config | grep -i <VARIABLE_NAME>
```

Проверить: переменная присутствует в результате с ожидаемым значением. Если значение пустое или отсутствует — задача не переходит в статус «готово».

---

### Правило 3 — Единый источник истины для env подстановок

`docker-compose.yml` substitution (синтаксис `${VAR}`) читает переменные строго из корневого `.env` файла проекта.

Файлы `.env.dev`, `.env.staging`, `.env.test` используются **только** через директиву `env_file:` внутри секции `services:` — для передачи переменных внутрь контейнера при его запуске. Они не влияют на подстановку в `docker-compose.yml` во время `docker compose up`.

Любая переменная, которая нужна как для compose-подстановки, так и внутри контейнера, должна присутствовать в корневом `.env`.

## Последствия и влияние

### Ускорение теряем, надёжность приобретаем

Verification gate добавляет 30–60 минут к каждой live-активации: время на прогон трёх шагов и фиксацию доказательств. Взамен исключается класс инцидентов «silent broken activation», который может оставаться незамеченным от суток до нескольких недель.

Соотношение: 1 час overhead при активации vs N × (часы реакции на инцидент + время восстановления + объём потерянных данных). При broken WAL archiving — потенциальная потеря резервных копий за всё время простоя.

### Требуемые обновления документации

- `docs/agents/departments/infrastructure.md` §live-activation — добавить раздел «DoD live-активации» со ссылкой на этот ADR и описанием трёх шагов
- Runbook live-activation (ответственный: devops-head) — обновить до следующей live-активации, но не позднее начала M-OS-1 Sprint 2
- `CLAUDE.md` — правило уже добавлено в секцию «Процесс (runtime)» (инцидент 2026-04-19)

### Взаимодействие с CI-gate (Альтернатива B)

При введении автоматизированного CI-gate (Альтернатива B, follow-up задача devops) Правило 1 данного ADR может быть частично делегировано pipeline. Ручной triggered-event verify при этом остаётся обязательным для первой активации на каждом окружении (dev / staging / prod), даже если CI-тест зелёный: CI проверяет поведение системы в тестовом контексте, ручной verify — в живом.

### Взаимодействие с Monitoring (Альтернатива C)

Monitoring alert (Prometheus archiving lag, Sentry uptime) действует как defense-in-depth: обнаруживает деградацию после активации, когда gate уже пройден. Gate и monitoring — два независимых слоя, не заменяют друг друга.

## Риски

| Риск | Вероятность | Влияние | Митигация |
|---|---|---|---|
| Исполнитель пропускает шаги gate под давлением срока | Средняя | Высокое (инцидент повторяется) | Правило 1 входит в Definition of Done; задача не закрывается reviewer без доказательств |
