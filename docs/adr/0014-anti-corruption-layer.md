# ADR 0014 — Anti-Corruption Layer: каркас адаптеров с жизненным циклом и безопасностью по умолчанию

- **Статус**: proposed (ожидает governance)
- **Дата**: 2026-04-17
- **Автор**: backend-director (субагент L2)
- **Утверждающий**: governance-director, затем Владелец (Мартин) — отдельная заявка
- **Контекст фазы**: M-OS-1 «Скелет», Волна 1 Foundation
- **Связанные документы**:
  - ADR 0009 (pod-архитектура) — принцип изоляции подов от внешнего мира
  - ADR 0011 (Foundation) — multi-company, RBAC
  - ADR-0013 (Migrations Evolution Contract) — правила схемы для `integration_catalog`
  - ADR-0015 (Integration Registry) — реестр адаптеров (пишется в Волне 2, ADR-0014 является его фундаментом)
  - `docs/m-os-vision.md` §3.4 — список внешних интеграций Vision
  - `docs/agents/CODE_OF_LAWS.md` ст. 45а/45б — запрет живых вызовов без production-gate
  - `docs/pods/cottage-platform/m-os-1-foundation-adr-plan.md` v3 — план Волны 1

---

## Проблема

M-OS-1 требует одной живой интеграции (Telegram-бот) и нескольких «на полке» (банки, ОФД, 1С, Росреестр, КриптоПро). На текущий момент нет формального изолирующего слоя между внешними API и ядром системы.

Это создаёт три взаимосвязанных риска.

**Риск 1 — Проникновение внешних API в ядро.** Без явного слоя-адаптера разработчик пишет HTTP-вызов к внешнему сервису прямо в бизнес-сервисе. Через 6 месяцев такой код размазан по 15 местам, и замена одного банка на другой требует ревизии всей кодовой базы.

**Риск 2 — Случайный сетевой вызов из dev/test/CI к продуктивному endpoint-у.** Это напрямую запрещено CODE_OF_LAWS ст. 45а/45б. Без технической защиты правило держится только на дисциплине разработчика. При добавлении новых сотрудников (субагентов, людей) правило неизбежно будет нарушено.

**Риск 3 — Отсутствие явного жизненного цикла адаптера.** Нет общего ответа на вопрос: «в какой среде этот адаптер сейчас активен и почему?». Это затрудняет диагностику инцидентов, аудит безопасности и принятие решения о production-gate.

---

## Контекст

**Ограничения Владельца (CODE_OF_LAWS ст. 45а/45б):**
- Живые вызовы к банкам, Росреестру, 1С, ОФД запрещены до прохождения production-gate.
- Production-gate включает проверку юристом и явное решение Владельца.
- Единственная разрешённая живая интеграция в M-OS-1 — Telegram-бот (уже работает).

**Что уже есть:**
- ADR 0009 зафиксировал изоляцию подов: поды не вызывают внешние системы напрямую.
- ADR 0011 дал RBAC: есть понятие `owner`, который принимает решение о production-gate.
- Telegram-адаптер фактически существует как код, но без формального каркаса ACL.

**Что нужно закрыть данным ADR:**
- Единый базовый класс и интерфейс адаптера.
- Три явных состояния адаптера, понятных в коде и реестре.
- Техническая блокировка сетевых вызовов на уровне кода и CI.
- Разделение ответственности: этот ADR описывает каркас; реестр (таблица `integration_catalog`, CRUD, seed) — ADR-0015.

## Рассмотренные варианты

### Вариант A — Каркас адаптеров с тремя состояниями, mock-режимом и CI-защитой (принятый)

Базовый абстрактный класс `IntegrationAdapter` с явным перечнем состояний (`written`, `enabled_mock`, `enabled_live`). Переход между состояниями — только через реестр (ADR-0015). Runtime-guard выбрасывает `AdapterDisabledError` при попытке сетевого вызова в неразрешённом состоянии. CI-защита через `pytest-socket`. Iptables egress-блокировка на уровне инфраструктуры — требует отдельной заявки к infra-director через Координатора.

Плюсы: технически невозможно случайно «пробить» защиту; чёткий жизненный цикл; тестируемость по умолчанию.

Минусы: добавляет обязательный overhead при разработке каждого нового адаптера (mock-режим, регистрация).

### Вариант B — Только feature flags в `.env` без базового класса

Feature flag `ADAPTER_TELEGRAM_ENABLED=true` в конфигурации. Каждый адаптер читает свой флаг при инициализации. Никакого общего базового класса.

Плюсы: просто, знакомо.

Минусы: флаги в `.env` могут случайно оказаться `true` в тестовом окружении; нет единого места для проверки состояния; нет ни одной технической гарантии против сетевого вызова; CODE_OF_LAWS ст. 45б требует, что переход в `enabled=True` — явное решение Владельца, а в `.env` это не отслеживается.

**Отклонён.**

### Вариант C — Service mesh с sidecar-прокси (Envoy, Istio)

Все исходящие вызовы проходят через sidecar, который реализует белый список.

Плюсы: максимальная надёжность egress-контроля; не зависит от кода приложения.

Минусы: избыточно для MVP-стадии; требует Kubernetes или аналогичного оркестратора; операционная сложность несоразмерна текущему масштабу.

**Отклонён для M-OS-1. Может быть рассмотрен в M-OS-3+.**

## Решение

**Принят вариант A — каркас адаптеров с тремя состояниями, mock-режимом и многоуровневой CI/runtime-защитой.**

### Три состояния адаптера

Каждый адаптер находится в одном из трёх состояний. Состояние хранится в реестре интеграций (таблица `integration_catalog`, ADR-0015). Код адаптера в своём методе-транспорте обязан проверять состояние через реестр до открытия сетевого соединения.

**`written` — написан, не подключён нигде.**
Код адаптера существует. Интерфейс реализован. Mock-транспорт работает. Ни в одной среде адаптер не подключён к реальному endpoint-у. Запись в `integration_catalog` имеет `state='written'`, `enabled=False`. Адаптер в этом состоянии может существовать сколько угодно долго — это «полка» до принятия решения.

**`enabled_mock` — подключён в dev/test только с mock-транспортом.**
Адаптер зарегистрирован в `integration_catalog` с `state='enabled_mock'`, `enabled=True`. Транспорт принудительно подменяется на `MockTransport`. Любая попытка использовать live-транспорт (открыть реальный сокет) в этом состоянии — немедленный `AdapterDisabledError`. В prod-среде этот переход недопустим: если `APP_ENV=production` и `state='enabled_mock'` — приложение не стартует (fail-fast при инициализации).

**`enabled_live` — живые вызовы разрешены в prod после production-gate.**
Перевод в это состояние возможен только при одновременном выполнении трёх условий: прохождение production-gate (юрист + Владелец, CODE_OF_LAWS ст. 45б), `APP_ENV=production` в конфигурации, запись в `integration_catalog` с `state='enabled_live'` и заполненным `credentials_ref`. В dev и staging `state='enabled_live'` технически невозможен: guard возвращает `AdapterDisabledError` до открытия соединения, даже если записать состояние вручную.

Единственный адаптер со статусом `enabled_live` в M-OS-1 — Telegram-бот.

### Базовый класс и интерфейс адаптера

Каждый адаптер обязан наследоваться от `IntegrationAdapter` (расположение: `backend/app/core/integrations/base.py`).

Интерфейс содержит:
- `adapter_name: str` — уникальный идентификатор в реестре (например, `"telegram"`, `"sberbank"`)
- `get_state() -> AdapterState` — чтение текущего состояния из реестра
- `call(...)` — публичный метод вызова, обязательно проходит через guard перед вызовом транспорта
- `_live_transport(...)` — абстрактный метод реального сетевого вызова (реализуется в конкретном классе)
- `_mock_transport(...)` — абстрактный метод mock-ответа (реализуется в конкретном классе, обязателен)

Запрещено: реализовывать транспорт непосредственно в методе `call()`. Запрещено: делать сетевой вызов обходя guard. Линтер (`ruff` + custom rule) проверяет отсутствие прямых `httpx.get`, `requests.post` и аналогов вне `_live_transport`.

### Feature flag на каждый адаптер

По умолчанию `enabled=False` для всех адаптеров в `integration_catalog`. Переход `enabled=False → enabled=True` фиксируется в AuditLog (ADR 0007 + ADR 0011 crypto chain) с указанием пользователя, времени и основания. Значение флага хранится в БД (реестр, ADR-0015), не в `.env`. `.env` содержит только технические параметры подключения (`APP_ENV`, `DATABASE_URL`), не решения о включении адаптеров.

### CI-защита: pytest-socket

В `conftest.py` подключается `pytest-socket` с блокировкой всех сетевых вызовов:

```
# conftest.py (корневой)
import pytest
@pytest.fixture(autouse=True)
def disable_network(socket_disabled):
    pass
```

Это означает, что любой тест, который случайно вызывает реальный сокет, упадёт с явной ошибкой `SocketBlockedError`. Тесты, которым нужна сеть (только интеграционные тесты Telegram в специальном CI-окружении), явно помечаются `@pytest.mark.allow_network` и запускаются отдельным пайплайном.

### Runtime-guard: AdapterDisabledError

Guard встроен в базовый класс `IntegrationAdapter.call()`. Логика:

1. Прочитать `state` из реестра (кешировано в памяти, TTL 60 сек; при изменении реестра — инвалидация через business-шину, ADR-0016).
2. Если `state == 'written'` — выбросить `AdapterDisabledError("Adapter is in 'written' state, not enabled in any environment")`.
3. Если `state == 'enabled_mock'` — вызвать `_mock_transport()`, не открывать сокет.
4. Если `state == 'enabled_live'` И `APP_ENV != 'production'` — выбросить `AdapterDisabledError("Live adapter can only be called in production environment")`.
5. Если `state == 'enabled_live'` И `APP_ENV == 'production'` — вызвать `_live_transport()`.

`AdapterDisabledError` наследуется от `RuntimeError`, логируется в AuditLog с action=`adapter_call_blocked`.

### Обязательный mock-режим

Каждый адаптер без реализованного `_mock_transport()` не проходит CI-шаг `test_all_adapters_have_mock`. Адаптер без mock-режима нельзя смержить в main — это жёсткое правило, не рекомендация.

Mock должен возвращать детерминированные ответы, соответствующие контракту реального API. Минимальный набор mock-ответов: успешный ответ, ответ с ошибкой сети, ответ с бизнес-ошибкой внешнего сервиса.

### Egress-блокировка на уровне инфраструктуры (iptables)

Исходящие сетевые соединения к внешним endpoint-ам должны быть закрыты по умолчанию на уровне ОС. Разрешены только endpoint-ы адаптеров в состоянии `enabled_live`, записанные в whitelist.

**Это требование выходит за границы backend-отдела и требует решения infra-director.** Backend-director формулирует запрос к Координатору: активировать infra-director и поставить задачу настройки iptables egress-правил на основе whitelist из `integration_catalog`. До получения этого решения runtime-guard обеспечивает программную защиту; iptables — второй эшелон.

### Адаптеры M-OS-1: стартовый перечень

| Адаптер | Состояние при seed | Комментарий |
|---|---|---|
| `telegram` | `enabled_live` | Единственный live-адаптер M-OS-1 |
| `sberbank` | `written` | До production-gate банковского модуля |
| `tinkoff` | `written` | До production-gate банковского модуля |
| `ofd` (ОФД) | `written` | До production-gate фискальной интеграции |
| `1c` | `written` | До production-gate интеграции с 1С |
| `rosreestr` | `written` | До production-gate интеграции с Росреестром |
| `kryptopro` | `written` | До production-gate КриптоПро |

Seed-миграция создаёт все записи в `integration_catalog` с `state='written'`, `enabled=False`, кроме Telegram (`state='enabled_live'`, `enabled=True`).

## Последствия

### Положительные

- Случайный сетевой вызов к продуктивному банку из dev/test/CI становится технически невозможным: три независимых уровня защиты (guard, pytest-socket, iptables).
- CODE_OF_LAWS ст. 45а/45б соблюдается технически, а не только на бумаге.
- Добавление нового внешнего сервиса — стандартный паттерн: написать класс-наследник, реализовать `_live_transport` и `_mock_transport`, добавить запись в реестр со статусом `written`. Больше ничего.
- Жизненный цикл адаптера аудируется: каждый переход состояния попадает в AuditLog с hash-chain.

### Отрицательные

- Разработка каждого нового адаптера требует обязательного mock-режима. Это правильно, но добавляет время.
- Egress-блокировка через iptables зависит от infra-director — отдельная заявка через Координатора.
- Guard читает реестр при каждом вызове адаптера (кешировано, но всё же зависимость от БД при старте кеша или его инвалидации).

### Нейтральные

- ADR-0015 (Integration Registry) является дополнением к данному ADR: таблица `integration_catalog`, CRUD-эндпоинты, seed-скрипт — там. Данный ADR описывает каркас и жизненный цикл, ADR-0015 — хранилище состояния.
- Telegram-адаптер, который уже существует в коде, нужно привести к новому каркасу в рамках DoD этого ADR.

## Влияние на существующие ADR

**ADR 0009** (pod-архитектура) — данный ADR конкретизирует механизм изоляции подов от внешнего мира: явно запрещает прямые HTTP-вызовы вне слоя адаптеров. ADR 0009 изменений не требует, но следует добавить ссылку в его раздел «связанные документы» при следующем amendment.

**ADR 0011** (Foundation) — `AdapterDisabledError` пишется в AuditLog, наследующий crypto-chain из ADR 0011. Изменений в ADR 0011 не требуется.

**ADR-0013** (Migrations) — таблица `integration_catalog` (ADR-0015) создаётся миграцией Alembic по правилам ADR-0013. Прямого изменения ADR-0013 не требуется.

**ADR-0015** (Integration Registry) — является прямым продолжением данного ADR. Волна 2. Данный ADR является предусловием для ADR-0015.

## Открытый вопрос: egress-блокировка через iptables

Требование к infra-director: настроить исходящие iptables-правила так, чтобы по умолчанию все исходящие соединения к внешним IP-адресам были закрыты. Разрешённые endpoint-ы берутся из поля `base_url` записей `integration_catalog` со статусом `enabled_live`.

Механизм обновления whitelist: при переводе адаптера в `enabled_live` — событие в `business_events_bus` (ADR-0016) с типом `AdapterActivated`; infra-скрипт подписан на это событие и обновляет iptables-правила.

**Данный вопрос требует решения Координатора**: активировать infra-director с задачей проектирования и реализации egress-блокировки как части M-OS-1 производственной готовности. До решения runtime-guard (`AdapterDisabledError`) обеспечивает программный заслон — это достаточно для dev и staging; для production требуется iptables как второй рубеж.
