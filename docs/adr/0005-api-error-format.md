# ADR 0005 — Формат ошибок API (Phase 3)

- **Статус**: proposed
- **Дата**: 2026-04-15
- **Автор**: Архитектор (Claude Code, субагент `architect`)
- **Утверждающий**: Владелец (Мартин)
- **Контекст фазы**: Phase 3 — базовые CRUD API, кросс-срезовый компонент
- **Связанные ADR**: ADR-0002 (FastAPI, Pydantic v2), ADR-0004 (трёхслойная архитектура, `NotFoundError` / `ConflictError`)

---

## Контекст

FastAPI возвращает ошибки в разных форматах в зависимости от источника: `HTTPException` даёт `{"detail": "..."}`, Pydantic validation error — `{"detail": [{"loc": [...], "msg": "...", "type": "..."}]}`, необработанные исключения Python — 500 без тела. Без единого стандарта Swagger будет показывать разные схемы ошибок, фронтенд будет обрабатывать несколько форматов, а Владелец при ручном тестировании не поймёт, в чём разница между ошибками.

Фаза 3 вводит 15 роутеров — это момент, когда формат нужно зафиксировать один раз для всего API.

---

## Проблема

Нужно выбрать единый формат JSON-ответа для всех ошибок (4xx, 5xx), который:

1. Является машиночитаемым: фронтенд различает типы ошибок по полю, не по тексту сообщения.
2. Swagger-friendly: ошибки документируются через стандартные response-схемы Pydantic.
3. Поддерживает будущую i18n: текст сообщения можно заменить без изменения кода клиента.
4. Совместим с FastAPI `HTTPException` (не требует замены встроенного механизма целиком).
5. Содержит структурированные детали для ошибок валидации (поле, причина).

---

## Рассмотренные альтернативы

### Вариант A: RFC 7807 Problem Details (`application/problem+json`)

Стандарт IETF (RFC 7807, обновлён RFC 9457 в 2023). Формат:

```json
{
  "type": "https://coordinata56.local/errors/not-found",
  "title": "Resource Not Found",
  "status": 404,
  "detail": "Project with id=42 not found",
  "instance": "/api/v1/projects/42"
}
```

**Плюсы:**
- Международный стандарт — при интеграции с внешними системами понятен без документации.
- Поле `type` — URI, уникальный идентификатор типа ошибки: машиночитаем и расширяем.
- Поддерживается OpenAPI 3.1 через `content-type: application/problem+json`.

**Минусы:**
- Требует поддержки специального `Content-Type`, что усложняет FastAPI-интеграцию (нужно переопределять `Response`).
- `type` как URI — избыточно для внутренней системы без внешних интеграций в MVP.
- Поле `title` дублирует `detail` для большинства случаев.
- Фронтенд всё равно будет смотреть на `type` (URI), а не на `status` — придётся поддерживать маппинг URI → текст ошибки. Это сложнее простого строкового кода.
- Тяжелее объяснить Владельцу: «что такое `type` как URL, который никуда не ведёт».

**Почему отклонено**: избыточность стандарта для внутренней ERP без внешних API-потребителей на MVP.

---

### Вариант B: FastAPI default `{"detail": "..."}` или `{"detail": [...]}`

Формат «из коробки» FastAPI: строка для `HTTPException`, список объектов для ошибок валидации Pydantic.

```json
{"detail": "Not found"}
{"detail": [{"loc": ["body", "code"], "msg": "field required", "type": "missing"}]}
```

**Плюсы:**
- Нулевые затраты на реализацию — уже работает.
- Все примеры FastAPI-документации используют этот формат.

**Минусы:**
- Нет машиночитаемого кода ошибки: фронтенд различает ошибки только по тексту строки (`"Not found"`) или HTTP-статусу. Изменение текста ломает клиентский код.
- Два разных формата в одной системе: строка для `HTTPException` и список для валидации — фронтенд должен обрабатывать оба случая.
- Нет места для `field` в ошибках валидации в едином формате.
- Нет кода ошибки (`NOT_FOUND`, `CONFLICT`) — невозможно реализовать i18n без изменения API.

**Почему отклонено**: нет машиночитаемого кода ошибки — это фундаментальный изъян для любого нетривиального фронтенда и i18n.

### Вариант C (выбран): Собственный JSON `{"error": {"code": ..., "message": ..., "details": [...]}}`

Формат:

```json
{
  "error": {
    "code": "NOT_FOUND",
    "message": "Project with id=42 not found",
    "details": []
  }
}
```

Для ошибок валидации:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request validation failed",
    "details": [
      {"field": "code", "message": "Field is required"},
      {"field": "name", "message": "Must be at most 255 characters"}
    ]
  }
}
```

**Плюсы:**
- `code` — строковый константный идентификатор: фронтенд switch/if по `error.code`, не по тексту.
- Единый формат для всех ошибок (HTTP-исключения и ошибки валидации).
- Swagger документирует одну схему `ErrorResponse` для всех `4xx`/`5xx`.
- `message` — человекочитаемый текст: при i18n заменяется на фронтенде или через middleware, `code` остаётся неизменным.
- `details` — список для множественных ошибок валидации: поле + причина.
- Вписывается в FastAPI через `exception_handler` без изменения встроенного механизма роутинга.

**Минусы:**
- Нестандартный формат — документировать явно.
- `error` как обёртка — дополнительный уровень вложенности.

**Почему выбрано**: максимальная простота реализации при полной машиночитаемости и единообразии.

---

## Решение

Принимается **Вариант C**: собственный JSON с полем `error.code`.

### Pydantic-схемы ответа

```python
class ErrorDetail(BaseModel):
    field: str | None = None
    message: str

class ErrorBody(BaseModel):
    code: str
    message: str
    details: list[ErrorDetail] = []

class ErrorResponse(BaseModel):
    error: ErrorBody
```

### Реестр стандартных кодов ошибок

| Код | HTTP-статус | Описание |
|---|---|---|
| `NOT_FOUND` | 404 | Запрошенный ресурс не существует или мягко удалён |
| `PERMISSION_DENIED` | 403 | Пользователь не имеет роли для данной операции |
| `VALIDATION_ERROR` | 422 | Тело запроса не прошло Pydantic-валидацию |
| `CONFLICT` | 409 | Нарушение уникальности или бизнес-инварианта (дубль, ссылочная целостность) |
| `IMMUTABLE_RESOURCE` | 409 | Попытка изменить ресурс в финальном состоянии (например, Payment со статусом `approved`) |
| `UNAUTHENTICATED` | 401 | JWT отсутствует или невалиден |
| `INTERNAL_ERROR` | 500 | Непредвиденная ошибка сервера (детали не раскрываются клиенту) |

### Интеграция с FastAPI

Регистрируются два `exception_handler` в `app/main.py`:

1. Обработчик `RequestValidationError` (ошибки Pydantic) → `VALIDATION_ERROR` с полем `details`.
2. Обработчик базового класса `AppError` (наши доменные исключения из ADR-0004) → соответствующий код из реестра.

Встроенный `HTTPException` FastAPI остаётся только для 500 (необработанные исключения) — его обработчик тоже переопределяется для единообразия формата.

### Правило маскировки секретов (MUST)

Поле `message` в ответах 5xx **не раскрывает** трассировку стека, имена таблиц БД или пути файловой системы. В `details` — только то, что явно разрешено для отдачи клиенту (поля запроса, бизнес-сообщение). Полные детали ошибки пишутся в лог приложения.

## Последствия

**Положительные:**
- Фронтенд Фазы 4 получает предсказуемый `error.code` для каждого сценария ошибки.
- Swagger автоматически документирует единую схему `ErrorResponse`.
- Добавление нового кода ошибки — просто новая константа в реестре, без изменения схемы.
- i18n в будущем: фронтенд хранит словарь `{code: translated_message}` и показывает перевод по `code`, игнорируя `message` из API.

**Отрицательные:**
- Нужно явно переопределить три стандартных обработчика FastAPI (`RequestValidationError`, `HTTPException`, `Exception`) в `main.py`.
- Все кастомные исключения (из ADR-0004) должны наследоваться от `AppError` и содержать `code` из реестра.

**Риски:**

| Риск | Митигация |
|---|---|
| Субагент использует `raise HTTPException` напрямую вместо `AppError` | Ревьюер проверяет по чек-листу; `HTTPException` разрешён только в слое роутера для 401/403 от `require_role` |
| В `details` утекают внутренние детали (имя таблицы, SQL-ошибка) | Обработчик 500 всегда возвращает фиксированный текст; детали только в логе |

---

## Открытые вопросы

1. **Локализация `message` на стороне API**: если в будущем потребуется отдавать переведённое сообщение из backend, нужен заголовок `Accept-Language` и словарь сообщений. Откладывается до явного требования от Владельца.
2. **Коды для бизнес-специфичных ошибок** (`HOUSE_STAGE_REGRESSION`, `PAYMENT_APPROVED_IMMUTABLE`): расширять реестр при реализации батчей Фазы 3. Каждый новый код — отдельная строка в таблице выше с указанием HTTP-статуса.
