# ADR 0009 — Pod-архитектура M-OS: общее ядро, контракты данных, шина событий

- **Статус**: черновик, ожидает утверждения governance-комиссией
- **Дата**: 2026-04-16
- **Автор черновика**: Архитектор (Claude Code, субагент `architect`)
- **Утверждающий**: governance-director, затем Владелец (Мартин)
- **Контекст фазы**: Фаза M-OS-0 «Reframing»
- **Связанные документы**: ADR 0008 (определение M-OS), docs/m-os-vision.md §3, §8.15, §11

---

## Проблема

M-OS охватывает пять бизнес-направлений холдинга: коттеджи, АЗС, металл, МКД, карьер. Если каждое направление строить независимо — через год получим несвязанные системы с дублирующимися справочниками, разными схемами пользователей и аудита, невозможностью консолидированной отчётности.

Нужно зафиксировать: что является общим для всех направлений, что принадлежит только своему направлению, и как направления обмениваются данными без прямых зависимостей между собой.

---

## Контекст

ADR 0008 зафиксировал: M-OS состоит из независимых модулей — подов. Каждый под отвечает за одно бизнес-направление. Поды разделяют общее ядро (пользователи, роли, аудит, справочники).

Принцип 10 из m-os-vision.md §2 («Разомкнутая архитектура, pluggability + редактируемость») требует: новый под добавляется без правки ядра. Администратор подключает новое направление через интерфейс, не через код. Этот же принцип распространяется на все бизнес-сущности пода: роли, права, лимиты, процессы, шаблоны документов, формы, отчёты, уведомления, бизнес-правила — они хранятся как данные и редактируются через UI, не через код (Telegram msg 856, 2026-04-16).

Принцип 7 («Эволюция без breaking changes») требует: изменения схем данных и контрактов не ломают работающие поды.

Принцип 1 («Всё записывается») требует: аудит охватывает действия во всех подах единообразно.

На Фазе M-OS-1 будут два активных пода: `cottage-platform-pod` (пилот) и `gas-stations-pod-lite` (проверка модульности). Архитектура должна выдержать это без переделки ядра.

---

## Решение

### 1. Принципы pod-архитектуры

Один под = одно бизнес-направление холдинга.

Под изолирован по предметной логике: модели данных, бизнес-процессы и экраны интерфейса для строительства коттеджей не связаны с моделями для продажи топлива. Ни один под не импортирует напрямую модели другого.

Поды разделяют общее ядро. Общее ядро — это код и данные, которые одинаково нужны всем направлениям.

Новый под добавляется без правки ядра. Под регистрируется как запись в таблице конфигурации, создаёт свои таблицы через миграцию, объявляет свои роли — и ядро начинает их обслуживать.

### 2. Состав общего ядра

Общее ядро включает ровно то, что одинаково нужно всем бизнес-направлениям без исключения:

| Компонент | Что делает |
|---|---|
| Пользователи и аутентификация | Один аккаунт на сотрудника, JWT-токены, сессии |
| Роли и права | Доступ = роль + объект + действие, включая межподовые права (принцип 6) |
| Журнал действий (AuditLog) | Запись каждого действия с криптографической цепочкой хэшей (принцип 1) |
| Справочники организаций | Юрлица холдинга, контрагенты, банки, статьи затрат — единые для всех подов (shared master data) |
| Уведомления | Telegram, email, push — единый хаб, поды только генерируют события |
| Процессный движок (BPM) | Согласования, таймеры, эскалации — ядро исполняет, поды описывают процессы |
| Интеграционный слой (anti-corruption) | Буфер между M-OS и внешними системами (банки, ОФД, 1С, Росреестр) |
| Чат и комментарии к объектам | Переписка привязана к объекту любого пода по единой схеме |

Что не входит в ядро: модели данных, специфичные для одного направления (Дом, Договор стройки, Диспенсер, Тип топлива). Это принадлежит своему поду.

Граница ядра — не расширяется молча. Любое добавление компонента в ядро оформляется через ADR и проходит через governance-комиссию.

**Граница `core/` vs `pods/core_pod/`:** директория `core/` содержит код, одинаково нужный всем бизнес-направлениям без исключения (аутентификация, роли, аудит, справочники, уведомления, BPM, интеграционный слой, чат). Директория `pods/core_pod/` содержит предметную логику общехолдинговых функций (HR, финансы холдинга, CRM) — функции, которые по смыслу являются подом с общехолдинговым scope, но не частью технического ядра. Вопрос «куда класть код» решается однозначно: если это нужно только HR или финансам — в `pods/core_pod/`; если это инфраструктура, на которую опираются все поды — в `core/`.

### 3. Состав отдельного пода

Каждый под содержит:

**Модели данных (специфичные для домена):**
- `cottage-platform-pod`: `House`, `HouseConfiguration`, `Stage`, `Contract`, `Payment`, `Contractor`
- `gas-stations-pod`: `Station`, `Dispenser`, `FuelType`, `Shift`, `FuelSale`
- Остальные поды — по аналогии

**Бизнес-процессы:** схемы согласований и рабочих процессов, специфичные для домена

**Роли домена:** `cottage-sales-manager`, `site-foreman`, `gas-station-cashier` и т.д. — объявляются в конфигурации пода, регистрируются в ядре

**Экраны интерфейса:** компоненты React, специфичные для направления; ядро предоставляет layout и общие компоненты

**Миграции БД:** миграции пода хранятся внутри пода в `backend/app/pods/<pod_name>/migrations/`

### 4. Контракты данных между подами

Поды не обращаются к таблицам друг друга напрямую. Если поду нужны данные другого пода — он использует опубликованный контракт.

**Принципы контракта:**

Схема версионируется: `ContractorV1`, `ContractorV2`. Версия в имени типа, не в URL.

Pod-издатель публикует контракт: объявляет Pydantic-схему данных, которые он готов предоставлять другим.

Pod-подписчик подписывается на конкретную версию контракта. Код подписчика знает только версию, на которую подписан.

Breaking change = явная миграция. При несовместимом изменении:
1. Издатель публикует `V2` рядом с `V1`
2. Подписчики переходят на `V2` в своём темпе
3. `V1` снимается только когда все подписчики перешли
4. Факт снятия оформляется через ADR-amendment

Это реализует принцип 7 из m-os-vision.md §2: «эволюция без breaking changes».

**Пример контракта:**
```
# Издатель: core-pod
ContractorV1:
  id: int
  inn: str
  name: str
  legal_form: str

# Подписчик: cottage-platform-pod
# Получает ContractorV1 через шину событий или API ядра
# Не обращается напрямую к таблице contractors
```

Справочники (организации, контрагенты, банки, статьи затрат) живут в ядре и являются общей мастер-данными для всех подов — это исключение: поды читают их напрямую через ядровое API, не через контракт.

### 5. Шина событий (event bus)

Поды общаются событиями, не прямыми API-вызовами друг к другу.

**Зачем:** если pod-A вызывает API pod-B напрямую, то падение pod-B роняет pod-A. Шина событий обеспечивает изоляцию отказов: при отключении пода другие продолжают работу.

**Примеры событий M-OS:**

| Событие | Издатель | Подписчик |
|---|---|---|
| `payment.approved` | core-pod (BPM) | cottage-platform-pod, gas-stations-pod |
| `contract.created` | cottage-platform-pod | core-pod (уведомления, аудит) |
| `stage.closed` | cottage-platform-pod | core-pod (BPM, уведомления) |
| `fuel_sale.registered` | gas-stations-pod | core-pod (аудит, финансовый учёт) |

Каждое событие содержит: тип, идентификатор объекта-источника, метку времени, идентификатор пода-издателя. Все события фиксируются в журнале действий ядра.

**Реализация по фазам:**

M-OS-1 (Скелет): минимальная реализация внутри одного процесса. Очередь в памяти или таблица PostgreSQL. Поды не знают о существовании очереди — они вызывают `EventBus.publish(event)` через интерфейс. Конкретный транспорт скрыт за интерфейсом.

M-OS-2+: выделенный брокер сообщений (технология — отдельный RFC через R&I). Переход прозрачен для подов: меняется только реализация за интерфейсом `EventBus`.

Это реализует принцип 10 («pluggability»): транспорт шины заменяется без правки кода подов.

### 6. Редактируемость через конфигурацию (Configuration-as-data layer)

Принцип 10 m-os-vision.md §2 включает не только pluggability, но и editability — редактируемость бизнес-сущностей через интерфейс администратора без участия разработчика (Telegram msg 856, 2026-04-16).

**Правило для всех подов:** любая бизнес-сущность, которая может меняться в процессе эксплуатации, хранится в БД как данные, а не вшита в код.

К таким сущностям относятся: роли и права доступа, лимиты операций, процессы согласования (схемы BPM), шаблоны документов, конфигурации форм, параметры отчётов, правила уведомлений, бизнес-правила расчётов, реестр интеграций.

**Архитектурные следствия:**
- Код читает конфигурацию из БД и исполняет; логика поведения — в данных, не в коде
- Администратор меняет поведение системы через UI, не через деплой
- Каждое изменение конфигурации фиксируется в том же журнале действий (AuditLog), что и любое другое действие в системе (принцип 1)
- Реестр интеграций — тоже editable: администратор подключает, настраивает и отключает интеграции через UI

**Реализация:** Configuration-as-data layer реализуется в рамках Фазы M-OS-1 как обязательный must-have (m-os-vision.md §9). В текущем пилоте `cottage-platform-pod` (Фаза 3 coordinata56) хардкод допустим как временная мера до M-OS-1.

### 7. Как добавляется новый под: пример gas-stations-pod

**Без pod-архитектуры (было бы):**
- Пришлось бы создавать отдельное FastAPI-приложение с собственными пользователями
- Дублирование таблиц аудита, ролей, справочников
- Никакой возможности консолидированной отчётности
- Отдельный деплой, отдельная авторизация

**С pod-архитектурой (стало):**

Шаг 1 — Структура файлов:
```
backend/app/pods/gas_stations/
    __init__.py
    models.py          # Station, Dispenser, FuelType, Shift
    schemas.py         # Pydantic-контракты пода
    api/               # Роутеры FastAPI этого пода
    migrations/        # Alembic-миграции только для таблиц пода
    events.py          # События, которые публикует этот под
    roles.py           # Роли домена: gas_station_cashier и т.д.
```

Шаг 2 — Миграция: создаются только таблицы пода, таблицы ядра не трогаются.

Шаг 3 — Регистрация ролей: `gas_station_cashier`, `station_manager` добавляются в таблицу ролей ядра через seed-скрипт пода при его первом запуске. Ядровый модуль ролей не изменяется.

Шаг 4 — Подключение роутеров: в `backend/app/main.py` добавляется одна строка включения роутера пода. Остальной код `main.py` не меняется.

Шаг 5 — Проверка: сотрудник с ролью `station_manager` входит в M-OS и видит экраны АЗС. Сотрудник с ролью `site-foreman` видит только коттеджи. Пользователи, аудит, уведомления — общие, работают без дополнительной настройки.

Критерий успеха: `cottage-platform-pod` и его тесты не изменились ни в одном файле при добавлении `gas-stations-pod`.

### 8. Структура директорий в репозитории

```
backend/app/
    core/                    # Общее ядро (нужно всем направлениям без исключения)
        users/
        roles/
        audit/
        notifications/
        bpm/
        integrations/        # Anti-corruption layer
        chat/
        master_data/         # Организации, контрагенты, банки, статьи затрат
    pods/                    # Модули по бизнес-направлениям
        cottage_platform/
            models.py
            schemas.py
            api/
            migrations/
            events.py
            roles.py
        gas_stations/        # Добавляется в M-OS-1 как pod-lite
            ...
        metal/               # Dormant, структура создаётся при активации
        mkd/
        quarry/

    main.py                  # Точка входа, подключает роутеры всех подов
```

## Последствия

### Положительные

**Изоляция.** Ошибка в коде `gas-stations-pod` не влияет на `cottage-platform-pod`. Тесты пода проверяют только его логику.

**Параллельная работа команд.** Субагент backend-dev может работать над cottage-pod, пока другой рабочий процесс готовит gas-stations-pod, без конфликтов в файлах.

**Подключаемость (принцип 10).** Новое направление = новая папка в `pods/`, миграция, регистрация ролей. Ядро не трогается.

**Единый аудит.** Все действия во всех подах фиксируются через одну крипто-цепочку ядра (принцип 1). Консолидированная отчётность работает «из коробки».

**Эволюция без поломок (принцип 7).** Версионирование контрактов гарантирует, что изменение одного пода не ломает другие.

**Редактируемость без деплоя.** Администратор меняет роли, процессы, лимиты, шаблоны через UI — разработчик не нужен. Это обязательное требование принципа 10 (Telegram msg 856).

### Отрицательные

**Дисциплина контрактов данных.** Разработчики должны помнить: нельзя обращаться к чужим таблицам напрямую. Это поведение должно быть зафиксировано в департаментском регламенте backend и проверяться на ревью.

**Шина событий добавляет сложность.** Отладка асинхронных событий сложнее прямых вызовов. На M-OS-1 это смягчается минимальной реализацией в одном процессе.

**Разрастание ядра — главный риск.** При давлении сроков соблазн добавить «временно в ядро» то, что принадлежит поду. Контрмера: любое изменение состава ядра — через ADR, не молча. Это явно зафиксировано в m-os-vision.md §9.

### Риски

| Риск | Вероятность | Влияние | Контрмера |
|---|---|---|---|
| Ядро разрастается — поды тащат в него свою логику | Средняя | Высокое | Любое добавление в ядро — через ADR. Reviewer проверяет: не лежит ли в `core/` то, что должно быть в `pods/` |
| Нарушение изоляции — прямое обращение к таблице чужого пода | Средняя | Среднее | Правило в регламенте backend. Автоматическая проверка через линтер (запрет import между pod-папками) |
| Версии контрактов расходятся — v1 не снимается вовремя | Низкая | Среднее | Реестр подписчиков хранится явно. Снятие версии — через ADR-amendment |
| Шина событий становится узким местом | Низкая на M-OS-1 | Высокое на M-OS-2+ | Интерфейс `EventBus` скрывает транспорт с первого дня. Замена транспорта — без изменений в подах |

## Что явно не входит в этот ADR

- Выбор конкретного брокера сообщений для M-OS-2 — отдельный RFC через R&I
- Схемы данных конкретных подов — в ADR каждого пода при его активации
- Детальная ролевая модель прав доступа — отдельный ADR (M-OS-1)
- Механизм оффлайн-синхронизации для мобильных — отдельный ADR
- Модель multi-company (несколько юрлиц холдинга, внутренние проводки, сделки между юрлицами) — отдельный ADR в рамках Фазы M-OS-1 (m-os-vision.md §9 Foundation)
- Миграция кода `backend/app/api/` → `backend/app/pods/cottage_platform/` — отдельная задача Фазы M-OS-1. До M-OS-1 новый код пишется в текущей структуре `backend/app/`, не в pod-структуре. Переход не блокирует продолжение работы по Батчу C.
