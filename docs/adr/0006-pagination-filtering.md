# ADR 0006 — Пагинация и фильтрация (Phase 3)

- **Статус**: proposed
- **Дата**: 2026-04-15
- **Автор**: Архитектор (Claude Code, субагент `architect`)
- **Утверждающий**: Владелец (Мартин)
- **Контекст фазы**: Phase 3 — базовые CRUD API, кросс-срезовый компонент
- **Связанные ADR**: ADR-0001 (модели, TimestampMixin), ADR-0002 (FastAPI, SQLAlchemy 2.0, Pydantic v2), ADR-0005 (формат ошибок)

---

## Контекст

15 роутеров Фазы 3 имеют эндпоинты `GET /resource` (список). Нужно определить единый способ:

1. **Разбивки на страницы** — чтобы не отдавать все записи одним ответом.
2. **Фильтрации** — чтобы клиент мог запросить подмножество записей (например, дома на определённой стадии).
3. **Формата ответа** — обёртка вокруг массива объектов или сырой массив.

Объём данных мал: 85 домов, порядка тысяч транзакций, до ~10 000 строк в самых больших таблицах на горизонте MVP. Производительность не является ограничивающим фактором.

---

## Проблема

### Подвопрос 1: механизм пагинации

Нужно выбрать механизм разбивки результатов на части.

### Подвопрос 2: механизм фильтрации

Нужно выбрать способ передачи фильтров в запросе.

### Подвопрос 3: формат ответа списка

Нужно решить, оборачивать ли массив объектов в конверт с метаданными.

---

## Рассмотренные альтернативы

### Подвопрос 1: Пагинация

#### Вариант A (выбран): `offset` / `limit` в query-параметрах

```
GET /houses?offset=0&limit=25
```

**Плюсы:**
- Стандарт де-факто для REST API: любой разработчик знает этот паттерн.
- Прямо поддерживается SQLAlchemy: `.offset(offset).limit(limit)`.
- Swagger автоматически описывает два параметра — ничего дополнительно документировать не нужно.
- Произвольный переход на страницу (Владелец может сразу открыть «страницу 5», передав `offset=100`).
- Для 85 домов и тысяч транзакций нет никаких проблем производительности.

**Минусы:**
- При добавлении новых записей в начало таблицы (сортировка по `created_at DESC`) страница может «съехать» — запись попадёт на другую страницу между двумя запросами. Для MVP с малым объёмом и не real-time UI — несущественно.
- Дорогой `COUNT(*)` для очень больших таблиц — на наших объёмах не проблема.

---

#### Вариант B: Cursor-based пагинация (keyset pagination)

```
GET /houses?after=eyJpZCI6IDQyfQ==&limit=25
```

Курсор — закодированное значение последней записи предыдущей страницы.

**Плюсы:**
- Стабильна при параллельных вставках/удалениях.
- Производительна на очень больших таблицах (нет `OFFSET N` с прогоном по индексу).

**Минусы:**
- Нельзя перейти произвольно на страницу N — только «следующая» / «предыдущая».
- Сложнее реализовать: нужна сериализация/десериализация курсора, нельзя просто `.offset(N)`.
- Избыточная сложность для 85 домов и тысяч записей.
- Труднее тестировать и объяснять.
- Swagger-параметр `cursor` непрозрачен для ручного тестирования Владельцем.

**Почему отклонено**: неоправданная сложность при нашем объёме данных. При росте до миллионов строк — пересмотреть.

---

#### Вариант C: `page` / `page_size`

```
GET /houses?page=2&page_size=25
```

**Плюсы:**
- Интуитивно: «страница 2 из 10» понятна без знания технических деталей.
- Легко рассчитать `offset = (page - 1) * page_size` на стороне backend.

**Минусы:**
- Внутри всё равно `OFFSET` — те же минусы, что у Варианта A, но скрытые.
- Добавляет вычислительный слой конвертации `page → offset` без реальной пользы.
- `offset`/`limit` — более гибко: клиент сам контролирует смещение, что удобно при реализации «загрузить ещё».

**Почему отклонено**: `offset`/`limit` даёт ту же семантику с большей гибкостью и меньшей конвертацией.

### Подвопрос 2: Фильтрация

#### Вариант A (выбран): Плоские query-параметры

```
GET /houses?stage_id=3&house_type_id=1&deleted=false
```

Каждый фильтр — отдельный query-параметр. Фильтры определяются явно в Pydantic-схеме `XxxFilter`.

**Плюсы:**
- Swagger автоматически документирует все фильтры — Владелец видит их в выпадающем списке параметров.
- Читается в браузерной строке и в логах без декодирования.
- Pydantic валидирует типы фильтров (например, `stage_id: int | None`).
- Для 15 сущностей набор фильтров фиксирован и невелик — нет необходимости в DSL.

**Минусы:**
- Сложные условия («дома, у которых план > факт», «платежи в диапазоне дат») не выражаются плоскими параметрами. Для MVP эти запросы не нужны (дашборд — Фаза 5).
- При большом числе фильтров URL становится длинным, но читаемым.

---

#### Вариант B: DSL `?filter[field]=value`

```
GET /houses?filter[stage_id]=3&filter[house_type_id]=1
```

**Плюсы:**
- Однородный паттерн для любого количества фильтров.
- Можно добавить операторы: `filter[amount_cents][gte]=10000`.

**Минусы:**
- Swagger не умеет автоматически документировать `filter[*]` — нужны ручные аннотации.
- Труднее реализовать парсинг на FastAPI (нестандартный синтаксис query-параметров).
- Избыточная универсальность для фиксированного набора фильтров MVP.

**Почему отклонено**: потеря Swagger-автодокументации критична для Фазы 3.

---

#### Вариант C: `POST /resource/search` с телом запроса

```http
POST /houses/search
{"stage_id": 3, "house_type_id": 1, "amount_from": 10000}
```

**Плюсы:**
- Тело запроса позволяет передавать сложные структуры (вложенные условия, массивы).
- Нет ограничений длины URL.

**Минусы:**
- Нарушает HTTP-семантику: `POST` семантически создаёт ресурс, не читает.
- Нельзя закешировать (ни браузер, ни CDN не кешируют POST).
- Добавляет нестандартный эндпоинт поверх стандартного `GET /resource` — путаница для потребителя API.
- Избыточно для MVP с простыми фильтрами.

**Почему отклонено**: нарушение HTTP-семантики и избыточность.

### Подвопрос 3: Формат ответа списка

#### Вариант A (выбран): Конверт с метаданными

```json
{
  "items": [...],
  "total": 85,
  "offset": 0,
  "limit": 25
}
```

**Плюсы:**
- Клиент знает общее число записей (`total`) — можно отрисовать пагинацию («страница 2 из 4»).
- Единый формат для всех коллекций — фронтенд пишет один компонент-таблицы.
- `offset` и `limit` в ответе — клиент знает, какой срез он получил, без парсинга URL.

**Минусы:**
- Дополнительный `COUNT(*)` запрос к БД. На наших объёмах несущественно.

---

#### Вариант B: Сырой массив

```json
[{"id": 1, ...}, {"id": 2, ...}]
```

**Плюсы:**
- Минимальный размер ответа.
- Проще сериализовать.

**Минусы:**
- Нет информации о пагинации — клиент не знает, есть ли ещё записи.
- Нельзя добавить метаданные (фильтры, агрегаты) без изменения формата — это будет ломающим изменением API.
- Фронтенд не может отрисовать «страница N из M» без `total`.

**Почему отклонено**: отсутствие `total` делает пагинацию бессмысленной для UI.

---

## Решение

Принимается: **offset/limit + плоские query-параметры + конверт с метаданными**.

### Pydantic-схемы

```python
class PaginationParams(BaseModel):
    offset: int = Field(default=0, ge=0)
    limit: int = Field(default=25, ge=1, le=200)

class PagedResponse(BaseModel, Generic[T]):
    items: list[T]
    total: int
    offset: int
    limit: int
```

`PagedResponse[HouseRead]` — конкретный тип для Swagger.

### Пример эндпоинта

```python
@router.get("/", response_model=PagedResponse[HouseRead])
async def list_houses(
    stage_id: int | None = Query(default=None),
    house_type_id: int | None = Query(default=None),
    pagination: PaginationParams = Depends(),
    session: AsyncSession = Depends(get_session),
    _: User = Depends(require_role(UserRole.read_only)),
):
    service = HouseService(HouseRepository(session))
    return await service.list(stage_id=stage_id, house_type_id=house_type_id, **pagination.model_dump())
```

### Ограничения (MUST)

1. Максимальный `limit` — `200`. Запросы сверх лимита возвращают `VALIDATION_ERROR`.
2. По умолчанию `limit=25`, `offset=0`.
3. Soft-deleted записи (`deleted_at IS NOT NULL`) **не включаются** в результат по умолчанию. Параметр `include_deleted=true` доступен только роли `owner`.
4. Сортировка по умолчанию: `ORDER BY created_at DESC` для транзакционных таблиц, `ORDER BY id ASC` для справочников. Явный параметр сортировки в Фазе 3 не вводится (Фаза 5).
5. Каждая сущность определяет свой набор фильтров в отдельной `XxxFilter`-схеме, задокументированной в `schemas/`.

## Последствия

**Положительные:**
- Swagger автоматически показывает все фильтры как параметры — Владелец может тестировать фильтрацию прямо в браузере.
- Единый `PagedResponse` на все 15 сущностей — фронтенд Фазы 4 пишет один компонент-список.
- При добавлении нового фильтра — только новый Pydantic-параметр и WHERE-условие в репозитории.

**Отрицательные:**
- `COUNT(*)` на каждый запрос списка. При таблице в 10 000+ строк — оценить производительность по итогам Батча A.
- Без параметра сортировки клиент не управляет порядком — ограничение MVP, снимается в Фазе 5.

**Риски:**

| Риск | Митигация |
|---|---|
| `COUNT(*)` медленный на больших таблицах (`audit_log`, `budget_plan`) | Добавить `EXPLAIN ANALYZE` в тест Батча A; при необходимости — оценочный count через `reltuples` |
| Клиент передаёт `limit=10000` и получает всю таблицу | Жёсткий лимит `200` на уровне Pydantic-валидации |
| Разные наборы фильтров у разных сущностей ведут к несогласованности | Каждая `XxxFilter`-схема документируется в `schemas/` с описанием каждого параметра |

## Открытые вопросы

1. **Составные фильтры** (`amount_from`/`amount_to` для Payment): реализуются как два отдельных query-параметра (`amount_cents_from: int | None`, `amount_cents_to: int | None`). Решить при реализации Батча C.
2. **Полнотекстовый поиск** по имени подрядчика, ФИО клиента: отложено до Фазы 5; при необходимости — добавить параметр `q: str | None` с `ILIKE '%q%'` или `pg_trgm`.
3. **`total` через `reltuples`**: для `audit_log` при большом объёме возможно использование приблизительного count PostgreSQL вместо точного. Решить при появлении проблемы производительности.
