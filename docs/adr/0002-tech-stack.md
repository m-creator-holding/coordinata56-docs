# ADR 0002 — Выбор технологического стека MVP

- **Статус**: принято
- **Дата**: 2026-04-14
- **Автор черновика**: Архитектор (Claude Code, субагент `architect`)
- **Утверждающий**: Владелец (Мартин)
- **Контекст фазы**: Phase 0–4, MVP1 для `coordinata56`
- **Связанные ADR**: ADR-0001 (модель данных v1)

---

## Контекст

Проект «Координата 56» — операционно-управленческая ERP для коттеджного посёлка (85 домов бизнес-класса, 4 типа с опциями, полный цикл от ГПЗУ до сдачи). Задача системы: максимальная детализация план/факт по бюджету, стадиям строительства, закупкам, подрядчикам и договорам.

**Условия, определяющие выбор:**

1. **Единственный продуктовый разработчик — Мартин**, у которого нет опыта программирования. Код пишут ИИ-субагенты (Claude Code), но Мартин должен читать, понимать и принимать решения по коду. Стек не должен требовать глубоких специализированных знаний для поддержки.
2. **Команда из 17 ИИ-субагентов** работает параллельно. Технологии должны иметь широкую документацию и большую базу примеров в обучающих данных LLM — это напрямую влияет на качество генерируемого кода.
3. **Объём данных мал**: 85 домов, тысячи транзакций, до 10 млн строк за 10 лет. Производительность не является ограничивающим фактором на MVP1.
4. **Поддерживаемость и читаемость** важнее максимальной производительности.
5. **Дорожная карта (ROADMAP.md)** уже зафиксировала основной стек — этот ADR документирует обоснование выбора и рассмотренные альтернативы с trade-offs для каждого слоя.
6. ADR-0001 зафиксировал: PostgreSQL 16, SQLAlchemy 2.0, целочисленные PK, деньги в копейках (bigint), мягкое удаление, TIMESTAMPTZ. Стек backend должен быть согласован с этим решением.

---

## Проблема

Нужно зафиксировать выбор технологий для каждого слоя системы с обоснованием, чтобы:
- Все субагенты работали в единых рамках без разночтений
- Мартин понимал, почему выбрана именно эта технология, а не другая
- Будущие участники (или сам Мартин через год) могли понять контекст решения без потери времени на реконструкцию причин

Слои для выбора:
1. Язык и веб-фреймворк backend
2. ORM (объектно-реляционный маппер)
3. База данных
4. Frontend-фреймворк
5. UI-kit (библиотека компонентов интерфейса)
6. Сборщик/инструмент разработки frontend

---

## Рассмотренные альтернативы

### Слой 1. Язык + веб-фреймворк backend

#### Вариант A (выбран): Python 3.12 + FastAPI

**Что это**: Python — самый популярный язык в мире по индексу TIOBE 2024–2025. FastAPI — современный асинхронный фреймворк, построенный на стандарте ASGI и аннотациях типов Python.

**Плюсы:**
- Python читается почти как человеческий язык — минимальный барьер для новичка
- FastAPI автоматически генерирует интерактивную Swagger-документацию из кода — Мартин видит все API прямо в браузере без дополнительных усилий
- Встроенная валидация через Pydantic (библиотека проверки данных): декларируешь схему — фреймворк сам проверяет входящие данные
- Строгая типизация через аннотации Python — IDE и ИИ-агенты лучше «понимают» код, меньше ошибок при генерации
- Огромный массив документации, туториалов, примеров — LLM-агенты генерируют FastAPI-код высокого качества
- Асинхронность (async/await) из коробки — не нужно ничего настраивать для параллельной обработки запросов
- Отличная интеграция с SQLAlchemy 2.0 (уже зафиксированной в ADR-0001)

**Минусы:**
- Python медленнее Java/Go по сырой вычислительной мощности, но для нашего объёма данных это несущественно
- GIL (Global Interpreter Lock) ограничивает истинную параллельность вычислений, но для I/O-нагрузки (запросы к БД) это не проблема

**Почему выбран**: оптимальное сочетание читаемости, экосистемы, качества генерации кода субагентами и автоматической документации.

---

#### Вариант B: Python + Django (+ Django REST Framework)

**Что это**: Django — «батарейки включены», монолитный фреймворк с ORM, админкой, миграциями и всем остальным из коробки. DRF (Django REST Framework) добавляет инструменты для построения REST API.

**Плюсы:**
- Встроенная административная панель (Django Admin) — можно быстро посмотреть данные без фронтенда
- Очень зрелый, проверен в production миллионами проектов
- Встроенные миграции (не нужен Alembic отдельно)
- Огромное сообщество

**Минусы:**
- Django имеет собственную ORM, которая конфликтует с выбором SQLAlchemy 2.0 из ADR-0001: либо отказываться от SQLAlchemy, либо подключать Django «не по-Django»
- «Магия» Django (неявные поведения, Class-based Views, сигналы) сложнее понять новичку, чем явный FastAPI-код
- Производительность ниже FastAPI при одинаковой нагрузке
- Для чистого REST API Django избыточен: тащит много ненужного (шаблонизатор, сессии, CSRF и т.д.)
- Смешанная типизация — DRF-сериализаторы и Pydantic — два разных подхода в одном проекте

**Почему отклонён**: конфликт с SQLAlchemy из ADR-0001, избыточность для нашего сценария (чистый REST API + SPA-фронт), скрытая «магия» хуже для обучения новичка.

---

#### Вариант C: Python + Flask

**Что это**: минималистичный микрофреймворк, предоставляет только маршрутизацию и основные утилиты. Всё остальное добавляется вручную.

**Плюсы:**
- Предельно простой, почти нет «магии»
- Гибкость — можно собрать именно то, что нужно

**Минусы:**
- Нет встроенной валидации, типизации, документации — нужно подключать отдельно Flask-RESTX, marshmallow и т.д.
- Нет нативной асинхронности (async-поддержка через Quart — отдельный форк)
- Для нашего проекта потребует написать «вручную» то, что FastAPI даёт из коробки
- Ниже качество генерации кода субагентами по сравнению с FastAPI из-за меньшего числа современных примеров

**Почему отклонён**: требует больше ручной работы по настройке, нет автодокументации, нет нативной асинхронности. FastAPI решает те же задачи с меньшими усилиями.

### Слой 2. ORM (объектно-реляционный маппер)

**Определение**: ORM — прослойка между кодом Python и базой данных. Вместо написания SQL-запросов вручную описываешь таблицы как Python-классы, а ORM переводит операции с объектами в SQL автоматически.

#### Вариант A (выбран): SQLAlchemy 2.0

**Уже зафиксирован в ADR-0001.** Приводим здесь для полноты стека.

**Плюсы:**
- Стандарт де-факто в мире Python для работы с реляционными БД
- Версия 2.0 (2023) — полностью переработана с поддержкой async и строгой типизации
- Alembic (инструмент миграций от тех же авторов) — лучшая интеграция, нет конфликтов
- Позволяет при необходимости писать «сырой» SQL внутри ORM-контекста
- Отличная документация, огромное число примеров в LLM-обучающих данных

**Минусы:**
- Высокий порог входа для самостоятельного изучения (но для нас код пишут субагенты)
- Verbose (многословный) код по сравнению с Active Record подходом

---

#### Вариант B: Tortoise ORM

**Что это**: асинхронная ORM, изначально спроектированная для async/await. Синтаксис похож на Django ORM.

**Плюсы:**
- Нативная асинхронность без дополнительных адаптеров
- Более простой синтаксис для базовых операций
- Хорошая интеграция с FastAPI через Aerich (инструмент миграций)

**Минусы:**
- Значительно меньшая экосистема и документация по сравнению с SQLAlchemy
- Меньше примеров в обучающих данных LLM → ниже качество генерации кода субагентами
- Aerich (миграции) менее зрелый, чем Alembic
- Конфликт с уже принятым ADR-0001: модели уже описаны через SQLAlchemy

**Почему отклонён**: конфликт с ADR-0001, меньшая экосистема, худшее качество генерации кода.

---

### Слой 3. База данных

#### Вариант A (выбран): PostgreSQL 16

**Уже зафиксирован в ADR-0001.** Приводим для полноты.

**Плюсы:**
- Лучшая в классе поддержка JSON/JSONB (используем в ADR-0001 для contacts и audit_changes)
- TIMESTAMPTZ с корректной обработкой временных зон
- Надёжные FK с RESTRICT/CASCADE, check-констрейнты
- Полнотекстовый поиск через pg_trgm и GIN-индексы (понадобится в фазе M2+)
- Открытый исходный код, нет вендорной зависимости
- Самая богатая документация и поддержка от сообщества

**Минусы:**
- Тяжелее в настройке, чем SQLite
- Требует запущенного сервера (решено через Docker в Фазе 0)

---

#### Вариант B: MySQL 8 / MariaDB

**Что это**: популярная реляционная СУБД, долгое время доминировала в вебе (WordPress, Drupal, phpBB).

**Плюсы:**
- Широко известна, много хостингов поддерживают «из коробки»
- Производительность чтения в простых сценариях сопоставима с PostgreSQL

**Минусы:**
- Поддержка JSON значительно слабее (нет JSONB, нет GIN-индексов по JSON)
- TIMESTAMP без поддержки временных зон на уровне типа (решается через DATETIME + соглашения, но требует дисциплины)
- Исторически слабее в соответствии стандарту SQL (поведение NULL, GROUP BY без ONLY_FULL_GROUP_BY и т.д.)
- Меньше поддержка в SQLAlchemy и экосистеме Python для современных возможностей
- MariaDB с 2023 года расходится с MySQL всё сильнее — документация не взаимозаменяема

**Почему отклонён**: уступает PostgreSQL по JSONB, временным зонам, строгости SQL-стандарта. Нет преимуществ для нашего сценария.

---

#### Вариант C: SQLite

**Что это**: встраиваемая СУБД, хранит всю базу в одном файле. Не требует отдельного сервера.

**Плюсы:**
- Нулевая настройка, нет сервера, файл можно скопировать как бэкап
- Идеален для разработки и тестирования

**Минусы:**
- Нет конкурентной записи (WAL-mode помогает, но ограниченно)
- Нет полноценных временных зон, ограниченный ALTER TABLE
- Не подходит для production при одновременном доступе нескольких пользователей или сервисов
- Ограниченная поддержка типов (нет нативного boolean, нет TIMESTAMPTZ)

**Почему отклонён**: подходит только для локальной разработки и тестов. В production не применим. Можно использовать в тестах через `--database sqlite:///:memory:`, но это отдельное решение.

### Слой 4. Frontend-фреймворк

**Определение**: фреймворк для построения пользовательского интерфейса в браузере. Берёт на себя отрисовку, реакцию на действия пользователя, работу с состоянием.

#### Вариант A (выбран): React 18 + TypeScript

**Что это**: React — библиотека Meta для построения UI через компонентный подход. TypeScript — надмножество JavaScript с типами (описываешь, какие данные ожидаешь — компилятор проверяет).

**Плюсы:**
- Самая распространённая frontend-технология в мире: огромная база примеров, компонентов, туториалов
- LLM-агенты генерируют React+TS код наилучшего качества среди всех frontend-опций
- TypeScript добавляет статическую типизацию: IDE подсказывает ошибки до запуска — меньше багов, которые Мартин не может сразу отловить
- shadcn/ui (выбранный UI-kit) официально построен на React
- Экосистема: React Query (запросы к API), Zustand/Jotai (состояние), React Router (навигация) — зрелые, хорошо документированные
- Компонентный подход: Мартин может читать компонент как независимый «блок» интерфейса
- Hooks (useState, useEffect) — более явная модель состояния по сравнению с Vue Options API

**Минусы:**
- React сам по себе не даёт роутинг, управление состоянием, данными — нужны дополнительные библиотеки (это усложняет начало, но зато каждая библиотека понятна отдельно)
- JSX (HTML внутри JavaScript) — непривычный синтаксис на старте, но быстро читается

---

#### Вариант B: Vue 3 + TypeScript

**Что это**: прогрессивный фреймворк с хорошо структурированным «всё в одном файле» подходом (Single File Components). Composition API (Vue 3) близок к React Hooks по концепции.

**Плюсы:**
- Более «батарейки включены» по сравнению с React: роутер и state management — официальные пакеты от команды Vue
- SFC (Single File Component) — HTML, CSS, JS в одном `.vue` файле: многие находят это нагляднее
- Ниже порог входа для чтения шаблонов (ближе к обычному HTML с директивами)
- Vite — официальный инструмент сборки для Vue (тот же инструмент, что мы выбрали)

**Минусы:**
- Значительно меньше экосистема, чем у React (в 3–5 раз меньше пакетов, вакансий, примеров)
- LLM-агенты генерируют Vue-код худшего качества: меньше обучающих данных
- shadcn/ui (наш выбранный UI-kit) не имеет официальной Vue-версии. Существуют порты (shadcn-vue), но они отстают от оригинала и имеют меньшую поддержку
- Vue 2 → Vue 3 был болезненной миграцией, что создаёт риск повторения при Vue 4

**Почему отклонён**: главные причины — несовместимость с shadcn/ui и существенно худшее качество генерации кода субагентами.

---

#### Вариант C: Svelte / SvelteKit

**Что это**: компилируемый фреймворк — пишешь компоненты, компилятор генерирует минимальный «чистый» JavaScript без рантайм-библиотеки.

**Плюсы:**
- Наименьший размер бандла (итогового файла JavaScript)
- Синтаксис ближе к HTML + CSS + JS без JSX
- Высокая производительность отрисовки

**Минусы:**
- Наименьшая экосистема среди трёх вариантов
- LLM-агенты имеют наименьшую обучающую базу по Svelte → качество генерации хуже всего
- Нет официальной версии shadcn/ui под Svelte (есть порты, но незрелые)
- SvelteKit (full-stack фреймворк) — дополнительная концепция, усложняющая стартовый контекст
- Нишевый — сложнее найти примеры для специфических задач

**Почему отклонён**: наименьшая экосистема, худшее качество генерации кода для нашей команды субагентов, нет поддержки shadcn/ui.

---

### Слой 5. UI-kit (библиотека компонентов интерфейса)

**Определение**: готовые визуальные компоненты (кнопки, формы, таблицы, модалки, выпадающие списки), которые не нужно рисовать с нуля.

#### Вариант A (выбран): shadcn/ui

**Что это**: не традиционная «библиотека», а коллекция компонентов на основе Radix UI (примитивы без стилей) + Tailwind CSS (утилитарный CSS-фреймворк). Компоненты копируются в проект как исходный код — они ваши, а не внешняя зависимость.

**Плюсы:**
- Полный контроль над кодом компонентов: можно изменить любую деталь без борьбы с закрытой библиотекой
- Tailwind CSS + shadcn/ui — самое популярное сочетание в React-экосистеме 2024–2025
- Высокое качество генерации кода субагентами: огромная база примеров
- Radix UI обеспечивает доступность (accessibility) «из коробки» — правильное поведение для скринридеров, клавиатурная навигация
- Компоненты хорошо выглядят «по умолчанию» с минимальной настройкой
- Активная разработка, регулярные обновления

**Минусы:**
- Требует Tailwind CSS — дополнительная технология для изучения (набор утилитарных CSS-классов)
- Компоненты копируются в проект — при обновлении shadcn/ui нужно вручную мержить изменения

---

#### Вариант B: Ant Design (antd)

**Что это**: полноценная дизайн-система от Alibaba с сотнями компонентов, включая сложные (таблицы с сортировкой, деревья, графики, формы с валидацией).

**Плюсы:**
- Огромный набор готовых компонентов — меньше нужно делать самому
- Встроенные сложные компоненты: таблицы с пагинацией, фильтрами, сортировкой — нужны для нашего ERP
- Хорошая документация, много примеров
- Не требует Tailwind — всё через собственную систему стилей

**Минусы:**
- «Китайский» дизайн-язык — внешний вид отличается от современных западных SaaS-продуктов, потребуется кастомизация темы
- Тяжёлый: библиотека весит много, даже с tree-shaking
- Сложнее кастомизировать компоненты: нужно изучать систему токенов и CSS-переопределений
- Генерация кода субагентами хуже, чем для shadcn/ui (меньше современных примеров)

**Почему отклонён**: избыточная сложность кастомизации, устаревший дизайн-язык по умолчанию, и shadcn/ui покрывает все нужные компоненты для MVP1.

---

#### Вариант C: Material UI (MUI)

**Что это**: React-реализация Google Material Design — стандарт интерфейса Android-приложений.

**Плюсы:**
- Очень зрелая, огромная экосистема
- Полный набор компонентов
- Хорошая документация

**Минусы:**
- Сильный «Google-стиль» — ERP бизнес-класса с Material выглядит как мобильное приложение, потребует глубокой кастомизации
- Сложная система тематизации (sx prop, styled-components, ThemeProvider) — много неявного для новичка
- Тяжёлый бандл

**Почему отклонён**: дизайн-язык не соответствует нашим требованиям к бизнес-инструменту, сложная система кастомизации.

---

### Слой 6. Сборщик / инструмент разработки frontend

**Определение**: инструмент, который берёт исходный TypeScript + React код, склеивает модули, оптимизирует и выдаёт финальные файлы HTML/CSS/JS для браузера.

#### Вариант A (выбран): Vite

**Что это**: сборщик нового поколения, использует нативные ES-модули браузера в режиме разработки — не нужно пересобирать весь проект при каждом изменении файла.

**Плюсы:**
- Мгновенный старт сервера разработки (секунды вместо десятков секунд у webpack)
- Мгновенное HMR (Hot Module Replacement) — изменил компонент, видишь результат в браузере без перезагрузки страницы
- Официально рекомендован React-командой для SPA (Single Page Application) с 2023 года
- Простая конфигурация: vite.config.ts на 10 строк против webpack.config.js на 100+
- Отличная интеграция с shadcn/ui (официальный шаблон)
- Производит оптимизированный бандл для production через Rollup

**Минусы:**
- Не даёт SSR (Server-Side Rendering) «из коробки» — но для ERP с JWT-аутентификацией SSR не нужен
- Относительно молод по сравнению с webpack (но уже полностью зрелый для production)

---

#### Вариант B: Next.js

**Что это**: полноценный React-фреймворк со встроенными роутингом, SSR/SSG, API Routes, оптимизацией изображений и многим другим.

**Плюсы:**
- «Батарейки включены»: роутинг, SSR, оптимизация изображений — из коробки
- Очень популярен, огромная экосистема
- App Router (Next.js 13+) — современная модель компонентов с Server Components

**Минусы:**
- Избыточен для нашего сценария: ERP работает за аутентификацией, SEO не нужен, SSR не даёт нам ничего
- App Router (Next.js 13+) — принципиально новая модель, отличная от классического React. Server Components сложно понять новичку и генерировать субагентами
- Next.js добавляет слой абстракций поверх React: нужно изучать «правила Next.js» в дополнение к React
- При проблемах с Next.js труднее понять, это баг в React, Next.js или в вашем коде
- Накладывает ограничения на деплой (Vercel-ориентирован для полной функциональности)

**Почему отклонён**: избыточен для внутреннего ERP (SSR/SEO не нужны), App Router добавляет сложность, Vite+React полностью закрывает наши потребности с меньшим числом абстракций.

---

#### Вариант C: Create React App (CRA)

**Что это**: официальный «стартер» от команды React, обёртка над webpack. Был стандартом в 2018–2022.

**Плюсы:**
- Исторически самый известный способ начать React-проект

**Минусы:**
- **Официально устарел** (deprecated с 2023 года, команда React рекомендует Vite или Next.js)
- Медленный: webpack под капотом, нет нативных ES-модулей в dev-режиме
- Конфигурация скрыта за `eject` — или используешь как есть, или открываешь всё сразу без пути назад
- Не получает обновлений безопасности

**Почему отклонён**: официально устарел, медленнее Vite в 10–30 раз в dev-режиме, не получает поддержки.

## Решение

Принимается следующий технологический стек для MVP1 проекта `coordinata56`:

| Слой | Технология | Версия |
|------|-----------|--------|
| Язык backend | Python | 3.12 |
| Web-фреймворк backend | FastAPI | >= 0.111 |
| Валидация данных | Pydantic | v2 (встроен в FastAPI) |
| ORM | SQLAlchemy | 2.0 |
| Миграции БД | Alembic | >= 1.13 |
| База данных | PostgreSQL | 16 |
| Язык frontend | TypeScript | >= 5.0 |
| Frontend-фреймворк | React | 18 |
| Сборщик frontend | Vite | >= 5.0 |
| UI-kit | shadcn/ui | актуальный на момент инициализации |
| CSS-утилиты | Tailwind CSS | v3 |
| Контейнеризация | Docker + Docker Compose | актуальный |

**Дополнительные библиотеки (рекомендованы, финальный список фиксируется при реализации каждой фазы):**

- Backend: `python-jose` / `python-jwt` (JWT-токены, Фаза 2), `bcrypt` (хеширование паролей), `httpx` (HTTP-клиент для тестов), `pytest` + `pytest-asyncio` (тестирование)
- Frontend: `react-router-dom` v6 (роутинг), `@tanstack/react-query` v5 (серверное состояние и кеш запросов к API), `axios` или `fetch` (HTTP), `react-hook-form` + `zod` (формы и валидация на клиенте)

## Причины выбора

**Главный критерий — пригодность для команды «один новичок + LLM-субагенты»:**

1. **Качество генерации кода** — React+TS+FastAPI — вершина по насыщенности обучающими примерами в LLM. Субагенты генерируют код лучшего качества с меньшим числом галлюцинаций именно для этих технологий.

2. **Читаемость для Мартина** — Python читается почти как псевдокод. TypeScript в связке с shadcn/ui-компонентами — максимально декларативный код. Мартин может понять, что делает код, без опыта программирования.

3. **Автодокументация** — FastAPI генерирует Swagger UI автоматически. Мартин в браузере видит все API и может их тестировать вручную без дополнительных инструментов.

4. **Согласованность с ADR-0001** — PostgreSQL 16 + SQLAlchemy 2.0 уже зафиксированы. FastAPI — лучший партнёр для SQLAlchemy в экосистеме Python.

5. **Принцип минимальной достаточности** — каждая выбранная технология закрывает конкретную потребность. Не выбраны Next.js (SSR не нужен), Django (ORM конфликтует с ADR-0001), Ant Design (избыточная сложность кастомизации).

6. **Единый рантайм в контейнере** — все сервисы запускаются через `docker compose up`, что исключает «работает у меня, не работает у тебя» и упрощает будущий деплой на staging (Фаза 9).

## Последствия

**Положительные:**
- Вся команда субагентов работает в единых технологических рамках без разночтений
- Автоматическая Swagger-документация снижает потребность в отдельной API-документации
- TypeScript на фронтенде ловит ошибки типов до запуска — меньше runtime-ошибок
- shadcn/ui компоненты можно кастомизировать без борьбы с закрытой библиотекой
- Стек легко масштабируется на другие бизнесы холдинга (M8 в роадмапе)

**Отрицательные:**
- Tailwind CSS потребует начальной кривой освоения (читать классы типа `flex gap-4 px-2 rounded-lg`)
- Два языка (Python backend + TypeScript frontend) — Мартин читает код в двух синтаксисах
- Без SSR нет автоматической SEO-оптимизации (но ERP не нужно индексировать поисковиками)

**Риски:**

| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|-----------|
| Мартин не может читать сгенерированный Tailwind-код | Средняя | Низкое | Субагент `tutor` объясняет паттерны по запросу; shadcn-компоненты изолируют Tailwind внутри |
| Конфликт версий shadcn/ui при обновлении | Низкая | Среднее | Компоненты хранятся как исходный код в репозитории, обновление — ручное и контролируемое |
| Pydantic v2 breaking changes vs v1 при использовании сторонних библиотек | Низкая | Среднее | Фиксировать версии зависимостей в `pyproject.toml`; проверять совместимость при добавлении библиотеки |
| React 18 Concurrent Features — сложные кейсы при генерации кода | Низкая | Низкое | Субагенты используют только стабильные паттерны (useState, useEffect, React Query) |
