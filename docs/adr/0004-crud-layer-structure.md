# ADR 0004 — Структура CRUD-слоя (Phase 3)

- **Статус**: proposed
- **Дата**: 2026-04-15
- **Автор**: Архитектор (Claude Code, субагент `architect`)
- **Утверждающий**: Владелец (Мартин)
- **Контекст фазы**: Phase 3 — базовые CRUD API для 15 бизнес-сущностей
- **Связанные ADR**: ADR-0001 (модели, SoftDeleteMixin), ADR-0002 (FastAPI, SQLAlchemy 2.0, Pydantic v2), ADR-0003 (роли, JWT-dependency)

---

## Контекст

Фаза 3 вводит 15 бизнес-роутеров. 16 моделей уже описаны в `backend/app/models/`. Задача — выбрать единый паттерн организации кода между HTTP-слоем (FastAPI-роутер) и базой данных (SQLAlchemy-сессия), который:

- Применяется одинаково к каждой из 15 сущностей — нет «особых» роутеров с разным устройством.
- Читается Владельцем без опыта программирования: структура файлов объясняет сама себя.
- Масштабируется без переписывания при добавлении бизнес-логики в Фазах 5–8.
- Позволяет тестировать слои независимо (сервис можно тестировать без HTTP, репозиторий — без сервиса).

---

## Проблема

Нужно определить, сколько слоёв кода стоит между роутером FastAPI и вызовом `session.execute()` / `session.add()`, и как именно они взаимодействуют.

---

## Рассмотренные альтернативы

### Вариант A (выбран): Router → Service → Repository (три слоя)

**Устройство:**

```
app/
  routers/
    projects.py      # HTTP: принимает запрос, возвращает ответ, вызывает service
  services/
    project_service.py   # Бизнес-логика: валидация инвариантов, оркестрация
  repositories/
    project_repo.py  # Доступ к данным: все SQLAlchemy-запросы здесь
```

**Контракт каждого слоя:**

- **Router**: принимает HTTP-запрос, вызывает `Depends` для сессии, user_id и ролей, передаёт управление сервису, возвращает Pydantic-схему. Не знает про SQLAlchemy.
- **Service**: содержит бизнес-правила (например, нельзя удалить Project, если есть Houses). Вызывает методы репозитория. Не знает про HTTP.
- **Repository**: содержит SQLAlchemy-запросы: `get_by_id`, `list_paginated`, `create`, `update`, `soft_delete`. Не знает про бизнес-правила.

**Скелет на примере сущности `Project`:**

```python
# repositories/project_repo.py
class ProjectRepository:
    def __init__(self, session: AsyncSession):
        self.session = session

    async def get_by_id(self, project_id: int) -> Project | None:
        return await self.session.get(Project, project_id)

    async def list_paginated(self, offset: int, limit: int) -> tuple[list[Project], int]:
        ...  # SELECT + COUNT

    async def create(self, data: dict) -> Project:
        ...

    async def soft_delete(self, project: Project) -> None:
        project.deleted_at = datetime.utcnow()


# services/project_service.py
class ProjectService:
    def __init__(self, repo: ProjectRepository, audit_service: AuditService):
        self.repo = repo
        self.audit = audit_service

    async def get_or_404(self, project_id: int) -> Project:
        project = await self.repo.get_by_id(project_id)
        if not project or project.deleted_at:
            raise NotFoundError("Project", project_id)
        return project

    async def delete(self, project_id: int, actor_id: int) -> None:
        project = await self.get_or_404(project_id)
        # Бизнес-правило: нельзя удалить, если есть дома
        if await self.repo.has_houses(project_id):
            raise ConflictError("PROJECT_HAS_HOUSES")
        await self.repo.soft_delete(project)
        await self.audit.log(actor_id, "delete", "Project", project_id, before=project)


# routers/projects.py
router = APIRouter(prefix="/projects", tags=["projects"])

@router.delete("/{project_id}", status_code=204)
async def delete_project(
    project_id: int,
    session: AsyncSession = Depends(get_session),
    current_user: User = Depends(require_role(UserRole.owner)),
):
    service = ProjectService(ProjectRepository(session), AuditService(session))
    await service.delete(project_id, actor_id=current_user.id)
```

**Плюсы:**
- Ответственность каждого слоя однозначна — разработчик (субагент) знает, куда класть любой новый код.
- Сервис тестируется с мок-репозиторием без поднятия БД (быстрые unit-тесты).
- При добавлении бизнес-логики в Фазах 5–8 не нужно переделывать структуру — просто добавляются методы в сервис.
- Читаемость для Владельца: структура папок `routers/`, `services/`, `repositories/` объясняет назначение каждого файла.

**Минусы:**
- Больше файлов на старте — для 15 сущностей получится 45 файлов (по 3 на сущность) вместо 15.
- Для простых справочников (`Stage`, `HouseType`, `BudgetCategory`) сервис фактически пустой — он просто делегирует в репозиторий. Это «шаблонный» код.

### Вариант B: Router → Service (два слоя, прямые запросы в сервисе)

**Устройство:**

```
app/
  routers/
    projects.py
  services/
    project_service.py   # Здесь и бизнес-логика, и SQLAlchemy-запросы
```

Сервис содержит и бизнес-правила, и `session.execute()`. Роутер вызывает сервис.

**Плюсы:**
- На 33% меньше файлов (30 вместо 45 для 15 сущностей).
- Меньше переходов при чтении кода для простых операций.

**Минусы:**
- Сервис нельзя протестировать без реальной (или in-memory) БД — медленные тесты.
- По мере роста бизнес-логики сервис превращается в God Object, где вперемешку идут запросы и правила.
- При необходимости переиспользовать один запрос из двух разных сервисов — либо дублирование, либо приходится всё равно выделять слой доступа к данным.

**Почему отклонено**: экономия 15 файлов не оправдывает потерю тестируемости и неизбежный рефакторинг при росте логики.

---

### Вариант C: Router напрямую (плоско, через Depends-сессию)

**Устройство:**

```
app/
  routers/
    projects.py   # Здесь роутер, валидация, SQLAlchemy-запросы, аудит — всё
```

Роутер сам выполняет `session.execute()`, сам пишет в `audit_log`.

**Плюсы:**
- Минимум файлов: 15 штук на 15 сущностей.
- Быстрее на старте для тривиальных CRUD.

**Минусы:**
- Любое бизнес-правило (иммутабельность Payment, проверка совместимости опций) смешивается с HTTP-кодом. Через два батча роутеры станут нечитаемыми.
- Невозможно переиспользовать логику между эндпоинтами (например, `get_or_404` придётся дублировать).
- Тесты требуют полного HTTP-стека — нет быстрых unit-тестов.
- Противоречит принципу единственной ответственности, что критично для 17 параллельных субагентов: два агента могут одновременно менять один роутер по разным причинам.

**Почему отклонено**: при 15 сущностях и гарантированном росте бизнес-логики — неприемлемый технический долг уже к концу Фазы 3.

---

## Решение

Принимается трёхслойная архитектура: **Router → Service → Repository**.

Структура директорий:

```
backend/app/
  routers/           # HTTP: маршруты, статус-коды, Pydantic-схемы ответа
  services/          # Бизнес-логика: инварианты, оркестрация, аудит
  repositories/      # Данные: все SQLAlchemy-запросы
  schemas/           # Pydantic-схемы: Request/Response, отдельно от ORM-моделей
```

**Обязательные соглашения (MUST):**

1. SQLAlchemy-запросы пишутся **только** в `repositories/`. Ни роутер, ни сервис не импортируют `select`, `insert`, `update`.
2. Каждая сущность имеет три файла: `routers/X.py`, `services/X_service.py`, `repositories/X_repo.py`.
3. Сервис **не знает** про FastAPI (`Request`, `Response`, `status`). Роутер **не знает** про SQLAlchemy.
4. Ошибки транслируются через собственные исключения (`NotFoundError`, `ConflictError`, `PermissionError`) — роутер перехватывает их и преобразует в HTTP-ответы (см. ADR-0005).
5. Вызов `audit_service.log(...)` — в **сервисном** слое, не в роутере.
6. Конструирование зависимостей (`ProjectService(ProjectRepository(session), ...)`) — в теле endpoint-функции через `Depends`, не через глобальные синглтоны.
7. Для справочных сущностей без бизнес-правил (`Stage`, `HouseType`, `BudgetCategory`) — сервисный файл создаётся и содержит хотя бы `get_or_404`. Это сохраняет единообразие и оставляет место для будущих правил.

---

## Последствия

**Положительные:**
- Единый паттерн для 15 сущностей — субагенты не принимают архитектурных решений при написании каждого роутера.
- Быстрые unit-тесты сервисов с мок-репозиторием.
- Явная точка вставки аудита (сервисный слой) — нельзя случайно пропустить.
- Структура папок самодокументируема: Владелец видит `routers/`, `services/`, `repositories/` и понимает назначение без объяснений.

**Отрицательные:**
- 45 файлов бизнес-кода вместо 15. Для простых справочников часть кода — шаблонная.
- Нужен явный DI (внедрение зависимостей) в каждом endpoint — три строки `Depends` вместо одной.

**Риски:**

| Риск | Митигация |
|---|---|
| Субагенты нарушают слои (кладут SQLAlchemy в сервис) | Линтер-правило: `import select` запрещён вне `repositories/`; ревьюер проверяет по чек-листу |
| Сервисы справочников становятся пустыми файлами | Минимальный контент — `get_or_404` + docstring с описанием сущности |
| Зависимости между репозиториями (Payment требует Contract) | Репозитории не вызывают друг друга; кросс-сущностная логика — только в сервисе |

## Открытые вопросы

1. **Базовый класс `BaseRepository`**: стоит ли выделить общий класс с методами `get_by_id`, `list_paginated`, `soft_delete`? Рекомендация: да, реализовать в Батче A на примере `ProjectRepository` и наследовать остальные. Решение принять при начале реализации.
2. **`BaseService` с `get_or_404`**: аналогично — общий базовый класс снижает дублирование. Решить при реализации Батча A.
3. **Инъекция сервисов**: конструирование в теле endpoint избыточно при большом числе зависимостей. При появлении необходимости — рассмотреть FastAPI `lifespan` + dependency factory, не раньше.

## Amendments

### Amendment 2026-04-15 — директория `api/` вместо `routers/`

**Причина**: Фаза 2 уже поместила auth-роутер в `app/api/` (файл `app/api/auth.py`). Изменение директории на `routers/` создало бы несогласованность внутри проекта — два разных имени для одного слоя. Унификация важнее буквального следования имени, зафиксированному в ADR.

**Решение**: директория `app/routers/` → `app/api/`. Все роутеры Фазы 3+ создаются в `app/api/`. Структура `services/` и `repositories/` остаётся без изменений.

**Не влияет** на безопасность, производительность и совместимость API.

**Согласовано**: Координатором (ревью round 2, 2026-04-15).

### Amendment 2026-04-15 — хелпер `_make_service` в роутере

**Причина**: п.6 ADR 0004 предписывал конструирование зависимостей в теле каждой endpoint-функции. На практике для 5 endpoint-функций одного роутера это означает 5 идентичных блоков из 2–3 строк. Такое дублирование является источником ошибок (можно забыть передать `AuditService` в одном из endpoint).

**Решение**: вынесение конструирования в приватную функцию `_make_service(db)` внутри того же модуля роутера. Функция не является синглтоном — создаётся заново при каждом вызове. Семантически эквивалентно конструированию в теле endpoint.

**Применяется**: ко всем роутерам Батча A и далее.

**Согласовано**: Координатором (ревью round 2, 2026-04-15).

### Amendment 2026-04-18 — типизированные предикаты фильтрации в сервисном слое (уточнение MUST #1)

**Автор**: backend-director
**Дата**: 2026-04-18
**Триггер**: архитектурный аудит ADR-согласованности 2026-04-18, пункт C-03 (P1) — формальный конфликт MUST #1 ADR 0004 с `CompanyScopedService` из ADR 0011 §1.3.
**Связанные документы**: `docs/reviews/adr-consistency-audit-2026-04-18.md` §C-03, ADR 0011 §1.3.

**Причина**. MUST #1 в исходной редакции сформулирован буквально: «ни роутер, ни сервис не импортируют `select`, `insert`, `update`». По букве это запрещает сервису любую работу с типами SQLAlchemy. По духу — запрещает строить SQL-запросы вне репозитория.

Практика Батча A и Foundation M-OS-1 показала разрыв между буквой и духом:

1. Сервисы `Project`, `Contract`, `Contractor`, `Payment` формируют SQLAlchemy-выражения-предикаты (например, `Contract.status == status`, `Project.status != "archived"`) и передают их в репозиторий через параметр `extra_conditions: list[ColumnElement[bool]]`.
2. Базовый класс `CompanyScopedService` из ADR 0011 §1.3 работает по тому же паттерну: метод `_scoped_query_conditions(user_context)` возвращает `[Model.company_id == user_context.company_id]` — это **предикат**, а не запрос.
3. В обоих случаях запрос (`select(Model).where(...)`, `COUNT(*)`, `offset/limit`, `session.execute`) полностью строит репозиторий в `BaseRepository.list_paginated`.

Два разработчика, читающие ADR 0004 и ADR 0011 буквально, приходят к взаимоисключающим реализациям. Нужно зафиксировать договор явно.

**Решение**. MUST #1 уточняется двумя частями:

**MUST #1a (строго запрещено в сервисе и роутере):**
- Строить или выполнять SQLAlchemy-запросы: `select(...)`, `insert(...)`, `update(...)`, `delete(...)`, `session.execute(...)`, `session.get(...)`, `.subquery()`.
- Вычислять `COUNT(*)`, применять `offset`/`limit`, `order_by` вне репозитория.
- Импортировать `sqlalchemy.select`, `sqlalchemy.insert`, `sqlalchemy.update`, `sqlalchemy.delete`, `sqlalchemy.func` в модулях `services/` и `api/`.

**MUST #1b (разрешено в сервисе):**
- Формировать **типизированные предикаты** в виде `ColumnElement[bool]` — это объекты-выражения SQLAlchemy, не запросы. Примеры: `Contract.status == "active"`, `Project.company_id == ctx.company_id`, `Payment.amount_cents > 0`, комбинации через `and_`/`or_`.
- Импортировать ORM-модели (`from app.models.contract import Contract`) — они уже импортируются сервисом для тайп-хинтов возвратов.
- Импортировать `from sqlalchemy import ColumnElement` и опционально `and_`, `or_` для композиции предикатов.
- Передавать предикаты в репозиторий через параметр `extra_conditions: list[ColumnElement[bool]]` метода `BaseRepository.list_paginated`.

**Обоснование разрешения.** Предикат — это не запрос. Он не генерирует SQL и не обращается к БД. Репозиторий принимает его как параметр и сам строит запрос с `.where(predicate)`. Контроль над запросами (как и что выбирается, как считается `total`, как применяется `offset/limit`) остаётся в репозитории — а именно это защищает MUST #1 по духу: централизованность запросов, тестируемость, невозможность случайно обойти пагинацию или фильтр soft-delete.

**Применяется** к `CompanyScopedService`, `BaseService` и всем сервисам предметной логики, начиная с Батча A (ретроактивно, так как уже использовалось де-факто) и ко всем последующим разработкам M-OS-1+.

**Что НЕ разрешено даже в рамках MUST #1b:**
- Вызывать `.execute()`, `.scalar()`, `.scalars()`, `.all()` в сервисе — это обращение к БД, а не предикат.
- Строить подзапросы (`.subquery()`, `exists()`) в сервисе — это уже запрос, даже если он используется как предикат. Подзапросы выносятся в репозиторий как отдельный метод (например, `ContractRepository.has_active_payments_subquery()`), который возвращает `Select` или `ScalarSelect`, и сервис получает уже готовый объект для передачи обратно в `extra_conditions` через репозиторный интерфейс.
- Импортировать `AsyncSession` в сервис — сессия передаётся только репозиторию при конструировании.

**Impact на код.** Нулевой — текущая реализация `CompanyScopedService` и сервисов Project/Contract/Contractor/Payment уже соответствует MUST #1b. Это легализация существующего паттерна, а не введение нового.

**Согласовано**: backend-director (2026-04-18), передано Координатору на governance-подачу.
