# ADR 0017 — Hooks as Defense-in-Depth Layer: механическая защита от повторяющихся ошибок агентов

- **Статус**: proposed (черновик на ревью Координатора; не утверждать без governance approval)
- **Дата**: 2026-04-18
- **Автор**: architect (субагент-советник)
- **Утверждающий**: governance-director, затем Владелец (Мартин)
- **Контекст фазы**: M-OS-1 «Скелет» / горизонтальный: действует поверх всех фаз
- **Связанные документы**:
  - `docs/agents/CODE_OF_LAWS.md` — операционный кодекс, «вербальный» слой защиты
  - `docs/CLAUDE.md` — антипаттерник, фиксирует повторяющиеся ошибки
  - ADR 0010 (таксономия субагентов) — хуки применяются к действиям всех типов субагентов
  - ADR 0007 (audit-log) — события блокировки хуков пишутся в AuditLog
  - `docs/governance/verdicts/` — governance-verdicts для принятых ADR

---

## Проблема

В M-OS действует регламентная защита: CODE_OF_LAWS, CLAUDE.md, инструкции субагентов. Это «вербальный» слой — набор правил, которые агент должен помнить и применять самостоятельно.

Практика показала: вербального слоя недостаточно. Одни и те же ошибки повторяются многократно, независимо от того, сколько раз правило добавлялось в CLAUDE.md или регламент:

| Ошибка | Повторений | Источник фиксации |
|---|---|---|
| Литеральные пароли в коде/тестах | 3 | Phase 2 Round 2 BLOCKER-1; Phase 3 Batch A R1 P0-2; CLAUDE.md §Секреты |
| Raw SQL вместо ORM (фильтр после LIMIT) | 2 | Phase 3 Batch A R1 P0-1; CLAUDE.md §Данные |
| SendMessage к dormant субагентам | 7 | CLAUDE.md §Процесс (инцидент 2026-04-17 вечер — 7 отправок, 0 результатов) |

Паттерн ошибок один: агент на момент действия «не думает» о правиле, даже если оно прописано в регламенте. Чем сложнее задача, тем выше риск «проскочить» правило в рабочем потоке.

**Ключевой тезис**: регламент нельзя «автоматизировать» до конца — слишком много контекстно-зависимых правил. Но 80% повторных ошибок приходятся на небольшой класс механически проверяемых паттернов. Именно эту часть нужно перевести из «вербальной» в «механическую» защиту.

---

## Контекст

**Что такое Claude Code Hooks.** Это shell- или Python-скрипты, которые запускаются автоматически до или после действий Claude Code (Bash, Write, Edit, Read, вызов Agent и т.д.). Хук получает на stdin JSON-событие с описанием действия, проверяет его и возвращает код выхода:

- `0` — разрешить действие;
- ненулевой — заблокировать действие, вернуть агенту сообщение об ошибке.

Хуки настраиваются в `~/.claude/settings.json` (глобальный уровень) или `.claude/settings.json` внутри проекта (уровень репозитория).

**Ограничения стека и среды**: Python 3.12, bash, Docker. Хуки живут в файловой системе хоста, запускаются в том же процессе, что и Claude Code.

**Принцип минимальной достаточности**: создаём только хуки для подтверждённых повторных ошибок. Не пишем превентивный каталог на все возможные нарушения — это ведёт к false-positive-шуму и maintenance-бремени.

**Что уже решено Владельцем**: Hooks Phase 0 pilot одобрен (msg 1411). ri-director пишет список 5–7 хуков первого приоритета. Данный ADR определяет архитектурную рамку над этим планом.

## Рассмотренные варианты

### Вариант A — Только регламентная защита (текущее состояние)

**Механика.** Все правила записываются в CODE_OF_LAWS, CLAUDE.md, инструкции субагентов. Агент читает документы в начале сессии и следует им самостоятельно. При нарушении — добавляем новое правило в CLAUDE.md.

**Плюсы**:
- Нулевые инфраструктурные затраты: никаких скриптов, конфигов, логов
- Неограниченная гибкость: любое правило можно выразить текстом
- Один источник правды — текстовые документы, которые читает и архитектор, и агент

**Минусы**:
- Уже не работает: три класса ошибок повторились 12 раз после фиксации в регламенте
- Нет механической проверки: агент «помнит» правило с вероятностью, снижающейся при росте контекста задачи
- Нет логирования попыток нарушения: невозможно посчитать «сколько раз агент чуть не ошибся»
- Добавление правил не снижает повторение ошибок — лишь удлиняет CLAUDE.md

**Отклонён**: задокументированное несоответствие между «правило есть» и «ошибка всё равно происходит» доказывает недостаточность вербального слоя.

---

### Вариант B — Только хуки, отказ от регламента

**Механика.** Все правила поведения агентов кодируются в хуки. Регламентные документы сводятся к минимуму. CODE_OF_LAWS упраздняется или превращается в артефакт для людей, не для агентов.

**Плюсы**:
- Детерминированная проверка: хук либо блокирует, либо нет — нет «нечёткого следования»
- Аудит: каждая попытка нарушения логируется

**Минусы**:
- Большинство архитектурных и процессных правил нельзя выразить в хуке: «не добавляй в scope то, чего нет в требованиях», «сначала думай, потом пиши» — это контекст, не паттерн
- Хук может проверить синтаксис действия, но не семантику бизнес-решения
- Полный перевод регламента в хуки потребует сотен скриптов, которые невозможно поддерживать
- При росте числа хуков до 50+ — maintenance становится отдельным engineering-проектом
- Риск «накормить хуки» — агент, зная правило хука, может изменить форму действия так, чтобы хук не поймал нарушение

**Отклонён**: хуки покрывают структурные и синтаксические паттерны, но не заменяют смысловой регламент.

---

### Вариант C — Гибрид: регламент + хуки + code review (рекомендуется)

**Механика.** Три независимых слоя защиты, каждый ловит свой класс нарушений:

1. **Регламентный слой** (CODE_OF_LAWS, CLAUDE.md, инструкции субагентов) — правила смысла и контекста, которые нельзя автоматизировать.
2. **Хуки** — механические проверки структурных и синтаксических паттернов. Блокируют действие до его выполнения.
3. **Code review** (review-head) — семантическая проверка результата: архитектурное соответствие ADR, IDOR, бизнес-логика.

Слои дополняют, а не заменяют друг друга. Каждое правило живёт ровно в одном слое:
- Пароль в коде → хук (паттерн в тексте файла, механически проверяемо)
- IDOR в эндпоинте → code review (семантика, нужен взгляд на логику)
- Агент не согласовал ADR-отклонение → регламент + governance

**Плюсы**:
- Покрывает три класса нарушений, которые не перекрываются между слоями
- Хуки снижают нагрузку на code review: механическое отсеивается до review, ревьюер смотрит только на семантику
- Регламент не раздувается правилами, которые эффективнее автоматизировать
- Постепенный рост: Phase 0 — 5-7 хуков по реальным инцидентам; каталог расширяется только при документированных повторных ошибках

**Минусы**:
- Три слоя требуют координации: правило должно быть в одном месте, не дублироваться
- Хуки добавляют latency к каждому действию агента (ожидаемо < 100 мс, некритично)
- Maintenance хуков — ответственность governance (новый compliance-процесс)

## Решение

**Принят Вариант C — гибридная трёхслойная защита.**

### Архитектурная схема трёх слоёв

```
Действие агента (Write / Bash / Edit / Agent-вызов / SendMessage)
        │
        ▼
┌───────────────────────────────────┐
│  Слой 1: ХУК (pre-tool)           │  ← механическая блокировка
│  Паттерн? → блокировать + лог     │    до выполнения действия
│  Нет? → пропустить                │
└────────────────┬──────────────────┘
                 │ разрешено
                 ▼
┌───────────────────────────────────┐
│  Действие выполнено               │
└────────────────┬──────────────────┘
                 │
                 ▼
┌───────────────────────────────────┐
│  Слой 2: CODE REVIEW (post-tool)  │  ← семантическая проверка
│  review-head смотрит на результат │    результата
└────────────────┬──────────────────┘
                 │ одобрено
                 ▼
┌───────────────────────────────────┐
│  Слой 3: РЕГЛАМЕНТ (всегда)       │  ← контекстные правила,
│  CODE_OF_LAWS, CLAUDE.md, ADR     │    не покрытые хуками
└───────────────────────────────────┘
```

### Расположение файлов хуков

```
~/.claude/settings.json                       — глобальная регистрация хуков
/root/coordinata56/.claude/
  settings.json                               — project-level регистрация (переопределяет глобальную)
  hooks/
    pre-write-no-hardcoded-secrets.sh         — блокировка литеральных паролей
    pre-write-no-raw-sql-after-limit.sh       — блокировка raw SQL с postprocessing
    pre-bash-no-git-add-all.sh               — блокировка git add -A без проверки status
    pre-agent-no-sendmessage-dormant.py       — предупреждение о SendMessage к dormant
    pre-commit-secrets-scan.sh               — сканирование перед коммитом
    post-write-log.sh                         — лог каждого Write-события (для аудита)
    _lib/
      common.sh                              — общие функции: лог, форматирование ошибки
```

### Типы хуков и их семантика

| Тип | Когда срабатывает | Логика |
|---|---|---|
| `pre-tool` (Write, Edit) | До записи файла | Проверяет содержимое на паттерн; блокирует при совпадении |
| `pre-tool` (Bash) | До выполнения команды | Проверяет команду на запрещённый паттерн |
| `pre-tool` (Agent) | До вызова субагента | Предупреждает о проблемах конфигурации вызова |
| `post-tool` (Write) | После записи файла | Логирует факт записи (без блокировки) |
| `pre-commit` (git hook) | До `git commit` | Сканирует staged-файлы на секреты |

### Контракт хука

Каждый хук обязан соответствовать:

```
Входные данные:  stdin — JSON-событие Claude Code
Выход:           exit code 0 (разрешить) / non-zero (блокировать)
                 stderr — сообщение для агента при блокировке
Идемпотентность: повторный запуск на том же входе даёт тот же результат
Побочные эффекты: только append в log-файл; никаких изменений кода
Тайм-аут:        < 2 секунд; зависание = auto-pass (не должен блокировать работу)
```

Формат сообщения при блокировке (stderr):

```
HOOK BLOCKED: <hook-name>
Rule: <одна строка — какое правило нарушено>
Evidence: <цитата из кода/команды — то, что сработало>
Fix: <конкретное действие для исправления>
Ref: <ссылка на правило: CLAUDE.md §X или ADR-NNNN>
```

### Логирование

Все события блокировки пишутся в `/var/log/coordinata56-hooks/hooks.log`:

```
<ISO-8601 timestamp> [BLOCKED|PASSED|WARN] <hook-name> <agent-session-id> <action-type> <brief>
```

Ротация: `logrotate`, 7 дней, gzip. Директория создаётся при установке пилота.

Критические блокировки (секреты, коммиты с паролями) дополнительно пишутся в AuditLog (ADR 0007) через вызов `POST /internal/audit` при наличии запущенного backend. При недоступности backend — только в файл.

### Governance хуков

Аналогично ADR: каждый новый хук требует:

1. Описания прецедента — какая повторная ошибка зафиксирована и сколько раз
2. Черновика хука с описанием логики
3. Approval governance-director
4. Записи в Hooks Catalog (`docs/hooks-catalog.md`)

Хуки не создаются профилактически. Только при наличии документированного повторного нарушения.

### Правило «одного места»

Правило, переведённое в хук, **удаляется из CLAUDE.md** (чтобы не дублировалось). В CLAUDE.md остаётся только краткая ссылка: `# Секреты → автоматически контролируется хуком pre-write-no-hardcoded-secrets`. Это принципиально: два источника одного правила создают рассинхронизацию при обновлении.

## Последствия

### Положительные

- Механическая блокировка 80% повторных ошибок: паттерны секретов, raw SQL, git add -A — проверяются до выполнения действия
- Снижение нагрузки на code review: review-head перестаёт ловить структурные нарушения, фокусируется на семантике
- Аудит попыток нарушения: впервые появляется возможность считать сколько раз хук поймал нарушение (метрика качества агентов)
- CLAUDE.md сокращается: автоматизированные правила выносятся в хуки, файл остаётся живым и читаемым

### Отрицательные

- Latency на каждое действие агента: < 100 мс на хук (bash), < 500 мс (python). При 10 хуках — до 1 с накладных расходов на действие
- False positives: хук может заблокировать легитимное действие (например, тестовая строка с паролем в комментарии). Каждый false positive должен быть разобран и либо исправлен в хуке, либо добавлен в allowlist
- Maintenance burden: хуки нужно обновлять при изменении регламента. Governance-director несёт ответственность за Hooks Catalog
- Новый compliance-слой: кто пишет хуки (ri-director / backend-head), кто ревьюит (governance-director), кто проверяет coverage (governance-auditor) — роли должны быть закреплены

### Нейтральные

- Хуки не изменяют результат работы агента — только блокируют или пропускают действие
- Агент видит причину блокировки и может исправить действие самостоятельно
- Phase 0 (5-7 хуков) не требует изменений в схеме БД и API

## Риски

| ID | Описание | Вероятность | Последствие | Митигация |
|---|---|---|---|---|
| R1 | False positive блокирует легитимное действие агента | Средняя | Агент застревает, задача не выполняется | Allowlist в хуке; procedure — при блокировке агент эскалирует к Координатору с описанием |
| R2 | Хук падает с ошибкой / зависает | Низкая | Если auto-pass — нарушение проходит; если auto-block — агент застревает | Тайм-аут 2 с + auto-pass; тест хука перед регистрацией |
| R3 | Правило дублируется в хуке и в CLAUDE.md — рассинхронизация | Средняя | Агент следует устаревшему правилу в CLAUDE.md, хук блокирует по новому | Правило «одного места»: при переводе в хук — удалить из CLAUDE.md |
| R4 | Рост Hooks Catalog до 50+ без governance → maintenance-ад | Низкая для Phase 0, высокая без контроля | Хуки начинают конфликтовать, false positive растёт, агенты теряют производительность | Governance-gate на каждый новый хук; quarterly review Catalog |
| R5 | Агент обходит хук, изменив форму действия | Низкая | Хук бесполезен для smart-обхода | Паттерн хука должен покрывать форму намерения, не только синтаксис; review при каждом подозрении |
| R6 | `/var/log/coordinata56-hooks/` не создана — хуки молча падают | Средняя | Нет логирования блокировок | Скрипт инициализации создаёт директорию при Phase 0 setup |

## Путь миграции (Migration Path)

**Phase 0 (Hooks Pilot, ~2 дня):**
- ri-director разрабатывает 5–7 хуков первого приоритета по подтверждённым инцидентам
- Регистрация в `/root/coordinata56/.claude/settings.json`
- Создание `/var/log/coordinata56-hooks/`, настройка logrotate
- Тест каждого хука: (а) легитимное действие → пропускает; (б) нарушение → блокирует с нужным сообщением
- governance-director: approval Hooks Catalog v0.1 (5–7 записей)
- Правила, ушедшие в хуки, помечаются в CLAUDE.md ссылкой на хук

**Phase 1 (Hooks Catalog, ~1 неделя после Phase 0):**
- По результатам Phase 0 — анализ false positives и false negatives
- Создание постоянного документа `docs/hooks-catalog.md` с таблицей: hook name / прецедент / дата добавления / ответственный
- Выработка governance-процедуры для новых хуков (аналог ADR-процесса, упрощённый)
- При накоплении 3+ новых инцидентов — создание следующей волны хуков

**Phase 2 (Hooks in CI, по мере готовности CI-pipeline):**
- Перенос части pre-commit хуков в GitHub Actions (`jobs: hooks-check`)
- Цель: проверки работают не только локально, но и при PR в main
- Не обязателен для M-OS-1; рассматривается при установке полноценного CI-gate (ADR 0013 §CI)
