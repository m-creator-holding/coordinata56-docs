# CLAUDE.md — coordinata56

Живой документ-антипаттерник: правила, которые родились из реальных ошибок в этом проекте. Принцип Бориса Черни (Anthropic): каждый раз, когда я ошибаюсь — добавляем сюда строку, чтобы не повторял. Ориентир по объёму — ~100 строк; при росте переносим устоявшееся в регламенты субагентов / хуки / скилы.

Этот файл загружается Claude Code автоматически при работе в `/root/coordinata56/`.

## Перед каждой задачей — обязательно прочесть

1. Этот CLAUDE.md (антипаттерник проекта)
2. **Свод законов**: `docs/agents/CODE_OF_LAWS.md` (навигация по всем регламентам)
3. **Регламент своего уровня иерархии**: `docs/agents/regulations/<coordinator|director|head|worker>.md`
4. **Регламент своего направления**: `docs/agents/departments/<своё>.md` (если применимо)
5. ADR, относящиеся к задаче
6. Должностная инструкция своей роли в `~/.claude/agents/<имя>.md` (если ты субагент)

## Процесс

- **Reviewer — до `git commit`**, не после. Работает на `git diff --staged`. См. регламент v1.3 §1.
- **Отклонения от ADR требуют согласования Координатора** — не молча. Если ADR неоднозначен — сначала amendment, потом код. Регламент v1.3 §2.
- **Фаза за фазой, без перепрыгиваний.** Следующая фаза не стартует, пока текущая не закрыта по DoD.
- **Координатор не задаёт Владельцу вопросы, ответы на которые уже есть в документах.** Перед эскалацией Владельцу — обязательно проверить: phase-N-scope.md, User Stories, модели (`backend/app/models/*`), ADR, CLAUDE.md, предыдущие батчи. Если ответ выводится из документов — Координатор решает сам и идёт дальше. К Владельцу — только реально блокирующие бизнес-решения, не описанные нигде. *(Правило от Владельца 2026-04-15 msg 566: «работай по регламенту дорожной карты, не допускай таких ошибок».)*
- **Строгая цепочка делегирования: Президент → Директор → Начальник отдела → Сотрудник.** Координатор (Президент) не передаёт задачи Сотрудникам или Начальникам отделов напрямую — только Директорам направлений. Директор распределяет дальше. Исключение — Советники (architect, analyst, legal, construction-expert, tech-writer, memory-keeper, tutor, data-analyst): они вне иерархии, к ним Координатор может обращаться напрямую (это консультация, не задача). Если у направления нет активного Директора — его надо активировать, не идти в обход. *(Правило от Владельца 2026-04-16 msg 665: «ты президент ты можешь ставить задачу только директорам». Отменяет маршруты XS→Worker и S→Head→Worker из v1.4 §5.)*
- **Pod-архитектура M-OS** — ADR 0009. Таксономия субагентов — ADR 0010 (5 типов: executive, core_department, domain_pod, governance, advisory). Pod-specific документы — в `docs/pods/<pod>/`, M-OS-wide — в `docs/`.
- **Паттерн «Координатор-транспорт» (v1.6).** В Claude Code субагенты технически не могут вызывать других субагентов (официальное ограничение Anthropic: «subagents cannot spawn other subagents»). Цепочка реализуется через Координатора-транспорт: Координатор последовательно делает Agent-вызовы по стадиям — Брифинг у Директора → Распределение у Head → Исполнение Workers → Ревью Head → Вердикт Директора → Коммит. Координатор не принимает решений за них, только передаёт бриф/результат дословно между уровнями. Ключевое правило: **Координатор формулирует Директору «напиши бриф для <role>», а не «сделай <work>»** — иначе Директор примет задачу на исполнение и нарушит свою статью 15. Если Координатор ошибся в формулировке и сказал «сделай» — Директор обязан отказать и вернуть бриф. Полный регламент паттерна — `docs/agents/regulations_addendum_v1.6.md`. *(Правило от Владельца 2026-04-16 msg 733 «директор не искал сам а лишь поручал» + msg 748 «реализовывай путь 1 и начинай писать новый регламент». Инцидент-пример: 2026-04-16 ri-director sensing.)*

## Данные и БД

- **Никаких фильтров коллекций постобработкой в Python после `LIMIT`.** Если фильтруем список — условие в SQL `WHERE`; `total` считается тем же `WHERE` через `SELECT COUNT(*)`. Иначе `total` врёт и пагинация на фронтенде сломается. *(Поймано: Phase 3 Batch A step 2, Round 1, P0-1)*
- **Enum в миграции Alembic — значения должны совпадать с `.value` Python-enum** (регистр включительно). При `native_enum=False` создаётся CHECK, несовпадение ломает raw-SQL INSERT. *(Поймано: Phase 3 Batch A step 1)*
- **Миграции — обязательный round-trip**: `alembic downgrade -1 && alembic upgrade head` чисто, без ошибок. Без round-trip миграция не считается готовой.

## Секреты и тесты

- **Никогда не литералить пароли, токены, секреты** — ни в `src/`, ни в `tests/`, ни в `conftest.py`, ни в фикстурах. Пароли — `secrets.token_urlsafe(16)`. Конфиги — `os.environ.get(...)` с дефолтом для dev. *(Регламент v1.3 §3. Повторяющаяся ошибка: Phase 2 Round 2 BLOCKER-1, Phase 3 Batch A step 2 Round 1 P0-2.)*
- **Testовые фикстуры не должны содержать production-подобных паролей.** Всегда случайная генерация на сессию pytest.

## API

- **Формат ошибок — всегда ADR 0005**: `{"error": {"code", "message", "details"}}`. Никаких `{"detail": "..."}` в ответах. `HTTPException` и `RequestValidationError` перехватываются глобальными handler-ами и приводятся к единому формату.
- **Пагинация — всегда envelope ADR 0006**: `{"items", "total", "offset", "limit"}`. Никогда голый массив. `limit` клиппится к 200.
- **Write-эндпоинт без `audit_service.log()` — это дефект.** Аудит — в той же транзакции, что и запись. ADR 0007.
- **Вложенные ресурсы (`/parents/{pid}/children/{cid}`) — всегда проверяй принадлежность.** `child.parent_id == pid` обязательно. Несовпадение → 404 (не 403 — не раскрывай существование чужого ресурса). *(Поймано: Phase 3 Batch A step 4, P1-1, IDOR в HouseConfiguration.)*

## Код

- **Никаких `# type: ignore` / `# noqa` без комментария-обоснования.** Если действительно нужно — одна строка с причиной.
- **Комментарии — только "почему".** "Что" — читается по коду. Никаких комментариев типа `# создаём пользователя` над `create_user(...)`.
- **Директория FastAPI-роутеров — `backend/app/api/`** (не `routers/`). Исторически, зафиксировано в ADR 0004 Amendment от 2026-04-15.

## Git

- **`git add -A` запрещён без просмотра `git status --short`** — риск закоммитить `.env`, временные файлы, чужие правки.
- **Когда фоновые субагенты работают — НЕ делать `git add -A`.** Они могут застейджить свои файлы, и ваш коммит подберёт чужую работу до её ревью. Стейджить только конкретный список файлов: `git add path/to/file1 path/to/file2`. *(Поймано: 2026-04-15, qa Батча A попал в коммит регламента v1.4 без ревью.)*
- **Коммит-сообщения — почему, а не что.** "Adds X" — плохо. "Add X because the review caught that Y breaks when Z" — хорошо.

## Как добавлять правила сюда

1. Ошибка зафиксирована в ревью / замечена в работе.
2. Короткая строка в соответствующей секции: сам паттерн + курсивом краткая ссылка на случай, где поймалось.
3. Если правило уже автоматизировано (pre-commit hook, lint rule, регламент субагента v1.x) — удалить отсюда, чтобы не дублировать.
4. При росте файла >150 строк — рефакторить: устоявшиеся правила переносить в регламент, в CLAUDE.md оставлять только свежие.
