## [0.5.1] — 2026-04-19 (Sprint 2 Volna A)

### Добавлено

**Sprint 2 Volna A — Event Bus и интеграционный слой (US-04 … US-07)**

- **US-04 BusinessEventBus:** шина деловых событий — маршрутизация событий между модулями (платёж одобрен, договор подписан). Подписчики регистрируются декоратором, ядро не знает о конкретных обработчиках.
- **US-05 AgentControlBus:** шина управления ИИ-субагентами — сигналы запуска, приостановки и завершения агентов проходят через единую точку, не напрямую между сервисами.
- **US-06 ACL IntegrationAdapter:** слой защиты от внешних форматов (Anti-Corruption Layer). Входящие данные из внешних систем (1С, банки) преобразуются в внутренние модели ещё на границе — внутренний код никогда не видит чужие структуры.
- **US-07 Pluggability container:** контейнер подключаемых модулей. Любой новый адаптер, агент или интеграция регистрируется в одном месте и подключается/отключается без правки ядра.

**ADR и архитектурные решения**

- **ADR-0024 Verification Gate:** трёхшаговый DoD для live-активаций: `SHOW <param>` в runtime PostgreSQL + triggered-event verify в объектном хранилище + E2E smoke. Ни один live-режим не считается включённым без прохождения всех трёх шагов.
- **ADR-0025 1С (draft):** архитектурное решение об интеграции с 1С через Anti-Corruption Layer. Адаптер находится «на полке» (60/60 тестов написаны, живых вызовов нет) до production-gate.

**1С adapter skeleton**

- Адаптер 1С: 60 тестов покрывают сценарии импорта платёжных поручений, справочника контрагентов и актов выполненных работ. Живых вызовов нет — файл лежит готовым для подключения.

**Operations UI — вайрфреймы (3 экрана)**

- Дашборд операционного отдела: сводные показатели по домам, договорам, платежам за период.
- Экран «Дома»: список объектов с фильтрацией по стадии и ответственному прорабу.
- Экран «Отчёты»: выгрузка план/факт в разрезе статей и периодов.

**M-OS-2 планирование**

- Voice brief: концепция голосового журнала прораба для полевого ввода данных.
- Innovation digest: обзор приоритетов M-OS-2 (1С первым, Voice флагманом).
- Benchmarks: сравнение подходов к интеграции голосового ввода.

### Безопасность

- **Email-mask утилита (`mask_email`):** функция маскирования адресов электронной почты добавлена к существующим 13 полям ПД. Формат: `j***@example.com`.

## [0.5.0] — 2026-04-19 (Wave 11 Round 3 + Round 4)

### Исправлено

**Wave 11 — регрессия закрыта (Round 3 + Round 4)**

- **BUG-R4-1:** ошибка авторизации Владельца при обращении к матрице прав через RBAC v2 — исправлена проверка `is_holding_owner` в декораторе `require_permission`.
- **BUG-R4-2:** некорректная фильтрация пользователей по компании в эндпоинте `/admin/users` — условие перенесено из Python-кода в SQL `WHERE` (нарушение правила из CLAUDE.md).
- **BUG-R4-3:** миграция US-01 завершалась ошибкой при повторном применении — добавлена проверка существования колонки `company_id` перед `ALTER TABLE`.
- **BUG-R4-4:** IDOR-уязвимость: Владелец холдинга мог через API прочитать объект другой компании без явной проверки — добавлена валидация `child.company_id == active_company_id` даже для holding owner в read-операциях.

**Резервное копирование — WAL Fix B**

- Непрерывный WAL-архив в Yandex Object Storage: PostgreSQL переведён в режим `archive_mode=on`, архивная команда копирует каждый завершённый сегмент журнала транзакций в облако. Данные не теряются даже при аварии между ежедневными дампами.
- Ежедневный SQL-дамп (`pg_dump`) с хранением 30 дней (было 14). Выполняется в 03:00 UTC автоматически.
- Итого три уровня защиты данных: WAL-архив (непрерывно) + SQL-дамп (ежедневно) + том Docker (локально).

### Безопасность

- **Hardcoded password sweep:** фиксированные пароли удалены из всех mock-фикстур и тестовых файлов. Пароли в тестах теперь генерируются через `secrets.token_urlsafe(16)` (правило из CLAUDE.md и Конституции ст. 79).

## [0.4.0] — 2026-04-19

### Добавлено

**Wave 11 — Admin UI (4 страницы)**

- `/admin/permissions` — визуальная матрица прав: кто (роль) что может делать (действие) с какими данными (ресурс). Режим редактирования с батчевым сохранением изменений, фильтрация по роли через URL-параметр, экспорт в CSV, история изменений прав конкретной роли.
- `/admin/companies` — реестр компаний с тремя вкладками: детали, пользователи компании, настройки. Вкладка «Настройки» содержит 7 полей: режим НДС (RadioGroup), валюта (disabled, задаётся при создании), часовой пояс, рабочая неделя, система единиц, бизнес-сегмент, ответственный. Бухгалтер видит вкладку в режиме только чтение.
- `/admin/users` — список пользователей с поиском и фильтрацией по роли. Назначение роли inline через Select прямо в строке таблицы (v1.1). Блокировка пользователя и просмотр детальной карточки.
- `/admin/rules` — бизнес-правила в трёх категориях: Финансы, Кадры, Процессы. Дерево категорий слева, таблица правил справа, просмотр исторических снимков значений.

**Sprint 1 — Foundation (US-01, US-02, US-03)**

- **US-01 Multi-company isolation:** поле `company_id` добавлено во все таблицы с деловыми данными. Базовый класс `CompanyScopedService` автоматически фильтрует запросы по активной компании пользователя. Суперадмин (`is_holding_owner=True`) получает bypass фильтра.
- **US-02 JWT с компаниями:** токен содержит клеймы `company_ids: list[int]` и `is_holding_owner: bool`. Клиент передаёт активную компанию через заголовок `X-Company-ID`. `UserContextMiddleware` обрабатывает заголовок и передаёт `UserContext` в сервисы через `ContextVar`.
- **US-03 RBAC v2 (fine-grained):** таблица `role_permissions` — матрица прав как данные, не код. Функция `can(user_context, action, resource) -> bool`. Декоратор `require_permission(action, resource_type)` заменяет `require_role` на всех write-эндпоинтах. `require_role` сохранён как deprecated alias — все существующие тесты продолжают работать.

### Изменено

- Все write-эндпоинты API переведены с `require_role` на `require_permission`.
- JWT-структура расширена: добавлены клеймы `company_ids` и `is_holding_owner`.
- Онбординг-документация: обновлён `docs/knowledge/onboarding/developer.md` с описанием Sprint 1 и паттерна `CompanyScopedService`.

### Техническое

- Admin UI работает через MSW (Mock Service Worker) для страницы правил — подключение к backend API запланировано в Sprint 2.
- Настройки компании (вкладка Settings): frontend реализован по wireframe v1.1 (7 полей), синхронизация с backend API — после Wave 11 (задача backend-director).

## [0.3.0] — 2026-04-17

### Добавлено

- **M-OS-0:** реструктуризация в pod-архитектуру. Первый pod — `cottage-platform`. ADR 0008–0010 приняты. CODE_OF_LAWS v1.1.

- Структура директорий: `backend/app/routers/` переименована в `backend/app/api/` (ADR 0004 Amendment).
- Frontend Admin: страницы Companies (FE-W1-1), Users (FE-W1-2), Roles (FE-W1-3) влиты в main.
